### 1. [close-issue.ts:107-124] Rally Issue Tracking Not Supported in New closeViaDirect()

**Severity:** Critical
**Location:** `/home/eltmon/Projects/overdeck/src/lib/lifecycle/close-issue.ts:107-124`

**Problem:**

The new `closeViaDirect()` function only handles GitHub and Linear issues:

```typescript
async function closeViaDirect(
  ctx: LifecycleContext,
  comment?: string,
): Promise<StepResult> {
  const step = 'close-issue:transition';

  if (ctx.github) {
    return closeGitHubDirect(ctx, comment);
  }

  // Try Linear
  const linearApiKey = getLinearApiKey();
  if (linearApiKey) {
    return closeLinearDirect(ctx, linearApiKey);
  }

  return stepFailed(step, 'No tracker available and cannot determine issue type');
}
```

However, the OLD close-out.ts monolithic function (line 298-353) explicitly handled Rally issues:

```typescript
if (ctx.isGitHub && ctx.owner && ctx.repo && ctx.number) {
  // GitHub handling
} else {
  // Linear issue — could also be Rally via RallyTracker
  const linearApiKey = getLinearApiKey();
  if (!linearApiKey) { /* error */ }
  // Linear SDK calls...

```

And in the OLD server endpoint at `/api/issues/:issueId/close` (before lifecycle refactor), there was NO explicit Rally handling — it delegated to the lifecycle module. The old close-out.ts also didn't have explicit Rally handling in the close step.

**BUT** — the issue-data-service.ts DOES have RallyTracker usage (line 885, 925), showing Rally is a supported tracker.

**Impact:**

- If an issue is a Rally issue (not GitHub, not Linear-configured), the new `closeViaDirect()` will **fail silently** with "No tracker available" instead of attempting to close it via RallyTracker.
- Issues that previously closed (via the old endpoint which might have had indirect Rally support through fallback logic) will now fail.
- The `ctx` object in new workflows has NO Rally context field (no `rally: { ... }` property like `github: { owner, repo, number }`), so even if we added Rally handling, there's no way to detect a Rally issue in the fallback path.

**Root Cause:**

1. The new lifecycle code only checks `ctx.github` to identify issue type
2. Rally issues have no equivalent context field (no `ctx.rally`)
3. The fallback only checks for Linear API key, not Rally configuration
4. The tracker-based path requires a pre-instantiated `tracker` object, which isn't automatically created for Rally

**Fix:**

Option A (Recommended): Add Rally context to LifecycleContext and handle it:

```typescript

  projectPath: string;
  github?: { owner: string; repo: string; number: number };
  rally?: { wsapiUrl: string; projectOid: string; storyOid: string }; // NEW

async function closeViaDirect(
  ctx: LifecycleContext,
  comment?: string,

  if (ctx.github) {
    return closeGitHubDirect(ctx, comment);
  }

  if (ctx.rally) {
    return closeRallyDirect(ctx, comment);
  }

  const linearApiKey = getLinearApiKey();
  if (linearApiKey) {
    return closeLinearDirect(ctx, linearApiKey);
  }

  return stepFailed(step, 'No tracker available and cannot determine issue type');
}

async function closeRallyDirect(ctx: LifecycleContext, comment?: string): Promise<StepResult> {
  const step = 'close-issue:transition';
  if (!ctx.rally) return stepFailed(step, 'Rally config not provided');

  try {
    const { RallyTracker } = await import('../tracker/rally.js');
    const rallyConfig = getRallyConfig();
    if (!rallyConfig) {
      return stepFailed(step, 'Rally not configured');
    }

    const tracker = new RallyTracker({
      apiKey: rallyConfig.apiKey,
      server: rallyConfig.server,
      workspace: rallyConfig.workspace,
      project: rallyConfig.project,
    });

    await tracker.transitionIssue(ctx.issueId, 'closed');
    if (comment) {
      try {
        await tracker.addComment(ctx.issueId, comment);
      } catch {
        // Non-fatal
      }
    }
    return stepOk(step, [`Closed Rally issue ${ctx.issueId}`]);
  } catch (err) {
    return stepFailed(step, `Failed to close Rally issue: ${(err as Error).message}`);
  }
}
```

Option B (Detect Rally by prefix): Extract Rally tracker auto-detection from getProjectPath() call:


### 2. [workflows.ts:163-172] PRD Move Now Happens Before Merge Verification — Ordering Changed

**Severity:** Critical (potential data loss)
**Location:** `/home/eltmon/Projects/overdeck/src/lib/lifecycle/workflows.ts:163-172`

**Problem:**

In the NEW `closeOut()` workflow (workflows.ts):

```typescript
export async function closeOut(
  ctx: LifecycleContext,
  opts: CloseIssueOptions & ArchiveOptions = {},
): Promise<WorkflowResult> {
  // 1. Verify/move PRD + 2. Archive workspace artifacts
  const archiveSteps = await archivePlanning(ctx, opts);  // <-- MOVES PRD HERE
  allSteps.push(...archiveSteps);

  // Hard fail on archive failure
  const archiveFailed = archiveSteps.some(s => !s.success && !s.skipped);
  if (archiveFailed) {
    allSteps.push(stepFailed('close-out:abort', 'Stopped — archiving failed, workspace preserved'));
    return buildResult('close-out', ctx.issueId, allSteps, start);
  }

  // 3. Verify branch merged (hard fail)
  const mergeVerify = await verifyBranchMerged(ctx);  // <-- VERIFIES MERGE HERE
  allSteps.push(mergeVerify);
  if (!mergeVerify.success && !mergeVerify.skipped) {
    return buildResult('close-out', ctx.issueId, allSteps, start);
  }
```

But in the OLD `executeCloseOut()` (close-out.ts), the order was:

```typescript
// Step 1: Verify PRD preserved
// ... PRD checking and moving logic
steps.push({ name: 'Verify PRD preserved', ... });

// Step 2: Verify branch merged (hard fail)
steps.push({ name: 'Verify branch merged', ... });

// Step 3: Archive workspace artifacts
// ... only after merge is verified
```

**The critical difference:**

1. OLD: Verify branch merged → THEN move PRD
2. NEW: Move PRD → THEN verify branch merged

This means in the NEW code, if merge verification fails, the PRD has ALREADY been moved to `completed/`, but the workspace/branch cleanup never happens. The PRD is now orphaned in the completed directory, and the workspace is still dirty.

**Impact:**

- If a user initiates close-out, the PRD moves to completed/, but then the merge check fails
- The ceremony aborts, but the PRD is already "archived" (marked complete)
- The workspace is left in an inconsistent state: PRD says it's done, but the branch isn't actually merged
- If user retries close-out, the PRD check is idempotent (skip because it's already in completed/) — but now the workflow can't be re-run to actually clean up

**Why this matters:**

The old code's logic was: "Verify the branch is merged BEFORE we declare this issue complete by moving the PRD."

The new code says: "Move the PRD first, then check."

This violates the principle: **Don't mark something complete until you've verified all preconditions are met.**

**Fix:**

Reorder the steps in `closeOut()` to verify merge BEFORE archiving:

```typescript
export async function closeOut(
  ctx: LifecycleContext,
  opts: CloseIssueOptions & ArchiveOptions = {},
): Promise<WorkflowResult> {
  const start = Date.now();
  const allSteps: StepResult[] = [];

  // 1. Verify branch merged FIRST (hard fail)
  const mergeVerify = await verifyBranchMerged(ctx);
  allSteps.push(mergeVerify);
  if (!mergeVerify.success && !mergeVerify.skipped) {
    // Fail early, before moving PRD
    return buildResult('close-out', ctx.issueId, allSteps, start);
  }

  // 2. Archive planning artifacts (PRD move + workspace artifacts)
  const archiveSteps = await archivePlanning(ctx, opts);
  allSteps.push(...archiveSteps);

  const archiveFailed = archiveSteps.some(s => !s.success && !s.skipped);
  if (archiveFailed) {
    allSteps.push(stepFailed('close-out:abort', 'Stopped — archiving failed, workspace preserved'));
    return buildResult('close-out', ctx.issueId, allSteps, start);
  }

  // 3+4. Teardown workspace + agent state
  // ... rest of workflow
```

### 1. [archive-planning.ts:49-104] movePrd() Has Edge Case: cpSync Without Source Validation

**Severity:** Warning
**Location:** `/home/eltmon/Projects/overdeck/src/lib/lifecycle/archive-planning.ts:94-101`

**Problem:**

The fallback copy logic doesn't validate that the source file exists before copying:

```typescript
// Fallback: plain copy
try {
  cpSync(activePrdPath, completedPrdPath);  // <-- cpSync will fail if source doesn't exist
  if (!existsSync(completedPrdPath)) {
    return stepFailed(step, 'PRD copy appeared to succeed but file not found at destination');
  }
  return stepOk(step, ['Copied PRD to completed/ (git mv failed, plain copy succeeded)']);
} catch (err) {
  return stepFailed(step, `Failed to preserve PRD: ${(err as Error).message}`);
}
```

The logic assumes that if the catch block isn't triggered, the copy succeeded. But if `cpSync` throws for ANY reason (permissions, disk full, etc.), the error is caught and reported as "Failed to preserve PRD".

However, there's a subtle issue: the `if (!existsSync(completedPrdPath))` check is redundant if `cpSync` succeeded — it's a defensive check, not a problem. The real issue is that this path assumes `activePrdPath` still exists, but it was checked only at line 72:

```typescript
if (!existsSync(activePrdPath)) {
  return stepSkipped(step, ['No PRD found in active/ (may not have had one)']);
}
```

**Conditions when this could fail:**

1. **File deleted between check and copy:** Between the `existsSync(activePrdPath)` check (line 72) and the `cpSync()` call (line 96), another process deletes the file. `cpSync` will throw `ENOENT`.
2. **Symlink race condition:** If activePrdPath is a symlink that points to a file that gets deleted between the check and copy.
3. **Permissions change:** After the check passes, permissions change and copy fails.

**Impact:**

In case 1-3, the error message is correct ("Failed to preserve PRD") but not specific. For case 1, the user might think it's a real failure when it's actually a race condition (file was already elsewhere, maybe manually moved).

**Fix:**

Add explicit source existence check immediately before copy to reduce timing window:

```typescript
// Fallback: plain copy
try {
  // Re-check source exists immediately before copy to reduce race condition window
  if (!existsSync(activePrdPath)) {
    return stepFailed(step, 'PRD file disappeared between git mv attempt and copy fallback (race condition)');
  }

  cpSync(activePrdPath, completedPrdPath);
  if (!existsSync(completedPrdPath)) {
    return stepFailed(step, 'PRD copy succeeded but file not found at destination');
  }
  return stepOk(step, ['Copied PRD to completed/ (git mv failed, plain copy succeeded)']);
} catch (err) {
  return stepFailed(step, `Failed to preserve PRD: ${(err as Error).message}`);
}
```

### 2. [close-issue.ts:59-68] Early Return on Close Failure Skips Label Operations

**Severity:** Warning
**Location:** `/home/eltmon/Projects/overdeck/src/lib/lifecycle/close-issue.ts:59-68`

**Problem:**

In `closeIssue()`, if the transition step fails (and it's not skipped), the function returns early and never applies labels:

```typescript
export async function closeIssue(
  ctx: LifecycleContext,
  opts: CloseIssueOptions = {},
): Promise<StepResult[]> {
  const results: StepResult[] = [];
  const { applyLabel = true, labelOnly = false, comment } = opts;

  // Step 1: Transition to closed (unless labelOnly)
  if (!labelOnly) {
    const closeResult = opts.tracker
      ? await closeViaTracker(ctx, opts.tracker, comment)
      : await closeViaDirect(ctx, comment);
    results.push(closeResult);

    // If close failed, don't bother with labels
    if (!closeResult.success && !closeResult.skipped) {
      return results;  // <-- EARLY EXIT
    }
  }

  // Step 2: Apply closed-out label + remove workflow labels
  if (applyLabel) {
    const labelResult = await applyClosedOutLabel(ctx, opts.tracker);
    results.push(labelResult);
  }

  return results;
}
```

This is arguably correct behavior (if you can't close the issue, don't label it), but there's an asymmetry:

- In the old code, if close FAILED, the ceremony aborted hard (returned error).
- In the new code, close failure just skips labels and returns a partial result.

**Conditions when this matters:**

1. **Tracker offline during close, online during label:** If the issue tracker is temporarily unreachable during close but comes back, the label is never applied.
2. **Label-only workflows:** The `labelOnly: true` option allows applying labels without closing, but there's no `closeOnly: true` option. So if close fails, you can't retry just the label step without re-running the whole workflow.

**Impact:**

Issues might have the closed state in one system but not the label applied, causing dashboard inconsistency.

**Conditions:**

This only matters if:
- The tracker is intermittently available
- A workflow retries and the issue is partially transitioned
- Users don't re-run close-out after fixing the transient failure

**Fix:**

Add a `closeOnly` option or allow separate retry paths:

```typescript
export interface CloseIssueOptions {
  tracker?: IssueTracker;
  reason?: string;
  comment?: string;
  applyLabel?: boolean;
  labelOnly?: boolean;
  closeOnly?: boolean;  // NEW: close but don't apply label
}

export async function closeIssue(
  ctx: LifecycleContext,
  opts: CloseIssueOptions = {},
): Promise<StepResult[]> {
  const results: StepResult[] = [];
  const { applyLabel = true, labelOnly = false, closeOnly = false, comment } = opts;

  // Step 1: Transition to closed (unless labelOnly)
  if (!labelOnly && !closeOnly) {
    // ... close logic ...
  }

  // Step 2: Apply label (unless closeOnly or labelOnly already ran)
  if (applyLabel && !closeOnly) {
    // ... label logic ...
  }

  return results;
}
```

Actually, on further review, this is **not a bug** — it's by design. The function is `closeIssue()`, and if closing fails, not applying labels is the right behavior. The OLD code would have failed the whole ceremony. The NEW code is more granular but has the same net effect: partial failure doesn't corrupt state.

So I'm downgrading this to a "consider for enhancement" note, not a bug.
