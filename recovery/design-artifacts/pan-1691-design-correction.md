## Design corrections from review (2026-06-09)

**1. Conflicts are resolved by an AGENT, never punted to a human.** Earlier framing ("mark blocked so a human looks") is wrong. When a sibling rebase genuinely conflicts, an agent resolves it. The original inspiration for a dedicated merge agent was "whoever did the last merge has the most context about what changed" — but it's not settled that the merge agent is best-positioned vs. the issue's own work agent. **Decision: who resolves is a secondary/configurable choice; the firm requirement is that whoever resolves MUST load and understand what BOTH features intended across BOTH changesets, and resolve so neither feature is degraded.**

**2. The per-issue toggle's tri-state is only half-wired.** `isAutoMergeEligible` currently acts only on `autoMerge === false` (Hold). `undefined` ("default") and `true` ("Auto") do nothing extra — "default" does NOT consult the global `flywheel.require_uat_before_merge` setting the label implies, and `Auto` does not override a require-UAT default. To close: make `undefined` follow the global Require-UAT setting and `true` override it, then make the badge hovertip state what "default" resolves to. Note: the default is GLOBAL (`flywheel.require_uat_before_merge`), not per-project; a true per-project default would be new work (see PAN-1693).
