## 0. Glossary (define before use)

- **Delivery door** — `deliverAgentMessage(agentId, message, caller, deliveryMethod)` in
  `src/lib/agents/delivery.ts` (grep anchor: `export async function deliverAgentMessage`).
  The single orchestrator→agent message primitive, re-exported via `src/lib/agents.ts`.
- **Tier** — one transport attempt inside the door. Auto mode tries, in order:
  **supervisor** (PTY supervisor unix socket `pty-<id>.sock`) → **channels** (legacy MCP
  bridge socket `agent-<id>.sock`, gated by `state.channelsEnabled`) → **tmux**
  (load-buffer + paste-buffer via `sendKeys` in `src/lib/tmux.ts`). An explicit
  `deliveryMethod` pins one tier; explicit `supervisor`/`channels` throw on failure
  today, explicit/fallback `tmux` never does — that asymmetry is the bug.
- **Verified delivery / evidence** — per-tier proof the payload reached the agent's PTY:
  supervisor = **echo confirmation** (the supervisor's `injectPtyMessage` watches PTY
  output for `confirmationPrefix(content)` before sending Enter, 2 attempts with
  `purgePtyInput` between; grep anchor: `input echo confirmation failed after`); tmux =
  **paste verification** (`sendKeys` polls `capturePaneText` for the payload's verify
  line, 2 attempts × 1.5 s + a 200-line wide-window check; grep anchor: `pasteVerified`);
  channels = the bridge's 200 response (accepted as-is — legacy tier, unchanged here).
- **Silent success** — returning `{ ok: true }` without evidence. The tmux tier does this
  today (grep anchor in `src/lib/tmux.ts`: `Sending Enter anyway to avoid orphaned input`).
- **Kickoff** — the initial work prompt delivered right after spawn. Persisted to
  `<agentDir>/initial-prompt.md` (grep anchor in `src/lib/agents/spawn.ts`:
  `join(getAgentDir(agentId), 'initial-prompt.md')`) so resume paths can redeliver it.
- **Kickoff-redelivery machinery** — `state.kickoffDelivered === false` marks a work
  agent whose kickoff never landed. `getResumeMessage` (grep anchor in
  `src/lib/agents/supervisor-channels.ts`: `redeliveringKickoff`) swaps the resume
  message for the persisted kickoff; the deacon's patrols key off the flag
  (`shouldRetryUndeliveredKickoff`, `isPreKickoffLaunchDeath`,
  `nudgeStalledResumeWorkAgents` in `src/lib/cloister/deacon-auto-resume.ts`).
- **Failed state** — in this PRD: `status: 'stopped'` **plus** failure fields set by
  `recordAgentFailure` (`consecutiveFailures`, `lastFailureReason`, troubled gate after
  `maxConsecutiveFailures`). NOT a new enum value. The `AgentState.status` union is
  `'starting' | 'running' | 'stopped' | 'error'` (grep anchor in
  `src/lib/agents/agent-state.ts`: `status: 'starting' | 'running' | 'stopped' | 'error'`).
- **Side-door** — a production call that sends a *message* to an agent session via raw
  `sendKeys` instead of the door, bypassing tier selection and (until this PRD) any
  failure contract. §2.4 enumerates all of them.
- **Zombie kickoff (PAN-2179)** — the incident class this PRD closes: session alive,
  kickoff never delivered, tmux fallback reported ok:true, agent stayed `running`,
  liveness checks passed, and the flywheel "merged" issues with zero `src/` changes.
- **`MessageDeliveryFailed`** — existing error class in `src/lib/tmux.ts` (grep anchor:
  `export class MessageDeliveryFailed extends Error`) carrying `sessionName` and
  `paneSnapshot`. This PRD makes it the door's single failure type.
- **TENET-10** — pipeline-machinery changes require the FULL `npm test` suite green
  before merge and source-introspection tests repointed in the same PR.

## 1. Problem (verified evidence at `506756378f`)

Four distinct silent-failure gaps, each verified:

1. **The tmux tier reports success without evidence.** `sendKeys` in `src/lib/tmux.ts`
   verifies the paste landed, but when verification fails after both attempts it logs a
   warning and proceeds: grep anchor `Sending Enter anyway to avoid orphaned input` —
   then the Effect succeeds and `deliverAgentMessage` returns
   `{ ok: true, path: 'tmux' }`. The `failure?` field carries only *earlier-tier*
   failures. This is the exact PAN-2179 mechanism.
2. **Kickoff failure leaves the agent `running`.** `recordKickoffDeliveryFailure` in
   `src/lib/agents/supervisor-channels.ts` records a failure, then sets
   `failedState.status = 'running';` (grep anchor — the literal assignment appears
   twice in that function). A work agent whose kickoff verifiably never landed is
   indistinguishable from a healthy one to every status surface.
3. **Role spawns ignore kickoff failure entirely.** In `spawnRun`
   (`src/lib/agents/spawn.ts`, grep anchor: `'spawnRun:initial-prompt'`), a ready-signal
   timeout only `console.error`s and the ok-ignored `deliverAgentMessage` result is
   discarded; execution falls through to `markAgentRunning(state)`. A review/plan/test
   role whose prompt never landed becomes a zombie session the reactive scheduler must
   detect by other means.
4. **~20 side-door `sendKeys` message sends bypass the door** (§2.4): deacon nudges,
   CI-failure feedback, context-overflow recovery, planning-session messages, the
   graceful-restart warning. Each re-inherits gap 1 and never benefits from the
   supervisor tier even when a supervisor socket exists.

The issue's fix directive: *fallback is allowed, silent success is not* — every tier
returns verified delivery or the failure propagates as a hard error, and spawn paths
treat kickoff failure as fatal (agent marked failed, not running).

## 3. Locked design decisions

- **D1 — The door returns verified success or throws; nothing in between.**
  `deliverAgentMessage` returns `DeliveryResult` ONLY when the succeeding tier produced
  evidence (§2.1). Every failure — including the auto chain exhausting all tiers —
  throws `MessageDeliveryFailed` (imported from `src/lib/tmux.ts`) whose message
  aggregates each attempted tier's failure reason (e.g.
  `supervisor: socket-missing; channels: channels-disabled; tmux: paste unverified after 2 attempts`).
  The `ok` field stays (always `true` on return) so the ~10 `delivery.ok` call sites
  don't churn; `ok: false` is never constructed by the door. `deliverInitialPromptWithRetry`
  keeps its `{ok: false, failure}` return — it is kickoff *orchestration*, its callers
  (§2.2 #2, #4, #5) already branch on it.
- **D2 — tmux paste verification becomes authoritative.** In `sendKeys`
  (`src/lib/tmux.ts`), when `pasteVerified` is false after both attempts and the
  wide-window check: (a) do NOT send Enter; (b) purge the possibly-landed composer text
  by writing DEL characters via the same mechanism `purgePtyInput` uses in
  `src/lib/channels/pty-supervisor.ts` (grep anchor: `async function purgePtyInput`) —
  `send-keys` the `\x7f` sequence sized to the payload — so a retry or supervisor
  fallback can never stack a duplicate copy (the PAN-1769 duplicate-submit race); (c)
  throw `MessageDeliveryFailed(message, sessionName, paneSnapshot)`. **Caveat (the only
  sanctioned evidence-free success):** payloads whose verify line is under 3 chars are
  unverifiable by pane text (grep anchor: `verifyLine.length >= 3`); the existing
  heuristic delay path stays but must log `evidence: 'unverifiable-short-content'` to
  the delivery log. The post-Enter submit-verification re-send stays as-is.
- **D3 — Work-agent kickoff failure ⇒ failed state, session killed.** Rewrite
  `recordKickoffDeliveryFailure`: keep `recordAgentFailure` (failure fields, troubled
  accumulation) and `kickoffDelivered: false`, but set `status: 'stopped'` (never
  `'running'`) and kill the tmux session via `stopAgent`. Decision rule and rationale:
  - `'running'` is rejected — it is the PAN-2179 illusion itself.
  - `'error'` is rejected — the deacon's resume machinery keys on `'stopped'`
    (`handleAgentStoppedEvent` grep anchor: `status=${state.status} (not stopped)`);
    `'error'` would strand the agent outside autonomous recovery.
  - `'stopped'` + killed session converts kickoff failure into the ORDINARY recovery
    path: `handleAgentStoppedEvent` resumes it, `getResumeMessage` redelivers the
    persisted kickoff (`initial-prompt.md`), `isPreKickoffLaunchDeath` accumulates
    failures so a crash-looping spawn trips the troubled gate instead of looping
    forever. No new machinery — the fix is telling the truth to machinery that
    already exists.
  - Strike keeps today's stricter contract (stop + throw to the caller).
- **D4 — `spawnRun` kickoff failure is fatal.** In the `spawnRun` prompt block (grep
  anchor: `'spawnRun:initial-prompt'`): ready-timeout OR a door throw ⇒ `stopAgent` +
  throw `new Error(...)` naming the role, issue, and tier failures. `spawnRun` already
  throws on earlier failures (session spawn, beads gate), so callers face no new error
  category. `markAgentRunning` must be unreachable on this path.
- **D5 — Side-doors route through the door.** Every §2.4 "migrate" row becomes
  `await deliverAgentMessage(<session>, <message>, '<caller-tag>')` (auto tier). A
  caller that today deliberately tolerates failure (graceful-restart warning) keeps its
  try/catch — D1 changes *honesty*, not each caller's tolerance policy. New caller tags
  follow the existing `module:purpose` convention.
- **D6 — Guard against side-door regression.** New unit test asserting the sweep
  `git grep -n "runPromise(sendKeys(" src/ ':!src/lib/tmux.ts' ':!src/lib/agents/delivery.ts' ':!tests' ':!**/__tests__/**'`
  returns nothing (pattern: follow `tests/unit/lib/overdeck/no-loss-matrix.test.ts`, a
  sanctioned source-introspection test). If PAN-2231's introspection baseline is live,
  add the new test file to `scripts/source-introspection-baseline.txt` in the same
  commit (audited addition — the commit references this issue).
- **D7 — Every delivery attempt is logged with its outcome.** `appendChannelDeliveryLog`
  today logs supervisor success, channels success, and the channels→tmux fallback — but
  NOT the explicit-tmux early return, the final-fallback tmux success/failure, or D2
  throws. Extend it to log every outcome (tier, evidence kind, failure reason, caller)
  on every path through the door. Best-effort (existing try/catch stays) — the log is
  evidence for incident forensics, not a delivery gate.

## 4. Requirements

- **FR-1** `deliverAgentMessage` never returns without tier evidence: all failure paths
  (per-tier and exhausted-auto-chain) throw `MessageDeliveryFailed` whose message names
  every attempted tier and its reason (D1).
- **FR-2** The tmux tier treats paste verification as authoritative: unverified after
  retries ⇒ purge + throw, never Enter, never ok (D2); the sub-3-char caveat is logged.
- **FR-3** A work-agent kickoff failure leaves the agent in the failed state — status
  `'stopped'`, session killed, failure fields set, `kickoffDelivered: false` — and
  never `'running'` (D3). Redelivery-on-resume machinery keeps working.
- **FR-4** A role (`spawnRun`) kickoff failure stops the session and throws; the agent
  is never marked running (D4).
- **FR-5** Every §2.4 "migrate" site sends through the door; the D6 guard test proves
  no production side-door remains.
- **FR-6** Every delivery attempt (success or failure, every tier) lands one line in the
  per-agent delivery log with tier, evidence kind, failure reason, and caller (D7).
- **FR-7** No new transport. The three tiers, their ordering, and `resilientDeliveryMethod`
  semantics are unchanged; only failure semantics and call-site routing change.
- **NFR-1** All new/changed tests exercising timeouts, retries, or delays use
  `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()` — the door is full of 1.5–8 s
  waits; real timers here recreate the OOM-by-parallel-suites failure mode.
- **NFR-2** Server-reachable code stays async-only: no `execSync`, no new `sendKeysSync`
  callers; migrated sites keep `await`-based flow.
- **NFR-3** Full `npm test` green before merge (TENET-10); typecheck+lint+build green on
  every intermediate commit.
- **NFR-4** No file grows past its baseline; no new file ≥1,000 lines (current sizes:
  `delivery.ts` 468, `tmux.ts` 1,258 — baselined; check `scripts/file-size-baseline.txt`
  before touching, lower entries if shrunk).
