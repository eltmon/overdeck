# PAN-2228 — One delivery door with loud failure semantics for agent message transport

**Issue:** https://github.com/eltmon/overdeck/issues/2228
**Verified-Against:** main @ `bc95c5956c6ea9a47f7e5629ac3f0772be0853c3`
**Queue phase:** Phase 4 — deep foundations (item 17 of `docs/codebase-health/REFACTOR-QUEUE.md`)
**Status:** draft

> **TENET-10, supervised dispatch only.** Delivery is how every orchestrator message —
> kickoff briefs, review feedback, resume prompts — reaches every agent. A regression
> here silently severs the pipeline's nervous system. This is a BEHAVIOR-CHANGING deep
> fix (contrast the Phase 2/3 relocations): failure semantics change on purpose; the
> design decisions below are locked — execute them, don't re-derive them.

## 0. Glossary

- **Delivery tiers** — the three transports `deliverAgentMessage`
  (`src/lib/agents/delivery.ts`, 468 lines — the single primitive; it already exists)
  tries in `auto` mode, in order: **supervisor** (POST to
  `~/.overdeck/sockets/pty-<id>.sock`; the PTY supervisor,
  `src/lib/channels/pty-supervisor.ts`, writes into the agent's PTY and
  echo-confirms server-side), **channels** (legacy MCP bridge socket
  `agent-<id>.sock`), **tmux** (paste-buffer via the Effect `sendKeys` from
  `src/lib/tmux.ts`).
- **Evidence** — proof the payload reached the agent's PTY/transcript. Three kinds in
  this design: `supervisor-echo` (the supervisor's intrinsic echo confirmation — grep
  anchor: `export function confirmationPrefix(content: string)` and
  `INPUT_ECHO_CONFIRM_ATTEMPTS` in `pty-supervisor.ts`), `transcript-user-record` (a
  new user record lands in the session JSONL — existing machinery
  `captureTranscriptUserRecordSnapshot`/`hasNewTranscriptUserRecord` from
  `src/lib/transcript-landing.js`), `pane-echo` (the message's confirmation prefix
  appears in `tmux capture-pane` output — the supervisor's normalize/prefix helpers
  reused client-side).
- **Kickoff** — the initial work brief delivered by
  `deliverInitialPromptWithRetry` (same file) after `waitForPromptReady`.
- **DOA session** — a tmux session whose pane process has exited but which still
  "exists" (sessions run with `remain-on-exit on`), so
  `sessionExists` (grep anchor: `export const sessionExists = (` in `tmux.ts`) returns
  true for a dead agent. `isPaneDead` (anchor: `export const isPaneDead = (`) is the
  discriminator — currently UNUSED in the kickoff path.
- **Zombie-kickoff incident (PAN-2179)** — flywheel-auto-started issues "completed"
  with zero src/ changes: tmux fallback returned `ok:true` for pastes that never
  landed, the DOA supervisor session looked alive to `sessionExists`, and the agent was
  marked running.
- **Redelivery patrol** — `src/lib/cloister/deacon-auto-resume.ts` re-nudges work
  agents with `kickoffDelivered === false` (grep anchors:
  `return state.role === 'work' && state.kickoffDelivered === false;` and
  `nudgeStalledResumeWorkAgents`). This machinery is KEPT (D5).

## 1. Problem (verified evidence — all anchors checked at the sha)

1. **The tmux tier lies.** `deliverAgentMessage`'s fallback ends with
   `await Effect.runPromise(sendKeys(normalizedId, message)); return { ok: true, path: 'tmux', failure: … }`
   — `ok:true` with zero verification, in three places (explicit `tmux` method, the
   auto-chain fallback, and the trailing default).
2. **Kickoff failure leaves agents "running" BY DESIGN.**
   `recordKickoffDeliveryFailure` (`src/lib/agents/supervisor-channels.ts`, grep
   anchor: `export async function recordKickoffDeliveryFailure`) calls
   `recordAgentFailure(...)` then sets `failedState.status = 'running'` and
   `kickoffDelivered = false` — feeding the redelivery patrol, but indistinguishable
   from a healthy agent on every dashboard/liveness surface.
3. **The kickoff liveness check can't see DOA sessions.**
   `deliverInitialPromptWithRetry` gates on `waitForPromptReady` then
   `sessionExists` (anchor: `const alive = await Effect.runPromise(sessionExists(` in
   `delivery.ts`) — never `isPaneDead`. A supervisor session whose child exited at
   spawn looks alive.
4. **The supervisor tier is already sound**: server-side echo-confirmation (2 attempts,
   input purge on failure — anchors: `purgePtyInput`, `appendEchoFailureLog`) and the
   supervisor process exits with its child (anchor:
   `const result = await childExited;` → `process.exit(result.exitCode` at EOF). Its
   2xx IS evidence.
5. **The verification machinery already exists but only guards resume**:
   `deliverResumeMessageWithTranscriptConfirmation` polls for a new transcript user
   record after delivery (2 attempts). Kickoff and ordinary messages get nothing.
6. **No unit tests cover `delivery.ts`** (verified: no test file imports
   `deliverAgentMessage`/`deliverInitialPromptWithRetry` outside src; the supervisor
   has its own `__tests__`).

## 2. Locked design decisions

- **D1 — Evidence-typed result; `ok:true` is unrepresentable without evidence.**
  ```ts
  export type DeliveryEvidence = 'supervisor-echo' | 'transcript-user-record' | 'pane-echo';
  export type DeliveryResult =
    | { ok: true; path: 'supervisor' | 'channels' | 'tmux' | 'pi' | 'codex'; evidence: DeliveryEvidence }
    | { ok: false; path?: 'supervisor' | 'channels' | 'tmux'; failure: string;
        tierFailures: Partial<Record<'supervisor' | 'channels' | 'tmux', string>> };
  ```
  The type system enforces the acceptance criterion "no code path returns ok:true
  without an ack".
- **D2 — Uniform confirmation rule.** A tier with INTRINSIC confirmation (supervisor
  echo) returns its evidence directly. Tiers WITHOUT intrinsic confirmation (channels
  2xx = "bridge accepted", tmux paste = fire-and-forget) must obtain EXTRINSIC
  confirmation after sending: transcript-user-record landing when a session id is
  resolvable (reuse `waitForTranscriptUserRecordLanding` — hoist it from the resume
  helper, unchanged semantics: 3s / 100ms poll), else pane-echo (capture the pane via
  the async tmux capture primitive, normalize with the supervisor's exported
  `confirmationPrefix` + normalize helpers, match the prefix). No confirmation within
  timeout ⇒ that tier FAILED.
- **D3 — Auto-chain: fallback allowed, silent success dead.** Each tier's failure
  reason accumulates in `tierFailures`; when no tier produces evidence, return
  `{ ok:false, failure: <last>, tierFailures }`. Explicit strict methods
  (`'supervisor'`/`'channels'`) keep their existing throw contract (PAN-1769);
  explicit `'tmux'` now confirms per D2 instead of blind `ok:true`.
  `resilientDeliveryMethod` (PAN-1988) is unchanged.
- **D4 — Kickoff sees dead panes.** In `deliverInitialPromptWithRetry`, the
  alive-check becomes `sessionExists && !isPaneDead` (both from `tmux.js`); a dead
  pane maps to the existing `SESSION_EXITED_BEFORE_KICKOFF` failure constant.
- **D5 — Kickoff fatality is split by liveness (the issue's "fatal" is interpreted
  against the DOA incident; the live-session redelivery patrol is deliberate machinery
  and stays):**
  - **Dead session/pane** (`SESSION_EXITED_BEFORE_KICKOFF`): spawn calls
    `stopAgent`, records the failure, and the final state is stopped/failed — NEVER
    `running`, and `markAgentRunning` is not reached. This satisfies the issue's
    "simulated dead-PTY kickoff leaves the agent in a failed state".
  - **Live session, unconfirmed delivery**: keep `recordKickoffDeliveryFailure` +
    `kickoffDelivered=false` so the deacon redelivery patrol retries — but the loud
    part is now real (the result was honest `ok:false`, activity log error emitted,
    failure counters incremented via `recordAgentFailure`).
  - The existing strike-role behavior (stop + throw) extends to ALL roles for the dead
    case; strike keeps throwing for the live-unconfirmed case too (verified current
    behavior — anchor: `if (role === 'strike') {` after
    `recordKickoffDeliveryFailure` in `spawn.ts`).
- **D6 — Caller sweep is part of the change.** D1 changes the result shape; every
  consumer of `DeliveryResult`/`deliverAgentMessage`/`deliverInitialPromptWithRetry`
  compiles against the new type and handles `ok:false` (verified consumer list:
  `src/lib/agents/spawn.ts` (kickoff), `src/lib/agents/resume.ts` (2 call sites),
  `src/lib/agents/recovery.ts`, `routes/agents.ts` message/tell/poke handlers — or the
  `agent-messaging.ts` door module if PAN-2147 landed — plus
  `deliverResumeMessageWithTranscriptConfirmation`'s internal use). Handlers surface
  `ok:false` as HTTP 502 with `tierFailures` in the body, not 200.
- **D7 — `deliverAgentPermissionDecision` and Pi/codex FIFO paths are out of scope.**
  Permission decisions already throw loudly on every failure (verified); Pi uses its
  `rpc.in` FIFO, codex kickoff uses the file-backed brief (anchor:
  `.pan/kickoff.md` block in `deliverInitialPromptWithRetry` — that block is kept
  verbatim; the pointer message it delivers is confirmed like any other message).

## 3. Requirements

### Functional
- **FR-1** `DeliveryResult` has the D1 shape; the `evidence` field is present on every
  success (compile-time enforced).
- **FR-2** The tmux tier (explicit and fallback) confirms per D2 and returns
  `pane-echo`/`transcript-user-record` evidence or fails.
- **FR-3** The channels tier confirms per D2 after its 2xx.
- **FR-4** The supervisor tier returns `supervisor-echo` on 2xx (no extra round trip).
- **FR-5** Auto mode returns `ok:false` with per-tier `tierFailures` when nothing
  confirmed; it never throws for auto (callers branch on `ok`), and strict modes keep
  throwing.
- **FR-6** `deliverInitialPromptWithRetry` treats a dead pane as
  `SESSION_EXITED_BEFORE_KICKOFF` (D4).
- **FR-7** Spawn: dead-session kickoff ⇒ agent stopped + failure recorded, status is
  never `running` (D5); live-session unconfirmed kickoff ⇒ honest failure +
  redelivery-patrol state (unchanged machinery).
- **FR-8** Message/tell/poke HTTP handlers return 502 + `tierFailures` on `ok:false`.

### Non-functional
- **NFR-1** No `execSync`; async tmux primitives only (capture-pane via the async
  helper; never add a sync variant).
- **NFR-2** Every new/changed test that involves the confirmation timeouts, retry
  loops, or backoff MUST use `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()` —
  this PRD's tests are exactly the class the fake-timers rule exists for.
- **NFR-3** Extra latency budget: a confirmed tmux delivery may take up to ~3s
  (transcript poll) — acceptable; do not raise the supervisor client timeout (8s,
  anchor: the `8_000` in `deliverAgentMessage` — its comment explains the
  duplicate-submit race it prevents).
- **NFR-4** `delivery.ts` stays under 1,000 lines (it grows ~150; fine). No new
  baseline entries.

## 5. Intersecting repo rules (restated)

- **Fake timers (NFR-2)** — non-negotiable here; the confirmation polls are 3s×2 and
  the kickoff loop waits up to 30s×2. Real timers would OOM the suite.
- **Async-only / no `execSync`;** async tmux primitives only; the paste-buffer
  pattern lives inside the tmux helpers — do not re-implement it.
- **No hardcoded model fallbacks / no behavior smuggling:** this change alters failure
  semantics ONLY; message content, tier order, timeouts (except the new confirmation
  windows), and the codex file-backed kickoff stay byte-identical.
- **TENET-10:** full `npm test` before merge; supervised dispatch; verify against
  origin HEAD. The deacon consumes these states — after merge, `npm run build` before
  any dashboard restart (Node 22 dist).
- **File-size guard** (<1,000 for any new file); **ESLint `--no-inline-config`**;
  never `--no-verify`; never `git stash`. commitlint: subjects all-lowercase, scope
  from the enum — **use `infra` for the delivery/spawn commits** (agent transport has
  no dedicated scope) **and `dashboard` for the WI-6 route-handler commit**, e.g.
  `refactor(infra): evidence-typed delivery result (pan-2228)`.
