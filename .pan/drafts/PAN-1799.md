# PAN-1799 — Codex work-agent boot deaths: fail fast, diagnose, and finish the spawn path

**Issue:** [PAN-1799](https://github.com/eltmon/overdeck/issues/1799)
**Status:** Planned (auto mode, 2026-06-12)
**Planner:** agent:claude-fable-5

## Glossary

- **Work agent** — the implementation-role agent spawned by `pan start` via `spawnAgent()` (`src/lib/agents.ts:3378`), living in a tmux session named `agent-<issue>` on the `overdeck` socket.
- **work-tui** — the codex work-agent mode (`codexMode: 'work-tui'`): a persistent interactive `codex` TUI, launched by `buildCodexCommand` (`src/lib/launcher-generator.ts:609-623`). Sandbox/approvals come from the per-agent `config.toml` seeded by `initCodexHome`, not CLI flags.
- **Kickoff delivery** — pasting the initial work prompt into the agent's session after it reaches its input prompt. For codex this is a short pointer message; the full brief sits in `.pan/kickoff.md` (`deliverInitialPromptWithRetry`, `src/lib/agents.ts:1614-1671`).
- **Ghost lifecycle** — the existing recovery contract for "session alive but kickoff never landed": state stays `running` with `kickoffDelivered: false`; deacon resume re-delivers the original kickoff (`buildResumeMessageForAgent`, `src/lib/agents.ts:1677-1700`; locked by `tests/integration/agent-spawning.test.ts:357-512`).
- **Boot death** — the spawned harness process exits seconds after launch, so the tmux session disappears before (or right after) kickoff.
- **Rollout / thread-id** — codex writes a `rollout-<ts>-<threadId>.jsonl` per session under `CODEX_HOME/sessions/`; the thread id is the codex session id, persisted to `~/.overdeck/agents/<id>/codex-thread-id` by `writeThreadId` (`src/lib/runtimes/codex.ts:76`).

## Incident and what already landed

Observed 2026-06-12 07:54Z (`pan start PAN-1704 --model gpt-5.5 --harness codex --fresh --force`): codex work agent's tmux session vanished 13s after spawn; `pan start` printed `✔ Agent spawned` and then hung >20 min.

Root cause of the boot death — `codex exec -s workspace` (raw sandbox token rejected by the codex CLI) plus the whole exec-mode work-agent design — was fixed the same day by hotfixes **already on `origin/main`**:

| Commit | Fix |
| --- | --- |
| `163e63c01` | translate sandbox token to `workspace-write`; init codex home for work agents |
| `eb79b04cf` | run codex work agents as persistent TUI sessions (`work-tui`) |
| `5e05abc06` | work agents inherit Settings codex permission level + pre-trust workspace |
| `b0a223c72` | detect work-tui readiness by input prompt + status line |
| `31760795c` | deliver kickoff via `.pan/kickoff.md` pointer, not a large TUI paste |
| `9857b1ac1` | `pan start --fresh` reset-session clears codex-thread-id (unwedges catch-22) |
| `fc17d1510` | pi: provider-qualify work-agent model ids |

**This plan does NOT re-do any of that.** It covers what remains:

1. **False success** — when kickoff delivery fails, `recordKickoffDeliveryFailure` (`src/lib/agents.ts:1707-1723`) sets status `running` and `spawnAgent` returns normally (`agents.ts:3684-3687`); `start.ts` then prints the full success block (`src/cli/commands/start.ts:~1127`) and exits 0. A dead agent is reported as a successful spawn.
2. **Dead-session waits** — `waitForCodexTuiReady` already bails when the session is gone (`agents.ts:262`), but `waitForReadySignal` (`agents.ts:783-804`, claude-code path) polls the full timeout against a dead session, and `deliverInitialPromptWithRetry` retries a second time against a session known to be dead, reporting only the generic `ready-signal-timeout`.
3. **CLI process hang** — in the incident `pan start` hung >20 min *after* printing success: undrained event-loop handles (fire-and-forget tracker HTTP in `transitionIssueToInProgress`, TLDR daemon exec, Effect runtime). Several pan commands already end with explicit `process.exit(0)` (`close.ts:217`, `health.ts:175`, `dev.ts:678`); `start.ts` does not.
4. **Zero boot diagnostics** — when the session dies, the pane (with the harness's fatal stderr, e.g. clap's "invalid value 'workspace'") is destroyed. The incident took four blind hotfix commits to root-cause. tmux `remain-on-exit` can preserve the dead pane long enough to capture it.
5. **No thread-id for work-tui agents** — only exec-mode specialists capture the rollout thread-id (`spawnRun`, `agents.ts:3303-3324`, explicitly skips `work-tui`). Nothing writes `codex-thread-id` for work-tui agents, so `getLatestSessionIdSync` fallback #4 (`agents.ts:2387-2396`) never fires and `CodexTranscriptSource.resolveAgentTranscript` returns null (`src/lib/memory/transcript-source.ts:226-233` via `readCodexThreadId:350-357`) — codex work agents are invisible to memory capture and session-id resolution.

## Requirements

### Functional

- **FR-1** Readiness waits detect a dead session early: `waitForReadySignal` returns within one poll interval when the tmux session no longer exists; `deliverInitialPromptWithRetry` reports a distinct `session-died` failure and does not attempt redelivery to a dead session.
- **FR-2** `spawnAgent` distinguishes boot death from kickoff failure: when the session is dead after delivery resolution, it marks the agent state `stopped` and throws a typed `AgentBootError` (agentId, harness, phase, diagnostic tail).
- **FR-3** Kickoff-failed-but-alive keeps existing ghost-lifecycle semantics (state `running`, `kickoffDelivered: false`, deacon redelivers) but `pan start` surfaces it as an explicit warning, never as unqualified success.
- **FR-4** `pan start` exits 1 with a diagnostic on boot death, and on success terminates promptly with exit code 0 (explicit `process.exit(0)` after the summary).
- **FR-5** Boot-window diagnostics: work-agent sessions launch with `remain-on-exit on`; on boot death the dead pane's tail is captured to `~/.overdeck/agents/<id>/boot-failure.log` and embedded in the `AgentBootError`; the remained session is then killed; after successful readiness the option is switched off so post-boot death semantics are unchanged.
- **FR-6** Codex thread-id resolves lazily: a `resolveCodexThreadId(agentId)` helper reads `codex-thread-id`, else scans the per-agent `codex-home/sessions/` for the newest rollout, writes the file back, and returns the id; wired into `getLatestSessionIdSync` fallback #4 and `readCodexThreadId` in transcript-source.

### Non-functional

- **NFR-1** No unbounded waits anywhere in the `pan start` spawn path; every poll loop has a deadline and a dead-session short-circuit.
- **NFR-2** All delay-based tests use `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()` (repo rule — never real `setTimeout` delays).
- **NFR-3** Deacon orphan-patrol and auto-resume semantics outside the boot window are unchanged.

## Decisions (auto mode)

- **Throw on dead session, return on alive-but-undelivered.** `AgentBootError` propagates naturally: `start.ts`'s catch → `spinner.fail` + exit 1; the dashboard start route → non-2xx. The alive case keeps the ghost lifecycle untouched.
- **`remain-on-exit` boot window over stderr redirection.** Redirecting the TUI's stderr risks breaking rendering; a remained dead pane preserves exactly what the user would have seen. The window is bounded: option flipped off right after readiness; dead session killed right after capture.
- **Lazy thread-id resolution over a background capture at spawn.** A post-spawn 30s poller would hold the CLI process open — the very hang this issue fixes. Per-agent `CODEX_HOME` makes "newest rollout" unambiguous.
- **Plain `process.exit(0)`** without flushing the fire-and-forget tracker transition: by the time the spawn summary prints, the transition has had the entire boot duration to land; it is best-effort and catch-logged by design.
