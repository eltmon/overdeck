## 1. Background & motivation

PAN-1791 landed the tiered-execution library layer: config-validated tier table, plan-time scheduling, lazy standing tiers, everyone-hears-everything ingestion-only commit feed, standing supervisor with verdict surface reuse, deterministic replay/compaction, and trivial fast-track. Operator direction (2026-07-02) plus the Devin Fusion comparison identified five gaps:

1. **Listeners are gagged.** The feed message orders total silence, so a warm tier that *sees* the working agent land a wrong commit cannot say so. Input tokens are cheap and cached; the marginal cost of vigilance is ~0 until a call-out is actually warranted.
2. **The feed has zero noise filtering.** `broadcastCommit` ships raw `git show` (up to 16MB) to every tier. A lockfile or generated-artifact commit blasts useless megabytes into every standing context — filtering should be *minimal* (only what doesn't help seeing the code), but not absent.
3. **Compaction is a wasted switching point.** Replay already pays the full cache miss. Devin Fusion's insight: that is the free moment to change models. Today we respawn onto exactly the same tier definition captured at spawn time.
4. **Difficulty misjudgment has no runtime correction.** Routing is fixed at plan time. A `simple`-labeled bead that is actually `complex` fails cheaply and repeatedly with no promotion path. (Fusion's failure case — a judgment-heavy task delegated to the sidekick — is exactly a mislabeled bead.) Fast-track has fail-open escalation; the other tiers have nothing.
5. **Inspection ignores the warm senior model.** `pan inspect` spawns a fresh ephemeral specialist with zero context while a standing supervisor — already warm on the entire commit feed — sits idle. Additionally, the `byKind` leg of the resolution chain has **no config surface**: nothing in `TieredExecutionConfig` supplies it (verified: no consumer of `byKind` outside `resolve-tier.ts` and tests), so kind-based routing (e.g. `design`/`spike` → top tier) is currently unreachable.

Everything below is config-gated with defaults that preserve current behavior, so each capability can be tuned or turned off as evaluation data accumulates.

## 2. Decisions (made here; do not re-litigate during planning or implementation)

| # | Decision | Rationale |
| --- | --- | --- |
| D1 | **Broadcast stays; filtering is subtractive noise removal only.** Per-tier relevance filtering (PAN-1791's original D3) stays dead. The filter removes: excluded pathspecs from diffs, whole commits by subject prefix, and oversized diffs (replaced by `--stat` + truncation note). | Operator intent: "filtering should be minimal — just stuff that wouldn't help it see the actual code." Input tokens are cheap; situational awareness is the product. |
| D2 | **One shared diff renderer for live feed and replay.** A single `renderCommitFeedDiff(workspace, sha, feedConfig)` is called by both `broadcastCommit` and replay's `loadReplayCommits` path. Excluded paths are dropped git-natively via `:(exclude)` pathspecs on `git show`. | Replay must reconstruct byte-identical feed messages (`composeCommitFeedMessage` is documented deterministic, `tier-feed.ts:52-55`, search `Deterministic over`); two filter implementations would drift. |
| D3 | **Call-outs are raised by models, adjudicated by policy — never self-executed.** The feed prompt (callouts enabled) permits exactly one structured call-out per delivery: a `curl POST` to a new `/api/tiered/callouts` endpoint. Policy decides the consequence: `notify` records + surfaces it; `corroborate` additionally triggers a supervisor review of that commit (via `deliverCommitForReview`) even if outside its subscription. A call-out NEVER blocks dispatch directly — only a supervisor BLOCKED verdict blocks. | Preserves PAN-1791 D1 (router, not listeners): a listener's opinion is an *event*; deterministic policy decides what happens. Listeners never implement, never self-assign. |
| D4 | **Escalation promotes the bead's effective difficulty one step up the ladder, on defined trigger events only.** Triggers: (a) supervisor BLOCKED verdict for the bead's commit; (b) verification-gate failure attributed to the bead; (c) floundering — no commit within the configured per-difficulty time budget (trigger inactive for a difficulty with no configured budget). Policy knobs: `retries_at_tier` (attempts at the current tier before promoting), `max_promotions` (per bead). Promotion re-routes via `difficultyToTier[nextDifficulty]` — validation already guarantees the mapping exists. An `expert` bead that triggers escalation cannot promote: it blocks and notifies the operator. | Deterministic event → rule → re-route; no model decides who works (PAN-1791 D1 intact). Extends the fast-track fail-open shape (`FastTrackEscalationReason`) to all tiers. Backward-movement detection is deferred (needs monitor infra, see Non-goals). |
| D5 | **Effective difficulty lives in workspace continue.json (`tierOverrides`), never in the spec.** New map keyed by bead id: `{ effectiveDifficulty, promotions, history[] }`. Routing reads it by overlaying `metadata.difficulty` before `resolveTier`. | PAN-1124: the spec on main is immutable after planning; workspace continue.json is the established runtime channel (`statusOverrides` precedent, `src/lib/vbrief/io.ts`, search `flat \`statusOverrides\` map`). |
| D6 | **Compaction re-tiering re-resolves against current config + effective difficulties, and decommissions unneeded sessions.** With `compaction_reroute: on`, any replay (compaction or crash) first recomputes the remaining schedule — `computeTierRunSchedule` over items not completed per statusOverrides, with tierOverrides applied. If the target tier no longer appears in the remaining schedule, skip the respawn (return null; the session is done). Otherwise respawn with the tier's *current* (model, harness) from the freshly validated config. | This is Fusion's "model switching for free": the replay already pays the cache miss. Re-resolving picks up (1) escalation promotions and (2) mid-run operator tuning of the tier table, deterministically. Off (default) = exact current behavior. |
| D7 | **Supervisor owns inspection when `supervisor.owns_inspection: true`.** With tiered execution enabled for the issue and the flag on, `spawnInspectAgent` routes the request to the standing supervisor via `deliverCommitForReview` instead of spawning an ephemeral specialist. If the standing supervisor session is not alive, spawn it (`spawnTierSupervisor`) — fail loudly if that fails; NEVER silently fall back to the ephemeral path while the flag is on. Verdicts land on the same inspect-status surface either way, so `onInspectComplete` and the foreman wait path are unchanged. | Operator: "the senior model does the inspection." The supervisor is already warm on the full feed — an ephemeral inspector re-derives context the supervisor already holds. Same verdict plumbing = zero downstream change. No-bandaids: a silent fallback would hide a broken supervisor. |
| D8 | **`by_kind` becomes real config.** Add `by_kind?: Partial<Record<VBriefItemKind, string>>` to `TieredExecutionConfig`, validate (known kinds, tier names exist), thread through `ValidatedTieredExecutionConfig` so `spawn-prep.ts`'s `assignDispatchTier` call picks it up structurally. Docs RECOMMEND routing `design` and `spike` to the top tier; this is a documented recommendation, never a hardcoded default. | The chain leg exists (`resolve-tier.ts:49`, search `byKind?: Partial`) but is unreachable from config. Fusion's "judgment is the deliverable" failure is prevented by kind-routing judgment-shaped items to the senior tier. Hardcoding it would violate the no-hardcoded-model-fallback rule. |
| D9 | **All new knobs default to off/absent; parity is a test, not a promise.** Every new config field's default reproduces post-PAN-1791 behavior byte-for-byte. The parity suite is extended to lock this. | The operator tunes over time; the baseline must be provably unchanged until a knob is turned. |
| D10 | **Runtime loop wiring stays out of scope.** Like PAN-1791, this issue delivers pure/library seams, config, prompts, and one route. The foreman loop (PAN-2203) calls them. The floundering trigger ships as a pure predicate over timestamps, not a timer. | Avoids re-implementing the foreman; keeps every piece unit-testable without wall-clock. |

## 3. Intersecting repo rules (restated — do not assume you recall them)

- **No hardcoded model fallbacks.** No code path may fall through to a model literal. Promotion, corroboration, and inspection routing all resolve through the validated tier table / supervisor config or fail loudly.
- **Fake timers** for any retry/backoff/delay test: `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()`. The floundering predicate is pure over timestamps precisely so its tests need no timers at all; any test that *does* exercise delays must use fake timers.
- **No `execSync`** in dashboard-server-reachable code (the callout route!). Use `execFile`/`exec` promisified, as `tier-feed.ts` and `tier-replay.ts` already do.
- **Async delivery primitives only.** All agent messaging via `deliverAgentMessage`; no new transport, no raw tmux writes.
- **PAN-1124 invariant.** Specs on main are immutable; all runtime state (statusOverrides, NEW tierOverrides) goes to workspace continue.json.
- **One bead = one commit.** Nothing here may weaken it; the feed, call-outs, supervision, and escalation all assume it.
- **Registered agents only.** Standing tiers/supervisor spawn through `spawnRun` — never harness `Agent`-tool subagents.
- **Karpathy rules.** Minimum code; no speculative abstraction; surgical diffs; every changed line traces to a work item below.
- **Commit often on main** during implementation; each work item its own commit. Update `docs/TIERED-EXECUTION.md` in the same effort (WI-10).

## 4. Requirements

### Functional

- **FR-1 — Feed noise filter.** `broadcastCommit` and replay deliver diffs rendered by one shared function honoring: `feed.exclude` (pathspec globs removed from the diff via `git show <sha> -- . ':(exclude)<glob>'…`), `feed.exclude_subjects` (commit-subject prefixes whose commits are skipped entirely — no delivery, no replay), and `feed.max_diff_bytes` (oversized diffs replaced by `git show --stat <sha>` output plus an explicit truncation note naming the byte size and the config knob). Defaults: all three absent → current raw `git show` behavior.
- **FR-2 — Vigilant-listener prompt.** With `feed.callouts` ≠ `off`, the ingestion-only feed message gains a call-out clause: stay silent by default; if you believe this commit is wrong (spec misread, hollowed test, broken invariant), you may raise AT MOST ONE call-out via the documented `curl POST` to `/api/tiered/callouts`, then return to silence. With `feed.callouts: off` (default) the message is byte-identical to today's.
- **FR-3 — Call-out endpoint + policy.** `POST /api/tiered/callouts` accepts `{ issueId, sha, beadId?, tierName, agentId, claim }`, validates the issue has tiered execution enabled, records the call-out (event log + review-status note), and applies policy: `notify` → record + surface (activity/TTS event, feedback line to the foreman session via `deliverAgentMessage`); `corroborate` → additionally trigger `deliverCommitForReview` for that commit to the standing supervisor regardless of its subscription policy. A call-out never mutates dispatch state, never blocks, never edits tracker status.
- **FR-4 — Escalation engine.** A pure decision function maps (trigger event, bead, escalation config, current tierOverrides) → `{ action: 'retry' | 'promote' | 'block' , nextDifficulty?, reason }` implementing D4's ladder, `retries_at_tier`, and `max_promotions`. Trigger wiring: (a) supervisor BLOCKED — in the specialists route where a failed inspect verdict with a `"Bead <id>"` prefix is recorded (`specialists.ts:514-527` region); (b) verification-gate failure attributed to the in-flight bead (seam invoked by the foreman loop with `runVerificationForIssue`'s result); (c) floundering — pure predicate `isFloundering(dispatchedAt, now, budgetMinutes)` active only when a budget is configured for the bead's effective difficulty.
- **FR-5 — Effective-difficulty state.** `tierOverrides` map in workspace continue.json: `{ [beadId]: { effectiveDifficulty, promotions, history: [{at, from, to, reason}] } }`, written by a new io helper sibling to the statusOverrides writers; routing applies it by overlaying `metadata.difficulty` before calling `resolveTier` / `computeTierRunSchedule`. The spec on main is never touched.
- **FR-6 — Compaction re-tiering.** With `compaction_reroute: on`, every replay path (compaction and crash) recomputes the remaining schedule (per D6) before respawn: decommission if the tier is no longer needed; otherwise respawn with the currently-configured (model, harness) for the tier. With the flag off (default), behavior is exactly today's `replayStandingAgent`.
- **FR-7 — Supervisor-owned inspection.** With tiered execution enabled for the issue and `supervisor.owns_inspection: true`, the `spawnInspectAgent` path delivers the inspection as a supervisor review request (`deliverCommitForReview`) to the standing supervisor, spawning it first if absent; on spawn failure the inspect request fails loudly (no silent ephemeral fallback). Flag off (default): the ephemeral inspect specialist path is byte-identical to today's.
- **FR-8 — `by_kind` config.** `tiered_execution.by_kind` maps `VBriefItemKind` → tier name; validated (kind values against the `VBriefItemKind` union, tier names must exist in `tiers`); carried on `ValidatedTieredExecutionConfig` so `assignDispatchTier` (via `spawn-prep.ts:237`, search `assignDispatchTier(item, tiered, planMetadata)`) resolves it with zero call-site changes.
- **FR-9 — Config validation for every new knob.** All new fields validate in `validateTieredExecutionConfig` with fail-loud errors (unknown policy strings, non-positive byte/minute values, unknown kinds/difficulties/tier names), following the existing `TieredExecutionConfigError` style.

### Non-functional

- **NFR-1 — Parity.** With every new knob at its default, all existing behavior is byte-identical: feed messages, replay deliveries, inspect spawning, replay respawn, dispatch routing. Locked by extending the parity suite (WI-9).
- **NFR-2 — Replay determinism.** For the same (commit set, feed config), replayed feed messages are byte-identical to the live ones — including filtered and truncated forms.
- **NFR-3 — Output-cost discipline.** Feed deliveries remain ingestion-only by default; the call-out clause bounds listeners to at most one short call-out per delivery.
- **NFR-4 — No new transports, no hardcoded models, no `execSync` in server-reachable code.**

## 5. Design detail

### 5.1 Config surface after this issue (complete)

```yaml
tiered_execution:
  enabled: false                    # existing
  replay_threshold: 0.5             # existing
  tiers:                            # existing
    junior:  { model: <id>, harness: <name>, difficulties: [trivial, simple] }
    mid:     { model: <id>, harness: <name>, difficulties: [medium] }
    senior:  { model: <id>, harness: <name>, difficulties: [complex, expert] }
  supervisor:                       # existing block, one new key
    model: <id>
    harness: <name>
    subscribe: all | flagged | sampled
    owns_inspection: false          # NEW (FR-7, D7)
  by_kind:                          # NEW (FR-8, D8) — optional; recommend design/spike -> senior tier in docs
    design: senior
    spike: senior
  feed:                             # NEW block (FR-1..FR-3)
    callouts: off | notify | corroborate    # default off
    exclude: []                     # pathspec globs, e.g. ["bun.lock", "dist/**", "*.map"]
    exclude_subjects: []            # commit-subject prefixes, e.g. ["chore(beads):", "chore(state):"]
    max_diff_bytes: null            # null = uncapped (current behavior)
  escalation:                       # NEW block (FR-4..FR-5)
    enabled: false
    retries_at_tier: 1              # attempts at current tier before promoting
    max_promotions: 1               # per bead
    flounder_budget_minutes: {}     # per-difficulty, e.g. { simple: 20, medium: 45 }; unset difficulty = trigger inactive
```

Per-issue override: `plan.metadata.tiered_execution: 'on' | 'off'` already exists (`resolveTieredExecutionEnabled`, `tier-table.ts:67`, search `export function resolveTieredExecutionEnabled`) and gates ALL of the above for the issue.

### 5.2 Feed message — before/after (FR-2)

Before (current, `tier-feed.ts:56-71`, `composeCommitFeedMessage`):

```
This is an ingestion-only feed delivery. Read the diff below to stay
current with work landing on this issue. Do NOT respond to this message,
do NOT take any action, and do NOT produce output — wait for your next
dispatch.
```

After (callouts `notify`/`corroborate`; `off` keeps the text above byte-identical):

```
This is an ingestion-only feed delivery. Read the diff below to stay
current with work landing on this issue. By default: do NOT respond, do
NOT take any action, do NOT produce output — wait for your next dispatch.

ONE exception. If you believe this commit is WRONG — it misreads the bead's
spec, hollows out a test, or breaks a stated invariant — you may raise at
most one call-out, then return to silence:

  curl -X POST <apiUrl>/api/tiered/callouts \
    -H "Content-Type: application/json" \
    -d '{"issueId":"<issueId>","sha":"<sha>","tierName":"<yourTier>","agentId":"<yourAgentId>","claim":"<one sentence: what is wrong and where>"}'

Do not fix it yourself. Do not edit files. A call-out is a flag, not a task.
```

The signature grows to `composeCommitFeedMessage(sha, beadTitle, diff, callout?: {apiUrl, issueId, tierName, agentId})` — the optional argument keeps every existing call site and replay byte-compatible when callouts are off. Replay passes the same callout context so NFR-2 holds.

### 5.3 Escalation ladder mechanics (FR-4, D4)

Promotion moves the bead's **effective difficulty** one index up `TIERED_EXECUTION_DIFFICULTIES` (`tier-table.ts:10`); the tier follows from `difficultyToTier`, which validation guarantees is total and unique. Sketch of the pure engine (new file `src/lib/agents/tier-escalation.ts`):

```ts
export type EscalationTrigger =
  | { kind: 'supervisor-blocked'; beadId: string; sha: string }
  | { kind: 'verification-failed'; beadId: string; detail: string }
  | { kind: 'floundering'; beadId: string; dispatchedAt: string; now: string };

export type EscalationAction =
  | { action: 'retry'; attempt: number }
  | { action: 'promote'; from: VBriefDifficulty; to: VBriefDifficulty; reason: string }
  | { action: 'block'; reason: string };  // expert bead or max_promotions reached -> operator

export function decideEscalation(
  trigger: EscalationTrigger,
  bead: Pick<VBriefItem, 'id' | 'metadata'>,
  config: ValidatedEscalationConfig,
  overrides: TierOverridesMap,
): EscalationAction
```

`isFloundering(dispatchedAtIso, nowIso, budgetMinutes)` is a pure timestamp predicate — the PAN-2203 foreman loop supplies the clock. The engine emits; the router acts (mirrors `escalateFastTrackItem`'s "this module only emits the signal; the foreman acts on it" contract, `fast-track.ts`, search `This module only emits the signal`).

### 5.4 Compaction re-tiering (FR-6, D6)

`compactAtTierRunBoundary` currently stops the old agent and calls `replayStandingAgent(target)` with the tier captured at spawn time. With `compaction_reroute: on`, insert a re-resolution step:

```ts
// inside the reroute-enabled path, before spawnReplayTarget:
const remaining = remainingItems(doc, statusOverrides, tierOverrides);   // not completed
const schedule  = computeTierRunSchedule({ ...doc, plan: { ...doc.plan, items: remaining } }, currentConfig);
if (!tiersNeededForSchedule(schedule).includes(target.tierName)) return null;  // decommission
const tier = currentConfig.tiers[target.tierName];                       // freshly-validated model+harness
```

Crash replay (`replayCrashedStandingAgent`) goes through the same gate so a dead session for a no-longer-needed tier is not resurrected.

### 5.5 Supervisor-owned inspection (FR-7, D7)

`spawnInspectAgent` (in `src/lib/cloister/inspect-agent.ts`; sub-role selection at `:144`) gains a routing gate at its top: resolve the issue's tiered-execution state (`resolveTieredExecutionEnabled` + validated config); if enabled and `supervisor.owns_inspection`, resolve the standing supervisor agent id (`supervisorAgentId(issueId)`, `tier-supervisor.ts:55`, search `export function supervisorAgentId`), ensure the session is alive (spawn via `spawnTierSupervisor` if not), and call `deliverCommitForReview` with the bead's item, sha (from the inspect checkpoint diff base machinery already imported by `inspect.ts`, search `getDiffBase`), and PRD markdown (`loadPrdDraft`). The verdict path is unchanged: the supervisor already posts `"Bead <id>"`-prefixed verdicts to `/api/specialists/done`, which drives `update.inspectStatus` and `onInspectComplete` exactly as the ephemeral inspector does.

## 8. Measurement & evaluation

PAN-1791's `tier-measurement` item (feed-delivery stamping + per-tier cost rows) is the substrate; if it did not merge with PR #2223, it becomes a prerequisite bead here. The pilot for v2 should publish Devin-Fusion-style headline numbers on one issue class: (1) total cost tiered vs. monolithic at equal review outcome, (2) % beads escalated and the cost of escalated rework vs. their original tier savings, (3) call-out precision (call-outs that a supervisor corroborated / total call-outs), (4) warm-cache delivery fraction with the noise filter on vs. off. These numbers drive tuning of `retries_at_tier`, budgets, sample rates, and the exclude lists — the reason everything is config.
