# PAN-1134 — Pi harness hook-equivalent event channel

## Problem

Pi agents are silent to the dashboard's domain-event system. The dashboard tracks every Claude Code agent through nine Claude Code hooks (`PreToolUse`, `PostToolUse`, `Stop`, `SessionStart`, `Notification`, `UserPromptSubmit`, `PreCompact`, `PostCompact`, `PermissionRequest`). Each hook POSTs to `/api/agents/:id/heartbeat`, which decodes the body into a typed `DomainEvent` and folds it into `AgentRuntimeSnapshot`.

Pi has no hooks. Pi agents do not appear in activity tracking, cost tracking, permission relay, or completion detection. `pan admin hooks install` writes the same registry regardless of harness, so nothing warns the user that Pi agents run with degraded telemetry.

This issue closes the gap.

## The right architecture: extension POSTs, no poller

The Pi extension already runs as a process loaded by `pi --extension`. It already writes filesystem markers (`ready.json`, `completed`, `heartbeat.json`) for the cloister to read. **Do not invent a new channel for domain events.** The dashboard already runs an HTTP server. The hooks already POST to a validated endpoint. The Pi extension can do the same thing.

```
Claude Code:    hook script  ──POST──►  /api/agents/:id/heartbeat ──►  bodyToEvent → decodeDomainEvent → appendAsync
Pi:             extension    ──POST──►  /api/agents/:id/heartbeat ──►  (same path, same validation, same fold)
```

Same endpoint, same validation, same code path. No JSONL file. No poller. No race window. No schema-validation bypass. No new background service in the dashboard.

**Filesystem markers stay for what they are: spawn-readiness signals (`ready.json`) and completion markers (`completed`) that cloister reads on demand.** Domain events flow through HTTP because they must flow continuously and reach the same reducer.

## What the Pi extension must POST

Map Pi's lifecycle events to the existing `bodyToEvent` kinds. The mapping is constrained by what Pi actually emits today (`session_start`, `tool_execution_end`, `turn_end`):

| Pi event | POST body `kind` | Notes |
|---|---|---|
| `session_start` | `model_set` | `model: "pi"`, `claudeSessionId: event.sessionId ?? undefined` (the field is a misnomer; rename in a follow-up if you must, but don't block this issue on it) |
| `session_start` (second POST) | `activity` with `activity: "idle"` | The Claude `session-start-hook` does the same — bootstrap snapshot to `idle` |
| `tool_execution_end` | `activity` with `activity: "working"`, `tool: event.toolName` | Maps to Claude's PostToolUse heartbeat |
| `turn_end` | `activity` with `activity: "idle"` | Maps to Claude's Stop hook |

The six events that Pi cannot fire (`PreToolUse`, `Notification`, `PreCompact`, `PostCompact`, `UserPromptSubmit`, `PermissionRequest`) stay unfired. **Document the gap in `docs/HOOKS.md`. Do not work around it.** The follow-up work (permission relay, completion detection, specialist auto-completion, cost tracking) is intentionally out of scope for this issue — track them as separate beads.

## Network failure handling

The Claude Code hooks share a small library at `scripts/pan-hook-lib.sh` that handles dashboard-down cases by buffering events to a pending file on disk and retrying on the next hook fire. **The Pi extension needs the same behavior.** Don't drop events when the dashboard is unreachable.

Concrete approach:

1. On POST failure (any reason — connection refused, non-2xx, timeout), append the body to `~/.panopticon/agents/<agentId>/pending-events.jsonl`.
2. On every subsequent successful POST, flush pending events first in FIFO order (and continue on failure — pending events stay pending).
3. Use a small timeout (1s). The extension must never block Pi.

This puts the buffer in the extension process where the events originate, not in a new dashboard-side consumer.

## Files to change

### Add to the Pi extension (`packages/pi-extension/src/index.ts`)

A small HTTP client function `postEvent(agentId, body)` that:

- Reads the dashboard URL from env (`PANOPTICON_DASHBOARD_URL`, default `http://localhost:3010`)
- POSTs `{ ...body, timestamp: ISO-now }` to `/api/agents/:id/heartbeat`
- On any failure, appends to `~/.panopticon/agents/<agentId>/pending-events.jsonl`
- On success, flushes pending events FIFO

## Why not POST from within Pi itself?

The Pi extension runs inside the Pi process. The extension API exposes lifecycle hooks (`pi.on('tool_execution_end', ...)`) that Pi calls synchronously from its turn loop. As long as the POST is non-blocking (fire-and-forget with timeout), Pi keeps running normally even if the dashboard is down. The pending-events buffer means events still arrive in order once the dashboard comes back.

This is the same trade-off the Claude Code hooks make — they POST and disk-buffer on failure. The Pi extension inherits the same model with the same guarantees.
