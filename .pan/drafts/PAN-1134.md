# PAN-1134 — Pi harness hook-equivalent event channel (full Claude Code parity)

> **Scope expansion (2026-05-27).** Originally this issue covered only the
> architectural cut-over (extension-POSTs vs JSONL+poller). It now also folds
> in the four feature gaps that were previously deferred as follow-ups —
> work-agent completion detection, specialist auto-completion, per-tool cost
> tracking, and harness-aware installer — so Pi reaches functional parity
> with Claude Code in a single ship. See "Why bundle these now" below.

## Problem

Pi agents are silent to the dashboard's domain-event system. The dashboard
tracks every Claude Code agent through nine Claude Code hooks (`PreToolUse`,
`PostToolUse`, `Stop`, `SessionStart`, `Notification`, `UserPromptSubmit`,
`PreCompact`, `PostCompact`, `PermissionRequest`). Each hook POSTs to
`/api/agents/:id/heartbeat`, which decodes the body into a typed `DomainEvent`
and folds it into `AgentRuntimeSnapshot`.

Pi has no hooks. Pi agents do not appear in activity tracking, cost tracking,
permission relay, or completion detection. `pan admin hooks install` writes
the same registry regardless of harness, so nothing warns the user that Pi
agents run with degraded telemetry.

The Pi extension API surface is narrower than Claude Code's hook set —
`pi.on(...)` exposes three lifecycle events (`session_start`,
`tool_execution_end`, `turn_end`) — so full Claude-Code parity is bounded
by what Pi exposes. Where Pi has no equivalent event, document the gap and
file an upstream request; do not paper over it.

## Why bundle these now

The original split was: land the architecture, then file four follow-ups for
the feature gaps. Reverting the split because:

1. **No autonomous Pi pipeline without completion detection.** A Pi work
   agent that finishes its work never advances. Every Pi run requires manual
   review-handoff. Shipping the architecture without completion detection
   ships a harness no one will use.
2. **Specialist auto-completion has the same problem at review/test/merge.**
   The Pi pipeline is end-to-end broken without it. Same root cause, same
   fix shape — bundling avoids three round-trips through plan/work/review.
3. **The four follow-ups touch the same files.** `packages/pi-extension/src/index.ts`,
   `docs/HOOKS.md`, `src/dashboard/server/routes/agents.ts`, and
   `src/lib/admin/hooks.ts` get edited in each one. Splitting forces four
   merge cycles through the same code.
4. **No feature is shippable in isolation.** Cost tracking without
   completion detection is half a Pi pipeline; harness-aware installer
   without the features it gates on is dead UX.

## The right architecture: extension POSTs, no poller

The Pi extension already runs as a process loaded by `pi --extension`. It
already writes filesystem markers (`ready.json`, `completed`, `heartbeat.json`)
for the cloister to read. **Do not invent a new channel for domain events.**
The dashboard already runs an HTTP server. The hooks already POST to a
validated endpoint. The Pi extension can do the same thing.

```
Claude Code:    hook script  ──POST──►  /api/agents/:id/heartbeat ──►  bodyToEvent → decodeDomainEvent → appendAsync
Pi:             extension    ──POST──►  /api/agents/:id/heartbeat ──►  (same path, same validation, same fold)
```

Same endpoint, same validation, same code path. No JSONL file. No poller.
No race window. No schema-validation bypass. No new background service in
the dashboard.

**Filesystem markers stay for what they are: spawn-readiness signals
(`ready.json`) and completion markers (`completed`) that cloister reads on
demand.** Domain events flow through HTTP because they must flow
continuously and reach the same reducer.

## What the Pi extension must POST

Map Pi's lifecycle events to the existing `bodyToEvent` kinds:

| Pi event | POST body `kind` | Notes |
|---|---|---|
| `session_start` | `model_set` | `model: "pi"`, `claudeSessionId: event.sessionId ?? undefined` |
| `session_start` (second POST) | `activity` with `activity: "idle"` | Bootstraps snapshot to `idle` — matches Claude `session-start-hook` |
| `tool_execution_end` | `activity` with `activity: "working"`, `tool: event.toolName` | Maps to Claude PostToolUse heartbeat |
| `tool_execution_end` | `cost-event` with per-tool cost | New — see "Cost tracking" below |
| `turn_end` | `activity` with `activity: "idle"` | Maps to Claude Stop hook |
| `turn_end` (work agents) | completion classification → `/api/agents/:id/work-complete` | New — see "Completion detection" below |
| `turn_end` (specialist agents) | pattern match → `/api/specialists/:type/auto-complete` | New — see "Specialist auto-completion" below |

Pi-API gaps (`PreToolUse`, `Notification`, `PreCompact`, `PostCompact`,
`UserPromptSubmit`) stay unfired. **Document the gap in `docs/HOOKS.md`.
Do not work around it.** File a separate upstream-Pi request issue capturing
what events we'd like Pi to add.

`PermissionRequest` has no Pi equivalent because Pi has no permission system.
Document as N/A, not deferred work.

## Feature scope (folded back in)

### 1. Work-agent completion detection (was deferred)

Mirror `scripts/work-agent-stop-hook` four-phase logic inside the extension's
`turn_end` handler, gated on agent role (`OVERDECK_AGENT_ROLE === 'work'`):

1. **Evidence scan** — read `<workspace>/.pan/continue.json` and
   `<workspace>/.beads/issues.jsonl`; if all beads are closed and the work
   vBRIEF shows all items satisfied, POST `work-complete` with reason
   `evidence-clean`.
2. **Structured reply scan** — tail Pi's session log (the same one Pi
   writes for its turn loop) for a structured completion marker the work
   role's prompt emits. If found, POST `work-complete` with the marker text.
3. **LLM fallback classification** — when evidence + structured reply are
   inconclusive, the dashboard already exposes
   `/api/agents/:id/classify-completion` which invokes the classifier model.
   Extension POSTs to that endpoint and forwards the verdict.
4. **Stuck escalation** — track consecutive `turn_end` events with no
   progress (no tool calls, no bead transitions, no continue.json delta).
   After N (config: `pi.stuckTurnThreshold`, default 3) POST
   `/api/agents/:id/stuck` so the deacon can mark the agent troubled.

Same endpoint targets as Claude Code's `work-agent-stop-hook`. New code
in the extension lives next to `handleTurnEnd` and routes on
`OVERDECK_AGENT_ROLE`.

### 2. Specialist auto-completion (was deferred)

Mirror `scripts/specialist-stop-hook` for review/test/merge/inspect/uat
agents. On `turn_end`, when `OVERDECK_AGENT_ROLE` is a specialist role:

- Tail Pi's session log for the same marker patterns the Claude Code hook
  matches (`INSPECTION PASSED`, `INSPECTION BLOCKED`, `TESTS PASSED`,
  `TESTS FAILED`, merge-success / merge-blocked, etc. — exact patterns live
  in `scripts/specialist-stop-hook` today).
- POST to `/api/specialists/<role>/auto-complete` with the matched verdict
  and any extracted summary.

Pattern table moves out of the bash script into a shared TypeScript module
(`src/lib/specialist-completion-patterns.ts`) consumed by both the
extension and a thin Claude Code hook wrapper. Removes the bash/TS split.

### 3. Cost tracking per tool (was deferred)

`tool_execution_end` POSTs an `activity: working` event today. Extend so it
also POSTs a `cost-event`:

- New body kind in `bodyToEvent`: `cost-event` with fields
  `{ tool: string, usage: { inputTokens, outputTokens, ... } | null,
  costUsd: number | null, agentRole, agentId, timestamp }`.
- Pi exposes per-turn token usage in the `tool_execution_end` payload
  (verify against Pi 1.x API surface in the spike; if not exposed,
  approximate from session-log tail). If neither path yields a number,
  emit the event with `usage: null` and let the dashboard backfill from
  Pi's session log on completion.
- Mirror the `flock` serialization from `record-cost-event.js`: the
  extension writes to a per-agent append-only file
  (`~/.overdeck/agents/<id>/cost-events.jsonl`) using
  `proper-lockfile` (already a dashboard dep). Dashboard reads on demand;
  no poller.

If Pi's per-tool cost data is not available in the documented API surface,
narrow this gap-fill to a per-turn cost event from `turn_end` and capture
the per-tool granularity gap as an upstream-Pi request.

### 4. Harness-aware installer (was deferred)

`pan admin hooks install` today writes the Claude Code hook registry
regardless of which harness will run. Add Pi awareness in
`src/lib/admin/hooks.ts`:

- Detect installed harnesses by probing `which claude` and `which pi`.
- New `--harness <claude-code|pi|both>` flag. Default: `both` if both
  binaries exist, otherwise the one that exists, otherwise hard-fail with
  an install hint.
- For Pi: verify the Pi extension is built (`packages/pi-extension/dist/index.js`
  exists) and the user's Pi config (`~/.pi/config.json` or env override)
  registers the extension path. If not, print the exact commands to fix it.
- For both: install Claude Code hooks **and** verify Pi extension wiring.
  Warn if one is present but the other isn't and the other binary is
  installed.
- `pan admin hooks status` shows per-harness registration so the user can
  see at a glance which agents will emit events.

## Network failure handling

The Claude Code hooks share a small library at `scripts/pan-hook-lib.sh`
that handles dashboard-down cases by buffering events to a pending file on
disk and retrying on the next hook fire. **The Pi extension needs the same
behavior** (already in scope from the original architecture work — keep it).

1. On POST failure (any reason — connection refused, non-2xx, timeout),
   append the body to `~/.overdeck/agents/<agentId>/pending-events.jsonl`.
2. On every subsequent successful POST, flush pending events first in FIFO
   order (and continue on failure — pending events stay pending).
3. Use a small timeout (1s). The extension must never block Pi.

For the new endpoints (`work-complete`, `specialists/:type/auto-complete`,
classify-completion, stuck): same buffer-on-failure model. They all flow
through one shared `postEvent` helper.

## Validation

```bash
npm run typecheck
npm run lint
npm test
```

All three must pass.

Manual validation:

1. `pan up` — start the dashboard.
2. Spawn a Pi work agent on a small test issue (use an issue with a
   single trivial bead so completion fires quickly).
3. Verify activity ticker goes `idle → working → idle` in real time.
4. Verify the agent advances to review without manual intervention once
   work is complete (completion-detection success path).
5. Spawn a Pi review specialist on a clean PR. Verify it auto-completes
   to `passed` and the merge agent picks it up.
6. Open the agent's cost panel — per-tool entries must be visible (or, if
   Pi doesn't expose per-tool usage, per-turn entries with a documented
   downgrade banner).
7. `pan admin hooks install` on a fresh box with only `pi` installed —
   skips Claude registry, configures Pi extension. With only `claude` —
   skips Pi. With both — does both.
8. Kill the dashboard mid-conversation, drive a tool call, restart —
   pending events arrive on the next POST.

## Acceptance criteria

- [ ] Pi extension POSTs `model_set` and three `activity_changed` event
      kinds to `/api/agents/:id/heartbeat`.
- [ ] No filesystem-based event channel (`events.jsonl`) exists in the
      Pi extension; no background poller exists in the dashboard server;
      `getPendingEvents` is gone from `PiRuntime`.
- [ ] Pi events flow through `bodyToEvent → decodeDomainEvent → appendAsync`.
- [ ] Pi extension buffers to `pending-events.jsonl` on POST failure and
      flushes on next success.
- [ ] **Pi work agents auto-advance** to review on completion (four-phase
      detection).
- [ ] **Pi review / test / merge specialists auto-complete** via pattern
      match; pattern table extracted to `src/lib/specialist-completion-patterns.ts`
      and shared with the Claude Code hook path.
- [ ] **Per-tool (or per-turn fallback) cost events** flow into the same
      cost view the Claude Code agents populate; serialization holds
      under concurrency.
- [ ] **`pan admin hooks install`** detects installed harnesses and
      configures the right registry; `--harness` flag exposes per-harness
      install; `pan admin hooks status` shows per-harness registration.
- [ ] All test cases pass under `npm test`; root `vitest.config.ts`
      discovers `packages/**` tests.
- [ ] `docs/HOOKS.md` updated: HTTP-POST architecture, per-feature parity
      matrix, three documented Pi-API gaps, `PermissionRequest` N/A note,
      `notification-hook` / `pre-compact-hook` row corrections, missing
      agents added to per-agent inventory.
