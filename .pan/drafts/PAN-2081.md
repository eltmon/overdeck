## Glossary

- **Epic** — a GitHub issue that is a *container* of child issues, not directly workable.
  Detected by a `[EPIC]` title prefix or an `epic` label. Carries `isEpic: true` on its
  `SequenceNode`. Example: [#2075](https://github.com/eltmon/overdeck/issues/2075).
- **`contains` edge** — a membership edge `epic → child` (`SequenceEdge.type === 'contains'`).
  The single source of truth for parent/child membership (Phase 1 deliberately did **not**
  add `epicId`/`childIssues` node fields — membership lives on the edge only).
- **sequence.md** — `.pan/backlog/sequence.md`, the source of truth for the ranked backlog.
  Holds a human-readable markdown table **and** a fenced machine-readable JSON block
  (`SequenceDoc`). `parseSequenceMd` reads the JSON block; `renderTable` writes the table.
- **`backlog_sequence` cache** — a SQLite table (`src/lib/database/backlog-sequence-db.ts`)
  that mirrors the *scalar ranking columns* of sequence.md for fast reads. It is a
  **disposable cache** rebuilt from sequence.md; it does **not** store edges (edges are read
  live from the parsed doc on every call).
- **DAG view** — `BacklogDAG.tsx`, the ReactFlow graph fed by `GET /api/backlog/sequence`.
- **Forecast view** — `BacklogForecast.tsx`, the waves/lanes pickup forecast fed by
  `GET /api/backlog/forecast`.
- **Pickable queue / waves / lanes** — `pickup.ts` computations over the *non-epic* nodes
  the Flywheel would actually pick up. Epics are already excluded (Phase 1), so they never
  appear in a wave or lane.

## Problem (Phase 2)

After Phase 1, an epic is correctly *modeled* (`isEpic`, `contains` edges) and correctly
*excluded* from pickup — but it is **invisible** to the operator. Both UIs render flat:

1. **DAG** (`BacklogDAG.tsx`) — epic nodes look identical to leaf issues; `contains` edges
   either don't render distinctly or aren't styled as membership. The operator can't see that
   #2075 is a container or which issues belong to it.
2. **Forecast** (`BacklogForecast.tsx`) — children of an epic scatter across waves/lanes with
   no indication they belong to a shared parent. The epic-as-unit-of-intent is invisible.

This is the issue's AC-4. It is **presentation-only over the same data** — no change to ranking,
pickup, or the pickup data model.

## Data flow (verified, file:line grounded)

Tracing how `isEpic` and `contains` reach each UI today:

### DAG path — `GET /api/backlog/sequence`

`src/dashboard/server/routes/backlog.ts:46-142` reads via
`getBacklogSequenceForRoot(projectRoot)` (`src/lib/database/backlog-sequence-db.ts:83-108`).

- **Edges (incl. `contains`) already reach the frontend.** `getBacklogSequenceForRoot`
  returns `edges: result.doc.edges.map((e) => ({ from, to, type }))` (`backlog-sequence-db.ts:97`)
  — `type` carries `'contains'` verbatim. The route passes `edges` straight through
  (`backlog.ts:137`). The frontend `SequenceEdge.type` is `string` (`BacklogDAG.tsx:53-57`).
  ✅ No backend change needed for `contains` edges on the DAG.
- **`isEpic` is dropped at the cache boundary.** The node read goes through
  `getBacklogSequence(projectKey)` (`backlog-sequence-db.ts:110-139`), which reads the SQLite
  `backlog_sequence` table whose columns (`schema.ts:711-726`) have **no** `is_epic`. The route's
  node map (`backlog.ts:118-134`) therefore can't emit `isEpic`. ❌ Must be threaded.

### Forecast path — `GET /api/backlog/forecast`

`src/dashboard/server/routes/backlog.ts:289-351` parses sequence.md **directly**
(`parseSequenceMd(readFileSync(seqPath))` `backlog.ts:303`) — so it already has full access to
`isEpic` (via `classifyIssue(...).epic`) and `contains` edges (`parsed.doc.edges`). But:

- Waves/lanes are built from `pickableQueue` (`pickup.ts:143-148`), which **excludes epics**
  (Phase 1). So epics never appear in `waves`/`lanes`. ❌ The response carries no epic list and
  no membership, so the UI has nothing to group by.

## Design — minimal-first

Two backend threads (one foundation bead) feed two independent UI beads, plus one independent
md-table bead. **No DB migration** (see Decision D2). Presentation only.

### Decisions made in this document

- **D1 — Scope is Phase 2 only.** AC-1/2/3 are done/structural-done on `main`. This plan
  delivers AC-4 (DAG + Forecast grouping) and the optional epic column. Do not touch Phase 1 code
  except to read from it.
- **D2 — Thread `isEpic` to the DAG via a live-doc join, NOT a DB column/migration.**
  `getBacklogSequenceForRoot` already reparses sequence.md on every call when it exists
  (`backlog-sequence-db.ts:89-99`) and reads **edges live from the parsed doc** rather than from
  the cache. `isEpic` is structurally identical — a node attribute the UI needs for presentation.
  Mirror the edge pattern: build an `isEpicByIssue` map from `result.doc.nodes` and attach
  `isEpic` to the returned nodes. This keeps `backlog_sequence` a disposable scalar-ranking cache,
  avoids a `SCHEMA_VERSION` bump (currently 57, `schema.ts:24`) and avoids touching the locked
  schema test (`src/lib/database/__tests__/backlog-sequence-schema.test.ts`). The only degradation
  is the rare md-absent fallback path (`backlog-sequence-db.ts:102-107`), which **already** returns
  `edges: []` — there, `isEpic` is reported `false`. Acceptable and symmetric: the DAG is purely
  presentational and epic *pickup safety* (AC-1) reads sequence.md directly via `pickup.ts`, never
  this cache.
- **D3 — Pure transform in `pickup.ts`.** Add a pure `computeEpicGroups(nodes, edges, lk)` to
  `pickup.ts` (the single-source-of-truth pure module) returning `{ epics, contains }`. Routes
  stay thin and the logic is unit-tested without booting the Effect HTTP server.
- **D4 — Forecast grouping is scatter-robust.** Children of an epic are not guaranteed to land in
  one wave. So the **primary** signal is a per-card epic membership **tag** (`▣ #2075`) that is
  always visible wherever the child lands. *Within* a wave, same-epic children are additionally
  ordered contiguously with a thin epic header. Tag is the robust core; the header is the visual
  grouping. Both read from `contains` — no new server computation.
- **D5 — md epic column is a display column only.** The JSON block remains the source of truth and
  already round-trips `isEpic`/`contains` (Phase 1). The table column is cosmetic; it does not
  change parse/write semantics.

### Work item 1 (foundation) — thread epic data through both read endpoints

**Files:** `src/lib/backlog/pickup.ts`, `src/lib/database/backlog-sequence-db.ts`,
`src/dashboard/server/routes/backlog.ts`.

**1a. `pickup.ts` — add a pure helper.** After `computeStats` (`pickup.ts:235-256`), add:

```ts
export interface EpicGroups {
  /** Epic container nodes (isEpic flag or `epic` label), rank-ordered. */
  epics: { issue: string; rank: number }[];
  /** Membership edges epic → child, from `contains` edges. */
  contains: { epic: string; child: string }[];
}

/** Epic containers + their membership, for the UI grouping layer (presentation only). */
export function computeEpicGroups(
  nodes: readonly SequenceNode[],
  edges: readonly { from: string; to: string; type: string }[],
  lk: ClassifyLookups,
): EpicGroups {
  const epics = nodes
    .filter((n) => classifyIssue(n, lk).epic)
    .map((n) => ({ issue: n.issue, rank: n.rank }))
    .sort((a, b) => a.rank - b.rank);
  const contains = edges
    .filter((e) => e.type === 'contains')
    .map((e) => ({ epic: e.from, child: e.to }));
  return { epics, contains };
}
```

`getBacklogSequenceForRoot` (`:83-108`), inside the `result.ok` branch, build a membership map


## Out of scope (avoid over-engineering)

- No DB schema migration / `is_epic` column (Decision D2).
- No `epicId`/`childIssues` node fields — membership stays on the `contains` edge (Phase 1 decision).
- No hierarchical dagre nesting in the DAG — render `contains` edges distinctly; visual nesting is a
  later refinement (issue "OUT of scope").
- No epic-affinity rewrite of `computeLanes`/`computeWaves`/`computeCohort` — grouping is
  presentation over the existing flat pickup data (issue "OUT of scope").
- No new POST route for manual epic linking — an `operator`-source `contains` edge already covers it
  (issue "OUT of scope").
- No change to pickup/ranking behavior — epics stay excluded from waves/lanes (Phase 1).

## Requirements

Functional (Phase 2):

- **FR-1** — `GET /api/backlog/sequence` returns `isEpic: boolean` on every node (sourced from
  sequence.md) and `contains` edges with `type: "contains"`.
- **FR-2** — `GET /api/backlog/forecast` returns `epics: { issue, title }[]` and
  `contains: { epic, child }[]`.
- **FR-3** — `BacklogDAG` renders an epic node with a visible `EPIC` affordance distinct from leaf
  nodes, and renders `contains` edges in a style distinct from `unblocks` and `informs`.
- **FR-4** — `BacklogForecast` tags each child card with its epic id and visibly groups a wave's
  same-epic children (contiguous + epic header).
- **FR-5** — `renderTable` emits an `epic` column in the human-readable sequence.md table.

Non-functional:

- **NFR-1** — No `backlog_sequence` schema migration; `isEpic` is read live from the parsed doc at
  the read door, mirroring the existing live-edge read.
- **NFR-2** — Phase 1 pickup behavior is unchanged: epics remain excluded from the pickable
  queue / waves / lanes. Phase 2 is presentation-only.
- **NFR-3** — Pure transform logic (`computeEpicGroups`, the wave grouping helper) lives in testable
  modules and is unit-tested without booting the HTTP server. No new direct store access (the
  single-source-of-truth CI guard stays green).
- **NFR-4** — UI follows `pan-style-guide` color-signal discipline (no decorative color; theme
  tokens only). Rendering is self-verified via Playwright screenshots, not operator eyeballing.
