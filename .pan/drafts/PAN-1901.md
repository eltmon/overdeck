## Glossary

- **Merge driver** — a program git invokes to perform a 3-way merge of a file matched by a
  `merge=<name>` attribute in `.gitattributes`. Git only invokes a *custom* driver named
  `<name>` if `merge.<name>.driver` is set in git config. If the attribute names a driver that
  is not configured (and is not a built-in), git silently ignores the attribute and falls back
  to its default line-based 3-way merge — leaving conflict markers on true divergence. **This is
  exactly the PAN-1841 bug.**
- **Built-in merge driver** — git ships three driver names that need NO config: `text` (default
  3-way), `binary` (mark conflicted, no textual merge), and `union` (take lines from *both*
  sides, no conflict markers). `union` is verified config-free in this repo (see Evidence).
- **Inert merge attribute** — a `.gitattributes` line `path merge=<name>` where `<name>` is
  neither a built-in nor a configured custom driver. The line looks like protection but does
  nothing. PAN-1901 exists because `merge=beads` is inert.
- **pan-dir auto-commit** — `src/lib/pan-dir/auto-commit.ts`. Background process that stages and
  commits `.pan/` and `.beads/` state changes. **It commits only when HEAD is `main`**
  (`doCommit`, `auto-commit.ts:250`: `if (branchResult !== 'main') return 'not on main'`).
  Feature branches therefore never auto-commit tracked `.pan/specs|continues|records` files.
- **JSONL vs single-JSON** — `.beads/issues.jsonl` is JSON-Lines (one independent object per
  line) → safe to `union` (concatenate lines; `bd` reconciles duplicate IDs on load). The
  `.pan/**` state files are *single JSON objects* → `union` would concatenate hunks into invalid
  JSON. They need a take-one-whole-side strategy, never union.

## Problem

PAN-1841 (CLOSED) added `.beads/issues.jsonl merge=beads` to `.gitattributes` to stop
`.beads`/`.pan` state from conflict-storming on integration merges. **That fix is inert:**

- `.gitattributes` declares `.beads/issues.jsonl merge=beads` (verified — `cat .gitattributes`).
- `git config --get merge.beads.driver` → **unset** (verified on the primary main worktree).
- No code configures it: `grep -rn "merge.beads.driver|git config merge" src scripts sync-sources infra` → **nothing**.
- `bd` has **no merge-driver subcommand** — `bd merge` is not a command (verified: only
  `merge-slot` and `duplicates` exist). So "wire a real `bd` driver" is **not available**; the
  named `beads` driver can never be configured to anything real.

Live consequence (issue Evidence, RUN-35): the flywheel's `git merge origin/main` conflicted on
`.beads/issues.jsonl` and was resolved by hand. The conflict recurs on essentially every
divergent integration. The `.pan/specs`, `.pan/continues`, and `.pan/records` single-JSON files
(also tracked, also auto-committed) have **no merge driver at all**.

### Evidence captured this session

- `git merge-file --union` on two divergent JSONL appends, with **no git config**, produced the
  union (`a,b,c,d`, no conflict markers, exit 0). The same inputs through the **default** driver
  conflicted (exit 1). → `union` is the correct, zero-config fix for the beads file.
- `.pan/specs/*.vbrief.json` (233 tracked), `.pan/continues/*.vbrief.json` (147 tracked),
  `.pan/records/*.json` (42 tracked) are all single JSON objects beginning with `{`.

## Proposal

Fix per file-class, and — critically — **make inertness detectable** so this class of bug
cannot silently recur:

1. **`.beads/issues.jsonl` → `merge=union`** (built-in, zero config). Replaces the inert
   `merge=beads`. This alone kills the observed storm.
2. **`.pan/{specs,continues,records}` single-JSON state → `merge=ours`** + wire
   `merge.ours.driver true` (the documented git recipe: the `true` command exits 0, keeping the
   current branch's whole file → never corrupts JSON). These files are owned by the on-`main`
   auto-commit and never edited on feature branches, so "keep ours" is safe; any transiently
   stale side self-heals on the next auto-commit flush (which re-derives state from sources of
   truth — GitHub, records, transcripts).
3. **Guard (the root-cause fix for PAN-1841's inertness):** a `pan doctor` check + unit test that
   fails when `.gitattributes` declares any `merge=<driver>` that is neither a git built-in
   (`union`/`text`/`binary`) nor backed by a configured `merge.<driver>.driver`. After this, an
   inert attribute is a loud, detected condition — not a silent no-op.
4. **Backfill existing clones:** wire the `merge.ours.driver true` config into a single
   idempotent helper, applied at workspace/worktree creation **and** unconditionally per primary
   repo during `pan sync`, so one `pan sync` heals every already-cloned project. (Today
   `sync.ts:329` sets git config only inside the beads-DB-missing `catch`, so it never re-runs on
   initialized repos — backfill must be unconditional.)

## Requirements

### Functional
- **FR-1** — `.beads/issues.jsonl` merges conflict-free via the built-in `union` driver, with no
  git config required.
- **FR-2** — The inert `merge=beads` attribute is removed from `.gitattributes`.
- **FR-3** — `.pan/specs/*.vbrief.json`, `.pan/continues/*.vbrief.json`, and `.pan/records/*.json`
  declare `merge=ours` and are never subjected to a corrupting `union` merge.
- **FR-4** — `merge.ours.driver true` is configured at every workspace/worktree creation site so
  the `merge=ours` attribute is active (not inert) in agent worktrees.
- **FR-5** — `pan doctor` detects and reports, as an `error`, any inert merge attribute (a
  `merge=<driver>` whose `merge.<driver>.driver` is unset and which is not a built-in), naming
  the offending pattern and driver and the command to fix it.
- **FR-6** — `pan sync` backfills `merge.ours.driver true` onto every configured project's
  primary repo unconditionally (not gated behind beads-DB init), healing existing clones.

### Non-Functional
- **NFR-1** — The guard's built-in allowlist is exactly `{union, text, binary}`; built-ins are
  never flagged as inert.
- **NFR-2** — `merge=ours` resolution never produces invalid JSON; a stale-but-valid side is
  acceptable because the pan-dir auto-commit re-stamps state from sources of truth on next flush.
- **NFR-3** — All git-config writes are idempotent and best-effort (failure is non-fatal, matching
  the existing `git config beads.role contributor` call sites).

## Auto Decisions (recorded — no interactive Q&A in `--auto`)

- **AD-1 — `union` for beads, not a "real beads driver."** The issue offered "wire a real
  `merge.beads.driver`" as option 1, but `bd` has no merge-driver subcommand (verified). `union`
  is the only viable conflict-free strategy and needs zero config. Forced choice.
- **AD-2 — `merge=ours` (config `true`), not a bespoke "take-newest by timestamp" script driver.**
  A timestamp-aware driver is more "correct" but requires a script present on PATH in every
  worktree — fragile and the exact failure mode of PAN-1841. `ours` is the documented config-free
  recipe, never corrupts JSON, and is safe because the auto-commit re-derives state. Recorded as
  hazard H2.
- **AD-3 — Scope `union` narrowly to `.beads/issues.jsonl`, never `*.json`.** A blanket
  `*.json merge=union` would corrupt the single-JSON `.pan` state files (operator's explicit
  correction in the issue comments).
- **AD-4 — Extend `merge=ours` to `.pan/records/*.json` too.** The issue names specs + continues;
  records (42 tracked) are the same single-JSON, auto-committed, per-issue class and would
  conflict identically. Including them is the complete fix.
- **AD-5 — Add `git config` per-site (one line beside each `beads.role` call) via a shared
  helper, rather than refactoring the four sites into one config function.** Keeps the change
  surgical while still centralizing the driver list in `git-merge-drivers.ts`.
- **AD-6 — `pan doctor` guard is detect-only (no `--fix`).** It reports the inert attribute and
  the heal command (`pan sync`); the fix path is bead 5. Avoids scope creep into a doctor
  auto-repair surface.

- The auto-commit/rebase race (PAN-1770) and `.pan/specs` mutated on feature branches (PAN-1728).
- Refactoring the four scattered `git config beads.role contributor` call sites into one helper

- A timestamp-aware / 3-way semantic JSON merge driver for `.pan` state (AD-2).
- Healing `beads.role` config on existing repos (only merge-driver config is in scope).
- Any change to `bd` itself.
