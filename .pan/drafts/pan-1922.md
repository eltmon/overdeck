# PAN-1922 — Make pipeline verdicts reconstruction-safe

> **PRD:** `.pan/drafts/pan-1922.md` — this file. Issue: [PAN-1922](https://github.com/eltmon/panopticon-cli/issues/1922).
> Part of the state-model simplification (`reference/state-model.mdx`). Sibling of [PAN-1919](https://github.com/eltmon/panopticon-cli/issues/1919). Depends on [PAN-1920](https://github.com/eltmon/panopticon-cli/issues/1920).

## 0. TL;DR for the executor — read this first

This PRD is **smaller than it looks**, because half of it already exists. Do **not** "build a durable home for verdicts." That writer already exists and is wired (PAN-1908): every `review_status` write mirrors the durable verdict block into the per-issue git record. **Do not reinvent it.**

What is missing is the **read / reconstruction direction**: when the SQLite cache is lost, nothing repopulates the `review_status` table from the durable record or from GitHub. `getReviewStatusSync` is a pure DB read with no fallback. This PRD builds that one missing arrow.

Your three deliverables:

1. A library function `restoreReviewStatusFromRecords()` that rebuilds `review_status` rows from the per-issue git records (the **durable verdict** subset).
2. A live GitHub re-derivation of the **PR-owned merge-state** subset (blockers) during that restore, reusing the existing `refreshMergeStateFromGitHub()`.
3. A `pan admin db restore-verdicts` CLI command that runs (1)+(2), mirroring the existing `pan admin db backfill-records` command.

Plus a **no-loss audit test** that enumerates every `review_status` column and proves each is either restored from a named source or explicitly on the ephemeral allowlist.

## 2. Problem

Pipeline **verdicts** live only in the SQLite `review_status` table. They are the one piece of pipeline state that is **not** derivable from the workspace alone (the workspace has the code and the plan, but not "review approved" or "tests passed"). On a database loss they vanish, and the PAN-1920 reconstruction path has nothing to repopulate them from.

Verified evidence of the gap:

- **Read path has no fallback.** `getReviewStatusSync()` (`src/lib/review-status.ts:467-469`) is:
  ```ts
  export function getReviewStatusSync(issueId: string): ReviewStatus | null {
    return getReviewStatusFromDbSync(issueId) ?? null;
  }
  ```
  If the row is gone, the answer is `null` — there is no read-through to the record or GitHub.

- **The write/mirror side already exists** (this is the load-bearing fact). On every status write, `setReviewStatusSync()` mirrors the durable verdict block into the record:
  ```ts
  // src/lib/review-status.ts:342-345
  // PAN-1908: project durable review_status verdicts into the infra repo's
  // per-issue permanent record. Fire-and-forget …
  void updateIssueRecordForIssue(issueId, updated);
  ```
  And `projectPipeline()` (`src/lib/pan-dir/records.ts:129-162`) projects review/test/verification/inspect/merge statuses, notes, PR identity, `reviewedAtCommit`, `lastVerifiedCommit`, `autoMerge`, `deaconIgnored*`, and `blockerReasons` into `PanIssuePipelineRecord`.

So the durable home is **built and continuously populated**; only the inverse arrow (record/GitHub → cache) is missing.

## 3. Decision: where each verdict's durable truth lives (and why)

The issue proposed two candidate durable sources and a "recommended split." We verified the split against the code and **narrowed it**, because the loose phrasing ("read PR-native verdicts from GitHub") does not match what is actually representable on a GitHub PR.

### 3.1 Verified constraint that drives the decision

**GitHub's PR `reviewDecision` does NOT capture Panopticon's internal review verdict.** Panopticon submits a GitHub PR approval (`gh pr review … --approve`) **only at merge time**, not when its internal review transitions to `passed`:

- `approveReviewArtifact()` (the only thing that runs `gh pr review --approve`, `src/lib/forge.ts:293-300`) has exactly one product caller: the forge-approve step inside the merge flow at `src/dashboard/server/routes/workspaces.ts:5770`.
- The internal review verdict (`review_status.review_status = 'passed'`) is set far earlier by the review pipeline and is **never** reflected as a GitHub PR review at that moment.
- Likewise `panopticon/review` commit-status is posted **only** on the `readyForMerge` transition (guarded at `src/lib/review-status.ts:306`), not on intermediate review-passed.

Therefore a freshly-cloned machine reading the live PR **cannot** reconstruct an intermediate "review passed, test pending" verdict from GitHub. The record can — `projectPipeline()` captures it. This kills "read review/test verdicts from GitHub."

Conversely, **merge-state IS PR-owned and IS live-readable.** `refreshMergeStateFromGitHub()` (`src/lib/webhook-handlers.ts:206-260`) already derives the GitHub-native blocker set from `gh pr view --json mergeable,mergeStateStatus,isDraft,statusCheckRollup`. A stored snapshot of mergeability/checks would be *stale and wrong* to trust after a clone; it must be re-derived live.

### 3.2 The decision (state plainly; this is the rule the executor follows)

| Verdict category | Durable source on reconstruction | Why |
| --- | --- | --- |
| **Internal pipeline verdicts** — `review_status`, `test_status`, `verification_status`, `inspect_status`, `merge_status`, their `*_notes`, plus PR identity (`pr_url`, `pr_number`, `pr_head_sha`), `reviewed_at_commit`, `last_verified_commit`, `auto_merge`, `deacon_ignored*` | **The per-issue git record** (already mirrored by PAN-1908). Restore these verbatim. | These are Panopticon-internal judgements with no faithful PR representation (§3.1). The record is their only durable home. |
| **PR-owned merge-state** — `blocker_reasons` (mergeability, required checks, draft, branch protection) | **GitHub PR, read LIVE** via the existing `refreshMergeStateFromGitHub()`. | The PR is the moving source of truth; a stored snapshot would be stale. Re-derive, never restore-as-stored. |
| **`ready_for_merge`** | **Recomputed**, not restored. Runs through the normal write path *after* the record verdicts are restored and *after* blockers are refreshed live, so PAN-1650 + PAN-905 derive it correctly. | It is a derived predicate, not a stored fact. Restoring a stale value would diverge from the gates. |
| **Ephemeral bookkeeping** — all `*_count` / `*_retry_count`, `verification_cycle_count`, `recovery_started_at`, `review_spawned_at`, `conflict_resolution_dispatched_at`, transient `stuck*`, `merge_step` | **None — reset on rebuild.** Documented as acceptable loss. | Loop-prevention, not data. Worst case: an agent gets a fresh set of retry attempts. (`reference/state-model.mdx` §"Retry counters … safe to lose".) |

`review_status` thus becomes a **cache over the union** of (record-durable verdicts) ∪ (live PR-owned merge-state), with ephemeral counters reset. This is exactly the State-Model contract.

### 3.3 Two classification calls the executor must NOT get wrong

- **`deacon_ignored*` and `auto_merge` are DURABLE, not ephemeral.** They are human decisions, and `projectPipeline()` already mirrors them into the record (`records.ts:157-159, 156`). Restore them from the record. Do **not** put them on the ephemeral allowlist even though they "feel" like bookkeeping.
- **`status_history` is accepted as lost.** It is a separate table (`schema.ts:272-287`), is **not** projected into the record, and is an append-only UI log, not a verdict. Reconstruction does **not** restore it; the restored row starts with an empty history. This is a deliberate decision, documented here so it is not a silent gap.

## 5. Requirements

### Functional

- **FR-1** A library function `restoreReviewStatusFromRecords(options)` reconstructs `review_status` rows from the per-issue git records for the requested issue(s), restoring the **durable subset** (§4) verbatim and leaving ephemeral columns at their schema defaults.
- **FR-2** During restore, for each issue with a `pr_url`/`pr_number`, the **PR-owned merge-state** (`blocker_reasons`) is re-derived **live** from GitHub via the existing `refreshMergeStateFromGitHub()` — never restored from the record snapshot.
- **FR-3** `ready_for_merge` is **recomputed** by the normal write path after durable verdicts are restored and blockers refreshed; it is never written directly from a stored value.
- **FR-4** A CLI command `pan admin db restore-verdicts` runs FR-1–FR-3, with `--issue-id <id>`, `--dry-run`, and `--verbose` flags, mirroring `pan admin db backfill-records`.
- **FR-5** `restoreReviewStatusFromRecords()` is exported so the PAN-1920 umbrella reconstruction (`pan admin db rebuild`) can call it as a step. The seam is a single exported function; PAN-1920 wires the call (out of scope here).
- **FR-6** When a record is missing for an in-flight issue (e.g. PAN-1919 not yet activated, or a never-reviewed issue), restore **skips** that issue and reports it; it does not error or fabricate a row.

### Non-functional

- **NFR-1** Restore is **idempotent**: running it twice yields the same `review_status` rows (modulo `updated_at`/live blocker timing). Re-running over an already-correct cache is a no-op for the durable subset.
- **NFR-2** No new SQLite columns, no schema migration. The durable home (the record) and the cache table both already exist.
- **NFR-3** No `execSync` anywhere reachable from the dashboard server; the GitHub read uses the existing async `refreshMergeStateFromGitHub()` (which already uses `execFileAsync`). The CLI command may run async freely.
- **NFR-4** The no-loss audit is a **test**, not a comment — it enumerates every `review_status` column and fails closed on any unclassified column.

## 7. Intersecting repo rules (restated — do not assume recall)

- **Refactor with an explicit no-loss audit.** Any replacement of an existing surface needs an audit gate that blocks until every old item is accounted for. Here: Work Item 4's column-enumeration test. "Is anything lost?" is answered by that test, not by reasoning.
- **No `execSync` in dashboard-reachable code.** The GitHub read reuses `refreshMergeStateFromGitHub()`, which already uses `execFileAsync`. Do not add `execSync`. (`src/lib/webhook-handlers.ts:208-210`.)
- **Surgical changes (Karpathy #3).** Touch only what you must: one new lib file, one new test file, one new CLI command, one `export` keyword added to a shared collector, optionally one tiny `parsePrUrl` helper. Do **not** "improve" `records-backfill.ts`, the mirror write path, or `getReviewStatusSync`.
- **Agent state planes (PAN-1908).** `review_status` is the **runtime plane** (rebuildable). The per-issue record is the **permanent plane**. This PRD wires the rebuild arrow from permanent → runtime. Enumerate agents/issues from the DB + records, not from `state.json`. (`docs/AGENT-STATE-PLANES.md`.)
- **State-model contract.** The DB is a cache; ephemeral counters are "safe to lose." Restoring must not try to resurrect them. (`reference/state-model.mdx`.)
- **Commit often on `main`.** This PRD authoring touches no source; the executor working this issue commits each work item as it builds.
