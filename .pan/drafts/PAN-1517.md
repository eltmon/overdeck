# PAN-1517 — Replace Swarm Runtime with Work-Agent-Spawns-Children

## Problem

Panopticon's swarm runtime is a polling daemon embedded inside an HTTP route module. It dispatches parallel "slot" work agents through a DAG defined in the vBRIEF. Each slot lives in its own git worktree, opens its own PR against a transient feature branch, and runs through the full review/test/ship/merge pipeline.

The architecture has accumulated cost without earning value:

- **Module-level state in a route file.** `src/dashboard/server/routes/swarm.ts` carries an `activeSwarmIssueIds: Set<string>`, a `setTimeout` handle, a re-entrancy lock, and a started-flag as module globals. An HTTP route file doubles as a background daemon.

- **No failure ceiling.** PAN-1203's `swarmRuntime` currently records `autoAdvanceFailureCount: 1130` — eleven hundred consecutive auto-advance failures with no exit. The poll loop has retried `"Wave 0 does not exist for PAN-1203"` 1,130 times since the slot merged 16 hours ago. Nothing disables auto-advance when failures pile up. The dashboard log emits this stall message every ~30 seconds, every poll, forever.

- **No issue-state filter in discovery.** `discoverActiveSwarmIssueIds()` registers every `workspaces/feature-*` directory as an active swarm without checking whether the issue is closed on GitHub. Merged issues with leftover workspaces re-register on every dashboard restart.

- **Three-writer persistence dance.** Swarm state lives in `<workspace>/.pan/continues/<issue>.vbrief.json` under `swarmRuntime`. A legacy sidecar at `~/.panopticon/swarms/*.json` is read for one-time migration. A retry directory at `~/.panopticon/swarms/pending-slot-merges/` holds failed loopback callbacks. `saveSwarmState`, `persistSwarmRuntime`, and `onSlotMergeComplete` all converge through different paths.

- **Per-slot pipeline ceremony.** A 9-wave swarm with one slot per wave runs the review/test/ship/merge ceremony 10 times: nine slot-PRs into the parent feature branch, plus one final merge to main. For the workloads that actually get swarmed (PAN-1203 = "ship a docs RAG index"), most slots are sequential dependencies — the parallelism never materializes, but the orchestration tax does.

- **Per-slot worktrees.** Each slot gets its own git worktree (hundreds of MB once `node_modules` is built), its own devcontainer stack, its own traefik routes. Disk and orchestration cost scale linearly with slot count, even when slots are tiny config changes.

- **Migration scar tissue.** PAN-946 (workspace-spec isolation), PAN-977 (continue-vBRIEF authority — 14+ rounds of fix comments in `swarm.ts`), PAN-1124 (single-spec-on-main). Each invariant migration left behind compat code paths still in the file.

The root issue: Panopticon's swarm tried to build a workflow engine on top of the agent runtime. The workflow engine became a stuck polling loop nobody asked for. Most agent products land on "one agent per session, parallelism is the harness's job" because that's where the complexity/value curve crosses.

## Alignment with canonical vBRIEF

vBRIEF v0.5 (`github.com/deftai/vBRIEF`) defines a plan as `items[]` + `edges[]` forming a DAG, with `narratives` and a small set of statuses. The spec is **deliberately silent on execution runtime** — how a plan is run (sequentially, concurrently, in containers, with retries) is left to the executor.

Panopticon's swarm extension layered execution semantics onto the plan artifact: `waves[]`, `slots[]`, `swarmRuntime` (currentWave, autoAdvance, autoAdvanceFailureCount). None of these went upstream. The only Panopticon extension proposal filed at `deftai/vBRIEF#1` adds `difficulty`, `issueRef`, and `kind` to `PlanItem` — item-level metadata, not runtime fields.

This proposal **restores the canonical separation**:

- The vBRIEF stays a plan: `items[]` + `edges[]`. Canonical.
- The work agent stays an executor: reads the DAG, decides how to run it, fans out subagents in-context when the DAG marks items as mutually independent. Runtime, not plan.

The Kahn's-algorithm wave grouping at `/home/eltmon/Projects/panopticon-cli/src/lib/vbrief/dag.ts:53` (`groupItemsByWave`) is kept as a **read-only informational helper** the work agent can call to see the topological layering. It no longer drives dispatch.

## Proposed architecture: work-agent-spawns-children

A "swarm" stops being a Panopticon construct. Parallelism becomes an in-context concern owned by the work agent.

```
Today:
  pan plan  → vBRIEF with waves[] + slots[]
  pan swarm → dashboard's swarm runtime → fans out slot agents in worktrees
            → each slot opens slot-PR → review/test/ship/merge per slot
            → auto-advance loop polls for next wave → repeat

Proposed:
  pan plan  → vBRIEF with a flat bead list (no waves[], no slots[])
  pan start → one work agent in one workspace
            → work agent calls harness subagent primitive (Claude Code's
              `Agent` tool, etc.) for genuinely parallel chunks within
              its own context
            → integrates subagent outputs → commits → next bead
            → opens ONE PR for the whole issue → standard review pipeline
```

The work agent is the durable, supervised, dashboard-visible entity (its tmux session, its `state.json`, its pipeline plumbing). When it encounters genuinely parallel work, it uses the harness's existing subagent primitive to fan out. Subagents share the parent's filesystem, return results as messages, and the parent decides what to commit. Subagents are ephemeral — no `state.json`, no tracker entry, no PR, no review pipeline. They are "throwaway parallel chunks" the same way `Explore` subagents are "throwaway investigation" today.

## Why this works

- **No swarm runtime.** Parallelism is the harness's job, not Panopticon's. The DAG is implicit in the work agent's planning.
- **No worktrees per slot.** One workspace per issue. Subagents share the filesystem.
- **No slot PRs.** One PR per issue. One review pipeline run.
- **No auto-advance loop.** Advancement is the agent's sequential progression through beads. There is nothing to poll.
- **No persistence dance.** No `swarmRuntime` block in `continue.json`. No legacy sidecar. No retry directory.
- **No discovery scan.** No `activeSwarmIssueIds` registry.
- **Failure handling is local.** A subagent fails → the parent retries it or surfaces the failure. Same as any tool call.
- **Cost shaping at the right layer.** The work agent can choose "spawn 4 cheap Haiku subagents" or "do this serially" based on the actual task, not based on a swarm config 9 layers up.

## Phases

**Phase 1 — Inventory the deletion surface.**
Map every file that touches `swarmRuntime`, every endpoint under `/api/swarm/`, every dashboard component that renders swarm state, every CLI flag that targets swarm config. Output: a checklist; no code changes.

**Phase 2 — Clarify planner edge semantics (no behavior change).**
The planner today already produces canonical vBRIEF — `items[]` and `edges[]` only, no waves or slots. See `/home/eltmon/Projects/panopticon-cli/roles/plan.md:44` ("items, acceptance criteria, and dependency edges") and `/home/eltmon/Projects/panopticon-cli/sync-sources/agents/pan-planning-agent.md` (planning agent template — zero mentions of swarm/wave/slot). Wave materialization happens downstream at `/home/eltmon/Projects/panopticon-cli/src/lib/vbrief/dag.ts:53` (`groupItemsByWave`), called by the swarm runtime.

The only planner update is **edge hygiene**, motivated by the new executor behavior: the work agent will use absence of an edge between two items as a signal that they're parallelizable via subagents. So the planner needs to be deliberate about when to add edges. Add a section "Edge semantics for the executor" to `/home/eltmon/Projects/panopticon-cli/roles/plan.md`:

> Add a `blocks` / `blockedBy` edge between items A and B only when there is a real dependency — output → input, shared file mutation, ordering requirement. Do not add edges for narrative flow or readability. The work agent treats absence of an edge between two ready items as "you may run these in any order, including in parallel via subagents." A spurious edge silently forces serialization that was never intended.

No schema change. No new fields. The schema is already correct.

**Phase 3 — Teach the work role to fan out.**
This is where the bulk of the prompt change lives. `/home/eltmon/Projects/panopticon-cli/roles/work.md` today is 101 lines describing a strictly sequential per-bead workflow (line 51, "Per-Bead Workflow"). Add a new section after that, titled "Parallel work via subagents":

- **When to fan out:** the DAG (`.pan/spec.vbrief.json`'s `edges[]`) shows 2+ ready beads in the same dependency layer; the items operate on non-overlapping files; each item is substantial enough to amortize subagent startup.
- **When NOT to fan out:** items share files; one item's commit must be visible to the next; items are tiny; you're uncertain about ordering (default serial).
- **How to fan out:** call the harness's subagent primitive (Claude Code's `Agent` tool) once per parallel item with a focused prompt; subagents share your filesystem and read/write directly in the workspace; collect outputs as text replies; resolve conflicts if any subagent's changes collide.
- **Commit and bead close:** one commit per bead (existing rule on line 67 stays). After fan-out, the parent stages each subagent's contribution into its own bead-scoped commit and closes the bead via `bd close`.
- **Failure handling:** retry a failed subagent once with a clarified prompt; if still wrong, fall back to doing it serially yourself; record the fan-out decision and any failure in `.pan/continue.json` `sessionHistory`.
- **Cost shaping:** the parent picks the model for each fan-out — cheap Haiku subagents for mechanical work, the parent's own model for nuanced work.

Concrete trigger examples for the prompt: "regenerate 4 independent test fixtures", "audit these 6 files for the same anti-pattern", "summarize each of these N test failures", "update import paths across these non-overlapping modules."

The work agent retains full discretion. The DAG is information, not directive. Cover that explicitly so the agent doesn't over-fan-out.

**Phase 4 — Deprecate `pan swarm`.**
The CLI verb prints a deprecation message and delegates to `pan start <id>`. The dashboard's swarm UI hides behind a feature flag, still functional for any in-flight swarm.

**Phase 5 — Drain in-flight swarms.**
Manually finalize or close out any swarms still running. Confirm `~/.panopticon/agents/agent-pan-*-slot-*` directories are gone and no `swarmRuntime` blocks remain in any workspace `continue.json`.

**Phase 6 — Delete the runtime.**
Remove `src/dashboard/server/routes/swarm.ts` polling loop, registry, discovery, persistence. Remove `/api/swarm/*` endpoints. Remove `swarmRuntime` schema from vBRIEF type definitions. Remove `~/.panopticon/swarms/` directory handling. Remove swarm UI from the dashboard. Remove the `pan swarm` CLI verb. Remove the `pan-swarm` skill.

**Phase 7 — Cleanup tests and docs.**
Remove swarm-related tests (`src/dashboard/server/routes/__tests__/swarm.test.ts`, `src/lib/cloister/__tests__/deacon-swarm-slot-merge.test.ts`, `src/lib/vbrief/__tests__/continue-state-swarm.test.ts`). Update docs that reference swarms. Update `CLAUDE.md`, skill descriptions, and skill index.

## Non-goals

- **Replacing parallelism with overlayfs / EdenFS / containers-per-subagent.** Subagents share the parent's filesystem; integration is the parent's job. If a future workload needs filesystem isolation between parallel subagents, that's a follow-up — overlayfs is a ~50-line shell script and a small workspace-setup tweak.
- **Preserving per-slot reviewability.** Reviewers (specialist agents and the optional human approver) see the final integrated diff. Per-commit review can still run on the final stack if desired, just without per-slot merge ceremonies.
- **Backwards compatibility for in-flight swarms beyond Phase 5.** One-time drain. No long-lived compat shim.

## Risks

1. **Workloads that genuinely benefited from parallel slots lose wall-clock parallelism unless the work agent fans out via subagents.** Mitigation: the work agent prompt in Phase 3 explicitly recommends fan-out for independent chunks. Validate on 1–2 real issues before Phase 6 deletes the runtime.

2. **Subagent outputs are text-only.** Subagents that need to coordinate via shared file state must be sequenced by the parent. Most tasks fit this model. Tasks that do not will surface during Phase 3 prompt testing.

3. **Reverting if this turns out wrong.** The swarm code can come back from git history, but the workflow change is sticky. Phase 4's feature flag is the off-ramp during the migration window.

## Success criteria

- `src/dashboard/server/routes/swarm.ts` deleted, or reduced to ≤100 lines of legacy type definitions.
- No `setTimeout`-driven polling in any route file.
- No `[swarm]` log lines in `dashboard.log` after Phase 6 lands.
- `pan plan` produces vBRIEFs without `waves[]` / `slots[]` for new issues.
- A work agent on a representative issue (one that today would be swarmed) runs end-to-end with at least one in-context subagent fan-out, integrated and committed cleanly.
- `~/.panopticon/swarms/` directory removed from project state.
- Disk footprint per "would-have-been-swarmed" issue drops to a single workspace.
