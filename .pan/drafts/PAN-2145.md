## 0. Glossary (define before use)

- **Two-door tenet** — every domain has exactly one read path (resolver) and one
  write path; routes reach state only through them (`docs/API-SURFACE.md`,
  PAN-1936). The conversations door already exists:
  `src/lib/overdeck/conversations.ts` (1,526 lines, baselined) — Effect
  resolvers/writers (`ConversationsResolverLive`, `ConversationWriterLive`,
  `TranscriptsResolverLive`) plus the sync CRUD surface (`listConversations`,
  `getConversationByName`, `setFavorite`, `importLegacyConversation`, …). It owns
  PERSISTENCE. It does NOT own session runtime, delivery, forks, or diffs — those
  live inline in the route file today and move to NEW sibling door modules.
- **Thin adapter** — a route handler that only parses params/body, calls one door
  function, and maps result/error to `jsonResponse`. Rule of thumb: a handler over
  ~40 lines still contains domain logic.
- **Door module** — a file under `src/lib/overdeck/`, one domain concern each,
  <1,000 lines (ceiling applies to new files).
- **Route registration idiom** — `const <name>Route = HttpRouter.add('METHOD',
  '/api/…', …)` (34 sites; grep anchor: `const getConversationsRoute = HttpRouter.add(`),
  merged in `export const conversationsRouteLayer = Layer.mergeAll(` near EOF.
- **Fork pipeline** — the summary-fork/handoff machinery inline in the route file
  (grep anchors: `export async function runForkPipeline`, `injectForkSummary`,
  `recoverStuckForks`, `registerInFlightForkPipeline`). Collaborates with
  `src/lib/conversations/summary-fork.ts`.
- **conversation-service** — `src/dashboard/server/services/conversation-service.ts`
  (1,609 lines at the sha; PAN-2156 decomposes it into
  `services/conversation/{parser,types,watch,context-usage,compact-boundary,…}.ts`
  keeping `conversation-service.ts` as a barrel, so this file's imports of it resolve
  unchanged either way). It owns TRANSCRIPT PARSING — not touched by this PRD.
- **No-loss audit** — §2: every route, export, and behavior in today's file has a
  named home in the end state; WI-8's test locks the route set.
- **Repoint** — update an importer to the moved symbol's new module path in the same
  commit as the move. No shims.
- **TENET-10** — conversations live-control is pipeline machinery (agent sessions,
  delivery, forks): FULL `npm test` green before merge; supervised, sequenced,
  never batched with other pipeline refactors.

## 1. Problem (verified evidence)

`src/dashboard/server/routes/conversations.ts` is 5,316 lines (baselined at exactly
`5316` in `scripts/file-size-baseline.txt`; the issue said 4,898 — it grew +418 while

message transforms, fork pipeline, archived-list handling, control acks), then 34
`HttpRouter.add` registrations, then the layer merge. It exports ~45 symbols besides
the layer, and — the clearest smell — **the server BOOT path imports from a route
file**: `src/dashboard/server/main.ts` does
`import { recoverStuckForks, waitForInFlightForkPipelines } from './routes/conversations.js'`
(grep anchor). Every conversation-lifecycle change (PAN-2225's RPC poison, the
supervisor delivery work, fork recovery) lands inside this one file.

## 3. Locked design decisions

- **D1 — Route file stays; handlers thin; no barrel directory.** All 34
  registrations and `conversationsRouteLayer` remain in `routes/conversations.ts`.
  Fat goes to lib. (Supersedes the issue body's barrel ask — see the override note.)
- **D2 — Five new door modules under `src/lib/overdeck/`, one concern each, <1,000
  lines:** `conversation-runtime.ts` (spawn/stop/resume/restart/delete/switch-model),
  `conversation-delivery.ts` (message handling, transforms, delivery method,
  control channel + acks, compact/abort/thinking-level/plan-action/images),
  `conversation-forks.ts` (fork pipeline + handoff docs + recovery),
  `conversation-archive.ts` (archive/unarchive/archived-list),
  `conversation-diffs.ts` + `conversation-reads.ts` (reads, patch/retitle/about).
  These are behavior-preserving RELOCATIONS — they keep calling
  `src/lib/overdeck/conversations.ts` CRUD, `deliverAgentMessage`,
  conversation-service, and `summary-fork.ts` exactly as the handlers do today. No
  rewrite onto the Effect resolver services in this issue (PAN-1936's work).
- **D3 — The existing conversations door is NOT grown.** It is baselined at 1,526 —
  a baselined file may not exceed its recorded count, and persistence vs runtime are
  different concerns anyway. New modules extend the door SURFACE as siblings; if a
  moved function belongs semantically inside `conversations.ts`, it still goes to
  the sibling module with a doc comment pointing at the door (no audited baseline
  bump for this issue).
- **D4 — No re-export shims; repoint instead** (§2.2). ONE production importer
  exists (`main.ts`); everything else is tests. Shims would keep the route file a
  fake door.
- **D5 — One seam per commit, safest first** (WI order below); after each seam:
  `npm run typecheck && npm run lint && npm run build` + the seam's named tests;
  FULL `npm test` before merge (TENET-10). Never batch with another pipeline
  refactor (Phase 3 rule).
- **D6 — File-size baseline handling.** `routes/conversations.ts` shrinks massively:
  lower its `5316` baseline entry in the same PR (PAN-2227's check-mode forces this
  if landed). Every new door module <1,000 lines — if a cluster would exceed the
  ceiling (delivery is the risk: `handleConversationMessage` + transforms + acks),
  split it (`conversation-message.ts` off `conversation-delivery.ts`) and record the
  split in the PR description. NO new baseline entries.
- **D7 — Live-control safety.** These routes drive live agent/conversation sessions.
  Tests use the existing injection seams (`__setForkPipelineRuntimeOverridesForTest`
  moves with its cluster and keeps working); NEVER exercise moved code against live
  `overdeck`-socket sessions or the running dashboard. Boot verification happens on
  a throwaway server instance, never the live one (no hot-patching a boot-path
  change — `main.ts` is touched here).
