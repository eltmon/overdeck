## 0. Glossary

- **`merge-ops.ts`** — `src/dashboard/server/routes/workspaces/merge-ops.ts`
  (1,925 lines): 7 routes (sync-main, merge, forge-approve, forge-merge, approve,
  merge-queue GET, internal pipeline notify), the **merge engine** (`triggerMerge`,
  ~784 lines — verified span from `export async function triggerMerge` to the merge
  route registration), and PR/merge support helpers.
- **Merge engine** — `triggerMerge(issueId)`: rebase, post-rebase quality gate
  (the authoritative one — `src/lib/review-status.ts` comments defer to it), PR
  ensure/merge, queueing (`dequeueNextMerge`), and post-merge handoff. Pipeline runtime,
  not transport — its new home is `src/lib/cloister/`.
- **Forge routes** — `forge-approve` / `forge-merge`: operator "forge" variants of the
  approve/merge flow.
- **Barrel + thin routes hybrid** — the end state: `merge-ops.ts` keeps all 7
  registrations (thin handlers) AND re-exports the moved symbols so its five verified
  importers stay unchanged. This differs from Batch 2 (full repoint) because two
  production callers import `triggerMerge` DYNAMICALLY by this path and the queue wants
  this file to stop being fat, not to disappear.
- **TENET-10 / pure move / repoint** — as PAN-2149/2232/2233.

## 3. Locked design decisions

- **D1 — Three new lib modules, one thinned route file:**
  `src/lib/cloister/merge-trigger.ts` (engine, ≈850 — verify it lands <1,000 with
  imports; if not, extract `dequeueNextMerge` + queue plumbing to
  `merge-trigger-queue.ts` and report), `src/lib/cloister/merge-route-support.ts`
  (≈350), `src/lib/cloister/approve-ops.ts` (approve/forge/notify bodies, ≈650);
  `merge-ops.ts` ends ≈300 (7 thin handlers + layer + re-exports), baseline entry
  LOWERED.
- **D2 — Home is `src/lib/cloister/`,** beside `merge-agent.ts` (its machinery sibling
  — `postMergeLifecycle` consumers, quality gates). NOT `src/lib/overdeck/merge.ts`,
  which is baselined at 1,001 lines and may not grow.
- **D3 — Barrel compatibility:** `merge-ops.ts` re-exports every moved public symbol so
  all §2.2 consumers (including the two dynamic imports) resolve unchanged. Zero
  importer edits; the only test edits are the §1.4 introspection repoint and the §1.5
  mock-list follow.
- **D4 — `triggerMerge` moves VERBATIM as one block.** It is the authoritative
  post-rebase quality gate (per `src/lib/review-status.ts` comments) and contains merge
  queueing; no internal restructuring, no step reordering, no "simplification" — the
  784 lines relocate as-is. If PAN-2233 landed first, its `postMergeLifecycle` import
  path may already be `merge-agent-post-merge.js` via the barrel — imports move
  untouched either way.
- **D5 — Handler-body extraction shape:** approve/forge/notify bodies become exported
  functions in `approve-ops.ts` taking parsed params (issueId, body fields) and
  returning the response payload; the route shell keeps request parsing + origin/
  validation + `jsonResponse` mapping (same thin-adapter rule as PAN-2148 §0).
- **D6 — Cycle expectations:** today `merge-ops.ts > workspaces.ts` cycles exist
  (madge-verified in PAN-2230's inventory). Moving the engine to lib CHANGES cycle
  lines — if PAN-2230 is live, run `--update`; additions are audited by this issue's
  ref. Runtime ESM cycle in `npm run build` ⇒ stop and report (as PAN-2149 D6).
