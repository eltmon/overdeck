# PAN-1884 — Migrate panopticon agent operational rules from memory into the scope:dev rule/role layer

**Status:** draft (awaiting human + AI review before execution)
**Scope:** immediate. Executed together with PAN-1883 and PAN-1877.
**Audit basis:** [`docs/STATE-STORAGE-AUDIT.md`](../../docs/STATE-STORAGE-AUDIT.md)

> **Executor note (read first).** This migration must **commit immediately on execution** — once the
> rules are in the rule layer, the memory copies stop being read and go stale, so an uncommitted

---

## Glossary

- **Memory** — `~/.claude/projects/<proj>/memory/*.md`. Recalled as advisory background into **one**
  conversation at a time. Does NOT govern work/review/flywheel agents.
- **Rule layer** — content that ships into harness context files every agent reads:
  - **Bundled rule** — `sync-sources/rules/<name>.md` with `scope: dev` (panopticon-dev checkouts
    only) or `scope: universal` (everywhere). Rendered by `pan sync` into `~/.claude/CLAUDE.md`
    managed region, `pi-global.md`, `codex-global.md`.
  - **Role file** — `roles/flywheel.md` etc.; governs that specific spawned role.
- **scope:dev** — folds into the managed region only on a panopticon-cli source checkout; never ships
  to projects that merely use Panopticon. **All new rules in this issue are `scope: dev`.**

## Requirements

- **FR-1** — Each qualifying memory (panopticon agent operational rule) is materialized as a
  `scope: dev` bundled rule under `sync-sources/rules/`, OR added to `roles/flywheel.md` if it is
  flywheel-orchestrator-specific.
- **FR-2** — Memories already duplicated in the rule layer are slimmed to a one-line pointer.
- **FR-3** — The stale `planning_artifacts` memory is deleted (or corrected) — it cites deleted paths.
- **FR-4** — After each batch, `pan sync` renders the rules into the harness context files, verified,
  and the change is committed immediately.
- **NFR-1** — Rules are tight (rules cost context in every dev session); condense the verbose memory
  prose to the imperative rule + one-line why + how. Match the voice of existing `sync-sources/rules/`.
- **NFR-2** — Scope guard: migrate ONLY panopticon-dev rules clearly designed for/by agents. Do NOT
  migrate non-panopticon or purely-personal/interaction memories (see "Explicitly NOT migrated").

## Work items

### WI-1 — Create the seven `scope: dev` bundled rules

For each, create `sync-sources/rules/<name>.md` with `---\nscope: dev\n---` frontmatter and a tight
body in the existing imperative voice (look at two neighbors in `sync-sources/rules/` first, per the
rule-authoring rule). Condensed bodies to write:

- **no-oneshot-agents.md** — "Work agents run as live interactive sessions, never one-shot/headless.
  Codex work agents MUST use the TUI path (`codex`), never `codex exec` (one-shot: runs one turn,
  exits, lifecycle marks it orphaned). One-shot was rejected long ago — it cost hundreds of hours.
  Before spawning a codex work agent, confirm it's the TUI path; surface it if not."
- **additive-refactor-no-loss.md** — "Refactor plans for an existing surface (CLI verbs, UI actions,
  API endpoints) must be additive/superset, never silent replacement. Enumerate everything the old
  surface exposes; verify each has a home in the new one. Any deletion/replacement gets a no-loss
  audit gate (a test) that blocks until every old item is accounted for. 'Is anything lost?' is
  answered by the audit, not by reasoning."
- **facts-not-guessing.md** — "On system-state questions (permissions, flags reaching a harness, why
  an agent behaves a way), verify and state facts — don't theorize. Ground claims in `git grep`,
  `strings`, `/proc/<pid>/cmdline`, the resolved config value. Distinguish code vs resolved config vs
  already-running process — a live process keeps its launch-time flags after a config change. Get the
  fact before answering."
- **explain-fully-no-jargon.md** — "Operator-facing messages must stand alone. Define every pipeline
  term inline on first use (ready=1, verification gate, merge queue, advancing slot). Spell out
  consequences in full sentences. Never compress operator-facing output to save tokens, even at high
  context usage — durable docs may be terse, operator messages may not."
- **self-verify-rendering.md** — "Verify dashboard/terminal rendering yourself; don't ask the
  operator to eyeball it. Playwright screenshots are readable (`browser_take_screenshot` writes a PNG
  to the repo root; then `Read` it). For terminal ANSI colors, `tmux -L panopticon capture-pane -t
  <s> -e -p` emits raw escapes. Preview in a throwaway `panopticon`-socket session rather than
  attaching to (and resizing) a live one."
- **no-inspection-policy.md** — "When the operator sets a no-inspection policy on an issue,
  `requiresInspection` must be false on every bead AND no `pan inspect` path may run — including
  PostToolUse hooks or other auto-triggers. If a bead would need inspection to pass, mark it blocked,
  don't inspect. Disable the auto-trigger path; don't rely on restraint."
- **handoff-not-pipeline-agents.md** — "When asked to 'spawn agents' to work on issues from within a
  conversation, use `pan handoff` (creates handoff *conversations* — interactive, human-supervised),
  NOT `pan start`/`pan swarm`/`pan plan` (pipeline work agents that auto-advance through review).
  Isolated git worktrees as `--cwd` are fine. Tell handoff agents NOT to run `pan done`."

### WI-2 — Add three flywheel-orchestrator rules to `roles/flywheel.md`

Add under an appropriate existing section (the tick loop / merge area near `:88`, or a new
"### Red-main and merge discipline" subsection). Tight versions:

- **Red main empties the merge gate** — "Each tick verify main CI *conclusion*, not just HEAD sha:
  `gh run list --branch main --workflow CI --limit 1 --json conclusion`. A green `Main HEAD: <sha>`
  line does NOT mean CI is green. When main is RED, every feature PR inherits the failing `test`
  check, so nothing reaches `readyForMerge` and the merge gate looks empty for a non-obvious reason.
  Red main is P0 — fix it first; it unblocks the whole gate at once."
- **Never admin-merge while main is red** — "`gh pr merge --admin` bypasses the PR's required `test`
  check. Only do it when main is already GREEN. If main is red, a PR's red `test` may be its OWN new
  failure, indistinguishable from the inherited stale-red — fix main green first, then a red PR check
  unambiguously means the PR's own break. Reverting a squash-merge that broke main is clean
  (`git revert <sha>`)."
- **Heartbeat cadence** — "Schedule the next tick with `ScheduleWakeup(delaySeconds: ~1000)`, NOT
  1200 — runtime drift pushes 1200 to ~1251s, past the 20-min watchdog threshold that flags the
  orchestrator as stuck. Emit a status every tick even when state is identical; never widen the
  interval to 1800/3600s."

### WI-3 — Slim the four already-promoted memories + delete the stale one

**Commit immediately** (`sync-sources/rules/` is package content — an uncommitted rule exists only on
