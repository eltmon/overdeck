# PRD: Merge Workflow Simplification

**Issue:** [PAN-1531](https://github.com/eltmon/panopticon-cli/issues/1531)

*Author: Edward Becker · Drafted: 2026-05-26*

---

## Vision

Replace Overdeck's six-actor merge choreography — work-agent → review specialists → ship-role LLM agent → human merge button → `postMergeLifecycle` → GitHub squash → three-oracle verifier — with a two-actor flow: the dashboard server and GitHub. Remove the ship-role entirely, take the rebase decision off the LLM, collapse three stash kinds to one, and let `pan done` and Start-Agent refuse dirty worktrees instead of papering over them.

The end state is a merge workflow a new contributor can explain in two sentences after reading one diagram.

## Motivation

The current pipeline has accreted over a year of incremental fixes, and the cumulative result is a workflow nobody can describe without reading code across six files. Five concrete problems compound:

### 1. An LLM is performing rebases

`roles/ship.md` and `src/lib/cloister/prompts/ship.md` instruct an LLM agent to:

> Rebase `${options.featureBranch}` onto `origin/${options.targetBranch}` using non-interactive rebase only. Resolve rebase conflicts if they are narrow and source-level. If conflicts are broad, abort and report SHIP BLOCKED.

Rebase conflict resolution is deterministic three-way merging — "prefer this side or that side," "this hunk applies cleanly, that one doesn't." LLMs are creative. Handing this decision to the component least suited to it is the single highest-leverage risk in the pipeline. The ship-role's only tools are *rebase* and *abort* — agent prompts forbid `git stash`, so there is no transactional safety net inside the rebase.

### 2. "Merged" has three definitions

[`merge-agent.ts:177`](/home/eltmon/Projects/panopticon-cli/src/lib/cloister/merge-agent.ts) `verifyMergedBeforeLifecycle` checks, in order:

1. Branch tip is an ancestor of `origin/main` (regular merge case)
2. GitHub PR API reports `mergedAt` or `mergeCommit` (squash and rebase merges)
3. `git diff origin/main...<ref>` returns empty (local-diff fallback)

We need three oracles because we don't trust any one of them. The cost of that hedging is that the meaning of "merged" is muddy — when bugs land, every diagnosis starts with "which oracle answered this time?"

### 3. `pre-spawn` stashing silently hides user work

When the user clicks Start Agent on a dirty worktree, [`routes/agents.ts:3000`](/home/eltmon/Projects/panopticon-cli/src/dashboard/server/routes/agents.ts) silently runs `git stash push -u -m "pre-spawn:PAN-XXX:..."` and proceeds. The dirty changes move into `refs/stash` with no UI feedback. The existence of the separate `salvageable:*` stash kind — the only kind the janitor will never auto-drop — is the system admitting it can lose user work this way.

### 4. `review-temp` stashing hides agent scratchwork

When an agent calls `pan done` with uncommitted edits, [`review-agent.ts:84`](/home/eltmon/Projects/panopticon-cli/src/lib/cloister/review-agent.ts) `ensureReviewTempStash` stashes the dirt, runs review against the committed-only diff, then pops on success. When review re-runs, the pattern repeats and orphans accumulate.

Evidence: this repo's `refs/stash` contains `review-temp:PAN-1418:1..10` — ten orphans from one issue's review cycles. The function's own header comment admits this is exactly the failure mode it was added to fix, and the fix didn't fix it.

### 5. Dead code defending against a holstered footgun

[`merge-agent.ts:32`](/home/eltmon/Projects/panopticon-cli/src/lib/cloister/merge-agent.ts) imports `buildStashMessage`, `createNamedStash`, `dropStash`, `listStashes` from the stashes module. Of these, only `dropStash` and `listStashes` are actually called — by `dropLingeringPreMergeStashes`, which cleans up `pre-merge:` stashes that nothing currently creates. The orchestrator-side stash/pop pattern was removed when the rebase moved into the workspace worktree, but the cleanup function defending against it was left in place. Anyone reading the code thinks the orchestrator still wraps merge in stash/pop. It doesn't.

### Triggering incident

On 2026-05-26 the dashboard activity feed flooded with 1,716 "Non-canonical stash in PAN-863" entries on startup. Root cause analysis revealed:

- Git worktrees share `refs/stash`. 41 worktrees × ~42 non-canonical stashes ≈ the 1,716 count.
- The startup scanner emitted one activity entry per finding (already patched in this conversation).
- The 42 non-canonical stashes are roughly half hand-typed debugging artifacts on `main`, half stashes from retired kinds (`pre-ship`, `pre-sync`, `pre-rebase`) and slot-suffix-style issue IDs that no longer parse.

The flood was a symptom. The underlying problem is that the stash subsystem exists primarily to compensate for choices the merge workflow shouldn't have made in the first place.

## Design Goals

- One person reading one diagram understands the merge flow in 60 seconds
- No LLM in the rebase decision path
- One authoritative answer to "is this PR merged"
- No silent state movement of user or agent work — every commit/discard/stash is an explicit choice
- The stash subsystem shrinks to just the kind that needs human judgment (`salvageable:*`)
- Rules visible to every agent that might touch Overdeck source, not just agents working inside this repo

## Non-Goals

- Changing GitHub's squash-merge behavior. Squash stays.
- Replacing the dashboard Merge button with auto-merge. Humans still pull the trigger.
- Pre-emptively cleaning up the 35 existing non-canonical stashes. That's a one-time `pan stash audit` follow-up, not in scope here.
- Touching the swarm slot-merge HTTP loopback. PAN-1517 is removing the swarm runtime entirely, which obsoletes the question.
- Changing the review specialist or test agent flows. Only the *handoff to merge* is in scope.

## Key Decisions

### Decision 1 — Remove the ship-role agent; rebase server-side

`roles/ship.md`, `src/lib/cloister/prompts/ship.md`, and all ship-role spawn machinery are deleted. The rebase moves into a small typed module at `src/lib/git/rebase.ts` (~40 lines): fetch `origin/main`, attempt `git rebase`, on success push the rebased branch, on conflict surface the conflict to the dashboard with a "Resume in workspace" action that drops the user (or the work-agent) back into the worktree to resolve manually.

The dashboard server already does heavier execAsync work (`gh pr merge`, Docker cleanup, GitHub API enrichment). One more `execAsync('git rebase ...')` does not violate the `no execSync in server code` rule from PAN-70.

### Decision 2 — Single merge oracle: GitHub PR API

`verifyMergedBeforeLifecycle` shrinks to one check: `gh pr view --json mergedAt,mergeCommit`. Ancestor check and diff fallback are deleted. If the PR API doesn't answer or is uncertain, we surface "Unable to verify merge state" to the dashboard and let the human resolve rather than guessing with a heuristic.

This eliminates a category of "the heuristics disagree" bugs (PAN-1024 was one — a squash merge wasn't recognized because the branch tip wasn't an ancestor of main).

### Decision 3 — Refuse dirty worktrees at boundaries

Two new boundary checks replace the silent stash machinery:

**Spawn-time refusal** in `POST /api/agents/:id/start`:
- If `git status --porcelain` is non-empty in the workspace, the route returns `409` with the diff and three actions.
- Frontend shows a modal with: `[Commit current changes]` (prompts for message), `[Discard (requires typed confirmation)]`, `[Stash as salvageable (prompts for description)]`.
- Nothing happens without explicit user choice. `Discard` requires the user to type the literal word `discard` to confirm — destructive operations get the same treatment as `deep-wipe`.

**`pan done`-time refusal** in the work-agent prompt:
- The work-role prompt updates to require `git status --porcelain` empty before `pan done` is allowed.
- If non-empty, the agent must either commit (with a meaningful message) or run `pan stash salvage <description>` to create a `salvageable:*` stash. Discard is a CLI option (`pan workspace discard`) requiring a typed confirmation flag.
- The `pan done` command itself adds a precheck that returns a non-zero exit code with a clear error if the worktree is dirty.

### Decision 4 — Stash taxonomy collapses to one kind

`CanonicalStashKind` becomes `'salvageable'` only. Removed kinds:

- `pre-spawn` — replaced by spawn-time refusal modal
- `review-temp` — replaced by `pan done` dirty-tree precheck
- `pre-merge` — already dead code in the current flow; this PRD formally deletes it

`buildStashMessage`, `parseCanonicalStashMessage`, and `createNamedStash` shrink accordingly. `dropStash` and `listStashes` stay — they're still useful for `pan stash audit`. The stash janitor in deacon becomes `auditSalvageableStashes` and only surfaces `salvageable:*` entries to the dashboard inspector. The startup scanner is deleted entirely (it was nagging about stashes the system itself no longer creates).

### Decision 5 — Rules move to global agent context

The remaining stash rule — *humans only create `salvageable:*` stashes; agents never run `git stash`; Overdeck's own code only creates `salvageable:*` and only via the typed builder* — moves into `/home/eltmon/.panopticon/context/global.md` so every agent across every project sees it, not just agents working inside Overdeck's own repo.

The `CLAUDE.md` "Stash Hygiene" section shrinks to a one-paragraph pointer to global.md.

## Phases

This is a single PR. Phases are organizational, not deliverables.

### Phase 1 — Server-side rebase module

Create `src/lib/git/rebase.ts` with a typed `rebaseFeatureBranch(workspacePath, targetBranch)` that returns `{ok: true, pushedSha} | {ok: false, conflictFiles, abortedAt}`. Unit-tested against a fixture repo.

### Phase 2 — Single-oracle merge verification

Replace `verifyMergedBeforeLifecycle` body with the PR API check. Delete ancestor and diff-fallback paths. Update callers (only `postMergeLifecycle`).

### Phase 3 — Spawn refusal modal

Update `POST /api/agents/:id/start` to return 409 + diff payload when workspace is dirty. Add frontend modal with three buttons. Discard button requires typed `discard` confirmation.

### Phase 4 — `pan done` precheck

Update work-agent role prompt to require clean tree before `pan done`. Add precheck to the `pan done` CLI. Add `pan stash salvage <description>` and `pan workspace discard --confirm` subcommands.

### Phase 5 — Delete the ship role

Remove `roles/ship.md`, `src/lib/cloister/prompts/ship.md`, ship-role spawn code in merge-agent.ts, ship-role state in review-status.ts. Delete `startShipRole` and all callers. Update `roles/work.md` if any cross-references exist.

### Phase 6 — Collapse stash taxonomy

Update `CanonicalStashKind` union. Delete `pre-spawn`, `pre-merge`, `review-temp` parsing paths in `parseCanonicalStashMessage`. Delete `ensureReviewTempStash` and `cleanupReviewTempStash` in review-agent.ts. Delete `pre-spawn` block in routes/agents.ts:3000. Delete `dropLingeringPreMergeStashes` in merge-agent.ts. Remove unused imports.

### Phase 7 — Move rules to global.md

Edit `/home/eltmon/.panopticon/context/global.md` to add the one-paragraph stash rule. Shrink `CLAUDE.md` "Stash Hygiene" section to a pointer. Run `pan sync` to render.

### Phase 8 — Write `docs/MERGE-WORKFLOW.md`

One state-machine diagram. Two actors (dashboard, GitHub). Four states (work-done → review-passed → rebased → merged). One paragraph each. Replaces the implicit pipeline contract scattered across six files.

### Phase 9 — Tests + cleanup

Update tests in `src/lib/cloister/__tests__/`, `src/dashboard/server/routes/__tests__/`. Add new tests for `rebase.ts`, the spawn refusal route, the `pan done` precheck. Delete tests for removed code paths.

## Risks

| Risk | Mitigation |
|---|---|
| Server-side rebase hits an edge case the LLM-driven ship-role handled | Unit-test `rebase.ts` against representative conflict scenarios from `refs/stash` history; conflict path always falls back to "Resume in workspace" — never silently fails |
| Deleting `pre-spawn` strands user work that was previously auto-stashed | Spawn refusal modal makes the choice explicit. The user gains visibility, not loses it |
| `pan done` precheck blocks a legitimate "I'm done, scratch is irrelevant" flow | `pan workspace discard --confirm` exists as the explicit "throw it away" path |
| GitHub PR API outage breaks merge verification | `postMergeLifecycle` surfaces "Unable to verify merge state — check manually" instead of guessing. The dashboard still works; only the auto-cleanup is deferred until the API responds |
| Existing in-flight PRs at merge time | Complete under the old flow; the dashboard checks `reviewStatus` for ship-role markers and routes accordingly during the cutover window |

## Resolved Open Questions

### Squash or rebase merges?

**Decision: Keep squash.** The dashboard Merge button continues to call `gh pr merge --squash`. No reason to change; rebase-merges would complicate conflict semantics (server-side rebase has to push the rebased branch *before* the merge, and ref-update races become a real concern). Status quo for this dimension.

### Discard button on `pan done` and Spawn refusal modal?

**Decision: Yes, offer Discard, with typed confirmation.** Following the project rule on destructive operations (deep-wipe also requires typed confirmation), `Discard` in both surfaces — the spawn modal and `pan workspace discard` CLI — requires the user to type the literal word `discard` to proceed. Single-click destructive actions are forbidden; the keystroke cost is the safety mechanism.

### Slot-branch merge loopback simplification?

**Decision: Moot.** [PAN-1517](https://github.com/eltmon/panopticon-cli/issues/1517) deletes the swarm runtime entirely, including all `/api/swarm/*` endpoints, the slot-merge HTTP loopback, and `swarmRuntime` persistence. The slot-aware regex hole I found during stash analysis (`PAN-1312-slot-1` not matching the issue-ID regex) becomes a non-issue after PAN-1517 lands. This PRD assumes PAN-1517's direction is firm and does not propose any slot-related code changes.

If PAN-1517 stalls, the only impact on this PRD is that the slot-suffix issue-ID drift continues to produce one non-canonical stash per slot-merge — but with the entire `pre-merge` kind being deleted by this PRD anyway, the regex no longer matters.

## Appendix A: PR-FAQ

*The original PR-FAQ that motivated this PRD, preserved for reference. Open questions in the original are now answered above in "Resolved Open Questions."*

### PRESS RELEASE

**Overdeck retires its agent-driven rebase pipeline and replaces six-actor merge choreography with one server-side flow**

*Overdeck HQ — 2026-06-15* — Today Overdeck shipped a redesigned merge workflow that removes the ship-role agent, eliminates two of three stash kinds, and reduces "merge" from six interacting actors to two: the dashboard and GitHub.

For the past three months Overdeck's merge pipeline asked an LLM agent to perform `git rebase` — a deterministic operation — and used three different oracles to decide afterward whether the merge had actually happened. The new design moves the rebase server-side where it's a 30-line TypeScript function, deletes the ship-role entirely, and trusts the GitHub PR API as the single source of truth. Two of the three canonical stash kinds (`pre-spawn:` and `pre-merge:`) are removed; the `review-temp:` kind is removed by simply refusing `pan done` from a dirty worktree.

> "I used to need a flowchart to explain how a PR got from 'agent done' to 'merged on main.' Now I can explain it in two sentences." — Edward Becker

The change reduces lines of code in `src/lib/cloister/merge-agent.ts` by roughly 60%, removes one role file and one agent prompt, and lets the canonical-stash-naming convention be deleted from `CLAUDE.md` since the only remaining stash kind (`salvageable:`) is the one created by hand and never by the system.

### FAQ

#### What is the problem you're solving?

The current merge workflow has five real problems that compound:

1. **An LLM is doing a rebase.** Conflict resolution is mechanical; LLMs are creative. This is the single highest-leverage decision in the pipeline given to the component least suited to it.
2. **"Merged" has three definitions.** `verifyMergedBeforeLifecycle` checks ancestor-of-main, then PR API, then diff fallback. We need three because we don't trust any one of them, and the cost is that nobody knows which one is authoritative.
3. **`pre-spawn` stashing hides user work.** When a workspace is dirty at spawn time, the system silently moves the changes into `refs/stash` where they're hard to find later. The existence of `salvageable:*` as a separate kind is the system admitting it can lose your work.
4. **`review-temp` stashing hides agent work.** When an agent calls `pan done` with uncommitted scratch, the system stashes it instead of forcing a commit-or-discard decision. Result: ten-deep `review-temp:PAN-1418:1..10` orphan stacks every time a review re-runs.
5. **The merge-agent module imports stash primitives it never calls.** Dead-defense code defending against a footgun that was already holstered.

#### Why now?

Two reasons:

- We just spent half an evening tracing where 1,716 non-canonical stashes came from. The answer revealed that one of the three canonical stash kinds is dead code and one is papering over a bug in the spawn flow.
- The fix is small. ~300 lines deleted, ~100 added. One role file removed. One doc rewritten.

#### What gets removed?

| Component | Why removed |
|---|---|
| `roles/ship.md` + ship prompts | LLM-driven rebase is the wrong tool |
| `pre-merge` from `CanonicalStashKind` | Already dead — no live creator in current code |
| `pre-spawn` auto-stash | Replaced with explicit refusal modal |
| `review-temp` stash machinery | Replaced with `pan done` dirty-tree precheck |
| Ancestor + diff-fallback merge checks | PR API only — single source of truth |
| Most of `CLAUDE.md` "Stash Hygiene" | Becomes a one-paragraph pointer to global.md |

#### What gets added?

- `src/lib/git/rebase.ts` — typed server-side rebase, ~40 lines
- `docs/MERGE-WORKFLOW.md` — one state machine, two actors, four states
- Spawn refusal modal — three explicit buttons, no silent state movement
- `pan done` precheck — refuses dirty worktree with three explicit options
- Stash rule in `/home/eltmon/.panopticon/context/global.md` — visible everywhere

#### Why not just fix the bugs and keep the structure?

We considered it. It's what got us here — every individual fix made sense in isolation, the cumulative result is a workflow nobody can explain. Each spot-fix locks in the existing architecture. The architecture itself is the problem.

#### Why is rebasing server-side OK if the server can't do blocking work?

Rebases are fast (seconds), already happen in worktrees, and use `execAsync`. The "no `execSync`" rule stays. The server already does heavier work (`gh pr merge`, Docker cleanup). One more `execAsync('git rebase')` doesn't change the contract.

#### What happens when a rebase conflicts?

Server surfaces it in the dashboard: "Rebase blocked on file X. [Open in editor] [Abort and request human merge]." LLMs never resolve conflicts. The work-agent that owned the branch can resume in its workspace, resolve, push, dashboard re-attempts.

This is strictly safer than the current "ship-role attempts narrow resolution" path.

#### Doesn't removing `pre-spawn` lose work when an agent crashes mid-edit?

Opposite. Today: agent crashes → workspace dirty → user clicks Start → silent stash → work lost in `refs/stash`. After: agent crashes → workspace dirty → user clicks Start → modal with diff + three buttons. User makes an informed choice.

#### Doesn't removing `review-temp` lose agent scratchwork?

If it matters, commit it. If not, discard it. `pan done` refusing dirty worktrees forces the decision. The `review-temp:PAN-1418:1..10` orphan stack is exactly what "defer forever" looks like in practice.

#### What about in-flight PRs / running agents at switchover?

- In-flight PRs complete under old flow (dashboard checks `reviewStatus` markers)
- Existing stashes stay; new scanner only watches `salvageable:*`
- Running agents are unaffected; they didn't stash anyway

#### What gets harder?

- Cleaning up existing non-canonical stashes is a one-time manual job (`pan stash audit` follow-up)
- Diagnosing rebase failures requires reading server logs instead of agent transcripts — trade: deterministic and cheaper
