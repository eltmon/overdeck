# PAN-1918 — CI/CD Reliability & Coverage Hardening

**Issue:** [PAN-1918](https://github.com/eltmon/panopticon-cli/issues/1918)
**Status:** draft
**Author:** orchestrator conversation (operator-directed), 2026-06-15
**Anchored on:** the frontend-suite coverage hole + `IssueMissionControl.test.tsx` open-handle hang discovered while shipping [PAN-1908](https://github.com/eltmon/panopticon-cli/issues/1908).

---

## 1. Problem

Two related failures keep biting the pipeline, and they share root causes:

1. **The full frontend unit suite runs nowhere.** On current `main`:
   - Root `npm test` = `vitest run` (root suite) + the frontend suite **limited to 3 files** (`package.json` `test`).
   - The Panopticon verification gate's `test` command was narrowed the same way (`src/lib/cloister/validation.ts:335`, via commit `10ba58a42`; ref [PAN-1857](https://github.com/eltmon/panopticon-cli/issues/1857)).
   - `.github/workflows/ci.yml` runs `npm test` + one frontend **Playwright** spec; no job runs the full frontend vitest suite.
   - Net: ~the entire dashboard UI suite (Command Deck, Kanban, EventRouter, Stage/cockpit — hundreds of tests) is validated by **nothing**. Frontend regressions ship undetected.

2. **The test suite is not hermetic, so it goes red under parallelism/load** — and a red `main` silently empties the merge gate. The recurring red-main/flaky cluster ([PAN-1824](https://github.com/eltmon/panopticon-cli/issues/1824), [PAN-1720](https://github.com/eltmon/panopticon-cli/issues/1720), [PAN-1880](https://github.com/eltmon/panopticon-cli/issues/1880), [PAN-683](https://github.com/eltmon/panopticon-cli/issues/683), [PAN-1903](https://github.com/eltmon/panopticon-cli/issues/1903), [PAN-1698](https://github.com/eltmon/panopticon-cli/issues/1698)) is all one disease: tests that leak handles, share mutable global state across files, depend on real timers / real `~/.panopticon` / real DB, or hardcode fixtures that drift.

The band-aid for #1 (narrow the frontend run to 3 files) was applied precisely because of #2 (the suite hangs/flakes). Fixing #2 is what lets us undo the band-aid in #1.

**Why this matters now:** every red-main also blocks every PR's verify/ship gate (`a red main → no PR reaches `readyForMerge` → Flywheel merge gate empties`), so this is substrate, not cosmetics.

## 2. Glossary

- **Verification gate / quality gate:** Panopticon's server-side check (`src/lib/cloister/validation.ts`, PAN-174) that runs after a work agent signals "done", *before* advancing to review. Defined by `DEFAULT_GATES` or a per-project `quality_gates` in `projects.yaml`. Distinct from GitHub CI.
- **GitHub CI:** `.github/workflows/ci.yml`, run on PRs/main. The final authority on a PR.
- **`--changed {{CHANGED_BASE}}`:** vitest flag scoping the run to test files affected by the diff vs a base ref (`origin/main`). A change to a widely-imported module pulls in many tests.
- **Hermetic test:** a test whose result depends only on its own setup — no shared mutable globals across files, no real timers, no real filesystem/DB/process state, and which releases every handle it opens.
- **Open handle:** a timer/socket/interval left running after a test's assertions finish; it keeps the worker process alive so vitest never exits → the run "hangs".
- **Test pollution:** one test file mutating module-level/global state that another file reads, making pass/fail depend on execution order or parallel scheduling.
- **`maxForks:1`:** running vitest single-fork to hide cross-file pollution — a symptom-masking workaround, not a fix.
- **Red main empties the merge gate:** when `main`'s CI `test` job is red, every PR inherits the failing check, so none reach `readyForMerge`; the Flywheel has nothing to merge.

## 3. Goals / Non-goals

**Goals**
- The full frontend vitest suite runs on every PR and reliably passes.
- The whole suite (root + frontend) is hermetic and parallel-safe — no test needs `maxForks:1`, no test leaks a handle, no test depends on real timers/env/DB.
- The verification gate cannot hang the box: it reaps the test process group on timeout/failure and bounds concurrency.
- A red `main` is surfaced as its own alarm and does not silently block unrelated PRs.

**Non-goals (explicitly out of scope; tracked separately)**
- [PAN-607](https://github.com/eltmon/panopticon-cli/issues/607) — evaluate the Ultimate Bug Scanner *tool* for the gate. This is a tooling evaluation, not a reliability fix.
- [PAN-1101](https://github.com/eltmon/panopticon-cli/issues/1101) — permission-safety CI guard / runtime tripwire. A security feature with its own design.
- [PAN-1609](https://github.com/eltmon/panopticon-cli/issues/1609) and [PAN-666](https://github.com/eltmon/panopticon-cli/issues/666) are **closed**; their intent (CI must not ship a red suite; the gate must not block on pre-existing failures) is honored here as **design constraints**, not new work items.

## 4. Requirements

**Functional**
- **FR-1** The full frontend vitest suite runs in GitHub CI on every PR and on `main`.
- **FR-2** The verification gate runs the full frontend suite (remove the 3-file narrowing) once FR-4/FR-5 make it reliable.
- **FR-3** On timeout or failure, the verification gate terminates the entire test **process group** — no orphaned `vitest`/`forks.js` workers survive a gate run.
- **FR-4** Every test is hermetic: no reliance on real timers for delay/retry/backoff; no shared mutable global state across files; no reads of real `~/.panopticon`, real beads DB, or live process/tmux state; no `process.exit` reaching the worker.
- **FR-5** A mechanical guard fails CI when a test leaks a handle, uses a forbidden real-timer pattern, or otherwise breaks isolation — so this can't silently regress.
- **FR-6** Tests that assert against registry/schema counts derive the expected value from the source of truth, not a hardcoded literal.
- **FR-7** The pipeline distinguishes a *pre-existing red-main* failure from one a PR introduced: a PR is not blocked solely by an inherited red-main check, and red main raises a distinct, loud alarm.

**Non-functional**
- **NFR-1** Full suite (root + frontend) completes on the 24-core dev box without thrash or leaked workers, within the gate's timeout.
- **NFR-2** The suite passes under full parallelism — no file requires `maxForks:1`.
- **NFR-3** No new flaky tests: changes use fake timers and deterministic fixtures per the repo's fake-timers rule.

## 5. Work items

> Each item names exact files, the change, and the test that proves it. `traces` lists the FR/issue it satisfies.

### WI-1 — Fix the `IssueMissionControl.test.tsx` open-handle hang *(anchor)*
- **Files:** `src/dashboard/frontend/src/components/Stage/cockpit/IssueMissionControl.test.tsx`, and the query hooks it mounts via `src/dashboard/frontend/src/components/CommandDeck/ZoneCOverviewTabs/queries.ts` (`useReviewStatusQuery`, `useActivityQuery`) and the inline `useQuery` in `IssueMissionControl.tsx:612`.
- **Root cause:** the test renders the real `IssueMissionControl`, which mounts React Query hooks. A per-render `QueryClient` (test lines 95–100) is never torn down, so its GC/refetch timers (and any `refetchInterval` in the query hooks) keep the worker's event loop alive after assertions pass. The default reporter then buffers all results waiting for an end that never comes → the whole `--changed` run looks like it "hangs at zero." Verified: a verbose run completes 67/68 files; only this one never reports completion. File is unchanged vs `main` (pre-existing).
- **Change:**
  1. Tear down React Query per test — share one client and `client.clear()` + `cleanup()` in `afterEach`, and set `gcTime: Infinity` (no GC timer scheduled) on the test client's `defaultOptions.queries`.
  2. If `useReviewStatusQuery`/`useActivityQuery` use a real `refetchInterval`, mock them in the test (the module is already mocked at line 8 — ensure the mock returns static data with no interval).
  3. Add an explicit `afterEach(() => { vi.useRealTimers() })` only if fake timers are introduced.
- **Proves it:** `npx vitest run src/dashboard/frontend/src/components/Stage/cockpit/IssueMissionControl.test.tsx` exits 0 and the **process exits promptly** (no hang). Add `--reporter=verbose` to confirm the file reports completion.
- **traces:** FR-2, FR-4, [PAN-1918](https://github.com/eltmon/panopticon-cli/issues/1918)

### WI-2 — Make the verification gate hang-proof (reap workers; bound concurrency)
- **Files:** `src/lib/cloister/validation.ts` (gate runner `runQualityGatesPromise`, the `execAsync` call and the 5-minute timeout at ~`:446`).
- **Root cause:** today the gate kills the parent `vitest` on timeout but orphans its `forks.js` workers, which pin cores; each failed retry leaks more, load climbs, the next run is slower → doom loop (observed on PAN-1908: ~19–24 orphaned workers, load 27).
- **Change:**
  1. Spawn the test command in its own **process group** (`detached: true`) and on timeout/failure send the signal to the **negative PID** (`process.kill(-pid, 'SIGTERM')` then `SIGKILL`) so all workers die with the parent.
  2. Bound frontend concurrency at the gate (e.g. pass `--max-workers` appropriate to a loaded box) rather than defaulting to one fork per core (24).
  3. Size the timeout to the real full-suite duration (the 5-min hard cap is too tight once the full frontend suite runs); make it configurable.
- **Proves it:** new unit test in `src/lib/cloister/__tests__/` asserting that after a simulated gate timeout, no child remains in the process group. Manual: a forced-timeout gate run leaves zero `forks.js` workers.
- **traces:** FR-3, NFR-1, [PAN-1918](https://github.com/eltmon/panopticon-cli/issues/1918) (the "system bug")

### WI-3 — Run the full frontend suite in CI and the gate (close the hole)
- **Files:** `.github/workflows/ci.yml` (test job), root `package.json` (`test` script), `src/lib/cloister/validation.ts:335` (`DEFAULT_GATES.test`).
- **Change:**
  1. Add a CI job/step that runs `cd src/dashboard/frontend && npx vitest run` (full suite), with bounded workers and a real timeout. Keep it required.
  2. Once WI-1/WI-4/WI-5 are green, **remove the 3-file narrowing** in `package.json` `test` and in `DEFAULT_GATES.test`, restoring full-frontend coverage to the gate.
  3. Update `verification-gate.test.ts` expectations accordingly (ref [PAN-1857](https://github.com/eltmon/panopticon-cli/issues/1857)).
- **Decision rule:** WI-3 step 2 lands **only after** the full frontend suite is green in CI (WI-3 step 1) for at least one run — never re-enable a suite that still hangs.
- **Proves it:** CI shows a green full-frontend job; `verification-gate.test.ts` passes with the un-narrowed command.
- **traces:** FR-1, FR-2, [PAN-1918](https://github.com/eltmon/panopticon-cli/issues/1918), [PAN-1609](https://github.com/eltmon/panopticon-cli/issues/1609) (constraint)

### WI-4 — Make the flaky/polluting tests hermetic *(the cluster)*
One sub-item per known offender; each is small and independently verifiable.

Replace real ~5s waits with `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()`. *traces:* FR-4, NFR-3, [PAN-1824](https://github.com/eltmon/panopticon-cli/issues/1824)
- **WI-4b — cross-file state pollution.** `src/lib/cloister/__tests__/pan-871-auto-resume.test.ts`, `tests/lib/cloister/auto-resume-gating.test.ts`, `tests/lib/cloister/failure-tracking.test.ts`. Find the shared module-level mock/state (likely a `resumeAgent`/state-store mock) and reset it in `beforeEach`/`afterEach`; isolate so order doesn't matter. *traces:* FR-4, NFR-2, [PAN-1720](https://github.com/eltmon/panopticon-cli/issues/1720)
- **WI-4c — `process.exit` reaching the worker.** `src/cli/commands/__tests__/start-sync-main-conflict.test.ts`. Mock/stub `process.exit` so the conflict path doesn't kill the fork under `maxForks:1`. *traces:* FR-4, [PAN-1880](https://github.com/eltmon/panopticon-cli/issues/1880)
- **WI-4d — real `~/.panopticon` dependence.** `tests/lib/shadow-state.test.ts`. Replace global `getPendingSyncCount()` assertions with per-issue `needsSync(id)` checks (fix sketched in the issue). *traces:* FR-4, [PAN-683](https://github.com/eltmon/panopticon-cli/issues/683)

### WI-5 — De-brittle stale-fixture tests
- **Files:** `tests/lib/model-fallback.test.ts` (and the schema-version / substrate-smoke fixtures named in the issue).
- **Change:** derive expected model/schema counts from the registry/source of truth (or assert membership, not length) so adding a model (e.g. Fable 5) doesn't redden `main`.
- **Proves it:** adding a model to the registry does not break these tests.
- **traces:** FR-6, [PAN-1698](https://github.com/eltmon/panopticon-cli/issues/1698)

### WI-6 — Mechanical isolation guard + red-main discrimination
- **Files:** a new lint rule or vitest reporter config; the pipeline's red-main handling (`src/lib/cloister/*`).
- **Change:**
  1. **Open-handle/teardown guard:** fail CI if a test leaves the process hanging (e.g. vitest `--hanging-process` reporter, or a forced-exit + handle assertion) — so WI-1's class of bug can't silently return.
  2. **Forbidden-pattern lint:** extend lint to flag real `setTimeout`/`retry-after` in test files (the existing fake-timers rule, mechanically enforced).
  3. **Red-main vs PR-break:** when the gate fails, compare against `main`'s current CI status; if the same test is already failing on `main`, surface it as a *red-main alarm* and do not count it as the PR's blocker (honors the closed [PAN-666](https://github.com/eltmon/panopticon-cli/issues/666) intent).
- **traces:** FR-5, FR-7, NFR-2

## 7. Sequencing (phases)

1. **Stop the bleeding (parallel):** WI-4a–e + WI-5 — make `main` green and parallel-safe. WI-2 — gate can't hang the box. These are independent and can run concurrently.
2. **Guardrails:** WI-4f + WI-6 — lock the hermeticity in so it can't regress.
3. **Close the hole:** WI-1 (fix the anchor hang) → WI-3 step 1 (full frontend in CI, green for one run) → WI-3 step 2 (remove the 3-file narrowing from gate + `npm test`).

Phase 3 step 2 is gated on phase 3 step 1 being green — never re-enable a suite that still hangs.

## 8. Acceptance criteria

- **AC-1 (FR-1/FR-2):** CI runs the full frontend vitest suite on every PR; the gate's `test` command no longer hard-limits the frontend to 3 files. `verification-gate.test.ts` reflects the un-narrowed command.
- **AC-2 (FR-3/NFR-1):** A forced gate timeout leaves **zero** orphaned `vitest`/`forks.js` workers; load returns to baseline. Covered by a new unit test.
- **AC-3 (FR-4/NFR-2):** `npm test` (full, default parallelism) passes on the dev box and in CI with **no** `maxForks:1` and no real-timer waits in the named files.
- **AC-4 (FR-5):** The isolation guard fails CI on a deliberately-leaked-handle / real-timer test fixture (negative test).
- **AC-5 (FR-6):** Adding a model to the registry does not redden `model-fallback.test.ts`.
- **AC-6 (FR-7):** A PR whose only failing check is a pre-existing red-main test is not marked as PR-blocked; red main raises a distinct alarm.
- **AC-7:** `IssueMissionControl.test.tsx` runs in isolation and as part of the full suite without hanging.

## 9. Restated repo rules the executor must honor

- **Fake timers:** any retry/backoff/delay test MUST use `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()` — never real `setTimeout`. Never drop `maxForks` to 1 to mask OOM/pollution; that hides the bug this PRD removes.
- **No `execSync`** in server-reachable code; use `execAsync`/`spawn`.
- **Surgical changes:** each changed line traces to a work item; don't refactor adjacent code.
