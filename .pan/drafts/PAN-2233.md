## 0. Glossary

- **`merge-agent.ts`** — `src/lib/cloister/merge-agent.ts` (1,414 lines): merge handoff
  machinery. Four clusters (verified structure map): sync-main (merging latest main into
  a feature workspace, incl. auto-commit-before-sync and main-preferred conflict
  resolution), post-merge lifecycle (the non-destructive handoff after a PR merges),
  merge-output parsing/telemetry (parsing agent output, git-pattern scanning, history +
  activity logging, stranded-merge salvage), and project quality gates.
- **`postMergeLifecycle`** (grep anchor: `export async function postMergeLifecycle`) —
  after merge: mark `verifying_on_main`, apply labels, pause agents, preserve
  workspace/state/vBRIEF/branches, and stop Docker containers + networks. Destructive
  completion belongs to close-out, NOT here — the function is non-destructive by design.
- **In-flight guard** — `createInFlightGuard()` from
  `src/lib/cloister/in-flight-guard.ts`, applied as `postMergeGuard` in
  `src/dashboard/server/routes/specialists.ts` (grep anchor:
  `const postMergeGuard = createInFlightGuard();`). NOT part of this file — this
  decomposition must not touch it; the barrel keeps `postMergeLifecycle` resolving for
  its call sites so the guard wiring is unaffected.
- **Barrel decomposition / pure move + re-export** — as PAN-2232 §0; all importers
  resolve unchanged through `merge-agent.ts` re-exports.

## 2. Locked design decisions

- **D1 — Pure moves only; barrel re-exports; zero importer edits** (as PAN-2232 D1/D2).
- **D2 — Four seam modules** sized from the verified map: `merge-agent-sync-main.ts`
  (≈540), `merge-agent-post-merge.ts` (≈420), `merge-agent-output.ts` (≈370),
  `merge-agent-quality-gates.ts` (≈60). `merge-agent.ts` ends as a barrel (≈80) and its
  baseline entry is lowered.
- **D3 — `postMergeLifecycle` moves VERBATIM as one block**, including its numbered-step
  comments and the entire Docker cleanup step (anchors:
  `// 4. Stop Docker containers + networks` …
  `const { stopWorkspaceDocker } = await import('../workspace-manager.js');` …
  `logActivity('docker_cleanup'`). No step reordering, no step removal, no "cleanup" of
  the dynamic import. AC-3 mechanically checks the step survived.
- **D4 — Cross-seam private helpers** (`logActivity`, `shellQuote`,
  `captureTmuxOutput`): home = primary caller's seam, exported to the other seams;
  note each in the commit message. Types (`MergeResult`, `MergeConflictContext`,
  `SyncMainResult`) live with their primary cluster and are re-exported via the barrel.
- **D5 — Idempotency surfaces are behavior, not style.** `resetPostMergeState` and any
  `_completedPostMerge` / `mergeStatus` checks inside the moved code are part of the
  PAN-328 defense-in-depth — move verbatim; if anything about them looks redundant,
  that redundancy is deliberate.
- **D6 — Cycle discipline as PAN-2149 D6 / PAN-2232 D5** (build per seam; PAN-2230
  `--update` with audited additions; runtime ESM cycle ⇒ stop and report).
