## Glossary

- **Convoy / full mode** — today's review: a `review` (synthesis) parent plus four sub-reviewers (`security`, `correctness`, `performance`, `requirements`) spawned in parallel. The parent idles, collects four `REVIEWER_*` signals, reads four reports, and emits one verdict. Convoy spawn: `Promise.all(REVIEW_SUB_ROLES.map(...))` at `src/lib/cloister/review-agent.ts:493`; synthesis-parent spawn `spawnRun(opts.issueId, 'review', {...})` at `:473`.
- **Quick mode (this PRD)** — a single `review` role agent that reads the diff and reviews it **itself** in one combined pass (correctness + security + requirements + performance), then emits the verdict directly. **No convoy spawn, no synthesis fan-out, no sub-reviewer sessions.**
- **Review entry point** — `spawnReviewRoleForIssue()` in `src/lib/cloister/review-agent.ts` — the single function every review caller goes through (≈10 callers: `routes/specialists.ts:1583`, `routes/workspaces.ts` ×5, `cloister/deacon.ts` ×4). Branching the mode here covers all callers with no per-call-site logic.
- **Verdict signal** — `pan admin specialists done review <id> --status passed|blocked --notes "..."` — how the synthesis parent emits its verdict today (the agent prompt at `review-agent.ts:206-207`). Quick mode reuses this verbatim — the single agent runs it itself.
- **Re-review cycle** — a review triggered after the work agent commits fixes. HEAD changed → new `runId` → today the **stale-runId** branch fires (`review-agent.ts:325`, `:339`, `:352-360`) and `killAllReviewerSessions()` + full respawn.
- **`remain-on-exit on`** — tmux setting (PAN-830) that keeps an agent's Claude process alive after a turn so a later cycle can resume it via `sendKeysAsync`.
- **Session continuity (this PRD)** — on a re-review cycle in quick mode, the same review agent is **resumed** (its live session fed the new diff + the work agent's response) instead of killed and respawned, so it keeps the context of its prior review and the fixes.

---

## Problem

**P1 — The convoy is heavyweight for the value it returns here.** Full mode spawns five agents (parent + four reviewers), each opening files independently. For our current needs a single self-review pass is wanted; the convoy is overkill.

**P2 — Every re-review throws away context.** A fixes-commit hits the stale-runId branch (`review-agent.ts:352-360`) which kills the whole convoy and respawns fresh, discarding the reviewer's accumulated understanding of the change. The reviewer re-reads everything from zero each cycle.

```ts
// review-agent.ts:352-360 (verified 2026-06-19)
if (!paneDead && !opts.force && !staleRunId) {
  return { success: true, message: `Review already in progress: ${reviewSessionName}` };
}
// Session pane is dead, force mode, or stale runId — kill the whole convoy and respawn.
const reason = opts.force ? 'force-killed for re-review' : paneDead ? 'pane is dead' : 'stale runId';
await Effect.runPromise(killAllReviewerSessions(undefined, opts.issueId).pipe(...));
```

**P3 — There is no review mode switch.** `RoleConfig` (`src/lib/config-yaml.ts:403`) has no `mode`; review is always the convoy. There is no way to ask for a lighter review.

## Requirements

### Functional

- **FR-1 (Review mode config)** — Add `roles.review.mode ∈ { full, quick }` to `RoleConfig` (`src/lib/config-yaml.ts:403`), the settings-API normalizer (`src/lib/settings-api.ts`), and the loader. Resolves through the existing global + per-project `roles.review` merge (per-project wins). A `resolveReviewMode(config)` helper returns the effective mode. **Default `quick`** (Decision 1).
- **FR-2 (Quick = the review parent reviews itself)** — When mode is `quick`, `spawnReviewRoleForIssue()` spawns the **one** `review` role agent with a self-review prompt and does **not** spawn the convoy or a separate synthesis agent. The agent reads the diff, performs a single combined correctness + security + requirements + performance pass, and emits its verdict directly via `pan admin specialists done review` (the existing path). No `REVIEWER_*` signal machinery is used.
- **FR-3 (Self-review prompt)** — The quick-mode `review` agent gets a prompt that (a) tells it to review the change itself across all dimensions in one pass, (b) explicitly tells it **not** to spawn or wait for sub-reviewers, and (c) gives it the existing context manifest's Tier-1 summary (risk-ranked changed files + acceptance criteria, already built by `buildReviewContext()`) as a starting point, then read the diff/files itself. Reuse the manifest — do not rebuild it.
- **FR-4 (Session continuity across re-review)** — The quick-mode review agent's session uses `remain-on-exit on` and survives after it emits a verdict. On a re-review cycle (stale-runId branch, `review-agent.ts:352-360`), if mode is `quick` and the agent's session is alive, **resume** it: deliver the new diff + the work agent's response + a "re-review — build on your prior review" framing via `sendKeysAsync` (load-buffer + paste-buffer pattern), instead of `killAllReviewerSessions` + respawn. Only fall back to a fresh spawn when the session is gone.
- **FR-5 (Verdict + lifecycle unchanged)** — The verdict path is unchanged: the agent runs `pan admin specialists done review --status passed|blocked`, which advances the lifecycle (review → test, or → `readyForMerge`) exactly as today. No gate change, no new `reviewStatus` value.

### Non-functional

- **NFR-1 (Single entry point)** — The mode branch lives only at `spawnReviewRoleForIssue()`, so all ≈10 callers honor it without change.
- **NFR-2 (No regression to full mode)** — `mode: full` is byte-for-byte today's convoy. The quick path is additive; `killAllReviewerSessions` still serves terminal lifecycle events (merge/reset/cancel/abort) unchanged.
- **NFR-3 (Async tmux only)** — re-review resume uses `sendKeysAsync()` with load-buffer + paste-buffer; never `sendKeys()`.
- **NFR-4 (No new sessions on resume)** — the re-review resume re-attaches to the *same* Claude session; it must not rotate to a fresh one. This rides on PAN-1980's `ALLOW_SESSION_ROTATION_ON_RESUME=false` — a quick-review resume is a normal `--resume`, not a compact/drift rotation.
- **NFR-5 (Idempotency / restart-safe)** — a duplicate review trigger for the same alive quick session resumes (or no-ops if a turn is in flight); it never double-spawns.

## Proposal

### A — Review mode config (FR-1)

`RoleConfig` gains `mode?: 'full' | 'quick'`. Add `resolveReviewMode(cfg)` (defaults `quick`). Thread through `settings-api.ts` normalizer + the YAML loader so global + per-project both carry it.

### B — Branch the entry point (FR-2)

At the top of `spawnReviewRoleForIssue()` (before the convoy spawn at `:473`/`:493`):

```ts
// PAN-1981: quick mode — the review agent reviews the diff itself; no convoy.
const mode = resolveReviewMode(cfg); // 'full' | 'quick', default 'quick'
if (mode === 'quick') {
  return spawnQuickReviewForIssue(opts); // one 'review' agent, self-review prompt, no fan-out
}
// mode === 'full' → existing convoy (unchanged)
```

`spawnQuickReviewForIssue(opts)`:
1. Reuse the existing context-manifest build (`buildReviewContext` / Tier-1 summary) that the full path already does — do not rebuild.
2. `spawnRun(opts.issueId, 'review', { workspace, prompt: buildQuickReviewPrompt({ ...opts, tier1Summary }), remainOnExit: true, ... })`.
3. Do **not** spawn sub-reviewers. The agent emits the verdict via `pan admin specialists done review` (FR-5).

### C — Self-review prompt (FR-3)

Add `buildQuickReviewPrompt()` (or a `phase: 'quick'` branch in the existing review-prompt builder) that produces a prompt instructing the `review` agent to: review the change itself across correctness/security/requirements/performance in one pass; NOT spawn or await sub-reviewers; use the Tier-1 summary as the risk map, then read the diff/files directly; emit the verdict via `pan admin specialists done review`. Add a `roles/review-quick.md` body (or a quick section in `roles/review.md`) carrying the dimension checklist + severity/verdict vocabulary, mirroring the convoy reviewers' rigor in one agent.

### D — Session continuity on re-review (FR-4)

In the stale-runId branch (`review-agent.ts:352-360`), before `killAllReviewerSessions`, add the quick path:

```ts
// PAN-1981: quick mode — resume the review agent across re-review runs instead
// of kill+respawn, so it keeps the context of its prior review.
if (mode === 'quick' && (await reviewSessionAlive(opts.issueId))) {
  await resumeQuickReview(opts.issueId, { newDiffSummary, workAgentResponse }); // sendKeysAsync
  return { success: true, message: 'Resumed quick review with new changes' };
}
// else fall through to the existing kill+respawn (session gone / full mode)
```

`resumeQuickReview` delivers, via `sendKeysAsync` (load-buffer + paste-buffer), a "re-review" message: what changed since the last review + the work agent's response + "build on your prior review; re-emit your verdict." `killAllReviewerSessions` is untouched for terminal lifecycle events (NFR-2).

## Decisions made in this document (do not re-litigate)

1. **Default review mode is `quick`** — per the operator directive ("for now we only do quick review"). `full` (the convoy) is the explicit opt-in via `roles.review.mode: full` (global or per-project). The convoy code is **not removed** — only switched off by default.
2. **The review *parent* reviews itself** — quick mode spawns the `review` role agent (not a separate `review-quick` agent as #1862 FR-15 sketched). One agent, one session, which is also the thing we resume across runs. Fewest moving parts and it's what makes FR-4 clean.
3. **No fork / cache-sharing here** — that is #1862 Phase A; quick mode has nothing to fork (single agent). Out of scope.
4. **No `mode: none` / `reviewStatus: 'skipped'`** — quick review always produces a real `passed`/`blocked` verdict, so no gate or schema change. `none` stays in #1862.
5. **Re-review rides on resume, not re-fork** — the single agent's live session already holds the prior review; resuming it (FR-4) is both cheaper and higher-fidelity. Fresh spawn is only the fallback when the session is gone.
