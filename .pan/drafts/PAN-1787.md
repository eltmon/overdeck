## Glossary (read first)

| Term | Meaning |
| --- | --- |
| **Harness** | The coding-agent CLI an agent runs inside: `claude-code`, `pi`, or `codex`. Type `RuntimeName` (`src/lib/runtimes/types.ts:20`) and `Harness` (`packages/contracts/src/types.ts:49-51`). |
| **Provider** | The model vendor: `anthropic`, `openai`, `google`, `kimi`, `minimax`, `zai`, `mimo`, `openrouter`, `nous`, `dashscope`. Registry: `PROVIDERS` in `src/lib/providers.ts`. A model's provider is derived via `getProviderForModelSync(modelId)` (`src/lib/providers.ts:~180`). |
| **Role** | Pipeline role of a spawned agent: `plan`, `work`, `strike`, `review`, `test`, `ship`, `flywheel`. Config: `RolesConfig` (`src/lib/config-yaml.ts:347-365`). |
| **Spawn site** | Any code path that launches an agent and must therefore decide its harness. The five sites are listed in "Current State". |
| **Policy gate** | `canUseHarnessSync(harness, model, authMode)` (`src/lib/harness-policy.ts:69-97`). Blocks ToS-violating combos (Pi + Anthropic + subscription auth). Blocked ⇒ collapse to `claude-code`. NEVER bypass or weaken it. |
| **continue.json** | `<workspace>/.pan/continue.json` — the agent's own session-state file (progress, decisions, hazards, resume point). The re-onboarding mechanism after any restart. |
| **PTY supervisor** | `dist/pty-supervisor.js` wrapper that owns Claude's PTY for claude-code agents/conversations. tmux keystrokes still reach the harness TUI through it. |
| **Transcript converter** | `convertConversationTranscript` (`src/lib/session-format-converter.ts:185-308`) — converts a conversation transcript between harness JSONL formats when a conversation switches harness. |
| **Switch tiers** | `decideSwitchStrategy` (`src/lib/conversations/switch-strategy.ts:41-60`): tier 1 = in-session `/model` command; tier 2 = respawn+resume (provider routing changed); tier 3 = compact then respawn (context overflow); tier 4 = harness changed (uses the converter). |

## Vision

One coherent harness-routing story: every agent spawn resolves its harness through a
single, predictable chain — explicit request → role override → provider override →
**built-in provider default** — and every surface that shows a provider or harness
shows its real brand logo, consistently. Switching the model or harness of an
already-running agent is session-safe by construction.

## Motivation

1. **No smart defaults.** Every unset harness silently falls back to `claude-code`,
   even for OpenAI/Google/Kimi models. `DEFAULT_CONFIG.providerHarnesses` is `{}`
   (`src/lib/config-yaml.ts:967`) and the terminal fallback at every spawn site is
   hardcoded `'claude-code'`.
2. **Role harness override exists in schema but not in UI.** `RoleConfig.harness`
   (`src/lib/config-yaml.ts:349`) is honored at every spawn site, but Settings →
   Roles exposes a harness picker only for Flywheel (`RolesPanel.tsx:512-525`).
3. **Logos are inconsistent.** Only Anthropic and OpenAI have real SVG marks
   (`src/dashboard/frontend/src/components/chat/ProviderIcons.tsx`); eight providers
   are colored letter badges; Settings → Providers uses generic lucide icons
   (`SettingsPage.tsx:423-433`); harness rows are text-only; Pi agents render with
   the **Anthropic** starburst (`AgentIcon.tsx:14-19` has no pi branch).
4. **Live-switch semantics are implicit.** Restarting an agent with a new
   model/harness happens to start a fresh session, but nothing guarantees it; the
   graceful grace period (30s) is too short when the agent is mid-action; and
   conversation harness switching silently runs an unsupported transcript converter.

## Goals

1. Built-in provider→harness defaults: `anthropic → claude-code`,
   `openai → codex`, **all other providers → pi**; user-overridable per provider.
2. Role-level harness override UI for every role.
3. One unified, exported `resolveHarness()` used by every spawn site.
4. Session-safe live switching: 60s graceful grace + Escape interrupt; fresh-session
   invariant for any agent model/harness change; conversation harness switching
   labeled experimental.
5. Real brand logos for all providers and harnesses from a single shared registry.

## Non-Goals

- Sub-role harness overrides (`RoleSubConfig` has only `model`, `config-yaml.ts:339-341`).
- New harnesses beyond claude-code / pi / codex.
- Per-project harness configuration (`projects.yaml`).
- Changing conversation **model**-switch behavior (tiers 1–3 stay exactly as-is).
- Removing or improving the transcript converter (kept, labeled experimental — WI-6).

import type { RuntimeName } from './runtimes/types.js';
import type { Role } from './config-yaml.js';

export interface ResolveHarnessInput {

}

/**

### WI-5: Fresh-session invariant — never `--resume` across a model/harness change

**Facts:** `restartAgent` already always builds a fresh session (it never passes
`resumeSessionId` into `buildAgentLaunchConfig`) and then prompts:
*"You are resuming work on `<issue>`. Read `.pan/continue.json` for context and pick
up where you left off."* (`agents.ts:4787`). The JSONL-resume path is separate:
`spawnMode: 'resume'` + `resumeSessionId` (`agents.ts:2647,3107-3142`) produces
`--resume <id>` in the launcher (`launcher-generator.ts:427,529`) and is used by
deacon auto-resume and specialist resumes.

**Changes:**

1. Persist, at session-creation time, the model+harness the JSONL session was
   created with (extend agent runtime state where `claudeSessionId` is stored —
   `agents.ts:2037,2159-2167`).
2. In every path that sets `spawnMode: 'resume'`: if the agent's current
   `state.model` or `state.harness` differs from the persisted session
   model/harness, **drop the resume id** (fall back to fresh session + the
   continue.json re-onboarding prompt) and log why.
3. Decision rule, stated plainly: **a model change, a harness change, or both ⇒
   fresh harness session. JSONL `--resume` is only ever used when both are
   unchanged.** No transcript conversion for agents, ever (the converter is
   conversation-only).

**Tests:** unit — restart with `newModel` only / `newHarness` only / both ⇒
launcher script contains no `--resume`; resume path with unchanged model+harness
keeps `--resume`; resume path with drifted model drops it.

## Behavior Change (call out loudly)

With built-in defaults active, spawns with **no** explicit/role/provider override
change harness: OpenAI models `claude-code` (CLIProxy) → **codex**; Google/Kimi/
Z.ai/MiniMax/MiMo/OpenRouter/Nous/DashScope models → **pi**. Mitigations:

- Explicit settings always win; one Settings change restores old behavior
  (e.g. `provider_harnesses.openai = claude-code`).
- Policy gate unchanged; blocked combos still collapse to `claude-code`.
- Conversations unaffected (harness pinned at creation).
- Emit a one-line activity-log notice the first time a spawn uses a built-in
  default ("harness codex chosen by provider default — override in Settings →
  Providers"), plus a release-notes entry.
