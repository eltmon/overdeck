# PAN-1787: Provider Default Harnesses, Role-Level Harness Overrides, and Real Brand Logos

**Issue:** [PAN-1787](https://github.com/eltmon/panopticon-cli/issues/1787)
**Status:** Draft
**Date:** 2026-06-11

## Vision

One coherent harness-routing story: every agent spawn resolves its harness through a
single, predictable chain — explicit request → role override → provider override →
**built-in provider default** — and every surface that shows a provider or harness
shows its real brand logo, consistently.

## Motivation

Three gaps, one theme (harness identity is half-built):

1. **No smart defaults.** Every unset harness silently falls back to `claude-code`,
   even for OpenAI, Google, Kimi, etc. models. The natural pairing — Anthropic →
   Claude Code, OpenAI → Codex, everything else → Pi — does not exist anywhere.
   `DEFAULT_CONFIG.providerHarnesses` is `{}` (`src/lib/config-yaml.ts:967`) and the
   terminal fallback at every resolution site is hardcoded `'claude-code'`.
2. **Role harness override exists in schema but not in UI.** `RoleConfig.harness`
   (`src/lib/config-yaml.ts:~349`) is read at every spawn site, but the Settings →
   Roles panel only exposes a harness picker for the Flywheel role
   (`RolesPanel.tsx:512`). Plan, Work, Strike, Review, Test, and Ship can only be
   configured by hand-editing YAML.
3. **Logos are inconsistent.** Only Anthropic and OpenAI have real SVG marks
   (`src/dashboard/frontend/src/components/chat/ProviderIcons.tsx`); the other eight
   providers render as colored letter badges. The Settings Providers list uses
   generic lucide icons (`Code`, `Lightbulb`, `Globe`, `Zap`, `Brain` —
   `SettingsPage.tsx:423-433`). Harness rows in both model pickers are text-only.
   Pi agents render with the **Anthropic** starburst
   (`AgentIcon.tsx providerForAgent()` maps anything containing "claude" → anthropic
   and has no pi branch).

## Goals

1. Built-in provider→harness defaults: `anthropic → claude-code`, `openai → codex`,
   **all other providers → pi**. User-overridable per provider (storage already exists).
2. Role-level harness override UI for every role, matching the existing model override UX.
3. One unified, exported resolution function used by every spawn site.
4. Real brand logos for all providers and all harnesses, from a single shared registry,
   used consistently across every surface.

## Non-Goals

- Sub-role harness overrides (`RoleSubConfig` has only `model`; review sub-roles are
  inlined prompt templates, not separate spawns). Note for future work only.
- Changing the conversation-pinning rule (a conversation keeps its harness; only an
  explicit switch via the picker changes it).
- New harnesses beyond claude-code / pi / codex.
- Per-project harness configuration (`projects.yaml`).

## Design

### 1. Built-in provider defaults

Add a `defaultHarness: RuntimeName` field to each entry of the `PROVIDERS` registry
in `src/lib/providers.ts` (the provider registry is the natural home — each provider
declares its own default):

```ts
anthropic:  { ..., defaultHarness: 'claude-code' },
openai:     { ..., defaultHarness: 'codex' },
google:     { ..., defaultHarness: 'pi' },
kimi:       { ..., defaultHarness: 'pi' },
minimax:    { ..., defaultHarness: 'pi' },
zai:        { ..., defaultHarness: 'pi' },
mimo:       { ..., defaultHarness: 'pi' },
openrouter: { ..., defaultHarness: 'pi' },
nous:       { ..., defaultHarness: 'pi' },
dashscope:  { ..., defaultHarness: 'pi' },
```

### 2. Unified `resolveHarness()`

Extract a single exported function (new module `src/lib/harness-resolve.ts`, or
exported from `agents.ts`) and make every spawn site call it:

```ts
/**
 * Resolution order (first defined wins):
 *   1. explicit       — --harness flag / dashboard body / spawn options
 *   2. role           — config.roles[role].harness
 *   3. provider user  — config.providerHarnesses[provider]   (Settings override)
 *   4. provider built-in — PROVIDERS[provider].defaultHarness
 *   5. 'claude-code'  — terminal fallback (unknown provider)
 * The winner always passes through canUseHarnessSync(); a blocked combination
 * collapses to 'claude-code' (current behavior, kept).
 */
export async function resolveHarness(opts: {
  explicit?: RuntimeName;
  role?: Role;
  model: string;          // provider derived via getProviderForModelSync(model)
}): Promise<RuntimeName>
```

Call sites to migrate (delete the duplicated inline logic):

- `src/lib/agents.ts:329-336` (`resolveEffectiveHarness` becomes/feeds this)
- `src/lib/agents.ts` `spawnAgent` + `spawnRun` (`options.harness ?? roles[role].harness`
  prefixes move inside `resolveHarness`)
- `src/lib/planning/spawn-planning-session.ts:513-516`
- `src/lib/agents.ts:4736-4737` (restart keeps `agentState.harness` as the explicit arg)
- `src/dashboard/server/routes/agents.ts:3156-3165` (validate body, pass as explicit)

**Harness availability:** if the resolved harness binary is not installed (pi/codex),
collapse to `claude-code` and emit an activity-log warning — same shape as the policy
collapse. `pan doctor` already surfaces installation status.

**`specialist_harnesses` reconciliation (PAN-636):** the cloister
`model_selection.specialist_harnesses` block is a parallel, pre-Codex mechanism
(claude-code|pi only). Fold it into the unified chain: treat it as a deprecated alias
for `roles[role].harness` (read-if-set with a deprecation warning) or migrate-on-load.
Decide during planning; it must not remain a fourth, undocumented resolution tier.

### 3. Settings UI — provider default harness

The existing per-provider "Default harness" select (`SettingsPage.tsx:1821-1832`)
currently renders `value={provider_harnesses[p] ?? 'claude-code'}`, which (a) lies
about the effective default and (b) cannot distinguish "unset" from an explicit
`claude-code` choice. Change to:

- First option: `Default (Codex)` / `Default (Pi)` / `Default (Claude Code)` —
  value `''`, meaning *unset*; label computed from the built-in default.
- Selecting the empty option **deletes** the override key (today's save path keeps
  whatever is set; `providerConfigForSave` at `settings-api.ts:731-739` must handle
  removal).
- Explicit options remain Claude Code / Pi / Codex, now rendered with harness logos.
- The Flywheel pattern already does exactly this (`<option value="">Provider default</option>`,
  `RolesPanel.tsx:517`) — reuse it.

### 4. Settings UI — role-level harness for all roles

Add the harness select (same component as Flywheel's, including the
`"" = Provider default` option) to every role card in `RolesPanel.tsx`: plan, work,
strike, review, test, ship, flywheel. Backend persistence already works via
`saveRoleConfig(role, { harness })`. Placement: next to the existing per-role
ModelPicker in the role card's right column.

The shared `ModelPicker` (`components/shared/ModelPicker/ModelPicker.tsx`) already

cleaner than a separate select, mirror the Command Deck composer's two-panel picker.
UX decision left to planning; requirement is only: every role exposes harness override
with a visible "Provider default" state.

## Behavior Change (call out loudly)

With built-in defaults active, spawns that previously fell back to `claude-code`
change harness when **no explicit/role/provider override** is set:

- OpenAI models (e.g. gpt-5.5): `claude-code` (via CLIProxy) → **codex**.
- Google/Kimi/Z.ai/MiniMax/MiMo/OpenRouter/Nous/DashScope models: `claude-code`
  (via CLIProxy) → **pi**.

Mitigations:

- Explicit settings always win; users who prefer gpt-5.5 under claude-code+CLIProxy
  set `provider_harnesses.openai = claude-code` (or the role override) once.
- Policy gate unchanged: blocked combos still collapse to `claude-code`.
- Conversations are unaffected (harness pinned per conversation).
- Release notes + a one-line activity-log notice on first spawn that used a built-in
  default ("harness codex chosen by provider default — override in Settings →
  Providers") make the change observable instead of silent.

## Acceptance Criteria

1. With an empty `provider_harnesses` and no role overrides: spawning an Anthropic
   model resolves `claude-code`; an OpenAI model resolves `codex`; any other
   provider's model resolves `pi` — verified at plan kickoff, `pan start`,
   `spawnRun(review|test)`, restart, and the dashboard start route.
2. Resolution precedence holds end-to-end: explicit > role > provider override >
   built-in default; unit-tested in one place (the new `resolveHarness` tests), with
   the duplicated inline logic in `spawn-planning-session.ts` deleted.
3. ToS gate untouched: Pi + Anthropic + subscription still collapses to
   `claude-code` regardless of which tier requested pi (existing
   `harness-policy.ts` tests keep passing).
4. Settings → Providers: each provider's harness select shows
   `Default (<built-in>)` when unset, persists explicit overrides, and clearing back
   to default removes the YAML key (round-trip test through
   `saveSettingsApi`/`loadSettingsApi`).
5. Settings → Roles: every role card (plan, work, strike, review, test, ship,
   flywheel) exposes a harness override with a "Provider default" state; saving
   round-trips to `config.roles[<role>].harness`.
6. All ten providers render a real SVG mark (no letter badges for known providers)
   in: composer picker (sidebar, trigger, dots), Settings Providers list, shared
   role ModelPicker. All three harnesses render marks in every harness picker/chip.
7. Pi agents/conversations no longer display the Anthropic starburst anywhere
   (agent dock, conversation feed).
8. `PROVIDER_LABELS`/color/icon maps exist exactly once (the registry);
   `ProviderCard.tsx` dead code is removed or rewired.
9. `specialist_harnesses` no longer functions as an independent fourth tier:
   deprecated-alias or migrated, with its tests updated accordingly.
10. Quality gates green: `npm run typecheck`, `npm run lint`, `npm test`.
