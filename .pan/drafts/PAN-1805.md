## Glossary

- **Harness** — the coding-agent runtime an agent runs under: `claude-code`, `pi`, or `codex`
  (`RuntimeName` in `src/lib/runtimes/types.ts`).
- **Rollout** — codex's session transcript JSONL, written under
  `~/.panopticon/agents/<id>/codex-home/sessions/<y>/<m>/<d>/rollout-*.jsonl`.
- **Thread-id** — codex session identifier; persisted per agent at
  `~/.panopticon/agents/<id>/codex-thread-id` for the fast-path rollout lookup.
- **Subagent / guardian rollout** — a rollout written by a codex-internal subagent thread
  (e.g. the approval supervisor), marked by `session_meta.payload.thread_source === 'subagent'`.
  It shares the codex-home with the user thread and can be newer by mtime.
- **Command Deck conversation view** — the dashboard session panel
  (`src/dashboard/frontend/src/components/CommandDeck/SessionView/SessionPanel.tsx`) fed by
  `src/dashboard/server/routes/command-deck.ts`, which renders the transcript when a
  `jsonlPath` resolves and shows "No conversation data available" otherwise.

## Problem

Codex work agents (real since PAN-1803) showed a blank conversation view. Root cause
(verified in the issue): `resolveJsonlPath` in
`src/dashboard/server/routes/jsonl-resolver.ts` was claude-code-only — it called
`resolveClaudeSessionId`, got null for codex agents, and returned null, so
`command-deck.ts` (`resolveJsonlPath` call at line 326) had no transcript source even
though the rollout JSONL existed on disk. The conversation panel had already solved this
for *conversations* (PAN-1690: `resolveCodexSessionFile` + `findLatestRollout` fallback +
`codex-conversation-parser.ts`), but the work-agent path predated codex work agents.

## Solution (as implemented in `5a0ce548d`)

1. **Resolver-level dispatch** — `resolveJsonlPath` now dispatches on the harness recorded
   in the agent's `state.json` (`resolveAgentHarness`). Codex agents resolve via
   `resolveCodexRolloutPath` (`src/dashboard/server/routes/jsonl-resolver.ts:170-194`):
   persisted `codex-thread-id` fast path (`findRolloutPath`), then `findLatestRollout`
   lazy fallback. The dispatch happens BEFORE claude resolution
   (`jsonl-resolver.ts:211`) so a stale claude session for the same agent id can't shadow
   the rollout. All callers inherit the fix: `command-deck.ts`, `projects.ts`,
   `reviewer-tree.ts`.
2. **Conversations agent-name fallback** — the `/:name/messages` agent-name fallback in
   `src/dashboard/server/routes/conversations.ts` was also claude-only; codex-harness
   agents now resolve there too (`conversations.ts:744-746`), and the previously local
   `resolveCodexSessionFile` was replaced by the shared `resolveCodexRolloutPath`
   (import at `conversations.ts:27`). Parsing uses the existing
   `parseCodexConversationMessages` (`conversations.ts:144`).
3. **Subagent-rollout exclusion** — `findLatestRollout` in `src/lib/runtimes/codex.ts`
   now skips subagent (guardian) rollouts using the `session_meta` `thread_source`
   discriminator (`codex.ts:318-330`), falling back to the newest rollout when ALL are
   subagent threads (`codex.ts:370`) and treating missing `thread_source` (older codex)
   as user threads. `waitForCodexRollout` reuses the same preference so spawn-time
   capture also prefers the user thread.
4. **spawnAgent thread-id capture** — `spawnAgent` (`src/lib/agents.ts:3699`) gained a
   non-blocking post-kickoff background task: `waitForCodexRollout` (120s) →
   `extractThreadIdFromRollout` → `writeThreadId`. Previously only `spawnRun` captured
   thread-ids (exec specialists only, gated at `agents.ts:3304`).
5. **No frontend change required** — SessionPanel already renders jsonl-backed messages;
   live verification confirmed the codex transcript renders end-to-end.

Tests: `src/dashboard/server/routes/__tests__/jsonl-resolver.test.ts` (+118 lines),
`src/lib/runtimes/__tests__/codex.test.ts` (+27 lines). All 51 pass post-main-sync.

## Requirements

- **FR-1:** For a codex-harness agent, `resolveJsonlPath` returns the rollout path — via the
  persisted thread-id when present, else the latest user-thread rollout under
  `codex-home/sessions/` — and returns null when the agent has no `codex-home`.
- **FR-2:** claude-code agents resolve exactly as before, and codex agents never fall
  through to claude-code resolution (no shadowing by a stale claude session file).
- **FR-3:** `GET /api/conversations/:name/messages` resolves codex-harness agents' rollouts
  through the shared `resolveCodexRolloutPath` and returns messages parsed by
  `parseCodexConversationMessages`.
- **FR-4:** `findLatestRollout` and `waitForCodexRollout` prefer user-thread rollouts over
  subagent rollouts; tolerate missing `thread_source`; fall back to newest when all
  rollouts are subagent threads.
- **FR-5:** `spawnAgent` captures the codex thread-id post-kickoff in a non-blocking
  background task; capture failure never fails or delays the spawn.
- **FR-6:** The Command Deck conversation view renders a codex work agent's transcript
  (command-deck payload carries `hasJsonl: true`/`jsonlPath`; no
  "No conversation data available").
- **NFR-1:** Quality gates pass: `npm run typecheck`, `npm run lint`, `npm test` with no
  new failures.
- **NFR-2:** Branch hygiene — the session-scoped `.pan/handoff-1805.md` does not ship to
  main as part of this PR.
