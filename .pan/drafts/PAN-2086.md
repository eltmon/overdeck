# PAN-2086 — finish `overdeck up` speedup & reboot experience

**Issue:** [PAN-2086](https://github.com/eltmon/overdeck/issues/2086)
**Status:** planning → proposed

## Glossary

- **startup sync** — the deferred `pan sync` child spawned at the end of
  `startPostLaunchSidecars` (`src/cli/index.ts:973`) after the dashboard is
  listening. Distinct from a user-invoked `pan sync`.
- **sync inputs** — the on-disk sources `pan sync` renders from: `sync-sources/`
  (skills, agents, rules, hooks, templates, mcp), `~/.overdeck/context/global.md`,
  and each registered project's `.pan/context/project.md`.
- **`[boot-timing]`** — the ISO-stamped boot probes added in `2da7150c2`
  (`module graph loaded at +Xms`, `HTTP server listening at +Yms`) written to
  `~/.overdeck/logs/dashboard.log`.
- **ReadModel bootstrap** — the in-memory projection merge of ~1600 issues +
  ~128 agents that runs before the HTTP server listens
  (`src/dashboard/server/read-model.ts`).
- **Traefik URL** — `https://<traefikDomain>` (default `overdeck.localhost`),
  routed through the `overdeck-traefik` Docker container.

## Problem

Follow-up to the `overdeck up` startup-speed work. Two fixes already landed on
`main`: boot-timing instrumentation (`2da7150c2`) and deferring `pan sync` off
the critical path (`23eae91b0`, 30s+ → ~6s to health). The remaining waste and
UX gaps:

- The deferred `pan sync` still burns ~22s of CPU on every `overdeck up`, almost
  always producing zero changes.
- `overdeck up` runs `docker compose up -d` for Traefik on every start even when
  the container is already healthy (~1–3s for nothing).
- After a cold reboot, `http://localhost:3011` serves well before
  `https://overdeck.localhost`, yet `up` health-gates only on `localhost:3011`
  and then advertises the Traefik URL — a dead-link-then-works experience.
- `overdeck up` opens no window when the Electron app is not installed.
- A reboot reconciles every prior-day conversation at once, logging one
  `[conversation-lifecycle] Session … gone — marking ended` line per
  conversation — a burst of near-identical lines.
- The clean cold-boot path has not yet been measured with the new probes.

## Proposal

Decompose into nine beads across the seven PRD tasks. Tasks 1, 2, 5, 6, 7 are
implementation; Task 3 is measurement that gates the optional Task 4.

### Task 1 — incremental startup sync (primary, FR-1/2/3)

**Profile first, then pick the strategy** (the issue mandates this — do not
guess):

- Instrument the phases in `src/lib/sync.ts` / `syncCommand` with
  `performance.now()` milestones and attribute the ~22s to either the plan/diff
  phase or the write phase.
- **Decision rule** (state the result in the bead's commit/notes):
  - If the **plan/diff phase is cheap** and writing dominates → compute the plan
    and **skip execution when the plan is empty** (inherently correct, no cache
    to drift).
  - If the **plan/diff phase itself is expensive** → gate on a **content-hash
    manifest** of the sync inputs stored at `~/.overdeck/.sync-manifest.json`;
    skip when the combined hash is unchanged.

**Guardrails:** add an explicit `--if-changed` flag used ONLY by the startup
spawn at `src/cli/index.ts:973`. Default `pan sync` and `pan sync --force` always
do a full sync. Hash inputs conservatively; a missed input means stale context —
when in doubt, fall back to a full sync.

### Task 2 — skip Traefik compose when healthy (low effort, FR-4)

Before `execSync('docker compose up -d')` at `src/cli/index.ts:813`, run a cheap
`docker ps --filter name=overdeck-traefik --format '{{.Names}}'` and skip the
compose call when the container is present. Keep current behavior when absent.

### Task 5 — close the cold-reboot Traefik gap (FR-5)

When `traefikEnabled`, health-gate (and the Task 6 auto-open) on the **Traefik
URL** (`https://<domain>/api/health`), not just `localhost:3011`, so "ready"
means the URL handed to the user actually serves 200. Add a Traefik-readiness
poller (extend `platform-lifecycle.ts`). If Traefik readiness is still warming
when the dashboard is up, point the user at the ready `http://localhost:3011` in
the interim rather than advertising a dead link. Surface irreducible
Docker-cold-start delay honestly.

### Task 6 — auto-open the dashboard (FR-6/7)

After the dashboard is confirmed ready (Task 5 readiness), have `overdeck up`:
launch the Electron app if installed (current behavior), **else** open the
default browser via `openBrowser` (as `serve` does), pointed at the ready URL.
Gate behind `--no-open` (**default-on** — matches the "I expected a browser"
feedback). Document how to install the Electron desktop app.

### Task 7 — batch the lifecycle log (FR-8)

In `conversation-lifecycle.ts`, emit a single summary line per poll tick —
`[conversation-lifecycle] marked N conversation(s) ended (sessions gone)` —
using the existing `endedConversations` array, instead of one line per
conversation. Fold the keep-alive-corpse case into the same summary with a
sub-count; keep per-conversation detail only at a debug level. UI status changes
unchanged.

### Task 3 — cold-boot breakdown (measure, FR-9)

After a cold start, capture the boot from `~/.overdeck/logs/dashboard.log`
(per-line ISO stamps + the two `[boot-timing]` anchors), attribute spawn→listen
time (prime suspects: `cache.db` WAL replay, FTS index open, ReadModel merge),
and record a documented breakdown with an explicit **go/no-go** on Task 4.

### Task 4 — listen before the full ReadModel merge (optional, FR-10)

**Conditional on Task 3's go/no-go.** Only if the breakdown attributes a
meaningful (~2s+) share of spawn→listen to the ReadModel merge: serve an
initial/partial snapshot as soon as HTTP is up and hydrate the full merge in the
background, with no change to eventual snapshot contents. If the breakdown says
no-go, mark this bead cancelled referencing the breakdown — do not implement
speculatively.

## Requirements

- **FR-1** A second `overdeck up` with no source changes runs the startup sync in
  well under 2s (logs `[sync] skipped — inputs unchanged` or equivalent).
- **FR-2** `pan sync` and `pan sync --force` always do a full sync (unchanged).
- **FR-3** Editing any sync input (`sync-sources/rules/*.md`, `global.md`, a
  project's `project.md`) causes the next startup sync to re-render that output.
- **FR-4** With Traefik already running, `overdeck up` does not invoke
  `docker compose up -d`, and Traefik still serves `/api/health` 200.
- **FR-5** After a cold reboot, `overdeck up` does not announce/open the Traefik
  URL until it returns 200, or points the user to `localhost:3011` meanwhile.
- **FR-6** `overdeck up` (without `--no-open`) opens the dashboard — Electron when
  installed, browser otherwise — at a URL already serving 200; `--no-open`
  suppresses it.
- **FR-7** Docs describe installing the Electron desktop app so `up` uses it.
- **FR-8** A reboot that ends many conversations logs one summary line (with the
  count) per poll tick, not one per conversation.
- **FR-9** A documented cold-boot breakdown (which phase owns the time) with an
  explicit Task-4 go/no-go decision.
- **FR-10** (Optional, gated by FR-9) spawn→listen drops by roughly the merge
  cost (~3s warm) with no change to eventual snapshot contents.

- **NFR-1** Dashboard server is Node 22 only; run `npm run build` before
  restarting after server-code changes.
- **NFR-2** No `execSync` in dashboard-server-reachable code; async exec/spawn
  only (Task 7 is server code — logging only, no new exec).
- **NFR-3** Reuse the `[boot-timing]` / `dashboard.log` conventions for any new
  timing code.
- **NFR-4** `npm run typecheck`, `npm run lint`, `npm test` all green.
- **NFR-5** Any changed `pan <verb>` flag/help updates `sync-sources/skills/pan-<verb>/SKILL.md`
  in the same commit (lint-skills lockstep).

## NonGoals

- Do NOT change default `pan sync` / `pan sync --force` behavior (always full).
- Do NOT implement Task 4 (defer ReadModel merge) unless Task 3's breakdown gives
  an explicit go; no speculative refactor of the ReadModel bootstrap.
- Do NOT regress snapshot correctness the dashboard relies on.
- Do NOT silence or remove the conversation-lifecycle status changes — only batch
  the logging.
- Do NOT add open-on-`up` without a `--no-open` opt-out.
- Do NOT switch the dashboard server off Node 22 / built `dist/`.
