## Vision

Agents generate mergeable code far faster than a human can review-and-merge it one PR at a
time. The bottleneck moves to the human's last-mile check: UAT. Overdeck's answer is the
**UAT batch train**: at any moment there is **one assembled, testable batch** of every
ready feature, served by a **live frontend**, with a **checklist of what to verify**, and a
**single button that merges exactly the tree the operator tested**. Testing is never put on
hold while new work arrives — the next batch assembles in the background.

The operator's founding complaint, verbatim: *"I looked at the merge queue and I had no
idea what any of it meant, how I would UAT it, or a way to UAT it."* Every surface in this
design answers one of those three.

## The model

| Term | Meaning |
| --- | --- |
| **Queue** | All features that passed review/test (verb `shipping`), in merge order. Reference data, not the primary UI object. |
| **Batch (generation)** | A throwaway branch `uat/<codename>-<MMDD>` off current main, containing **as many ready features as possible, in merge order, with cross-feature conflicts resolved inside the batch** by the assembly agent. |
| **Assembly agent** | An ephemeral agent that builds a batch in a dedicated worktree. Deterministic `git merge`s run in code; the agent is invoked **only when a merge conflicts**, resolves it on the batch branch, and the build continues. Timeboxed; on failure the feature is **held out**. |
| **Held out** | A feature excluded from a generation because its conflict couldn't be resolved confidently. Shown on the card with the reason. It is retried in subsequent generations (e.g. after the conflicting predecessor merges). |
| **Generation chain** | Batches accumulate (`sea-monkey` → `brass-donkey` → `copper-fox`…). A new generation assembles whenever the ready-set changes; the newest **ready** batch is "current", older ready batches remain testable and promotable. Unbounded — the operator may prefer an earlier, smaller batch. |
| **Promote** | THE merge. Merging a tested batch to main lands exactly the tree the operator exercised — conflict resolutions included. Member issues then flow through the normal per-issue post-merge lifecycle (`verifying_on_main`, labels, close-out). |
| **Escape hatch** | Merging a single feature to main directly. Still supported; it invalidates live batches and triggers automatic reassembly. |
| **UAT stack** | A live dashboard stack serving a batch branch at `uat-<codename>-<MMDD>.pan.localhost`, spun on demand from the batch's worktree (existing workspace-stack machinery). Max 2 concurrent; oldest torn down first. |

## Lifecycle walkthrough

1. `PAN-123` and `PAN-124` pass review → assembly agent builds `uat/sea-monkey-0630`
   (123 + 124, one conflict resolved in-branch). Card: **sea-monkey · ready**, with
   "Open UAT frontend", a per-issue "What to UAT" checklist, and **Merge batch (2) to main**.
2. `PAN-125` becomes ready → `uat/brass-donkey-0630` (123 + 124 + 125) assembles in the
   background. Sea-monkey stays fully testable the whole time. When brass-donkey is ready
   it becomes current; sea-monkey is marked superseded but remains testable/promotable.
3. Operator opens brass-donkey's frontend, runs the checklist, clicks
   **Merge batch (3) to main**. Main receives the tested tree in one merge; 123/124/125
   close out through the standard post-merge flow; the chain resets for the next ready set.
4. Alternative: operator promotes sea-monkey instead (125 wasn't wanted yet). 125 is
   reassembled onto new main as the next generation.
5. Alternative: operator merges `PAN-124` alone via the escape hatch. All live batches are
   invalidated; reassembly starts automatically.

## Architecture

### Generation state

New persistent store (SQLite, alongside existing flywheel state): one row per generation —
`name`, `baseSha` (main at assembly), `members[]`, `heldOut[] {issueId, reason}`,
`resolutions[] {issueIds, files}`, `status: assembling | ready | superseded | invalidated |
promoted | failed`, timestamps. The chain is append-only; invalidation/supersession are
status flips, never row deletion (auditable history).

### Assembly

- Trigger: flywheel tick (and queue-affecting events) compares the ready-set against the
  newest generation's members ∪ heldOut. Difference → assemble next generation.
- Build: fresh worktree, branch off current main, merge each feature branch in queue order.
  Clean merge → continue. Conflict → spawn the **assembly agent** scoped to that worktree
  with a single-merge mission (resolve this conflict; change nothing beyond resolution;
  commit with a `uat-assembly:` marker). Agent fails/timeboxes → `git merge --abort`,
  feature held out with reason, build continues.
- Naming: codename per generation (existing `makeUatCandidateName` machinery, seeded per
  generation rather than per day).
- Branch hygiene: `uat/*` branches of promoted/invalidated generations are deleted (local +
  origin) after their stack is torn down; keep the newest K=3 ready/superseded generations.

### Promote

- Precondition: generation `baseSha == origin/main` HEAD. Main moved → generation is stale;
  the button offers "Reassemble first".
- Merge `uat/<name>` into main (no-ff), push. Then for each member issue: invoke the same
  per-issue post-merge lifecycle used today (`verifying_on_main`, label, workspace pause,
  Docker cleanup). The PAN-328 in-flight guard and its lock test must stay green — promote
  fans out post-merge work per issue and must be idempotent per issue.
- The merge-train per-item `merge-next` path remains as the escape hatch and triggers
  invalidation + reassembly.

### UAT stacks

- `POST` on a generation's stack: worktree already exists → render devcontainer
  (`ensureDevcontainerSync`) → compose up (reuse `rebuildWorkspaceStack` internals,
  parameterized by folder/project instead of issue id) → Traefik host
  `uat-<codename>-<MMDD>.pan.localhost` falls out of the folder-name pattern.
- Button behavior: not running → click starts it (~1 min, progress on the button) and opens
  when ready; running → plain link.
- **Hard constraint:** Docker's default address pool fits ~31 networks. Max 2 UAT stacks
  concurrently; starting a third tears down the oldest. Teardown of a generation's stack is
  mandatory on invalidation/promotion. Never let UAT stacks accumulate.

### Card UI (Flywheel rail, replaces MergeQueueCard's internals)

Per the mockup, four zones in one RailCard ("UAT batches"):

1. **Plain-language intro** — "N features passed review & tests. They're assembled into the
   test batches below — open a batch's frontend, run its checklist, then merge that batch
   to main."
2. **Batches, newest first** — per batch: codename, status (ready / assembling with live
   assembly-agent line / superseded / stale), member issue ids, conflict-resolution notes,
   actions: Open UAT frontend · Merge batch (N) to main · rebuild.
3. **What to UAT — current batch** — acceptance criteria grouped per member issue, sourced
   from the shared vBRIEF AC extractor (`src/lib/vbrief/acceptance-criteria.ts`, the same
   source as PAN-1686 — no second parser). Conflict-resolved pairs get an explicit "verify
   the touchpoint" item.
4. **Ready features (merge order)** — reference list: order, issue id, `feature/<id>`
   branch (monospace), PR link, per-feature chips (`✓ resolved in batch`, `⏳ resolving…`,
   `held out: <reason>`).
5. **Escape hatch** — "Merge one feature to main…" (quiet button → picker + `useConfirm()`
   dialog naming exact semantics). The promote button gets its own confirm dialog naming
   the members and stating "lands exactly the tree you tested".

All confirm dialogs via the existing `useConfirm()`/DialogProvider. No `window.confirm`.

### API surface

- `GET /api/flywheel/uat-generations` — the chain (newest first) + per-generation stack
  status. Replaces `GET /uat-candidate` as the card's primary read.
- `POST /api/flywheel/uat-generations/:name/stack` — ensure the live stack (idempotent).
- `POST /api/flywheel/uat-generations/:name/promote` — promote to main.
- `POST /api/flywheel/assemble-uat` — repurposed: force-reassemble now ("Rebuild").
- `POST /api/flywheel/merge-next` — unchanged engine, demoted to escape hatch in the UI.
- `GET /api/flywheel/merge-queue` — keeps the PAN-1737 enrichment (`branchName`, `prUrl`).

## Design decisions (from the operator session)

1. **Honest language everywhere.** No "Ship batch". Buttons name their exact effect;
   consequences are written underneath in sentences.
2. **Assembly is automatic.** No manual gate on having a testable batch. The manual control
   is demoted to "rebuild this generation".
3. **Conflicts are resolved inside the batch, not used to exclude features.** The
   disjoint-only ("batch-safe") model is superseded. Held-out is the fallback, not the rule.
4. **Generations are unbounded**, newest-first, all ready ones testable and promotable.
5. **Promote-the-batch is the primary merge** — "we are testing how those conflicts were
   resolved", so what merges must be what was tested.
6. **Out-of-band single merges invalidate** live batches and trigger reassembly.
7. **UAT stacks on demand, max 2, mandatory teardown** (Docker network pool).
8. **AC checklists come from the shared extractor** — coordinate with PAN-1686, never fork
   the parsing.
9. **PAN-1738 is absorbed** (its core — a stack serving the assembled branch — is zone 2's
   "Open UAT frontend").

## Hazards

- **Docker network pool (~31)** — stack teardown is correctness, not hygiene. Enforce the
  max-2 invariant in code; tear down on invalidation/promotion.
- **postMergeLifecycle idempotency (PAN-328)** — promote fans out per-issue post-merge.
  The in-flight guard test must stay green; promote must not re-trigger itself via
  specialists/done.
- **PAN-1736 verb contract** — the queue is empty unless pipeline items have
  `verb === 'shipping'`. All new surfaces must degrade gracefully on an empty queue.
  PAN-1736 owns the fix; do not fix it here.
- **Trusting agent conflict resolutions** — every resolution is surfaced in the What-to-UAT
  checklist as a "verify the touchpoint" item; resolutions are committed with
  `uat-assembly:` markers for auditability.
- **Stale generations** — main advancing invalidates `baseSha`; promote must hard-check and
  offer reassembly instead of merging a stale tree.
- **Assembly agent runaway** — timebox per conflict; abort → held-out, never a wedged build.
- **uat/* branch litter on origin** — deletion policy above; the flywheel must not spam
  collaborators' fetches with dead branches.

## Acceptance criteria (operator-visible)

1. With ≥2 ready features, a UAT batch exists without any human action, named
   `uat/<codename>-<MMDD>`, containing every ready feature whose conflicts were resolved
   (or that merged cleanly), in merge order.
2. When a new feature becomes ready, the current batch remains testable (frontend stays up,
   merge button stays active) while the next generation assembles; the card shows the
   assembling generation with a live status line.
3. A conflict between two ready features does not exclude either from the batch when the
   assembly agent can resolve it; the resolution is visible in the checklist ("verify the
   touchpoint"). An unresolvable conflict holds the feature out with a human-readable
   reason on the card.
4. Clicking "Open UAT frontend" on a ready batch yields a browser tab serving that batch's
   exact tree at `uat-<codename>-<MMDD>.pan.localhost`, starting the stack first if needed
   (progress shown). No more than 2 UAT stacks run concurrently — verified by attempting a
   third.
5. "Merge batch (N) to main" shows a confirm dialog naming every member; confirming
   produces a single merge to main whose tree equals the batch tree, and each member issue
   advances through the standard post-merge flow exactly once.
6. Promoting an older (superseded) generation works; features not in it reassemble into a
   fresh generation afterward.
7. Merging a single feature via the escape hatch marks live batches invalidated and a new
   generation assembles without human action.
8. The card never shows "Ship batch"; every action's label and confirm dialog state its
   exact effect; the What-to-UAT checklist renders the acceptance criteria of every member
   issue via the shared extractor, with a graceful "No UAT steps in plan" fallback.
9. Empty queue (including the PAN-1736 empty-verb case) renders the explanatory empty
   state, never an error.
10. Visual verification (isolated Playwright profile) of the three mockup states: steady,
    assembling, promote-confirm.

## Out of scope

- Remote (fly.io) UAT stacks — local Docker only.
- Cross-project batches — a batch belongs to one project.
- Automated browser UAT of the batch (agent-run UAT) — future; this PRD is about giving the
  human the surface.
- The PAN-1736 verb-contract fix (owned there).
- Stacked-PR publication of batch contents to GitHub — the batch is a local test vehicle;
  PRs remain per-feature.

## Implementation guidance (single deliverable — no partial ship)

Suggested build order within the one issue: generation store + assembly engine (code-path
merges, held-out, no agent yet) → assembly-agent conflict resolution → promote path +
invalidation triggers → UAT stack lifecycle → card UI + dialogs → Playwright visual pass.
PRD phases are sequencing guidance, not separate deliverables: PAN-1737 is done when all
ten acceptance criteria pass.
