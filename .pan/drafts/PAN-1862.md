## Glossary

- **Convoy** ‚Äî the four parallel code-review sub-agents: `security`, `correctness`, `performance`, `requirements`. Type `ReviewSubRole` at `src/lib/cloister/review-monitor.ts:22`; the list `REVIEW_SUB_ROLES` at `src/lib/cloister/review-monitor.ts:24-29`.
- **Synthesis / parent** ‚Äî the `review` role session (`roles/review.md`), spawned by `spawnReviewRoleForIssue()`. Today it does **no discovery**; it idles waiting for `REVIEWER_READY`/`REVIEWER_FAILED`/`REVIEWER_TIMEOUT` signals, reads the four reports, and emits one verdict. Spawned at `src/lib/cloister/review-agent.ts:443-460`.
- **Context manifest** ‚Äî the pre-built shared diff/AC object written once per run by `buildReviewContext()` (`src/lib/cloister/review-context.ts`, builder ~`:437`; shape `:70-92`). Contains risk-ranked `changedFiles`, acceptance criteria, diff stat ‚Äî **not** the raw diff or file contents (PAN-1125). `formatTier1Summary()` (`:339-436`) renders the inline summary that goes into each reviewer's spawn prompt.
- **Discovery** ‚Äî (new) the parent reads the file *contents* the convoy will all need (diff hunks + high-risk changed files + key surrounding code), so that context lives in its session history before the fork.
- **Warm cache** ‚Äî Anthropic's prompt cache populated as the parent does discovery. The cache key is a **content hash of the prefix** (system prompt + tool defs + message history) for a given model ‚Äî **not** the session id ‚Äî so a byte-identical copied history resumed under a new session id hits the same cache entry. TTL 5 min (default).
- **Fork** ‚Äî copy a Claude Code session's `.jsonl` to a new session id and relaunch with `claude --resume`. Existing machinery (conversation panel): `copySessionFromCompactBoundaryPromise()` (`src/lib/conversations/summary-fork.ts:627-649`), `reserveSummaryForkSessionPromise()` (`:559-571`, `randomUUID()` + session-file path), `runForkPipeline()` (`src/dashboard/server/routes/conversations.ts:3862-3998`), launch via `spawnConversationSession(..., resume=true)` (`:1406-1656`), `--resume` emitted by `src/lib/launcher-generator.ts:429-433`.
- **Re-review cycle** ‚Äî a review triggered after the work agent commits fixes. Always a new HEAD, so today it hits the **stale-runId** branch (`src/lib/cloister/review-agent.ts:340-365`) which `killAllReviewerSessions()` + full fresh respawn.
- **`remain-on-exit on`** ‚Äî tmux setting (PAN-830) that keeps a reviewer's Claude process alive after it finishes a turn so a later cycle can resume it via `sendKeysAsync`. The `killAllReviewerSessions` doc claims rounds reuse this (`src/lib/cloister/review-agent.ts:511-520`) but the stale-runId kill defeats it on every real re-review ‚Äî see Problem.
- **Per-reviewer verdict** ‚Äî (new) a stored `passed|blocked` per sub-role at a given commit. Today only the **aggregate** `reviewStatus` exists (`src/lib/review-status.ts:42-100`); synthesis collapses four reports into one verdict and nothing records which reviewer blocked.
- **Re-review scope** ‚Äî (new) config governing which reviewers re-run on a re-review cycle: `all` | `changed` | `blockers`.
- **Review mode** ‚Äî (new) `roles.review.mode` ‚àà `{ full, quick, none }` governing what *kind* of review runs on the first cycle. `full` = the four-reviewer convoy with the cache-sharing of this PRD (default). `quick` = a single lightweight reviewer, no convoy/synthesis fan-out, no fork (does not use the cache-sharing path). `none` = skip the AI review entirely (the pre-review verification gate still runs). Orthogonal to **Re-review scope** (which only governs re-run within `full`).
- **Cache miss (this feature)** ‚Äî a forked reviewer's first request returns `cacheRead == 0` when a warm-cache hit was expected (TTL expired, model mismatch, or prefix drift). Cache tokens are already captured per request: `CostEvent.cacheRead` / `CostEvent.cacheWrite` (`src/lib/costs/events.ts:17-44`), populated from `usage.cache_read_input_tokens` / `cache_creation_input_tokens` in `sync-sources/hooks/record-cost-event.ts:187-188`.

## Problem

**P1 ‚Äî No cross-reviewer cache sharing.** `spawnReviewRoleForIssue()` builds the manifest once (good, PAN-1059) but then spawns four reviewers (`src/lib/cloister/review-agent.ts:463-486`) that each open the changed files and surrounding code independently via Read/Grep. The overlapping reads are paid 4√ó at full input price. Because each reviewer is a **separate session with a different prompt prefix** (different `roles/review-<subRole>.md` inlined + sub-role named in the preamble, `buildConvoyPromptPromise` `:89-145`), there is no shared prefix, so Anthropic's cache never bridges them.

**P2 ‚Äî Re-review respawns the whole convoy.** A fixes-commit changes HEAD ‚Üí new `runId` ‚Üí the stale-runId branch fires and calls `killAllReviewerSessions()` then re-spawns all four fresh (`src/lib/cloister/review-agent.ts:354-365`):

```ts
// review-agent.ts:354-365 (verified)
if (!paneDead && !opts.force && !staleRunId) {
  return { success: true, message: `Review already in progress: ${reviewSessionName}` };
}
const reason = opts.force ? 'force-killed for re-review' : paneDead ? 'pane is dead' : 'stale runId';
await Effect.runPromise(killAllReviewerSessions(undefined, opts.issueId).pipe(...));
```

This discards every reviewer's accumulated context and re-runs all four even when only one blocked ‚Äî directly contradicting the `remain-on-exit`/resume design described at `:511-520`.

**P3 ‚Äî Selective re-review is not expressible.** `ReviewStatus` (`src/lib/review-status.ts:42-100`) stores a single aggregate `reviewStatus` and a per-issue `history[]` of `{type: 'review'|'test'|...}` entries ‚Äî **no per-reviewer verdict**. So the system cannot know which reviewer to skip.

**P4 ‚Äî Model split silently defeats caching.** Sub-role models are independently configurable (`RoleSubConfig` `src/lib/config-yaml.ts:339-341`; resolver `resolveModel()` `:1576-1592`) and default to a mix (synthesis/security `workhorse:expensive`, others `workhorse:mid` ‚Äî frontend metadata `RolesPanel.tsx:120-126`). Since the cache is per-model, mismatched models would silently disable fork cache-sharing with no operator signal.

## Requirements

### Functional

- **FR-1 (Discovery)** ‚Äî Before the convoy starts, the parent (synthesis) session performs **shared discovery**: it reads the diff and the file contents the convoy collectively needs (the changed files above the manifest risk threshold, their changed hunks, and the immediately surrounding code), bounded by the existing large-changeset guard in the manifest. The discovery instructions live in `roles/review.md` as an explicit first phase and state **why** (so the forked reviewers reuse this context instead of re-reading).
- **FR-2 (Discovery-ready signal)** ‚Äî When discovery completes and the parent session is quiescent (turn boundary), it signals readiness exactly once (CLI signal, e.g. `pan admin specialists discovery-ready review <id>`, mirroring the PAN-977 launcher-signal pattern), so the server knows the JSONL is safe to copy.
- **FR-3 (Fork into convoy)** ‚Äî On the discovery-ready signal, the server forks the parent's session into four reviewer sessions using a **refactored, reusable fork primitive** (extracted from the conversation-panel machinery ‚Äî see NFR-3), one per sub-role. Each fork is launched with `claude --resume <newSessionId>` and its kickoff message is the `roles/review-<subRole>.md` template plus an instruction that the codebase context is already in history.
- **FR-4 (Parent survives as synthesis)** ‚Äî Forking copies the JSONL; the parent session continues unmodified and resumes its existing role as synthesis ‚Äî waiting for convoy signals, reading reports, emitting the verdict (unchanged from today downstream of the fork).
- **FR-5 (Reviewer prompt explains the WHY)** ‚Äî The convoy kickoff and `roles/review-<subRole>.md` tell the reviewer that a shared discovery pass already loaded the diff and high-risk files into context, why that exists (cost + consistency), and that they should build on it rather than re-running a broad `git diff` or re-reading the same files.
- **FR-6 (Per-reviewer verdicts)** ‚Äî Synthesis records a per-sub-role verdict (`passed` | `blocked`, with the reviewed commit and a findings pointer) into review status, in addition to the existing aggregate `reviewStatus`. Extend `ReviewStatus` (`src/lib/review-status.ts:42-100`) and the `review_status` table (`src/lib/database/schema.ts`).
- **FR-7 (Re-review scope config)** ‚Äî A new config `roles.review.reReviewScope` ‚àà `{ all, changed, blockers }`, default `changed`:
  - `all` ‚Äî re-run all four reviewers every cycle (today's behavior).
  - `changed` (default) ‚Äî re-run reviewers that blocked **plus** any reviewer whose prior-flagged files **or** domain files changed since `reviewedAtCommit`.
  - `blockers` ‚Äî re-run only reviewers that blocked in the prior cycle.
  Add to `RoleConfig` (`src/lib/config-yaml.ts:347-365`), the settings API normalizer (`src/lib/settings-api.ts`), and the loader.
- **FR-8 (Resume, don't respawn)** ‚Äî On a re-review cycle, the in-scope reviewers are **resumed** (their `remain-on-exit on` sessions, fed the new diff + their prior findings + the work agent's response via `sendKeysAsync`) instead of killed + respawned. Out-of-scope reviewers carry their prior `passed` verdict forward without running. Replaces the `killAllReviewerSessions` call in the stale-runId branch (`src/lib/cloister/review-agent.ts:354-365`) for the re-review path.
- **FR-9 (Synthesis re-arm)** ‚Äî On a re-review cycle, synthesis is told which sub-roles are running this cycle and which verdicts are carried forward, and waits for terminal signals only from the in-scope reviewers before re-synthesizing.
- **FR-10 (Model-uniformity banner)** ‚Äî Settings ‚Üí Roles renders a **red banner** when the five review roles (synthesis + the four convoy reviewers) do not all resolve to the same model. The banner names the differing roles and states the ramification: fork cache-sharing is disabled, so each reviewer re-reads the full discovery context at full price (~N√ó more input tokens per review). Implemented in `src/dashboard/frontend/src/components/Settings/RolesPanel.tsx` near the sub-role pickers (`:603-638`).
- **FR-11 (Cache-miss detection)** ‚Äî After the convoy launches, the server detects a fork cache miss: a reviewer's first request shows `cacheRead == 0` (read from its `CostEvent`, `src/lib/costs/events.ts:17-44`) when a warm-cache hit was expected. A proactive secondary heuristic: warn if the gap between discovery-ready and fork launch approaches the 5-minute TTL.
- **FR-12 (Cache-miss notification)** ‚Äî On a detected miss, emit (a) an Awareness/activity-log entry via `emitActivityEntrySync({ source, level: 'warn', message, issueId, link })` (`src/lib/activity-logger.ts:107-121`) and (b) a **desktop notification**. The desktop path requires wiring (App.tsx currently only fires `new Notification()` for AskUserQuestion, `:892-897`) ‚Äî see WI-C2.

### Functional ‚Äî review mode (full / quick / none)

> These options do **not** use the cache-sharing path; they are added now because the same paths and config surface are open. `mode` resolves through the existing `roles.review` merge (global + per-project), default `full`.

- **FR-13 (Review mode config)** ‚Äî A new config `roles.review.mode` ‚àà `{ full, quick, none }`, default `full`. Add to `RoleConfig` (`src/lib/config-yaml.ts:347-365`), the settings-API normalizer (`src/lib/settings-api.ts`), and the loader. All three modes are decided at the single review entry point `spawnReviewRoleForIssue()` (`src/lib/cloister/review-agent.ts:31`, "the only review entry point") so every caller ‚Äî the trigger route (`src/dashboard/server/routes/workspaces.ts:3804`) and the Deacon re-dispatch sites (`src/lib/cloister/deacon.ts:1969`, `:2173`) ‚Äî honors it without per-call-site logic.
- **FR-14 (No Review = `mode: none`)** ‚Äî When `mode === 'none'`, `spawnReviewRoleForIssue()` does **not** spawn any reviewer. It sets `reviewStatus: 'skipped'` (new union value, FR-16), emits an activity-log entry recording the skip, and invokes the same lifecycle advance that an *approved* review triggers (transition toward `test`, or toward `readyForMerge` if test is also skipped). The **pre-review verification gate still runs** (`runVerificationForIssue`, `src/dashboard/server/routes/workspaces.ts:3755`) ‚Äî `none` skips the AI review, never the typecheck/lint/test floor.
- **FR-15 (Quick Review = `mode: quick`)** ‚Äî When `mode === 'quick'`, `spawnReviewRoleForIssue()` spawns **one** reviewer (no convoy fan-out, no separate synthesis agent, no discovery/fork) that performs a single combined pass and signals the verdict directly via `pan admin specialists done review`. Add a `roles/review-quick.md` prompt (a focused correctness+security+requirements pass with the same severity/verdict vocabulary as the convoy). Quick mode is independent of Phases A/C/D ‚Äî it has nothing to fork or cache-share.
- **FR-16 (`reviewStatus: 'skipped'`)** ‚Äî Add `'skipped'` to the `reviewStatus` union (`src/lib/review-status.ts:44`) and update the `readyForMerge` gate (`:282-295`) so a skipped review passes the gate, mirroring the existing `testStatus === 'skipped'` precedent (`:284`, `:449`) and `verificationStatus: 'skipped'` (`:53`).
- **FR-17 (Mode UI)** ‚Äî Settings ‚Üí Roles exposes the review `mode` selector and the `reReviewScope` selector (`RolesPanel.tsx`). The model-uniformity banner (FR-10) renders **only when `mode === 'full'`** ‚Äî `quick`/`none` neither fork nor cache-share, so the banner is irrelevant and must not show.

### Non-functional

- **NFR-1 (Quality preserved)** ‚Äî `reReviewScope: changed` must never skip a reviewer whose relevant files changed. Verdicts carried forward must be invalidated by `reviewedAtCommit` drift on their files.
- **NFR-2 (Graceful degradation)** ‚Äî If forks do not produce cache hits (TTL expired, model mismatch, prefix drift), reviews remain **correct** ‚Äî only the cost optimization is lost. The cache-miss notification fires; behavior degrades to today's independent reads. No review is ever blocked or skipped because of a cache concern.
- **NFR-3 (Proper refactor, no hack)** ‚Äî The fork primitive must be extracted from the conversation-panel path (`conversations.ts`) into a reusable module callable from the role/agent spawn path. No copy-paste of fork logic into `review-agent.ts`. (Repo "No Bandaids" rule.)
- **NFR-4 (Same-model invariant for sharing)** ‚Äî Cache sharing is only claimed when synthesis + all four convoy reviewers resolve to the same model. The banner (FR-10) is the operator surface for this; the system does not silently rewrite models.
- **NFR-5 (Idempotency)** ‚Äî The discovery‚Üífork‚Üíconvoy orchestration must be idempotent under retries and dashboard restarts (the architecture's restart invariant, docs/REVIEW-AGENT-ARCHITECTURE.md ¬ß"Dashboard restart invariant"). A second discovery-ready signal for the same run is a no-op.

## Proposal

Four phases, one issue. Phase A is the first-cycle cache win; Phase D is the re-review win; B and C are the operator-facing guardrails that make A safe and observable.

### Phase A ‚Äî Warm-parent discovery + fork (FR-1‚Ä¶FR-5, NFR-3)

**Orchestration change** in `spawnReviewRoleForIssue()` (`src/lib/cloister/review-agent.ts:443-486`). Today: spawn synthesis, then immediately `Promise.all` spawn four reviewers. Target: spawn synthesis with a discovery prompt; **do not** spawn the convoy yet. The convoy is spawned by a discovery-ready handler that forks the parent.

Before (`src/lib/cloister/review-agent.ts:443-486`, abridged):

```ts
const run = await spawnRun(opts.issueId, 'review', { workspace, prompt /* synthesis idle-wait */, ... });
run.reviewRunId = runId;
const reviewerResults = await Promise.all(REVIEW_SUB_ROLES.map(async (subRole) => {
  return spawnReviewSubRoleForIssue({ issueId, workspace, subRole, runId, ... }); // independent reads
}));
```

After (shape ‚Äî work agent matches surrounding style):

```ts
// PAN-1862: spawn the parent with a discovery-first prompt. The convoy is no
// longer spawned here ‚Äî it is forked from the parent once discovery completes,
// so the four reviewers inherit the warm prompt cache instead of each re-reading.
const run = await spawnRun(opts.issueId, 'review', {
  workspace, prompt: buildReviewRolePrompt({ ...opts, runId, reviewDir, contextManifestPath, tier1Summary, phase: 'discovery' }), ...
});
run.reviewRunId = runId;
await saveAgentState(run);
// Convoy is launched by handleDiscoveryReady() (FR-3) on the discovery-ready signal.
```

New `handleDiscoveryReady(issueId)` (server, in/near `review-agent.ts`):
1. Confirm the parent session is quiescent (turn complete) ‚Äî required for a clean JSONL copy.
2. For each sub-role: `forkReviewerFromParent(parentSessionId, subRole, runId)` using the refactored primitive (NFR-3) ‚Üí copies the parent JSONL to a new session id, launches `claude --resume <id>` with `remain-on-exit on`, delivers the kickoff (template + "context already loaded; review and write to <outputPath>; signal contract unchanged").
3. Reviewer launchers signal synthesis on exit exactly as today (PAN-977). Synthesis (the still-alive parent) collects and emits the verdict (FR-4) unchanged.

**Refactor (NFR-3):** extract `copySessionFromCompactBoundaryPromise` + session-id reservation + resume-launch into a reusable `forkSession({ sourceSessionId, cwd, model, harness, kickoffMessage, remainOnExit })` primitive (new module under `src/lib/conversations/` or `src/lib/`), and refactor `runForkPipeline()` (`conversations.ts:3862-3998`) to call it so the conversation panel and the review convoy share one code path. For the convoy, fork the **full** parent history (not just from the last compact boundary) since discovery is the payload we want cached.

**Cache mechanics (decided here):** the cache key is the content hash of the prefix for a given model, not the session id (Glossary). The parent writes the cache incrementally during discovery; the four forks resume a byte-identical prefix and read it ‚Äî provided (i) same model (Phase B), (ii) within 5-min TTL (Phase C), (iii) the kickoff is appended *after* the cached history (it is, as a new user turn). **Implementation checkpoint:** validate empirically that forked `--resume` reviewers report `cacheRead > 0` on their first request. **Fallback:** if not, NFR-2 holds ‚Äî reviews are still correct, the miss notification fires, and we revisit whether to insert an explicit cache breakpoint or pre-warm differently. This checkpoint does not block merge; it gates whether the savings claim is realized.

### Phase B ‚Äî Model-uniformity banner (FR-10, NFR-4)

In `RolesPanel.tsx`, after rendering the sub-role pickers (`:603-638`), compute the resolved model for synthesis + the four reviewers (reuse the same resolution the backend uses, or read resolved values from the settings payload) and render a red banner when they are not all equal:

```tsx
// PAN-1862: cache-sharing requires one model across the whole review convoy.
const reviewModels = ['synthesis','security','correctness','performance','requirements']
  .map((s) => resolvedReviewModel(s));
const uniform = new Set(reviewModels).size === 1;
{!uniform && (
  <Banner tone="danger">
    Review reviewers use different models ({/* list differing role‚Üímodel */}).
    Prompt-cache sharing across the convoy is <b>disabled</b>: each reviewer re-reads the
    full diff and high-risk files at full price (~{reviewModels.length}√ó the input tokens
    per review). Assign all review roles the same model to enable cache sharing.
  </Banner>
)}
```

Match the dashboard style guide (no decorative color; red = genuine warning) ‚Äî see the `pan-style-guide` skill.

### Phase C ‚Äî TTL miss detection + notification (FR-11, FR-12, NFR-2)

- **Detection (server):** in the convoy monitor (Deacon review-convoy monitor, `src/lib/cloister/deacon.ts` review section; or the orchestration that already tracks reviewer state), read each reviewer's first `CostEvent` and flag `cacheRead == 0`. Record `discoveryReadyAt` and `convoyLaunchedAt` on the run; if `convoyLaunchedAt - discoveryReadyAt` approaches 300 s, flag proactively.
- **Notify (FR-12):**
  - Activity log: `emitActivityEntrySync({ source: 'review', level: 'warn', message: `Review cache miss for ${issueId} ‚Äî convoy did not reuse the warm cache (TTL/model). Reviews proceed at full cost.`, issueId, link })`.
  - Desktop: extend the frontend notifier. Add a notify-eligible field to the activity entry (e.g. `desktop?: boolean`) or a dedicated lightweight event; have `App.tsx` (the existing `new Notification()` site, `:892-897`) fire on it. Reuse the existing permission flow (`:865`).
  - Optional TTS via `emitActivityTtsSync` (`src/lib/activity-logger.ts:153-166`) at info priority ‚Äî leave behind the existing `tts.enabled` gate; do not narrate if disabled.

### Phase D ‚Äî Resumable, selective re-review (FR-6‚Ä¶FR-9, NFR-1)

- **Per-reviewer verdicts (FR-6):** extend `ReviewStatus` with `reviewerVerdicts?: Record<ReviewSubRole, { status: 'passed'|'blocked'; atCommit: string; findingsPath?: string }>` (`src/lib/review-status.ts:42-100`) and the `review_status` table (`src/lib/database/schema.ts`). Synthesis writes them when it reads each report ‚Äî it already reads all four reports, so it has the per-reviewer outcome; persist it instead of only aggregating. The signal path is `pan admin specialists done review` (extend to accept per-reviewer outcomes, or have synthesis write them directly before signaling).
- **Scope resolution (FR-7):** a helper `reviewersToRerun(issueId, scope, changedFilesSinceReviewedCommit, priorVerdicts) ‚Üí ReviewSubRole[]`:
  - `all` ‚Üí all four.
  - `blockers` ‚Üí sub-roles with prior `status: 'blocked'`.
  - `changed` ‚Üí blockers ‚à™ { sub-role : its prior-flagged files OR its domain files ‚àà changedFiles }. "Domain files" = the changed files relevant to that reviewer (e.g. all changed files are in scope for `correctness`; security-sensitive paths for `security`). Keep the mapping simple and documented; default conservative (when unsure, include the reviewer).
- **Resume instead of respawn (FR-8):** in the stale-runId branch (`src/lib/cloister/review-agent.ts:354-365`), for the re-review path: compute the in-scope set, then for each in-scope reviewer whose session is still alive, **resume** it (deliver the new diff + its prior findings + the work agent's response via `sendKeysAsync`, `src/lib/tmux.ts`) rather than `killAllReviewerSessions`. Out-of-scope reviewers keep their carried-forward verdict. Only fall back to fresh spawn (+ optional re-fork) when a reviewer's session is gone. `killAllReviewerSessions` remains for the terminal lifecycle events it already serves (merge/reset/cancel/deep-wipe/abort, `:511-524`).
- **Synthesis re-arm (FR-9):** the re-review kickoff to synthesis lists the in-scope sub-roles and the carried-forward verdicts; synthesis waits for terminal signals only from the in-scope reviewers, then re-synthesizes using fresh + carried verdicts.

### Phase E ‚Äî Review mode: full / quick / none (FR-13‚Ä¶FR-17)

A single branch at the top of `spawnReviewRoleForIssue()` (`src/lib/cloister/review-agent.ts:31`) on `resolveReviewMode(config)`:

```ts
// PAN-1862: decide the review mode at the single entry point so the trigger
// route AND the Deacon re-dispatch sites all honor it.
const mode = resolveReviewMode(cfg); // 'full' | 'quick' | 'none', default 'full'
if (mode === 'none') {
  setReviewStatusSync(opts.issueId, { reviewStatus: 'skipped', reviewNotes: 'Review mode: none' });
  emitActivityEntrySync({ source: 'review', level: 'info', message: `Review skipped (mode=none) for ${opts.issueId}`, issueId: opts.issueId });
  await advanceAfterReviewApproved(opts.issueId); // same lifecycle advance an approved review triggers
  return { success: true, message: 'Review skipped (mode=none)' };
}
if (mode === 'quick') {
  return spawnQuickReviewForIssue(opts); // one reviewer, no convoy/synthesis/fork
}
// mode === 'full' ‚Üí existing convoy (Phases A‚ÄìD)
```

`readyForMerge` gate change for FR-16 (`src/lib/review-status.ts:282-295`):

```ts
// before
const gatesPassed = merged.reviewStatus === 'passed' && (merged.testStatus === 'passed' || merged.testStatus === 'skipped') && ...
// after
const gatesPassed = (merged.reviewStatus === 'passed' || merged.reviewStatus === 'skipped') && (merged.testStatus === 'passed' || merged.testStatus === 'skipped') && ...
```

Verification ordering is unchanged: work done ‚Üí verification gate (`runVerificationForIssue`, `workspaces.ts:3755`) ‚Üí review mode branch. `none` and `quick` both run *after* the verification floor.

## Decisions made in this document (do not re-litigate)

1. **Defaults are not silently downgraded.** Shipping leaves the existing per-sub-role model defaults (`RolesPanel.tsx:120-126`) unchanged. The savings are **opt-in** via aligning models; the banner (FR-10) is the nudge. Rationale: forcing synthesis onto a cheaper model to unlock caching would trade review quality for cost without the operator's consent ‚Äî violates "no quality sacrifice."
2. **The parent is the discovery source AND the synthesis agent.** One session does discovery, is forked, and survives to synthesize. No separate discovery session. Rationale: fewest moving parts; the parent's discovery context is also useful for synthesis.
3. **Fork is a first-cycle optimization; re-review rides on resume.** We do not re-fork on re-review ‚Äî the reviewer sessions already hold the discovery + prior-cycle context, so resuming them (Phase D) is both cheaper and higher-fidelity. Re-fork is only the fallback when a session is gone.
4. **Default re-review scope is `changed`** (safe): blockers + reviewers whose relevant files changed. `strict`/`all` are opt-in.
5. **Same-model requirement spans all five review roles** (synthesis + four reviewers), because the forks inherit the parent/synthesis model's cache.
6. **Cache-hit realization is an implementation checkpoint, not a merge gate** (Phase A). Correctness (NFR-2) does not depend on the cache landing.
7. **Default review mode is `full`; No Review never skips the verification floor.** `mode` defaults to `full` (the convoy). `none` skips only the AI review, not the pre-review typecheck/lint/test gate. `quick`/`none` are deliberately *not* wired to the cache-sharing machinery ‚Äî they have nothing to fork.
8. **`mode` and `reReviewScope` are project-scoped now; per-issue is PAN-1874.** Both resolve through the existing global+per-project `roles.review` merge. Per-issue override is a separate, dependent issue ‚Äî not in PAN-1862.
