# PAN-1834 — Blocked reviewer/sub-agent must surface a needs-you triangle + activity-feed/TTS notification

**Issue:** https://github.com/eltmon/panopticon-cli/issues/1834
**Mode:** `pan plan --auto` (no interactive Q&A; choices recorded in `## Auto Decisions`)
**Flywheel-Run-Id:** RUN-23

## Problem

On 2026-06-13 a convoy sub-reviewer (`agent-pan-1696-review-correctness`, codex/gpt-5.5)
blocked on an interactive gpt-5.5 rate-limit / model-switch modal
(*"Switch to gpt-5.4-mini / Keep current model …"*) waiting for a keypress. The whole
review synthesis stalled, but:

- **No needs-you triangle** appeared on the agent in the dashboard.
- **No activity-feed event / TTS** announced that an agent was blocked on input.

The operator only noticed because synthesis never progressed. An autonomous fleet must
surface "agent X is blocked waiting for input" loudly.

This issue is the **visibility** side. The *signal* side (the modal blocking
`REVIEWER_READY`) is #1830; suppressing the modal entirely on non-interactive role
sessions is #1830 ask 2 — both out of scope here.

## Glossary

- **Awaiting-input detection** — the machinery that decides an agent is blocked waiting
  on the operator. Three contributing detections, merged in
  `computeAgentEnrichmentPromise` (`src/lib/agent-enrichment.ts:442-474`):
  `questionDetection` (JSONL AskUserQuestion), `runtimeDetection`
  (`runtime.json state === 'waiting-on-human'`), and `paneDetection`
  (`detectAwaitingInputForAgent` → `detectAwaitingInputFromPaneSync`).
- **Pane signature** — a regex over a cleaned tmux `capture-pane` snapshot that
  recognizes a specific TUI dialog. Lives in
  `src/lib/agent-input-detection.ts:detectAwaitingInputFromPaneSync`.
- **`AwaitingInputReason`** — union of detection reasons
  (`src/lib/agent-input-detection.ts:5`): `tool_permission | user_question |
  disambiguation | confirmation | planning_done | session_resume | other`.
- **`PendingInputKind`** — the *actionable* unified kinds that drive the needs-you
  triangle and the "Needs you" list. Defined **twice** and kept in sync:
  server `src/lib/agent-enrichment.ts:45` and frontend
  `src/dashboard/frontend/src/lib/pendingInput.ts:12`. Current values:
  `askUserQuestion | permissionRequest | exitPlanMode | enterPlanMode | sessionResume`.
- **Needs-you list / triangle** — driven by `selectPendingInputSubjects`
  (`src/dashboard/frontend/src/lib/store.ts:322`), which is **kind-gated**:
  `kinds.length > 0` (PAN-1591). The fuzzy `hasPendingQuestion` superset bool is
  deliberately ignored there. A pane detection that sets only `hasPendingQuestion`
  (no kind) does **not** appear in the list and raises **no** triangle.
- **Enrichment poller** — `pollOnce` in
  `src/dashboard/server/services/agent-enrichment-service.ts:59`, runs every 10 s, scans
  every live agent, emits `agent.enrichment_changed`. It does **not** emit any
  activity-feed or TTS event today.
- **`hasActiveSpecialist`** — a boolean passed into `computeAgentEnrichment`. Computed at
  three call sites (poller `:142`, read-model `read-model.ts:674`, route
  `routes/agents.ts:765`) as `reviewStatus==='reviewing' || testStatus==='testing' ||
  mergeStatus==='merging'`. When true, `computeAgentEnrichmentPromise` skips **all**
  pending-input population (`agent-enrichment.ts:475` and the `if (!hasActiveSpecialist)`
  block at `:483`).
- **Convoy sub-reviewer** — `agent-<issue>-review-<subRole>` (e.g.
  `agent-pan-1696-review-correctness`), spawned via `spawnRun(issueId,'review',{subRole})`
  in `src/lib/cloister/review-agent.ts:264`. It **does** get a `state.json`
  (`saveAgentState(run)` at `review-agent.ts:281`) → it **is** enumerated and scanned.
- **Inspect agent** — `inspect-<issue>-<beadSlug>`, spawned in
  `src/lib/cloister/inspect-agent.ts:230`. It writes only `runtime.json`
  (`saveAgentRuntimeState`, `inspect-agent.ts:237`) and **never** a `state.json`.

## Root-cause analysis (verified against code at 4aeb458dc)

1. **No pane signature for the rate-limit/model-switch modal.**
   `detectAwaitingInputFromPaneSync` (`agent-input-detection.ts:167`) matches Claude
   permission menus, Codex approval headers, Pi permissions, generic y/n, planning-done,
   and session-resume — but **nothing** for the codex/gpt-5.5 rate-limit modal. Even when
   the sub-reviewer is scanned, the modal yields `null` → no detection → no surface.

2. **Pane detections (other than `session_resume`) produce no actionable kind.**
   In `computeAgentEnrichmentPromise`, only `detection.reason === 'session_resume'` is
   promoted into `pendingInputKinds` (`agent-enrichment.ts:502`). All other pane reasons
   set only the fuzzy `hasPendingQuestion` bool, which the needs-you list ignores
   (PAN-1591). So a new modal reason must **also** be promoted to a new kind, or the
   triangle/list still won't fire.

3. **`hasActiveSpecialist` suppresses the active specialist's own pending-input.**
   During the convoy, `PAN-1696`'s `reviewStatus === 'reviewing'` → `hasActiveSpecialist`
   is `true` for **every** agent on that issue, including the sub-reviewers themselves.
   `computeAgentEnrichmentPromise` then skips all `pendingInputKinds` population
   (`agent-enrichment.ts:483`) for the sub-reviewer → no kind → no triangle. The guard's
   intent was to silence the *parked* work/plan agent during review, not the active
   specialist. (Fixing inside `computeAgentEnrichmentPromise` corrects all three call
   sites at once.)

4. **Inspect agents are never enumerated.** `listRunningAgents`
   (`agents.ts:3858`/`3890`) enumerates only dirs with a readable `state.json`
   (`getAgentStateSync`, `agents.ts:1041`). Inspect agents write only `runtime.json`, so
   the poller never sees them → they cannot surface a needs-you state at all.
   The deacon's patrols filter on `agent-`/`specialist-`/`planning-` prefixes
   (`deacon.ts:978,2108,3908,4332,5430`) and **never** match `inspect-`, so giving an
   inspect agent a minimal `state.json` does **not** make the deacon auto-resume it.

5. **The poller emits no activity-feed / TTS event on awaiting-input.** `pollOnce` emits
   only `agent.enrichment_changed`. `emitActivityEntrySync` / `emitActivityTtsSync`
   (`src/lib/activity-logger.ts:107,153`) exist and are used elsewhere
   (work-agent start, review-status transitions) but never for an awaiting-input edge.

## Requirements

Functional:

- **FR-1** Detect the harness rate-limit / model-switch modal as an awaiting-input state
  via a conservative pane-content signature, returning a new
  `AwaitingInputReason: 'rate_limit'`.
- **FR-2** Surface a `rate_limit` pane detection as a new `PendingInputKind: 'rateLimit'`
  in both the server and frontend kind unions, with a human label, and promote it into
  `pendingInputKinds` so the needs-you triangle + "Needs you" list fire.
- **FR-3** A convoy sub-reviewer blocked on input surfaces needs-input even while its
  issue is `reviewing` — i.e. do not apply the `hasActiveSpecialist` suppression to an
  agent that is itself the active specialist.
- **FR-4** Inspect agents are enumerable by the enrichment poller so they can surface
  needs-input the same way sub-reviewers do.
- **FR-5** When an agent transitions into awaiting-input (rising edge of
  `pendingInputCount`), emit one `activity.entry` naming the agent, its issue, and what
  it is waiting for.
- **FR-6** Emit one `activity.tts` utterance for the same rising edge so a blocked agent
  is announced by the narrator.

Non-functional:

- **NFR-1** No new false-positive needs-you rows: the rate-limit signature requires the
  modal's distinctive option phrasing **and** a recent/current prompt index (mirroring
  the existing detections), so an answered/scrolled-away modal does not re-fire.
- **NFR-2** Pane detection reuses the existing `detectAwaitingInputForAgent` path
  (cache + concurrency cap) — no new tmux capture cadence.
- **NFR-3** The activity/TTS notification fires once per rising edge, not on every poll
  while the agent stays blocked.

## Auto Decisions

- **AD-1 — Fix `hasActiveSpecialist` suppression inside `computeAgentEnrichmentPromise`,
  not at the three call sites.** Centralizing keeps the three sites consistent and avoids
  drift; the function already knows the agent's id/role. An agent is treated as its own
  active specialist when its derived `role ∈ {review, test, ship}`. Rationale: sub-reviewers
  and the synthesis agent both carry `state.role === 'review'`; the parked work/plan agent
  (`role === 'work'/'plan'`) keeps existing suppression.
- **AD-2 — Inspect agents become enumerable by writing a minimal `state.json` at spawn**
  (mirroring `review-agent.ts:281`), rather than a bespoke poller-only enumeration.
  Rationale: it is the root-cause fix (the gap is "no state.json"), reuses every existing
  enrichment path (JSONL + pane + runtime), and is deacon-safe because deacon patrols
  ignore the `inspect-` prefix. Accepted consequence: inspect agents will also appear in
  other `listRunningAgents` consumers (e.g. the session tree) — desirable for the issue's
  visibility goal.
- **AD-3 — The activity/TTS notification fires for ANY awaiting-input kind, not only the
  rate-limit modal.** Rationale: the underlying gap (no activity/TTS on awaiting-input) is
  general; a kind-agnostic rising-edge emitter fixes the whole class with one change and
  covers the observed incident.
- **AD-4 — TTS priority = 1 (warn).** A blocked agent is actionable and time-sensitive but
  not an error; warn is prominent without interrupting error-priority speech.
- **AD-5 — Rate-limit signature keys on the option phrasing.** Match requires both a
  "Keep current model" option line and a "Switch to <model>" option line near the bottom
  of the pane. Rationale: this two-option pairing is specific to the model-switch modal
  and avoids matching prose that merely mentions rate limits. Implementation checkpoint:
  if a captured real pane shows different exact wording, widen the regex to the captured
  text and keep the two-option-pairing + recent-index guard. Fallback if no live capture
  is available: ship the phrasing above (taken verbatim from the issue) with a fixture
  test, since the modal is intermittent.
