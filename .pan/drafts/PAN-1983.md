## 0. Glossary (define before use)

- **panopticon.db** — the legacy SQLite cache at `~/.overdeck/panopticon.db`, opened by
  `getDatabase()` in `src/lib/database/index.ts` (grep anchor:
  `return join(getOverdeckHome(), 'panopticon.db');`). Superseded by overdeck.db.
- **overdeck.db** — the current runtime cache, opened by `getOverdeckDatabaseSync()` in
  `src/lib/overdeck/infra.ts`; schema = `drizzle/overdeck/0000_overdeck_init.sql`
  (30 `CREATE TABLE`s), which runs ONLY on a fresh database (guard: the `agents` table
  existing skips it — grep anchor: `WHERE type = 'table' AND name = 'agents'`).
- **Runtime top-up (PAN-2220 pattern)** — idempotent DDL applied on every open for
  objects added after a DB was created: `ensureRuntimeIndexesSync` in
  `src/lib/overdeck/infra.ts` (grep anchor: `CREATE INDEX IF NOT EXISTS
  \`cost_session_id_idx\``). New tables for existing DBs use this same mechanism.
- **Legacy cluster** — the dead store modules in `src/lib/database/` (agents-db,
  merge-queue-db, schema.ts, index.ts, …) slated for deletion in PAN-1984.
- **`driver.ts`** — `src/lib/database/driver.ts`, the GENERIC `openDatabase(path)`
  SQLite driver. Explicitly KEPT (issue body); used by 14 production files including
  the overdeck layer itself, cache-service, memory FTS, and the embeddings DB. Not a
  panopticon.db dependency.
- **Legacy-import feature** — `src/lib/overdeck/legacy-import.ts` (312 lines) + routes
  `GET/POST /api/settings/legacy-import/conversations` in
  `src/dashboard/server/routes/settings.ts` + Settings UI (`LegacyImportDialog.tsx`,
  `ExperimentalSection.tsx`). Operator-triggered, one-time, READ-ONLY import of
  conversations from an explicitly chosen legacy .db file
  (`openDatabase(path, { readOnly: true })`) into the overdeck conversations door.
  Added after this issue was filed.
- **Boundary gate** — `scripts/lint-overdeck-boundaries.sh`, run in `npm run lint` as
  `lint:overdeck-boundaries` with `PAN_OVERDECK_BOUNDARY_LINT=1`. Fails when a consumer
  reaches the old DB — but structurally exempts `src/lib/overdeck/**` AND
  `src/lib/database/**` (grep anchor: `':!src/lib/overdeck/**'`).
- **Door** — a module under `src/lib/overdeck/` owning one domain's canonical
  read/write path (two-door tenet).

## 1. Problem (verified evidence at `506756378f`)

Sweeping every real import of a legacy-cluster module across `src/` and `scripts/`
(`grep -rEn "from '[^']*database/<module>" src/ scripts/` for all 18+1 modules) finds
exactly ONE production import left:

1. **`src/lib/overdeck/backlog.ts` is a one-line legacy leak through a door.** Its
   entire content:
   ```ts
   export { getBacklogSequenceForRoot, clearBacklogSequence } from '../database/backlog-sequence-db.js';
   ```
   `backlog-sequence-db.ts` calls `getDatabase()` — panopticon.db. Consumer:
   `src/dashboard/server/routes/backlog.ts` (grep anchors:
   `getBacklogSequenceForRoot(projectRoot)` and `clearBacklogSequence(parsed.doc.project)`),
   which imports them **from the door** (`from '../../../lib/overdeck/backlog.js'`). So
   the dashboard's backlog sequence cache still reads/writes panopticon.db today, while
   every status surface says the migration is done. The `backlog_sequence` table does
   NOT exist in `drizzle/overdeck/0000_overdeck_init.sql` (verified: `grep -n backlog`
   → no match) — the door was stubbed as a re-export instead of migrated.
2. **Why the gate missed it:** the boundary gate exempts `src/lib/overdeck/**` (the
   doors themselves) and `src/lib/database/**` (the old layer) — a door that re-exports
   the old layer is invisible to it. This is the structural hole to close, not just the
   one leak (fix the system, not the symptom).
3. **The legacy-import feature post-dates the issue** and is the one deliberate reader
   of legacy .db files. Whether it survives is an operator decision (§3 D3) — the
   issue's "delete all code that talks to panopticon.db" was written before this
   feature existed.
4. **Actively misleading references:** `src/lib/costs/sync-wal.ts` header claims "import
   cost events into panopticon.db" (grep anchor: `import cost events into panopticon.db`)
   while the code writes through `CostDoorLive`/`CostWriter` from `overdeck/cost.js`
   (verified import list); `src/lib/paths.ts` documents `getLegacyHome()` as "Used ONLY
   by the opt-in \`--seed-from-legacy\` importer" (grep anchor: `--seed-from-legacy`) —
   that importer was deleted; the actual caller today is the settings legacy-import
   route (grep anchor in `routes/settings.ts`: `join(getLegacyHome(), 'panopticon.db')`).

Out of the issue's original four scope bullets: db↔db migration code — GONE (verified:
no `cutover.ts`, no `conversations-export.ts`); seed-from-legacy — GONE (no
`--seed-from-legacy` in `src/cli/`); recover-costs scripts — GONE from `scripts/`
(stale exemption lines for them remain in the boundary gate's EXCLUDES — cleanup in
WI-4); legacy SQLite layer — deleted in PAN-1984 after the test tail.

(verified by `grep -n "export function" src/lib/database/backlog-sequence-db.ts` and a

|---|---|---|

`sequence.md` (grep anchor: `join(projectRoot, '.pan', 'backlog', 'sequence.md')`) and

which moves with the code (WI-2).

Consumer route behavior (`routes/backlog.ts`) is untouched — same door import path,
same functions, same shapes.

## 3. Locked design decisions

- **D1 — `backlog_sequence` moves to overdeck.db via BOTH schema paths.** Add the table
  + index to `drizzle/overdeck/0000_overdeck_init.sql` (fresh DBs) AND to a new
  `ensureRuntimeTablesSync(db)` in `src/lib/overdeck/infra.ts` called next to
  `ensureRuntimeIndexesSync` (existing DBs — the init migration is skipped once
  `agents` exists, so a migration-file-only change would silently never run on every
  live machine). DDL is the legacy one verbatim (grep anchor in
  `src/lib/database/schema.ts`: `CREATE TABLE IF NOT EXISTS backlog_sequence`) — same
  columns, same `UNIQUE(project_key, issue_id)`, same
  `idx_backlog_sequence_project_rank` index.
- **D2 — NO data migration, by design.** The table is a disposable cache of
  `sequence.md` (§2); on first read after cutover, `getBacklogSequenceForRoot`
  repopulates it from the file. Copying rows from panopticon.db would recreate exactly
  the db↔db migration this issue exists to abolish ("There was never supposed to be any
  DB↔DB import/export migration"). The old rows in panopticon.db are simply abandoned
  (the FILE is kept, per the issue).
- **D3 — Legacy-import is the ONE sanctioned legacy reader — operator checkpoint.**
  **Implementation checkpoint:** ask the operator whether any machine still needs the
  Settings → legacy-import path. **Default (no answer / yes-needed): KEEP** — it is
  read-only, explicit-path, driver-based (no `getDatabase()`, no cluster imports), and
  it imports INTO the overdeck doors; document it in the boundary gate's header comment
  as the sanctioned exception. If the operator confirms deletion: remove
  `src/lib/overdeck/legacy-import.ts`, its two tests
  (`src/lib/overdeck/__tests__/legacy-import.test.ts`, `legacy-import-write.test.ts`),
  the two `/api/settings/legacy-import/conversations` routes in `routes/settings.ts`,
  the `LegacyImportDialog.tsx` component + its test + its `ExperimentalSection.tsx`
  wiring, `tests/dashboard/settings-legacy-import.test.ts`, and `getLegacyHome()` in
  `src/lib/paths.ts` — as ONE later commit, so the keep-path PR is not blocked on the
  answer.
- **D4 — Close the structural hole in the boundary gate.** Add a check to
  `scripts/lint-overdeck-boundaries.sh`: no file under `src/lib/overdeck/**` may import
  from `src/lib/database/` EXCEPT `driver.js` (the generic driver used by `infra.ts`
  and `legacy-import.ts`). This makes a door-level re-export of a legacy store a lint
  failure — the class of leak in §1, not just the instance. Follow the script's
  existing check style (numbered sections, `comment_filter`, count-and-fail). Also
  drop the stale `recover-costs*.mjs` EXCLUDES lines (the files no longer exist).
- **D5 — Comment hygiene, misleading-only.** Fix exactly two doc surfaces: the
  `sync-wal.ts` header (says panopticon.db; writes overdeck.db) and the
  `getLegacyHome()` doc in `paths.ts` (names a deleted feature; point it at the
  settings legacy-import route instead). Historical "Drop-in for database/X" notes in
  the overdeck doors stay — they are accurate provenance, and PAN-1984 deletes their
  referents anyway (surgical-change rule).
- **D6 — Sequencing with PAN-1984.** This PRD deletes ONLY `backlog-sequence-db.ts`
  (after its replacement lands + test moves). The other legacy modules and the whole
  test tail are PAN-1984's — do not pull them forward; a partial pull-forward reds the
  typecheck gate (tests are included in `npm run typecheck`).

## 6. Intersecting repo rules (restated — do not assume recall)

- **No DB↔DB migration, ever** (this issue's founding rule; D2). The DB is a
  disposable cache rebuilt from git/files — never state that as a "delete X and
  observe" test.
- **Never delete .db FILES** — code only. Deep-wipe and destructive endpoints are
  never invoked in verification.
- **Two-door tenet** — the fix direction is always "make the door real", never "add
  another consumer exemption to the gate".
- **No `execSync` in server-reachable code; async tmux only** (no tmux here, but the
  door is dashboard-reachable — keep it dependency-light).
- **Fake timers** for any retry/delay test (none expected in this scope; restated for
  the moved test if it gains waits).
- **commitlint**: all-lowercase, scoped (suggested scope `db` for WI-1/2/3, `infra`
  for WI-4/5, `dashboard` for WI-6's delete branch); every commit references
  `pan-1983`. Never `--no-verify`; never `git stash`; feature workspace, never the
  primary worktree.
