## 1. Problem

Every work agent today binds to one fixed model — `roles.work.model`, defaulting to
`workhorse:mid` → `claude-sonnet-4-6` ([`config-yaml.ts:389`](../../src/lib/config-yaml.ts),
[`:381`](../../src/lib/config-yaml.ts)). A one-line typo fix and a 40-file architectural change
draw the *same* model. That wastes tokens on trivial work and under-powers hard work. We want to
route each issue, after planning, to the **cheapest model capable enough to do the job** —
reducing token cost without quality loss.

A per-bead `metadata.difficulty` field already exists on every plan item
([`vbrief/types.ts:67`](../../src/lib/vbrief/types.ts), values at
[`:29`](../../src/lib/vbrief/types.ts) — `trivial|simple|medium|complex|expert`), but it is **only
turned into a `difficulty:<level>` bead label** today ([`vbrief/beads.ts:345`](../../src/lib/vbrief/beads.ts))
and is **never read by the spawn path**. There is **no difficulty→model rubric anywhere** in live
code (verified by grep, 2026-06-13). This PRD builds that missing rubric.

## 2. Glossary

- **Difficulty** — a property of the *work* (`trivial|simple|medium|complex|expert`), stamped by
  the planning agent onto the spec. NEW in this PRD: an **issue-level** difficulty at
  `plan.metadata.difficulty`, in addition to the existing per-item field.
- **Capability** — a measured property of a *model*: how well it resolves real coding tasks.
  Sourced from external SWE benchmarks (see *Eval source*). Expressed on a 0–100 scale.
- **Capability floor** — the minimum capability a model must have to be eligible for a given
  difficulty. The output of the `difficultyTiers` map.
- **Cost** — a model property: `costPer1MTokens` ([`model-capabilities.ts`](../../src/lib/model-capabilities.ts)
  `ModelCapability`), or DeepSWE's per-task cost. The quantity we *minimize* subject to the floor.
- **Eval source** — the external benchmark we borrow capability scores from: **DeepSWE**
  (https://deepswe.datacurve.ai/, canonical) → **SWE-bench Pro**
  (https://labs.scale.com/leaderboard/swe_bench_pro_public, per-model fallback) → **operator
  override** (final).
- **Basis expression** — how a model's score is *stored*: never a bare number, always an
  expression that resolves deterministically to one (e.g. `between:kimi-k2.6,gpt-5.5`). See FR-1.
- **Workhorse slot** — the existing 3-rung tier abstraction (`cheap|mid|expensive`) at
  [`config-yaml.ts:327`](../../src/lib/config-yaml.ts), today hardcoded to Anthropic models at
  [`:381`](../../src/lib/config-yaml.ts). This PRD **replaces** that hardcoded map with
  data-driven, capability-floor band membership.
- **Decision record** — a structured log of one model-selection event (floor used, candidates
  considered, model picked, basis/provenance), keyed so a later eval effort can join it to the
  run's gate outcome.
- **Stakes** — a per-issue override of the whole tiering: `critical` (force top model + full
  workflow) or `express` (hotfix lane: minimal friction to prod).
- **Convoy** — the parallel review reviewers: `security|correctness|performance|requirements`
  ([`cloister/review-monitor.ts:22`](../../src/lib/cloister/review-monitor.ts) *(verify)*).
- **Risk floor** — review reviewers that must run regardless of change size when a high-risk file
  is touched (auth/crypto/payments/migrations/infra).

## 3. The two-axis model (the design's spine)

**Capability is a constraint; cost is the objective.** Selection is one rule:

```
floor   = difficultyTiers[ plan.metadata.difficulty ].minCapability
eligible = { m ∈ enabledModels : capability(m) ≥ floor }
pick    = cheapest(eligible)          # PAN-1832 weighted distribution may sample within eligible
```

Consequences that resolve the cost-vs-difficulty conflation:

- A "tier" is a **capability band**, not a price band. `frontier` = highest capability; it is
  expensive *today* only because that is what top capability costs now.
- A cheap model that scores high lands in a high band automatically — band membership is **never**
  inferred from provider or version (FR-5).
- The hardcoded `cheap→haiku / mid→sonnet / expensive→opus` map
  ([`config-yaml.ts:381`](../../src/lib/config-yaml.ts)) is **deleted**; membership is derived from
  eval scores.

## 4. Requirements

### Functional

- **FR-1 — Basis-expression scoring.** A model's capability is stored as a `basis` expression that
  resolves deterministically to a 0–100 score. Grammar: `absolute:N`, `deepswe:pass@1=N`,
  `swebench-pro=N`, `between:A,B` (midpoint of anchors' resolved scores), `like:X`, `delta:X,±N`.
  References to other models resolve topologically with cycle detection.
- **FR-2 — Three-source resolution.** DeepSWE is canonical; SWE-bench Pro fills per-model gaps only
  and is **tagged as a different scale** (never arithmetically blended with a DeepSWE number);
  operator override wins over both.
- **FR-3 — Heavy-refresh dev skill.** A Panopticon dev skill drives the current top model to ingest
  both leaderboards, reconcile their model names to our `ModelId`s, propose a `basis` per model with
  provenance, and flag models on no leaderboard for operator anchoring.
- **FR-4 — Targeted operator update.** An operator can add/edit a single model's score via a
  relative basis (e.g. `between:A,B`) without running the heavy skill or re-scraping.
- **FR-5 — No version inference.** Every effective score MUST trace to a benchmark row or an
  explicit operator basis. No code may infer capability from a version string or provider.
- **FR-6 — Planning stamps issue difficulty.** On planning completion, `plan.metadata.difficulty`
  is written to the spec (derived or planner-assigned; see Decision D-2).
- **FR-7 — Capability-floor work resolver.** Work-agent model = cheapest enabled model clearing
  `difficultyTiers[difficulty].minCapability`, implemented as a new dimension on the PAN-1832
  spawn-time resolver seam (NOT the deterministic static `resolveModel`).
- **FR-8 — Availability gating.** Selection considers only **enabled and available** models. A
  pulled model (e.g. `claude-fable-5`, still in the catalog at
  [`model-capabilities.ts:178`](../../src/lib/model-capabilities.ts) but withdrawn by Anthropic)
  MUST be excludable so it never gets picked, and "most advanced available" for planning skips it.
- **FR-9 — Decision record.** Each selection emits a record (difficulty, floor, candidate set,
  chosen model, basis/provenance) under a key joinable to the run's `CostEntry`
  ([`cost.ts`](../../src/lib/cost.ts), already carries issue/agent IDs).
- **FR-10 — Escalation on failure.** If a work agent fails the verification gate (PAN-174) its
  configured number of times, it re-spawns one capability band higher. This is the safety net that
  makes cheapest-above-floor acceptable.
- **FR-11 — Stakes override.** Per-issue `stakes: critical` forces top model + full convoy + full
  planning and disables down-tiering; `stakes: express` selects the hotfix lane (FR-12). A global
  default **aggressiveness** lever in Settings chooses between "cheapest qualifier" and "cheapest in
  the top half of qualifiers".
- **FR-12 — Express hotfix lane.** `express` issues skip heavy planning (reuse the existing
  `pan start --auto` minimal-vBRIEF path), select the cheapest work band, and get a light review
  (FR-13 minimum) — the lowest-friction path to production.
- **FR-13 — Review composition rubric.** The number of review reviewers scales with change
  size/risk computed from the already-built review context manifest
  ([`cloister/review-context.ts`](../../src/lib/cloister/review-context.ts): `diff.stat`,
  `changedFiles[].additions/deletions`, per-file `riskScore`): tiny+low-risk → one general
  reviewer; large → today's full convoy. **Risk floor:** any high-risk file forces the `security`
  reviewer regardless of size.
- **FR-14 — Synthesis adapts to the spawned subset.** The synthesis agent and review-monitor signal
  accounting MUST wait only for the reviewers actually spawned — not the hardcoded four
  ([`roles/review.md:74`](../../roles/review.md) currently says "wait for all four").
- **FR-15 — `difficultyTiers` config replaces the hardcoded workhorse→Anthropic map.** New config
  keys (`difficultyTiers`, `stakes`, aggressiveness) added; scalar `model:` refs remain valid
  (back-compat).

### Non-functional

- **NFR-1 — Determinism.** No `Math.random`/`Date.now` in resolution. Within-band sampling reuses
  PAN-1832's agent-id-seeded picker. Basis resolution is pure.
- **NFR-2 — Offline eval data.** Capability scores come from a cached artifact
  (`model-evals.json`), refreshed by the skill/script — **never** fetched at spawn time.
- **NFR-3 — Back-compat.** Existing scalar and `workhorse:` refs keep working.
- **NFR-4 — Effort-ready.** The capability/score data model accepts an optional `effort` dimension
  now (v2 `(model,effort)` co-optimization per PAN-1142), even though v1 fixes effort per band.
- **NFR-5 — Provenance.** Every score record carries `source` (deepswe|swebench-pro|operator),
  `asOf`, and an optional human `note`.
- **NFR-6 — Repo rules.** No `execSync` in server-reachable code (use `execAsync`/`spawn`); async
  tmux primitives only; retry/backoff tests use fake timers; no `--model` passed at spawn unless an
  override is explicitly set.

## 5. Architecture & key changes

### 5.1 Eval data store + resolver (FR-1, FR-2, FR-5, NFR-2, NFR-5)

New artifact `model-evals.json` (checked in) + `model-evals.overrides.json` (operator-owned).
Shape:

```jsonc
"gpt-5.5":        { "basis": "deepswe:pass@1=70", "effort": "xhigh", "source": "deepswe", "asOf": "2026-06-13" },
"claude-opus-4-8":{ "basis": "deepswe:pass@1=58", "effort": "max",   "source": "deepswe", "asOf": "2026-06-13" },
"gpt-5.4":        { "basis": "swebench-pro=59",   "source": "swebench-pro", "note": "fallback scale" },
"kimi-k2.7-code": { "basis": "between:kimi-k2.6,gpt-5.5", "source": "operator",
                    "note": "operator 2026-06-13: midway Kimi 2.6 ↔ GPT-5.5" }
```

New lib (e.g. `src/lib/model-evals.ts`): `resolveCapability(modelId): { score, source, asOf }`,
implementing the basis grammar with topological deref + cycle detection. Seeds/augments — or
replaces — the hand-authored `skills['code-generation']` numbers currently in
[`model-capabilities.ts`](../../src/lib/model-capabilities.ts) (whose provenance is a stray
`vals.ai` code comment at [`:13`](../../src/lib/model-capabilities.ts), almost certainly
agent-authored — replacing it with provenance-tracked data is a strict improvement).

### 5.2 Name aliasing + ingestion (FR-3, FR-4)

- `model-evals.aliases.json`: `{ "claude-opus-4.8": "claude-opus-4-8", "gpt-5.5 [xhigh]": "gpt-5.5", … }`.
- **Dev skill** `skills/pan-model-eval/SKILL.md` (or similar): top-model-driven heavy refresh.
  **Implementation checkpoint:** both leaderboards are HTML-only (no API, confirmed 2026-06-13), so
  ingestion is fetch-and-parse (or operator-paste); the skill must tolerate layout change and fail
  loud rather than silently mis-score. Fallback: operator pastes the table.
- **Targeted update:** a thin path — either a `pan models score <id> --between A,B --note "…"` verb
  (if added, its `/pan-models` wrapper skill changes in the same commit per the skills convention)
  or a documented one-line edit to `model-evals.overrides.json` validated on config load.

### 5.3 Capability-floor work resolver (FR-7, FR-8, FR-9, FR-15)

Extends the **PAN-1832 spawn-time resolver** (NOT the static deterministic `resolveModel` at
[`config-yaml.ts:1576`](../../src/lib/config-yaml.ts), which several callers require to stay
single-valued — see PAN-1832 draft §"Static callers"). New spawn-only function, conceptually:

```ts
// inputs: role='work', difficulty, config, agentId (seed), enabled+available model set
const floor = difficultyTiers[difficulty].minCapability;
const eligible = enabledAvailableModels.filter(m => resolveCapability(m).score >= floor);
const pick = aggressiveness === 'conservative'
  ? cheapestInTopHalf(eligible)
  : cheapest(eligible);                 // default
// if roles.work.model is a PAN-1832 distribution, restrict it to `eligible`, then sample by agentId
emitDecisionRecord({ issueId, agentId, difficulty, floor, candidates: eligible, pick, basis });
return pick;
```

Wired into [`determineModel`](../../src/lib/agents.ts) *(≈:2676, verify)* via the already-present
but unused `SpawnOptions.difficulty` field *(≈:2564, verify)*, and into `spawnAgent`
*(≈:3444, verify)* / `spawnRun` *(≈:3131, verify)*. `getCheapestModelsSync()`
([`model-capabilities.ts:1161`](../../src/lib/model-capabilities.ts)) is a reusable building block.

### 5.4 Planning (FR-6, planning pin)

- On planning completion (`complete-planning` flow), write `plan.metadata.difficulty`.
- Planning's OWN model is **never** floor-selected — it resolves to the **most advanced available**
  model (top of the eval ranking, after availability gating per FR-8).
  [`planning/spawn-planning-session.ts:556`](../../src/lib/planning/spawn-planning-session.ts)
  currently does `resolveModel('plan', …)`; replace the "expensive workhorse" default with
  "top available by capability".

### 5.5 Escalation on failure (FR-10)

Hook the verification-gate retry loop (PAN-174; feedback sent to tmux, completion marker withheld,
bypass after 3). After the configured failure count, re-spawn the work agent at the next band up
(reuse the `WORK_AGENT_BROKEN_MODELS` safety-net pattern in `determineModel`). Record the
escalation in the decision record.

### 5.6 Stakes + express lane (FR-11, FR-12)

- Per-issue `stakes` (tracker label or review-status field): `critical | express | (default)`.
- `critical`: top model, no down-tiering, full convoy, full planning.
- `express`: `pan start --auto` minimal plan + cheapest work band + FR-13 light review.
- Settings: global `aggressiveness: cheapest | conservative` default.

### 5.7 Review composition rubric (FR-13, FR-14)

- Compute size/risk from the manifest **before** the convoy spawns (manifest is already built first;
  the unconditional `REVIEW_SUB_ROLES.map(...)` spawn is at
  [`cloister/review-agent.ts:463`](../../src/lib/cloister/review-agent.ts) *(verify)*).
- Filter `REVIEW_SUB_ROLES` per the rubric; the dead `ReviewAgentConfig.enabled` field
  ([`cloister/config.ts`](../../src/lib/cloister/config.ts)) can back per-role opt-out.
- **Risk floor:** if any `changedFiles[].riskScore` ≥ high, force-include `security`.
- Update [`roles/review.md`](../../roles/review.md) + review-monitor signal accounting to wait only
  for spawned reviewers (FR-14) — the one place a light review could otherwise hang forever.
- **Out of scope:** tiering the reviewers' *models* (composition/count only).

## 6. Eval strategy (the operator-requested paragraphs)

**Borrow, don't build.** We do not stand up an eval harness here. Capability-floor selection needs
only the *relative ordering* of models for code work; external SWE benchmarks (DeepSWE canonical,
SWE-bench Pro fallback) plus operator relative-anchoring give that cheaply and with provenance. The
honest limitation: SWE-bench measures generic GitHub-issue resolution — a good proxy for the **work**
role and a poor one for review/plan, which is exactly why planning is pinned to the top model and
review stays out of model-tiering.

**Instrument now, evaluate later.** The benchmark that ultimately matters is *"can model M handle
Panopticon-difficulty-D work in our codebase."* Every work run already yields that tuple
`(difficulty, model+effort, gate outcome, rework count, token cost, review verdict)`. We will NOT
collect or analyze it in this deliverable, but FR-9's decision record + the `CostEntry` shared key
make the future join possible without retrofitting. That single architectural choice is the whole
"data-collection influence on architecture" the operator asked for.

## 10. Decisions made in this document

- **D-1 Issue-level, not per-bead (v1).** One model per work agent, chosen from the issue's
  difficulty. Per-bead routing is PAN-1791.
- **D-2 Planning stamps difficulty** (vs. deriving from bead max). Planner assigns
  `plan.metadata.difficulty` holistically; if absent, fall back to max of item difficulties.
- **D-3 Four conceptual bands, expressed as floors** (`trivial/simple→low, medium→mid,
  complex→heavy, expert→frontier`), but the engine is a continuous `minCapability` floor, not pinned
  models.
- **D-4 Cheapest-above-floor is the default**, made safe by escalation-on-failure (FR-10) and the
  Settings aggressiveness lever (FR-11).
- **D-5 DeepSWE canonical, SWE-bench Pro per-model fallback only**, never blended.
- **D-6 No version inference** — every score traces to a benchmark row or explicit basis (FR-5).
- **D-7 Planning always top-available; review tiering out of scope** (composition rubric only).
