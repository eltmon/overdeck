## 1. Glossary (read first)

- **Three planes.** Every datum lives in exactly one of:
  - **Permanent plane (git, the "infra repo")** — the durable, portable per-issue record. Survives forever, moves machine-to-machine.
  - **Runtime plane (local SQLite, `~/.panopticon/panopticon.db`)** — ephemeral per-machine state for the agents/pipeline running *on this host now*. Never leaves the machine; rebuildable.
  - **Liveness oracle (tmux)** — "is this agent actually running?" = does a `panopticon`-socket session of that name exist. Unchanged by this work.
- **Infra repo.** The git repository (and path within it) that holds a project's permanent `.pan/` records. Declared per project in `projects.yaml`. Monorepo → the repo itself; polyrepo → a named repo (existing `infra`, or `api`/`fe`).
- **`agents` table.** New SQLite table that becomes the authoritative runtime registry of agents, replacing `~/.panopticon/agents/<id>/state.json` as the *enumeration + status* source. (The per-agent **directory stays** for artifacts — logs, launcher, transcripts, pty-token.)
- **Directory-as-registry.** Today's anti-pattern: `state.json` files are simultaneously the data *and* the only way to enumerate agents, forcing O(all-agents-ever) synchronous scans on hot paths. This PRD ends it.
- **Owner-URI lease.** The single runtime field that DOES live in the git record: a URI naming the machine currently driving an issue (`pan://node-3`), used as a distributed lease (mutual exclusion), the transfer baton (machine-to-machine handoff), an addressable monitoring target, and a reclaim signal.
- **Event-driven.** State changes are pushed as events to the existing event store; consumers react to events. **No polling, no directory scanning** (one possible narrow exception under evaluation — §7).
- **vBRIEF artifacts** (existing, reused — canonical ref `docs/VBRIEF.md`): **spec** (immutable plan after planning), **continue** (mutable per-issue working memory: decisions/hazards/feedback). This PRD adds **pipeline status** and **close-out** blocks to the per-issue record (§4) and routes runtime fields out to SQLite.

## 2. Vision, motivation, goals

**Vision.** Agent and per-issue state moves off the filesystem-as-registry. The **runtime plane** becomes an authoritative, event-driven SQLite `agents` table on each machine (no scans, no polling). The **permanent plane** is a per-issue vBRIEF record autocommitted to git in a declared **infra repo**, making issues portable across machines.

**Motivation (verified incidents):**
- Directory-as-registry → O(all-agents-ever) synchronous scans on 5s/60s hot paths → dashboard event-loop stalls (24s `/api/resources`), watchdog restarts, WebSocket reconnect storms. ([PAN-1711](https://github.com/eltmon/panopticon-cli/issues/1711), [PAN-1436](https://github.com/eltmon/panopticon-cli/issues/1436), [PAN-1846](https://github.com/eltmon/panopticon-cli/issues/1846))
- State divergence across `state.json` / SQLite / tmux / `review-status.json` → phantom incidents (the false [PAN-1883](https://github.com/eltmon/panopticon-cli/issues/1883)). ([PAN-1888](https://github.com/eltmon/panopticon-cli/issues/1888))
- No portability / cross-machine handoff; no canonical infra repo. ([PAN-541](https://github.com/eltmon/panopticon-cli/issues/541) master FS→SQLite migration, [PAN-111](https://github.com/eltmon/panopticon-cli/issues/111) cross-machine sync, [PAN-1325](https://github.com/eltmon/panopticon-cli/issues/1325) canonical artifact repo)

**Requirements.**

Functional:
- **FR-1** A SQLite `agents` table is the authoritative runtime registry; agent enumeration/count/status come from indexed queries, never directory reads.
- **FR-2** All agent-state reads/writes go through the existing choke-point functions (`getAgentState[Sync]`, `saveAgentState[Sync]`, `listRunningAgentsSync`, `countRunningAgents`), now backed by SQLite.
- **FR-3** Agent lifecycle transitions (spawn/start/stop/status-change/fail) **emit events**; consumers (deacon reactions, dashboard, schedulers) react to events — no 60s patrol scan, no `/api/resources` `state.json` sweep, no `autoResume`/`countRunning` directory read.
- **FR-4** Each project declares an **infra repo** in `projects.yaml`; the permanent per-issue record is written there and autocommitted.
- **FR-5** The permanent per-issue record reuses vBRIEF (spec + continue) and adds **pipeline status**, **close-out**, and **owner-URI lease** blocks. Runtime-only fields are removed from it.
- **FR-6** Close-out aggregates per-event token/cost telemetry into the git record as **raw token counts keyed by `provider/model` and by pipeline stage** (re-priceable), plus forge-agnostic **merge URL strings** (array, polyrepo).
- **FR-7** Legacy/unused fields identified in the audit are removed (not migrated).
- **FR-8** Conversations are **out of scope**; they remain machine-local SQLite (`conversations` table) and are not made portable here.

Non-functional:
- **NFR-1** No hot path performs O(all-agents-ever) work; agent-set queries are O(matching rows), indexed.
- **NFR-2** Writes to the `agents` table are transactional (no partial-write corruption — replaces the non-atomic `writeFileSync` at `src/lib/agents.ts:1080`).
- **NFR-3** The runtime plane is reconstructable: losing the SQLite DB loses no permanent data (git record + tmux are sufficient to rebuild).
- **NFR-4** Documentation of the new architecture is updated in the same change (§9).

## 3. Architecture — three planes + the event spine

```
            PERMANENT (git: infra repo .pan/)         RUNTIME (local SQLite)            LIVENESS
            per-issue vBRIEF record                    panopticon.db                     tmux (-L panopticon)
            ─────────────────────────────             ──────────────────────────        ───────────────────
   spec      immutable plan (vBRIEF)                    agents          (registry)        session exists?
   continue  decisions/hazards/feedback                 review_status   (ephemeral cols)   = is it running
   status    phase + verdicts + merges                  cost_events     (per-event)
   closeOut  token/cost by stage, ranOn                 health_events, events, ...
   owner     owner-URI lease  ◄── the one runtime
                                  field kept in git
                        ▲                                      ▲
                        │ autocommit on durable transition     │ event-driven upsert on lifecycle event
                        └──────────────── events ◄─────────────┘
                                   (existing `events` table is the spine)
```

**The event-driven spine already exists — extend it, do not invent it.** (Full map: Appendix B.)
- **Event store + emitter:** append-only `events` table + an `EventEmitter` in `src/dashboard/server/event-store.ts:131-353` (`append` / `appendAsync` / `emitOnly` / `subscribe` / `streamEvents`).
- **Taxonomy:** ~80 event `type` tags in `packages/contracts/src/events.ts`.
- **Consumers:** the read-model folds events via `packages/contracts/src/event-reducers.ts` (`src/dashboard/server/read-model.ts:429`, wired in `services/domain-services.ts:168`); `projection_cache` table; WS `subscribeDomainEvents` (`src/dashboard/server/ws-rpc.ts:500`).
- **A reactive scheduler ALREADY drives dispatch:** `src/lib/cloister/service.ts:821-846` subscribes to the event store and dispatches the next role on lifecycle transitions, fed by the `pipeline-notifier` bus (`review.approved` / `test.passed`). **The 60s deacon patrol (`src/lib/cloister/deacon.ts:184`) is explicitly the *safety-net*, not the primary driver.**

So the mandate is concrete: make the reactive scheduler + event store the **sole** driver, **project the `agents` table from lifecycle events**, and **delete the patrol scans** (§6.1) — the implementer extends existing wiring, not greenfield.

## 4. Data model — permanent plane (git, infra repo)

One file per issue (so autocommits never collide), reusing vBRIEF. **Consolidation decision:** the separate `.pan/status/<id>.json` idea is dropped — pipeline status is *mutable per-issue state*, so it joins the **continue** record, not the immutable spec. One durable per-issue record, cradle-to-grave.

### 4.1 Per-issue record shape (continue + status + closeOut + owner)

```jsonc
{
  "issueId": "PAN-1696",
  "owner": "pan://node-3",                  // owner-URI lease (only runtime field in git)
  // ── vBRIEF continue (REUSED, durable subset) ──
  "decisions": [{ "id": "...", "summary": "...", "recordedAt": "..." }],
  "hazards":   [{ "id": "...", "summary": "...", "mitigation": "..." }],
  "feedback":  [],
  // ── pipeline status (NEW; mutable; was review_status durable cols) ──
  "pipeline": {
    "phase": "review",                       // plan|work|review|test|merge|done
    "appliesToCommit": "a1b2c3d",
    "review": { "verdict": "changes_requested", "atCommit": "a1b2c3d" },
    "test":   { "verdict": "pending" },
    "merge":  { "ready": false, "blockers": ["failing_checks"] }
  },
  // ── close-out (NEW; written once at close-out; immutable thereafter) ──
  "closeOut": {
    "outcome": "merged",
    "openedAt": "...", "mergedAt": "...", "wallClock": "6h30m",
    "ranOn": "pan://node-3",
    "merges": [
      "https://github.com/eltmon/panopticon-cli/pull/1701",
      "https://gitlab.com/myorg/myn-api/-/merge_requests/88"
    ],
    "usage": {
      "byStage": {
        "plan":   { "anthropic/claude-opus-4-8":  { "input": 210400, "output": 41200, "cacheRead": 880300, "cacheWrite": 60100 } },
        "work":   { "kimi-coding/kimi-k2.7-code":  { "input": 3120300, "output": 198700, "cacheRead": 7840100, "cacheWrite": 151000 } },
        "review": { "openai-codex/gpt-5.5":        { "input": 1450200, "output": 96400, "cacheRead": 2010800, "cacheWrite": 88000 } },
        "test":   { "anthropic/claude-haiku-4-5":  { "input": 142600, "output": 9100, "cacheRead": 60200, "cacheWrite": 4000 } },
        "merge":  { "anthropic/claude-sonnet-4-6": { "input": 88300, "output": 12400, "cacheRead": 41100, "cacheWrite": 6000 } }
      }
      // + "totals" roll-up per provider/model = sum across stages
    },
    "costAtCloseOut": { "usd": 31.77, "pricingAsOf": "2026-06-13" }   // derived snapshot, recomputable from usage
  }
}
```

Design rules:
- **`merges`** are forge-agnostic **URL strings**, array (polyrepo + GitHub/GitLab/etc.); serialized from the live `merge_set_repos` rows (which already carry `forge`, `source_branch`, `target_branch`). Replaces the bare numeric `pr`.
- **`usage`** stores **raw token counts** keyed `provider/model` and bucketed by the four `cost_events` token columns (`input`/`output`/`cacheRead`/`cacheWrite`), grouped **by pipeline stage** (`cost_events.session_type`). `costAtCloseOut.usd` is a derived convenience snapshot, never the source of truth.
- **REMOVED from the record** (these go to the runtime SQLite plane, never git): the continue file's `gitState.dirty`/`sha`, `resumePoint`, `sessionHistory`, `agentModel`, `beadsMapping`. (Keep `gitState.branch` if useful, or derive.)
- The vBRIEF **spec** stays as today (immutable after planning, `plan.status` the only mutable field) — unchanged by this PRD except it now lives in the infra repo.

### 4.2 Infra-repo declaration (`projects.yaml`)

Add a per-project `infra_repo` selector naming where `.pan/` records are written:
- **monorepo** (e.g. panopticon-cli): defaults to the repo itself (`.`).
- **polyrepo** (e.g. MYN, Auricle — which already declare `- name: infra`): names which repo holds records (`infra` | `api` | `fe`).

```yaml
# projects.yaml (per project)
pan_records:
  repo: infra            # name of the repo in `repos:` (polyrepo) or "." (monorepo)
  path: .pan             # subdir within that repo
```

> Grounded in current structure: monorepo `panopticon-cli` (`type: monorepo`, `github_repo`), polyrepo `myn`/`auricle` (`type: polyrepo`, `repos:` incl. `- name: infra / path: infra`).

## 6. Event-driven mandate — remove all polling/scanning

**Rule:** no production *steady-state* code enumerates agents by reading the directory, and no orchestration step runs on a fixed-interval scan of state files. The `agents` table is updated by lifecycle **events**; readers query the table; the deacon **reacts** to events instead of patrolling.

**Migration exception (named, so AC-2 doesn't fail itself):** the one-time versioned **backfill command** (§8) MAY enumerate `~/.panopticon/agents/` — that is the *only* permitted directory enumeration, it runs once at cutover, and it must NOT be reachable from any dashboard/deacon hot path (a test proves this).

## 7. Work items  (exact call-sites to touch: Appendix A §3)

> Each item gets: files touched (from the call-site map in `/tmp/prd-legacy.md`), step-by-step changes, before/after, and named tests. Skeleton:

1. **Schema + DB module** — add `agents` table + indexes; new `src/lib/database/agents-db.ts` (mirror `conversations-db.ts`); migration. Tests: agents-db unit.
2. **Back the choke-points with SQLite** — rewrite `getAgentState[Sync]`, `saveAgentState[Sync]`, `listRunningAgentsSync`, `countRunningAgents` against `agents-db` (FR-2, NFR-2). **Do NOT rip out `getAgentDir`** — it has ~67 call sites that build paths to per-agent *sidecar artifacts* (`pty-token`, transcripts, `output.log`, `launcher.sh`), which keep living in the directory (§5 keeps the dir for artifacts); only the `state.json` read for *enumeration/status* migrates to SQLite. Tests: existing agents/concurrency suites stay green.
3. **Emit lifecycle events + write-through projection (transactional boundary — DECIDED).** Every lifecycle transition (`agent.started`, `agent.status_changed`, `agent.stopped`, `agent.heartbeat_dead`) **upserts the `agents` row AND appends the event in ONE SQLite transaction** (write-through: the row is authoritative the instant the tx commits; subscribers/dashboard react to the event but never gate the row's correctness). This is the decision — NOT append-then-async-project, which would let a crash between append and projection leave the registry stale, the exact `state.json` failure we are removing. **Per-event payloads** (add to `packages/contracts/src/events.ts` with reducer + emit sites): `agent.started` = full initial row; `agent.status_changed` = `{agentId, status, …only-changed fields}`; `agent.stopped` = `{agentId, issueId, stoppedBy, stoppedAt}`; `agent.heartbeat_dead` = `{agentId, lastSeen}`. The projection **upserts by `id`, merging only present fields** (a partial event never nulls absent columns) and is **idempotent** (replay yields the same row). (FR-3; AC-3, AC-11)
4. **Convert deacon patrol → event reactions** — remove `autoResumeStoppedWorkAgents`/`checkOrphaned*` scans; subscribe to events (§6.1).
5. **`/api/resources` + dashboard read-model → query `agents`** (delete the `readdir` sweep).
6. **Permanent record writer** — write per-issue vBRIEF record (continue+pipeline+closeOut+owner) to the infra repo. **Extend `queueAutoCommit()` at `src/lib/pan-dir/auto-commit.ts:142`** (debounced 2s, main-only, rebase-after-commit; today driven by `pan-dir/{specs,continues,drafts}.ts` + the deacon beads sweep) to support **repo-target selection** — commit `.pan/` records to the *declared infra repo*, not unconditionally the project's own repo. **No infra-repo concept exists today** (only the polyrepo `workspace.repos[]` for code repos), so this adds it. (FR-4, FR-5)
7. **Pipeline status → git** — route durable `review_status` verdicts into the record's `pipeline` block on change; keep ephemeral cols in SQLite (§5.3).
8. **Close-out aggregation** — the record is **per issue**, so the query MUST filter by issue: `SELECT … FROM cost_events WHERE issue_id = ? GROUP BY session_type, provider, model` → `closeOut.usage.byStage`. Rows with a null/unknown `session_type` bucket under stage `"other"` (never dropped, never mis-attributed). Serialize `merge_set_repos` → `merges[]` forge-agnostic URLs; stamp `ranOn`. (FR-6)
9. **`projects.yaml` infra-repo declaration** + resolver (§4.2).
10. **Owner-URI lease** — claim/release/reclaim on the git record (mutual exclusion + transfer).
11. **Remove legacy fields** — delete the confirmed-legacy `AgentState` fields and `review_status` columns and their dead code (FR-7).

## 8. Big-bang migration / cutover

Single migration, not dual-write:
1. Create `agents` table + indexes.
2. **Backfill** the table once from existing `~/.panopticon/agents/<id>/state.json` files (and reconcile status against live tmux).
3. Switch every reader/writer (work items 2,4,5) to SQLite in the same change.
4. Delete the `readdir`-based enumeration and the patrol scans.
5. Write existing in-flight issues' permanent records into the infra repo (backfill from current `.pan/continues` + `review_status`).
6. The per-agent **directory remains** for artifacts; `state.json` is no longer read for enumeration/status (may be left as a derived debug dump or dropped — decide in work item 2).

### 8a. Rollback / safety net (mandatory — big-bang has no dual-write)

- **Backup:** snapshot `panopticon.db` (file copy) before migrating, and keep `state.json` written through the cutover (CP-2 = *keep*, not drop, until the new registry is proven) so the old enumeration source still exists.
- **Idempotent migration:** the backfill upserts by `id` — re-running converges, never duplicates; safe to re-run after a partial failure.
- **Rebuild path:** the `agents` table is fully reconstructable from `state.json` + live tmux (NFR-3) — provide `pan admin rebuild-agents` to re-derive it; permanent records are reconstructable from git + `.pan/continues`.
- **Kill switch:** a single flag (e.g. `PANOPTICON_NO_RESUME=1` + an `agents`-table-off fallback) disables the event-driven deacon and drops to safe no-resume mode if the dashboard wedges, with no data loss.

## 4. `.pan` autocommit + infra-repo mechanism (what the PRD must extend)

### Current autocommit entrypoint
`queueAutoCommit({ projectRoot, paths, subject })` in **`src/lib/pan-dir/auto-commit.ts:142`** — fire-and-forget, debounced 2s, coalescing. Flush via `flushAutoCommits()` (`:193`). It is **main-only** (`:242` aborts "not on main"), fetches `origin/main` and rebases after commit to avoid divergence (`:247`), and refuses excluded paths (`AUTO_COMMIT_EXCLUDED_PATHS` `:47`: `.pan/kickoff.md`, `.pan/continue.json`, `.pan/handoff-*.md`, `.pan/spec.vbrief.json`, `.claude/rules/`, `.claude/skills/`).

**Callers (the only write sites that auto-commit `.pan/`):**
- `src/lib/pan-dir/specs.ts:153` — `.pan/specs/*.vbrief.json`
- `src/lib/pan-dir/continues.ts:56` and `:100` — `.pan/continues/<issue>.vbrief.json`
- `src/lib/pan-dir/drafts.ts:59` — `.pan/drafts/<issue>.md`
- `queueBeadsAutoCommit()` (`auto-commit.ts:179`) → called from `cloister/deacon.ts:5280` patrol — `.beads/issues.jsonl`, `.beads/export-state.json`

So today `.pan/` durable artifacts (specs, continues, drafts) and beads exports are committed to **the project's own git repo on `main`**, by the deacon/pan-dir write path. Workspace-side files (`continue.json`, `spec.vbrief.json`, kickoff/handoff) are explicitly excluded.

### `pan sync` — distinct mechanism, NOT autocommit
`src/cli/commands/sync.ts` syncs **bundled skills/agents/context/hooks/statusline** from `sync-sources/` into `~/.claude/` and registered project workspaces (`planSyncSync`/`executeSyncSync`/`syncContextLayersSync`, `sync.ts:12`). It renders context layers into harness files and mirrors `.pan/skills/`. It does **not** commit per-issue state and is unrelated to the agent-state store.

### Multi-repo / "infra repo" reality

## Glossary

- **Event store** — SQLite `events` table + in-memory `EventEmitter`. The append-only domain event log. `src/dashboard/server/event-store.ts`.
- **Domain event** — a `{ type, timestamp, payload, sequence }` record. The `type` taxonomy is the union in `packages/contracts/src/events.ts`.

## 4. Existing Reactive / Event-Driven Dispatch (vs the 60s patrol)

**A reactive scheduler already exists.** The pipeline does NOT advance solely by the 60s patrol — there is an event-triggered dispatch path:

- `CloisterService.subscribeToDomainEvents()` — `service.ts:821-846`. Calls `initEventStore()` then `store.subscribe((event) => Effect.runPromise(handleCloisterDomainEvent(event)))` at `:827-836`. Log line: "Cloister reactive lifecycle scheduler subscribed to domain events" (`:837`).
- `handleCloisterDomainEvent` → `handleCloisterDomainEventPromise` (`service.ts:417-421`): maps the event to an issue-state change via `issueStateChangeFromDomainEvent` (`:385-417`), then `onIssueStateChange(issueId, state)`.
- `issueStateChangeFromDomainEvent` maps: `issue.statusChanged`→canonicalStatus, `agent.completed`(role=work)→`in_review`, `work.completed`→`in_review`, `review.approved`→`testing`, `test.passed`→`shipping`, `issue.closed`→`closed` (`:390-416`).
- `ROLE_RUN_STATES` (`:151-161`) maps lifecycle state → role: `in_planning→plan`, `in_progress→work`, `in_review→review`, `testing→test`, (`shipping`/`closed`→null, server-side merge handoff).
- `onIssueStateChange` (~`:250-379`) does idempotency guards (`activeRoleRunExists`, zombie-session kill `:316-332`) then dispatches: review via `spawnReviewRoleForIssue` (`:343`), test via `dispatchTestAgentAndNotify` (`:353`), else `spawnRun(issueId, role, ...)` (`:362`).
- **Pipeline-notifier bus** (`src/lib/pipeline-notifier.ts`): `notifyPipelineSync(event)` (`:39-88`) calls the in-process `handler` if set (dashboard) else POSTs to `/api/internal/pipeline/notify`. `review.approved`/`test.passed` are emitted here from `review-status.ts:20-26` — this is how a CLI/specialist process feeds the reactive scheduler when it isn't the server.

**So work IS advanced reactively on lifecycle transitions.** The 60s deacon patrol is the *safety net* for events that were dropped, processes that crashed mid-transition, or state set outside the API. The patrol comment at `deacon.ts:184` says exactly this: "Safety net — immediate processing happens via pipeline events."

The PRD's goal is to widen the reactive surface so the patrol's scan-based steps become event-handlers, shrinking (not the scheduler — that's already reactive) the *polling/scanning* steps in §5.

The work is (a) emit the missing transition events at their source, (b) move each patrol scan into a `store.subscribe` handler keyed on that event, and (c) keep the 60s patrol only as a thin reconciler/safety-net for dropped events.
