# PRD: Fork-into-worktree from conversation branch chip

**Issue:** [PAN-1533](https://github.com/eltmon/overdeck/issues/1533)
**Status:** Draft
**Author:** Edward Becker (operator)
**Date:** 2026-05-26

## Problem

The branch/worktree chip we added in #1523 is read-only. To start an exploratory conversation in a different worktree today, a user has to either:

1. Manually `git checkout` in their main checkout — risks disturbing in-flight work and breaks any other conversation rooted there.
2. Provision a full Overdeck agent workspace via `pan start <issue>` — heavy ceremony (planning, beads, review pipeline) for "I want to discuss this code on a side branch."

There is no lightweight path from "I'm looking at this conversation" to "give me a sibling conversation in a worktree."

## Goal

Make the branch chip a useful affordance: click it, pick (or create) a worktree, fork the conversation in there.

## Non-goals

- **In-place cwd switching.** Considered and rejected. Claude's session-resume is keyed off the cwd-derived project dir (`~/.claude/projects/<encoded-cwd>/...`). Switching cwd mid-conversation means a fresh JSONL in the new dir; the existing transcript lives in the old dir disconnected from the new session. The UX of "this conversation now exists in two places, neither of which can resume the other" is worse than forking and being honest about it.
- **Worktree cleanup / lifecycle.** Out of scope. Treat `worktrees/` as human-managed for now. A future issue can add explicit "delete worktree" when all its child convs are archived.
- **Worktree selection for agent sessions.** Work agents are pinned to `feature/<issue-id>` by construction. The picker should not render on agent ConversationPanel headers.

## Design

### User flow

```
User clicks branch chip on a non-embedded ConversationPanel header
  └── popover opens with:
      ├── ✓ feature/main             ← current, pre-selected
      ├── feature/foo                ← existing worktree
      ├── feature/bar (in use)       ← existing worktree, has an active agent
      ├── ─────────────────────────
      └── + Create new worktree…
            └── inline form: branch name [____________] from [main ▾] [Create]
                  └── on success: opens fork modal with targetWorktreePath set
```

Picking an existing worktree (other than the current) opens the existing fork modal pre-seeded. "Create new worktree…" runs `git worktree add` then opens the modal.

### Directory convention

```
<projectRoot>/
├── workspaces/feature-pan-1450/   ← pipeline-managed agent worktree
├── workspaces/feature-pan-1242/   ← pipeline-managed agent worktree
├── worktrees/conv-20260526-7188/  ← conv-spawned worktree (THIS PRD)
└── worktrees/conv-20260526-3595/  ← conv-spawned worktree
```

**Why a separate `worktrees/` dir, not `workspaces/`?**

Agent workspaces have a strict lifecycle (deacon patrols, postMergeLifecycle, close-out, cleanup). Conversation worktrees should be lighter — closer to scratch directories — and shouldn't be subject to the same lifecycle policing. Mixing them means every code path that touches `workspaces/` has to discriminate by name pattern, and a future cleanup pass that doesn't know about the convention can nuke a user's conv worktree.

Naming: `conv-<sourceConversationName>`. The conversation `name` is already a stable, greppable, collision-free slug (e.g. `20260526-7188`).

### Backend

1. **`GET /api/projects/:key/worktrees`**
   - Source: `git worktree list --porcelain` in the project root.
   - Annotate each entry with `{ path, branch, hasActiveAgent }` (cross-reference resource-discovery's agent map).
   - Cache TTL 30s, same pattern as `resolveAgentGitInfo`.
   - Returns `[{ path: string, branch: string, isPrimary: boolean, hasActiveAgent: boolean }]`.

2. **`POST /api/projects/:key/worktrees`**
   - Body: `{ branch: string, base?: string }`.
   - Resolves the target path as `<projectRoot>/worktrees/conv-<sanitized-slug>` where `<sanitized-slug>` is derived from the conversation name passed in the body (or a fresh slug if creating without a source).
   - Rejects paths outside `<projectRoot>/worktrees/`.
   - Runs `git worktree add <path> <branch>` async. If `branch` does not exist, also pass `-b <branch> [base]` (defaults to current HEAD of `<base>` or `main`).
   - Validates branch name against `^[a-zA-Z0-9._/-]+$` (git's own rules are looser, but we want predictable filesystem paths).
   - Returns `{ path: string, branch: string, isPrimary: false }` on success.

3. **`POST /api/conversations/:name/fork` — extend**
   - Accept optional `targetWorktreePath` in the body.
   - If provided, validate it points to a real git worktree of the same project as the source conversation's cwd.
   - Spawn the fork's Claude session with `cwd = targetWorktreePath`.
   - Store the new conversation's `cwd` accordingly.

4. **Resource discovery**
   - New scanner pass for `<projectRoot>/worktrees/` that surfaces conv worktrees as a sibling collection to `workspaces/` — does NOT use beads / vBRIEF / agent-state checks.
   - Existing scanners stay unchanged.

5. **`.gitignore`**
   - Add `worktrees/` at the repo root.

### Frontend

1. **`<WorktreePickerMenu>`** (new component)
   - Lightweight popover anchored to the branch chip.
   - Lists worktrees fetched from the new endpoint via TanStack Query.
   - Visual: branch name + small icon for `(in use)` when `hasActiveAgent`.
   - "Create new worktree…" entry at the bottom (separator above it).
   - Keyboard: Esc closes; ArrowUp/Down navigate; Enter selects.

2. **`<CreateWorktreeForm>`** (new component, inline within picker)
   - Branch name input.
   - Base branch picker (dropdown of existing branches).
   - Submit calls `POST /api/projects/:key/worktrees`, on success closes the form and opens fork modal.
   - Inline error rendering on validation / creation failure.

3. **`<ForkModal>` — extend**
   - Accept `targetWorktreePath` prop.
   - When set, render a confirmation line above the launch button: "Forking into `<branch>` at `<path>`."
   - Pass the path through to the fork POST.

4. **`ConversationRow` and `ConversationPanel` — wire the chip click**
   - Replace the read-only `<span>` chip with a `<button>` that opens the picker.
   - Anchor positioning: above the chip (toolbar is at the bottom of the panel for ConversationPanel — popover renders upward).
   - Skip the button on agent / specialist / planning conversation names (regex match — same gating as the work-agent rule in #1523).

### Edge cases

- **Source conv is in main checkout, all worktrees already in use.** Picker still works — user picks "Create new worktree…" or picks an existing one.
- **Source conv is itself in a worktree.** Picker shows that worktree as current; other worktrees + main are alternatives.
- **Worktree creation fails (branch already checked out elsewhere, dirty path).** Inline error in the form. No partial state — `git worktree add` is atomic.
- **Fork into a worktree whose Claude session is busy.** Not applicable — a worktree is a filesystem location, not a session. The fork is a new conversation; it doesn't collide with anything happening inside the cwd.
- **User picks current worktree.** Treat as no-op; close the picker.

## Implementation steps (proposed beads)

1. **Backend: worktree listing endpoint.** `GET /api/projects/:key/worktrees` + tests. Use the existing `worktree list --porcelain` parser already implicitly in `git-info.ts` if extractable; otherwise add a focused parser.
2. **Backend: worktree creation endpoint.** `POST /api/projects/:key/worktrees` + tests. Validation, async git command, error reporting.
3. **Backend: fork endpoint extension.** Accept `targetWorktreePath`, validate, plumb into spawn.
4. **Backend: resource-discovery scanner for `worktrees/`.** New scan pass + tests.
5. **Gitignore + repo housekeeping.** Add `worktrees/` to root `.gitignore`.
6. **Frontend: `<WorktreePickerMenu>`.** New component + tests.
7. **Frontend: `<CreateWorktreeForm>`.** New component + tests.
8. **Frontend: ForkModal extension.** Accept and render `targetWorktreePath`.
9. **Frontend: chip → picker wire-up in `ConversationRow` and `ConversationPanel`.** Gate by conv-type regex.
10. **Manual end-to-end verification.** Fork into existing worktree; fork into newly-created worktree; verify the source conv is untouched and the fork's Claude session has the right cwd.

## Open questions

- **Should we auto-archive the conv worktree when the spawned conversation is archived?** Defaulting to NO for v1 — keep `worktrees/` strictly human-managed until we see usage patterns.
- **Where does base-branch detection come from for "Create new worktree…"?** Suggest: the project's default branch (`main` for overdeck, configurable later). Fetch via `git symbolic-ref refs/remotes/origin/HEAD` with a fallback to `main`.

## Risks

- **Disk usage.** Each worktree is a full checkout. For a 200MB repo, 5 worktrees = ~1GB. We don't show this to the user; should add an indicator in a future issue.
- **Confusion with agent workspaces.** Mitigated by the separate dir. Worth a tooltip or docs note: "Conversation worktrees live in `worktrees/`. Pipeline-managed agent workspaces live in `workspaces/`."
