## Glossary

- **Barrel** — a module that contains only `import`/`export` statements, no logic.
  After this refactor, `conversation-service.ts` becomes a barrel that re-exports
  the same public symbols it exports today, so every existing `import ... from
  './services/conversation-service.js'` resolves unchanged.
- **Public surface** — the set of symbols currently exported from
  `conversation-service.ts` and reachable by importers. There are **16** today
  (11 functions + 5 interfaces, enumerated below). The barrel must re-export all 16.
- **Internal-shared symbol** — a non-exported helper/type/constant used by more
  than one extracted module (e.g. `JsonlEntry`, `MAX_READ_BYTES`). These are
  `export`ed from their new module so siblings can import them, but are **not**
  re-exported through the barrel — they stay off the public surface.
- **File-size guard** — the codebase-health rule that each source file stays under
  1,000 lines.
- **Seam** — one extraction (one cluster of code moved to a new module + barrel
  re-export updated). Each seam is one bead and must leave the tree green.

## Problem

`conversation-service.ts` is 1,609 lines — over the 1,000-line file-size guard.
It mixes nine distinct concerns: JSONL session-file discovery, provider
detection, message-text filtering, the core incremental JSONL parser, activity
summarization, compact-boundary detection, context-window usage computation, and
file-watching. The parser alone is ~725 lines.

## Proposal

Pure relocation into a sibling directory
`src/dashboard/server/services/conversation/`, leaving `conversation-service.ts`
as a thin re-export barrel. No logic edits, no renamed exports, no importer edits.
Behavior identical, verified by the existing comprehensive test suite
(`conversation-service.test.ts`, which exercises the parser, context usage,
activity summary, watch, gate, and discovery through the barrel).

### Dependency DAG (no cycles)

```
types.ts ──────────────► (every other module)
provider.ts ───────────► parser.ts
message-filters.ts ────► parser.ts
compact-boundary.ts ───► context-usage.ts, parser.ts
parser.ts ─────────────► activity-summary.ts, watch.ts
session-files.ts ──────► (barrel only)
context-usage.ts ──────► (barrel only)
```

Extraction order follows this DAG leaf-first, so each new module imports only
already-extracted modules (or stdlib/external libs), never the barrel — avoiding
any runtime import cycle.

### The 16 public symbols the barrel MUST re-export

## Ownership decisions (cross-cluster symbols)

Verified against the file at planning time (`grep -n` for each symbol):

- `LatestAssistantUsage` (declared at line 1364) is referenced by `ParseResult`
  (line 47) and `ParseState` (line 91), so it MUST live in `types.ts`, not
  `context-usage.ts`. `context-usage.ts` imports it from `types.ts`.
- `JsonlEntry` is used by the parser (421, 423) AND `readLatestAssistantUsage`
  in context-usage (1397, 1399) → `types.ts`, imported by both.
- `MAX_READ_BYTES` / `MAX_FALLBACK_BYTES` are used by the parser AND
  `findLastCompactBoundary` → `types.ts`, imported by parser + compact-boundary.
- `ModelCapability` (line 23) is used only by context-usage (1278, 1289) → stays
  in `context-usage.ts`.
- `ContentBlock`, `JsonlUsage` are parser/JSONL-shape types; grouped into
  `types.ts` with the other JSONL interfaces for cohesion.

## Importer / test impact: NONE

Every external importer and every test reaches `conversation-service.ts` through
the barrel and uses only public exports — verified:

- Non-test importers: `ws-rpc.ts`, `routes/agents.ts`, `routes/conversations.ts`,
  `conversation-message-resolver.ts`, `conversation-compaction.ts`, and the
  `pi`/`codex`/`ohmypi` parsers (the last three `import type { ParseResult }`).
- `conversation-service.test.ts`, `agents-conversation.test.ts`,
  `ws-rpc-resolved-stream.test.ts`, `codex-conversation-parser.test.ts` all
  `import` / `await import('../conversation-service.js')` — barrel only.

So no importer or test file needs editing. The requirement note "repoint any test
importing moved internals" does not trigger here.
