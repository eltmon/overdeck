# PAN-2173 — Flywheel under-plans new ready backlog while draining a cohort

**Issue:** [PAN-2173](https://github.com/eltmon/overdeck/issues/2173)
**Type:** bug(flywheel)
**Planned by:** plan role (`--auto`, claude-opus-4-8), 2026-06-29

## Glossary

- **Flywheel / orchestrator** — the singleton LLM agent (role `flywheel-orchestrator`) that
  drives the Fix-All loop, following `roles/flywheel.md`. It loops *ticks* internally within
  one session; its standing brief is `roles/flywheel.md` + `docs/flywheel-brief.md`, re-read
  on spawn/resume only.
- **Tick** — one revolution of the orchestrator loop: inventory → diagnose → suggest → launch → emit-status.
- **Cohort (PAN-2006)** — the set of issues a Run commits to draining, frozen at Run start
  (`computeCohort` = in-flight ∪ current+next wave). "The Run is complete once this cohort drains;
  mid-run pickups don't extend it." Computed in `src/lib/backlog/pickup.ts:190`.
- **needsPlanning (PAN-2006)** — an issue that is `ready` but not `planned`, and not
  parked/vetoed/objected/in-pipeline. Already computed as a *count* in `computeStats`
  (`src/lib/backlog/pickup.ts:252`). These are the items that get starved.
- **Release gate (PAN-2059)** — a Planned item is auto-pickable (startable) only after the
  operator applies the `released` label (`isAutoPickable` requires `ready && planned && released`).
  **Planning happens BEFORE release** — so planning new backlog never bypasses operator control.
- **Awaiting-release queue** — items that are Planned but not yet Released; the queue the
  operator picks from. Step 5 of `roles/flywheel.md` wants this kept deep via aggressive planning.
- **Relevance vet (PAN-2059)** — the pre-launch check (already done / refs stale / deps unmet /
  net-negative) the orchestrator runs before planning/starting any backlog item.

## Problem

Observed 2026-06-29 (RUN-37/38, opus): the flywheel spent every tick draining the in-flight
cohort (closing out PAN-2063/2171, re-dispatching PAN-1982 review) and **never planned the new
ready backlog**. PAN-1894 and PAN-2076 sat **unplanned for hours** despite free capacity (3–7 of
20 agents) and no objection/parked labels — until manually kicked via `pan plan --auto`.

This contradicts `roles/flywheel.md` step 5 (≈line 110): *"keep the awaiting-release queue deep by
planning everything that passes the relevance vet."*

### Root cause (two contributing factors)

1. **Soft prose, no enforceable floor.** Step 5 only says "keep planning aggressively." There is no
   concrete per-tick planning floor, no SLA, and the PAN-2006 cohort-freeze wording
   (*"the Run is complete once this cohort drains; mid-run pickups don't extend it"*) biases the
   LLM toward drain-only behavior. When the cohort is busy, the orchestrator reads "finish the
   cohort" and deprioritizes planning new work indefinitely.

2. **No per-tick surface of the starved set.** The orchestrator recomputes state each tick from
   API surfaces (`pan review pending`, `GET /api/flywheel/merge-blockers`, etc.). The
   `needsPlanning` set is already computed by the shared pickup model but is exposed only as a
   *count* in `stats.needsPlanning`, never as a *list*. `GET /api/backlog/forecast` returns
   `inFlight`, `waves` (auto-pickable), and `cohort` — but `needsPlanning` issues are neither
   in-flight nor auto-pickable (they're unplanned), so **they never appear in any list the
   orchestrator reads each tick.** Nothing puts the starved backlog in front of it.

A soft-prose fix alone would not work: RUN-37/38 already had the soft prose and ignored it. The
fix must make the floor **data-driven and surfaced every tick**, and **codify an enforceable rule**.

## Proposal

Make the planning floor observable and mandated, reusing the existing pure pickup model.

1. **Pure selector** — add `selectNeedsPlanning(nodes, lk, { cap })` in `src/lib/backlog/pickup.ts`,
   mirroring the existing `selectUnblockTargets` (`pickup.ts:215`). Returns the ranked
   `needsPlanning` issues (`ready && !planned && !parked && !vetoed && !objection && !inPipeline`),
   capped. Single source of truth for "what should be planned."

2. **Surface the list** — `GET /api/backlog/forecast` (`src/dashboard/server/routes/backlog.ts:292`)
   gains a `needsPlanning` array (enriched: issue/rank/title/why) alongside `inFlight`/`waves`/`cohort`.
   This is the per-tick surface the orchestrator reads.

3. **Mandate the floor** — `roles/flywheel.md` step 5 gains an explicit **Planning floor** rule:
   each tick, read `needsPlanning[]` from the forecast and run `pan plan <id> --auto` for up to
   **2** of them that pass the author/assignee + parked/objection gates — **even while draining the
   cohort**. Reconcile with PAN-2006 (planning fills the awaiting-release queue; it does NOT
   auto-start and does NOT extend the running cohort). State the SLA: a ready, vetted,
   capacity-available issue is planned within ~1–2 ticks.

### Decisions made in this doc

- **Floor cap = 2 planning launches per tick** (mirrors `selectUnblockTargets` default cap of 2).
- **The floor applies regardless of `auto_pickup_backlog`.** Planning stops at the Release gate
  (PAN-2059) and never auto-starts, so it cannot bypass operator control — step 5 already states
  planning should be aggressive unconditionally.
- **The forecast endpoint stays a pure projection of the shared model** — it is NOT author/assignee
  filtered. The eltmon author/assignee security gate is applied by the orchestrator per item before
  planning (consistent with how the role already gates every launch). The classifier already
  excludes parked/vetoed/objection/in-pipeline.
- **No code auto-fires planning agents.** The orchestrator remains the launcher (its #1 job per
  `roles/flywheel.md`); the floor is *surfaced + mandated*, not auto-launched by a second uncontrolled path.
- **Use `pan plan --auto`, never `--auto-start`** — the floor only PLANS; starting stays behind the
  operator's Release gate.

## Requirements

- **FR-1** — A pure selector returns the ranked set of vetted ready-but-unplanned backlog issues
  (`needsPlanning`), respecting a cap.
- **FR-2** — `GET /api/backlog/forecast` exposes the `needsPlanning` list (ranked, enriched) so the
  orchestrator can read it each tick. Existing forecast fields are unchanged.
- **FR-3** — `roles/flywheel.md` mandates planning up to 2 `needsPlanning` items per tick — decoupled
  from cohort drain, applying the author/assignee + parked/objection gates, using `pan plan --auto`.
- **FR-4** — The role rule states the SLA: a ready, vetted, capacity-available issue is planned
  within ~1–2 ticks, and reconciles the floor with the PAN-2006 cohort-freeze.
- **NFR-1** — No change to existing forecast fields, `stats.needsPlanning` count semantics, the
  Forecast UI, the Release gate (PAN-2059), cohort-freeze / run-completion (PAN-2006), or auto-start.
