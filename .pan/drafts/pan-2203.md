## 0. Glossary (define-before-use)

- **Slot** — one parallel work agent inside a single issue's swarm. Named `agent-<issue>-slot-N`, on branch `feature/<issue>-slot-N`, in worktree `<featureWorkspace>-slot-N`. `N` is a 1-based slot index.
- **Wave** — a set of vBRIEF items with no unresolved `blocks` dependencies on un-merged items; computed by `groupItemsByWave` (Kahn's algorithm). Waves run sequentially; slots within a wave run in parallel.
- **Foreman / orchestrator / coordination pass** — the loop that drives a swarm: dispatch a wave of slots → detect completion → verify+merge each slot's branch → advance to the next wave → recover failed slots. Today it exists only as **prose** the work-agent LLM executes (`roles/work.md:70-107`). This epic makes it a **deterministic** function, `coordinateSwarmSlots()`.
- **Registered slot** — the current (PAN-1762) slot model: `spawnRun(issueId, 'work', { slotIndex, slotItemId })` resolves naming/branch/worktree via `resolveRegisteredSlotSpawn` and spawns the agent. Contrast the deleted `SlotAssignment`/`SwarmState` model.
- **Dispatch tier** — `chooseDispatchTier(item)` → `'in-context'` (do it in the parent agent) or `'registered-slot'` (spawn a slot). Reads `item.metadata`.
- **Swarm readiness** — `analyzeSwarmReadiness(doc)` → per-item slot-eligibility + all-pairs `files_scope` overlap matrix; `swarmEligible` true iff ≥1 item is slot-eligible AND currently dispatchable.
- **Derive-don't-store** — the design rule for this orchestrator: hold **no** persistent swarm state. Each tick re-derives merged/in-flight/pending from live git slot branches + tmux + the immutable main-side vBRIEF, via `reconcileSlotState`. The only durable state is the vBRIEF item status, which the pipeline already owns.
- **Synthesis phase** — for a DAG convergence item (>1 blocking parent), a first "synthesis" slot summarizes parents' outputs into a context blob that the item's implementation slot then consumes.
- **Failed-merge recovery** — when a slot branch cannot merge (conflict), the operator/loop chooses **retry** (re-run the slot), **drop** (mark item done, proceed), or **handoff** (pause auto-advance for a human). This is the PAN-1194 concern.

## 1. Problem

PAN-1762 ("swarm-v2") shipped the *primitives* and a **prose** foreman loop, but nothing wires them into an autonomous driver:

- `verifyAndMergeSlot` (`src/lib/agents/slot-merge.ts:36`), `reconcileSlotState` (`src/lib/agents/slot-reconcile.ts:52`), and `spawnRun({slotIndex})` (`src/lib/agents/spawn.ts:78,84`) have **zero non-test callers**. `analyzeSwarmReadiness` (`src/lib/vbrief/swarm-readiness.ts:73`) is called only by `quality-lint.ts` and the finalize readiness report — not by any orchestrator.
- There is **no `swarm` command** in `src/cli/index.ts`.
- The only "foreman" is the LLM-interpreted protocol at `roles/work.md:70-107` ("no auto-advance poller, no per-slot PR") — a work agent hand-driving waves. Fragile; unsafe unattended.

A full old orchestrator existed (`src/dashboard/server/routes/swarm.ts`, 2598 lines) and was **deleted** in `59f1e6cd4a` (PAN-1517), partly for cause (`pan-1194-swarm-no-recovery-when-a-slot's-PR-becomes-unmergeable`). Its coordination *logic* is the reference; its stateful bloat is not to be revived.

## 2. Locked design decisions

- **D1 — Derive-don't-store (NFR-1).** No new persisted swarm state. `reconcileSlotState` re-derives buckets from git+tmux+vBRIEF each tick. The deleted `SwarmState.slots` + dual `continue.json`/sidecar persistence is not recreated. Rationale: it was the largest bug surface of the old route.
- **D2 — Deacon-integrated, not a fat route.** The loop is a `coordinateSwarmSlots()` pass in a new `src/lib/cloister/deacon-swarm.ts`, invoked once per `runPatrol()` tick — the modern successor to the deleted `detectMergedSwarmSlots` patrol (`git show 59f1e6cd4a^:src/lib/cloister/deacon.ts:2802`). No new HTTP surface.
- **D3 — CLI is a thin trigger.** `pan swarm <id>` validates swarm-readiness and marks the issue's swarm active (or nudges the first wave); the deacon does the ongoing coordination. No client-side coordination logic.
- **D4 — Reactive fast-path is additive, patrol is the safety net.** An optional `agent.stopped` branch (`src/lib/cloister/service.ts:423`) calls `coordinateSwarmSlots({issueId})` for near-instant advance; the 60s patrol remains the backstop (mirrors how `checkPendingTestDispatch` backs up the reactive review/test handoffs).
- **D5 — Scope split.** v1 = local slots (this epic). Remote Fly slots = Phase 2 = PAN-1773 (depends on this + PAN-1676). Model/difficulty routing = PAN-1791's axis, not here.
- **D6 — Recovery is first-class (PAN-1194).** A slot whose branch cannot merge blocks auto-advance until an explicit retry/drop/handoff. Never silently halt; never silently skip.
- **D7 — Per-slot verification is tests-only; the aggregate review is load-bearing.** A slot merges on its `verify_commands` passing, not on a per-slot code review. Per-unit review is deliberately out of scope for v1 — the issue's single downstream PR is reviewed once, on the merged result. Trade: throughput (one PR / one review) over per-unit review; cross-slot interaction bugs surface at fan-in, so the downstream review carries the weight. (deft directive gates *every* unit's PR on review-clean before integrating; we diverge for autonomy/throughput — see §9.)

## 3. Architecture

### 3.1 The deacon tick pass — `coordinateSwarmSlots()`

Insertion point in `runPatrol()` (`src/lib/cloister/deacon.ts:2571`), in the merge-check cluster immediately after `checkFailedMergeRetry` (`deacon.ts:2899`) and before `reconcileStaleMergeStatus` (`deacon.ts:2904`), matching the existing idiom at `deacon.ts:2894-2901`:

**Before** (`deacon.ts` ~2899-2904):
```ts
    const failedMergeRetryActions = await checkFailedMergeRetry();
    actions.push(...failedMergeRetryActions);
    for (const a of failedMergeRetryActions) addLog('action', a, state.patrolCycle);

    const staleMergeActions = reconcileStaleMergeStatus();
```

**After:**
```ts
    const failedMergeRetryActions = await checkFailedMergeRetry();
    actions.push(...failedMergeRetryActions);
    for (const a of failedMergeRetryActions) addLog('action', a, state.patrolCycle);

    // PAN-2203: deterministic swarm coordination — detect completed slots,
    // verify+merge them, advance the wave, recover failed slots. Derives all
    // state from git+tmux each tick (no persisted SwarmState).
    const swarmActions = await coordinateSwarmSlots();
    actions.push(...swarmActions);
    for (const a of swarmActions) addLog('action', a, state.patrolCycle);

    const staleMergeActions = reconcileStaleMergeStatus();
```

`coordinateSwarmSlots` is imported + re-exported alongside the other cloister passes (`deacon.ts:35` import cluster / `deacon.ts:142` re-export cluster), following the `deacon-merge.ts` module pattern.

### 3.2 Each tick, per swarm

1. **Enumerate swarms.** `listFeatureWorkspaces()` (`deacon.ts:878`) → `{issueId, workspacePath}`; skip `-slot-N` sub-workspaces; keep issues whose main-side vBRIEF has `analyzeSwarmReadiness(doc).swarmEligible === true`. (Replaces the deleted `discoverActiveSwarmIssueIds`.)
2. **Reconcile / detect completion.** `reconcileSlotState(issueId, workspace, doc)` (`slot-reconcile.ts:52`) → `{merged, inFlight, pending}` from live git slot branches (`listSlotBranches`) + tmux slot agents (`listSlotAgents`) + slot-eligible items. For ready-vs-crashed, cross-check `listSessionNames()` (`tmux.ts:970`) + `isPaneDead()` (`tmux.ts:1237`) + pane exit status against `agent-<issue>-slot-N` (the signals the deleted `refreshSwarmSlotStatuses` used).
3. **Verify + merge ready slots.** For each slot whose agent pane exited `0` and whose branch is unmerged: `verifyAndMergeSlot(issueId, slotIndex, item)` (`slot-merge.ts:36`) — runs `item.metadata.verify_commands` in the slot workspace, then `git merge --no-ff feature/<issue>-slot-N` into the feature workspace. On `merged:true`, apply the canonical vBRIEF `done` for the item via `applyTaskOperationToPlanFile` (the write door; the same op the deleted `onSlotMergeComplete` made). On `conflicts:true`, emit a failed-merge action and leave the item claimed for recovery (D6).
4. **Next-wave dispatch.** `mergedItemIds` = reconcile `merged` bucket → `getDispatchableItems(doc, mergedItemIds)` (`dag.ts:217`). For each newly-ready item with a free slot index and no live slot agent: run the **capacity+overlap selection** (defer by `assertRegisteredSlotCap` ceiling or by `files_scope` overlap against running+selected items — from `analyzeSwarmReadiness().overlapMatrix`), reserve budget with `tryReserveAdvancingSlot()` (`deacon.ts:147`), claim the item via `applyTaskOperationToPlanFile` type `claim`, then `spawnRun(issueId, 'work', { slotIndex, slotItemId })` (`spawn.ts:78`); release the claim + budget on spawn failure.
5. **Idempotency + backoff.** Module-level per-slot-branch cooldown map (like the deleted `recentSlotMergeFires`/`SLOT_MERGE_REFIRE_COOLDOWN_MS`) so an in-flight merge is not re-fired within the tick window. After ≥3 consecutive advance failures for an issue, set a 60s cooldown before retrying (like `recordAutoAdvanceFailure`/`isAutoAdvanceCoolingDown`).

### 3.3 `pan swarm <id>` (CLI trigger, D3)

New `src/cli/commands/swarm.ts` registered in `src/cli/index.ts`. `pan swarm <id>`: resolve id → load main-side plan → `analyzeSwarmReadiness`; if not `swarmEligible`, print why and exit non-zero; else ensure the feature workspace exists (reuse `pan start`'s workspace setup) and dispatch wave 0 (call the same first-wave selection the deacon uses), then print the dispatched slots and hand ongoing coordination to the deacon. `pan swarm recover <id> <slotIndex> --action retry|drop|handoff` mirrors the deleted `recoverSwarmCommand`. Wrapper skill `/pan-swarm` restored (`skills/pan-swarm/SKILL.md`) and cross-checked by `scripts/lint-skills.sh`.

## 4. Requirements

### Functional
- **FR-1** `pan swarm <id>` triggers a swarm for a swarm-ready plan; refuses (non-zero, with reason) a non-eligible plan.
- **FR-2** Each deacon tick enumerates active swarms (feature workspaces whose vBRIEF is `swarmEligible`).
- **FR-3** Slot completion is detected deterministically from tmux pane exit status (0 → ready-to-merge; non-zero / vanished → failed).
- **FR-4** A completed slot's branch is verified (`verify_commands`) then `--no-ff` merged into the feature workspace; on success the vBRIEF item is marked `done` via the write door.
- **FR-5** Next-wave dispatch selects newly-dispatchable items, deferring by concurrency ceiling and by `files_scope` overlap against in-flight/selected items.
- **FR-6** The vBRIEF item is `claim`ed before slot spawn and released (`unblock`) if spawn throws — no orphan claims.
- **FR-7** Slot-branch unmergeability (conflict) is detected (git merge dry result / `gh pr` state) and transitions the slot to `failed-merge`.
- **FR-8** Failed-merge recovery supports retry / drop / handoff; a `failed-merge` slot blocks auto-advance until resolved (PAN-1194). ← folds PAN-1336, PAN-1447.
- **FR-9** Auto-advance is idempotent: no double-dispatch of a `(slot,item)`; per-slot-branch cooldown; per-issue backoff after repeated failures. ← folds PAN-1336.
- **FR-10** DAG convergence items (>1 blocking parent) run a synthesis-phase slot first, whose output context feeds the implementation slot.
- **FR-11** Overlapping-scope items in the same wave are serialized (never dispatched concurrently). ← folds PAN-1193.
- **FR-12** Terminal (merged) slot worktrees + branches are GC'd after a wave converges. ← folds PAN-1202.
- **FR-13** Each slot has a distinct, navigable dashboard identity (`agent-<issue>-slot-N` surfaced on the issue's pipeline row / drawer). ← folds PAN-1263.
- **FR-14** Inspection/review verdicts route to the owning slot agent, not a missing parent. ← folds PAN-1472.
- **FR-15** An optional `agent.stopped` reactive fast-path advances a swarm without waiting for the next patrol.
- **FR-16** `docs/SWARM.md` (or equivalent) is rewritten to describe swarm-v2 + this orchestrator. ← folds PAN-1555.
- **FR-17** A slot that is **pane-alive but not progressing** (no new commit on its branch AND no pane output within a configurable threshold) is detected as `stalled` and escalated to recovery. Completion detection must not rely on pane-exit alone — a wedged-but-alive slot must never hang the wave. *(We have hit exactly this — the kimi wedged-vs-live-pane problem; deft directive solves it with a heartbeat/staleness contract — §9.)*
- **FR-18** Before (re)spawning a slot for an item, verify no live slot agent **and** no unmerged slot worktree already exist for it — never double-spawn onto a reconnecting/paused slot (two agents on one worktree corrupt the tool_use/tool_result chain). *(deft directive's "Duplicate-Agent Failure Mode" doctrine.)*

### Non-functional
- **NFR-1** Derive-don't-store: no new persisted swarm state; all runtime state re-derived from git+tmux+vBRIEF each tick (D1).
- **NFR-2** Loop-safety: idempotent; bounded by `assertRegisteredSlotCap`; per-slot cooldown; no re-dispatch storms. (Cf. the 24,626-call incident the in-flight guards protect against.)
- **NFR-3** Server-reachable code (the deacon is) uses **async** primitives only: `sendKeysAsync` never `sendKeys`; `execAsync`/`spawn` never `execSync`; no event-loop blocking.
- **NFR-4** Every retry/backoff/cooldown/delay test uses `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()` — never real `setTimeout`.
- **NFR-5** Single write door: vBRIEF item status is changed only via `applyTaskOperationToPlanFile`; the main-side spec is never mutated directly (PAN-1124 invariant).
- **NFR-6** Worktree discipline: slot worktrees created via `ensureRegisteredSlotWorktree`; never `git checkout` inside a workspace.
- **NFR-7** Builds + runs under Node 22 (deacon runs in the dashboard server bundle); `npm run build` before any restart; never run the server under Bun.

## 6. Intersecting repo rules (restated — do not assume recall)

- **Async tmux only.** Use `sendKeysAsync` from `src/lib/tmux.ts`; never `sendKeys`. Server-reachable code (the deacon) must never use the sync variants.
- **No `execSync` in server-reachable code.** Use `execAsync`/`spawn`; for delays `await new Promise(r => setTimeout(r, ms))`. `execSync` freezes the deacon's event loop (PAN-70).
- **Fake timers for delay/retry/backoff tests** (NFR-4) — `vi.useFakeTimers()` + `advanceTimersByTimeAsync`; never real `setTimeout`. A suite that "needs" `maxForks:1` is hiding this bug.
- **Single write door / single-spec-on-main.** vBRIEF item status changes only through `applyTaskOperationToPlanFile`; the main-side spec is immutable after planning (PAN-1124). Never write a store directly.
- **Worktree discipline.** Slot worktrees via `ensureRegisteredSlotWorktree`; never `git checkout` inside a workspace; verify branch before edits.
- **Node 22 only for the dashboard/deacon.** `npm run build` before restart; never run under Bun (node-pty + circular ESM).
- **Idempotency guards are structural.** Keep the post-merge/in-flight guard tests green (`tests/unit/lib/cloister/in-flight-guard.test.ts`); this loop must not reopen a re-dispatch cycle.

## 9. Prior-art cross-check — deft directive (`/home/eltmon/Projects/directive`)

deft directive (deftai, authors of the vBRIEF spec) ships a mature swarm as a **human-supervised monitor agent** running a 113 KB skill (`skills/deft-directive-swarm/SKILL.md`) + deterministic `task swarm:*` Python helpers (`scripts/swarm_readiness.py` et al.). No daemon; approval gates at allocation/assignment/release. Independent-design agreement + deliberate divergences:
