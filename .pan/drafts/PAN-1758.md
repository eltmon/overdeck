# PAN-1758 — UAT-off merge lane: continuous, deterministic convergence

**Issue:** [PAN-1758](https://github.com/eltmon/panopticon-cli/issues/1758) — bug(cloister):
ship lane cannot converge on a continuously-moving main.

> Author note (planning, `--auto`): the issue's original "Fixes (layered)" #1–#3
> targeted an **agent-based ship lane** that no longer exists. The ship patrol
> that caused the 37 re-dispatches was removed in commit `30f545272`
> (2026-06-10) and PAN-1747 (missing `roles/ship.md`) is **completed**
> (`a6410d602`). Shipping is now **server-side**. This plan therefore implements
> the operator's **RUN-35 design comment** (2026-06-14): make the UAT-off merge
> lane converge **continuously and deterministically**, the way UAT-on already
> does via the `uat-reconciler`.

## Glossary

- **readyForMerge** — boolean on a `ReviewStatus` row (`src/lib/review-status.ts:61`).
  Since PAN-1650 it is **event-driven**: re-derived from `gatesPassed`
  (review=passed, test=passed/skipped, verification not failed, no blockers,
  mergeStatus pending/queued) on every `setReviewStatus` write
  (`review-status.ts:268-295`).
- **pending_auto_merge** — a row in the `pending_auto_merges` SQLite table
  (`src/lib/database/schema.ts:508`). One **active** (`pending`/`merging`) row per
  issue (unique index). The queue entry that drives a UAT-off auto-merge.
- **auto-merge executor** — `tickAutoMergeExecutor`
  (`src/dashboard/server/services/auto-merge-executor.ts`), a 30 s interval
  (`AUTO_MERGE_EXECUTOR_INTERVAL_MS = 30_000`) that picks up **due** `pending`
  entries, checks `isAutoMergeEligible`, and calls `triggerMerge`. This is the
  **UAT-off conveyor**.
- **triggerMerge** — `src/dashboard/server/routes/workspaces.ts:4861`. The
  authoritative merge: serializes per-project (PAN-632 SQLite queue), rebases the
  feature branch onto current main, re-runs quality gates, then squash-merges.
  Already rebases + re-verifies at merge time — a **stale-but-clean** PR merges
  fine when its turn comes.
- **isAutoMergeEligible** — `src/lib/cloister/auto-merge-eligibility.ts:50`.
  Gate: readyForMerge=true, autoMerge≠false, PR not merged/closed/draft, CI not
  failing/pending, mergeable≠false, no blocker label.
- **startup repair sweep** — `fixStuckReadyForMerge`
  (`src/lib/review-status.ts:510`), called **once at boot** (`main.ts:525`).
  Restores readyForMerge for review+test-passed issues stuck at false (excludes
  `mergeStatus='failed'`).
- **strand** — a queue entry left in a terminal-ish state (`blocked`/`failed`)
  that nothing ever moves back to `pending`, so the executor never retries it.
- **transient vs terminal ineligibility** — *transient* = will likely clear on
  its own (CI pending, GitHub `mergeable` still computing, a momentary
  not-mergeable while main moves, a recoverable merge attempt). *terminal* = a
  human/recovery is required (PR closed/merged, held-for-UAT, blocker label,
  genuine persistent conflict).
- **PAN-1691 merge order** — `orderMergeCandidates` / `planMergeTrain`
  (`src/lib/flywheel-merge-order.ts:39,64`): disjoint (no-conflict) candidates
  first; conflicting ones broadest-file-footprint first; issue number tiebreak.
- **UAT-on vs UAT-off** — when `flywheel.require_uat_before_merge=true`, ready
  features feed the `assemble-uat` batch branch (human ships); the
  `uat-reconciler` continuously rebuilds it onto current main, so it stays
  converged. When false, ready PRs feed **direct auto-merge** via the executor.

## Problem

The UAT-off auto-merge lane cannot converge while main moves continuously
(flywheel tick-docs / beads-sync / strike commits land on main every few
minutes). The concrete mechanism, verified against current code:

1. A ready PR is scheduled into `pending_auto_merges` (5 min cooldown).
2. When the executor tick fires, `isAutoMergeEligible` can transiently fail —
   CI re-running after a main move (`checksPending`), GitHub `mergeable` still
   computing, or a momentary not-mergeable. The executor calls **`markBlocked`**
   → row status `blocked`.
3. **Nothing ever moves a `blocked`/`failed` row back to `pending`.**
   `requeueToPending` only transitions `merging`→`pending`;
   `listDuePendingAutoMerges` only selects `pending`. The row **strands**.
4. Recovery today is non-deterministic: either the **LLM flywheel** happens to
   re-POST `/api/flywheel/auto-merge/schedule` (a fresh `pending` row, since the
   active-index excludes `blocked`), or the **server restarts** and the startup
   sweep + a re-schedule resurrect it. Hence the title symptom: *readyForMerge
   only ever flips via the startup repair sweep.*

`roles/flywheel.md:88` even instructs the LLM: *"The failed/blocked entry stays
in the table until cancelled; do not auto-clear it"* — i.e. the LLM is the
fragile, manual convergence engine today (the RUN-35 "rolling-rebase
whack-a-mole").

UAT-on does not have this problem because the `uat-reconciler` continuously
rebuilds the batch branch onto current main. UAT-off has no equivalent
self-healing loop.

## Proposal

Make the **auto-merge executor self-heal**, so convergence is continuous and
deterministic — no server restart, no LLM re-scheduling required. Reuse the
existing engine; add no new subsystem (Simplicity First). Six changes:

1. A pure **classifier** that labels each `isAutoMergeEligible` ineligibility
   reason `retryable` or `terminal`.
2. A **schema migration** adding an `attempts` counter to `pending_auto_merges`,
   plus DB primitives to defer a `pending` row (push out `scheduledMergeAt`) and
   resurrect a stranded `blocked`/`failed` row to `pending`.
3. Executor: on **transient** ineligibility, **defer** the still-`pending` row
   with exponential backoff (cheap GitHub re-check — *not* counted against the
   merge cap); on **terminal** ineligibility, `markBlocked` as today. Cap
   **expensive failed merge attempts** (a `triggerMerge` that returned failure)
   at **3**, then surface "stuck" (TTS + activity + the problems list), mirroring
   `checkPendingTestDispatch`'s 3-retry cap.
4. Executor: each tick, **resurrect** stranded `blocked`/`failed` rows that are
   **eligible again** back to `pending` (respecting operator-`cancelled` and the
   attempt cap). This is the deterministic replacement for the startup sweep /
   LLM re-schedule.
5. Executor: when **multiple** rows are due, process them in **PAN-1691 order**
   (`orderMergeCandidates`) — disjoint first, conflicts broadest-footprint first
   — instead of FIFO.
6. **Docs / role-doc** update carving the actor boundary: the **server** owns
   transient retry + resurrection of recoverable entries; the **LLM flywheel**
   investigates only genuinely-stuck (capped) entries — so they never double-act
   on the same row.

The convergence-vs-churn trap is avoided structurally: the executor **never
spawns a review convoy**. Transient ineligibility is a cheap GitHub state
re-check. The expensive path (`triggerMerge`'s rebase + quality gate) runs only
when GitHub already reports the PR green + mergeable, and `triggerMerge`
preserves verification across a clean rebase. Genuine conflicts are out of scope
(PAN-1240). Issue fix #1 ("don't require exact main-currency") is **already
satisfied** by `triggerMerge` rebasing at merge time + eligibility keying on
GitHub `mergeable` (not exact-currency) — no change needed.

## Requirements

### Functional

- **FR-1** — A pure function classifies every `AutoMergeEligibility` ineligibility
  reason as `retryable` or `terminal`.
- **FR-2** — On **transient** (retryable) ineligibility, the executor keeps the
  entry `pending` and pushes out `scheduledMergeAt` with exponential backoff
  instead of stranding it via `markBlocked`.
- **FR-3** — A `triggerMerge` attempt that returns failure is retried up to **3**
  times; on the 3rd failure the entry is marked terminally blocked and surfaced
  as "stuck" (TTS + activity entry + `GET /api/flywheel/auto-merge/problems`).
- **FR-4** — Each tick, stranded `blocked`/`failed` entries that are **eligible
  again** (and under the attempt cap, not operator-`cancelled`) are resurrected
  to `pending`; convergence requires neither a server restart nor an LLM
  re-schedule.
- **FR-5** — When ≥2 entries are due in one tick, they are processed in
  `orderMergeCandidates` order (disjoint first, then conflicting
  broadest-footprint first), not FIFO.
- **FR-6** — `roles/flywheel.md` (and any operator-facing doc that describes the
  auto-merge problems loop) states the server/LLM boundary: server owns
  transient retry/resurrection; the LLM acts only on genuinely-stuck entries.

### Non-functional

- **NFR-1** — The executor never spawns a review/test convoy; clean orthogonal
  main movement does not trigger re-review (no churn).
- **NFR-2** — Every retry/backoff/interval test uses `vi.useFakeTimers()` +
  `vi.advanceTimersByTimeAsync()` (repo rule); never real waits, never
  `maxForks: 1`.
- **NFR-3** — All new server-reachable code uses async `exec`/`spawn`; no
  `execSync`.
- **NFR-4** — Convergence is server-side and deterministic — it does not depend
  on the LLM flywheel re-POSTing `auto-merge/schedule`.

## Planning Q&A (auto-mode decisions)

`--auto` mode: no interactive questions. Decisions taken and their rationale:

- **Scope = operator's RUN-35 direction, not the issue body's #1–#3.** Git proves
  the agent-based ship lane was removed (`30f545272`) and PAN-1747 is done
  (`a6410d602`); the operator launched this `--auto` plan *after* writing the
  RUN-35 comment (`962e6e0b7`). Building the continuous-convergence direction.
- **No new reconciler subsystem; extend the existing `auto-merge-executor`.** The
  lifecycle trace shows the strand is a queue-entry problem; `triggerMerge`
  already rebases at merge time. Extending the executor is the root-cause fix.
- **Drop the originally-planned "continuous `fixStuckReadyForMerge`" idea.** In
  current code the flag is event-driven and `markBlocked` never touches it;
  genuine-conflict failures set `mergeStatus='failed'` + `blockerReasons`, which
  the sweep deliberately ignores. A continuous sweep would be a no-op for every
  in-scope case. The fix lives entirely in the queue lane.
- **Two cap regimes.** Cheap eligibility re-checks back off generously (GitHub
  polls, no agent spawn); only expensive `triggerMerge` failures are hard-capped
  at 3 — honoring issue fix #2 without recreating "37 attempts."

## Non-goals

- Active rebasing of genuinely-**conflicting** ready PRs — **PAN-1240** (recovery half).
- Conflict-flagged churn handling — **PAN-1765**.
- Decoupling the merge train into a per-project pipeline feature — **PAN-1696**.
- The agent-based ship lane / `roles/ship.md` — **PAN-1747** (already removed/done).
- UAT-on (`require_uat_before_merge`) convergence — already handled by the
  `uat-reconciler`; this issue is **UAT-off only**.
- Changing `triggerMerge`'s rebase or quality-gate behavior.
- Auto-clearing operator-`cancelled` entries.
