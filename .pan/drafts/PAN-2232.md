## 2. Locked design decisions

- **D1 — Pure moves only.** Every function moves verbatim with its imports; no logic
  edits, no renames, no signature changes, no visibility changes. If a move seems to
  require an edit, stop — the seam is wrong; pick the boundary that doesn't.
- **D2 — Six seam modules** (names + contents in §5), each well under 1,000 lines;
  `specialists.ts` ends as a barrel (≈100 lines of re-exports) and its baseline entry
  is LOWERED.
- **D3 — `loadRegistry`/`saveRegistry`/`migrateRegistryIfNeeded` move together** into
  the registry seam — one reader/writer pair, one module (single-write-surface shape).
- **D4 — Private helpers move with their (only) callers.** E.g.
  `deterministicUUID` → registry seam; `scheduleLogCleanup` → lifecycle seam;
  `formatFeedbackForAgent` → feedback seam; `readRecordedClaudeSessionId` → spawn seam.
  If a private helper turns out to have callers in TWO seams, put it in the seam of its
  primary caller and export it to the other — note it in the commit message.
- **D5 — Bundler-cycle tolerance as in PAN-2149 D6:** `npm run build` per seam is the
  real cycle check; PAN-2230's ratchet (if live) gets `--update` per seam with
  additions audited by this issue's ref; a runtime ESM cycle error = stop and report.
