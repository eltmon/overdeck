## 0. Glossary (read first)

Define every term of art before it is used.

| Term | Meaning in this document |
| --- | --- |
| **Durable state** | State that must survive a process restart and (per the State Model) be derivable from / mirrored into git. The per-issue resume/progress record, the immutable plan spec, and agent runtime state are durable. By contrast `output.log`, `health.json`, `session.id`, `kickoff` prompt files, and tmux transcripts are **runtime scratch**, not durable state. |
| **State file** | A file on disk that holds durable state: `.pan/continue.json` (workspace), `.pan/continues/<issue>.vbrief.json` (project), `.pan/records/<issue>.json` (per-issue record), `.pan/specs/<…>.vbrief.json` (immutable plan), and `~/.panopticon/agents/<id>/state.json` (agent runtime). |
| **Write surface** | The small set of functions that are *allowed* to call a raw filesystem write/rename primitive and land bytes at a state-file path. Everything else must call one of these functions, never `fs.writeFile*` / `fs.rename` directly. |
| **The record writer** | The single approved writer for the per-issue durable record. After [PAN-1919](https://github.com/eltmon/overdeck/issues/1919) lands it is expected at `src/lib/pan-dir/record.ts`. Until/unless #1919 renames it, the existing writer is `writeIssueRecordSync` in `src/lib/pan-dir/records.ts:240` (verified 2026-06-16). See **§2 Precondition / checkpoint.** |
| **Ad-hoc write** | A call to a raw fs write/rename primitive (`fs.writeFileString`, `fs.writeFileSync`, `fs.rename`, `fs.renameSync`) that lands at a state-file path from a file that is **not** on the approved write surface. This is exactly what this issue forbids and what the guard detects. |
| **Raw fs write/rename primitive** | One of: `writeFileString` / `writeFile` (Effect `FileSystem`), `writeFileSync` / `writeFile` / `writeFileString` (node `fs`), `rename` / `renameSync` (the second half of the atomic tmp-write-then-rename pattern — this is the call that actually lands bytes at the state path). |
| **Atomic tmp-write-then-rename** | The dominant write idiom in `pan-dir`: write to `<path>.tmp`, then `rename(tmp, path)`. The byte that *lands at the state path* is the **rename**, not the `writeFileString` to the tmp file. The guard must account for both halves. |
| **Guard** | An automated, CI-wired test/script that **fails** when an ad-hoc write is introduced, so the invariant cannot silently regress. Modeled on `scripts/lint-permissions.sh` (a git-grep allowlist guard wired into `npm run lint`). |
| **No-loss audit** | A focused test asserting that routing a write site through the approved writer **drops no field** that the old ad-hoc site persisted. Answers "is anything lost?" mechanically, not by reasoning. |
| **PAN-1124 single-spec-on-main** | The plan spec (`.pan/specs/`) is immutable after planning; only `plan.status` may change via `updateSpecStatus`. The spec is therefore a *legitimate single-writer* state file — in scope here only to **confirm** it routes through one writer (`writeSpec` in `specs.ts`), never to make it mutable. |

## 1. Problem

The State Model (`reference/state-model.mdx:71`) states the invariant plainly:

> **All state mutations go through the database API** — never ad-hoc writes to scattered files. The database is the single write surface; the per-issue git record is its durable mirror.

Today that invariant is **documented but not mechanical.** Durable state is written directly to several scattered files (workspace continue, project continue, agent `state.json`, and — legitimately, but unverified — the spec). When two writers touch the same conceptual state, the SQLite cache and the filesystem drift out of sync. Nothing fails a build when a new ad-hoc write is added, so the invariant erodes silently.

This issue makes the invariant **mechanical**: a single approved write surface, plus a CI-wired **guard** that fails the moment a new ad-hoc state-file write appears outside that surface, plus a **no-loss audit** so any routing this issue performs persists every field the old site did.

## 5. The guard design (load-bearing — specify precisely)

A naive "ban `writeFile*` to a state path" grep fails here for two reasons, both confirmed in the inventory:

1. **Dynamic paths.** The writes target *variables* (`continuePath`, `path`, `contextPath`), not string literals — you cannot grep for "writeFile to a state-path literal."
2. **Atomic tmp-write-then-rename.** The call that lands bytes at the state path is the **`rename`**, not the `writeFileString` to the `.tmp` file. A guard that only watches `writeFile*` misses the landing write.

Therefore the guard uses a **scope-based primitive ban with an allowlist**, exactly mirroring `lint-permissions.sh`'s shape (git-grep candidates → perl filter → allowlist of files):

### 5.1 Rule 1 — `pan-dir` raw-primitive ban

Within `src/lib/pan-dir/*.ts` (and the sync continue writer `src/lib/vbrief/io.ts`), the raw fs write/rename primitives —
`writeFileString`, `writeFileSync`, `writeFile(`, `renameSync`, `.rename(` —
**may appear ONLY in the allowlisted writer files.** Allowlist (the approved write surface):

```
src/lib/pan-dir/record.ts        # the record writer (or records.ts pre-rename — see §2 fallback)
src/lib/pan-dir/specs.ts         # immutable plan single-writer (PAN-1124)
src/lib/pan-dir/auto-commit.ts   # commit fan-in helper, not a state write
src/lib/pan-dir/drafts.ts        # OUT OF SCOPE — human PRD markdown
src/lib/pan-dir/context.ts       # OUT OF SCOPE — context-layer config
# feedback.ts / sessions.ts: include ONLY if #1919 left them as separate single-writers (WI-1 checkpoint)
# continue.ts / continues.ts / vbrief/io.ts continue writer: NOT allowlisted —
#   they must be retired by #1919; any surviving primitive here is a violation.
```

Any raw primitive in `src/lib/pan-dir/*.ts` outside this allowlist is a violation. This catches "someone added a new scattered state file under pan-dir." Note that path-scoping to `pan-dir/` is **deliberately insufficient on its own** for the continue file — its sync writer lives at `src/lib/vbrief/io.ts`, outside `pan-dir/` — so the continue file is guarded by the literal-keyed Rule 3 below, not by this path scope.

### 5.2 Rule 2 — `agents.ts` `state.json` single-writer

Key on the **`'state.json'` literal** rather than a blanket `writeFile*` ban (so the many legitimate `agents.ts` runtime writes — `output.log`, health, `session.id`, kickoff — are not false-positives). A quoted `'state.json'` co-located with a write/rename primitive may appear only inside `writeAgentStateJsonSync`.

> **Matcher must be multiline-aware.** The verified write is split across lines (`agents.ts:1110-1111`):
> ```
> 1110  writeFileSync(
> 1111    join(dir, 'state.json'),
> ```
> A line-oriented grep for `writeFileSync.*'state.json'` matches **zero** lines and silently passes — do not ship that. Concrete strategy the executor must implement: (1) `git grep -n "'state.json'" -- src/lib/agents.ts` to find every line referencing the literal; (2) determine the `[start,end)` line range of `writeAgentStateJsonSync` (grep its `export function writeAgentStateJsonSync` opening line; find the next top-level `export function`/`export const` line as the end bound, or brace-balance); (3) **fail** if any `'state.json'` line that is also within a write/rename emission (i.e. the literal appears in a `writeFileSync(`/`renameSync(`/`writeFileString(` argument list — detect by checking the literal's line OR the immediately preceding non-comment line for a write-primitive token, since the call may be multiline) falls **outside** that range. The function-scope check is the load-bearing part; spell it out in awk/perl, do not leave it as a placeholder.

### 5.3 Rule 3 — continue-file literal guard (catches the off-`pan-dir` regression)

The workspace continue file is mutated by **two** writers — the Effect writer `writeWorkspaceContinue` (`src/lib/pan-dir/continue.ts`) and the sync writer `writeWorkspaceContinueSync` (`src/lib/vbrief/io.ts:113-114`) — and the sync one lives outside `pan-dir/`, so Rule 1's path scope cannot see it. Both are **retired by #1919**, so the correct post-#1919 allowlist for the continue literal is **empty**. Guard it the same way as Rule 2 but scoped to all of `src/`: grep across `src/` for a write/rename primitive co-located with the continue-file literal (`PAN_CONTINUE_FILENAME`, or the string `continue.json`, or `continuePath` assigned from `PAN_CONTINUE_FILENAME`) and **fail on any match** outside the allowlist. This catches a continue-write regression wherever it reappears — `pan-dir/`, `vbrief/io.ts`, or a new file — which path-scoping alone would miss. (If WI-1 finds #1919 left a read-only legacy continue *reader* with no write primitive, that is fine and does not match.)

### 5.4 Phase-2 comment filter (copy from lint-permissions.sh)

All three rules reuse the `perl` filter shape from `lint-permissions.sh:76-97`: skip lines beginning with `//`, `*`, or `/*` so prose/JSDoc mentioning a primitive name is not a false positive.

### 5.5 Self-verification the guard must satisfy (state in the script header)

- Running against the **current tree** (post-#1919) prints `✓` (no violations) — the allowlist exactly covers the legitimate writers.
- (Rule 1) Adding a test fixture line `fs.writeFileString(somePath, x)` to a **non-allowlisted** `pan-dir` file makes it print a violation and `exit 1`.
- (Rule 2) Adding a **multiline** `writeFileSync(\n  join(dir, 'state.json'), x)` outside `writeAgentStateJsonSync` makes it `exit 1` (proves the matcher is multiline-aware, not line-oriented).
- (Rule 3) Adding `writeFileSync(continuePath, x)` to `src/lib/vbrief/io.ts` **or any file outside `pan-dir/`** makes it `exit 1` (proves the continue guard is not path-scoped to `pan-dir/`).
- A legitimate `agents.ts` write (`writeFileSync(join(agentDir, 'output.log'), …)`) does **not** trigger it.

#!/usr/bin/env bash
#
# lint-state-writes.sh — repo-wide guard: all durable state mutations go
# through one approved write surface; never an ad-hoc fs write/rename to a
# state file. Enforces reference/state-model.mdx. See PAN-1921 / PAN-1919.
#
# Approved write surface (raw fs primitives may appear here only):
#   - src/lib/pan-dir/record.ts (or records.ts pre-#1919-rename) — the record writer
#   - src/lib/pan-dir/specs.ts        — immutable plan single-writer (PAN-1124)
#   - src/lib/pan-dir/auto-commit.ts  — commit fan-in, not a state write
#   - src/lib/pan-dir/drafts.ts, context.ts — OUT OF SCOPE (human/config files)
#   - src/lib/agents.ts writeAgentStateJsonSync — the sole state.json writer
set -euo pipefail
cd "$(dirname "$0")/.."

# ── Rule 1: pan-dir raw-primitive ban outside the writer allowlist ──
EXCLUDES=(
  ':!src/lib/pan-dir/record.ts'
  ':!src/lib/pan-dir/records.ts'        # fallback writer name pre-#1919 rename
  ':!src/lib/pan-dir/specs.ts'
  ':!src/lib/pan-dir/auto-commit.ts'
  ':!src/lib/pan-dir/drafts.ts'
  ':!src/lib/pan-dir/context.ts'
  ':!src/lib/pan-dir/__tests__/*'
)
candidates=$(
  { git grep -nE -e 'writeFileString' -e 'writeFileSync' -e 'renameSync' -e '\.rename\(' \
      -- 'src/lib/pan-dir/' "${EXCLUDES[@]}"; } || true
)
# Phase-2 perl filter: drop comment/JSDoc lines (copy lint-permissions.sh:76-97).
violations=$( printf '%s\n' "$candidates" | perl -ne '
    next unless length;
    my ($prefix, $content) = /^([^:]+:\d+:)(.*)$/ ? ($1, $2) : ("", $_);
    next if $content =~ m{^\s*(?://|\*|/\*)};
    print;
' )
if [[ -n "$violations" ]]; then
  echo "✗ ad-hoc state write under src/lib/pan-dir/ outside the approved writer:" >&2
  echo "$violations" >&2
  echo "Route durable mutations through the record writer (src/lib/pan-dir/record.ts)." >&2
  exit 1
fi

# ── Rule 2: state.json single-writer in agents.ts ──
# MULTILINE-AWARE: the write is split across lines (agents.ts:1110-1111), so a
# single-line "writeFileSync.*'state.json'" grep matches NOTHING. Find every
# 'state.json' literal line, find the [start,end) line span of
# writeAgentStateJsonSync, and fail on any 'state.json' write/rename emission
# outside that span. awk does both the span-tracking and the emission check.
violation=$(
  awk '
    # Record the function span.
    /^export function writeAgentStateJsonSync\(/ { start=NR; depth=0; inFn=1 }
    inFn { depth += gsub(/{/,"{"); depth -= gsub(/}/,"}"); if (depth==0 && NR>start) { end=NR; inFn=0 } }
    # Track whether the current logical statement is a write/rename emission:
    # set a flag on a write-primitive token line; the literal may appear on this
    # or the next argument line.
    /writeFileSync|writeFileString|renameSync|\.rename\(/ { armed=NR }
    # On a state.json literal, flag a violation if we are armed (this or prev line
    # was a write primitive) AND we are outside [start,end).
    /'\''state\.json'\''/ {
      if ((armed==NR || armed==NR-1) && !(NR>=start && NR<=end)) {
        print FILENAME ":" NR ": state.json write outside writeAgentStateJsonSync"
      }
    }
  ' src/lib/agents.ts
)
if [[ -n "$violation" ]]; then
  echo "✗ state.json written outside writeAgentStateJsonSync:" >&2
  echo "$violation" >&2
  exit 1
fi

# ── Rule 3: continue-file literal guard (scans ALL of src/, not just pan-dir) ──
# The sync continue writer lives at src/lib/vbrief/io.ts, OUTSIDE pan-dir/, so
# Rule 1's path scope can't see it. Both continue writers are retired by #1919,
# so the post-#1919 allowlist for the continue literal is EMPTY. Fail on any
# write/rename primitive co-located with the continue-file literal anywhere in src/.
cont_candidates=$(
  { git grep -nE -e 'PAN_CONTINUE_FILENAME' -e "continue\.json" -e 'continuePath' \
      -- 'src/' ':!src/**/__tests__/*' ':!*.md'; } || true
)
cont_violations=$( printf '%s\n' "$cont_candidates" | perl -ne '
    next unless length;
    my ($path,$line,$content) = /^([^:]+):(\d+):(.*)$/ ? ($1,$2,$3) : next;
    next if $content =~ m{^\s*(?://|\*|/\*)};
    # Keep only lines that ALSO emit a write/rename primitive on the continue path.
    next unless $content =~ /writeFileSync|writeFileString|renameSync|\.rename\(/;
    print "$path:$line:$content\n";
' )
if [[ -n "$cont_violations" ]]; then
  echo "✗ ad-hoc write to the workspace continue file (retired by #1919):" >&2
  echo "$cont_violations" >&2
  exit 1
fi

echo "✓ state-write lint passed (single write surface intact)"
```
*(The awk span-tracking in Rule 2 and the perl filters in Rules 1/3 follow the shape of `lint-permissions.sh`; the executor finalizes the brace-depth edge cases and proves all three with the WI-3 fixtures in §5.5.)*

## 8. Requirements

### Functional
- **FR-1** Every durable, cache-mirrored state mutation flows through exactly one approved writer: the record writer (per-issue record), `writeSpec` (immutable plan), `writeAgentStateJsonSync` (`state.json`). No other file lands bytes at a state-file path.
- **FR-2** A CI-wired guard (`scripts/lint-state-writes.sh`, in `npm run lint`) **fails** when a new ad-hoc raw-primitive write to a `pan-dir` state file appears outside the writer allowlist.
- **FR-3** The guard **fails** when a `state.json` write appears outside `writeAgentStateJsonSync`.
- **FR-4** The guard does **not** false-positive the legitimate single-writers, the out-of-scope human/config writers (`drafts.ts`, `context.ts`), or `agents.ts` runtime scratch writes (`output.log`, health, `session.id`, kickoff).
- **FR-5** Any write site this issue *routes* through the record writer drops no field the old ad-hoc site persisted (no-loss audit test).
- **FR-6** This issue does **not** re-implement [PAN-1919](https://github.com/eltmon/overdeck/issues/1919)'s continue→record rewrites nor its field-superset test; its no-loss audit is scoped to sites #1921 itself routes.
- **FR-7** If [PAN-1919](https://github.com/eltmon/overdeck/issues/1919) is not merged, the work stops at the §2 checkpoint and surfaces the blocker; no guard is shipped against an inconsistent tree.

### Non-functional
- **NFR-1** The guard is an **automated test/script**, not a convention or doc note (the issue is explicit: mechanical, not advisory).
- **NFR-2** The guard runs in the existing `npm run lint` chain with no new heavyweight dependency — git-grep + perl/awk, matching `lint-permissions.sh`.
- **NFR-3** Changes are surgical: no rewrite of the continue/record machinery owned by #1919, no touching of unrelated writers, no deletion of pre-existing dead code.
- **NFR-4** No `execSync` is introduced into `src/dashboard/server/`-reachable code (the guard is a standalone build script and is exempt).
