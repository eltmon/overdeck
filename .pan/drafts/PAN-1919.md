## Glossary

- **Record** — the single per-issue git-tracked file at
  `<workspace>/.pan/records/<issue-lowercase>.json`, written by
  `src/lib/pan-dir/record.ts`. The superset of all prior per-issue progress
  state. Lives on the **feature branch** (not `main`) and travels via `git push`.
- **Project-side continue** — legacy `<projectRoot>/.pan/continues/<issue>.vbrief.json`,
  committed on `main`. Holds `decisions`, `hazards`, `resumePoint`,
  `beadsMapping`, `sessionHistory`, `feedback`. 132 historical files exist.
- **Workspace-side continue** — legacy `<workspace>/.pan/continue.json`,
  **gitignored** (never travels). Holds `statusOverrides` (bead completion) and
  session state.
- **`state.json`** — machine-local `~/.panopticon/agents/<id>/state.json`. Held
  the only non-derivable fields (`harness` + `model`). Authoritative runtime
  registry is now the SQLite `agents` table (PAN-1908); `state.json` is a
  rollback/rebuild source.
- **statusOverrides** — map of `itemId → status` recording which beads/items are
  complete (the merged-view overlay on top of the immutable spec).
- **Single-writer discipline** — only the pipeline writes the record; agents
  never hand-edit it.
- **No-loss audit** — a focused test that blocks until the new record schema is a
  **superset** of every field in (project continue ∪ workspace continue ∪
  `state.json` harness/model). "Is anything lost?" is answered by the test.

---

## Problem

Per-issue resume/progress state is scattered across five places; two are a
confusing duplicate ("continues"), and the one non-derivable field
(`harness`/`model`) lived only in machine-local `state.json` that never travels.
An agent on a second machine (`git clone` + GitHub status only) cannot reliably
know what has been done, and a local DB loss strands resume state.

## Proposal

Consolidate into **ONE git-tracked per-issue record** on the feature branch.
The foundation already exists (commit `e9914c9429`). Close the remaining gaps so
that every write path lands in the record, the legacy continues become read-only
legacy, harness/model are sourced from the record on a fresh machine, and a
cross-machine resume is provable by test.

## Requirements

**Functional**

- **FR-1** — Every status-override write (sync CLI **and** async dashboard
  paths) persists to the per-issue record; no path writes `statusOverrides` to
  the workspace `continue.json`.
- **FR-2** — Every resume-context write (decisions, hazards, resumePoint,
  sessionHistory, feedback) across all surfaces (done, planning spawn, cloister
  prompt/handoff, dashboard routes) persists to the record; corresponding
  readers read from the record.
- **FR-3** — `harness` + `model` travel in the record and are the source on a
  fresh machine with no SQLite row and no `state.json` (already implemented;
  verified by FR-6 test).
- **FR-4** — A one-time backfill migrates each **in-flight** issue's
  (project-continue ∪ workspace-continue ∪ `state.json` harness/model) into its
  record at `schemaVersion: 2` with no field loss; historical closed continues
  remain read-only legacy.
- **FR-5** — Workspace `.pan/continue.json` and project `.pan/continues/` become
  read-only legacy: no production code path **writes** them; readers may remain
  for legacy/backfill.
- **FR-6** — Cross-machine: a record written on machine A, read on machine B with
  no DB and no `state.json`, resumes full harness/model/decisions/hazards/
  statusOverrides/sessionHistory from the record alone.

**Non-functional**

- **NFR-1** — The record is committed to the feature branch and stays off `main`
  during work (preserves the PAN-1124 single-spec-on-`main` invariant). Landing
  the record on `main` at merge is acceptable (consistent with how continues
  land today).
- **NFR-2** — A no-loss/no-remaining-writers assertion test blocks if any record
  field or any retired-writer regresses. (This is a one-shot test, **not** the
  enforced CI write-surface guard — that is PAN-1921.)
- **NFR-3** — Root-cause fixes only (no fallback chains tolerating the split
  brain). Async and sync paths converge on the same record writer.

## Planning decisions (`--auto`)

- **D1 — Name = `record`.** Already implemented (`record.ts`, `.pan/records/`).
  Unifies with the PAN-1908 records plane; the maintainer disliked "continues".
- **D2 — Pipeline verdicts stay as the existing `pipeline` mirror field; not
  expanded here.** The record schema already carries `pipeline`
  (`PanIssuePipelineRecord`, from PAN-1908). Making verdicts reconstruction-safe
  is **PAN-1922**. This issue keeps the record about *work* progress.
- **D3 — Backfill targets only in-flight issues** (live workspace/branch);
  the 132 historical closed continues remain read-only legacy and are not
  migrated. Resume only matters for in-flight work.
- **D4 — Continues become read-only legacy, not mass-deleted.** AC6's "removed
  **or** read-only legacy" escape hatch is taken to avoid destroying historical
  files and the backfill source.
- **D5 — No `pan_records` entry added to `projects.yaml`.** The record is
  branch-resident; its path resolves from the workspace, needing no config.
  Backfill's leftover infra-repo coupling is reconciled to the branch path.

## Non-goals (enforced by review as must-not constraints)

- CI guard and the enforced single-write-surface API are **PAN-1921**; this issue
  only stops legacy writes far enough to make the continues read-only.
- Reconstructing the cache from git + GitHub (boot replay replacement) is **PAN-1920**.
- Making pipeline verdicts reconstruction-safe is **PAN-1922**.
- Keeping agent transcripts (JSONL) in git; beads cleanup; cache pruning/VACUUM (PAN-1847).
- Rebuilding the record schema or writer (already landed in `e9914c9429`).
- Mass-deleting the 132 historical `.pan/continues/` files.

## Acceptance criteria (map 1:1 to work items)

- One git-tracked record file per issue replaces both continues as the write target.
- `harness`/`model` resume from the record on a fresh machine (no DB/state.json).
- No-loss test passes; no production path writes the legacy continues.
- Cross-machine: `git clone` + `pan start <issue>` resumes decisions/progress/status.
- Backfill migrates in-flight issues without field loss at `schemaVersion: 2`.
- `.pan/continues/` and workspace `.pan/continue.json` are read-only legacy.

## Hazards

- **H1 — Split-brain already shipped.** Until work item 1 lands, sync and async
  status-override writes disagree (record vs continue.json). Fix item 1 first.
- **H2 — Migrating planning-spawn persistence (item 3) changes the path that
  wrote this very session's continue.json.** Verify planning resume still works.
- **H3 — Backfill data loss risk** (item 6): a wrong field map permanently loses
  state once continues are retired (item 7). Hence item 6 is inspection-gated and
  item 7 is blocked on it.
- **H4 — Record lands on `main` at merge.** Acceptable (continues do today), but
  must not churn `main` mid-work — keep writes on the branch only.
