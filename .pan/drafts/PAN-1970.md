# PRD: Minimal New Project flow (PAN-1970)

> Part of the **minimum new-user onboarding** (demo readiness). **Explicitly NOT the full #654 wizard** â€” no Setup Agent, templates, archetypes, `SetupRpcGroup`, or `pan setup`. Clone-from-GitHub/GitLab is a **later** release.

## Summary
A minimal dashboard flow to create/register a project, with two modes:
1. **Add existing** â€” browse to an existing folder â†’ register it.
2. **New project** â€” type a name, pick a parent folder (default `~/Overdeck/<name>`) â†’ create folder + `git init` â†’ register.

End state: an entry in `~/.overdeck/projects.yaml`, visible in the sidebar + Home (PAN-1969), ready to open. Mirrors the reference flow in the issue screenshots (add existing / new project / default folder / choose parent folder â†’ Start workspace).

## Glossary
- **Project** â€” entry in `~/.overdeck/projects.yaml` (`ProjectConfig`).
- **registerProjectSync(key, config)** â€” the one helper that writes a project to projects.yaml.
- **Project key** â€” slug derived from name/folder: `name.toLowerCase().replace(/[^a-z0-9-]/g, '-')`.
- **Git initialization target** â€” the fully canonicalized directory path that will be passed as `cwd` to `git init`.

## Problem
Projects can only be registered via `pan project add <existing-path>` in a terminal (`src/cli/commands/project.ts:85`). That command **requires the path to already exist** and **does not run `git init`**. There is **no dashboard endpoint** to register a project, **no server-side folder browser**, and **no "create folder + git init"** path. A brand-new user has no UI way to get their first project in.

## Goal / Non-goals
- **Goal:** from PAN-1969's "+ New Project", a user can Add existing OR New project, ending with a registered project visible in sidebar + Home.
- **Non-goals (explicit):** Setup Agent, templates/archetypes, `SetupRpcGroup`, `pan setup` (those are #654); clone from GitHub/GitLab (later); tracker config / quality gates / workspace-build config in the wizard (use minimal defaults, editable later in projects.yaml).

## Requirements
### Backend
- **FR-1** â€” `POST /api/projects` â€” body `{ mode: 'existing' | 'new', path?, parentDir?, name? }`. Registers the project and returns the created project record.
  - `existing`: validate `path` exists; derive key; dup-check; `registerProjectSync`; `ensureProjectLayer`; `preTrustDirectory`.
  - `new`: target = `<parentDir>/<slug(name)>`; resolve and validate the final target itself before any write; reject symlinks, files, non-empty directories, existing Git repositories, and any canonical target outside the user's home; `mkdir -p`; `git init` (async); then the same register path.
  - Reuse the registration core from `projectAddCommand` (factor a shared helper rather than duplicating).
- **FR-2** â€” `GET /api/fs/list-dirs?path=<abs>` â€” returns immediate subdirectories of `path` (`{ name, path }[]`) plus the parent, for the folder picker. **Auth-gated** (internal token, like other routes). Default start path = `~` (or `~/Overdeck`). Must **not** traverse outside the user's home unless explicitly navigated there.
- **FR-3** â€” `git init` runs **asynchronously** (`execAsync`/`spawn`), never `execSync`.
- **FR-6** â€” `git init` is a deliberate, confirmed operation. Before `POST /api/projects` with `mode='new'` is sent, the UI must show a confirmation dialog that displays the exact canonical Git initialization target directory, the derived project key, the selected parent directory, and the fact that Overdeck will create the folder if needed, run `git init` there, then register the project. The user must explicitly confirm. The server must still enforce all target-safety checks; the dialog is not a substitute for backend validation.
### Frontend
- **FR-4** â€” A "New Project" modal/page with a top toggle **Add existing | New project**:
  - **Add existing** â†’ folder picker (FR-2) â†’ **Start** â†’ POST `mode='existing'`.
  - **New project** â†’ name field + parent-folder picker (default `~/Overdeck/`, "choose parent folderâ€¦") â†’ live preview `<parent>/<slug>` â†’ **Start** â†’ confirmation dialog (FR-6) â†’ POST `mode='new'`.
- **FR-5** â€” On success, invalidate/refetch `['command-deck-projects']` + `['registered-projects']` and select/open the new project.
### Non-functional
- **NFR-1** â€” No `execSync` in server code; async only (PAN-70 rule).
- **NFR-2** â€” Single write door: register **only** via `registerProjectSync`; do not write projects.yaml directly anywhere new.

## Acceptance criteria
- **AC-1 (FR-1/4):** "New project", name "Bugs", parent `~/Overdeck` â†’ creates `~/Overdeck/bugs`, runs `git init`, registers key `bugs` in projects.yaml; appears in the sidebar + Home.
- **AC-2 (FR-1/4):** "Add existing" â†’ browse to an existing repo folder â†’ registers it.
- **AC-3:** Duplicate key, or a non-empty `new` target â†’ clear error, **no partial write**.
- **AC-4 (FR-2):** The folder picker lists host subdirectories and navigates; gated by auth; cannot escape the user's home without explicit navigation.
- **AC-5 (NFR-1):** No `execSync` introduced; `git init` is async.
- **AC-6 (NFR-2):** `projects.yaml` is written only through `registerProjectSync`.
- **AC-7 (FR-1/3/6):** `mode='new'` never runs `git init` in an existing repository, in a symlink-resolved escape, in a file path, or in a non-empty directory. Tests cover each rejection and assert no `.git` creation and no `projects.yaml` mutation. The frontend test covers the confirmation dialog and verifies the POST happens only after explicit confirmation while showing the exact target path.

## Intersecting repo rules
- **Server:** NO `execSync` (PAN-70) â€” async git init / readdir.
- **Git initialization safety:** treat `git init` as a potentially dangerous filesystem write. The backend must be the source of truth: canonicalize the target itself, reject symlinks and existing repos, and run `git init` only in a verified empty directory under home.
- **Single source of truth:** register via `registerProjectSync` only (the write door).
- **Security:** the folder-browse endpoint must not allow arbitrary host traversal beyond the user's home without explicit navigation; auth-gate it like other internal routes.
- **Tests:** route tests for `POST /api/projects` (both modes + dup + error) and `GET /api/fs/list-dirs`; frontend Vitest for the modal.

## Planning Q&A

Planning ran in **`--auto` (non-interactive) mode** on 2026-06-19 (agent: claude-opus-4-8), so no questions were asked. The choices normal planning would have surfaced were resolved as defensible defaults and recorded below + in `.pan/continue.json` decisions. None of these contradicted the PRD/issue.

1. **Folder-browser scope â€” clamp to home vs. allow explicit out-of-home navigation?** FR-2's "must not traverse outside the user's home *unless explicitly navigated there*" is ambiguous. **Decision: hard-clamp `GET /api/fs/list-dirs` to the user's home** (default start `~`, reject `..`/absolute escapes with 400). Rationale: cleanly satisfies AC-4, safe for a demo, and this user's projects live under `~/Projects` (within home). Out-of-home browsing is a NonGoal (deferred).
2. **Where does the shared registration core live?** **Decision: a new server-safe module `src/lib/project-registration.ts` (`registerProjectFromPath`)**, consumed by both `projectAddCommand` and `POST /api/projects`. Rationale: single-sources the write door (NFR-2) without bloating `projects.ts`; no chalk/console/execSync.
3. **Sync vs async in the registration core.** **Decision: keep the sanctioned sync write door (`registerProjectSync`) + sync hook symlinks; make ONLY `git init` async** (`execAsync`/spawn). Rationale: PAN-70/NFR-1 targets unbounded subprocess blocking; the yaml write is the established single write door.
4. **No-partial-write and safe-target ordering for 'new' mode.** **Decision: dup-check, canonical target validation, symlink/file/existing-repo rejection, and non-empty-target check reject BEFORE any `mkdir`/`git init`** (D3). Rationale: AC-3 and AC-7.
5. **Route placement + auth.** **Decision: `POST /api/projects` in the existing `routes/projects.ts` (CSRF gate `rejectUnsafeDashboardMutationRequest`); `GET /api/fs/list-dirs` in a new `routes/fs.ts` (read gate `rejectUnauthorizedDashboardRequest`), merged into `server.ts` `makeRoutesLayer`.** Rationale: auth is per-route here, not global; match the established gates.
6. **PAN-1969 coordination.** `onNewProject()` / the "+ New Project" button come from sibling issue PAN-1969, which may not be merged yet (only PRDs are on main). **Decision: the wiring bead is self-contained â€” implements `onNewProject()` to open the modal and ensures exactly one "+ New Project" affordance exists (reuse PAN-1969's if present, else add it).** Rationale: PAN-1970 stays independently shippable/demoable.
7. **Decomposition.** Split into **8 beads**: `register-core` (foundation, inspection-gated) â†’ `api-projects-existing` â†’ `api-projects-new`; `fs-list-dirs` â†’ `folder-picker` â†’ `new-project-modal` â†’ `wire-new-project`. Rationale: each bead is independently reviewable, mergeable, and testable; the registration core is the one foundation other beads encode assumptions about.
