# PRD: Minimal New Project flow (PAN-1970)

> Part of the **minimum new-user onboarding** (demo readiness). **Explicitly NOT the full #654 wizard** — no Setup Agent, templates, archetypes, `SetupRpcGroup`, or `pan setup`. Clone-from-GitHub/GitLab is a **later** release.

## Summary
A minimal dashboard flow to create/register a project, with two modes:
1. **Add existing** — browse to an existing folder → register it.
2. **New project** — type a name, pick a parent folder (default `~/Overdeck/<name>`) → create folder + `git init` → register.

End state: an entry in `~/.overdeck/projects.yaml`, visible in the sidebar + Home (PAN-1969), ready to open. Mirrors the reference flow in the issue screenshots (add existing / new project / default folder / choose parent folder → Start workspace).

## Glossary
- **Project** — entry in `~/.overdeck/projects.yaml` (`ProjectConfig`).
- **registerProjectSync(key, config)** — the one helper that writes a project to projects.yaml.
- **Project key** — slug derived from name/folder: `name.toLowerCase().replace(/[^a-z0-9-]/g, '-')`.

## Problem
Projects can only be registered via `pan project add <existing-path>` in a terminal (`src/cli/commands/project.ts:85`). That command **requires the path to already exist** and **does not run `git init`**. There is **no dashboard endpoint** to register a project, **no server-side folder browser**, and **no "create folder + git init"** path. A brand-new user has no UI way to get their first project in.

## Goal / Non-goals
- **Goal:** from PAN-1969's "+ New Project", a user can Add existing OR New project, ending with a registered project visible in sidebar + Home.
- **Non-goals (explicit):** Setup Agent, templates/archetypes, `SetupRpcGroup`, `pan setup` (those are #654); clone from GitHub/GitLab (later); tracker config / quality gates / workspace-build config in the wizard (use minimal defaults, editable later in projects.yaml).

## Requirements
### Backend
- **FR-1** — `POST /api/projects` — body `{ mode: 'existing' | 'new', path?, parentDir?, name? }`. Registers the project and returns the created project record.
  - `existing`: validate `path` exists; derive key; dup-check; `registerProjectSync`; `ensureProjectLayer`; `preTrustDirectory`.
  - `new`: target = `<parentDir>/<slug(name)>`; reject if it exists and is non-empty; `mkdir -p`; `git init` (async); then the same register path.
  - Reuse the registration core from `projectAddCommand` (factor a shared helper rather than duplicating).
- **FR-2** — `GET /api/fs/list-dirs?path=<abs>` — returns immediate subdirectories of `path` (`{ name, path }[]`) plus the parent, for the folder picker. **Auth-gated** (internal token, like other routes). Default start path = `~` (or `~/Overdeck`). Must **not** traverse outside the user's home unless explicitly navigated there.
- **FR-3** — `git init` runs **asynchronously** (`execAsync`/`spawn`), never `execSync`.
### Frontend
- **FR-4** — A "New Project" modal/page with a top toggle **Add existing | New project**:
  - **Add existing** → folder picker (FR-2) → **Start** → POST `mode='existing'`.
  - **New project** → name field + parent-folder picker (default `~/Overdeck/`, "choose parent folder…") → live preview `<parent>/<slug>` → **Start** → POST `mode='new'`.
- **FR-5** — On success, invalidate/refetch `['command-deck-projects']` + `['registered-projects']` and select/open the new project.
### Non-functional
- **NFR-1** — No `execSync` in server code; async only (PAN-70 rule).
- **NFR-2** — Single write door: register **only** via `registerProjectSync`; do not write projects.yaml directly anywhere new.

## Acceptance criteria
- **AC-1 (FR-1/4):** "New project", name "Bugs", parent `~/Overdeck` → creates `~/Overdeck/bugs`, runs `git init`, registers key `bugs` in projects.yaml; appears in the sidebar + Home.
- **AC-2 (FR-1/4):** "Add existing" → browse to an existing repo folder → registers it.
- **AC-3:** Duplicate key, or a non-empty `new` target → clear error, **no partial write**.
- **AC-4 (FR-2):** The folder picker lists host subdirectories and navigates; gated by auth; cannot escape the user's home without explicit navigation.
- **AC-5 (NFR-1):** No `execSync` introduced; `git init` is async.
- **AC-6 (NFR-2):** `projects.yaml` is written only through `registerProjectSync`.

## Intersecting repo rules
- **Server:** NO `execSync` (PAN-70) — async git init / readdir.
- **Single source of truth:** register via `registerProjectSync` only (the write door).
- **Security:** the folder-browse endpoint must not allow arbitrary host traversal beyond the user's home without explicit navigation; auth-gate it like other internal routes.
- **Tests:** route tests for `POST /api/projects` (both modes + dup + error) and `GET /api/fs/list-dirs`; frontend Vitest for the modal.
