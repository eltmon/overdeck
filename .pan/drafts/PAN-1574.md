# PRD: Codex as a First-Class Harness

**Issue:** [PAN-1574](https://github.com/eltmon/panopticon-cli/issues/1574)
**Status:** Draft
**Author:** Planning
**Date:** 2026-05-31

---

## 1. Overview

Add **OpenAI Codex CLI** as a third first-class agent **harness** in Panopticon,
alongside Claude Code (default) and Pi. "First-class harness" means the
plan / work / review / test / ship pipeline roles can **spawn, monitor, message,
recover, and cost-track** a Codex agent through the exact same `AgentRuntime`
abstraction that Claude Code and Pi implement today — not merely route a GPT
model through another harness.

Claude Code remains the default. Codex is opt-in per spawn (role config,
conversation picker, plan kickoff) exactly as Pi is.

## 2. Background & Motivation

### 2.1 What Codex is *today* in Panopticon (and why that is not a harness)

Codex currently exists on two axes, neither of which is the harness axis:

- **Auth axis (done):** ChatGPT-subscription / Codex sign-in tokens are bridged
  through CLIProxy so GPT models are reachable.
  Code: `src/lib/codex-auth.ts`, `src/lib/cliproxy.ts`,
  `src/dashboard/server/routes/codex-auth.ts`,
  `src/dashboard/frontend/src/hooks/useCodexAuthStatus.ts`,
  `useCodexAutoRetry.ts`. (PAN-690, PAN-913, PAN-1067 — landed.)
- **Sync-target axis (basic):** skills authored in `.pan/skills/` are written to
  `AGENTS.md` as named blocks via `multi-tool-sync.ts` `syncToCodex`.
  (PAN-240 — landed, basic.)

When a user picks a GPT model, the model runs **under the Pi or Claude Code
harness** — Pi via its own `openai-codex` provider
(`src/dashboard/server/routes/conversations.ts:468`), or Claude Code via
CLIProxy. The actual `codex` CLI binary is **never spawned as a Panopticon
agent.** There is no `codex` runtime.

### 2.2 Why add the harness

- **Native Codex behavior** — `codex exec` / Codex TUI bring OpenAI's own agent
  loop, tool surface, and reasoning-model ergonomics, distinct from driving GPT
  through Pi's or Claude's loop.
- **Harness diversity** — a third independent harness reduces single-vendor CLI
  lock-in and lets the pool route work by harness, not just by model.
- **Foundation reuse** — the auth and skill-sync work already exists; the
  remaining cost is the runtime + plumbing, which the Pi integration already
  blueprinted.

### 2.3 Prior art — superseded, not reopened

- **PAN-23** (Cloister: Codex CLI Runtime Support) — closed COMPLETED 2026-04-10,
  but `src/lib/runtimes/` proves no runtime was ever built. Aspirational close.
- **PAN-600** (Launch Codex CLI conversations) — closed **NOT_PLANNED**;
  deliberately abandoned.

Both are obsoleted by this PRD. Neither should be reopened — their genuine
deliverables (auth, sync) live under PAN-690/913/240 and are preserved.

### 2.4 Why PAN-636 (Pi) is the template — and where it fell short

PAN-636 created the runtime abstraction (`AgentRuntime` / `AgentRuntimeSync` in
`src/lib/runtimes/types.ts`) and widened the harness union. That skeleton is
directly reusable. **But PAN-636's scope was too narrow:** the context-layering,
hooks/heartbeat, skill-format, and session-format-conversion concerns were all
backfilled *after* Pi shipped, in separate later issues (PAN-1200/1201 context
system, PAN-1569 AGENTS.md preservation, the session-format converter). This PRD
folds those concerns into the **initial** scope so Codex does not repeat the
backfill churn.

## 3. Goals / Non-Goals

### Goals
- `codex` is a selectable harness everywhere Pi is selectable.
- A Codex work agent runs the full pipeline (plan → work → review → test → ship)
  with health monitoring, message delivery, cost tracking, and recovery.
- Codex context (`AGENTS.md`) is produced by the layered context system, not
  just the cosmetic skill-sync path.
- No regression to Claude Code or Pi spawns (default path bit-for-bit unchanged).

### Non-Goals
- Replacing CLIProxy auth (it stays; Codex harness consumes it).
- Reworking the Pi or Claude Code runtimes.
- OpenCode (a separate, later harness — explicitly out of scope; this PRD is
  Codex only).
- New OpenAI model catalog work beyond what the harness requires (PAN-1122 owns
  catalog trimming).

## 4. Detailed Design

The work is the union of "widen the harness type" + "implement a runtime" +
"thread it through the four backfill surfaces."

### 4.1 Type widening (the union of record)

Codex must be added to every place the harness/runtime union is enumerated:

| Location | Symbol |
| --- | --- |
| `src/lib/runtimes/types.ts` | `RuntimeName = 'claude-code' \| 'pi' \| 'codex'` |
| `packages/contracts/src/types.ts` | `Harness` union + `KNOWN_HARNESSES` set |
| `src/lib/launcher-generator.ts` | `LauncherHarness` |
| `packages/contracts/src/flywheel.ts` | `FlywheelHarness` |
| `packages/contracts/src/context-layers.ts` | `CONTEXT_PREVIEW_HARNESSES`, render map keys |
| `packages/contracts/src/artifacts.ts` | `ArtifactAgentHarness` |
| `src/lib/config-yaml.ts` | role `harness` enum + validation |
| `src/lib/settings-api.ts` | harness validation |

`getHarness()` normalization must keep defaulting unknown/legacy → `claude-code`.

### 4.2 `src/lib/runtimes/codex.ts` — the runtime adapter

Mirror `pi.ts`. Implement all `AgentRuntime` methods:

- `spawnAgent(config)` — mkdir agent dir → set up delivery transport (see 4.3)
  → write launcher via `generateLauncherScript({ harness: 'codex', ... })` →
  `tmux new-session` on the `panopticon` socket → wait for readiness marker.
- `sendMessage(agentId, msg)` — via the chosen transport (4.3).
- `killAgent`, `isRunning` — tmux session lifecycle (reuse `tmux.ts` helpers).
- `getSessionPath` — resolve the active Codex rollout JSONL (4.4).
- `getLastActivity` / `getHeartbeat` — heartbeat source (4.5).
- `getTokenUsage` / `getSessionCost` — parse Codex JSONL usage (4.4).
- `listSessions`.

Register in `src/lib/runtimes/index.ts` (`createCodexRuntimeSync` /
`createCodexRuntime`, both sync and Effect variants).

**Open question (OQ-1):** Codex run mode. Codex offers an interactive TUI and a
non-interactive `codex exec`. Decide which maps to *work* agents (structured
delivery, like Pi `--mode rpc`) vs *conversation* panels (interactive TUI, like
Pi `--mode tui`). Likely both, as Pi does.

### 4.3 Message-delivery transport

Each harness has a delivery primitive:
- Claude Code → PTY supervisor (`dist/pty-supervisor.js`), tmux paste-buffer
  fallback.
- Pi → `rpc.in` named FIFO (rpc mode) / paste-buffer (tui mode).

**OQ-2:** Does Codex expose a structured stdin command stream (FIFO-able) like
Pi rpc, or is tmux paste-buffer the only path? This determines whether work-agent
delivery is reliable or best-effort. Investigate Codex's stdin/exec contract
during planning; fall back to the `load-buffer`+`paste-buffer` async pattern
(`sendKeysAsync`) if no structured channel exists.

### 4.4 Session format & cost

Codex stores sessions at `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl` — a
**third** JSONL shape distinct from Claude Code and Pi.

- Extend `src/lib/session-format-converter.ts` (today `pi`↔`claude-code` only) to
  handle `codex` as both a `fromHarness` and `toHarness`, using the same
  lossless-of-content / lossy-of-structure transcript extraction the converter
  already uses.
- Add Codex usage parsing to `src/lib/cost.ts` (`getTokenUsage` /
  `getSessionCost`). **OQ-3:** confirm Codex rollout JSONL records per-message
  token usage; if not, cost falls back to estimate-from-transcript.

### 4.5 Context layering — CLAUDE.md vs AGENTS.md

This is the gap PAN-636 left and PAN-1200/1201/1569 only partly closed for the
existing harnesses.

- `renderGlobalLayer` / `renderProjectLayer` (`src/lib/sync.ts`) are
  **claude-code-only** today. Add a `codex` rendering path that targets
  `AGENTS.md` (global `~/.codex/AGENTS.md` or equivalent, project `AGENTS.md`).
- `context-layers.ts` render map (`{ "claude-code": ..., pi: ... }`) needs a
  `codex` key, and `CONTEXT_PREVIEW_HARNESSES` must include `codex` so the
  dashboard `/context` preview shows the Codex column.
- Honor the **pre-existing-`AGENTS.md` preservation contract** from PAN-1569
  (managed BEGIN/END region; never clobber human content).
- Workspace context bundle: the launcher must hand Codex the assembled
  `<workspace>/.pan/context/workspace.md` (the Codex equivalent of Claude's
  `--append-system-prompt-file` / Pi's session_start load). **OQ-4:** Codex's
  mechanism for injecting a system/context file.

Note: the existing `syncToCodex` skill→`AGENTS.md` named-block writer is the
*cosmetic* also-sync path and is orthogonal; it stays, but it is **not** the
harness context path.

### 4.6 Hooks / heartbeat

Claude Code installs heartbeat hooks (`pan admin hooks install`); Pi heartbeats
from its vendored extension. **OQ-5:** Codex hook surface. If Codex has no hook
mechanism, `getHeartbeat()` uses passive detection only (rollout-JSONL mtime →
tmux pane activity), and we document Codex agents as passive-heartbeat-only.

### 4.7 harness-policy gate

Add `{ codex, model, authMode }` cells to `src/lib/harness-policy.ts`
`canUseHarnessSync`. At minimum: Codex + non-OpenAI model — decide
allowed/blocked; Codex + GPT under subscription auth — allowed (its native case).

### 4.8 Pickers & spawn entry points

Every harness/model picker and spawn path must offer Codex:
Settings (role harness/model defaults), conversation composer, plan kickoff,
`pan start` / work start, and the flywheel harness selector. All must call
`canUseHarness()` before showing/accepting the option.

## 5. Phasing (single deliverable — phases are build order, not separate ships)

1. **Type widening + registry stub** → verify: `typecheck` green with `codex` in
   every union; `codex` runtime registered (methods may throw `not-implemented`).
2. **`codex.ts` runtime (spawn/kill/isRunning/sendMessage)** → verify: spawn a
   Codex agent in a workspace, deliver a message, kill it.
3. **Session format + cost** → verify: Codex agent shows live cost/tokens and
   survives a harness conversion round-trip.
4. **Context layering (AGENTS.md) + workspace bundle** → verify: `/context`
   preview shows Codex column; spawned agent receives layered `AGENTS.md`;
   pre-existing `AGENTS.md` content preserved.
5. **Heartbeat + harness-policy + pickers** → verify: dashboard health monitoring
   works for a Codex agent; Codex selectable in all pickers; full
   plan→work→review→test→ship run on a trivial benchmark issue.

Per the "deliver complete features" rule, **all five phases ship together** — a
half-wired harness provides zero user value.

## 6. Open Questions

- **OQ-1** Codex run modes → work (structured) vs conversation (TUI) mapping.
- **OQ-2** Structured stdin/exec command channel for reliable message delivery?
- **OQ-3** Per-message token usage in Codex rollout JSONL?
- **OQ-4** Codex context/system-file injection mechanism.
- **OQ-5** Codex hook/heartbeat surface.

These are investigated in the **plan** phase (`pan plan PAN-1574`) before beads
are cut — they materially shape the runtime adapter.

## 7. Risks

- **Codex CLI surface drift** — Codex is evolving fast; pin behaviors discovered
  during planning and gate on a known Codex version.
- **Delivery reliability** — if no structured channel (OQ-2), work-agent delivery
  degrades to paste-buffer best-effort (acceptable for conversations, risky for
  unattended work). May constrain Codex to conversation-only in v1 if so.
- **Single-deacon / tmux invariants** — Codex sessions live on the `panopticon`
  socket like all agents; no second orchestrator.

## 8. Success Criteria

- A Codex work agent completes a trivial benchmark issue end-to-end through the
  pipeline with visible health, cost, and recovery in the dashboard.
- Codex selectable in every harness picker, gated by `harness-policy`.
- Layered `AGENTS.md` context delivered; human `AGENTS.md` content preserved.
- Claude Code and Pi default paths unchanged (regression suite green).

## 9. Out of Scope / Follow-ups

- **OpenCode harness** — the next harness after Codex; separate umbrella.
- **QwenCoder runtime** (PAN-466) — independent open runtime track.
- OpenAI model-catalog trimming (PAN-1122).
