## 1. Overview

Add **OpenAI Codex CLI** (`codex`) as a third first-class agent **harness** in
Overdeck, alongside Claude Code (default) and Pi. "First-class harness" means
the plan / work / review / test / ship roles can **spawn, deliver messages to,
monitor health of, cost-track, resume, and recover** a Codex agent through the
same `AgentRuntime` abstraction Claude Code and Pi implement — not merely route a
GPT model through another harness.

Claude Code stays the default. Codex is opt-in per spawn (role config,
conversation picker, plan kickoff, flywheel), exactly as Pi is.

This is the third harness; **OpenCode is the next one after Codex** and is
explicitly out of scope here.

## 2. Background & the auth-vs-harness distinction

### 2.1 What "Codex" means in the repo today (NOT a harness)

Codex currently exists on two axes, neither of which is a harness:

- **Auth axis (done):** ChatGPT-subscription / Codex sign-in tokens bridged
  through CLIProxy so GPT models are reachable.
  `src/lib/codex-auth.ts`, `src/lib/cliproxy.ts`,
  `src/dashboard/server/routes/codex-auth.ts`, `useCodexAuthStatus.ts`,
  `useCodexAutoRetry.ts`. (PAN-690, PAN-913, PAN-1067 — landed.)
- **Sync-target axis (basic):** skills authored in `.pan/skills/` are mirrored
  into `AGENTS.md` as **named blocks** via `multi-tool-sync.ts` `syncToCodex`
  (`AlsoSyncTool` already contains `'codex'`). (PAN-240 — landed, cosmetic.)

When a user picks a GPT model, it runs **under the Pi or Claude Code harness** —
Pi via its `openai-codex` provider (`conversations.ts:468`) or Claude Code via
CLIProxy. **The `codex` binary is never spawned as a Overdeck agent.** There is
no `codex` runtime: `src/lib/runtimes/` is `claude-code.ts` + `pi.ts` only, and
both `Harness` and `RuntimeName` are `'claude-code' | 'pi'`.

> **Critical trap for the work agent:** `getHarness()`
> (`packages/contracts/src/types.ts:55-63`) normalizes any unknown runtime string
> to `'claude-code'`. So **any touch-point we miss silently degrades a Codex agent
> to Claude Code rather than erroring.** Completeness of Appendix A is the
> correctness contract — TypeScript `Record<Harness, …>` types catch some misses
> at compile time, but many branches are runtime `if`s that fail silent.

PAN-636 created the `AgentRuntime` abstraction (`src/lib/runtimes/types.ts`) and

### 3.2 The decisive findings (verbatim evidence)

- **Headless run, verified end-to-end:**
  `codex exec --skip-git-repo-check -s read-only -c approval_policy=never --json -o last.txt "Reply PROBE_OK"`
  emitted `thread.started` → `turn.started` → `item.completed{agent_message:"PROBE_OK"}`
  → `turn.completed{usage:{input_tokens, cached_input_tokens, output_tokens, reasoning_output_tokens}}`,
  and wrote `PROBE_OK` to `last.txt`. **⚠️ `exec` does NOT accept `-a/--ask-for-approval`**
  (TUI-only) — set approval headlessly via `-c approval_policy=never`.
- **Multi-turn headless, verified:** `codex exec resume <thread-id> "<prompt>"`
  replayed full history (recalled `PROBE_OK` from the prior turn), reused the same
  `thread_id`, and `input_tokens` grew to reflect replayed context. `--last`
  resumes the most recent. Prompt may come from stdin via `-`. **⚠️ `exec resume`
  does NOT accept `-s`** — use `-c sandbox_mode=read-only`.
- **`codex app-server`, verified live handshake:** JSON-RPC `initialize` returned
  `{userAgent:"...0.135.0...", codexHome:"/home/eltmon/.codex"}`. Method set
  includes `thread/start`, `turn/start {threadId, input}`,
  `turn/steer {threadId, expectedTurnId, input}`, `turn/interrupt`,
  `thread/resume`, `thread/inject_items`, with streaming notifications
  (`AgentMessageDeltaNotification`, `ItemCompletedNotification`,
  `TurnCompletedNotification`, `ThreadTokenUsageUpdatedNotification`) and approval
  round-trips. Listener forms: stdio (default), `--listen unix://PATH`,
  `--listen ws://IP:PORT` (ws non-loopback requires capability-token/JWT auth).
  Daemon lifecycle: `codex app-server daemon {start|stop|restart|…}`.
  **Tagged `[experimental]`** — regenerate bindings per upgrade via
  `codex app-server generate-json-schema --out <dir>` / `generate-ts --out <dir>`.
- **`codex proto` does NOT exist in 0.135.0** — replaced by `app-server` + `mcp-server`.
- **Session/rollout format:**
  `~/.codex/sessions/YYYY/MM/DD/rollout-<ISO>-<UUIDv7>.jsonl`. The **UUIDv7 is the
  thread/session id** used by `resume` and emitted as `thread_id` in `--json`.
  Each line `{timestamp, type, payload}`. First line `session_meta`
  (`{id, cwd, originator, cli_version, model_provider, base_instructions.text}`).
  Token usage in `event_msg/token_count.info.{total_token_usage,last_token_usage}`.
  `~/.codex/session_index.jsonl` maps `{id, thread_name, updated_at}` (named
  resume); `~/.codex/history.jsonl` is global prompt history.
  **⚠️ The headless `--json` event names (`thread.started`/`item.completed`/
  `turn.completed`) DIFFER from the rollout-file event names
  (`task_started`/`agent_message`/`token_count`) — two distinct schemas.**
- **AGENTS.md precedence (docs-verified):** global
  `~/.codex/AGENTS.override.md`→`AGENTS.md` (first non-empty wins); project walk
  from git root **down to cwd**, ≤1 file/dir, concatenated blank-line-joined,
  closer-to-cwd appears later (overrides), stops at `project_doc_max_bytes`
  (32 KiB). app-server `thread/start`/`turn/start` accept a `baseInstructions`
  string that overrides the built-in Codex system prompt.
- **`notify` hook (docs-verified):** only event today is `agent-turn-complete`;
  Codex appends a single JSON string as the next argv (keys: `type`, `thread-id`,
  `turn-id`, `cwd`, `input-messages`, `last-assistant-message`). The user's
  machine already wires `notify` to a bridge script — **a per-agent `CODEX_HOME`
  with its own `config.toml` lets Overdeck set a Overdeck-owned `notify`
  without touching the user's global config.**
- **Auth:** `~/.codex/auth.json` (mode 0600) here is `auth_mode:"chatgpt"` with
  `tokens.{id_token,access_token,refresh_token,account_id}`. The native binary
  uses its **own** auth.json, independent of CLIProxy. For programmatic/CI use,
  OpenAI recommends API-key mode.
- **Trust/sandbox:** per-dir `[projects."<abs>"].trust_level`. Headless levers:
  `-s read-only|workspace-write|danger-full-access`, `-c approval_policy=never`,
  `--skip-git-repo-check`. With `approval_policy=never` + a sandbox mode it runs
  without prompting regardless of trust (verified in an untrusted `/tmp` dir).
  **UNVERIFIED:** TUI trust prompt behavior in an untrusted dir *without*
  `approval_policy=never` — verify by launching `codex` in a fresh dir.

## 4. Key design decisions (resolving v1's open questions)

| # | Decision | Rationale (verified finding) |
|---|---|---|
| **D-1 Run modes** | **Work agents → `codex exec` headless; conversations → interactive TUI.** Mirrors Pi's rpc/tui split. | §3.2 headless verified; TUI is `codex` bare. |
| **D-2 Delivery transport** | **Phase 1: `codex exec resume <thread-id> <prompt>` one-shot per orchestrator message** (stable, simple, no daemon). **Phase 2 (follow-up): `codex app-server` unix-socket daemon** as the Pi-rpc-grade bidirectional channel. | The #1 question is resolved: Codex *has* a structured channel; we don't need paste-buffer. exec-resume is on the **stable** surface; app-server is **experimental**, so it is a deliberate phase-2 upgrade, not the v1 risk. |
| **D-3 Token/cost** | **Parse the rollout JSONL `token_count` records** for cost; also capture `turn.completed.usage` from `--json` stdout as a live signal. | §3.2: usage is present & detailed in both. |
| **D-4 Context injection** | **Write Overdeck's layered context to a per-agent `CODEX_HOME/AGENTS.md`** (Codex auto-loads it), and/or pass `baseInstructions` when on app-server. Per-agent `CODEX_HOME` isolates this from the user's global `~/.codex`. | AGENTS.md precedence verified; baseInstructions verified on app-server. Per-agent CODEX_HOME avoids the global-AGENTS.md collision (§7.5). |
| **D-5 Heartbeat / readiness / done** | **Use a per-agent `CODEX_HOME/config.toml` `notify` hook** pointed at a tiny Overdeck shim that writes `~/.panopticon/agents/<id>/{heartbeat.json,completed}` on `agent-turn-complete`. **Readiness:** detect first appearance of the rollout JSONL (or first `--json` `thread.started`) rather than a ready.json the harness can't write natively. | `notify` is turn-granular (verified) — sufficient for a turn-end heartbeat + done marker, the Pi-extension analog. No in-process extension API needed. |
| **D-6 Auth** | **Reuse the host ChatGPT-subscription `auth.json` by default; support API-key via per-agent `CODEX_HOME` + `codex login --with-api-key`.** Independent of CLIProxy. | auth.json structure verified; CODEX_HOME override verified. |
| **D-7 Session storage model** | Codex writes to `~/.codex/sessions/...` (or `$CODEX_HOME/sessions/...`), like **Claude Code's external store** — NOT under `<agentDir>/sessions` like Pi. So `getSessionPath` resolves by **thread-id → rollout file**, captured at spawn from `--json`/notify, closer to the Claude Code adapter than the Pi one. | Rollout path + UUIDv7 thread-id verified. |

**Net:** none of v1's OQs remain open. The one genuinely deferred lever is the
**app-server daemon (D-2 phase 2)**, intentionally sequenced after a stable
exec-resume v1.

### 7.2 ⚠️ The AGENTS.md collision (the single trickiest slot-in)
Once Codex lands, **three writers** may target a project's `AGENTS.md`:
1. Pi context — managed BEGIN/END region (`sync.ts:609`).
2. Codex context — also wants the managed region of `AGENTS.md`.
3. The cosmetic skill mirror — named `<!-- panopticon:skill -->` blocks
   (`multi-tool-sync.ts:135`).
(1) and (3) coexist (different markers). But **(1) and (2) both want the same
BEGIN/END region of the same file.** Resolution options for the work agent to pick
during planning: **(a)** render AGENTS.md for the pi∪codex union when their
harness-filtered output is identical; **(b)** designate a single "AGENTS.md owner"
per project; or **(c, recommended)** keep Codex's harness context in a **per-agent
`CODEX_HOME/AGENTS.md`** (D-4) so it never contends for the project file. Option
(c) sidesteps the collision entirely and is consistent with D-5's per-agent
CODEX_HOME.

## 9. Risks

- **Experimental surface drift** — `app-server`/`remote-control` are experimental;
  D-2 phase 1 deliberately avoids them. Pin the Codex version; regenerate
  JSON-schema bindings on upgrade (`codex app-server generate-json-schema`).
- **exec-resume context-replay cost** — each `exec resume` turn replays full
  history (verified: input_tokens grew 30k→60k). For long work-agent sessions this
  inflates token cost vs a persistent app-server thread. Mitigation: phase-2
  app-server, or Codex's own compaction (`context_compacted` events exist).
- **AGENTS.md contention** (§7.2) — mitigated by per-agent CODEX_HOME (option c).
- **Two JSON schemas** (`--json` stdout vs rollout file) — don't assume parity;
  parse each with its own shape.
- **Trust prompts in TUI** (UNVERIFIED) — conversation-mode Codex in an untrusted
  dir may prompt; resolve in planning by setting trust via per-agent config or
  `approval_policy`.
- **Single-deacon / tmux invariants** — Codex sessions live on the `panopticon`
  socket like all agents; no second orchestrator; `sendKeysAsync` only.

## 10. Success criteria

- A Codex work agent completes a trivial benchmark issue end-to-end with visible
  health, cost, resume, and recovery in the dashboard.

## Appendix B — `AgentRuntime` interface (what codex.ts must implement)

From `src/lib/runtimes/types.ts`. Two parallel interfaces; implement both like Pi.

- Sync `AgentRuntimeSync` (`:115-227`): `name`, `getSessionPath`,
  `getLastActivity`, `getHeartbeat`, `getTokenUsage`, `getSessionCost`,
  `sendMessage` (sync|Promise), `killAgent` (prefer async), `spawnAgent`
  (sync|Promise), `listSessions`, `isRunning` (sync|Promise).
- Effect `AgentRuntime` (`:254-266`): same introspection (sync) + `sendMessage`/
  `killAgent`/`spawnAgent`/`isRunning` returning `Effect.Effect<…, AgentRuntimeError>`
  where `AgentRuntimeError = ProcessSpawnError | ProcessTimeoutError | TmuxError |
  FsError`. Pi's wrapper (`pi.ts:406-481`) holds `inner: PiRuntimeSync` and lifts
  async methods via `Effect.tryPromise` — copy this delegation verbatim.

Shared dirs reused from Pi: `~/.panopticon/agents/<id>/`,
`~/.panopticon/heartbeats/<id>.json`. Codex-specific: per-agent `CODEX_HOME`
holding `config.toml` (notify) + `AGENTS.md`; sessions resolved by thread-id under
`$CODEX_HOME/sessions/...`. **Never delete a rollout JSONL** (the JSONL-is-sacred
rule; Pi's `killAgent` cleanup preserves session files — mirror that).
