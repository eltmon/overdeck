# PAN-1847 — Delete the dead `projection_cache` table + write-through paths

> **Title:** chore(dashboard): delete the dead projection_cache table + write-through paths (unused after PAN-1920)
> **Issue:** https://github.com/eltmon/panopticon-cli/issues/1847
> **Epic:** state-model epic #1919. Follow-up for the real ~1 GB disk reclaim: **PAN-1925** (events-table / `agent.output_received` retention).

## Glossary

- **`projection_cache`** — a SQLite table in `~/.panopticon/panopticon.db` (`key TEXT PRIMARY KEY, data TEXT, sequence INTEGER, updated_at TEXT`). Introduced by PAN-437 as a write-through cache so the dashboard could serve a snapshot instantly at boot.
- **Write-through cache** — a cache written on every state change so it can be read back later. Here, two writers feed it and **nobody reads it** after PAN-1920.
- **The `dashboard` key** — a single row (`key = 'dashboard'`) holding the full JSON `DashboardSnapshot`. Written by the read model (`read-model.ts`).
- **The `agent-runtime:<agentId>` keys** — one row per agent holding that agent's `AgentRuntimeSnapshot`. Written by `agent-state-service.ts`.
- **PAN-1920 reconstruction** — the boot path that rebuilds runtime state from durable sources (`state.json` + tmux + GitHub + per-issue records) via `src/lib/reconstruct/reconstruct-cache.ts`. It **replaced** the projection_cache as the boot input. After PAN-1920, `ProjectionCache.load()` has **zero callers**.
- **`load()` / `save()`** — the two methods of the `ProjectionCache` service (`projection-cache.ts`). `load()` is dead; `save()` is still called but feeds a cache nobody reads.
- **The reducer fold** — `applyEvent` in `packages/contracts/src/event-reducers.ts` folds runtime events into `agentRuntimeById`. This drives the **live UI** runtime snapshots and is **kept**.
- **The runtime mirror** — `setAgentRuntimeMirror()` in `src/lib/agent-runtime-mirror.ts`, an in-process snapshot read by lib-side adapters. **Kept.**

## Problem

PAN-1847 was originally filed to "prune dead projection_cache rows + VACUUM to reclaim ~1.4 GB". **The planning probe empirically disproved that premise** against the live DB (`~/.panopticon/panopticon.db`, 1379 MB):

| Original claim | Reality (measured 2026-06-16) |
| --- | --- |
| DB "dominated by ~12k dead-agent rows" | The 12,125 `agent-runtime:` rows are **1.93 MB** (0.14% of the file) |
| "VACUUM reclaims ~1.4 GB" | Freelist is **5.3 MB**; `auto_vacuum` is **already `INCREMENTAL`** with a 15-min reclaimer (`src/lib/database/index.ts:108`) |
| where the 1 GB actually is | `events` table = **1068 MB**, of which `agent.output_received` = **721 MB** — tracked separately as **PAN-1925** |

Separately, after PAN-1920 the **entire `projection_cache` table is dead**: `ProjectionCache.load()` has zero callers (`reconstruct-cache.ts` never touches the table), yet two write-through `save()` paths still run on every state change — the `dashboard` snapshot path was historically a CPU hog (the file header of `projection-cache.ts` documents flush at "30% of process self-time").

**Behavior-neutrality proof (state this in review):** `getSnapshot` in `read-model.ts` builds snapshots live via `buildSnapshot()` (`read-model.ts:467`) and never depended on `load()`. Removing the write-through changes nothing a client observes — the cache was write-only.

## Proposal

The repo's **no-bandaids / root-cause** rule says: because the table is unread, **delete it** — do not build a garbage collector for a cache nobody consumes. Concretely:

1. Remove the two `projection_cache` write-through `save()` paths (read-model `dashboard` key; agent-state-service `agent-runtime:` keys).
2. Delete the orphaned `projection-cache.ts` service module (`createProjectionCache` / `initProjectionCache` / `getProjectionCache` / `load` / `save`) and its imports.
3. Stop creating the table (remove the `CREATE TABLE` from the base schema, the Bun-path schema, and the bundled cost hook) and **drop it** on existing DBs via a forward migration.
4. Remove the dedicated unit test for the deleted service; clean the orphaned mock in the boot test; add a regression test that locks the deletion.

This is **zero behavioral change** — the cache was unread. Disk reclaim is **explicitly not** a goal (≈7.5 MB is freed and reclaimed automatically by the existing `incremental_vacuum`).

## Requirements

- **FR-1** — No live caller of `ProjectionCache` `load()`/`save()` remains in dashboard server code; the `projection-cache.ts` module is deleted.
- **FR-2** — `projection_cache` is not created on a fresh DB (base schema, Bun path, and bundled cost hook all stop creating it) and is dropped on existing DBs by a forward migration.
- **FR-3** — No runtime behavior change: the runtime reducer fold (`agentRuntimeById`) and the runtime mirror (`setAgentRuntimeMirror`) still update on every runtime event; PAN-1920 reconstruction is unaffected; snapshot serving is unaffected.
- **FR-4** — The bundled cost hook `sync-sources/hooks/record-cost-event.js` no longer contains any `projection_cache` schema after regeneration.
- **NFR-1** — `npm run typecheck`, `npm run lint`, and `npm test` are green with no new failures.

## Decisions (made in this doc — executor must not re-derive)

- **D1 — Delete, don't GC.** Operator-confirmed (Option 1). Root-cause per the no-bandaids rule.
- **D2 — Schema removal shape.** Remove BOTH the base-schema `CREATE TABLE projection_cache` (`schema.ts:486-492`) and the historical `if (currentVersion < 5)` create block (`schema.ts:817-827`); add a new `if (currentVersion < 56) { db.exec('DROP TABLE IF EXISTS projection_cache'); }` block immediately before `db.pragma(\`user_version = ${SCHEMA_VERSION}\`)`; bump `SCHEMA_VERSION` from `55` to `56` (`schema.ts:24`). Removing the historical block is safe — no migration test migrates a DB from version < 5 or expects `projection_cache` (verified in `tests/unit/lib/database/schema-migrations.test.ts`), and it keeps the regenerated hook grep-clean.
- **D3 — No `VACUUM`.** `auto_vacuum` is already `INCREMENTAL` and `src/lib/database/index.ts:108` already reclaims the freelist every 15 min. The ≈7.5 MB freed by `DROP` is reclaimed automatically. Do not add a `VACUUM` or `VACUUM INTO` step.
- **D4 — Regenerate the hook from source.** `sync-sources/hooks/record-cost-event.js` is a tsdown-bundled artifact (it transitively bundles `schema.ts`). NEVER hand-edit the `.js`. After editing `schema.ts`, run `npm run build:scripts` (which runs `cd sync-sources/hooks && tsdown`) and commit the regenerated artifact.
- **D5 — Keep the no-circular-source guard.** `tests/unit/lib/reconstruct/no-circular-source.test.ts` lists `projection_cache` and `projectionCache.load` as forbidden strings in `src/lib/reconstruct/*`. Leave it untouched — it stays green and guards against re-introduction.
- **D6 — Keep the runtime fold + `agent.state_restored`.** Only the DB write is removed from `agent-state-service.ts`. The `agent.state_restored` event type and the `RUNTIME_EVENT_TYPES` set stay (reconstruction emits state).

## Key before/after — B2 (`agent-state-service.ts`)

The one non-mechanical deletion. **Remove only the DB write; preserve the fold + mirror.**

**Before** (`applyEventToRef`, ~189-222):
```ts
function applyEventToRef(
  ref: SubscriptionRef.SubscriptionRef<Record<string, AgentRuntimeSnapshot>>,
  ev: StoredEvent,
  upsertStmt: UpsertStmt,
): Effect.Effect<void> {
  return SubscriptionRef.update(ref, (current) => {
    const fakeState = { ...INITIAL_READ_MODEL_STATE, agentRuntimeById: current };
    const nextState = applyReducerEvent(fakeState, ev as unknown as DomainEvent);
    const next = nextState.agentRuntimeById;

    for (const [id, snap] of Object.entries(next)) {
      if (current[id] === snap) continue;
      try {
        upsertStmt.run([
          `${AGENT_RUNTIME_KEY_PREFIX}${id}`,
          JSON.stringify(snap),
          snap.updatedAtSequence,
          new Date().toISOString(),
        ]);
      } catch (err) {
        console.warn(`[AgentStateService] projection_cache upsert failed for ${id}:`, err);
      }
    }

    Effect.runSync(setAgentRuntimeMirror(next));
    return next;
  });
}
```

**After:**
