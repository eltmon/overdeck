# PAN-2076 — Boot reconciliation: grace window + durable decision + dashboard modal

**Status:** Ready · **Type:** substrate-improvement · **Epic:** [PAN-2075](https://github.com/eltmon/overdeck/issues/2075)
**Architecture:** [`docs/FAILSAFE-CONTROL-ARCHITECTURE.md`](https://github.com/eltmon/overdeck/blob/main/docs/FAILSAFE-CONTROL-ARCHITECTURE.md) — read this first; it carries the rationale and the operator-approved decisions.
**Visual spec:** [`.pan/drafts/PAN-2076-mockup.html`](PAN-2076-mockup.html) — open in a browser. The countdown ticks; try "Review each" and the "Status panel" view. Build the modal to match.

This is the first slice of PAN-2075: it **retires no-resume-by-default** and replaces it with a boot-reconciliation **grace window** backed by a **durable decision**, plus the operator modal. Out of scope here (follow-on issues): the `pan boot status` CLI + status panel ([PAN-2078](https://github.com/eltmon/overdeck/issues/2078)) and the scoped-control (pause/ignore/troubled) unification.

## Operator-approved decisions (build to these — do not re-litigate)
- **Grace default = 30s; on expiry the default action is RESUME.** Configurable via a new `startup.reconciliation_grace_secs` in cloister config (default 30).
- **`OVERDECK_NO_RESUME` is demoted to a boot input only:** it pre-selects "Keep all stopped" / skips the grace. It MUST NOT gate any runtime behavior (relaunch, re-dispatch, orphan recovery). PAN-2160 already removed it from the flywheel path; this removes the default-on behavior at boot.
- **Freeze (`deacon.globally_paused`) is the one emergency stop.** The modal's "Freeze everything" maps to it. No new kill-switch.
- **The decision is durable** (SQLite) so a watchdog restart honors it instead of reverting (the env-loss bug class).

## Glossary
- **Reconciliation set** — agents that were running before this dashboard start (from the SQLite `agents` table) whose tmux sessions on socket `overdeck` are now gone (need a resume decision). Live sessions need no action.
- **Grace window** — a bounded interval after boot during which the deacon patrol runs but **holds resume/dispatch for the reconciliation set**, and the flywheel is not auto-launched, until the operator decides or the grace expires.
- **Durable control plane** — canonical control state in SQLite (`app_settings` or a `control_*` table), reached only through the issues/agents resolver + the single writer (the two-door tenet). NOT process env.

## Work items

### WI-1 — Durable boot-reconciliation decision (control plane)
**Files:** `src/lib/database/app-settings.ts` (+ migration if a new table) and the canonical resolver/writer.
- Persist: `boot_reconciliation.decision` (`pending` | `resume_all` | `hold_all` | `per_agent`), `boot_reconciliation.per_agent` (map issueId→`resume`|`hold`), `boot_reconciliation.decided_at`, `boot_reconciliation.boot_id` (so a decision is scoped to a boot and a later genuine reboot re-prompts).
- Expose read/write through the existing settings door only — no direct store access elsewhere (CI guard).

### WI-2 — Grace window in the boot sequence
**Files:** `src/dashboard/server/main.ts`, `src/lib/cloister/deacon-auto-resume.ts`.
- On boot, compute the reconciliation set. Stamp a new `boot_id` and set `decision=pending` unless `OVERDECK_NO_RESUME` is set (then `hold_all`) — see WI-5.
- Start the patrol, but `autoResumeStoppedWorkAgents` (and flywheel auto-launch) **skip the reconciliation set while `decision==pending`**.
- A timer of `startup.reconciliation_grace_secs` (default 30) flips `pending → resume_all` on expiry (the approved default), then resumes.

### WI-3 — Apply the decision
**Files:** `deacon-auto-resume.ts` (or a small `boot-reconciliation.ts`).
- `resume_all` → resume the whole set (subject to the existing PAN-1665 concurrency brakes). `hold_all` → leave stopped. `per_agent` → resume only the `resume`-marked ones.
- Idempotent: applying twice (e.g. across a patrol tick) is a no-op once `decided`.

### WI-4 — Dashboard modal (the surface — build to the mockup)
**Files:** dashboard frontend (new `BootReconciliationModal.tsx` + an overlay host), `ws-rpc.ts` (+ contracts).
- RPC: `getBootReconciliation` (set + decision + grace deadline) and `submitBootReconciliation(decision, perAgent?)`.
- Modal per `.pan/drafts/PAN-2076-mockup.html`: countdown ring to the grace deadline, the reconciliation set, actions **Resume all now / Keep all stopped / Review each / Freeze everything**. "Review each" = per-agent Resume/Keep toggle (the PAN-2076 per-agent disposition). "Freeze" calls the existing freeze.
- The countdown is display-only; the server-side timer (WI-2) is authoritative.

### WI-5 — Demote NO_RESUME to a boot input
**Files:** `src/lib/boot-gates.ts`, `src/lib/cloister/no-resume-mode.ts`.
- `OVERDECK_NO_RESUME` at boot ⇒ `decision=hold_all` (skip the grace). It must no longer be consulted by any runtime path. Remove/neuter `getNoResumeMode()` runtime callers (verify none remain that gate relaunch/dispatch — PAN-2160 already cleared the flywheel one).
- Default boot (no env) ⇒ `decision=pending` + grace (NOT hold). This is the retirement of no-resume-by-default.

### WI-6 — Watchdog preserves the decision
**Files:** `src/supervisor/watchdog.ts`.
- After a watchdog restart, the dashboard reads `boot_reconciliation` + resume mode from the **durable plane** for the same `boot_id`, so an already-decided boot is NOT re-prompted and the operator's choice is not reverted. (This is the proper fix for the env-loss death-loop class; the supervisor-env workaround can then be retired.)

## Acceptance criteria
- **AC-1:** With no env set, a restart leaves prior agents stopped only until the grace resolves, then **resumes by default**; no-resume is no longer the default. Test: simulate a reconciliation set + advance fake timers past the grace ⇒ resume applied. (Use `vi.useFakeTimers()` — restate of the repo's fake-timers rule for delay tests.)
- **AC-2:** A decision written to the durable plane **survives a simulated watchdog restart** (same `boot_id`) — no re-prompt, choice honored. Test asserts persistence.
- **AC-3:** `OVERDECK_NO_RESUME=1` ⇒ `hold_all` at boot AND does not gate any runtime relaunch/dispatch. Unit test asserts no runtime consults it.

## Planning Q&A (auto mode — 2026-06-29)

`pan plan --auto` made the following decisions (no operator prompts). Recorded here so downstream agents understand WHY the plan is shaped this way; the canonical machine record is `plan.autoDecisions[]` in the spec.

- **Scope = the PRD (WI-1…WI-6), not the broader issue body.** The GitHub issue body describes a richer surface (grouped-by-concern inventory, running-remote `$`/cost columns, per-row **Kill**, Operator-Inbox posting). The PRD scopes PAN-2076 down to *grace window + durable decision + modal* and explicitly defers the inventory backend, the cost/remote columns, per-row Kill, and the Operator Inbox to **sibling issues that do not exist in the repo yet**. Building the issue-body scope is therefore impossible today (it depends on unbuilt siblings). The PRD says "build to these — do not re-litigate," so the plan builds to the PRD and lists the deferred items as NonGoals. *Not a contradiction* → no auto-halt.
- **Transport = HTTP routes, not Effect RPC.** The surface being replaced (`NoResumeBanner` + `GET /api/no-resume-mode` + `POST /api/resume-all` in `routes/misc.ts`) is plain HTTP + react-query. WI-4 mentions RPC, but matching the adjacent precedent (HTTP) minimizes surface area and churn. New endpoints: `GET/POST /api/boot-reconciliation`.
- **Control state lives in `src/lib/overdeck/control-settings.ts` (the two-door layer), not raw `app-settings.ts`.** That file already owns `deacon.globally_paused` and the flywheel keys with sync read/write wrappers over `app_settings`; the new `boot_reconciliation.*` keys follow the same pattern (single resolver/writer per the single-source-of-truth tenet).
- **`boot_id` = system boot time** (Linux `/proc/stat` `btime`, fallback `floor(Date.now()/1000 - os.uptime())`). It is stable across in-session dashboard restarts (watchdog *or* manual) and changes only on a real OS reboot. This makes the boot decision idempotent-per-boot: a watchdog restart re-derives the *same* `boot_id`, finds the existing decision, and honors it instead of re-prompting (AC-2); a genuine reboot gets a fresh `boot_id` and re-prompts. Chosen over a watchdog-passed env id because env is exactly what the death-loop class loses.
- **Grace default = 30s, expiry action = RESUME**, and **Freeze (`deacon.globally_paused`) is the only emergency stop** — taken verbatim from the PRD's operator-approved decisions; not re-litigated.
