## Glossary

- **Fly Machine** — a Firecracker micro-VM that is really a *container*: its lifetime is bound to PID 1, and its **rootfs is ephemeral** (reset from the image on every start, proven empirically on PAN-1762's clone).
- **`pan-workspaces` app / image** — the Fly app + Docker image (`registry.fly.io/pan-workspace:latest`) hosting remote work agents. PID 1 is `sleep infinity` (keep-alive); the agent runs as a detached `tmux` child launched via the Machines exec API. Image must NOT run sshd (port 22 collides with Fly hallpass → PID-1 exit → kernel panic).
- **Machines exec API** — `POST /apps/{app}/machines/{id}/exec`. `HOME=/`, ~16 KB payload cap (`writeRemoteFile` chunks at 8 KB), per-machine rate limit (429 on bursts), `/wait` max 60 s.
- **Volume** — a Fly persistent block device (`POST /apps/{app}/volumes`), region-pinned, mountable at one path on one machine at a time. Survives stop/start/crash and rootfs reset.
- **Resiliency tier** — a per-issue user-selected posture trading cost against durability: **ephemeral** (cheap, winds down on disconnect) vs **durable** (volume + restart + persists when the laptop closes).
- **Continuous push** — a VM-side heartbeat daemon that `git commit`+`git push`es `/workspace` to its `feature/<issue>` branch on a short cadence, making `origin` the durable store independent of Fly.
- **REMOTE_DONE sentinel** — `/workspace/.pan/REMOTE_DONE`; the work prompt's completion signal that the host reap patrol polls (`remote-completion.ts`).
- **Reap patrol** — `reapCompletedRemoteAgents()` (`src/lib/remote/remote-completion.ts`), run by the deacon (`deacon.ts:4997`), that hands completed remote work to the local review pipeline and destroys the machine.

## Problem

A remote Fly **work agent** (PAN-1762, machine `7812076c571018`) died abnormally on Fly's host (`exit_code=-1, oom_killed=false, requested_stop=false`) on 2026-06-12 18:54Z. **100% of its generated code and the Fable compute spend were lost.** Root cause, verified:

1. Durability is **push-once-at-completion**: `remote-completion.ts` commits+pushes only at clean handoff. An abnormal death *before* completion skips that push.
2. The machine has **no volume + ephemeral rootfs + `restart.policy:"no"`** (`fly-provider.ts:188-195`, `createMachine` `fly-api.ts:104-129`), so the never-pushed branch and the disk evaporate on death. Reading a stopped machine requires a start, which resets it — no recovery path.

One ephemeral VM + one keep-alive + one end-of-run push ⇒ any abnormal death before the finish = total loss.

## Proposal

Land the operator-agreed durability fixes before remote Fly **work** agents run in production again, and expose a user-selectable resiliency posture:

- **#1 Continuous commit+push (required, both tiers)** — a VM-side heartbeat daemon pushes `feature/<issue>` every N minutes, independent of agent diligence *and* host presence. Converts "lose everything" into "lose minutes." Highest ROI; would have saved PAN-1762 alone.
- **#2 Persistent Fly volume at `/workspace` (required, durable tier)** — survives rootfs reset / stop / crash. `createVm` currently mounts none.
- **3a restart policy (durable tier)** — `restart.policy:"on-failure"` + bounded `max_retries`. Auto-recovers the "Fly host killed our VM" case; only meaningful with #2 (restart resets rootfs; the volume preserves the work).
- **Resiliency tiers (operator requirement)** — a user setting: **ephemeral** (no volume, `restart:"no"`, continuous push, winds down on disconnect/idle) vs **durable** (volume + `on-failure` + continuous push, persists across laptop close). Continuous push is on in **both** tiers.
- **Spend guardrail** — a cap on concurrent remote agents so "durable" can't silently run to bankruptcy.
- **Durability preflight gate** — refuse to spawn a remote work agent unless the selected tier's durability mechanisms are wired (encodes the "no remote work agents until #1+#2 land" acceptance gate as self-enforcing code).

### Continuous-push mechanism (Decision D2)

Three options: (a) agent-side prompt instruction, (b) host-side patrol, (c) VM-side daemon. Only **(c)** survives the durable tier's "laptop can close" promise *and* is independent of agent diligence (the crash that loses work is exactly when the agent never got to push). **Chosen: (c) a VM-side detached `tmux` heartbeat daemon installed at spawn**, with **(a)** a per-bead push instruction in the work prompt as a complement. (b) is redundant with (c) and the existing `salvageRemoteWork` crash path, so it is **not** added. WIP checkpoints use the existing `wip(remote):` commit prefix and are squashed at merge.

### Wind-down on disconnect (Decision D5)

An autonomous work agent does **not** go idle when the laptop closes, so idle-stop is insufficient. Ephemeral-tier wind-down uses a **host-heartbeat + VM-side watchdog**: the deacon patrol touches a freshness file on each ephemeral VM; a VM-side watchdog (sibling of the push daemon) self-stops the machine (kills PID 1; `restart:"no"` keeps it down) when the heartbeat goes stale beyond a threshold. Continuous push means whatever was done is already on `origin`. The **durable** tier does not install the watchdog.

### Resume-after-restart scope (Decision D6)

3a + volume guarantee the **work survives** an abnormal death (volume preserves `/workspace`; continuous push already got it to `origin`). Host-independent **auto-relaunch of the agent** after a restart, and restoring the Claude conversation JSONL (which lives on the wiped rootfs, not the volume), require the supervisor-as-PID-1 (3c) and are **out of scope** here.

## Requirements

### Functional
- **FR-1** — A remote work agent's `feature/<issue>` branch appears on `origin` within minutes of work starting and stays current via a VM-side continuous-push heartbeat (both tiers).
- **FR-2** — Durable-tier machines mount a Fly volume at `/workspace`; its contents survive a stop→start cycle (inverse of the clone test). The volume is destroyed when the machine is destroyed (no orphan/cost leak).
- **FR-3** — Durable-tier machines are created with `restart.policy:"on-failure"` + bounded `max_retries`; ephemeral-tier machines with `restart.policy:"no"`.
- **FR-4** — Resiliency tier (`ephemeral`|`durable`) is a user setting: a config default, a `pan start --remote --tier <…>` flag, and a dashboard Settings control. The chosen tier is surfaced to the operator.
- **FR-5** — A spend guardrail (max concurrent remote agents) is enforced at remote spawn: spawning beyond the cap is refused with a clear error.
- **FR-6** — Ephemeral-tier machines wind down (self-stop) when the host heartbeat goes stale beyond a threshold; durable-tier machines persist.
- **FR-7** — Remote work-agent spawn is refused with a clear, actionable error unless the selected tier's durability mechanisms are satisfied (continuous push for both tiers; volume for durable).

### Non-functional
- **NFR-1** — `npm run typecheck`, `npm run lint`, `npm test` pass with no new failures. Any cadence/heartbeat/backoff test uses fake timers (`vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()`), never real waits.
- **NFR-2** — All new server-reachable code uses async exec/spawn (no `execSync`); tmux interactions use async primitives.
- **NFR-3** — Skills stay in lockstep with the CLI: a new `pan start --tier` flag updates `skills/pan-start/SKILL.md` in the same change (`scripts/lint-skills.sh`).
- **NFR-4** — WIP checkpoint commits use the `wip(remote):` prefix (consistent with `salvageRemoteWork`), relying on the pipeline's squash-merge.

## NonGoals

- 3c supervisor-as-PID-1 (the intended self-healing end-state) — separate follow-up.
- #4 machine-health monitoring / abnormal-death alerting / "unpushed for N min" health warning — explicitly future per the operator comment.
- Host-independent auto-**relaunch** of the agent after a restart, and restoring the Claude conversation JSONL post-restart (work resumes from filesystem + origin with a fresh session).
- Migrating CLIProxy / non-Anthropic (gpt-5.5, codex, kimi) agents to remote — remote stays claude-code + Anthropic-only.
- Sharing one volume across multiple machines concurrently.
- A host-side push patrol (redundant with the VM-side daemon and the existing salvage path).
