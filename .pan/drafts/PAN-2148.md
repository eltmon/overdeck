## 0. Glossary (define before use)

- **Two-door tenet** — every domain has exactly one read path (resolver) and one write
  path (writer); routes/CLI/dashboard reach state only through them
  (`docs/DECISIONS.md`, `docs/API-SURFACE.md`, issue PAN-1936). The door modules for
  issues already exist in skeleton: `src/lib/overdeck/issues.ts` (375 lines) exports
  `IssuesResolver` (+`IssuesResolverLive`) and `IssueWriter` (+`makeIssueWriterLive`).
- **Thin adapter** — a route handler that only: parses params/query/body, validates
  (`parseIssueIdSync`), calls one door function, and maps the result/error to
  `jsonResponse`. Rule of thumb: a handler over ~40 lines still contains domain logic.
- **Door module** — a file under `src/lib/overdeck/` owning one domain's logic, each
  under 1,000 lines (file-size guard applies to NEW files at 1,000).
- **Route registration idiom** — `const <name>Route = HttpRouter.add('METHOD', '/api/…', httpHandler(Effect.gen(...)))`
  (grep anchor: `const getIssuesRoute = HttpRouter.add(`), all merged in
  `export const issuesRouteLayer = Layer.mergeAll(` near EOF.
- **No-loss audit** — the §2 inventory: every route, export, and behavior in today's file
  must have a named home in the end state; the WI-9 test locks it.
- **Repoint** — update an importer to the moved symbol's new module path in the same
  commit as the move.
- **TENET-10** — pipeline-machinery refactors (complete-planning is pipeline machinery)
  require the FULL `npm test` suite green before merge, and source-introspection tests
  repointed in the same PR.

## 3. Locked design decisions

- **D1 — Route file stays; handlers thin; no barrel directory.** All 29 `HttpRouter.add`
  registrations and `issuesRouteLayer` remain in `routes/issues.ts`. No
  `routes/issues/` submodule directory is created for handlers (that was the
  merge-ops.ts mistake — moving fat handlers between route files). Fat goes to `lib`.
- **D2 — Door modules under `src/lib/overdeck/`, one domain each, <1,000 lines:**
  `planning-promotion.ts`, `planning-sessions.ts`, `task-generation.ts`,
  `issue-transitions.ts`, `workspace-hygiene.ts`, `pull-requests.ts`, `discussions.ts`,
  `issue-reads.ts`. These are behavior-preserving RELOCATIONS of today's logic — they
  call existing lib (e.g. `executeCloseOut` in `src/lib/close-out.ts`,
  `src/lib/lifecycle/*`) exactly as the handlers do today; no rewrite onto
  `IssuesResolver`/`IssueWriter` Effect services in this issue (that deeper migration is
  PAN-1936's; these modules become its staging ground).
- **D3 — No re-export shims; repoint instead.** Only ONE production file imports from
  `routes/issues.js` (`reconstruct-cache.ts`, verified §2.2). Shims would preserve the
  route file as a fake door and defeat the tenet. Every move repoints its importers in
  the same commit.
- **D4 — The 10× project-path boilerplate dedups into ONE exported helper**,
  `resolveIssueProjectPathSync(id)` in `issue-reads.ts`, containing the EXACT current
  block (grep anchor, appears 10×: `const localPaths = getGitHubLocalPaths();`) —
  verbatim semantics, including its fallback order. Do NOT substitute
  `resolveProjectFromIssue()` from `src/lib/projects.ts` — its resolution rules differ;
  behavior-preserving means keeping today's block.
- **D5 — One seam per commit, safest first** (WI order below). After each seam:
  `npm run typecheck && npm run lint && npm run build` plus the seam's named tests; FULL
  `npm test` before merge (TENET-10).
- **D6 — File-size baseline handling.** `routes/issues.ts` shrinks massively — lower its
  baseline entry in the same PR (if PAN-2227 landed, check mode forces this via
  `--update`; otherwise edit the entry down manually). Every new door module must be
  <1,000 lines so NO new baseline entries appear — if a planned module would exceed the
  ceiling, split it (that is what separates this from the merge-ops failure) and record
  the split in the PR description.
- **D7 — Deep-wipe stays behaviorally identical and is never invoked in verification.**
  The `/api/issues/:id/deep-wipe` handler is destructive; tests may only unit-test the
  moved function with fakes — never POST to a live server.
