# PAN-1813 â€” `pan plan finalize` is not resilient to `bd` timeouts under multi-agent load

**Issue:** [PAN-1813](https://github.com/eltmon/panopticon-cli/issues/1813)
**Discovered in:** PAN-1807 planning finalize, 2026-06-12 (Flywheel RUN-23)

## Glossary

- **bd / beads** â€” `bd` is the beads CLI; an embedded Dolt database backs it. Only
  one OS process can hold the Dolt file lock at a time, so concurrent `bd` calls
  across agents serialize on that lock.
- **finalize** â€” `pan plan finalize` (`src/cli/commands/plan-finalize.ts`): converts
  the workspace vBRIEF (`.pan/spec.vbrief.json`) into beads, stamps
  `plan.status = "proposed"`, and POSTs `complete-planning` to promote the spec to
  main and transition the issue to Planned.
- **create path** â€” `createBeadsFromVBriefPromise()` in `src/lib/vbrief/beads.ts`:
  connectivity probe â†’ dedup (clear existing beads) â†’ topological sort â†’ `bd create`
  per item with `--deps` wiring.
- **dedup** â€” `clearBeadsForIssue()` (`beads.ts:85`): list existing beads for the
  issue label, `bd delete` each, then re-list to verify none remain.
- **dependency edge** â€” a `blocks` edge in `plan.edges`. `bd create <item> --deps
  <blockerBeadId>` records "item depends on blocker". A lost edge corrupts the DAG.
- **bd-mutex** â€” `withBdMutexPromise()` (`src/lib/bd-mutex.ts`): serializes bd calls
  **within one process only**. It does NOT serialize across agent processes â€” the
  cross-process Dolt lock is the real contention point.

## Problem

With ~15 agents running (load avg ~10, several concurrent planning sessions issuing
`bd list` bursts), a single trivial `bd create` took ~15s and a `bd list â€¦ --limit 0`
took 10â€“30s. The bd call sites in `src/lib/vbrief/beads.ts` use **fixed timeouts**
(`bd list` = 15s `beads.ts:80`, `bd delete` = 10s `:98`, `bd ping` = 8s `:206/:230`,
`bd create` = 30s `:382`). Under contention these are routinely exceeded, and the
failure handling makes a timeout *worse than a clean failure*. Four consecutive
finalize runs each failed differently (verbatim from the issue):

1. Dedup's post-delete `bd list` failed â†’ abort â€” but `plan.status` had **already**
   flipped to `proposed` (half-finalized).
2. Beads (3/6) and (5/6) "timed out after 30s" â€” **but both creates succeeded
   server-side**. The client gave up, so their bead IDs were never captured;
   downstream beads were therefore created **without their `--deps`**. Two of four
   dependency edges silently missing while all 6 beads exist â€” a corrupted DAG that
   would run beads out of wave order.
3. Dedup's post-delete list timed out â†’ abort, leaving 0 beads (delete succeeded,
   verification didn't).
4. Even the initial dedup `bd list` timed out â†’ abort.

## Root causes

- **R1 â€” fixed timeouts.** `beads.ts` hardcodes per-op timeouts that don't scale with
  observed Dolt contention.
- **R2 â€” create-timeout treated as create-failure.** A client timeout on `bd create`
  usually means the create *succeeded server-side*; the lost item is only the printed
  bead ID. Treating it as failure orphans the bead and every dependent loses its edge.
- **R3 â€” transient read failures abort the whole finalize.** A single timed-out
  `bd list`/`bd ping` aborts with no retry, even though re-issuing it usually succeeds.
- **R4 â€” phase ordering leaves half-finalized state.** `stampPlanForFinalization()`
  flips `plan.status = "proposed"` **before** beads are created
  (`plan-finalize.ts:167` runs before `:173`), so a beads failure leaves the spec
  claiming `proposed` with 0 (or partial) beads.

## Proposal

Make a single finalize run resilient to bd timeouts without changing the cross-process
contention model. Five independently reviewable changes:

1. **Adaptive/configurable bd timeout** (R1). A `resolveBdTimeout(workspacePath)`
   resolver: `PANOPTICON_BD_TIMEOUT_MS` env override wins absolutely; else
   `clamp(pingTotalMs Ã— MULTIPLIER, FLOOR, CEILING)` where `pingTotalMs` comes from
   `bd ping --json` (its `total_ms`; the probe itself waits on the Dolt lock, so its
   latency already reflects contention). Defaults: `MULTIPLIER = 20`, `FLOOR = 30000`
   (preserves today's 30s as the minimum â€” no regression on an idle box),
   `CEILING = 180000` (hard cap â€” never wait unbounded). Probe once per finalize and
   cache. Route every `bd` invocation in `beads.ts` through it.

2. **Retry transient bd reads** (R3). A bounded exponential-backoff wrapper applied to
   **idempotent** bd ops only (`ping`, the dedup `list`, the post-delete `list`,
   `delete`). 3 attempts, base 500ms, Ã—2 backoff. `bd create` is **never** auto-retried
   (would create duplicates) â€” see change 3.

3. **Recover bead ID by title after a create timeout** (R2). When a `bd create` times
   out or returns empty stdout, re-query `bd list --json -l <issueLabel>` and match the
   bead whose title equals the full title `${plan.id}: ${item.title}` to recover the ID
   before declaring failure. The dedup pass clears all prior beads first, so within a
   run titles are unique; if more than one bead matches, surface an error rather than
   silently picking one.

4. **Verify + repair dependency edges** (R2 belt-and-suspenders). After all creates,
   issue one **batched** `bd dep list <idâ€¦> --json` (records carry `issue_id`, so one
   call covers every bead) and compare the actual `blocks` edges against `plan.edges`.
   Repair each missing edge with `bd dep add <blocked> <blocker>`. If a repair fails,
   fail the finalize loudly â€” a corrupt DAG must never materialize silently.

5. **Stamp status only after beads succeed** (R4). Move `stampPlanForFinalization()` to
   run **after** `createBeadsFromVBrief()` succeeds (still before promote). A beads
   failure then leaves the workspace spec un-stamped and cleanly re-runnable.

## Requirements

Functional:
- **FR-1** â€” bd op timeout is adaptive (`clamp(ping.total_ms Ã— N, FLOOR, CEILING)`) and
  overridable by `PANOPTICON_BD_TIMEOUT_MS`; all `bd` calls in `beads.ts` use it.
- **FR-2** â€” a `bd create` that times out client-side but succeeded server-side has its
  bead ID recovered by exact-title re-query before failure is declared, so dependents
  keep their `--deps`.
- **FR-3** â€” after creation, every `plan.edges` `blocks` edge is verified present;
  missing edges are repaired via `bd dep add`; an unrepairable edge fails the finalize.
- **FR-4** â€” `plan.status` is stamped `proposed` only after beads creation succeeds.
- **FR-5** â€” idempotent bd reads (`ping`, dedup `list`, post-delete `list`, `delete`)
  retry with bounded exponential backoff on transient timeout.

Non-functional:
- **NFR-1** â€” `bd create` is never auto-retried (no duplicate beads).
- **NFR-2** â€” the resolved timeout is bounded by a hard CEILING (never unbounded).
- **NFR-3** â€” retry/backoff unit tests use `vi.useFakeTimers()` +
  `vi.advanceTimersByTimeAsync()` (repo rule â€” never real waits, never `maxForks:1`).
- **NFR-4** â€” edge verification uses a single batched `bd dep list` call, not one call
  per bead, to limit added contention.

## Decisions (made in this doc; --auto mode)

- **Re-query-by-title, not `bd create --id`.** `bd create` supports an explicit `--id`,
  which would make the ID timeout-proof, but client-generated IDs risk prefix collisions
  with the shared Dolt DB and change the ID scheme. Re-query-by-title is the issue's
  suggested fix and stays within the existing ID model.
- **Timeout floor = today's 30s.** Adaptive only ever *raises* the timeout above the
  current value on a busy box; an idle box behaves exactly as today.
- **Edge-repair failure is fatal.** Failing loudly and re-running beats materializing a
  corrupt DAG â€” this is the precise bug being fixed.
- **Scope is the finalize/create path only.** Other bd call sites (kanban planning-state
  queries, etc.) are out of scope.

## Out of scope (NonGoals)

- Eliminating cross-process Dolt contention (a lock daemon, Dolt config, or a global
  cross-agent bd queue). These changes make one finalize *survive* contention; they do
  not reduce systemic load.
- Client-generated explicit bead IDs (`bd create --id`).
- A true distributed transaction across bd + spec file + the HTTP promote phase. We make
  phases idempotent/re-runnable and order them so partial failure leaves a recoverable
  state â€” we do not add 2-phase commit.
- Changing the `complete-planning` HTTP promote behavior (it already fails gracefully and
  tells the operator to re-run `pan plan done`).
- Touching bd timeouts outside `src/lib/vbrief/beads.ts`.
- Removing the existing 3-consecutive-failure verification bypass or any unrelated gate.
