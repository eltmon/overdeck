## Background (why this issue changed)

The original PAN-1883 claimed `~/.panopticon/review-status.json` is the canonical review-status
store and that a wipe stranded 11 in-review issues. **Both claims are false**, verified:

- `src/lib/review-status.ts:156` — `loadReviewStatuses()` returns `getAllReviewStatusesFromDb()`.
  **SQLite (`~/.panopticon/panopticon.db`, table `review_status`) is authoritative.** The deacon
  and server never read the JSON file.
- The JSON file is written only by `src/lib/review-status-json.ts`, which **no production module
  imports** (verified: `grep -rn "from.*review-status-json" src | grep -v test` → none). It is
  test/CLI scratch.
- The 11 "stranded" issues were all present and current in `review_status`, blocked by real
  recorded reasons (`blocker_reasons` = `merge_conflict` ×8, `failing_checks` ×1; 2 genuinely
  `review=blocked`; 1 in-progress).

The actual bug: the Flywheel orchestrator (and the dashboard) can **misdiagnose pipeline state by
reading a legacy file instead of the SQLite-backed surfaces.** The orchestrator's live instructions
(`roles/flywheel.md`, `docs/flywheel-brief.md`) say what surfaces to USE but never say the JSON file
is a trap — so the RUN-34 orchestrator spelunked it and filed a false "wiped" bug.

**Guiding principle (operator-stated):** agents read pipeline truth through CLI/API/skills, never by
reading raw state files or the DB directly.

---

## Glossary

- **`review_status` table** — the authoritative SQLite store of every issue's review/test/merge gate
  state, in `~/.panopticon/panopticon.db`. Accessed via `src/lib/database/review-status-db.ts`.
- **review-status.json** — `~/.panopticon/review-status.json`. **Legacy/test-only scratch.** Not the
  store; usually empty or stale. Never read it to judge pipeline state.
- **The surfaces** — the sanctioned read paths for pipeline truth:
  - `pan review pending --ready` — issues with review+test green, not merged (CLI; reads SQLite).
  - `GET /api/flywheel/merge-blockers` — issues that passed review but cannot merge, with
    `reasons[].type` ∈ {`merge_conflict`, `failing_checks`, `not_mergeable`} (`src/dashboard/server/routes/flywheel.ts:594`).
  - the dashboard review-status snapshots (read model, SQLite-backed).
- **blockerReasons** — `ReviewStatus.blockerReasons: BlockerReason[]` (`src/lib/review-status.ts`),
  GitHub-native merge blockers persisted on the row.

## Requirements

- **FR-1** — The Flywheel orchestrator's live instructions must explicitly prohibit reading
  `review-status.json` for pipeline truth and name the sanctioned surfaces.
- **FR-2** — The Command Deck must distinguish blocker *kinds* in its label, not collapse all to
  "CHANGES REQUESTED": review-blocked → `CHANGES REQUESTED`; `merge_conflict`/`not_mergeable` →
  `MERGE BLOCKED`; `failing_checks` → `CI BLOCKED`.
- **FR-3** (optional, stretch) — Provide a CLI surface for blocked work (`pan review pending
  --blocked`) so agents/humans have a sanctioned command, not ad-hoc DB/JSON reads.
- **NFR-1** — No change to the SQLite store, its schema, or any write path. This issue is
  instructions + presentation only.

### WI-2 — Command Deck label distinguishes blocker kind (FR-2)

**File:** `src/dashboard/frontend/src/components/CommandDeck/ProjectOverview.tsx`.

**Current (verified `:418`, search for `'CHANGES REQUESTED'`):**

```tsx
  if (isBlockedFeature(entry.feature, entry.reviewStatus)) return { variant: 'CHANGES REQUESTED' };
```

Every blocked feature renders `CHANGES REQUESTED` — including merge-conflict and CI-failure blocks,
which is wrong and is the literal cause of the "Flywheel says blocked, tree doesn't match" confusion.

**Change:** introduce a helper that picks the label from the blocker kind, and call it here. Add near
the other classifier helpers (after `isBlockedFeature`, ~`:122`):

```tsx
function blockedVariant(reviewStatus: ReviewStatusSnapshot | undefined): string {
  const types = new Set((reviewStatus?.blockerReasons ?? []).map(b => b.type));
  if (types.has('merge_conflict') || types.has('not_mergeable')) return 'MERGE BLOCKED';
  if (types.has('failing_checks')) return 'CI BLOCKED';
  if (MERGE_BLOCKED_STATUSES.has(reviewStatus?.mergeStatus ?? '')) return 'MERGE BLOCKED';
  if (TEST_BLOCKED_STATUSES.has(reviewStatus?.testStatus ?? '')) return 'CI BLOCKED';
  return 'CHANGES REQUESTED'; // review failed/blocked or stuck with no merge-level reason
}
```

**After (`:418`):**

```tsx
  if (isBlockedFeature(entry.feature, entry.reviewStatus)) return { variant: blockedVariant(entry.reviewStatus) };
```

**Note:** `blockerReasons[].type` values are verified from live data (`merge_conflict`,
`failing_checks`) and the merge-blockers payload (`not_mergeable`). Confirm the `BlockerReason` type
union in `src/lib/review-status.ts` covers these before relying on them.

**Tests:** extend the existing ProjectOverview classifier test (search `tests/` for
`isBlockedFeature` / `ProjectOverview`). Add cases: a `merge_conflict` blockerReason → `MERGE
BLOCKED`; a `failing_checks` blockerReason → `CI BLOCKED`; `reviewStatus='failed'` with no
blockerReasons → `CHANGES REQUESTED`.
