## 1. Glossary

Define every term before it is used.

| Term | Definition |
| --- | --- |
| **Cache** | The local SQLite database `~/.panopticon/panopticon.db`. Per the State Model it is a **disposable derived cache**, not a system of record. |
| **Source of truth** | The four durable systems outside the cache: **GitHub Issues** (issue lifecycle), **git** (feature branches, beads, the per-issue record committed on the feature branch), **agent transcripts** (on disk per harness), **tmux** (the liveness oracle on socket `-L panopticon`). |
| **Event log** | The `events` table in SQLite, read/written by `src/dashboard/server/event-store.ts`. Today it is BOTH a live notification bus AND the boot reconstruction source. This issue removes the second role. |
| **Reducer** | The pure fold in `packages/contracts/src/event-reducers.ts` (`applyEvent`) that turns a stream of domain events into a `ReadModelState`. Unchanged by this issue. |
| **Projection cache** | Rows in the `projection_cache` SQLite table: key `dashboard` (the full `DashboardSnapshot`) and keys `agent-runtime:<agentId>` (per-agent runtime snapshots). A **persistence of the cache**, NOT a source of truth. |
| **AgentStateService** | `src/dashboard/server/services/agent-state-service.ts`. Owns the in-memory `Record<AgentId, AgentRuntimeSnapshot>` map (live activity: thinking/waiting/model). |
| **ReadModelService** | `src/dashboard/server/read-model.ts`. Owns the in-memory `ReadModelState` and serves `getSnapshot` (the `DashboardSnapshot` the frontend renders). |
| **IssueDataService** | `src/dashboard/server/services/issue-data-service.ts`. Already polls GitHub/Linear in the background and is the issue source for the read model. Its `bootstrap` Effect is already `Effect.void`. |
| **agents table** | The `agents` SQLite table — the authoritative runtime agent registry (PAN-1908 runtime plane). Rebuilt from `state.json` (rollback source) reconciled against live tmux by `backfillAgentsFromStateJsonSync`. |
| **state.json** | `~/.panopticon/agents/<id>/state.json`. Per PAN-1908 a **rollback/rebuild source** for the agents table, not the enumeration source. |
| **per-issue record** | `<infraRepo>/.pan/<recordsPath>/<ISSUE>.json` (`PanIssueRecord` in `src/lib/pan-dir/records.ts`). Holds the continue subset, the `pipeline` verdict block (review/test/merge statuses, `readyForMerge`, `prNumber`, `prUrl`), `closeOut`, and `owner`. Committed to git → portable. The PAN-1919 dependency. |
| **pipeline phase** | A derived label — `work` \| `review` \| `merge` (rendered `ship`) \| `done` — that the kanban/Command-Deck uses. **Derived, never stored.** Today computed by `getPipelineIssuePhase()` in `src/dashboard/frontend/src/lib/pipeline-state.ts`. |
| **in-flight issue** | An OPEN GitHub issue that has a `feature/<id-lowercase>` branch/worktree on disk. The set reconstruction enumerates. |
| **reconstructCache()** | The new routine this issue adds: rebuilds the agents table → in-memory runtime map → read-model snapshot from sources of truth, with **no event-log replay**. |
| **superset (no-loss)** | The audit property: reconstructing from sources must drop **no** issue, agent, or phase that the cache showed before. A behavioral test, not a static manifest. |

## 2. Problem & current state (verified 2026-06-16)

The dashboard reconstructs runtime state at boot by **seeding from `projection_cache` and replaying the event log**. Two bootstrap sites do this:

1. **AgentStateService** — `src/dashboard/server/services/agent-state-service.ts`. The forked `seedAndReplay` Effect (verified at `agent-state-service.ts:167`) calls `seedFromCache()` (line 124) to load `agent-runtime:` rows, then `store.readFrom(maxCachedSequence)` (line 181) to replay every runtime event since the last cached sequence:

   ```ts
   // agent-state-service.ts:180-188 (verified)
   const stored = store.readFrom(maxCachedSequence);
   let replayed = 0;
   for (const ev of stored) {
     if (isRuntimeEvent(ev)) {
       yield* applyEventToRef(ref, ev, upsertStmt);
       replayed++;
     }
   }
   ```

2. **ReadModelService** — `src/dashboard/server/read-model.ts`. Bootstrap runs inline during layer construction (the comment `// ── Bootstrap inline during layer construction ──` at `read-model.ts:484`). It has a **fast path** that loads the `dashboard` projection-cache row (`projectionCache.load()`, `read-model.ts:498`) and a **slow path** that calls `listRunningAgents()` from `src/lib/agents.js` (`read-model.ts:633`). The sequence is read from the event store (`getEventStore().getLatestSequence()`, `read-model.ts:752`).

Per the State Model ([`reference/state-model.mdx`](../../reference/state-model.mdx)): **the database is a cache, not a source of truth**; the pipeline phase is **a function of GitHub + the workspace, not a stored field** (the table at `state-model.mdx:40-46`). The event log should be **only a live notification bus**. Today it is also the reconstruction source — which is what produces the cold start and the unbounded snapshot graveyard ([PAN-1847](https://github.com/eltmon/overdeck/issues/1847)).

**Reusable enumerators that already exist** (verified):
- `src/lib/database/agent-backfill.ts` — `backfillAgentsFromStateJsonSync(db, opts)` at `agent-backfill.ts:166`: enumerates `~/.panopticon/agents/*/state.json`, reconciles each against live tmux (`reconcileAgentStatus`), upserts the `agents` table. **This is a sources-only chain: state.json + tmux → agents table.** Already wired to `pan admin db rebuild-agents` (`src/cli/commands/db.ts:627`).
- `src/lib/pan-dir/records-backfill.ts` — `collectInFlightIssueIds(projects)` at `records-backfill.ts:96`. **NOTE: today this reads the SQLite cache** (`collectAgentIssueIds` → `listAllAgents()`, `collectReviewStatusIssueIds` → `getAllReviewStatusesFromDb()`, plus continue files). It is therefore **NOT a valid reconstruction enumerator** — see FR‑2 / NFR‑1.
- `src/dashboard/server/routes/issues.ts` — `fetchIssuePullRequest(issueId)` at `issues.ts:3303` shells `gh pr view ... --json <GH_PR_VIEW_FIELDS>` where `GH_PR_VIEW_FIELDS` (line ~3163) already includes `reviewDecision` and `state`. This is the existing GitHub PR-state fetch we reuse for approval.

**There is no branch/worktree enumerator today.** A `grep` for `listWorktrees` / `git worktree list` / `for-each-ref` over `src/lib/` returned nothing reusable. Step 1 of reconstruction needs a new one (Work Item 1).

## 3. Decisions made in this document

State each rule plainly so the executor never guesses.

- **D1 — Reconstruction reads ONLY sources of truth.** The forbidden inputs are: the `events` table (`store.readFrom`), the `projection_cache` table (any key), the `review_status` table (`getAllReviewStatusesFromDb`), and `collectInFlightIssueIds` *as written* (it reads those caches). Allowed inputs: GitHub (issues + PR state), git (branches/worktrees, beads, the per-issue record), `state.json` (agents-table rollback source), and live tmux (oracle). The `agents` table is allowed **as an intermediate** because `backfillAgentsFromStateJsonSync` rebuilds it from state.json + tmux first — that chain is sources-only.
- **D2 — In-flight enumeration = OPEN GitHub issue ∩ has `feature/<id-lowercase>` branch/worktree.** The GitHub OPEN set comes from `IssueDataService` (already GitHub-sourced). The branch set comes from the new enumerator (Work Item 1). The intersection is the reconstructed in-flight set.
- **D3 — Phase derivation rule** (matches `state-model.mdx:40-46`):
  - Feature branch present + issue OPEN + **no PR** → **Work**.
  - PR OPEN + **not approved** → **Review**.
  - PR **approved** → **Merge** (rendered `ship`).
  - Issue **CLOSED** on GitHub → **Done**.
- **D4 — "Approved" is sourced from the per-issue record's `pipeline` block, enriched (not replaced) by GitHub `reviewDecision`.** `GitHubPullRequestState` (`src/lib/github-app.ts:42`) has **no** approval field, so approval cannot come from the bare PR object, and it must **not** come from `review_status` (that is the cache, D1). Primary: `record.pipeline.readyForMerge === true` OR `record.pipeline.reviewStatus === 'passed'` (these are the durable verdicts mirrored into the record). Enrichment: if `fetchIssuePullRequest` returns `pr.reviewDecision === 'APPROVED'`, treat as approved even when the record is stale. This keeps approval portable (record travels in git) and matches the PAN-1919 dependency framing.
- **D5 — Reconstruction does NOT mutate any spec.** PAN-1124 single-spec-on-main: it READS the main-side spec via `findSpecByIssue` if needed; it never writes one. The only legal spec mutation in the codebase remains `updateSpecStatus()`.
- **D6 — The event log stays a live bus.** `store.subscribe(...)` (`agent-state-service.ts:149`), the reducers, the WebSocket fan-out, AND `projection_cache` write-through (`applyEventToRef`'s `upsertStmt.run`, and `projectionCache?.save(...)` in read-model) **all remain unchanged**. This issue removes only the *replay-as-reconstruction-source* code path. Retention slashing and cache-persistence demotion are PAN-1921 — do not touch `compact()` or `readFrom`'s remaining callers here.
- **D7 — PAN-1919 may not be merged when this lands (it is OPEN).** Implementation checkpoint: if `readIssueRecord(project, issueId)` returns `null` (no record yet), fall back to `state.json` for harness/model (already the agents-table source) and to `fetchIssuePullRequest().reviewDecision` for approval. Reconstruction must not throw when a record is absent.

## 4. Requirements

### Functional

- **FR‑1** — Add `reconstructCache()` in a new module `src/lib/reconstruct/reconstruct-cache.ts` that rebuilds, in order: (a) the `agents` table from `state.json` + tmux; (b) the in-memory `AgentRuntimeSnapshot` map; (c) the `ReadModelService` snapshot — all **without** `store.readFrom`/event replay.
- **FR‑2** — Add `enumerateInFlightIssuesFromSources()` that returns the set of OPEN GitHub issue IDs that have a `feature/<id-lowercase>` branch/worktree on disk. It must NOT read `events`, `projection_cache`, `review_status`, or `collectInFlightIssueIds`.
- **FR‑3** — Add `derivePipelinePhase(issue, record, pr)` implementing D3/D4: returns `'work' | 'review' | 'merge' | 'done'` from GitHub issue state + per-issue record + GitHub PR state. No `review_status` / `projection_cache` reads.
- **FR‑4** — `reconstructCache()` runs at dashboard boot in place of event replay: AgentStateService's runtime map is seeded from the rebuilt agents table (not `seedFromCache` + `readFrom`); ReadModelService's bootstrap snapshot is built from `reconstructCache()` output (issues still from IssueDataService, agents from the rebuilt table).
- **FR‑5** — Add a `pan admin db rebuild` subcommand that runs `reconstructCache()` on demand and prints a summary (issues enumerated, agents rebuilt, phases counted).
- **FR‑6** — The live event bus is unaffected: `store.subscribe`, reducers, WebSocket fan-out, and `projection_cache` write-through continue to operate exactly as before (D6).

### Non-functional

- **NFR‑1 — No circular sourcing.** A unit test asserts `reconstructCache()` and its helpers never call `store.readFrom`, `getAllReviewStatusesFromDb`, `projectionCache.load`, or `collectInFlightIssueIds` (static-import / call-graph assertion — see Work Item 6).
- **NFR‑2 — No-loss superset (behavioral).** Snapshot the cache view (issues, agents, phases) → run `reconstructCache()` → assert the reconstructed view is a **superset**: no issue, agent, or phase present before is missing after (Work Item 7).
- **NFR‑3 — Node 22 only.** All server-reachable code uses `execAsync`/`execFileAsync` or `spawn`, never `execSync`. After changes, `npm run build` before restarting the dashboard.
- **NFR‑4 — Async tmux.** Any tmux interaction added uses `sendKeysAsync` / the async primitives in `src/lib/tmux.ts`, never sync.
- **NFR‑5 — Idempotent.** Re-running `reconstructCache()` (boot, then `pan admin db rebuild`) produces the same agents table and snapshot with no duplicate rows.

### Work Item 1 — Branch/worktree enumerator (FR‑2)

**What/why:** There is no helper that lists `feature/<id>` branches on disk; reconstruction needs the in-flight set. Intersect with OPEN GitHub issues (D2).

**Files/functions:**
- New: `src/lib/reconstruct/enumerate-in-flight.ts` — `enumerateInFlightIssuesFromSources(projects, openIssueIds): Promise<Set<string>>`.

**Steps:**
1. For each project, resolve `workspacesDir = join(project.path, workspaceConfig.workspaces_dir || 'workspaces')` — the exact expression used at `src/lib/workspace-manager.ts:571`.
2. Enumerate `feature-<slug>` worktrees via `git worktree list --porcelain` (run with `execAsync`, cwd = project root — NFR‑3) and/or by scanning `workspacesDir` for `feature-<id>` directories. Map each `feature-<slug>` → issue ID `<SLUG>.toUpperCase()` (the inverse of `featureFolder = feature-${featureName}` at `workspace-manager.ts:572`).
3. Intersect the branch-derived ID set with `openIssueIds` (passed in from IssueDataService, GitHub-sourced). Return the intersection.

**Before/after:** New file — no before.

```ts
// src/lib/reconstruct/enumerate-in-flight.ts (new, codebase style)
import { promisify } from 'node:util';
import { exec } from 'node:child_process';
import { join } from 'node:path';
const execAsync = promisify(exec);

/** OPEN GitHub issues that have a feature/<id-lowercase> branch on disk. */
export async function enumerateInFlightIssuesFromSources(
  projects: ProjectConfig[],
  openIssueIds: Set<string>,
): Promise<Set<string>> {
  const inFlight = new Set<string>();
  for (const project of projects) {
    const { stdout } = await execAsync('git worktree list --porcelain', { cwd: project.path }).catch(() => ({ stdout: '' }));
    for (const m of stdout.matchAll(/^worktree .*\/feature-([a-z0-9-]+)$/gim)) {
      const id = m[1].toUpperCase();
      if (openIssueIds.has(id)) inFlight.add(id);
    }
  }
  return inFlight;
}
```

**Proving test:** `tests/unit/lib/reconstruct/enumerate-in-flight.test.ts` — given a fake worktree list and an OPEN-issue set, asserts only the intersection is returned (closed-but-has-branch and open-but-no-branch are both excluded).

### Work Item 2 — Phase derivation from sources (FR‑3)

**What/why:** Phase is derived, not stored (D3). The frontend already derives it via `getPipelineIssuePhase()` (`src/dashboard/frontend/src/lib/pipeline-state.ts:116`); reconstruction needs a server-side derivation from **sources** (record + GitHub PR), per D4.

**Files/functions:**
- New: `src/lib/reconstruct/derive-phase.ts` — `derivePipelinePhase(args): 'work' | 'review' | 'merge' | 'done'`.

**Steps (implement D3/D4 exactly):**
1. If the GitHub issue state is closed/done → `'done'`.
2. Else fetch PR via `fetchIssuePullRequest(issueId)` (`src/dashboard/server/routes/issues.ts:3304`). If no PR → `'work'`.
3. Approved? `record?.pipeline?.readyForMerge === true || record?.pipeline?.reviewStatus === 'passed' || pr?.reviewDecision === 'APPROVED'` → `'merge'`.
4. Else (PR open, not approved) → `'review'`.

```ts
// src/lib/reconstruct/derive-phase.ts (new)
export function derivePipelinePhase(args: {
  issueClosed: boolean;
  hasPr: boolean;
  record: PanIssueRecord | null;
  reviewDecision: string | null;   // from gh pr view --json reviewDecision
}): 'work' | 'review' | 'merge' | 'done' {
  if (args.issueClosed) return 'done';
  if (!args.hasPr) return 'work';
  const approved =
    args.record?.pipeline?.readyForMerge === true ||
    args.record?.pipeline?.reviewStatus === 'passed' ||
    args.reviewDecision === 'APPROVED';
  return approved ? 'merge' : 'review';
}
```

**Proving test:** `tests/unit/lib/reconstruct/derive-phase.test.ts` — a table-driven test over all four branches, including the D7 fallback (null record but `reviewDecision === 'APPROVED'` → `'merge'`; null record + open PR + no approval → `'review'`).

// db.ts — new rebuildCommand (mirror rebuildAgentsCommand style)
async function rebuildCommand(options: { verbose?: boolean }): Promise<void> {
  const spinner = ora('Reconstructing cache from git + GitHub sources...').start();
  try {
    const db = getDatabase();
    const { reconstructCache } = await import('../../lib/reconstruct/reconstruct-cache.js');
    const r = await reconstructCache(db, { verbose: options.verbose });
    const phases = Object.entries(r.phaseCounts).map(([p, n]) => `${p}=${n}`).join(' ');
    spinner.succeed(`Reconstructed: ${r.issuesEnumerated} in-flight issue(s), ${r.agentsRebuilt} agent(s); phases ${phases}`);
  } catch (error: any) {
    spinner.fail(`Reconstruct failed: ${error.message}`);
    process.exitCode = 1;
  }
}
