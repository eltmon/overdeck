## 1. Background & motivation (read once, then skip)

Directive and Panopticon agree on the destination (machine-readable plans, small verifiable items, dependency DAGs) but differ on **where correctness is enforced**. Directive's axiom: *"Every rule MUST use the strongest applicable layer. Order: deterministic > Taskfile > vBRIEF > RFC2119 > prose."* It backs its planning rules with Python lints (`scripts/_vbrief_story_quality.py`, `scripts/vbrief_validate.py` in the Directive repo). Panopticon enforces plan quality with prompt prose only: today's finalize validation checks structure (two top-level keys, `plan.id`, item-count ↔ bead-count) and will happily promote a plan whose every AC is "works as expected".

Separately, the comparison surfaced five **drift bugs** — contradictions between role files, spawn templates, and runtime code — including one where work agents are instructed to hand-edit a file the verification gate stopped reading when PAN-1124 moved the source of truth to main.

This PRD fixes the drift bugs, adds structural protection against recurrence (a prompt lint, modeled on `scripts/lint-skills.sh`), adopts Directive's quality-lint layer at finalize, and ports its highest-value prompt practices. The swarm-contract material (`files_scope`, `verify_commands`, readiness lint) is **out of scope** — it belongs to PAN-1762.

## 3. Intersecting repo rules (restated — do not assume you recall them)

- **PAN-1124 invariant:** the spec on main is immutable after planning; only `plan.status` may change, via `updateSpecStatus()`. `updateItemStatus`/`updateSubItemStatus` write ONLY to `statusOverrides`.
- **Fake timers:** any test exercising retries/delays MUST use `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()`. Never real `setTimeout` waits; never `maxForks: 1` to mask OOM.
- **No `execSync`** in dashboard-server-reachable code. Use `execAsync`/`spawn`.
- **Commit often on main:** each work item below is a separate commit as soon as it builds; never batch the session into one commit. Push before ending the session.
- **Skills ↔ CLI convention:** if a CLI flag changes (`pan plan --probe`, `--no-quality-lint`), the wrapper skill (`skills/pan-plan/SKILL.md`) changes in the same commit; `scripts/lint-skills.sh` enforces this in CI.
- **Quality gates before done:** `npm run typecheck`, `npm run lint`, `npm test`.
- **`.pan/continue.json` is gitignored** — never tracked. `.pan/specs/`, `.pan/drafts/`, `.pan/continues/`, and (new) `.pan/context/` are tracked.
- **bd for tracking:** use `bd` for task tracking during implementation, not TodoWrite/markdown lists.

#### WI-2: Canonicalize the per-bead order in roles/work.md (close → inspect)

**Why:** [roles/work.md:53-65](../../roles/work.md) runs inspection (step 7) **before** `bd close` (step 10). The spawn template and `pan inspect`'s own description use close-first (decision D2). Two state machines for the agent's core loop.

**File:** [roles/work.md](../../roles/work.md), section `## Per-Bead Workflow` (search `For every bead:`).

**Before (steps 6-11):**


### Phase 3 — Finalize quality lint

#### WI-7: Story-quality lint module

**Why:** §1. Port of Directive's `_vbrief_story_quality.py`, adapted (no UserStory regex — D9).

**New file:** `src/lib/vbrief/quality-lint.ts`. Export:

```ts
export interface QualityIssue { itemId: string | null; rule: string; message: string; severity: 'error' | 'warn'; }
export function lintPlanQuality(doc: VBriefDocument): QualityIssue[];
```

(`VBriefDocument` from [src/lib/vbrief/types.ts:135](../../src/lib/vbrief/types.ts).)

**Rules (constants exported for WI-14's prompt rendering):**

```ts
export const PLACEHOLDER_AC_PATTERNS = ['acceptance criteria for', 'copy from parent', 'copy from specification', 'placeholder', 'refine from parent', 'tbd', 'to be defined', 'to refine', 'todo'];
export const DOCS_ONLY_AC_PATTERNS = ['docs updated', 'documentation updated', 'readme updated', 'update docs', 'update documentation', 'update readme'];
export const VAGUE_AC_PATTERNS = ['displays a message', 'handles errors', 'is implemented', 'is updated', 'passes tests', 'shows a message', 'updates the ui', 'works as expected', 'make it work', 'implement the feature', 'change the code', 'update the code'];
export const OBSERVABLE_TERMS = ['blocks', 'creates', 'deletes', 'displays', 'emits', 'fails', 'persists', 'records', 'redirects', 'rejects', 'renders', 'returns', 'saves', 'shows', 'stores', 'updates', 'validates', 'exits', 'prints', 'logs', 'throws', 'spawns', 'opens', 'closes', 'sends', 'receives', 'resolves', 'refuses', 'marks', 'syncs', 'commits', 'pushes', 'when ', 'given ', 'then '];
```

Per item (skip items with `status: 'cancelled'`):
1. **AC presence:** ≥1 subItem with `metadata.kind: 'acceptance_criterion'` → else error `ac-missing`.
2. **AC band:** 2–5 ACs, unless item has non-empty `metadata.acJustification` (string) → else error `ac-count`.
3. **Banned phrasing:** each AC title, lowercased, must not contain any `PLACEHOLDER_AC_PATTERNS`, `DOCS_ONLY_AC_PATTERNS`, or `VAGUE_AC_PATTERNS` substring → error `ac-banned-phrase` naming the matched pattern.
4. **Observable behavior:** each AC title, lowercased, must contain ≥1 `OBSERVABLE_TERMS` substring → else error `ac-not-observable`.
