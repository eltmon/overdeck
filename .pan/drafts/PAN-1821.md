# PAN-1821 — Linear responses never feed `updateRateLimit`; pollers have zero rate-limit awareness

**Issue:** [PAN-1821](https://github.com/eltmon/panopticon-cli/issues/1821)
**Parent context:** [PAN-1817](https://github.com/eltmon/panopticon-cli/issues/1817) (Linear quota exhaustion by IssueDataService polling)
**Planning mode:** `pan plan --auto` (no interactive Q&A; inferred choices recorded in `## Auto Decisions`)

---

## Glossary

- **Tracker** — an issue source: `'github'`, `'linear'`, or `'rally'`. Each has its own poll loop in `IssueDataService`.
- **`IssueDataService`** — `src/dashboard/server/services/issue-data-service.ts`. Owns the per-tracker poll loops (`pollGitHub`/`pollLinear`/`pollRally`), scheduling (`scheduleNext`), and diagnostics (`getDiagnostics`).
- **`CacheService`** — `src/dashboard/server/services/cache-service.ts`. SQLite-backed cache. Owns the `rate_limits` table and the methods `updateRateLimit`, `getRateLimit`, `getBackoffMs`, `shouldBackoff`.
- **`rate_limits` table** — one row per tracker: `(tracker, remaining, total, reset_at, updated_at)`.
- **Backoff** — `getBackoffMs(tracker, base)` returns an additive interval bump based on the remaining/total ratio (0, 1×, 4×, or 9× base). Already applied in `scheduleNext`.
- **Suspension** — a stronger state than backoff: when a tracker's quota is fully exhausted (`remaining ≤ 0`, `reset_at` in the future), polling stops entirely until the window resets, then resumes. **New in this issue.**
- **Reset epoch unit** — GitHub's `x-ratelimit-reset` is **Unix seconds**; Linear's request-window reset header is **Unix milliseconds**. This difference is a landmine — see FR-2 / D2.

---

## Problem

`updateRateLimit('linear', …)` is **never called**. The only `updateRateLimit` call sites are GitHub:

- `src/dashboard/server/services/issue-data-service.ts:861` (inside `fetchGitHubRepoIssues` paginate callback)
- `src/dashboard/server/services/substrate-bug-poller.ts:94` (`CacheServiceRateLimitStore.updateFromHeaders`)

The Linear fetch path — `pollLinear` → `fetchLinearIssues` (`issue-data-service.ts:949` / `:1010`) — issues a raw `fetch('https://api.linear.app/graphql', …)` (`:1089`) and **discards `response.headers` entirely**. It only reads the JSON body, and on `json.errors` it throws (`:1100-1102`).

Consequences:

1. `getRateLimit('linear')` always returns `null` ⇒ `getBackoffMs('linear', …)` always returns `0` (`cache-service.ts:337-339`) ⇒ `scheduleNext` never lengthens the Linear interval.
2. When Linear returns the 2,500-req/hr rate-limit error (observed in PAN-1817: "Only 2500 requests are allowed per 1 hour"), `pollLinear` catches the thrown error, sets `lastError`, and `scheduleNext` immediately reschedules at the **base** interval — an error loop, not a backoff. PAN-1817 logged 84+ such errors in one window.
3. Quota-exhaustion is invisible to operators: `pan close` / `pan wipe` tracker steps fail silently when the window is gone, and there is no doctor/health signal.

This issue is the **complement** to PAN-1817: PAN-1817 reduces *request volume* (batching, peer-dashboard cache-only); PAN-1821 adds *rate-limit awareness* so a host poller backs off and suspends instead of error-looping, regardless of what exhausts the quota.

## Requirements

### Functional

- **FR-1** — Successful Linear GraphQL responses populate the persisted `'linear'` rate-limit state (`remaining`, `total`, `resetAt`) by parsing the response headers and calling `updateRateLimit('linear', …)`, symmetric with the GitHub path.
- **FR-2** — The Linear reset timestamp is normalized correctly regardless of whether the header value is encoded in Unix **seconds** or **milliseconds** (Linear uses ms; GitHub uses s). The persisted `resetAt` is a valid future ISO-8601 string.
- **FR-3** — On a Linear rate-limit error (HTTP 429 and/or GraphQL `RATELIMITED`), the poller records quota exhaustion (`remaining: 0`) and a reset time — using the rate-limit headers if present, else the `Retry-After` header — **before** the existing error is thrown.
- **FR-4** — When any tracker's quota is exhausted (`remaining ≤ 0` and `reset_at` in the future), `scheduleNext` suspends that tracker's polling until the reset time (clamped to a safety ceiling), then resumes automatically. Non-exhausted trackers keep the existing graduated backoff.
- **FR-5** — `getDiagnostics()` (the `/api/cache-status` payload) exposes per-tracker `backoffMs`, `suspendedUntil` (ISO string or `null`), and `rateLimited` (boolean).
- **FR-6** — `pan doctor` shows a `warn` row when a tracker is rate-limited or suspended, naming the tracker and the reset time; `ok` otherwise.

### Non-functional

- **NFR-1** — No change to GitHub/Rally/Linear poll cadence under normal (non-exhausted) conditions. Suspension engages only at true exhaustion (`remaining ≤ 0`).
- **NFR-2** — Every test touching backoff/suspension/Retry-After timing uses `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()` (repo rule: `fake-timers-for-retry-tests`). No real `setTimeout` waits.
- **NFR-3** — No new SQLite table or schema migration. Suspension is derived from the existing `rate_limits` row.
- **NFR-4** — No `execSync` in any server-reachable code (PAN-70). Doctor's existing `CacheService` read pattern is fine (CLI-side).

## Decisions made in this document

- **D1 (header names)** — Linear's per-request rate-limit headers are `x-ratelimit-requests-limit`, `x-ratelimit-requests-remaining`, `x-ratelimit-requests-reset` (NOT the GitHub names `x-ratelimit-limit`/`-remaining`/`-reset`). The work agent MUST use the `-requests-` names. **Implementation checkpoint:** if a live capture of a Linear response header set is available, confirm these names; if the `-requests-` headers are absent, fall back to the bare `x-ratelimit-*` names, then to `Retry-After`. Parsing must be defensive: a missing/unparseable header simply skips the update (mirrors `parseIntegerHeader` returning `null`), never throws.
- **D2 (reset unit normalization)** — Do NOT blindly copy GitHub's `* 1000`. Normalize defensively: a reset value `< 1e12` is treated as **seconds** (multiply by 1000); `≥ 1e12` is treated as **milliseconds** (use as-is). This is unambiguous for any real date from ~2001 to ~33000 and is correct for both GitHub-style (seconds) and Linear-style (ms) without guessing.
- **D3 (suspend trigger)** — Suspension triggers when `remaining ≤ 0` **and** `reset_at` is in the future. Between `0 < remaining < 10%`, the existing `getBackoffMs` 9× tier still applies (no change). This keeps suspension surgical to true exhaustion.
- **D4 (suspension ceiling)** — Clamp suspension delay to a hard ceiling of **3,600,000 ms (1 hour)** so a malformed far-future `reset_at` can never wedge a poller forever. The Linear window is 1 hour, so this never truncates a legitimate suspension.
- **D5 (generic, not Linear-only)** — The suspension mechanism in `scheduleNext` and `getSuspensionMs` is tracker-generic, not Linear-special-cased. GitHub already feeds `updateRateLimit`, so it inherits the same protection; this is strictly an improvement and avoids a one-off Linear branch. (GitHub's 5,000/hr ceiling means it effectively never trips today.)
- **D6 (health surface split)** — PAN-1821 owns the *backend* signal: the doctor row (FR-6) and the `cache-status` diagnostics fields (FR-5). The *frontend* health-strip visual remains PAN-1817 ask #4's scope and is a NonGoal here.

### W1 — Feed `updateRateLimit('linear', …)` from Linear response headers *(foundation)*

**Files:** `issue-data-service.ts` (`fetchLinearIssues`, ~:1089-1102)

Parse the rate-limit headers off `response` immediately after the `await fetch(...)` resolves and **before** `response.json()` / the `json.errors` throw, so the update happens even when the body carries a GraphQL error. Mirror the GitHub callback at `:854-862` and the `parseIntegerHeader` helper at `substrate-bug-poller.ts:105-110`.

Before (`:1089-1102`):
```ts
const response = await fetch('https://api.linear.app/graphql', { … });
const json = await response.json();
if (json.errors) {
  throw new Error(json.errors[0]?.message || 'Linear GraphQL error');
}
```
After (shape):
```ts
const response = await fetch('https://api.linear.app/graphql', { … });
this.recordLinearRateLimit(response.headers);   // FR-1/FR-2 — never throws
const json = await response.json();
if (json.errors) {
  throw new Error(json.errors[0]?.message || 'Linear GraphQL error');
}
```
`recordLinearRateLimit(headers)` (new private method or inline helper): parse `x-ratelimit-requests-remaining` / `-limit` / `-reset` (D1 fallback chain), normalize reset per D2, and call `this.cache.updateRateLimit('linear', { remaining, total, resetAt })` only when remaining & total parsed. Reuse/extract a `parseIntegerHeader(headers, name)` helper.

### W2 — Record exhaustion on Linear 429 / `RATELIMITED` *(edge path; depends on W1)*

**Files:** `issue-data-service.ts` (`fetchLinearIssues`)

When `response.status === 429` **or** the GraphQL error indicates rate-limiting (message contains "rate limit" / "RATELIMITED", consistent with `linear-client.ts:139`), ensure `'linear'` is recorded as exhausted (`remaining: 0`) before the throw. ### W3 — `getSuspensionMs` on `CacheService` *(foundation for W4/W5/W6)*

**Files:** `cache-service.ts`

Add a pure method:
```ts
/** ms until the tracker's rate-limit window resets when fully exhausted; 0 otherwise. */
getSuspensionMs(tracker: string, now: number = Date.now()): number {
  const limit = this.getRateLimit(tracker);
  if (!limit) return 0;
  if (limit.remaining > 0) return 0;
  const resetMs = new Date(limit.resetAt).getTime();
  if (!Number.isFinite(resetMs) || resetMs <= now) return 0;
  return resetMs - now;     // caller clamps to ceiling (D4)
}
```
No schema change (NFR-3). `now` param makes it unit-testable with fake timers.

**ACs:** see vBRIEF `cache-suspension-method`.

### W4 — Honor suspension in `scheduleNext` *(depends on W3)*

**Files:** `issue-data-service.ts` (`scheduleNext`, :570-598)

After computing `effectiveInterval`, check suspension and let it win, clamped to the D4 ceiling:
```ts
const suspendMs = this.cache.getSuspensionMs(tracker);
const delay = suspendMs > 0
  ? Math.min(suspendMs, 3_600_000)          // D4 ceiling
  : effectiveInterval;
if (suspendMs > 0) {
  console.warn(`[IssueDataService] ${tracker} rate-limited; suspending polls for ${Math.round(delay/1000)}s`);
}
state.currentInterval = delay;
state.timer = setTimeout(/* … unchanged … */, delay);
```
Suspension bypasses the `intervals.max` (5-min) clamp; non-exhausted path is unchanged (NFR-1).

**ACs:** see vBRIEF `suspend-wire`.

## Auto Decisions (recorded because `--auto` suppressed Q&A)

1. **Scope = backend rate-limit awareness only.** Frontend health-strip is left to PAN-1817 ask #4 (D6). *Rationale:* the issue title and asks are about poller awareness + a doctor/health signal; duplicating PAN-1817's UI would create drift.
2. **Suspension made generic across trackers (D5)** rather than Linear-special-cased. *Rationale:* GitHub already feeds `updateRateLimit`; a generic mechanism is simpler, has no behavioral downside (github never trips), and avoids a one-off branch.
3. **Defensive seconds-vs-ms reset normalization (D2)** instead of asserting Linear's unit from docs alone. *Rationale:* cannot hit the live Linear API during planning without burning the very quota in question; the magnitude heuristic is correct for both encodings and removes the guess.
4. **No new schema/table (NFR-3); suspension derived from the existing `rate_limits` row.** *Rationale:* `remaining ≤ 0 && reset_at > now` fully determines suspension; a new column/table would be dead weight.
