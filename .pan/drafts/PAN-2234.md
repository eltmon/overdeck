## 0. Glossary (define before use)

- **PRD draft** — the free-form narrative implementation brief at
  `<projectRoot>/.pan/drafts/<ISSUE-ID>.md` (uppercase filename is canonical —
  `getIssueDraftPath` in `src/lib/pan-dir/drafts.ts` uppercases; a legacy minority of
  drafts on main are lowercase). First artifact of the four-artifact model; the vBRIEF is
  *lowered from* it.
- **Spec promotion** — writing the workspace's vBRIEF to
  `<projectRoot>/.pan/specs/<...>.vbrief.json` with `plan.status: "proposed"` and
  committing it on main. Happens inside `completePlanningArtifacts` (grep anchor:
  `emitCompletePlanningPhase(upperIssueId, 'specWrite', 'start'`).
- **complete-planning endpoint** — `POST /api/issues/:id/complete-planning` in
  `src/dashboard/server/routes/issues.ts`. The single promotion door: called by
  `pan plan finalize` (via `promotePlanning` in `src/cli/commands/plan-finalize.ts`), by
  `pan plan done` (same function, `src/cli/commands/plan-done.ts`), and by the dashboard
  Done button.
- **Finalize** — `pan plan finalize` (`planFinalizeCommand`): quality-lints the workspace
  vBRIEF, creates beads, stamps `status=proposed`, then chains to complete-planning.
- **Prompt-only vs mechanical gate** — `roles/plan.md` (this sha) instructs: *"Write the
  PRD draft … Do not proceed to the vBRIEF until the PRD is on disk"* (grep anchor:
  `Do not proceed to the vBRIEF until the PRD is on disk`). That is a prompt; prompts
  drift while gates hold (the soul-degradation lesson). This issue adds the gate.
- **Auto-start exemption** — `pan start --auto` synthesizes a minimal vBRIEF for trivial
  issues without ever running a planning agent; it must not be blocked (it structurally
  isn't — see D6).
- **Escape hatch** — `--no-prd`: a loud, explicit bypass for genuinely trivial issues that
  went through interactive planning anyway.

## 2. Locked design decisions

- **D1 — One shared gate helper, sync, in the drafts module.**
  `checkPrdGateSync(args)` lives in `src/lib/pan-dir/drafts.ts` next to the existing path
  helpers. Sync (`node:fs` `existsSync`/`readFileSync`/`statSync`) because the CLI caller
  is a sync flow and the route already uses sync fs at request time (grep anchor:
  `if (existsSync(remoteMetadataPath))` inside the same handler). The no-`execSync` rule
  bans child-process sync, not fs sync.
- **D2 — Search order: projectRoot uppercase → projectRoot lowercase → workspace
  uppercase → workspace lowercase; first existing file wins.** Rationale: projectRoot
  uppercase is the canonical location (`roles/plan.md`, `getIssueDraftPath`); lowercase
  covers the legacy minority of drafts on main; the workspace pair covers planning agents
  that wrote into their cwd (which is what the existing `readPrdDraftText` already reads).
  A draft found anywhere in that chain satisfies the gate.
- **D3 — Non-trivial = at least 20 lines.** `MIN_PRD_LINES = 20` exported from
  `drafts.ts`; check is `content.split('\n').length >= MIN_PRD_LINES` on the first found
  file. The issue says "> ~20 lines"; 20 exactly is the locked constant. A found-but-thin
  draft fails with a distinct reason (`too-short`) naming the path and its line count.
- **D4 — Two enforcement points; `completePlanningArtifacts` stays untouched.**
  (a) CLI: `planFinalizeCommand` fails fast BEFORE quality lint and beads creation, exit
  code **4** (1 = promotion failed, 2 = beads failed, 3 = quality lint — verified; 4 is
  free). (b) Server: the route's existing pre-check block (grep anchor:
  `assertPlanQuality(workspaceDoc);` inside `if (workspacePath) {` after
  `// Determine project path`) gains the PRD check and returns **422** in the exact shape
  of the `PlanQualityLintError` mapping beside it. `completePlanningArtifacts` is not
  modified: its only production caller is this route (verified:
  `grep -rn "completePlanningArtifacts" src/ --include="*.ts"` → route + tests only), and
  the route gate runs earlier.
- **D5 — Escape hatch `--no-prd`, propagated end-to-end, loud.** New commander negation
  flag on BOTH `plan finalize` and `plan done` (`--no-prd` arrives as `prd: false`).
  Finalize skips its local check AND passes `noPrd: true` in the complete-planning body
  (otherwise the endpoint would 422 the same run); `plan done` passes it through
  identically. The endpoint honors `body.noPrd === true` by skipping the check and
  emitting a `prdGate`/`skipped` phase event. CLI prints a yellow
  `⚠ PRD gate SKIPPED (--no-prd)` mirroring the `--no-quality-lint` treatment (grep
  anchor: `quality lint SKIPPED (--no-quality-lint)`).
- **D6 — The auto-start exemption is structural; write no code for it.** Verified:
  `pan start --auto` (`src/cli/commands/start.ts`, grep anchor:
  `await Effect.runPromise(writeAutoStartVBrief(projectRoot, workspace, issue));`) and the
  dashboard auto path (`src/dashboard/server/routes/agents.ts`, grep anchor:
  `yield* writeAutoStartVBrief(projectPath, workspacePath, {`) synthesize the spec and
  create beads directly — neither ever POSTs complete-planning, so the gate cannot block
  them. Document this in the gate's failure message docs, do not special-case it in code.
- **D7 — Observability follows existing idioms.** Server: extend the complete-planning
  phase union with `'prdGate'` and emit start/success/failure/skipped around the check
  (grep anchor for the union/emitter: `message: \`complete-planning.phase=${phase}\``).
  CLI: no new phase events — the quality-lint failure path emits none either (verified);
  match it.

## 4. Requirements

### Functional
- **FR-1** `checkPrdGateSync({ projectRoot, workspacePath, issueId })` returns
  `{ ok: true, path, lineCount }` when a draft is found per D2 with ≥ `MIN_PRD_LINES`
  lines; `{ ok: false, reason: 'missing', searched: string[] }` when no candidate exists;
  `{ ok: false, reason: 'too-short', path, lineCount }` when the first found draft is
  thinner than the minimum. `projectRoot` may be null/empty (workspace-only search).
- **FR-2** `pan plan finalize` without a qualifying PRD exits **4** before creating beads
  or stamping the plan, with this message (exact text, so agents can pattern-match):
  `✗ PRD-first gate: no PRD draft found for <ISSUE-ID>. Write <projectRoot>/.pan/drafts/<ISSUE-ID>.md first (roles/plan.md, Outputs #1), then re-run finalize. For a genuinely trivial issue use --no-prd.`
  (too-short variant names the found path and line count instead of "no PRD draft found").
- **FR-3** `pan plan finalize --no-prd` skips the local check, prints the yellow skip
  warning, and sends `noPrd: true` in the complete-planning body. `--no-prd` is also
  accepted by `pan plan done` and propagated the same way.
- **FR-4** `POST /api/issues/:id/complete-planning` without a qualifying PRD returns 422
  `{ error: 'PRD-first gate: no PRD draft for <ISSUE-ID>', prdGate: <FR-1 failure object> }`
  from the pre-check block — before any spec write, bead materialization, git commit, or
  session kill side effects beyond those the handler already performed earlier (the
  planning-agent stopped-marking at the top of the handler is unchanged and acceptable:
  it is cosmetic and idempotent).
- **FR-5** `body.noPrd === true` skips the endpoint check with a `prdGate`/`skipped`
  phase event; a passing check emits `prdGate`/`success`; a failing one
  `prdGate`/`failure`.
- **FR-6** With a qualifying PRD present, finalize and complete-planning behave byte-for-
  byte as today (no new output on the success path except the D7 phase events).
- **FR-7** `pan start --auto` remains unaffected (structural, D6) — covered by an
  assertion in review, not by new code.
- **FR-8** `sync-sources/skills/pan-plan/SKILL.md` documents `--no-prd` on both
  `finalize` and `done` in the same commit as the flag (lint-skills cross-checks
  documented flags against `pan plan finalize --help`).

### Non-functional
- **NFR-1** No `execSync`/child-process sync anywhere (server-reachable code); fs-sync
  only, per D1.
- **NFR-2** Gate adds no network calls and O(4) stat/read operations — negligible latency.
- **NFR-3** All modified files stay under their file-size baseline (`routes/issues.ts` is
  baselined at 4,065 — the gate adds ~25 lines, so the baseline must be BUMPED in the same
  commit **with this issue reference in the commit message** (PAN-2227's audited-bump
  rule; if PAN-2227 has landed, the pre-push audit enforces exactly this). Alternative if
  the executor prefers: put the gate call in a small local function and accept the bump
  anyway — `issues.ts` may not shrink by side quest; route thinning is PAN-2148.)
- **NFR-4** Retry/backoff tests in `plan-finalize.test.ts` already exist around
  `promotePlanning`; any test you add or touch there that involves its backoff MUST use
  `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()` — never real timers.

## 6. Intersecting repo rules (restated — do not assume recall)

- **No `execSync` in server-reachable code** (`routes/issues.ts`, `pan-dir/drafts.ts` are
  server-reachable): fs-sync is fine, child-process sync is not; delays via
  `await new Promise(r => setTimeout(r, ms))`.
- **Fake timers** for anything touching `promotePlanning`'s retry/backoff in tests:
  `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()`; never real `setTimeout`; never
  `maxForks: 1` to mask it.
- **Async tmux primitives only** — n/a; this change touches no tmux path (the session
  kill deferral in the route is untouched).
- **File-size guard**: `routes/issues.ts` is baselined (4,065 at authoring) — audited bump
  required (NFR-3). If PAN-2227 has landed, expect the pre-push hook to enforce the issue
  reference; if PAN-2231 has landed, your new tests must not read `src/` sources (they
  don't — they exercise behavior).
- **Single-spec-on-main / PAN-1124**: the gate must NOT write or mutate any spec or draft
  — it is read-only. Draft writes stay with the planning agent / `writeIssueDraft`.
- **ESLint `--no-inline-config`** — no disable comments.
- **commitlint**: all-lowercase subject, scope enum `cloister, dashboard, workspace, cli,
  review, beads, db, specialists, terminal, infra, deps`. Suggested:
  `feat(cli): mechanical prd-first gate in plan finalize and complete-planning (pan-2234)`.
- **Never `--no-verify`; never `git stash`; wrapper skill updated in the same commit as
