## Glossary

- **Sessions page** — the dashboard tab at `/sessions`, rendered by `ConversationsPage.tsx`. Browses *discovered sessions* and *managed-archived conversations*.
- **Discovered session** — a row in the `discovered_sessions` SQLite table (overdeck.db), produced by the scanner walking `~/.claude/projects/` (PAN-457). May or may not correspond to an Overdeck-managed conversation.
- **Managed conversation** — a row in the `conversations` table; a session Overdeck spawned (Mission Control list). **Managed-archived** = a managed conversation with `archived_at NOT NULL`.
- **Conversation view / `ConversationPanel`** — the canonical transcript renderer used by Command Deck, inspector panels, drawers, and popouts (`src/dashboard/frontend/src/components/chat/ConversationPanel.tsx`). PAN-1917's goal is to reuse it on /sessions.
- **`MessagesTimeline`** — the virtualized message list inside `ConversationPanel`; can be fed `ChatMessage[]` directly.
- **Feed** — the new unified, server-side-paginated list of discovered + managed-archived rows this PRD introduces (replaces the client-side merge).
- **Facet** — a filter group in the left sidebar (model, tags, tools, files, time, cost, enrichment level, source, workspace) with per-value counts.
- **Keyset cursor** — pagination by `WHERE (last_ts, id) < (:lastTs, :id) ORDER BY last_ts DESC, id DESC` instead of `OFFSET`; stable under concurrent inserts.
- **WS RPC** — the Effect RPC over WebSocket at `/ws/rpc`; methods declared in `packages/contracts/src/rpc.ts` (`WS_METHODS`), executed via the SQLite worker (`dashboard-db-worker.ts`).
- **Theme tokens** — the app-wide Tailwind semantic classes (`bg-background`, `bg-card`, `text-muted-foreground`, `border-border`) that make light/dark theming work.
- **No-loss audit** — repo rule: a redesign of an existing surface must enumerate every old affordance and prove each has a home in the new surface, gated by a test.

### Canonical conversation view (the reuse target)

- `ConversationPanel` props (`ConversationPanel.tsx:74-108`): the only **required** prop is `conversation: Conversation`; `embedded?: boolean` hides header chrome/composer. Six surfaces already embed it this way (e.g. `CommandDeck/SessionView/SessionPanel.tsx:377`, `drawer/DrawerAgentSession.tsx:221`).
- The exact embed recipe: `App/StandaloneRoutes.tsx:48-90` (`StandaloneConversationPopoutRoute`) — `fetch('/api/conversations/' + numericId)` → `Conversation` → `<ConversationPanel conversation={conversation} />`. `EventRouter` is already mounted globally (`App.tsx:648`), so live streaming works for any inline embed.
- Message data is hybrid: REST backfill `GET /api/conversations/:name/messages` (route at `src/dashboard/server/routes/conversations.ts:3327`) + WS `subscribeConversationMessages` merged in `useConversationMessagesStream.ts`.
- `MessagesTimeline` (`chat/messagesTimeline/MessagesTimeline.tsx`) is virtualized (`@tanstack/react-virtual`). Its props (`chat/messagesTimeline/types.ts:23-50`) require only `messages: ChatMessage[]`, `workLog: WorkLogEntry[]`, `streaming: boolean` — everything else optional. A transcript can be rendered without a `Conversation` object.

### The integration gap

`DiscoveredSessionSnapshot` (`packages/contracts/src/types.ts:628-659`) exposes `conversationTitle` but **no conversation id or name** — quote from the type: `// Human title of the tracked conversation this session belongs to, when the / // session id matches a conversation in the DB (joined via conversation_files).` The join already exists server-side: `attachConversationTitles` (`src/lib/overdeck/discovered-sessions.ts:228`), quote: `SELECT cf.locator AS sid, c.title AS title FROM conversation_files cf JOIN conversations c ON c.id = cf.conversation_id` — it just doesn't select `c.id`/`c.name`. Without those, the frontend cannot map a row to a `Conversation` for `ConversationPanel`.

Ad-hoc discovered sessions have **no** `conversations` row at all, so `ConversationPanel` / `/api/conversations/:name/messages` can never serve them; they need a messages-by-discovered-session endpoint (WI-4).

## Requirements

### Functional

- **FR-1** The page lists discovered and managed-archived sessions as ONE server-side-paginated feed, globally sorted by `last_ts DESC`, with keyset cursoring ("load more" reaches all rows).
- **FR-2** Facet counts (model, tags, tools, files, time, cost, enrichment, source) are computed server-side over the **full filtered corpus**, not the loaded page.
- **FR-3** Feed rows carry `conversationId` and `conversationName` when the session maps to a managed conversation (via the existing `conversation_files` join).
- **FR-4** Selecting a row with a managed conversation renders the canonical `ConversationPanel` (embedded mode) with live streaming — the same renderer as Command Deck.
- **FR-5** Selecting an ad-hoc (unmanaged) row renders its transcript via `MessagesTimeline`, served by a new messages-by-discovered-session endpoint.
- **FR-6** Selection is deep-linkable: `/sessions?session=<source>:<id>` selects on load and updates on click. A "Open in Command Deck" action navigates to `buildConversationUrl(conversationId)` for managed rows.
- **FR-7** Every existing affordance is retained: keyword search, semantic search toggle, Filters toggle, all nine facet groups, active-filter chips, stats header, source filter, ScanButton + progress, enrichment L1/L2/L3 actions, embed action, **Unarchive** button, metadata fields (summary/detailed/tags/tools/files/jsonlPath), search pagination.
- **FR-8** The page uses theme tokens and style-guide-compliant badges/typography (squared badges, no decorative color, no `font-semibold` headers, no `bg-gray-*`).

### Non-functional

- **NFR-1** The session list is virtualized; smooth scroll at 10,000 rows.
- **NFR-2** Feed list rows exclude `summaryDetailed`, `toolsUsed`, `filesTouched` (detail panel fetches the full row via existing `getDiscoveredSession`).
- **NFR-3** A 50-row feed page (query + mapping) completes in <100 ms against a 10,000-row corpus, measured by a bench test (baseline measured first — issue AC).
- **NFR-4** No `execSync` anywhere in server-reachable code (repo rule); all new DB work runs through the existing `runDashboardDbJob` worker.
- **NFR-5** Any retry/delay logic in new tests uses `vi.useFakeTimers()` (repo rule).

## Decisions (made here — do not re-litigate downstream)

- **D-1** Modify **only** the live overdeck data layer (`src/lib/overdeck/…`). The legacy twins (`src/lib/database/discovered-sessions-db.ts`, `conversations-db.ts`) are frozen; PAN-1983/PAN-1984 delete them.
- **D-2** Browse pagination is **keyset** (`(last_ts, id)` descending). FTS search keeps its existing offset paging (rank-ordered; already works).
- **D-3** New backend code goes in a **new module** `src/lib/overdeck/sessions-feed.ts` (file-size ratchet: don't grow `discovered-sessions.ts`/`conversations.ts`).
- **D-4** The detail region replaces `SessionDetail`'s panel role: a compact **metadata header** (collapsible, keeps ALL current fields + enrich/embed/unarchive actions) above the **transcript**. Managed → `ConversationPanel embedded`; unmanaged → `MessagesTimeline` with `workLog={[]}`, `streaming={false}`.
- **D-5** The REST `GET /api/conversations/archived` endpoint **stays** (CLI/API compat, PAN-1391) even though the page moves to the unified feed.
- **D-6** The new unmanaged-transcript endpoint is REST (`GET /api/discovered-sessions/:id/messages`) mirroring the response shape of `GET /api/conversations/:name/messages`, so frontend message types are reused unchanged. It parses with the **same transcript-adapter stack** that endpoint uses. Shape it so a harness parameter can be added later (PAN-2224) — but this PRD implements Claude Code parsing only, which covers 100% of today's discovered rows.
- **D-7** Facet time buckets: `24h / 7d / 30d / older`; cost buckets: `<$0.10 / $0.10–1 / $1–10 / >$10` (match the existing client-side `countTimeFacets`/`countCostFacets` buckets in `ConversationsPage.tsx` so the UI is unchanged).
- **D-8** Source chips remain `All / Discovered / Managed-archived`; implemented as a feed filter, not separate queries.

1. `listSessionsFeed(filter: SessionsFeedFilter): { rows: SessionsFeedRow[]; nextCursor: string | null }` — a UNION of:
   - `discovered_sessions` (projected columns: `id, jsonl_path, session_id, workspace_path, message_count, first_ts, last_ts, primary_model, token_input, token_output, estimated_cost, tags, summary, enrichment_level, enrichment_failed, overdeck_managed, pan_issue_id`), `source = 'discovered'`;
