# PRD: Multi-harness session discovery, search, enrichment + transcript resilience

**Issue:** PAN-2224 (https://github.com/eltmon/overdeck/issues/2224)
**Status:** draft
**Author:** operator conversation, 2026-07-02. All file/line references verified against main at commit time; each carries a search-quote for line-drift recovery.

> **Correction to the issue body:** the issue claims the embedded viewer "cannot render" archived pi/codex transcripts. Research shows the dashboard messages endpoint (`GET /api/conversations/:name/messages`) is **already harness-dispatched and renders pi/ohmypi/codex today** (see Current State §6). The real gaps are: discovery/enrichment/search never index non-Claude sessions; the archived-list `jsonlPath` field fabricates a nonexistent Claude path for them; and transcript resolution dies when an agent dir is cleaned up because there is no discovered-sessions fallback. This PRD supersedes the issue's gap list.

## Glossary

- **Harness** — the coding-agent runtime for a session: `claude-code`, `pi` (legacy), `ohmypi` (live pi-family), `codex`. The pi-family shares one on-disk format ("Pi v3").
- **Discovered session** — a row in `discovered_sessions` (overdeck.db) produced by the scanner (PAN-457). Today all 6,861 rows are Claude Code.
- **Ad-hoc session** — a session file created outside Overdeck management (a user running `claude`/`pi`/`omp`/`codex` directly).
- **Agent dir** — `~/.overdeck/agents/<agent-id>/`; pan-spawned pi-family agents write sessions under `<agentDir>/sessions/` (work/review pi agents: agent-dir **root**, PAN-1908); pan-spawned codex agents under `<agentDir>/codex-home/sessions/`.
- **Rollout** — a Codex session transcript: `$CODEX_HOME/sessions/YYYY/MM/DD/rollout-<uuid>-<threadId>.jsonl`, lines with nested `payload` objects.
- **Pi v3 format** — first line `{ type: "session", version: 3, id, timestamp, cwd }`; subsequent lines form an `id`/`parentId` tree; message lines are `{ type: 'message', message: { role, content } }` with `toolCall`/`toolResult` blocks. ohmypi uses the identical format under `~/.omp/`.
- **Locator** — `conversation_files.locator`: the harness-native session id linking a managed conversation to its transcript file (Claude session UUID, pi session id, codex thread id), alongside a `harness` column.
- **Enrichment tiers** — L1 (Haiku, sampled lines) / L2 (Sonnet, reservoir sample) / L3 (deep, near-full transcript); user-triggered LLM summarization of a discovered session.
- **`sessions_fts`** — external-content FTS5 table over `summary, summary_detailed, tags, files_touched`. **Only enriched rows are inserted** (synced manually in code, no SQL triggers).
- **Encoded-cwd** — the directory-name encoding of a workspace path used by Claude (`~/.claude/projects/<encoded>/`) and the pi-family (`sessions/<encoded-cwd>/`).

## Requirements

### Functional

- **FR-1** `discovered_sessions` records a `harness` for every row (`claude-code` | `pi` | `ohmypi` | `codex`); all pre-existing rows are backfilled `claude-code`.
- **FR-2** A system scan discovers session files from all harness roots: Claude projects dir; pi/ohmypi ad-hoc dirs; codex global sessions dir; and per-agent dirs (`sessions/`, agent-dir-root pi JSONLs, `codex-home/sessions/`).
- **FR-3** Per-harness metadata is extracted correctly: session id, message count, first/last timestamps, models, token in/out, estimated cost, tools used. (`filesTouched` is best-effort for pi/codex.)
- **FR-4** Discovered pi/codex sessions belonging to managed conversations are correlated (`overdeckManaged`, `panIssueId`) via locator matching — not Claude path reconstruction.
- **FR-5** Enrichment produces meaningful excerpts/summaries for pi-family and codex sessions, making them reachable via FTS search (which indexes only enriched rows).
- **FR-6** Semantic embeddings work for enriched non-Claude sessions with no embedding-side changes (embeds enriched metadata).
- **FR-7** Archived pi/codex conversation rows never expose a fabricated Claude `jsonlPath`; once scanned, they expose the real discovered path. Their transcripts render even after agent-dir cleanup, via a discovered-sessions fallback in `resolveSessionFile`.
- **FR-8** The Sessions page can facet/filter by harness, and rows show which harness produced them; the `harness` field flows through the REST list route, the WS RPC snapshot, and (if landed) the PAN-1917 unified feed.

### Non-functional

- **NFR-1** Claude Code discovery behavior and scan time are unchanged (existing scanner tests stay green; scan-time measured before/after on the ~6.9k-row corpus).
- **NFR-2** The scanner keeps its zero-sync-FS doctrine (`scanner.ts:12`, quote: `Zero sync FS calls — uses fs/promises throughout`); no `execSync` anywhere server-reachable.
- **NFR-3** The schema migration is idempotent (try/catch ALTER pattern) and applied in both `initDiscoveredSessionsSchema` and `runMigrations`.
- **NFR-4** Any delay/retry logic in new tests uses `vi.useFakeTimers()`.

## Decisions (made here — do not re-litigate downstream)

- **D-1 Harness values & attribution.** Harness is derived from the discovery root that yielded the file: `~/.claude/projects` → `claude-code`; `~/.pi/agent/sessions` → `pi`; `~/.omp/agent/sessions` → `ohmypi`; `~/.codex/sessions` → `codex`; agent dirs → the agent's recorded harness where knowable, else `ohmypi` for `sessions/`-layout pi-family files and `codex` for `codex-home/sessions/`.
- **D-2 Backfill.** Migration sets `harness='claude-code'` for all NULL rows — safe because only Claude was ever scanned (all 6,861 rows).
- **D-3 Discovery-source architecture.** Generalize `discoverJsonlFiles` into a table of per-harness discovery sources `{ harness, roots(), collect(dir) }`, one module (`src/lib/conversations/harness-discovery.ts`), keeping `scanner.ts` orchestration intact. `ScanOptions` keeps its external shape; `system` mode scans all sources.
- **D-4 Metadata parser dispatch.** New `src/lib/conversations/harness-metadata.ts` exporting `parsePiSessionMetadata` and `parseCodexSessionMetadata` returning the existing `SessionMetadata` shape; `scan()` dispatches on the file's harness tag. Reuse the cost parsers' field logic (`pi-parser.ts` / `ohmypi-parser.ts` / `codex-parser.ts`) for usage/cost; use the **nested-payload rollout schema** for codex (per Current State §12), never `extractCodexTranscript`.
- **D-5 Correlation by locator, not path.** For non-Claude files, `buildCorrelationMapSync` matches the parsed `sessionId` against the `conversation_files.locator` set (which already stores harness-native ids). The existing Claude path-reconstruction branch (`correlator.ts:51`) is untouched.
- **D-6 Workspace resolution.** Pi-family: `cwd` from the `type:"session"` first line. Codex: **implementation checkpoint** — verify the field name for cwd in the rollout's session-meta payload against a real file under `~/.codex/sessions/`; fallback if absent: resolve via the owning agent dir → agents-table workspace, else leave `workspacePath` null (nullable today).
- **D-7 Enrichment excerpts.** `buildConversationExcerpt` becomes harness-dispatched with two small per-harness excerpt builders (pi `type:'message'` lines with `toolCall`/`toolResult` blocks; codex `event_msg`/`response_item` payloads) in the enrichment module. Embeddings unchanged (FR-6).
- **D-8 `jsonlPath` fix.** `mapArchivedConversation` constructs the Claude fallback path **only when the row's harness is `claude-code`**; otherwise `discoveredJsonlPath ?? null`.
- **D-9 Transcript fallback.** `resolveSessionFile` gains a final fallback: look up `discovered_sessions.jsonl_path` by the conversation's locator when harness-specific resolution fails. One new read via the existing data layer; no new door.
- **D-10 No new CLI verbs.** The existing scan trigger (ScanButton / `POST /scan` / `scanConversations` RPC) now covers all harnesses; CLI parity is satisfied by that.
- **D-11 PAN-1917 coordination.** If the unified feed (`sessions-feed.ts`) has landed, WI-7 extends its row shape + facets with `harness` in the same change; if not, WI-7 targets `DiscoveredSessionSnapshot` + `FacetPanel` as they exist today. Either way the contracts field is the same optional `harness` string.
