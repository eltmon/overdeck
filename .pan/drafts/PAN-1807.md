# PRD: PAN-1807 â€” Handoff conversations get a completion contract + unsubmitted-work detection

## Problem

A handoff conversation (`pan handoff`) working in `workspaces/feature-pan-1800` fixed a broken
Playwright spec (commit `53d4ac20f`), pushed the branch, verified green locally, and reported done
in its pane. Then nothing happened: no PR, no `pan done`, no review request, no operator surface.
Main stayed red for ~a day; every open PR inherited the failure; the fix was found only by manual
workspace audit and merged by hand (`2148beaf7`).

Root cause: handoff conversations intentionally bypass the work-agent pipeline, but they have **no
completion contract at all**. The handoff brief delivered to the successor (the handoff doc authored
from `roles/handoff-external.md`) describes the work but never says how to *finish* it. Summary
forks get an explicit delivery-side instruction (`FORK_WAIT_INSTRUCTION`,
`src/lib/conversations/summary-fork.ts:89`); handoff briefs get nothing. And no system component
detects pushed-but-unsubmitted feature branches afterward.

## Glossary

- **Handoff conversation / successor**: the new conversation spawned by `pan handoff` (CLI:
  `src/cli/commands/handoff.ts` â†’ server route `POST .../summary-fork` with `forkMode: 'handoff'`
  in `src/dashboard/server/routes/conversations.ts`). The successor row in the conversations DB has
  `handoffDocPath` set (`src/lib/database/conversations-db.ts:75`).
- **Handoff brief**: the handoff doc text injected as the successor's first message by
  `injectForkSummary()` (`src/dashboard/server/routes/conversations.ts:3824`), built inside
  `runForkPipeline()` (`:3861`).
- **Feature workspace**: a git worktree at `<projectRoot>/workspaces/feature-<issue-lowercase>/`
  on branch `feature/<issue-lowercase>`. Path convention used at e.g.
  `src/cli/commands/inspect.ts:64`, `src/cli/commands/doctor.ts:444`.
- **Submit flow**: `pan done <issue>` (`src/cli/commands/done.ts`) â€” rebases onto the target
  branch, pushes, creates review artifacts (PR), writes the completion marker, sets review status
  to pending, and triggers the review pipeline.
- **Completion contract**: a delivery-side footer appended to the handoff brief telling the
  successor how to finish (run `pan done <issue>`; pushing is not done).
- **Unsubmitted handoff work**: a feature branch with commits pushed to origin ahead of the merge
  target, associated with a recent handoff conversation, with no submission signal (no review
  status, no PR, issue still open, no running work agent).

## Requirements

- **FR-1** â€” When a handoff brief is delivered to a successor whose cwd is inside a feature
  workspace, the brief MUST end with a completion contract that names the exact submit command
  (`pan done <ISSUE-ID>`, issue derived from the workspace path) and states that pushing commits
  alone is not completion. Applies to all handoff-mode forks, including reused docs
  (`readExistingHandoffDoc`) and handoffâ†’summary fallbacks (cwd is still the workspace).
- **FR-2** â€” `pan done <issue>` MUST work from a workspace with no registered agent state: when
  `~/.panopticon/agents/agent-<issue>/state.json` is absent, resolve the workspace as
  `<projectPath>/workspaces/feature-<issue-lowercase>` via `resolveProjectFromIssueSync()` and
  proceed (preflight, rebase, artifacts) if that directory exists. The hard error remains only when
  no workspace can be found anywhere.
- **FR-3** â€” A library function MUST detect unsubmitted handoff work: handoff successor
  conversations created within the last 7 days whose cwd is inside a feature workspace, where the
  workspace branch has â‰Ą1 commit on `origin/<branch>` ahead of the merge target, suppressing
  candidates that are already submitted/merged/in-flight (see Decision rules).
- **FR-4** â€” `pan review pending` MUST print an "Unsubmitted handoff work" section listing each
  detection (issue, workspace, branch, ahead-count, conversation, suggested action
  `pan done <issue>`) after the pending-reviews list, and print nothing extra when there are none.
- **NFR-1** â€” All new exec calls use async exec (`execAsync`/spawn); no `execSync` in any code
  importable by the dashboard server. Detection lives in `src/lib/` so the server or flywheel can
  adopt it later.
- **NFR-2** â€” Decision logic (which candidates are flagged) is a pure function over collected
  inputs, unit-tested without git/DB/network. I/O collectors are thin and best-effort.
- **NFR-3** â€” Any test exercising delays/retries uses vitest fake timers (repo rule).

## Decision rules (made here, not by the work agent)

1. **Issue id for the contract comes from the cwd path, not `conv.issueId`.** Match
   `/\/workspaces\/feature-([^/]+)(\/|$)/` against the successor's cwd; parse the captured segment
   with `parseIssueIdFromTextSync()`. `conv.issueId` is inherited from the parent and can be wrong
   or null (the PAN-1800 handoff had no reliable issueId); the workspace path is the ground truth
   of where commits land. No match â‡’ no contract (handoffs in main worktrees or arbitrary dirs get
   today's behavior).
2. **The contract is delivery-side, not template-side.** Append it in `runForkPipeline()` after the
   brief text is final (after :3976), as part of the same injected string. Do NOT edit
   `roles/handoff-external.md` / `handoff.md` / `handoff-external-pi.md` â€” a delivery-side footer
   covers all three authoring paths plus reused docs, and doesn't depend on the author model
   following instructions.
3. **Append on requested mode `handoff`, including fallback-to-summary.** When handoff authoring
   fails and the pipeline falls back to a summary brief (`effectiveForkMode = 'summary'`), the
   successor still sits in the feature workspace; the contract still applies. Key the append on the
   requested `forkMode === 'handoff'`, not `effectiveForkMode`. The contract footer is about how to
   finish; it does not conflict with `FORK_WAIT_INSTRUCTION`'s "wait before starting".
4. **Detection candidates = handoff successor conversations** (`handoffDocPath !== null`) created
   within the last **7 days**, any status (active or ended), cwd matching the feature-workspace
   pattern. Dedupe to one row per workspace (keep newest conversation).
5. **Merge target** = `git rev-parse --abbrev-ref origin/HEAD` in the workspace, mapped to
   `origin/<name>`; fall back to `origin/main` when origin/HEAD is unset.
6. **"Pushed ahead"** = `git rev-list --count <target>..origin/<branch>` â‰Ą 1, where `<branch>` is
   the workspace's current branch and must match `feature/*`. Remote-tracking refs are trustworthy
   because the push originated from this machine. Missing `origin/<branch>` â‡’ never pushed â‡’ skip
   (committed-but-unpushed detection is a NonGoal).
7. **Suppression rules** (any one suppresses the row):
   a. Review status for the issue exists with `reviewStatus` in {pending, reviewing, passed} or
      `mergeStatus === 'merged'` or a non-empty `prUrl` â€” already in the pipeline.
   b. A completion marker `~/.panopticon/agents/agent-<issue>/completed` exists â€” `pan done` ran.
   c. A running agent exists for the issue (`listRunningAgentsSync()`) â€” normal in-flight work.
   d. Best-effort: `gh pr list --head <branch> --state all --json number --limit 1` (run in the
      workspace cwd, 5s timeout) returns a PR â€” someone already opened/merged one. On gh
      error/absence: do not suppress, do not fail.
   e. Best-effort: the tracker issue is closed (`resolveGitHubIssueSync`-style state check). On
      error: do not suppress. This catches operator hand-merges like `2148beaf7` that squash or
      bypass PRs (ancestry alone misses squash merges).
8. **Contract text** names: the issue id, that pushing is not completion, the single command
   `pan done <ISSUE>` run from the workspace root, what it does (rebase+push+PR/review artifacts+
   review pipeline), the instruction to report a `pan done` failure verbatim in the final message
   instead of stopping silently, and an escape hatch ("if this work is intentionally not for
   submission, say so explicitly in your final message").

## Acceptance criteria (issue-level)

1. Forking with `forkMode: 'handoff'` into a cwd of `<proj>/workspaces/feature-pan-9999` produces
   an injected brief whose final section is the completion contract naming `pan done PAN-9999`.
2. The same fork into a non-workspace cwd produces a brief byte-identical to today's behavior.
3. `pan done PAN-X` from a workspace whose `agent-pan-x` state.json is absent proceeds through
   preflight/rebase/artifacts using `workspaces/feature-pan-x` and exits 0 (given clean tree).
4. `pan review pending` lists a pushed-but-unsubmitted handoff workspace with the action line, and
   stops listing it after `pan done` runs (suppression 7a/7b) or the PR/issue closes (7d/7e).
5. `npm run typecheck`, `npm run lint`, `npm test` pass.
