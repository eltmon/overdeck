# PAN-2155 — Decompose MessagesTimeline.tsx into focused modules

**Type:** Refactor (codebase-health). Behavior-preserving.
**Target:** `src/dashboard/frontend/src/components/chat/MessagesTimeline.tsx` (1620 lines).

## Glossary
- **Barrel** — a thin module that only re-exports symbols from other modules so existing
  `import` paths keep resolving unchanged.
- **Seam** — a clean cut point: a cohesive group of symbols with a one-directional
  dependency on the rest, so it can move to its own file with no import cycle.
- **File-size guard** — `scripts/lint-file-size.sh` (run by `npm run lint:file-size`). New
  non-test src files must stay under **1000** lines; a *baselined* file (listed in
  `scripts/file-size-baseline.txt`) may **shrink** but never grow.

## Problem
`MessagesTimeline.tsx` is 1620 lines — baselined at 1620 in `scripts/file-size-baseline.txt`.
It mixes several cohesive concerns in one file: 3 module constants, time/duration formatters,
search-highlight DOM utilities, the virtualized container component, the per-row dispatcher
(`TimelineRowRenderer`), the individual message-row renderers (user / assistant / context),
the work-log tool renderers, and the divider/indicator components.

A sibling `MessagesTimeline.logic.ts` already holds the data-transform layer (out of scope).

## Proposal
Pure relocation behind a barrel. Extract cohesive groups into sibling modules under a new
**camelCase** directory `src/dashboard/frontend/src/components/chat/messagesTimeline/`, and
leave `chat/MessagesTimeline.tsx` as a **thin re-export barrel**. No logic edits, no renamed
exports, no call-site changes. The directory `messagesTimeline/` (lowercase `m`) does not
collide with the barrel file `MessagesTimeline.tsx` (uppercase `M`) on this case-sensitive
filesystem, and `import './MessagesTimeline'` continues to resolve to the `.tsx` file.

### Verified facts (checked against the file at planning time)
- **Public surface used by importers** (must keep resolving from `chat/MessagesTimeline.tsx`):
  - `MessagesTimeline` (value, defined line 229) — imported by `TerminalPanel.tsx:6`,
    `ConversationPanel.tsx:11`, `__tests__/MessagesTimeline.test.tsx:14`.
  - `RoundMarker` (type, line 189) — imported by `CommandDeck/SessionView/SessionPanel.tsx:7`,
    `ConversationPanel.tsx:11`, `lib/deriveRoundMarkers.ts:19`, the test.
  - `MessagesTimelineProps` (type, line 198) — `export`ed; no external importer found, keep
    re-exported for safety.
- The tests import **only** the public `MessagesTimeline` + `RoundMarker` and render through
  the component (`MessagesTimeline.test.tsx`); `ConversationPanel.test.tsx` does
  `vi.mock('../MessagesTimeline', …)`. **No test imports an internal symbol of this file**, so
  no test needs repointing. (`MessagesTimeline.logic.test.ts` imports from
  `../MessagesTimeline.logic`, untouched.)
- The file contains **zero `any`** (grep verified) — the A1 ratchet /
  `eslint-any-allowlist.json` is a no-op here (no allowlist edits needed).
- `MessagesTimeline.logic.ts` is already a separate sub-1000 sibling; importers reference
  `./MessagesTimeline.logic`. **It is out of scope** — do not move or touch it.

### Dependency directions (no cycles — confirmed by usage-line scan)
- `formatTimestamp` (151) / `formatElapsed` (169) → used by message rows (UserMessageRow 952,
  AssistantMessageRow 1035/1085) and dividers (SlashCommandDivider 1580).
- `stringifyToolInput` (49) → used by `getRowSearchText` (139). `getRowSearchText` (128) +
  `extractSearchHighlightTerms` (62) + `clearSearchHighlights` (81) +
  `highlightSearchTermsInElement` (88) → used by the main component only (481/505/570-575).
- `MAX_VISIBLE_WORK_LOG_ENTRIES` (45) → work-log rows only (1131-1145);
  `ALWAYS_UNVIRTUALIZED_TAIL_ROWS` (46, used 356) / `AUTO_SCROLL_THRESHOLD_PX` (47, used 461)
  → main component only.
- Leaf row/divider components → consumed by `TimelineRowRenderer` (867-901);
  `TimelineRowRenderer` → consumed by the main component (414/451/642/679). `RoundDivider`

So the dependency order is: `types` / `helpers` (leaves) → `messageRows` / `workLogRows` /
`dividers` → `TimelineRowRenderer` → component → barrel. Putting `RoundMarker` in `types.ts`
(not in the component file) is what keeps `dividers.tsx` from importing the component
(which would be a cycle).

## Requirements

### Functional
- **FR-1** — `chat/MessagesTimeline.tsx` becomes a thin re-export barrel that still exports
  `MessagesTimeline` (value), `RoundMarker` (type), and `MessagesTimelineProps` (type).
- **FR-2** — Every internal symbol of the original file is relocated into cohesive modules
  under `chat/messagesTimeline/`, per the layout table, with no import cycle.
- **FR-3** — Zero call-site changes: `TerminalPanel.tsx`, `ConversationPanel.tsx`,
  `CommandDeck/SessionView/SessionPanel.tsx`, `lib/deriveRoundMarkers.ts`, and all `__tests__`
  resolve their imports unchanged.
- **FR-4** — Pure relocation: no logic edits, no renamed/removed exports, identical runtime
  behavior. Moved code is byte-for-byte the same except for added import lines.

### Non-functional
- **NFR-1** — Each new file under `chat/messagesTimeline/` is < 1000 lines; `npm run
  lint:file-size` exits 0.
- **NFR-2** — A1 any-ratchet stays green. No new `any`; no `eslint-any-allowlist.json` edit is
  expected (the source has no `any`). If a move surfaces an unavoidable `any`, add only the new
  file's path to the allowlist.
- **NFR-3** — After **each** seam, the full gate set is green (see below).
- **NFR-4** — No edit to `scripts/file-size-baseline.txt` and no edit to
  `MessagesTimeline.logic.ts`.

## Gates — run after EVERY seam (workspace root unless noted)
The issue's generic refactor checklist names only the root suite, but this is a **frontend**
file, and the root tooling does **not** cover the frontend:
- Root `npm run typecheck` (`tsc --noEmit`) **excludes** `src/dashboard/**` (root
  `tsconfig.json` exclude).
- Root `npx vitest run --configLoader runner` **excludes** `src/dashboard/frontend/**`
  (root `vitest.config.ts` exclude) — so it does **not** run the MessagesTimeline tests.

Therefore the per-seam gate set MUST be:
1. `npm run lint` — covers ESLint on the new `.tsx` files + `lint:file-size` + the any-ratchet.
2. `npx tsc --noEmit` **run from `src/dashboard/frontend/`** — the frontend typecheck
   (mirrors the pipeline's `frontend-typecheck` quality gate). This is the gate that proves the
   moved frontend code still type-checks.
3. `npm --prefix src/dashboard/frontend run test` — the frontend vitest suite (runs
   `MessagesTimeline.test.tsx`, `ConversationPanel.test.tsx`, `MessagesTimeline.logic.test.ts`,
   …). **This is the behavior-preserving proof** for this refactor; 0 failed.
4. `npm run typecheck && npx vitest run --configLoader runner` — the root node gates the issue
   names; they won't touch the frontend but must stay green (no node-side breakage).

All four must pass before moving to the next seam and before `pan done`. Commands are async
(no `execSync`); never `git checkout` / `git stash` in the workspace; conventional-commit
subjects.

## NonGoals
- No logic edits, no renamed/removed exports, no behavior change (pure moves + re-exports).
- Do NOT move or modify `MessagesTimeline.logic.ts`.
- No changes to any importing call site (TerminalPanel, ConversationPanel, SessionPanel,
  deriveRoundMarkers, the tests).
- No new tests, no test-behavior changes.
- No `eslint-any-allowlist.json` edit (the file has no `any`).
- No `scripts/file-size-baseline.txt` edit (the shrunk barrel already passes against its 1620
  baseline; touching the baseline is out of scope, matching the PAN-2150 convention).
- No re-architecture of virtualization, search, rounds, or any feature — relocation only.

## Decisions
- **D1** — Sibling **directory** `messagesTimeline/` (camelCase), matching the PAN-2150
  SettingsPage precedent (`Settings/sections/`), rather than dotted sibling files. The
  case difference from the `MessagesTimeline.tsx` barrel avoids any resolution ambiguity.
- **D2** — `RoundMarker` + `MessagesTimelineProps` live in `messagesTimeline/types.ts` (not the
  component file) so `dividers.tsx` can import `RoundMarker` without a cycle back to the
  component.
- **D3** — Three work items (seams), each independently green and gated; a linear `blocks`
  chain because all three mutate `chat/MessagesTimeline.tsx` (shared mutation).
- **D4** — `requiresInspection: false` on all items (pure relocation, fully covered by the
  4-part gate set + review), matching the proven PAN-2150 pattern.
