## 0. Glossary (define before use)

- **Two-door tenet** — every canonical-state domain has exactly one read path
  (resolver/door) and one write path; consumers never touch a store directly
  (`docs/API-SURFACE.md`, PAN-1936).
- **Agents read door** — `getAgentStateSync(agentId)` / `getAgentState` in
  `src/lib/agents/agent-state.ts` (grep anchor:
  `export function getAgentStateSync`) and the enumeration queries
  `listRunningAgentsSync`/`listRunningAgents` in `src/lib/agents/queries.ts`. They
  read the overdeck DB first with `state.json` as fallback/rebuild source
  (PAN-1908 planes model: SQLite `agents` table = runtime registry; `state.json` =
  rollback/rebuild source only).
- **Raw handle** — `getOverdeckDatabaseSync()` from `src/lib/overdeck/infra.ts`: a
  direct SQLite handle to overdeck.db. Inside `src/lib/overdeck/**` (the doors) it is
  the implementation; outside, it is a door bypass.
- **Guard-script convention** — `#!/usr/bin/env bash`, `set -euo pipefail`, repo-root
  `cd`, a `comment_filter` perl helper dropping comment lines from `git grep` output
  (both existing guards define it — copy the idiom verbatim), wired into the
  `npm run lint` chain (grep anchor in `package.json`:
  `&& npm run lint:overdeck-boundaries`), and locked by a fixture-repo unit test under
  `tests/unit/scripts/` (pattern: `tests/unit/scripts/lint-state-writes.test.ts` —
  `git init` a temp repo, install the script, seed violation/clean fixtures, assert
  red/green).
- **Allowlist entry** — a named file the guard exempts for a stated reason, written in
  the script next to the rule. Removals are free; additions are code-reviewed by
  construction (they edit the guard) and must carry a justification comment.
- **PAN-1832** — the motivating incident: a consumer hardcoded agent display state
  instead of calling the read door, and nothing failed at build time.

## 1. Problem (verified evidence at `506756378f`)

The write side is guarded; the read side is not:

1. **Direct `state.json` reads bypass the agents door in ~6 production sites**
   (sweep: `grep -rn "'state.json'" src/ --include='*.ts'` minus tests/door files):
   `src/lib/cloister/inspect-session-reaper.ts` (existence probe for `inspect-*`
   sessions), `src/lib/costs/migration.ts` (historical backfill scan),
   `src/lib/costs/reconciler.ts` (×2 — reads `issueId`/`role` for cost attribution),
   `src/dashboard/server/services/conversation-lifecycle.ts` (parses state for
   orphaned-specialist backfill, building the path with `join(homedir(), '.overdeck',
   'agents', sessionName, 'state.json')`), `src/dashboard/server/routes/jsonl-resolver.ts`
   (reads state for transcript resolution). Each is a PAN-1832-class bug waiting:
   they see the filesystem plane only, so a DB-registered agent whose `state.json`
   lags (or vice versa) gives them a different answer than the door gives everyone
   else.
2. **Raw overdeck handle outside the doors in 3 files** (sweep:
   `grep -rln "getOverdeckDatabaseSync" src/` minus `src/lib/overdeck/`):
   `src/lib/conversations/correlator.ts`, `src/dashboard/server/event-store.ts`,
   `src/dashboard/server/services/agent-projection.ts`. The boundaries gate's check #3
   catches raw `Db` *service imports* from `infra.js` but NOT the sync-handle
   function — a hole big enough that three callers live in it.
3. **Nothing enforces any of this in CI.** A new route handler doing
   `JSON.parse(readFileSync(.../state.json))` or `getOverdeckDatabaseSync().prepare(...)`
   lands green today — the exact failure mode of PAN-1832.
4. **Bit-rot in the neighboring guard:** `lint-overdeck-boundaries.sh` check #4's
   comment says it guards "the panopticon.db file path" with a sanctioned exemption
   for "the read-only cutover seed in dashboard main.ts" — but its pattern greps
   `overdeck\.db` (the NEW db) and the main.ts seed block was deleted weeks ago
   (grep anchor: `':!src/dashboard/server/main.ts'`). The check functions usefully
   (it catches direct overdeck.db path references) but its documentation lies and its
   exemption is stale.

Deliberately NOT in scope (the re-scope decision, §3 D6): `.pan/` canonical-state
READS and GitHub issue-state reads — their door consolidation is Phase-2/PAN-1936
territory and their read surfaces are still moving with the route-thinning PRDs.

## 3. Locked design decisions

- **D1 — New script, read-door scope: `scripts/lint-store-access.sh`.** A third
  member of the guard family, NOT an extension of `lint-state-writes.sh` (writes) or
  `lint-overdeck-boundaries.sh` (old-DB cutover) — one concern per script, matching
  how the family is already split. Name follows the issue's suggestion adjusted to the
  family's `lint-*` prefix (the `check-store-access.sh` name in the issue body is
  advisory).
- **D2 — Two rules at launch, structured for per-domain growth.**
  - **Rule 1 (agents / state.json reads):** any `'state.json'` string literal in
    `src/` (excluding `__tests__`, `*.md`) outside the allowed surface fails. Allowed
    surface: `src/lib/agents/agent-state.ts`, `src/lib/agents.ts`,
    `src/lib/overdeck/agents.ts`, plus §2's justified allowlist. Use `comment_filter`;
    literal-based (like the continue-file rule) so path-construction evasion
    (`join(dir, 'state' + '.json')`) is out of scope — accepted limitation, documented
    in the script header.
  - **Rule 2 (raw overdeck handle):** `getOverdeckDatabaseSync[[:space:]]*\(` in
    `src/` outside `src/lib/overdeck/**` and the §2 allowlist fails, with the fix-line
    "read through a door in src/lib/overdeck/ instead".
  - The script's layout makes a third rule (a future domain) a copy-paste section:
    header comment documents the recipe (pattern + allowed doors + allowlist).
- **D3 — Repoints land BEFORE the guard turns on** (§2's four REPOINT rows), one
  file per commit, behavior-preserving; the guard commit then lands with a minimal
  allowlist. Never land the guard with violations baselined-in that a one-line door
  call would fix — baselines are for expensive migrations (the 3 raw-handle services),
  not laziness.
- **D4 — Allowlist additions are audited.** Each entry carries an inline
  justification comment; the script prints the current entry count on success (a
  visible ratchet: reviewers see `✓ store-access lint passed (allowlist: 4)` change).
- **D5 — Fix the boundaries check #4 bit-rot** in the same PR (small, adjacent,
  same-family): rewrite its comment to describe what it actually guards (direct
  `overdeck.db` path references outside doors), drop the stale
  `':!src/dashboard/server/main.ts'` exemption (the seed block is gone — verify by
  grep before dropping), and keep its behavior otherwise identical.
- **D6 — Domains beyond agents are deferred, explicitly.** `.pan/` canonical READS
  and GitHub issue-state reads wait for the Phase-2 route-thinning wave to settle
  (their call sites are being relocated wholesale by PAN-2145/2147/2148). The script
  header names them as the next two sections so the deferral is visible in the code,
  not just in this PRD. This matches the REFACTOR-QUEUE re-scope note.

## 6. Intersecting repo rules (restated — do not assume recall)

- **Two-door tenet / PAN-1908 planes:** enumerate agents from the registry via the
  door; `state.json` is rebuild-source only — repoints must not invert that order.
- **Investigate before fixing:** if a repointed site's behavior differs under the
  door (different answer than the raw file), STOP — that is a plane-drift finding to
  surface, not to paper over (it is the exact bug class this guard exists for).
- **No `execSync` in server code; async-only primitives** — repoints keep the
  surrounding sync/async character of each file.
- **Fake timers** for any retry/delay test. **Never `--no-verify`; never `git stash`;**
  feature workspace, never the primary worktree.
- **commitlint:** all-lowercase subjects, scoped per WI, every commit referencing
  `pan-2008`.
- **Guard self-consistency:** a guard PR must leave ALL guards green at every commit
  (D3's ordering exists for this).
