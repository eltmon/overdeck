# PAN-2150 — Decompose `SettingsPage.tsx` (2043 lines) into focused submodules

**Issue:** [PAN-2150](https://github.com/eltmon/overdeck/issues/2150)
**Type:** Refactor — codebase-health (A3 file-size). Behavior-preserving.
**Campaign:** `docs/codebase-health/REMAINING-BACKLOG.md`

## Glossary

- **Barrel / thin entry** — a module whose job is to re-export symbols from
  other modules so existing import paths keep resolving. Here, after the
  refactor `SettingsPage.tsx` still exports `SettingsPage` and re-exports
  `buildMiniMaxFormData`, so no importer changes.
- **Section component** — a React component that renders one `<section>` of the
  settings page, receiving the state/handlers it needs as props. The repo
  already has five of these under
  `src/dashboard/frontend/src/components/Settings/sections/`
  (`VoiceSettingsSection`, `ConversationSettingsSection`,
  `ProviderManagementSection`, `TtsConfigurationSection`,
  `ConversationSearchSection`). This refactor extends that established pattern
  to the remaining inline `<section>` blocks.
- **Seam** — one extraction step that ends with the full build green. Each
  work item below is one seam = one commit.
- **A1 ratchet** — the no-new-`any` lint guard. Files allowed to contain `any`
  are listed in `eslint-any-allowlist.json`.
- **A3 file-size guard** — `scripts/lint-file-size.sh`: no non-test `src` file
  may exceed 1000 lines unless baselined in `scripts/file-size-baseline.txt`
  (where it may shrink but never grow). `SettingsPage.tsx` is baselined at 4200.

## Goal

Split `src/dashboard/frontend/src/components/Settings/SettingsPage.tsx`
(2043 lines) into cohesive submodules so each new file is under 1000 lines and
`SettingsPage.tsx` becomes a thin entry (target ≤ ~900 lines). Keep
`SettingsPage.tsx` as the import entry so every existing import resolves
unchanged — **zero call-site changes**. Pure relocation: no logic edits, no
renamed exports.

## Chosen approach (decision — do not re-litigate)

**Mirror the existing `sections/*Section.tsx` prop-driven pattern.** For each
inline `<section id="x">…</section>` block:

1. Create `src/dashboard/frontend/src/components/Settings/sections/<Name>Section.tsx`
   (e.g. `PermissionsSection.tsx`, `MemorySection.tsx`, `DiffSection.tsx`).
2. Move the `<section>…</section>` JSX **verbatim** into a new component
   `export function <Name>Section(props: <Name>SectionProps) { return (<section …>…</section>); }`.
3. Every identifier the JSX closes over that is **not** an import or
   module-global becomes a typed prop on `<Name>SectionProps`. Reuse the
   existing prop types from `./types` / `./modelCatalog` where they already
   exist (e.g. `SettingsConfig`, `BackgroundAiConfig`).
4. **Handler ownership rule (deterministic):** a handler referenced by
   **exactly one** section moves *into* that section component (relocated
   verbatim, now closing over props such as `formData` + `applySettings`
   instead of parent scope). A handler referenced by **two or more** sections
   stays in `SettingsPage` and is passed down as a prop. This keeps
   `SettingsPage` shrinking on both JSX and handler axes. typecheck enforces
   that every closed-over identifier is wired.
5. In `SettingsPage.tsx`, replace the inline block with
   `<XSection …props… />`, **in the same position/order**, and add the import.

Move module-level shared code into co-located modules:
- New `SettingsPage.types.ts` — the four local interfaces
  (`OpenRouterModelCatalog`, `OpenRouterCatalogResponse`,
  `SaveSettingsResponse`, `CloisterConfig`).
- New `settingsPageConstants.ts` — `TrackerType`, `TRACKERS`,
  `BG_FEATURE_COST_SOURCE`, `SETTINGS_NAV_ITEMS`. **Contains `any`** (`TRACKERS`
  `icon: any`) → add this file to `eslint-any-allowlist.json` in the same commit.
- New `voiceSettingsDefaults.ts` — `DEFAULT_VOICE_SETTINGS`,
  `DEFAULT_VOICE_HARDWARE_SETTINGS`, `VOICE_HARDWARE_STORAGE_KEY`,
  `loadVoiceHardwareSettings`, `normalizeVoiceSettings`.
- Move `buildMiniMaxFormData` into a co-located helper (e.g.
  `miniMaxFormData.ts`) **and re-export it from `SettingsPage.tsx`**
  (`export { buildMiniMaxFormData } from './miniMaxFormData';`) so the existing
  test import `from '../SettingsPage'` keeps resolving with no test edit (FR-1).

**Why not pull state into hooks?** Out of scope. Pure relocation only; moving
state ownership risks behavior change. Keep all `useState`/`useEffect` in
`SettingsPage` and pass setters down. The handler-ownership rule above moves
only handler *bodies*, not state.

**Why leave `model-routing` inline?** It is already a thin wrapper around two
extracted panels (~24 lines); extracting it adds a file for no size benefit.

## Requirements

**Functional**
- **FR-1** `SettingsPage.tsx` stays the import entry: `export function
  SettingsPage()` unchanged, and `buildMiniMaxFormData` still resolves from
  `'./SettingsPage'` (re-export). `App/AppRoutes.tsx` and `Settings/index.ts`
  importers unchanged; the `__tests__/SettingsPage.test.ts` import unchanged.
- **FR-2** Every inline `<section>` except `model-routing` is extracted to its
  own `sections/<Name>Section.tsx`, rendered in the same order and position in
  `SettingsPage`'s return.
- **FR-3** Behavior-preserving: pure relocation. No renamed/removed exports, no
  logic edits, no JSX/markup changes beyond mechanical prop substitution.
- **FR-4** Each new file < 1000 lines; `SettingsPage.tsx` materially smaller
  than 2043 (target ≤ ~900).

**Non-functional**
- **NFR-1** After **each** seam: `npm run typecheck` + `npm run lint` + the
  **full** suite (`npx vitest run --configLoader runner`, 0 failed). The
  verification gate runs these before review; a seam is not done until green.
- **NFR-2** A1 ratchet: introduce no new explicit `any`. For `any` that moves
  with extracted code, add the **new** file to `eslint-any-allowlist.json` in
  the same commit (`settingsPageConstants.ts`, `MaintenanceSection.tsx`,
  `ExperimentalSection.tsx`).
- **NFR-3** Async only — no `execSync` (n/a to this frontend code, but no new
  sync subprocess primitives).
- **NFR-4** Any test importing a moved internal is repointed in the same commit.
  With the `buildMiniMaxFormData` re-export (FR-1), no test edit is expected; if
  the executor chooses not to re-export, it must repoint the test import instead.

## Intersecting repo rules (restated for the executor)

- **Behavior-preserving refactor:** match existing style; touch only what the
  move requires; do not "improve" adjacent code (Karpathy surgical-changes).
- **No bandaids:** if a section won't extract cleanly, find the real reason
  (usually a missed closed-over identifier surfaced by typecheck) — do not add
  defensive shims.
- **Worktree discipline:** stay on `feature/pan-2150`; never `git checkout`
  another branch; commit per seam.
- **Full suite, not `--changed`:** large rename/move diffs defeat
  `vitest --changed`; run the full suite per NFR-1.
