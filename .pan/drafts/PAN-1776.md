**Status:** draft
**Issue:** PAN-1776
**Related:** PAN-1769 (delivery duplication + false-ended), PAN-1697/PAN-1700 (eaten input), PAN-1635 (eaten by compaction)

This PRD is written to be implementable without further design work. Every
file path, JSON shape, endpoint, and function signature below is normative.
Where a line number is given it is approximate (verify with grep â€” the file

## 1. Problem

The PTY supervisor is the process that owns a Claude Code session's terminal:

- `src/lib/launcher-generator.ts` generates `~/.panopticon/agents/<id>/launcher.sh`
  per spawn; when `useSupervisor` is true its final line is
  `exec node '<packageRoot>/dist/pty-supervisor.js' claude â€¦args`.
- `resolvePtySupervisorScriptPath()` (`src/lib/agents.ts`, ~line 1751) points at
  the shared build artifact `dist/pty-supervisor.js`.
- Node reads that file **once at process start**. Rebuilding `dist/` changes
  nothing for running sessions; only the next spawn/resume picks it up.
- The supervisor owns the PTY **master fd** (via
  `@homebridge/node-pty-prebuilt-multiarch`). If the supervisor exits, the
  kernel closes the PTY and the wrapped harness exits with it ("H1
  lifecycle"). So a supervisor process cannot simply be restarted in place.

Consequences observed in production (2026-06-07 â†’ 2026-06-11):

- PAN-1697/PAN-1700: fire-and-forget delivery ate messages; the fix shipped
  but only new sessions got it.
- PAN-1769: the echo-confirmation fix itself introduced duplicate submits;
  again, every running session kept duplicating until respawned, and nothing
  could enumerate which sessions were affected.
- PAN-1642's work agent lost â‰¥6 deliveries across four days of resume churn.

Two structural gaps:

1. **No observability**: nothing records which supervisor build a session runs.
2. **No update path**: the only way to change delivery behavior is respawning
   every session by hand.

## 2. Goals

- G1: Every running supervisor is identifiable (script hash + capabilities).
- G2: The system can enumerate stale supervisors (`pan doctor`, status APIs).
- G3: An operator (or the deacon, config-gated) can roll stale sessions
  through the existing stopâ†’resume machinery safely (idle-gated, verified).
- G4: Delivery *policy* (echo confirm, purge, retry, Enter) runs in the
  dashboard server, hot-updated by `pan reload`; the supervisor retains only
  dumb primitives (write bytes, expose output, resize).
- G5: Old and new supervisors coexist: capability negotiation, no flag day.

## Non-goals

- No fd-handoff / `exec`-in-place upgrades (node-pty has no attach-to-fd API).
- No removal of the legacy delivery endpoint in this issue.
- No changes for Pi (uses `rpc.in` FIFO, no supervisor), Codex (headless
  one-shot, see the `harness === 'codex'` branch in `deliverAgentMessage`),
  or Docker workspaces (excluded from supervisor wiring).

## 4. Part A â€” version stamp, staleness, rolling refresh

### A1. Supervisor identity sidecar

**New file written by the supervisor at startup** (immediately after the
socket binds), atomic (`write tmp` + `rename`):

`~/.panopticon/agents/<agentId>/supervisor.json`

```json
{
  "schemaVersion": 1,
  "pid": 94634,
  "scriptPath": "/home/eltmon/Projects/overdeck/dist/pty-supervisor.js",
  "scriptSha256": "ab12â€¦64 hex chars",
  "capabilities": ["legacy-post", "primitives-v1"],
  "startedAt": "2026-06-11T07:27:06.000Z"
}
```

- `scriptSha256` = sha256 of the file at `process.argv[1]` read at startup.
  (Tolerable race: a rebuild in the ms between `node` loading the script and
  the hash read; document, don't engineer around.)
- `capabilities` lists what the socket supports. This PRD's new supervisor
  writes `["legacy-post", "primitives-v1"]`. Old supervisors have no file at
  all â€” absence â‡’ legacy-only AND unknown version (treat as stale).
- Implement in `pty-supervisor.ts`: `async function writeSupervisorIdentity(agentId: string): Promise<void>`,
  pure helper `export async function computeScriptSha256(path: string): Promise<string>`
  (exported for tests). Never throw (best-effort, like the log appenders).
- Do **not** delete the file on shutdown â€” staleness checks must validate
  liveness independently (pid alive + socket exists), and the stale file is
  useful post-mortem.

### A2. Staleness detection (server side)

**New module** `src/lib/supervisor-status.ts`:

```ts
export interface SupervisorStatus {
  running: boolean;          // socket exists AND identity pid is alive
  identityPresent: boolean;  // supervisor.json exists and parses
  stale: boolean;            // running && (no identity || scriptSha256 !== current dist sha)
  capabilities: string[];    // [] when unknown
  runningSha?: string;
  distSha?: string;
}
export async function getSupervisorStatus(agentId: string): Promise<SupervisorStatus>;
export async function getCurrentDistSha(): Promise<string>;  // sha256 of resolvePtySupervisorScriptPath(), cached by (path, mtimeMs)
```

- pid liveness: `process.kill(pid, 0)` in try/catch.
- Surface:
  - Add `supervisorStale?: boolean` and `supervisorSha?: string` to the agent
    status payload (`src/dashboard/server/routes/agents.ts` status route) and
    the conversation payload (`src/dashboard/server/routes/conversations.ts`,
    same place `sessionAlive` is computed). Only for claude-code sessions
    with a supervisor socket; omit otherwise.
  - `pan doctor`: one row â€” "PTY supervisors: N running, M stale" (warn when
    M > 0). Doctor lives in `src/commands/doctor.ts` (verify path with grep).

### A3. `pan supervisor` CLI verb

New command file following existing command conventions (find the commander
registration in `src/cli.ts` / `src/commands/`):

```
pan supervisor status
    # table: id | kind (agent/conv) | sha (8 chars) | stale | idle | gates (paused/troubled)
pan supervisor refresh <id> [--force]
pan supervisor refresh --all-stale [--max <n>]   # default max 3
```

Refresh semantics per session, in order:

1. **Skip** (with printed reason) if: not claude-code; Docker workspace;
   paused or troubled gate set (`state.json`); supervisor not stale (unless
   `--force`).
2. **Idle gate**: agents â€” require idle via the runtime-mirror/idle signal
   used by `waitForAgentIdle`; conversations â€” require no in-flight turn
   (`summarizeConversationActivity.isWorking === false`, see
   `src/dashboard/server/services/conversation-service.ts`). `--force`
   overrides. Never interrupt a mid-turn session.
3. **Roll**: agents â€” existing stop (`stopAgent`/`pan kill` path) then
   `resumeAgent` (regenerates launcher â†’ new supervisor; auto-continue is
   PAN-1700 landing-verified). Conversations â€” the existing conversation
   resume flow (same code path as the dashboard Resume button; call the
   internal function, do not shell out to HTTP).
4. **Verify**: after resume, poll `getSupervisorStatus` until `stale ===
   false` (timeout 60s) and report per-session success/failure.

`--all-stale` processes sequentially (never parallel â€” resume storms are how
PAN-821-style thrash starts), `--max` caps per invocation.

**Skills convention:** adding a `pan` verb requires either a wrapper skill or
an exclusion-list entry â€” read `docs/SKILLS-CONVENTION.md` and keep
`scripts/lint-skills.sh` (runs in `npm run lint`) green. An exclusion entry
is appropriate here (`--help` is self-explanatory).

## 5. Part B â€” dumb-shim primitives + server-side delivery

### B1. Socket protocol v1

Extend `createPtySupervisorServer()` with URL-path routing (it currently
rejects all non-POST; change to route by `req.method + url.pathname`). All
endpoints require the same pty-token header. Unknown paths â†’ 404.

| Endpoint | Body / query | Response | Semantics |
| --- | --- | --- | --- |
| `POST /` | `{content, echo?, caller?, meta?}` | `200 "ok"` / `502` | **Unchanged** legacy full delivery (keep `injectPtyMessage` working) |
| `GET /v1/info` | â€” | `{schemaVersion, pid, scriptSha256, capabilities, outputEndOffset}` | identity + current absolute output offset |
| `POST /v1/write` | `{data: string, echoToStdout?: boolean}` | `{ok: true, outputEndOffset}` | raw `child.write(data)`; when `echoToStdout`, mirror to supervisor stdout exactly like `payload.echo` does today |
| `GET /v1/output?since=<n>` | â€” | `{startOffset, endOffset, data}` | ring-buffer slice; if `since` < buffer start, return from buffer start (caller detects the gap via `startOffset > since`) |
| `POST /v1/resize` | `{cols, rows}` | `{ok: true}` | `child.resize(cols, rows)` (bounds-check 1..1000) |

### B2. Ring buffer

In supervisor `main()`: one `child.onData` subscription appending to a ring
buffer, capacity 64 KiB, tracking `absoluteOffset` (total bytes ever seen).
Implement as a small exported class in `pty-supervisor.ts`
(`export class OutputRing { append(s: string): void; slice(since: number): {startOffset, endOffset, data} }`)
with direct unit tests (wrap-around, gap, empty).

### B3. Shared echo/purge helpers

Move the pure delivery helpers out of `pty-supervisor.ts` into a new
`src/lib/channels/pty-echo.ts` re-exported by both sides:

```ts
export const INPUT_ECHO_CONFIRM_TIMEOUT_MS / INTERVAL_MS / ATTEMPTS / PREFIX_CHARS;
export const INPUT_SETTLE_MS, INPUT_PURGE_MAX_CHARS, INPUT_PURGE_SETTLE_MS;
export const PASTE_PLACEHOLDER_RE;
export function stripAnsi(value: string): string;
export function normalizePtyText(value: string): string;
export function normalizeForEchoMatch(value: string): string;
export function confirmationPrefix(content: string): string;
export function purgeBackspaces(charCount: number): string;  // '\x7f'.repeat(min(n, MAX) + 8)
```

`pty-supervisor.ts` imports these (its tests keep passing unchanged);
the new client (B4) imports the same â€” one source of truth.

### B4. Server-side delivery engine

New `src/lib/channels/pty-delivery-client.ts`:

```ts
export interface PrimitiveDeliveryArgs {
  socketPath: string;
  token: string;
  agentId: string;
  content: string;
  echoToStdout: boolean;       // mirrors payload.echo !== false
  caller?: string;
}
export async function deliverViaPrimitives(args: PrimitiveDeliveryArgs): Promise<void>; // throws on failure
```

Algorithm â€” the exact `injectPtyMessage` state machine, but over HTTP
primitives:

1. `GET /v1/info` â†’ `baseOffset = outputEndOffset`.
2. `prefix = confirmationPrefix(trimmedContent)`; if empty, skip to (5).
3. For attempt 1..`INPUT_ECHO_CONFIRM_ATTEMPTS`:
   a. attempt > 1 â†’ `POST /v1/write {data: purgeBackspaces(len)}` + sleep
      `INPUT_PURGE_SETTLE_MS`.
   b. `POST /v1/write {data: trimmedContent}`.
   c. Poll `GET /v1/output?since=baseOffset` every
      `INPUT_ECHO_CONFIRM_INTERVAL_MS` until `INPUT_ECHO_CONFIRM_TIMEOUT_MS`:
      confirmed when `normalizeForEchoMatch(accumulated)` contains `prefix`
      OR `PASTE_PLACEHOLDER_RE` matches `stripAnsi(accumulated)`.
4. Not confirmed â†’ purge once more, log, throw (caller's ladder falls back to
   tmux exactly as today â€” the purge keeps that single-copy-safe).
5. Sleep `INPUT_SETTLE_MS`, `POST /v1/write {data: '\r', echoToStdout}` â€”
   note: echo the *content* to stdout (transcript), not the `\r`; replicate
   today's behavior by sending `echoToStdout` on the content write in (3b).
6. Append the same `socket_write` line to the bridge/supervisor logs the
   server already writes (`appendChannelDeliveryLog` with
   `path: 'supervisor-primitives'`).

Wire into `deliverAgentMessage()` supervisor tier (`src/lib/agents.ts`):

```ts
// pseudo
const info = await tryGetInfo(socketPath, token, 1_000);   // null on any error
if (info?.capabilities?.includes('primitives-v1')) {
  await deliverViaPrimitives({...});                        // server-side policy
  return { ok: true, path: 'supervisor-primitives' };
}
await postUnixSocketJson(socketPath, {content, meta:{caller}}, 8_000, token, PTY_TOKEN_HEADER);  // legacy
```

Client timeouts: each primitive call â‰¤ 2s; total engine budget must stay
under the ladder's expectations â€” keep the legacy 8s comment's invariant in
mind (client budget > worst-case write/confirm path).

## 7. Risks & edge cases

- **Two supervisor generations coexist** for days. Capability negotiation
  (B4) and keeping legacy POST `/` (B5) make this safe by construction.
- **Echo matching against transcript re-renders** can false-positive when the
  same text is visible in the prior screen (e.g. a retry of an identical
  message). Pre-existing behavior; primitives don't change it. The
  `since=baseOffset` window (B4 step 1) actually narrows it vs today.
- **Ring-buffer gap** (>64KB output mid-delivery, e.g. agent streaming):
  caller sees `startOffset > since` and simply keeps matching on the
  available window â€” same information the in-process 64KB buffer has today.
- **Refresh during pipeline activity**: idle gates + sequential processing +
  paused/troubled/freeze gates. Never auto-refresh conversations with
  unsubmitted composer state (cannot detect â€” conversations refresh only via
  explicit CLI, never via the A4 patrol; patrol scope is `agent-*` only).
- **Docker workspaces**: no host socket sharing; `getSupervisorStatus`
  reports `running: false` â€” refresh skips them (explicit skip reason).
- **Security**: all new endpoints behind the existing per-agent token; socket
  stays mode 0600; no new listening surface.
- **No execSync** anywhere in server-reachable code (existing rule); all new
  IO async.
