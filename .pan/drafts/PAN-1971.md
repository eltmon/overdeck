# PRD: Omnibox Claude/Codex launches open in terminal view (PAN-1971)

> Part of the **minimum new-user onboarding** (demo readiness). Frontend-only, ~1 file.

## Summary
Make the Home omnibox's **"Ask Claude Code"** and **"Ask Codex"** actions open the new session in **terminal view** (raw PTY/xterm) by default, so the agent's initial onboarding/auth TUI dialog is visible and interactive. Conversation view remains the default everywhere else.

## Glossary
- **Omnibox / Launcher** — the "Ask, run, or search…" input on a project Home (`Launcher.tsx`).
- **Conversation view** — parsed message UI (`ConversationView`); hides the raw TUI.
- **Terminal view** — raw xterm.js PTY (`XTerminal`) over `/ws/terminal`; shows the live TUI.
- **viewMode** — per-pane field on `WorkspacePane`: `'conversation' | 'terminal' | 'findings'`.

## Problem
Launching Claude Code/Codex from the omnibox opens an agent pane with `viewMode` undefined → it defaults to conversation view. Claude Code (and likely Codex) show an initial onboarding/login dialog that only renders in the raw TUI; in conversation view a new user sees nothing actionable and gets stuck.

## Goal / Non-goals
- **Goal:** omnibox agent launches (claude-code, codex) default to terminal view; user can toggle back.
- **Non-goal:** changing the default for non-omnibox conversation opens; detecting "is onboarded" (always open agent omnibox launches in terminal — simplest, correct for new users, harmless for existing).

## Requirements
- **FR-1** — Omnibox "Ask Claude Code" and "Ask Codex" launches open the resulting agent pane in **terminal** view.

- **FR-3** — Non-omnibox conversation opens are unchanged (default conversation view).
- **NFR-1** — Frontend-only; no backend change.
