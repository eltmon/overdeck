## 1. Vision

An AI evaluates the **entire open backlog** and produces one **planned sequence** of every
issue — ranked by real impact, sized by effort, wired into a dependency DAG. A new dashboard
page renders that sequence so you can see the whole backlog, in order, at a glance, with
visual signals for what is already in the pipeline, what has a PRD, what is ready to pick up,
and which issues the AI thinks need refinement or are no longer relevant.

The plan-of-record is a **repo-tracked markdown file** (`.pan/backlog/sequence.md`), committed
to `main`, so anyone who checks out the repo reproduces the identical visualization. SQLite is
a derived cache only.

This is the planning complement to the **Flywheel** (which keeps agents *working* on the next
item, by fixed rules). The Sequencer answers *"what is the next item, across the entire backlog,
and why?"* and makes that answer durable, inspectable, and reproducible.

## 2. Glossary

- **Backlog** — every open issue for a project across configured trackers (today: GitHub for PAN),
  excluding closed/cancelled. Pulled via `src/lib/tracker/`.
- **Sequence** — a total ordering of the backlog by AI-assessed importance, expressed as a DAG.
- **Importance / impact** — the AI's judgement of how much an issue matters, weighing **urgency**
  (time-sensitive / actively bleeding), **relevancy** (still applicable to where the project is now),
  **unblocking power** (does it gate other work), and **reach**. The GitHub priority field **and issue
  age** are inputs, never the answer — this deliberately replaces the Flywheel's age-based oldest-first
  tiebreak, which assumes neither urgency nor relevancy.
- **Size** — AI-estimated effort ∈ {XS, S, M, L, XL}. Encoded as node size.
- **In pipeline** — the issue already has a live workspace/branch/agent/PR or a non-pending review status.
- **PRD** — a `.pan/drafts/<ISSUE>.md` draft exists.
- **Ready** — a vBRIEF spec exists in `.pan/specs/` **and** beads exist, i.e. an agent could pick it up now.
- **`sequence.md`** — the repo-tracked source of truth at `.pan/backlog/sequence.md`: human-readable
  table + per-issue rationale + a fenced JSON block that is the parse target.
- **Sequencer agent** — the configurable-model AI role that (re)writes `sequence.md`.
- **Creation pass** — first-ever run: ranks the entire backlog from scratch.
- **Incremental pass** — default, cheap run: examines only new/updated/closed issues since the last run;
  slots them in, minimally re-ranks, preserves prior order + rationale.
- **Review pass** — periodic deep run: re-evaluates the entire backlog from first principles.
- **Condition** — the AI's assessment of the issue itself, independent of rank: `ok`,
  `needs-refinement` (too poorly specified to act on — a human must fix it), or `stale`
  (likely no longer relevant — a candidate to close).
- **Tier** — a derived presentation band (Now / Next / Later / Someday) computed from rank/score
  thresholds; not stored per-issue, reproducible from the ranking alone.
- **Pinned** — an in-pipeline issue the sequencer never re-ranks or rewrites (only its live state refreshes).
- **Pickup gate** — an operator-set per-issue override ∈ {auto, ready, blocked} stored in `sequence.md`:
  `ready` greenlights/promotes for auto-pickup, `blocked` holds it out, `auto` defers to heuristics.
  Operator-owned; the sequencer preserves it verbatim. `blocked` also sets a parked label.
- **Dependency edge** — a directed cross-issue relation in the sequence DAG (`unblocks`/`informs`) with a
  provenance `source` ∈ {github-ref, operator, ai-inferred}. Distinct from vBRIEF/beads *intra-issue* edges.

## 3. Motivation (verified references)

- **The Flywheel ranks reactively, by rules — no LLM evaluation.** `docs/flywheel-brief.md` ranks by
  fixed tiers (P0→P3) then oldest-first; it emits per-tick `FlywheelSuggestion[]` but does no holistic,
  content-aware evaluation of the whole backlog (`src/lib/cloister/flywheel.ts`,
  `src/lib/flywheel-merge-order.ts`). No surface shows the *entire* backlog sequenced and explained.
- **GitHub priority is thin.** The tracker layer returns `priority: undefined` for GitHub
  (`src/lib/tracker/github.ts:361`); Linear sets it (`src/lib/tracker/linear.ts:378`). We want
  impact-ranking derived from issue content.
- **Durable-markdown precedent exists.** `docs/FLYWHEEL-STATE.md` is a markdown durable-state file the
  orchestrator reads/writes and commits. A backlog sequence deserves the same treatment.
- **`.pan/` writes already auto-commit.** `src/lib/pan-dir/auto-commit.ts` (`queueAutoCommit`, introduced
  in PAN-1146, 2026-05-16) debounces and commits `.pan/{continues,specs,drafts}` writes to `main`.
  A new `.pan/backlog/sequence.md` rides the same mechanism.

## 6. `.pan/backlog/sequence.md` shape

the authoritative parse target.

for every issue; a full-paragraph `rationale` only for the active top tier (≈ top 80). The markdown

## 6a. Dependencies — provenance & reconciliation (FR-16)

Dependencies are first-class: they are the DAG edges (`dependsOn` per node + `edges`). But cross-issue
dependencies are **not available programmatically** the way one might assume:

- **GitHub has no native blocks/blocked-by**, and the tracker reads no relationships for GitHub
  (`src/lib/tracker/github.ts:342` — `getChildIssues` returns `[]`: "GitHub Issues does not support
  hierarchical parent-child relationships"). GitHub gives us only what's in the issue *body*: `#1234`
  references and prose ("blocked by PAN-1665", "after the throttle").
- **vBRIEF / beads edges are intra-issue** — dependencies *between an issue's own work items*
  (`blocks`/`informs`/`suggests`/`invalidates`, rendered by `PlanDAG.tsx`), **not** between backlog issues.
  Distinct graph; do not conflate.
- **Linear** (not PAN) does expose real issue relations — read them where a tracker provides them.

So cross-issue edges come from three sources, and each edge records its **provenance** (`source`):

| `source` | How obtained | Authority |
|---|---|---|
| `github-ref` | parsed from the issue body (`#N`, "blocked by PAN-N", "depends on", "after"); native relations/sub-issues where the tracker exposes them | authoritative |
| `operator` | a human added/edited the edge on the page | authoritative, preserved verbatim |
| `ai-inferred` | the AI proposed it from reading the bodies | advisory, recomputed each pass, carries `confidence` |

**Reconciliation rules:**
- Each pass **re-derives `github-ref` edges** (re-parse bodies / re-read relations) and **preserves
  `operator` edges verbatim** (same discipline as the pickup gate and pinning). `ai-inferred` edges are recomputed.
- **Authoritative edges win.** If an `ai-inferred` edge contradicts a `github-ref`/`operator` edge, the
  authoritative one stands and the conflict is surfaced (an "AI disagrees" marker) for the operator.
- **We read GitHub metadata; we do not write back.** GitHub has no dependency field to round-trip into, so
  `sequence.md` stays the source of truth for the *sequence*. Provenance is what keeps it honest rather than
  a pile of AI guesses. If bidirectional sync is wanted later, the place is the tracker layer
  (`github.ts` — read sub-issues / a `Depends-on:` body convention).
- The operator can **add / remove / override a dependency edge on the page** (→ `source: operator`),
  exactly like a pickup gate; it persists in `sequence.md` and the AI never clobbers it.

## 7. Scale & cost budget — panopticon-cli has 522 open issues (measured)

This feature is designed for a 500+ issue backlog from day one.

- **Output file:** rationale stored once + tiered detail → `sequence.md` ≈ **50–65k tokens** at 522 issues.
  The naive triplicated-rationale shape would be **~120k tokens** — explicitly rejected (NFR-5).
- **Input cost is the real driver:** average open issue (title+body) ≈ **715 tokens**; reading all 522
  ≈ **~370k tokens**, exceeding one context window. Therefore (NFR-6):
  - **Creation** and **Review** passes are **chunked map-reduce** — rank in batches against a running
    shortlist, then a final merge — never one prompt containing all bodies.
  - **Incremental** passes read only the prior `sequence.md` (~50–65k) + the few changed bodies.
- **Visualization (FR-12):** 522 nodes cannot render at once. The page renders only the selected **tier**
  (Now / Next / Later / Someday) plus dependency neighbors; the long tail collapses behind counts;
  segment pills + filters (importance, condition, in-pipeline, ready) pivot the full 522. See the
  scaled mockup.

## 8. The three passes

| Pass | When | Scope | Cost |
|---|---|---|---|
| **Creation** | first run / no file yet | entire backlog, from scratch | high (once) |
| **Incremental** | default; debounced on backlog delta | only changed issues; preserves prior order + rationale | low |
| **Review** | periodic (configurable) or on demand | entire backlog re-evaluated from first principles | high |

Creation runs automatically when no file exists; the page selects Incremental vs Full review for manual
runs. Each pass stamps `pass` + `generatedAt` so cadence/drift is auditable in git history.

## 9. Issue-condition assessment

- **`needs-refinement`** → too vague to plan. Amber `⚠ REFINE` node (amber = "a human must act"). Still ranked.
- **`stale`** → likely no longer relevant. `⊘ STALE`, surfaced as a candidate to close.
- **`ok`** → no flag.

Stored per node in `sequence.md`; surfaced in the UI; never auto-modifies the tracker.

**Placement of stale issues (FR-11).** A `stale` issue ranks **low** — it sinks to the bottom tier
(Someday) and appears under the **Stale-candidates** filter for one-click closing, so irrelevant work
never competes for the top and doesn't clutter the Now DAG. The **exception is an in-pipeline stale
issue**: it stays **pinned** at its position (FR-13) and is rendered **bright + flagged** (not dimmed) —
the ring + `⊘ STALE` carry the warning — so the operator can **abort** active work the AI now believes
is dead rather than finish it. (A non-pipeline stale issue only ever appears in Now if the backlog is so
small that the bottom tier reaches the top — acceptable, and the close action clears it.)

New top-level tab **Backlog** ("Backlog Sequencer"). DAG via ReactFlow 11 + `@dagrejs/dagre`
(same as `PlanDAG.tsx`), new `IssueNode`.

| Channel | Encodes | Treatment |
|---|---|---|
| Node size | AI size estimate | XS→XL scales footprint |
| Rank badge | sequence position | `#1`, mono, top-left |
| Left-border heat | importance | red→amber→neutral→dim |
| In-pipeline | already being worked | glowing ring in the phase color + live verb badge (reuse `.plan-glow`) |

| Rationale | why this rank | click node → side panel renders rationale from `sequence.md` |

Color follows the graph-viz precedent of `PlanDAG.tsx` while honoring style-guide signal semantics.

## 10a. Tiers (derived view, not an AI field)

The **Now / Next / Later / Someday** bands shown in the scaled mockup are a *derived presentation
layer* over the ranking — **not** a per-issue field the AI assigns. They are computed deterministically
from rank/score thresholds, so they reproduce from the ranking alone (NFR-1). Default bands
(configurable): **Now** = ranks 1–N (N defaults to the active agent capacity / `maxAgents`), **Next** =
the following band, **Later**, **Someday** = the long tail. The AI's job stays "rank + score + condition";
tiering is a function of that output. The page renders one tier's DAG at a time (FR-12).

## 10b. In-pipeline issues are pinned — never re-sequenced (FR-13)

Work already moving through the pipeline must not get reshuffled under the operator. The sequencer
**reads** each in-pipeline issue's live state (`derivePipelineState`) but **never re-ranks, re-orders,
or changes the `why`/`rationale`** of a pinned issue. Pinned = has a live workspace/branch/agent/PR or a
non-pending review status. New and changed issues slot *around* the pinned set; only a pinned issue's
flags/condition refresh from live state. This is a hard rule in the agent prompt and enforced in
`writeSequence` (a pinned node's rank/why are carried forward verbatim from the prior file). The page
marks pinned nodes with the in-pipeline ring and excludes them from "candidates" lists.

## 10c. Flywheel integration — auto-pickup reads the sequence (FR-14)

Today the Flywheel picks by fixed rule (tier P0–P3, then oldest-first; `docs/flywheel-brief.md`). With the
Sequencer, when `flywheel.auto_pickup_backlog` is enabled the Flywheel consults the **sequence** as its
ranked pickup source instead of oldest-first:

- New read API for the orchestrator: `getBacklogSequence(projectKey)` → the cache view (rank-ordered).
- **Eligibility filter (all must hold):** `pickup gate ≠ blocked` (§10d) · not parked
  (`needs-design`/`needs-discussion`) · not in-pipeline (pinned) · passes the existing author/assignee
  **security filter** (`flywheel-brief.md:37-47` — unchanged, never weakened) · `ready` or has a PRD
  (so there's something to act on).
- The Flywheel starts/plans the **highest-ranked eligible** issue (a `ready` one is startable; a
  PRD-only one is plannable) up to `minAgents`. Impact-ranked-first replaces oldest-first.
- Brief change: `docs/flywheel-brief.md` gains a "prefer the sequence when present" clause; the rule-based
  tiers remain the fallback when no `sequence.md` exists. This is additive — no existing safeguard removed.

## 10d. Operator pickup gate — mark Ready / Block from the page (FR-15)

The page is the natural place to greenlight or hold an issue. Each issue has an operator-set **pickup gate**
∈ {`auto`, `ready`, `blocked`}:

- **`ready`** — operator greenlight / promote: eligible for auto-pickup (and surfaced as "do this next") even
  if heuristics wouldn't reach it yet.
- **`blocked`** — operator hold: never auto-picked. The Block action also applies a parked label
  (`needs-design` by default) so the **existing** Flywheel skip honors it immediately, before the §10c
  integration lands.
- **`auto`** (default) — defer to heuristics/eligibility.

Storage: `gate` per node in `sequence.md`, **operator-owned**. The sequencer treats it as immutable input —
it must carry the gate forward verbatim and never clobber it (same discipline as pinning). UI: a
segmented control (Auto / ✓ Ready / ⛔ Block) in the rationale panel; nodes show a `📌 PROMOTED` or
`⛔ HELD` badge. Setting a gate writes through `POST /api/backlog/sequence/gate` → updates `sequence.md`
(+ parked label for Block) → auto-commit.

## 11. Requirements

**Functional**
- **FR-1** Sequencer writes a total ordering to `.pan/backlog/sequence.md` (table + rationale + JSON), auto-committed.
- **FR-2** Ranking is AI-derived from content/impact, with **urgency** and **relevancy** as explicit inputs; GitHub priority **and issue age** are inputs, never the sole determinant (this replaces the Flywheel's age-based oldest-first tiebreak).
- **FR-3** Incremental: given the prior file, preserve existing order unless a delta justifies change; each change carries a rationale.
- **FR-4** Sequencer model configurable via `roles.sequencer.model`, surfaced in Settings → Roles.
- **FR-5** A **Backlog** page renders the sequence as an interactive ReactFlow+dagre DAG.
- **FR-6** Each node encodes rank, size, importance, in-pipeline, has-PRD, ready.
- **FR-7** Selecting a node shows the AI's rationale.
- **FR-8** `GET /api/backlog/sequence` returns the parsed file joined with live pipeline/PRD/ready state; `POST /api/backlog/sequence/regenerate` triggers the sequencer.
- **FR-9** SQLite `backlog_sequence` is a cache fully rebuildable from `sequence.md`.
- **FR-10** Three passes (creation/incremental/review); creation auto on first run; incremental default; page selects incremental vs full review; each run stamps `pass`.
- **FR-11** Per-issue `condition` ∈ {ok,needs-refinement,stale}, stored in the file; `⚠ REFINE` (amber) and `⊘ STALE` surfaced; neither auto-modifies the tracker. A non-pipeline `stale` issue ranks into the bottom tier and is reachable via the Stale-candidates filter; an in-pipeline stale issue stays pinned in place, bright and flagged, so the operator can abort it.
- **FR-12** The page handles 500+ issues: top-tier default view + dependency neighbors, collapsed long tail, filters (importance, condition, in-pipeline, ready).
- **FR-13** In-pipeline issues are **pinned**: the sequencer never re-ranks, re-orders, or rewrites them; only their live flags/condition refresh. New/changed issues slot around the pinned set.
- **FR-14** When `flywheel.auto_pickup_backlog` is enabled, the Flywheel picks the highest-ranked **eligible** issue from the sequence (gate≠blocked, not parked, not in-pipeline, passes the author/assignee security filter, ready-or-has-PRD) — impact-ranked-first replacing oldest-first; rule-based tiers remain the fallback when no `sequence.md` exists.
- **FR-15** Each issue has an operator-set **pickup gate** ∈ {auto, ready, blocked}, stored in `sequence.md` and preserved verbatim by the sequencer. The page sets it (Auto / ✓ Ready / ⛔ Block); `blocked` also applies a parked label so the existing Flywheel skip honors it immediately; the Flywheel (FR-14) respects it.
- **FR-16** Cross-issue **dependencies** are DAG edges with provenance (`source ∈ {github-ref, operator, ai-inferred}` + `confidence`). `github-ref` edges are parsed from issue bodies/relations and re-derived each pass; `operator` edges are preserved verbatim and editable on the page; `ai-inferred` edges are advisory. Authoritative (`github-ref`/`operator`) edges override `ai-inferred` ones and conflicts are surfaced. GitHub metadata is **read, not written back**. vBRIEF/beads intra-issue edges are a separate graph and are not used as backlog edges.

**Non-functional**
- **NFR-1** Reproducible: a fresh checkout + the same `sequence.md` yields the identical visualization with no AI call.
- **NFR-2** No new frontend viz dependency — reuse ReactFlow 11 + `@dagrejs/dagre`.
- **NFR-3** `sequence.md` stays human-diffable; auto-commit subject `chore(state): update backlog sequence (<project>)`.
- **NFR-4** Style-guide conformance: DM Sans, `font-medium`, semantic tokens, signal-color discipline.
- **NFR-5** File-size budget: ≤ ~65k tokens at 522 issues — rationale once, paragraphs for active tier only. Parser test asserts per-issue footprint stays in budget.
- **NFR-6** Input-scale: Creation/Review are chunked map-reduce (never one prompt with all bodies); Incremental reads only the prior file + changed bodies.
