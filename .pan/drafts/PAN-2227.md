## 0. Glossary (define before use)

- **Ratchet** — a guard that lets a metric move only in the good direction: god files may
  shrink but never grow; the `any`-allowlist may lose entries but not gain them silently.
- **Baseline** — `scripts/file-size-baseline.txt`, format `<lines> <path>` one per line,
  path-sorted. A file listed there may not exceed its recorded line count. Files NOT listed
  are governed by the **ceiling**.
- **Ceiling** — 1,000 lines (`CEILING=1000` in `scripts/lint-file-size.sh`): the maximum
  size for any non-test `.ts`/`.tsx` file under `src/` that has no baseline entry.
- **Allowlist** — `eslint-any-allowlist.json`, a JSON array of 155 file paths (verify with
  `python3 -c "import json;print(len(json.load(open('eslint-any-allowlist.json'))))"`).
  `.eslintrc.cjs` loads it (grep anchor: `anyAllowlist = require('./eslint-any-allowlist.json');`)
  and turns `@typescript-eslint/no-explicit-any` OFF for the listed files.
- **Write point** — where a change enters the repo. This repo's dominant write pattern is
  direct-to-main bot commits (agents pushing to `main`), so CI-only enforcement means every
  violation is discovered AFTER it lands: a red-main outage or a silent baseline bump.
- **Audited bump** — a commit that RAISES a baseline entry (or adds one) or ADDS an
  allowlist entry, carrying an issue reference in its commit message so the erosion is
  traceable. Lowerings and removals are always free.
- **Auto-lower** — when a baselined file shrinks, the baseline entry is rewritten down to
  the new size (or dropped entirely once the file is under the ceiling), so the ratchet
  actually ratchets instead of keeping the old high-water mark.
- **Issue reference** — a token matching the extended regex `([A-Z]+-[0-9]+|#[0-9]+)`
  anywhere in the full commit message (subject or body). Matches `PAN-2227` and the
  `(#2223)` suffix GitHub appends to squash-merges.
- **Guard-script convention** — this repo enforces invariants with bash scripts wired into
  `npm run lint` (`scripts/lint-state-writes.sh`, `scripts/lint-overdeck-boundaries.sh`,
  `scripts/lint-file-size.sh`). Each new guard follows that exact style:
  `#!/usr/bin/env bash`, `set -euo pipefail`, `cd "$(dirname "$0")/.."`.
- **Range mode / last-commit mode** — the two entry modes of the new audit script (WI-2).
  Range mode audits every commit in `<base>..<head>` that touched a ratchet file (used by
  pre-push, which knows exactly what is being pushed). Last-commit mode audits only the most
  recent commit that touched each ratchet file (used by the `npm run lint` chain, local and CI).

## 2. Locked design decisions

- **D1 — Pre-push checks the working tree, not each pushed commit.** The pre-push hook runs
  `bash scripts/lint-file-size.sh` once (working-tree state). For direct-to-main writers the
  tree at push time IS what lands, so this is the right cheap check. Auditing every pushed
  commit's tree (`git archive` per commit) is rejected as slow and complex. Limitation
  accepted and documented in the hook comment.
- **D2 — Auto-lower is enforced by check-mode failure + a `--update` writer.** Check mode
  (`bash scripts/lint-file-size.sh`) FAILS when any baseline entry is stale (file shrank,
  file missing, or file now ≤ ceiling), printing the one-command fix. `--update` rewrites
  the baseline: lower stale entries, drop dead/sub-ceiling entries, **never raise, never
  add**. This makes "a PR that shrinks a god file lowers its baseline in the same change"
  mechanically verified — CI lint is red until the lowered baseline is committed.
- **D3 — An entry whose file drops to ≤ 1,000 lines is REMOVED, not frozen.** The file is
  then governed by the ceiling like every other file (it may regrow to 1,000 unnoticed —
  same rule as all normal files). Rationale: the ratchet's job is eliminating god files, not
  freezing decomposed files at their post-decomposition size; keeping exact-size entries for
  sub-ceiling files would make every ordinary edit to them a baseline-churn commit.
- **D4 — Raising a baseline entry stays a manual, audited edit.** `--update` never raises or
  adds entries. Accepting growth means hand-editing `scripts/file-size-baseline.txt` in a
  commit whose message carries an issue reference (that is the audit trail).
- **D5 — The bump audit is one script with two modes.** `scripts/lint-ratchet-audit.sh`
  audits increases to both ratchet files. Pre-push calls it in range mode with the exact
  pushed range (`<remote_sha>..<local_sha>` per branch ref). The lint chain calls it with no
  args → last-commit mode. A root commit (no parent) or a shallow clone with no reachable
  parent degrades to WARN + exit 0 — never a false red; pre-push range mode is the primary
  gate. Known accepted gap: last-commit mode sees only the newest touching commit, so an
  increase-then-decrease pair inside one PR can net-increase past it — the pre-push range
  mode catches that locally, and it is rare enough not to justify CI event plumbing.
- **D6 — Allowlist audit is set-based.** Parse the JSON array at `C^` and `C` with
  `node -e` (same idiom the pre-push hook already uses for `read_version`); any path present
  in new but not old is an increase requiring an issue ref; removals are free. Do NOT diff
  raw JSON text lines (reordering would false-positive).

## 4. Requirements

### Functional
- **FR-1** Every `git push` runs the file-size guard locally: a working tree containing a
  new >1,000-line src file or a grown baselined file refuses to push.
- **FR-2** Check mode of `scripts/lint-file-size.sh` fails on stale baseline entries —
  file shrank below its entry, file missing, or file now ≤ ceiling — and the failure
  message names the entry and prints the exact fix command
  (`bash scripts/lint-file-size.sh --update`).
- **FR-3** `bash scripts/lint-file-size.sh --update` rewrites the baseline: lowers each
  stale-high entry to the file's current line count, drops entries for missing or
  ≤-ceiling files, keeps violation entries unchanged (never raises), adds nothing, and
  preserves path-sorted `<lines> <path>` format.
- **FR-4** `scripts/lint-ratchet-audit.sh` fails when a commit raises a
  `scripts/file-size-baseline.txt` entry or adds a new one and that commit's message has no
  issue reference (`[A-Z]+-[0-9]+` or `#[0-9]+`). Lowering or removing entries never fails.
- **FR-5** The same script applies the same rule to `eslint-any-allowlist.json`: added
  paths require an issue reference in the adding commit's message; removals are free.
- **FR-6** The pre-push hook runs the audit in range mode over exactly the pushed range for
  every `refs/heads/*` ref (new-branch and deleted-branch edge cases per WI-3).
- **FR-7** `npm run lint` runs the audit in last-commit mode (`lint:ratchet-audit` chained
  after `lint:file-size`), and the CI lint job checkout uses `fetch-depth: 0` so the
  parent-diff works there.
- **FR-8** This change ships with the baseline reconciled by `--update` in the same PR, so
  check mode is green at merge (expected effect at authoring time: 7 entries dropped,
  10 lowered — see `## Re-verify at execution`; regenerate, don't copy these numbers).

### Non-functional
- **NFR-1** Guard-script convention: pure bash additions + `package.json` script wiring +
  hook/workflow edits. No production `src/**` code is modified.
- **NFR-2** Pre-push additions complete in under ~2 seconds on this repo (the file-size
  guard is `find`+`wc`; the audit is a handful of git plumbing calls per pushed ref).
- **NFR-3** No false reds: any situation where the audit cannot see history (root commit,
  shallow clone without parent) is WARN + exit 0, with the warning naming the limitation.
- **NFR-4** `.husky/pre-push` stays `#!/bin/sh`-compatible (it may invoke
  `bash scripts/...` as it does today, but the hook's own syntax stays POSIX sh).

## 5. Work items

### WI-1 — Staleness check + `--update` mode in `scripts/lint-file-size.sh` *(FR-2, FR-3)*

**Files:** `scripts/lint-file-size.sh`.

**Steps:**
1. Add arg parsing at the top (after `set -euo pipefail`): `MODE=check`; if `${1:-}` is
   `--update`, `MODE=update`.
2. In the existing per-file loop, the baselined branch currently only fails on growth:

   **Before** (grep anchor: `god files must shrink, not grow`):
   ```bash
   if [[ -n "$allowed" ]]; then
     if (( n > allowed )); then
       echo "✖ $f grew to $n lines (baseline $allowed) — god files must shrink, not grow."
       fail=1
     fi
   ```
   **After** — also detect stale-high entries:
   ```bash
   if [[ -n "$allowed" ]]; then
     if (( n > allowed )); then
       echo "✖ $f grew to $n lines (baseline $allowed) — god files must shrink, not grow."
       fail=1
     elif (( n < allowed )); then
       echo "✖ stale baseline: $f is $n lines but baselined at $allowed — run: bash scripts/lint-file-size.sh --update"
       stale=1
     fi
   ```
   (Track `stale` separately from `fail` so the summary can distinguish the two, but both
   set exit 1 in check mode.)
3. After the loop, add a pass over baseline entries themselves (the find loop only visits
   files that exist): every baseline path that is missing from disk, or whose current count
   is ≤ `CEILING`, is stale → check mode reports it (`stale baseline: <path> no longer
   exists` / `is under the ceiling`) and exits 1 with the `--update` instruction.
4. Implement `--update`: rebuild the baseline in place — for each existing entry, drop it
   if the file is missing or `n <= CEILING`; write `min(n, allowed)` otherwise; never add
   entries. Write path-sorted (`sort -k2`), then print a one-line summary
   (`✓ baseline updated: X lowered, Y dropped, Z unchanged`) and exit 0.
5. Update the header comment: document `--update`, D3 (sub-ceiling entries are dropped),
   and D4 (raising is a manual audited edit).

**Named test:** `tests/unit/scripts/lint-file-size.test.ts` (WI-6).

### WI-2 — New `scripts/lint-ratchet-audit.sh` + lint-chain wiring *(FR-4, FR-5, FR-7)*

**Files:** new `scripts/lint-ratchet-audit.sh`; `package.json`.

**Steps:**
1. Create the script in the guard-script convention. Interface:
   `lint-ratchet-audit.sh [--range <base>..<head>]`. Ratchet files:
   `scripts/file-size-baseline.txt`, `eslint-any-allowlist.json`.
2. Commit selection: range mode → `git rev-list --reverse '<range>' -- <file>` per ratchet
   file; last-commit mode → `git log -1 --format=%H -- <file>` (empty → pass for that file).
3. Per selected commit `C`: if `git rev-parse -q --verify "$C^" >/dev/null` fails (root
   commit or shallow history), print
   `⚠ ratchet audit: cannot see parent of ${C:0:12} — skipping (pre-push range mode is the primary gate)`
   and continue (NFR-3). Otherwise detect increases:
   - **Baseline:** materialize `git show "$C^:scripts/file-size-baseline.txt"` (missing at
     parent → treat as empty) and `git show "$C:..."`; compare per-path with awk/sort: a
     path only in new, or with a larger count in new, is an increase.
   - **Allowlist:** parse both revisions with the hook's existing `node -e` idiom (D6),
     e.g. `git show "$C:eslint-any-allowlist.json" | node -e "let d='';process.stdin.on('data',c=>d+=c).on('end',()=>JSON.parse(d).forEach(p=>console.log(p)))"`,
     then `comm -13 <(old sorted) <(new sorted)` → added paths are increases.
4. If a commit has increases, check `git log -1 --format=%B "$C"` against
   `grep -Eq '([A-Z]+-[0-9]+|#[0-9]+)'`. No match → print the offending commit sha+subject,
   each increase, and the requirement ("a ratchet increase must reference an issue —
   amend the commit message with e.g. PAN-XXXX"), exit 1.
5. Success output: `✓ ratchet audit passed (no unaudited baseline/allowlist increases)`.
6. Wire into `package.json`:

   **Before** (grep anchor — end of the `lint` script value):
   ```json
   "lint": "eslint src/ --ext .js,.ts,.tsx --no-inline-config && npm run lint:effect && npm run lint:permissions && npm run lint:state-writes && npm run lint:overdeck-boundaries && npm run lint:skills && npm run lint:prompts && npm run lint:file-size",
   ```
   **After:**
   ```json
   "lint": "eslint src/ --ext .js,.ts,.tsx --no-inline-config && npm run lint:effect && npm run lint:permissions && npm run lint:state-writes && npm run lint:overdeck-boundaries && npm run lint:skills && npm run lint:prompts && npm run lint:file-size && npm run lint:ratchet-audit",
   ```
   and add below the existing `lint:file-size` entry:
   ```json
   "lint:ratchet-audit": "bash scripts/lint-ratchet-audit.sh",
   ```

**Named test:** `tests/unit/scripts/lint-ratchet-audit.test.ts` (WI-6).

### WI-3 — Pre-push wiring *(FR-1, FR-6)*

**Files:** `.husky/pre-push`.

**Steps:**
1. After `set -e` and before `read_version() {`, run the working-tree file-size guard once,
   with stdin detached so it cannot consume the ref list:
   ```sh
   # PAN-2227: ratchets enforce at the write point — catch god-file growth
   # before origin does. Checks the WORKING TREE (what a direct-to-main
   # writer is about to make main), not each pushed commit.

   ```
2. Inside the existing ref loop (grep anchor:
   `while IFS=' ' read -r local_ref local_sha remote_ref remote_sha; do`), add a
   `refs/heads/*` case ABOVE the existing `refs/tags/v*` case:
   ```sh
   refs/heads/*)
     # PAN-2227: audited ratchet bumps — every pushed commit that raises a
     # baseline entry or adds an eslint-any-allowlist entry must reference
     # an issue in its message.
     zero=0000000000000000000000000000000000000000
     if [ "$local_sha" = "$zero" ]; then
       continue  # branch deletion — nothing to audit
     fi
     if [ "$remote_sha" = "$zero" ]; then
       base=$(git merge-base origin/main "$local_sha" 2>/dev/null || true)
     else
       base="$remote_sha"
     fi
     if [ -n "$base" ]; then
       bash scripts/lint-ratchet-audit.sh --range "$base..$local_sha" </dev/null
     else
       echo "⚠ ratchet audit: no base for $local_ref — skipping" >&2
     fi
     ;;
   ```
   `set -e` at the top of the hook makes a non-zero audit exit refuse the push.
