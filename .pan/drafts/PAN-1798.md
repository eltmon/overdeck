# PAN-1798: Shared tmux server must never live inside an agent's systemd scope

**Issue:** https://github.com/eltmon/overdeck/issues/1798
**Mode:** `pan plan --auto` (no interactive Q&A; defaults recorded in `plan.autoDecisions[]`)
**Authored:** 2026-06-12

## Glossary

- **Managed socket** — the tmux socket named `panopticon` (`tmux -L panopticon`); all Overdeck agent/conversation sessions live on one shared server bound to it. Name comes from `getManagedTmuxSocketName()` (`src/lib/tmux.ts:65`).
- **Managed config** — `~/.panopticon/tmux/panopticon.tmux.conf`, regenerated from `MANAGED_TMUX_CONFIG_CONTENT` (`src/lib/tmux.ts:26`).
- **Founding** — the act of starting the shared tmux server. Any `tmux new-session` against a dead socket forks a server that (a) inherits the calling process's cgroup and (b) carries the full `new-session ... launcher.sh` command line.
- **Dirty founding** — founding via an agent/conversation spawn's `new-session`: the server lands in that spawn's environment with an agent-specific cmdline. Clean founding = `tmux -L panopticon -f <conf> start-server` inside a dedicated transient systemd scope.
- **`tmux-spawn-*.scope`** — transient systemd scopes created by tmux 3.4's own systemd integration for **pane processes** (verified empirically 2026-06-12: descriptions read "tmux child pane N launched by process M"; the *server* is NOT moved — it stays in the founding client's cgroup).
- **The latch** — module global `tmuxContextPrepared` in `src/lib/tmux.ts` (checked at lines 164 and 172): once true, `ensureManagedTmuxConfigSync/Async` return immediately for the rest of the process lifetime.
- **Ensure helper** — the new exported `ensureOverdeckTmuxServer()` this plan introduces: check server aliveness, found cleanly if dead, run the founder guard.

## Problem

Live incident 2026-06-12 03:43: `pan kill` of four work agents destroyed the entire shared tmux server — all 5 conversations and two review convoys died. The server had been dirty-founded by an agent spawn, so it was killable both by cmdline match (`pgrep -f launcher.sh`) and by its membership in an ephemeral scope.

Commit `b7e4ab885` (main, 2026-06-12 08:35) landed a partial fix:
1. `set -g exit-empty off` in the managed config (`src/lib/tmux.ts:31`) — server survives zero sessions.
2. `foundManagedServerSync()` (`src/lib/tmux.ts:114`) — clean founding via `systemd-run --user --scope --collect --unit=panopticon-tmux-<Date.now()> --quiet tmux -L panopticon -f <conf> start-server`, fallback to plain `start-server`.
3. `isTmuxProcessSync()` filter in `src/lib/agents.ts` kill sweeps — never signal a tmux process.

**The remaining defect:** the founding preflight only runs inside `ensureManagedTmuxConfigSync/Async`, which are guarded by the once-per-process latch. The dashboard is long-lived; it latches at boot (`src/dashboard/server/main.ts:80` `await ensureManagedTmuxContextOnce()`). If the tmux server dies later, the next `new-session` from the dashboard (deacon resume, conversation spawn, role spawn) dirty-founds a new server. **Proof:** the current live server (PID 1947249, founded 08:53 — *after* the fix commit) carries cmdline `tmux ... new-session -d -s conv-20260612-3871 ... bash .../launcher.sh` — it was dirty-founded by a conversation spawn. (It has since been manually adopted into `panopticon-tmux-adopted.scope` by the operator.)

Also missing from the landed fix: the founder cgroup guard (issue comment part 3), a logged warning on the non-systemd fallback, and any tests at all.

## Verified codebase facts (2026-06-12, workspace `feature/pan-1798` = main for these files)

- `src/lib/tmux.ts:163-177` — `ensureManagedTmuxConfigSync/Async`: latch check → write config → `reloadManagedTmuxConfig*` (which contains the aliveness check + founding) → set latch.
- `src/lib/tmux.ts:92-99` — `isManagedServerAliveSync()`: `tmux -L panopticon list-sessions`, exit 0 ⇒ alive. (Non-creating commands like `list-sessions` do NOT found a server; only `new-session`/`start-server` do.)
- `src/lib/tmux.ts:239-249` — `tmuxExecAsync`/`tmuxExecSync` are the exec choke points; both call `ensureTmuxContextPrepared*` (latched) and then run tmux.
- `src/lib/tmux.ts:806-833` `createSession` (Effect) and `:385` `createSessionSync` both funnel through `tmuxExec*` with `buildNewSessionArgs()` (`:251`, `args[0] === 'new-session'`).
- **Complete founding-vector audit:** every `new-session` against the managed socket goes through `tmuxExec*`. External arg-builders (`buildTmuxArgs`/`getTmuxCommand`/`buildTmuxCommandString`) are used only for `attach-session` (ws-terminal.ts:496, terminal-service.ts:142,157), `display-message` (conversations/current.ts:39), and `set-environment` (spawn-planning-session.ts:190) — none can found. `src/lib/runtime/claude.ts:118` runs a raw `execa('tmux', ['new-session', ...])` on the **default** socket, but the module is imported nowhere outside `src/lib/runtime/` — dead legacy, out of scope (do not delete; Karpathy surgical-changes rule).
- Remote Fly agents (`src/lib/remote/remote-agents.ts:260`) run tmux on the remote host — out of scope.
- `tmux -V` = 3.4. Empirical probe (throwaway socket): a freshly founded server stays in the founding client's cgroup; panes get `tmux-spawn-<uuid>.scope`. `tmux -L <sock> display-message -p '#{pid}'` returns the server PID **even with zero sessions**.
- `getTmuxConfigMode()` (`src/lib/tmux.ts:182`) — all of this applies only to `managed` mode; `inherit-user` mode has no managed socket and must remain untouched.

## Requirements

- **FR-1** Exported ensure helper. `ensureOverdeckTmuxServer()` (async; thin sync wrapper where the legacy sync path needs it): if no server answers on the managed socket, found one cleanly — `systemd-run --user --scope --collect --unit=panopticon-tmux-<unique> --quiet tmux -L panopticon -f <managed conf> start-server` — then verify the socket answers before returning.
- **FR-2** De-latch founding. The aliveness-check + founding preflight runs before **every** session-creating tmux command (`args[0] === 'new-session'`) in `tmuxExecAsync` and `tmuxExecSync`. The once-per-process latch may continue to govern **config-file writing only**.
- **FR-3** Boot path. Dashboard boot (`main.ts:80`) keeps founding the server before `server.listen`, routed through the ensure helper.
- **FR-4** Fallback. When `systemd-run` is unavailable or fails, found via plain `tmux ... start-server` and log a single warning that scope isolation is unavailable (currently the fallback is silent — `src/lib/tmux.ts:123-126`).
- **FR-5** Founder guard. After ensuring a live server, resolve the server PID via `tmux -L panopticon display-message -p '#{pid}'`, read `/proc/<pid>/cgroup`; if the cgroup path's final component matches `tmux-spawn-*.scope`, log a loud warning naming the scope unit and server PID. Warn-only — never restart, signal, or migrate the server. At most one warning per server PID per process; a re-founded server (new PID) warns again. Silent no-op when `/proc` is unreadable (non-Linux).
- **FR-6** `exit-empty off` stays in the managed config (already landed — lock with a regression test).
- **NFR-1** No live-system mutation: nothing in this change may restart/kill/migrate an already-running server. Behavior in `inherit-user` mode is unchanged.
- **NFR-2** Test discipline: mocked exec (`vi.mock('child_process')`) or pure extracted functions; any real tmux interaction uses a throwaway `-L` socket name — never `panopticon`. Fake timers (`vi.useFakeTimers()`) for any retry/delay logic (repo rule).
- **NFR-3** Async-first: the preflight on the `tmuxExecAsync` path must use async exec for founding/aliveness (no new blocking `execFileSync` reachable from the dashboard's async flows — no-execSync rule, PAN-70). The sync path (`tmuxExecSync`, legacy) may keep sync exec for parity.
- **NFR-4** Cost: the aliveness probe runs only for session-creating commands, not on every `tmuxExec*` call (capture-pane polling etc. must not gain a subprocess per call).

## Decisions (made here; do not relitigate during implementation)

- **D1 — keep timestamped transient scopes**, not the issue's literal stable `--unit=panopticon-tmux-server` service. A `Type=forking` service with a stable name needs RemainAfterExit fiddliness and collides when a stale unit lingers; timestamped `--scope` units (already landed) are collision-proof and equally outside any agent's scope. Uniqueness suffix: keep `Date.now()`; collisions within 1ms are not a real risk, and the founding helper is rarely called concurrently in one process.
- **D2 — preflight placement**: inside `tmuxExecAsync`/`tmuxExecSync`, gated on `args[0] === 'new-session'`. One choke point covers every current and future call site; verified that nothing else can found.
- **D3 — guard scope**: warn on `tmux-spawn-*` cgroups only (per issue comment), not on other "wrong" cgroups; warn-once-per-PID via module state.
- **D4 — deacon auto-resume race** (resuming a just-killed agent, "has a resumable Claude session" failure) is the issue's "bonus race" — **out of scope**; file separately if it recurs.
- **D5 — post-found verification**: after founding, assert `isManagedServerAlive*()`; if still dead, log an error and continue (the subsequent tmux command will fail with its own error — do not add retry loops with real timers).

Before (tmux.ts:245-249):
```ts
function tmuxExecSync(args: string[], options?: Parameters<typeof execFileSync>[2]) {
  const mode = getTmuxConfigMode();
  ensureTmuxContextPreparedSync(mode);
  return execFileSync('tmux', [...getTmuxContextArgsForMode(mode), ...args], options);
}
```
After (shape):
```ts
function tmuxExecSync(args: string[], options?: Parameters<typeof execFileSync>[2]) {
  const mode = getTmuxConfigMode();
  ensureTmuxContextPreparedSync(mode);
  if (mode === 'managed' && args[0] === 'new-session') {
    ensureOverdeckTmuxServerSync();
  }
  return execFileSync('tmux', [...getTmuxContextArgsForMode(mode), ...args], options);
}
```
