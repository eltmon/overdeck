## 1. Background & motivation

### 1.1 Vision

Issues execute as a **tiered ensemble**: a cheap durable foreman runs the bead loop and all bureaucracy; standing tier agents (cheap â†’ frontier) are pre-instantiated and kept warm by plan-filtered commit feeds, stepping in only when a bead matching their tier comes up; an event-driven supervisor reviews diffs at commit boundaries instead of holding working context. Expensive-model attention becomes an interrupt handler, not a participant. No other orchestrator routes *context* per tier the way the vBRIEF plan lets us â€” the plan knows, before work starts, which future work depends on which commits.

### 1.2 Live evidence (2026-06-12, PAN-1788/PAN-1792 runs)

The PAN-1788 implementation ran exactly this shape manually:

| Role | Session | Cost / tokens | Output |
| --- | --- | --- | --- |
| Implementer | gpt-5.5 / codex (conv 2774) | **$31.77** | 25 beads + 4 fix commits, ~84 min across 4 chunks, self-recovered from 2 branch-reset incidents |
| Parallel implementer | kimi-k2.6 / pi (conv 2775) | â†‘82k â†“10k tokens, **99.5% cache hit** | 6 commits fixing 23 stale tests |
| Supervisor/orchestrator | Fable 5 (conv 2773) | $90.11 *total session* â€” supervision-specific marginal cost was **~30 commit-event wake-ups Ã— 500â€“2k tokens** | every spec deviation inspected within minutes of its commit; 2 branch-reset incidents caught by the monitor before the implementer noticed |

Supervision marginal cost ran ~1â€“5% of implementer cost. The catch-rate for "executor misread the spec" is highest at commit boundaries â€” a misread caught at commit 1 is one bead of rework, not seven. Warm-session chunk continuation beat fresh handoffs (zero re-onboarding for chunks 2â€“4). The commit stream worked as a semantic event bus exactly because one-bead-one-commit held.

### 1.3 What the design must avoid

The failure mode of the naive version â€” N standing models all "listening" and deciding whether to step in â€” is a race with N-way duplicated cost. Every piece of autonomy in dispatch is removed by design: the bead's metadata decides (D1), the foreman serializes (D2), the plan filters the feeds (D3).

## 4. Requirements

### Functional

- **FR-1 â€” Deterministic difficulty routing.** The next ready bead's `metadata.difficulty` ([types.ts:67](../../src/lib/vbrief/types.ts)) maps through a configured tier table (Â§5.1) to exactly one named tier; the foreman dispatches to that tier's standing agent. Tier assignment involves no model judgment; given the same plan and config, routing is reproducible. Beads missing `difficulty` fail plan-time scheduling validation (they cannot reach the router).
- **FR-2 â€” Foreman owns all bureaucracy.** The foreman session runs `bd ready` / `bd update <bead-id> --claim` / `bd close`, writes `statusOverrides`, composes commit messages, stages and commits each bead's change (D9), and performs tier dispatch and feed forwarding. Tier agents perform only implementation; their prompts contain no lifecycle commands.
- **FR-3 â€” Standing tier agents with plan-filtered feeds.** Tier agents are pre-instantiated registered sessions (one per tier appearing in the plan's schedule). Feed filtering derives from the plan-time relevance map: `blocks` edges ([types.ts:22](../../src/lib/vbrief/types.ts)), `metadata.foundationFor` ([docs/VBRIEF.md:346](../../docs/VBRIEF.md)), and `files_scope` overlap via `hasFileOverlap()` ([dag.ts:333](../../src/lib/vbrief/dag.ts)). **A tier agent with no relevant upcoming beads receives zero deliveries â€” zero tokens â€” during unrelated runs.** Tiers that never appear in the schedule are never instantiated.
- **FR-4 â€” Event-driven supervisor.** A standing supervisor session subscribes to commit events under a configured policy (`all` | `flagged` | `sampled`). For each subscribed commit it receives the diff plus the bead's ACs, and responds with an ack or a blocking finding on the existing inspect-status surface ([specialists.ts:399-401](../../src/dashboard/server/routes/specialists.ts)). It never implements. A blocking finding halts dispatch of dependent beads until the foreman lands a fix commit (mirroring today's `INSPECTION BLOCKED` flow).
- **FR-5 â€” Replay-based crash recovery.** If any standing session (tier agent or supervisor) dies, the foreman respawns it and replays its filtered commit feed reconstructed from git history + the relevance map. No salvage of the dead session's context is attempted. The replayed agent reaches dispatch-ready state without human intervention.
- **FR-6 â€” Plan-time tier scheduling.** At finalize, the scheduler computes the tier-run sequence from the DAG (`groupItemsByWave()`, [dag.ts:36](../../src/lib/vbrief/dag.ts)) and each item's difficulty: consecutive same-tier beads form one run; tier switches occur only at run boundaries. The schedule (tier runs + relevance map) is persisted as a plan artifact so the foreman and replay consume it without recomputation.
- **FR-7 â€” Delivery via the existing primitive.** All feed and dispatch messages to standing sessions go through `deliverAgentMessage` ([agents.ts:1371](../../src/lib/agents.ts)). Feed messages are marked ingestion-only (the message text instructs: do not respond). No new transport is introduced.
- **FR-8 â€” Tier table in config.** A `tiered_execution` config section defines tiers as (model, harness, difficulties served) plus the supervisor policy, generalizing today's claude-only `complexity_routing` ([src/lib/cloister/config.ts:138](../../src/lib/cloister/config.ts), search `complexity_routing`). Harness values resolve through the existing specialist-harness machinery ([src/lib/cloister/router.ts:187](../../src/lib/cloister/router.ts), search `export function getSpecialistHarness`), migrating to PAN-1787's unified `resolveHarness()` when that lands. Feature is off by default (`enabled: false`).

### Non-functional

- **NFR-1 â€” Near-zero per-event ingestion cost.** A feed delivery to a standing session should cost â‰ˆ (cached-prefix read + diff tokens) with ~0 output. **Measurement target for the pilot:** â‰¥90% of deliveries during an active tier run hit warm prompt cache (delivery-to-delivery gap < 5 min), and mean marginal cost per delivery â‰¤ 2k tokens. Measured per Â§6.3.
- **NFR-2 â€” Fly-slot compatibility.** Routing (FR-1), scheduling (FR-6), and delivery (FR-7) must work unchanged whether a tier agent is local or a remote Fly slot (PAN-1676). Concretely: nothing in the design may assume the tier agent shares the foreman's machine *except* the worktree-edit path in D9, which is the one acknowledged remote-mode delta (remote tiers would need the slot-branch model â€” deferred with D8/D12).
- **NFR-3 â€” Supervision cost ceiling.** Supervisor marginal cost per issue â‰¤ 10% of implementer cost under the `flagged` policy (evidence baseline: 1â€“5% under per-commit supervision of all commits). The pilot reports the measured ratio.

### 5.2 Foreman loop (FR-2, D2, D9)

The foreman is the PAN-1762 work-agent foreman with two added responsibilities: routing and feed forwarding. The per-bead workflow changes shape:

**Before** (today's single-agent loop, [roles/work.md:53-65](../../roles/work.md), abbreviated):

```
1. bd ready â€” next bead
2. bd update <bead-id> --claim
3. implement the bead yourself
4. git add â€¦ && git commit       (one bead = one commit)
5. bd close <bead-id>
6. if requiresInspection: pan inspect â€¦ ; wait for verdict
```

**After** (foreman loop â€” design sketch):

```
1. bd ready â€” next bead
2. router: bead.metadata.difficulty â†’ tier (deterministic table lookup)
3. flush pending feed batch to that tier agent if its tier run is starting (D10)
4. bd update <bead-id> --claim
5. dispatch: deliverAgentMessage(tierAgent, beadSpec + ACs); await "done + anomalies"
6. scope check: staged/changed files trace to this bead (else bounce back once)
7. compose commit message; git add â€¦; git commit   (one bead = one commit)
8. emit commit event: per supervisor policy â†’ supervisor; append diff to the
   pending batches of tier agents whose relevance map includes this bead
9. statusOverrides update + bd close <bead-id>
10. if supervisor filed a blocking finding on a dependency of the next bead:
    dispatch the fix to the owning tier first (new commit), then continue
```

Trivial-tier beads MAY be implemented by the foreman itself when the tier table maps them to the foreman's own model â€” a degenerate tier requiring no dispatch round-trip (decision left to the tier table, not to the foreman's judgment).

### 5.3 Relevance map and feed construction (FR-3, D3)

Computed once at finalize, persisted with the schedule (FR-6). For each plan item `i` and each tier agent `t`, item `i`'s commit is **relevant** to `t` iff any not-yet-started bead assigned to `t`:

1. is reachable from `i` via `blocks` edges, or
2. appears in `i.metadata.foundationFor`, or
3. has `files_scope` overlapping `i`'s `files_scope` per `hasFileOverlap()` ([dag.ts:333](../../src/lib/vbrief/dag.ts)) â€” glob-aware, hotspot files excluded per PAN-1762 Â§A2.

Feed message shape (ingestion-only; exact wording fixed at planning):

```
[TIER FEED PAN-XXXX Â· commit 7/12 Â· ingestion only â€” do NOT respond]
Bead pan-xxxx-3 "â€¦titleâ€¦" completed (tier: light).
Relevant to your upcoming beads: pan-xxxx-7 (blocks), pan-xxxx-9 (files_scope: src/lib/cloister/router.ts).
--- git show <sha> ---
<diff>
```

Properties that fall out: an expert-tier agent on a plan whose expert beads sit atop 3 of 20 commits ingests exactly 3 diffs; during a 10-bead trivial run it ingests nothing; and the full feed for any agent is reconstructible from `git log` + the persisted map (D5/FR-5).

### 5.4 Supervisor tier (FR-4, D4)

- **Event source:** the foreman (it composes every commit â€” D9 â€” so it *is* the event bus; no file-watcher or git-hook machinery).
- **Subscription policy** (config, Â§5.1): `all` (pilot default â€” matches the evidence run), `flagged` (only `requiresInspection: true` beads â€” steady-state default), `sampled` (rate in config).
- **Per event:** supervisor receives diff + bead ACs (+ the PRD's relevant FR text when `metadata.traces` is present), spends its 500â€“2k tokens, and posts ack or a blocking finding to the inspect-status surface ([specialists.ts:399-401](../../src/dashboard/server/routes/specialists.ts)); the existing rules hold â€” inspect failures do not change tracker status ([specialists.ts:609-610](../../src/dashboard/server/routes/specialists.ts), search `inspect failures don't change Linear status`), and passing saves the checkpoint ([specialists.ts:501-505](../../src/dashboard/server/routes/specialists.ts)).
- **Relationship to `pan inspect`:** the pull-based gate remains for non-tiered execution; under tiered execution the supervisor subsumes it (a `requiresInspection` bead's commit is always a subscribed event, so the foreman never needs to call `pan inspect`). The convoy review pipeline at `pan done` is untouched (non-goal Â§8).

### 5.5 Replay (FR-5, D5, D11)

To replay agent `t`: respawn via `spawnRun` with the standard tier kickoff prompt; reconstruct `t`'s feed as the relevance-filtered subset of `git log <base>..HEAD` in plan order; deliver as one batched ingestion-only message (or a few, if size-capped); mark dispatch-ready. Replay triggers: (a) crash/orphan detection of a standing session; (b) context threshold at a tier-run boundary (D11). The supervisor replays the same way (its "relevance map" is its subscription policy).

### 5.6 Plan-time tier scheduling (FR-6, D6)

At finalize, after DAG validation: linearize waves (`groupItemsByWave()`), annotate each item with its tier (FR-1 table), and cut tier runs at tier-change boundaries. Emit the schedule artifact: ordered runs `[{tier, beadIds}]` + the relevance map. Scheduling failures (missing difficulty, unroutable difficulty) are finalize-time errors in the same family as the existing quality-lint checks ([quality-lint.ts:80-86](../../src/lib/vbrief/quality-lint.ts)). Standing agents are instantiated lazily: a tier's session spawns when its first run is â‰¤1 run away (not all at `pan start` â€” avoids paying for sessions a cancelled run would strand).

## 6. Economics & validation

### 6.1 Cost model

Per issue: `C_tiered = Î£_beads C_impl(tier(bead)) + C_foreman + C_feeds + C_supervisor`, vs `C_mono = C_impl(one model for everything)`.

- `C_feeds` per delivery = cache-read of the stable prefix (~0.1Ã— input price) + diff tokens in, ~0 out (ingestion-only). On cache miss (>5 min gap): full prefix at input price.
- `C_supervisor` = events Ã— 500â€“2k tokens (evidence: ~30 events â‰ˆ 15kâ€“60k tokens for a 29-commit issue).
- Evidence anchor: implementer $31.77 / 29 commits â‰ˆ **$1.10 per commit** (gpt-5.5/codex); supervision marginal ran 1â€“5% of that.

### 6.2 Cache-miss exposure

The 5-minute prompt-cache TTL is the main economic hazard: sparse-commit issues (long beads, slow tiers) make every per-commit delivery a cache miss. D10's batching bounds this for implementation tiers (one miss per batch); the supervisor's per-commit policy accepts misses as the price of catch latency.

### 6.3 Measurement method (pilot must implement, not estimate)

1. Stamp every `deliverAgentMessage` feed delivery with timestamp + token count; compute the warm-hit fraction (gap < 300s) per agent.
2. Use `pan token-spend-report` rows per registered agent id (foreman, each tier agent, supervisor) for per-role totals â€” this already attributes cost per agent.
3. NFR-1/NFR-3 verdicts come from these measurements.

### 6.4 Pilot recommendation

Run one issue class â€” **medium-difficulty bugs** â€” N issues tiered vs N monolithic (same model as the `standard` tier), and compare per-issue totals from `pan token-spend-report` plus wall-clock and defect-catch latency. Success: tiered â‰¤ monolithic cost with equal-or-better catch latency, NFR-1/NFR-3 targets met. Known measurement gap to fix first or work around: pi harness has no cost rows (gap noted on the issue, 2026-06-12).
