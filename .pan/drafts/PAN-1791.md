## 0. Glossary

Read this first. Every term of art used below is defined here.

| Term | Definition |
| --- | --- |
| **Bead** | One vBRIEF plan item materialized as a `bd` task. The unit of dispatch. |
| **One-bead-one-commit** | Enforced invariant: each bead's work lands as exactly one commit ([roles/work.md:58](../../roles/work.md), search `one bead = one commit`). Makes the commit stream a *semantic* event stream. |
| **Difficulty** | `metadata.difficulty` on a plan item: `trivial \| simple \| medium \| complex \| expert` ([src/lib/vbrief/types.ts:29](../../src/lib/vbrief/types.ts), search `export type VBriefDifficulty`; field at line 67). Present on every item emitted since PAN-1788. |
| **Tier** | A named (model, harness) pair plus the set of difficulties it serves. Defined in config (§5.1), resolved per spawn. |
| **Tier agent** | A **standing** (pre-instantiated, long-lived) implementation session bound to one tier for the lifetime of an issue. Receives bead specs + its filtered commit feed; returns implemented changes. Never runs bureaucracy. |
| **Foreman** | The issue's durable work agent, per PAN-1762 §4.B1 ([.pan/drafts/pan-1762.md](pan-1762.md)). In this design it additionally becomes the **router and event bus**: claims beads, dispatches to tier agents, composes commits, forwards feed messages. |
| **Router** | The deterministic dispatch rule inside the foreman loop: next ready bead's `metadata.difficulty` → tier table → tier agent. No model ever chooses who works. |
| **Relevance map** | Plan-derived table: for each tier agent, the set of beads whose commits it must see, computed from `blocks` edges, `metadata.foundationFor`, and `files_scope` overlap. Fixed at plan time, durable. |
| **Plan-filtered commit feed** | The stream of commit diffs the foreman forwards to a tier agent, filtered by the relevance map. The warm-up primitive that replaces per-bead re-onboarding. |
| **Ingestion-only message** | A feed delivery that instructs the recipient NOT to respond — context warming with ~0 output tokens. |
| **Supervisor tier** | A standing frontier-model session subscribed to commit events. Reviews diff vs the bead's ACs; acks or files a blocking finding; never implements. Push-based generalization of the Jidoka inspect gate. |
| **Jidoka inspect gate** | Today's pull-based mid-implementation review: a work agent runs `pan inspect <id> --bead <beadId>` after closing a `requiresInspection` bead ([src/cli/commands/inspect.ts:25](../../src/cli/commands/inspect.ts), search `Request inspection of a completed bead`); verdicts flow through the inspect specialist status ([src/dashboard/server/routes/specialists.ts:399-401](../../src/dashboard/server/routes/specialists.ts), search `update.inspectStatus`). |
| **Wave** | A set of plan items with no unmet dependencies among them, computed by `groupItemsByWave()` ([src/lib/vbrief/dag.ts:36](../../src/lib/vbrief/dag.ts)). |
| **Difficulty wave / tier run** | Consecutive beads in DAG order that route to the same tier. The plan-time scheduling unit (§5.6). |
| **Replay** | Crash/compaction recovery for a standing session: respawn, then re-deliver its filtered commit feed from git history + the relevance map. Deterministic because both inputs are durable. |
| **statusOverrides** | The only legal runtime status channel: a map in `<workspace>/.pan/continue.json`; specs on main are immutable after planning (PAN-1124). Written via `syncBeadStatusToVBrief` ([src/lib/vbrief/beads.ts:432](../../src/lib/vbrief/beads.ts), search `syncBeadStatusToVBriefPromise`). |
| **`deliverAgentMessage`** | The single orchestrator-to-agent delivery primitive: `deliverAgentMessage(agentId, message, caller?, deliveryMethod?)` ([src/lib/agents.ts:1371](../../src/lib/agents.ts), search `export async function deliverAgentMessage`). Tries PTY supervisor socket → legacy Channels → tmux; routes codex agents via `CodexRuntimeSync.sendMessage`. |

## 1. Background & motivation

### 1.1 Vision

Issues execute as a **tiered ensemble**: a cheap durable foreman runs the bead loop and all bureaucracy; standing tier agents (cheap → frontier) are pre-instantiated and kept warm by plan-filtered commit feeds, stepping in only when a bead matching their tier comes up; an event-driven supervisor reviews diffs at commit boundaries instead of holding working context. Expensive-model attention becomes an interrupt handler, not a participant. No other orchestrator routes *context* per tier the way the vBRIEF plan lets us — the plan knows, before work starts, which future work depends on which commits.

### 1.2 Live evidence (2026-06-12, PAN-1788/PAN-1792 runs)

The PAN-1788 implementation ran exactly this shape manually:

| Role | Session | Cost / tokens | Output |
| --- | --- | --- | --- |
| Implementer | gpt-5.5 / codex (conv 2774) | **$31.77** | 25 beads + 4 fix commits, ~84 min across 4 chunks, self-recovered from 2 branch-reset incidents |
| Parallel implementer | kimi-k2.6 / pi (conv 2775) | ↑82k ↓10k tokens, **99.5% cache hit** | 6 commits fixing 23 stale tests |
| Supervisor/orchestrator | Fable 5 (conv 2773) | $90.11 *total session* — supervision-specific marginal cost was **~30 commit-event wake-ups × 500–2k tokens** | every spec deviation inspected within minutes of its commit; 2 branch-reset incidents caught by the monitor before the implementer noticed |

Supervision marginal cost ran ~1–5% of implementer cost. The catch-rate for "executor misread the spec" is highest at commit boundaries — a misread caught at commit 1 is one bead of rework, not seven. Warm-session chunk continuation beat fresh handoffs (zero re-onboarding for chunks 2–4). The commit stream worked as a semantic event bus exactly because one-bead-one-commit held.

### 1.3 What the design must avoid

The failure mode of the naive version — N standing models all "listening" and deciding whether to step in — is a race with N-way duplicated cost. Every piece of autonomy in dispatch is removed by design: the bead's metadata decides (D1), the foreman serializes (D2), the plan filters the feeds (D3).

## 2. Decisions (made here; do not re-litigate during planning or implementation)

| # | Decision | Rationale |
| --- | --- | --- |
| D1 | **Router, not listeners.** A deterministic router reads `metadata.difficulty` on the next ready bead and assigns its tier from the tier table. Models never decide who works. | Multiple standing agents "deciding to step in" is a race. Operator's explicit design intent. Difficulty is already on every item ([src/lib/vbrief/types.ts:67](../../src/lib/vbrief/types.ts), search `difficulty?: VBriefDifficulty`). |
| D2 | **Foreman owns the loop and all bureaucracy.** `bd ready`/`bd update --claim`/`bd close`, `statusOverrides` updates, commit-message composition, tier dispatch, feed forwarding. Claiming primitive: `bd update <bead-id> --claim` ([roles/work.md:56](../../roles/work.md)). |
| D2a | **Scope split while PAN-1762 is open.** Until PAN-1762 lands the durable foreman, PAN-1791 may implement only foreman-independent planning/config artifacts: tier config validation, plan-time tier-run scheduling, relevance-map generation/persistence, and read-only inspection surfaces for those artifacts. Runtime loop changes (claim/dispatch/commit/close), standing tier session lifecycle, supervisor delivery, replay, and pilot measurement are blocked by PAN-1762. | Option B from 2026-06-30 kickoff. It preserves the "do not design a second loop" tenet while still delivering reusable substrate that PAN-1762's foreman can consume later. |
| D3 | **Plan-filtered commit feeds are the warm-up primitive.** Tier agents are pre-instantiated, not spawned per bead. The foreman forwards each commit's diff only to agents whose upcoming beads depend on it, per the relevance map (`blocks` edges, `foundationFor`, `files_scope` overlap). During trivial-bead runs, expensive tiers receive nothing — zero tokens. | The key differentiating idea: agents "only see the content of the commits that were relevant to the section of work they would be doing" (operator). | D4 | **Supervisor tier is event-driven, not pull-based.** It subscribes to commit events (policy: `all` \| `flagged` = `requiresInspection` beads \| `sampled`), reviews diff vs the bead's ACs, then acks or files a blocking finding. It never implements. Verdicts land on the existing inspect-status surface so the foreman's wait-for-verdict path is unchanged. | Generalizes the Jidoka gate from pull ([src/cli/commands/inspect.ts:25](../../src/cli/commands/inspect.ts)) to push; reuses the verdict plumbing at [specialists.ts:399-401](../../src/dashboard/server/routes/specialists.ts) and the checkpoint hook at [specialists.ts:501-505](../../src/dashboard/server/routes/specialists.ts) (search `onInspectComplete`). |
| D5 | **Replay = crash recovery.** A standing agent's curated context = (relevance map × git log), both durable. If a tier session dies, respawn and replay its filtered feed. No salvage path is designed or needed. | This is the property that makes standing sessions safe for the lifetime of an issue. |
| D6 | **Plan-time tier scheduling.** Group consecutive same-tier beads into tier runs; switch tiers only at genuine difficulty-wave boundaries. The vBRIEF DAG knows the difficulty sequence before work starts. | Avoids per-bead tier-switch thrash and re-onboarding. Waves come from `groupItemsByWave()` ([dag.ts:36](../../src/lib/vbrief/dag.ts)). |
| D7 | **Delivery via `deliverAgentMessage`.** Per-commit diff delivery uses the existing primitive ([agents.ts:1371](../../src/lib/agents.ts)); feed messages are ingestion-only (no response requested) so per-event output cost ≈ 0. No new delivery path. Known delivery gaps are fixed in their own issues: pi kickoff delivery (PAN-1793), codex status tracking (PAN-1789). | Single delivery primitive already handles PTY supervisor / channels / tmux / codex routing; inventing a parallel path violates no-bandaids. |
| D8 | **Fly remote slots are a compatible non-goal.** Standing tier agents could run as remote Fly slots (PAN-1676) with the foreman local. The initial design must not require it, and the routing/delivery layer must not preclude it (NFR-2). | Future optimization; architecture compatibility is cheap now, retrofit is not. |
| D9 | **Tier agents edit in the shared issue worktree; the foreman stages and commits.** The tier agent implements the bead with normal file edits and replies "done + anomalies"; the foreman checks `git status` (staged files must trace to the bead, per the scope-discipline rule), composes the commit message, commits, updates `statusOverrides`, runs `bd close`. Direct-commit by tier agents is rejected. | Only one implementation agent is active at a time (D1/D2 serialize), so there is no worktree race. Foreman-composes keeps the one-bead-one-commit invariant in ONE deterministic place instead of N tier prompts, and keeps commit bureaucracy out of tier contexts. (When PAN-1762 parallel slots later compose with tiers, slot worktrees follow PAN-1762's rules — see D12.) |
| D10 | **Feed batching default: per-commit to the supervisor; per-tier-run-boundary batch to implementation tiers.** The supervisor gets every subscribed commit immediately (catch latency is its purpose). Implementation tiers only need to be warm *by dispatch time*, so their relevant diffs are accumulated and delivered as one batched message when their tier run is next up (or when the batch exceeds a size threshold). | Per-commit supervisor delivery is what produced the minutes-level catch latency in the evidence. Batching implementation feeds amortizes the 5-minute prompt-cache TTL: one cache miss per batch instead of one per sparse commit. The cache-window coalescing refinement (deliver immediately iff last delivery <5 min ago) is a pilot measurement, not a v1 requirement (§6.3). |
| D11 | **Replay is also the context-compaction mechanism, triggered at tier-run boundaries.** Trigger: at a tier-run boundary, if the standing session's estimated context exceeds a configured fraction of its model's window (default **0.5**), kill and replay. Crash always triggers replay. Never compact mid-bead. | Replay at a boundary loses nothing (no in-flight bead) and the replayed feed is bounded by construction (filtered diffs only). Answers the context-rot open question with a deterministic rule. |
| D12 | **V1 executes beads serially through standing tiers.** Composition with PAN-1762's *parallel* slot dispatch (a wave fanning out to multiple slots, each slot's model drawn from the tier table) is explicitly deferred; the tier table is designed so slots can consume it later without change. | The live evidence validates the serial shape. Parallel-slot composition multiplies worktree and feed-ordering questions that PAN-1762 owns; coupling the two designs would block both. |

## 4. Requirements

### Functional

- **FR-1 — Deterministic difficulty routing.** The next ready bead's `metadata.difficulty` ([types.ts:67](../../src/lib/vbrief/types.ts)) maps through a configured tier table (§5.1) to exactly one named tier; the foreman dispatches to that tier's standing agent. Tier assignment involves no model judgment; given the same plan and config, routing is reproducible. Beads missing `difficulty` fail plan-time scheduling validation (they cannot reach the router).
- **FR-2 — Foreman owns all bureaucracy.** The foreman session runs `bd ready` / `bd update <bead-id> --claim` / `bd close`, writes `statusOverrides`, composes commit messages, stages and commits each bead's change (D9), and performs tier dispatch and feed forwarding. Tier agents perform only implementation; their prompts contain no lifecycle commands.
- **FR-3 — Standing tier agents with plan-filtered feeds.** Tier agents are pre-instantiated registered sessions (one per tier appearing in the plan's schedule). Feed filtering derives from the plan-time relevance map: `blocks` edges ([types.ts:22](../../src/lib/vbrief/types.ts)), `metadata.foundationFor` ([docs/VBRIEF.md:346](../../docs/VBRIEF.md)), and `files_scope` overlap via `hasFileOverlap()` ([dag.ts:333](../../src/lib/vbrief/dag.ts)). **A tier agent with no relevant upcoming beads receives zero deliveries — zero tokens — during unrelated runs.** Tiers that never appear in the schedule are never instantiated.
- **FR-4 — Event-driven supervisor.** A standing supervisor session subscribes to commit events under a configured policy (`all` | `flagged` | `sampled`). For each subscribed commit it receives the diff plus the bead's ACs, and responds with an ack or a blocking finding on the existing inspect-status surface ([specialists.ts:399-401](../../src/dashboard/server/routes/specialists.ts)). It never implements. A blocking finding halts dispatch of dependent beads until the foreman lands a fix commit (mirroring today's `INSPECTION BLOCKED` flow).
- **FR-5 — Replay-based crash recovery.** If any standing session (tier agent or supervisor) dies, the foreman respawns it and replays its filtered commit feed reconstructed from git history + the relevance map. No salvage of the dead session's context is attempted. The replayed agent reaches dispatch-ready state without human intervention.
- **FR-6 — Plan-time tier scheduling.** At finalize, the scheduler computes the tier-run sequence from the DAG (`groupItemsByWave()`, [dag.ts:36](../../src/lib/vbrief/dag.ts)) and each item's difficulty: consecutive same-tier beads form one run; tier switches occur only at run boundaries. The schedule (tier runs + relevance map) is persisted as a plan artifact so the foreman and replay consume it without recomputation.
- **FR-7 — Delivery via the existing primitive.** All feed and dispatch messages to standing sessions go through `deliverAgentMessage` ([agents.ts:1371](../../src/lib/agents.ts)). Feed messages are marked ingestion-only (the message text instructs: do not respond). No new transport is introduced.
- **FR-8 — Tier table in config.** A `tiered_execution` config section defines tiers as (model, harness, difficulties served) plus the supervisor policy, generalizing today's claude-only `complexity_routing` ([src/lib/cloister/config.ts:138](../../src/lib/cloister/config.ts), search `complexity_routing`). Harness values resolve through the existing specialist-harness machinery ([src/lib/cloister/router.ts:187](../../src/lib/cloister/router.ts), search `export function getSpecialistHarness`), migrating to PAN-1787's unified `resolveHarness()` when that lands. Feature is off by default (`enabled: false`).

### Foreman-independent slice for this plan revision

PAN-1791's immediately executable scope is limited to FR-1, FR-3's **plan-time relevance-map computation**, FR-6, and FR-8. FR-2 and the runtime portions of FR-3 through FR-7 are blocked by PAN-1762. Implementers must not add a parallel bead loop, status writer, commit runner, supervisor dispatch loop, or standing-session runtime in PAN-1791.

### Non-functional

- **NFR-1 — Near-zero per-event ingestion cost.** A feed delivery to a standing session should cost ≈ (cached-prefix read + diff tokens) with ~0 output. **Measurement target for the pilot:** ≥90% of deliveries during an active tier run hit warm prompt cache (delivery-to-delivery gap < 5 min), and mean marginal cost per delivery ≤ 2k tokens. Measured per §6.3.
- **NFR-2 — Fly-slot compatibility.** Routing (FR-1), scheduling (FR-6), and delivery (FR-7) must work unchanged whether a tier agent is local or a remote Fly slot (PAN-1676). Concretely: nothing in the design may assume the tier agent shares the foreman's machine *except* the worktree-edit path in D9, which is the one acknowledged remote-mode delta (remote tiers would need the slot-branch model — deferred with D8/D12).
- **NFR-3 — Supervision cost ceiling.** Supervisor marginal cost per issue ≤ 10% of implementer cost under the `flagged` policy (evidence baseline: 1–5% under per-commit supervision of all commits). The pilot reports the measured ratio.

## 5. Architecture

### 5.1 Tier table (FR-1, FR-8)

Today's difficulty→model routing is claude-only and per-spawn:

**Before** ([src/lib/cloister/config.ts:136-144](../../src/lib/cloister/config.ts), search `complexity_routing`):

```ts
export interface ModelSelectionConfig {
  default_model: 'opus' | 'sonnet' | 'haiku';
  complexity_routing: {
    trivial: 'opus' | 'sonnet' | 'haiku';
    simple: 'opus' | 'sonnet' | 'haiku';
    medium: 'opus' | 'sonnet' | 'haiku';
    complex: 'opus' | 'sonnet' | 'haiku';
    expert: 'opus' | 'sonnet' | 'haiku';
  };
```

**After** (config sketch — exact schema fixed at planning):

```yaml
tiered_execution:
  enabled: false                # off by default
  tiers:
    light:    { model: haiku,   harness: claude-code, difficulties: [trivial, simple] }
    standard: { model: gpt-5.5, harness: codex,       difficulties: [medium, complex] }
    frontier: { model: opus,    harness: claude-code, difficulties: [expert] }
  supervisor:
    model: opus
    harness: claude-code
    subscribe: flagged          # all | flagged | sampled
  replay_threshold: 0.5         # fraction of model context window (D11)
```

Validation rules: every difficulty maps to exactly one tier (total, non-overlapping); unknown harness/model combinations rejected at config load; the pi+Anthropic+subscription ToS gate (`src/lib/harness-policy.ts`) applies to tier definitions as to any spawn.

### 5.2 Foreman loop (FR-2, D2, D9)

The foreman is the PAN-1762 work-agent foreman with two added responsibilities: routing and feed forwarding. The per-bead workflow changes shape:

**Before** (today's single-agent loop, [roles/work.md:53-65](../../roles/work.md), abbreviated):

```
1. bd ready — next bead
2. bd update <bead-id> --claim
3. implement the bead yourself
4. git add … && git commit       (one bead = one commit)
5. bd close <bead-id>
6. if requiresInspection: pan inspect … ; wait for verdict
```

**After** (foreman loop — design sketch):

```
1. bd ready — next bead
2. router: bead.metadata.difficulty → tier (deterministic table lookup)
3. flush pending feed batch to that tier agent if its tier run is starting (D10)
4. bd update <bead-id> --claim
5. dispatch: deliverAgentMessage(tierAgent, beadSpec + ACs); await "done + anomalies"
6. scope check: staged/changed files trace to this bead (else bounce back once)
7. compose commit message; git add …; git commit   (one bead = one commit)
8. emit commit event: per supervisor policy → supervisor; append diff to the
   pending batches of tier agents whose relevance map includes this bead
9. statusOverrides update + bd close <bead-id>
10. if supervisor filed a blocking finding on a dependency of the next bead:
    dispatch the fix to the owning tier first (new commit), then continue
```

Trivial-tier beads MAY be implemented by the foreman itself when the tier table maps them to the foreman's own model — a degenerate tier requiring no dispatch round-trip (decision left to the tier table, not to the foreman's judgment).

### 5.3 Relevance map and feed construction (FR-3, D3)

Computed once at finalize, persisted with the schedule (FR-6). For each plan item `i` and each tier agent `t`, item `i`'s commit is **relevant** to `t` iff any not-yet-started bead assigned to `t`:

1. is reachable from `i` via `blocks` edges, or
2. appears in `i.metadata.foundationFor`, or
3. has `files_scope` overlapping `i`'s `files_scope` per `hasFileOverlap()` ([dag.ts:333](../../src/lib/vbrief/dag.ts)) — glob-aware, hotspot files excluded per PAN-1762 §A2.

Feed message shape (ingestion-only; exact wording fixed at planning):

```
[TIER FEED PAN-XXXX · commit 7/12 · ingestion only — do NOT respond]
Bead pan-xxxx-3 "…title…" completed (tier: light).
Relevant to your upcoming beads: pan-xxxx-7 (blocks), pan-xxxx-9 (files_scope: src/lib/cloister/router.ts).
--- git show <sha> ---
<diff>
```

Properties that fall out: an expert-tier agent on a plan whose expert beads sit atop 3 of 20 commits ingests exactly 3 diffs; during a 10-bead trivial run it ingests nothing; and the full feed for any agent is reconstructible from `git log` + the persisted map (D5/FR-5).

### 5.4 Supervisor tier (FR-4, D4)

- **Event source:** the foreman (it composes every commit — D9 — so it *is* the event bus; no file-watcher or git-hook machinery).
- **Subscription policy** (config, §5.1): `all` (pilot default — matches the evidence run), `flagged` (only `requiresInspection: true` beads — steady-state default), `sampled` (rate in config).
- **Per event:** supervisor receives diff + bead ACs (+ the PRD's relevant FR text when `metadata.traces` is present), spends its 500–2k tokens, and posts ack or a blocking finding to the inspect-status surface ([specialists.ts:399-401](../../src/dashboard/server/routes/specialists.ts)); the existing rules hold — inspect failures do not change tracker status ([specialists.ts:609-610](../../src/dashboard/server/routes/specialists.ts), search `inspect failures don't change Linear status`), and passing saves the checkpoint ([specialists.ts:501-505](../../src/dashboard/server/routes/specialists.ts)).
- **Relationship to `pan inspect`:** the pull-based gate remains for non-tiered execution; under tiered execution the supervisor subsumes it (a `requiresInspection` bead's commit is always a subscribed event, so the foreman never needs to call `pan inspect`). The convoy review pipeline at `pan done` is untouched (non-goal §8).

### 5.5 Replay (FR-5, D5, D11)

To replay agent `t`: respawn via `spawnRun` with the standard tier kickoff prompt; reconstruct `t`'s feed as the relevance-filtered subset of `git log <base>..HEAD` in plan order; deliver as one batched ingestion-only message (or a few, if size-capped); mark dispatch-ready. Replay triggers: (a) crash/orphan detection of a standing session; (b) context threshold at a tier-run boundary (D11). The supervisor replays the same way (its "relevance map" is its subscription policy).

### 5.6 Plan-time tier scheduling (FR-6, D6)

At finalize, after DAG validation: linearize waves (`groupItemsByWave()`), annotate each item with its tier (FR-1 table), and cut tier runs at tier-change boundaries. Emit the schedule artifact: ordered runs `[{tier, beadIds}]` + the relevance map. Scheduling failures (missing difficulty, unroutable difficulty) are finalize-time errors in the same family as the existing quality-lint checks ([quality-lint.ts:80-86](../../src/lib/vbrief/quality-lint.ts)). Standing agents are instantiated lazily: a tier's session spawns when its first run is ≤1 run away (not all at `pan start` — avoids paying for sessions a cancelled run would strand).

## 6. Economics & validation

### 6.1 Cost model

Per issue: `C_tiered = Σ_beads C_impl(tier(bead)) + C_foreman + C_feeds + C_supervisor`, vs `C_mono = C_impl(one model for everything)`.

- `C_feeds` per delivery = cache-read of the stable prefix (~0.1× input price) + diff tokens in, ~0 out (ingestion-only). On cache miss (>5 min gap): full prefix at input price.
- `C_supervisor` = events × 500–2k tokens (evidence: ~30 events ≈ 15k–60k tokens for a 29-commit issue).
- Evidence anchor: implementer $31.77 / 29 commits ≈ **$1.10 per commit** (gpt-5.5/codex); supervision marginal ran 1–5% of that.

### 6.2 Cache-miss exposure

The 5-minute prompt-cache TTL is the main economic hazard: sparse-commit issues (long beads, slow tiers) make every per-commit delivery a cache miss. D10's batching bounds this for implementation tiers (one miss per batch); the supervisor's per-commit policy accepts misses as the price of catch latency.

### 6.3 Measurement method (pilot must implement, not estimate)

1. Stamp every `deliverAgentMessage` feed delivery with timestamp + token count; compute the warm-hit fraction (gap < 300s) per agent.
2. Use `pan token-spend-report` rows per registered agent id (foreman, each tier agent, supervisor) for per-role totals — this already attributes cost per agent.
3. NFR-1/NFR-3 verdicts come from these measurements.

### 6.4 Pilot recommendation

Run one issue class — **medium-difficulty bugs** — N issues tiered vs N monolithic (same model as the `standard` tier), and compare per-issue totals from `pan token-spend-report` plus wall-clock and defect-catch latency. Success: tiered ≤ monolithic cost with equal-or-better catch latency, NFR-1/NFR-3 targets met. Known measurement gap to fix first or work around: pi harness has no cost rows (gap noted on the issue, 2026-06-12).

## 7. Open questions (genuinely unresolved — everything else is decided in §2)

1. **Cache-window coalescing** (refines D10): deliver per-commit to implementation tiers when the last delivery was <5 min ago, batch otherwise. Worth the complexity only if the pilot shows batching materially delays warm-up. Decision deferred to pilot data.
2. **Sampling policy details** for `subscribe: sampled` (rate, stratification by difficulty). Not needed for the pilot (`all`) or steady state (`flagged`); specify only if/when sampling is wanted.
3. **Parallel-slot composition** (D12): when PAN-1762 Path B lands, how tier-table model assignment and feeds apply to concurrently-running slots (feed ordering across parallel commits is the hard part). Owned jointly with PAN-1762 at that time; nothing in v1 may preclude it.

## 8. Non-goals

- **Multiple models autonomously deciding to intervene.** Race-prone; the router decides (D1).
- **In-session model switching.** The PAN-1787 fresh-session invariant stands; tiers are separate standing sessions ([.pan/drafts/PAN-1787.md](PAN-1787.md), search `resolveHarness`).
- **Replacing the convoy review pipeline.** The supervisor tier complements `pan done` → convoy review; it does not replace it.
- **Remote Fly tier agents in v1** (D8/NFR-2): architecture-compatible, not delivered here.
- **Rebuilding any deleted swarm runtime machinery** (PAN-1762 §B4 list applies here too).
