# PAN-2172 — host-mode fresh respawn never delivers kickoff

**Issue:** [PAN-2172](https://github.com/eltmon/overdeck/issues/2172)
**Mode:** `pan plan --auto` (non-interactive)

## Glossary

- **Kickoff / initial prompt** — the first instruction (work-agent role brief + vBRIEF plan + decisions/hazards + beads context) delivered to a freshly spawned work agent. Written to `<agentDir>/initial-prompt.md` at spawn (`src/lib/agents.ts:3131`).
- **Host mode (`--host`)** — `pan start <id> --host` runs the work agent directly on the host (primary repo worktree) instead of inside a Docker workspace container. The documented fallback when the Docker workspace stack is unhealthy (currently common — PAN-2170).
- **PTY supervisor** — the default orchestrator→agent delivery transport: Claude is wrapped as `node dist/pty-supervisor.js claude …`, listens on `${OVERDECK_HOME}/sockets/pty-<id>.sock`, and writes delivered messages into Claude's PTY. Eligible for host-mode claude-code work agents; Docker workspaces are excluded (`decideSupervisorForWorkAgent`, `src/lib/agents.ts:1706`).
- **`deliverAgentMessage`** — the single delivery primitive (`src/lib/agents/delivery.ts:176`). Tries supervisor → channels → tmux paste-buffer fallback in `auto` mode.
- **`deliverInitialPromptWithRetry`** — kickoff-specific wrapper (`delivery.ts:345`): waits for prompt-ready, then delivers, up to 2 attempts.
- **`kickoffDelivered`** — per-agent state flag (`AgentState`, `src/lib/agents.ts:1081`). Set `false` at spawn (`agents.ts:3136`), `true` once delivery returns `ok` (`agents.ts:3281`). Persisted to state.json and the agents DB.
- **Deacon** — the lifecycle watchdog. `runPatrol()` (`src/lib/cloister/deacon.ts:2571`) runs nudge/recovery patrols each cycle.
- **Transcript user-record landing** — the confirmation mechanism already used for *resumes*: after delivering, poll the workspace JSONL transcript for a new `user` record to prove the message actually reached the agent (`waitForTranscriptUserRecordLanding`, `delivery.ts:288`; `deliverResumeMessageWithTranscriptConfirmation`, `delivery.ts:308`).

## Problem

`pan start pan-2086 --fresh --host --yes` reported success, spawned a live tmux session, and printed `Beads: ✓ 9 tasks` — but the work agent never received its kickoff. It sat at `ctx 0% (0/200k)`, `out 0`, `$0.0000` for ~45 minutes at an empty `❯` prompt. The operator/flywheel believed the agent was working; it was inert, burning a slot, and the flywheel role cannot use `pan tell` to recover it.

### Verified root causes (one per acceptance criterion)

**AC1 — first delivery can silently no-op.** `deliverInitialPromptWithRetry` (`delivery.ts:345`) trusts `deliverAgentMessage`'s `ok:true`. But the tmux fallback returns `ok:true` even when paste-verification failed (`src/lib/tmux.ts:1185-1191` — "Sending Enter anyway to avoid orphaned input"), and the PTY supervisor's echo-confirm can fail on large briefs (the ~50KB kickoff trips it; documented in PAN-2102, `deacon-auto-resume.ts:485`). Unlike resumes, the initial kickoff has **no post-delivery confirmation** that the message actually landed in the transcript. So `kickoffDelivered` can be set `true` when nothing landed.

**AC2 — silent success.** `src/cli/commands/start.ts:1373` calls `spinner.succeed(...)` unconditionally and never inspects `agent.kickoffDelivered`. `spawnAgent` (`src/lib/agents.ts:3284-3294`) records the failure but `return state` silently for non-`strike` roles. So `pan start` exits `0` and prints success even when the work agent never received its kickoff. (Strike agents already throw + stop; work agents do not — and must not, because the live session is recoverable.)

**AC3 — no deacon re-delivery for a live-but-inert fresh start.** A running work agent with `kickoffDelivered === false` and a *live* tmux session is invisible to every existing patrol:
- `isPreKickoffLaunchDeath` (`deacon-auto-resume.ts:107`) only accumulates a failure when the **tmux session is gone**. Here the session is alive, so the heartbeat-dead path returns early (`deacon-auto-resume.ts:160`).
- `nudgeStalledResumeWorkAgents` (`deacon-auto-resume.ts:397`) already knows how to re-deliver an undelivered kickoff (`buildStalledResumePrompt` reads `initial-prompt.md` when `kickoffDelivered === false`, `deacon-auto-resume.ts:384`) — but it is gated by `if (!state.lastResumeAt) continue;` (`deacon-auto-resume.ts:405`). A **fresh** start has no `lastResumeAt`, so it is skipped.
- `nudgeIdleWorkAgentsWithOpenBeads` (`deacon-auto-resume.ts:434`) is gated by `isAgentIdleForNudge` (`deacon-auto-resume.ts:452`), which returns **false** when `runtime.json` is absent (`src/lib/cloister/agent-idle.ts:9-11`). A never-kicked agent never fired any hook, so `runtime.json` does not exist — it is skipped too.

## Proposal

Three independent, additive fixes — one per AC. None refactors the working resume-nudge path; none restores the Docker path (out of scope, PAN-2170).

### Work item 1 (AC1): confirm initial-kickoff landing via transcript, then report truthfully

**Files:** `src/lib/agents/delivery.ts` (`deliverInitialPromptWithRetry`).

After a `deliverAgentMessage` returns `ok`, confirm a **new user record landed** in the workspace transcript before treating the attempt as a success — reusing the exact mechanism resumes already use (`captureTranscriptUserRecordSnapshot` + `waitForTranscriptUserRecordLanding`, both already in this file). On no-landing, the existing 2-attempt loop retries; if still unconfirmed, return `{ ok: false, failure: 'kickoff-not-confirmed' }`.

**Guard (no regression):** confirmation runs only when the agent's `sessionId` and `workspace` are known *and* the harness writes a confirmable transcript (claude-code / pi). For codex / ohmypi (file-pointer kickoff via `.pan/kickoff.md`, and ohmypi's `writeOhmypiAgentPrompt` path which does not go through this function) confirmation is skipped — current behavior preserved.

Snapshot the transcript **before** delivery (mirroring `deliverResumeMessageWithTranscriptConfirmation`) so the "new record since" comparison is correct.

Before:
```ts
const result = await deliverAgentMessage(agentId, deliveredPrompt, caller, deliveryMethod);
if (result.ok) return result;
lastFailure = result.failure ?? `delivery returned ok=false via ${result.path}`;
```
After (shape):
```ts
const result = await deliverAgentMessage(agentId, deliveredPrompt, caller, deliveryMethod);
if (result.ok) {
  const confirmed = await confirmKickoffLanded(state, before); // best-effort; true when not confirmable
  if (confirmed) return result;
  lastFailure = 'kickoff-not-confirmed';
} else {
  lastFailure = result.failure ?? `delivery returned ok=false via ${result.path}`;
}
```

### Work item 2 (AC2): `pan start` fails loudly when the kickoff was not delivered

**Files:** `src/cli/commands/start.ts` (work-spawn branch, ~:1362-1428).

After `spawnAgent` returns, if `agent.kickoffDelivered === false` (work role): replace `spinner.succeed(...)` with `spinner.fail(...)`, print a complete operator message, and set `process.exitCode = 1`. Do **not** kill the session — it is preserved so the deacon (work item 3) can re-deliver. The message states plainly: the session spawned but the kickoff did not land; the agent is idle; the deacon will retry within the stuck threshold, or run `pan tell <id> "<brief pointer>"`.

The tracker/vBRIEF side effects already executed inside `spawnAgent`; this bead changes only the success-vs-failure reporting and exit code, not the spawn.

### Work item 3 (AC3): deacon re-delivers undelivered kickoffs to live sessions

**Files:** `src/lib/cloister/deacon-auto-resume.ts` (new `redeliverUndeliveredKickoffs()`), `src/lib/cloister/deacon.ts` (`runPatrol` wiring + re-export).

New exported patrol, modeled on the existing nudge patrols but **not** gated on `lastResumeAt` or `isAgentIdleForNudge`. For each running `work` agent that is not paused/troubled, whose issue is open, whose tmux session is alive, with `kickoffDelivered === false`, and whose age since `startedAt` exceeds `KICKOFF_REDELIVERY_THRESHOLD_MS` (default 3 min — safely past the normal ready+delivery window so it never fires while the first delivery is still in flight), with a per-agent cooldown file (`.last-kickoff-redelivery`):

1. Re-read the brief from `<agentDir>/initial-prompt.md`.
2. Re-deliver via `deliverInitialPromptWithRetry` (which now confirms landing — work item 1).
3. On confirmed delivery, set `kickoffDelivered = true` and save state.
4. After a bounded number of consecutive failed re-deliveries, **surface loudly** — emit an activity error + TTS and mark the agent `troubled` — so it is never left silently inert.

Wire it into `runPatrol()` (`deacon.ts:2679`-ish, beside `nudgeStalledResumeWorkAgents`) and re-export from `deacon.ts:143`.

## Planning Q&A

Auto mode — no interactive questions. Inferred decisions recorded in `plan.autoDecisions[]` of the vBRIEF and in `continue.json` decisions. Key calls:

- **New deacon patrol vs. extend `nudgeStalledResumeWorkAgents`** → new patrol. Rationale: the resume nudge is resume-specific (two gates assume a prior resume); bolting fresh-start kickoff handling onto it would muddy a working path. A dedicated, additive patrol is the no-loss choice and keeps changes surgical.
- **AC2 preserves the session (no kill)** → yes. Rationale: the live session is recoverable; killing it would defeat AC3. Loud report + non-zero exit satisfies "no silent drop" without destroying recoverable state.
- **Transcript confirmation is best-effort, claude-code/pi only** → yes. Rationale: codex/ohmypi use file-pointer delivery; forcing transcript confirmation there would regress their kickoff. Guard on harness + sessionId/workspace presence.
- **Stuck threshold = 3 min; re-delivery cooldown = 10 min** → defensible defaults matching neighboring patrols (`STALLED_RESUME_NUDGE_COOLDOWN_MS = 10 min`).

## Requirements

- **FR-1** — `pan start <id> --fresh --host --yes` delivers the kickoff to the new session and the agent begins its first bead within the normal startup window.
- **FR-2** — when kickoff delivery is not confirmed, `pan start` reports failure (non-zero exit, `spinner.fail`, clear message) instead of printing success.
- **FR-3** — a running host-mode work agent with `kickoffDelivered === false` and a live session is re-delivered by the deacon within the stuck threshold, or surfaced loudly after repeated failure.
- **NFR-1** — no regression to codex/ohmypi kickoff, to resume delivery, or to the Docker-excluded supervisor decision.
- **NFR-2** — deacon code is server-reachable: async tmux primitives only, no `execSync`; delay-based tests use fake timers.

## Acceptance criteria → see vBRIEF items (1:1 mapping).

## NonGoals

- Restoring or fixing the Docker workspace path (PAN-2170) — host mode is the fallback this issue makes reliable.
- Adding PTY-supervisor wiring to Docker workspaces.
- Changing ctx-saturation detection (PAN-2169).
- Refactoring `nudgeStalledResumeWorkAgents` or `nudgeIdleWorkAgentsWithOpenBeads`.
- Auto-killing, deep-wiping, or respawning inert agents — AC3 re-delivers or surfaces; it never destroys recoverable state.
