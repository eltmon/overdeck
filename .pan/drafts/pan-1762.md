# PRD: Swarm v2 — Tracer-Bullet Planning Contract + Foreman-Driven Intra-Issue Swarms

- **Issue:** PAN-1762
- **Status:** Draft
- **Date:** 2026-06-11
- **Author:** Claude (from design conversation with Ed, 2026-06-10/11)

## 1. Background

### 1.1 What we had and deleted

PAN-977 built an intra-issue swarm: each vBRIEF item got its own worktree, a
slot branch (`feature/<issue>/slot-<n>`), a tmux session, and a registered
agent. A server-side runtime (2,598-line `routes/swarm.ts`) drove dispatch: an
auto-advance poll loop watched the DAG, a slot-merged HTTP callback fired on
merges, `SwarmRuntime`/`SynthesisOutput` state persisted in continue files,
and a deacon patrol detected merged slots.

PAN-1517 deleted all of it (~6,021 lines, commits `59f1e6cd4` +
`f128f41b2`) and replaced it with in-context subagent fan-out
(`roles/work.md` "Parallel work via subagents"): the single work agent spawns
harness subagents that share its filesystem and return text.

What survives on main:

- `src/lib/vbrief/dag.ts` — `groupItemsByWave()` (Kahn's algorithm),
  `hasFileOverlap()` (glob-aware, tested, currently **zero production
  callers**), critical-path analysis.
- `VBriefItem.metadata.files_scope?: string[]` in
  `src/lib/vbrief/types.ts` — never populated by the planner.
- `docs/SWARM.md` and the dispatch diagrams (describe the deleted system).
- `src/cli/commands/kill.ts` still recognizes `agent-<id>-<n>` slot naming.

### 1.2 What the replacement lost

In-context subagents cover the cheap 80% (mechanical parallel subtasks) but
cannot provide:

1. **Durability.** Subagents die with the parent session. A crash, context
   deadlock, or compaction three hours into a fan-out loses invisible work.
   Slots were registered agents — own state file, dashboard-visible,
   deacon-recoverable.
2. **Distribution.** Subagents share the parent's filesystem; they can never
   leave the machine. Slots had per-item worktrees and branches, so they
   could run anywhere — including Fly VMs (the per-issue remote machinery
   from PAN-1676 already exists).
3. **Continuous integration.** Slots merged into the feature branch as the
   DAG advanced, with synthesis at convergence points. Subagents hand
   everything to one parent who integrates in one context at the end.

### 1.3 What Deft Directive taught us

Directive's swarm (skills `deft-directive-gh-slice`, `deft-directive-decompose`,
`deft-directive-swarm`; `scripts/swarm_readiness.py`) parallelizes **stories
within an epic** — the same granularity as our flywheel parallelizes issues.
Its strength is not runtime supervision (ours was richer) but the **contract
before dispatch**:

1. Enforced **tracer-bullet decomposition** — thin vertical slices cutting
   end-to-end through all integration layers, never horizontal layer-slices.
2. Mandatory swarm metadata per story: `file_scope`,
   `file_scope_confidence`, `conflict_group`, `parallel_safe`, `depends_on`,
   `verify_commands`.
3. A machine **pre-flight readiness audit**: any two ready stories sharing a
   file without a dependency relationship are flagged; ill-formed cohorts
   don't dispatch.
4. **Per-story verification** (`verify_commands`) before a story's work
   counts.

Our deleted swarm had none of these. Slot merges into the feature branch ran
**zero quality gates** (verified against the deleted route source); the first
gate was per-issue, at the very end.

### 1.4 Why we tolerate overlap where Directive refuses

Directive's overlap gate is binary at the file level — one shared file trips
it the same as eighty. But git conflicts are **hunk-level**: two branches
touching the same file in different regions merge clean in a 3-way merge.
File-level overlap therefore over-predicts textual conflicts, and predicts
nothing about semantic conflicts (zero shared files can still break
together). Overdeck has what Directive lacks: the merge train
(PAN-1691) and UAT generations (PAN-1737) resolve real conflicts at
integration time, hold out unresolvables, and let the operator UAT the exact
combined tree before it lands on main.

**Therefore: overlap prediction is an ordering signal, never a refusal.**
Prevention reduces conflict frequency; the train is the hard guarantee.

### 1.5 Why now

Wall-clock latency of single large issues is becoming a real constraint
(several big issues upcoming). The flywheel parallelizes *across* issues but
cannot make any single issue finish faster. Path A makes big issues rarer by
splitting them at planning; Path B handles the genuinely unsplittable ones.

## 2. Design principles

1. **Contract before dispatch.** Swarm-eligible plans must declare per-item
   `files_scope` (+ confidence) and `verify_commands`. A readiness lint
   refuses ill-formed cohorts at `pan plan finalize`.
2. **Foreman, not external runtime.** The issue's durable work agent drives
   waves itself. No server poll loop, no slot-merged HTTP callback, no swarm
   route, no `SwarmRuntime` persistence. The old route was 2,598 lines
   *because* the runtime lived outside the agent; moving wave-advance into
   the foreman deletes the entire category.
3. **Verify-then-merge per slot.** Each item's `verify_commands` must pass
   before its slot branch merges into the feature branch.
4. **Overlap = ordering signal.** `hasFileOverlap()` serializes colliding
   items within a wave; everything else runs parallel. Predicted overlap
   never blocks dispatch outright.
5. **Reuse existing primitives.** `dag.ts` waves + overlap (already on
   main), `spawnRun` registered agents, harness worktree-isolated subagents,
   PAN-1676 Fly remote machinery, the existing dashboard resource-discovery
   (registered slot agents appear under their issue with no new UI).

## 3. Path A — Planning discipline (Directive's contract, our pipeline)

### A1. Tracer-bullet slicing mandate (prompt-only)

Update `roles/plan.md` and the planning skills (`pan-plan`, `write-vbrief`):

- Items MUST be vertical slices — each independently verifiable end-to-end —
  never layer-slices ("all the DB work" / "all the UI work").
- Every item MUST declare `metadata.files_scope` (globs allowed, per
  `hasFileOverlap()` semantics) plus a confidence marker
  (`files_scope_confidence: high|medium|low`). Both fit the existing
  `metadata` index signature — **no schema change**.
- Every item SHOULD declare `verify_commands` (commands that prove the slice
  works on its own).
- **Prefer splitting**: when a big issue decomposes into independently
  shippable slices, the planner proposes sibling issues instead of one mega
  vBRIEF. Sibling issues parallelize through the flywheel today (local +
  Fly per issue) with full per-unit review/test/UAT — strictly more rigor
  than any intra-issue mechanism.

### A2. Readiness lint at `pan plan finalize`

The Overdeck analogue of Directive's `swarm_readiness.py`, run at the
equivalent lifecycle point:

- Warn on items missing `files_scope`.
- Warn when two edge-unconnected sibling items declare overlapping scopes —
  planner must add an edge, merge the items, or explicitly accept
  serialization.
- Per-project **hotspot list** (e.g. `CHANGELOG.md`, `bun.lock`, generated
  files) configured in `projects.yaml`; hotspot files are excluded from
  overlap math (every feature touches them; they'd serialize everything)
  and owned by merge-train resolution conventions instead.
- Lint is advisory (warn) initially; `--strict` flag available for blocking.

### A3. Cross-issue soft signals

- **Issue footprint** = union of item `files_scope` from the spec on main.
- Flywheel launch ordering avoids co-starting issues with heavily
  overlapping predicted footprints (soft preference, not a gate).
- Merge queue / UAT train surfaces **predicted conflicts** for in-flight
  branches (same pairwise math as `flywheel-merge-order.ts` but on declared
  scopes), converging to actual `changedFilesVsMain` sets once branches
  exist.

### A4. Scope-drift feedback loop (improves on Directive — they have no runtime enforcement)

At `pan done` / the verification gate: diff
`git diff --name-only origin/main...HEAD` against the declared scope union;
record drift (files touched outside scope, scope files untouched) in the
workspace continue state and review status. Drift surfaces in review and
calibrates how much to trust that plan's predictions downstream.

## 4. Path B — Foreman-driven intra-issue swarm

For issues that are big but **unsplittable into independently shippable
slices** (deep refactors with one coherent deliverable, migrations whose
intermediate states can't merge), where wall-clock matters.

### B1. The foreman

The issue's existing durable work agent becomes the foreman. It:

1. Reads the spec's DAG and computes waves via `groupItemsByWave()`.
2. For each wave, partitions items by `hasFileOverlap()`: non-overlapping
   items dispatch in parallel; overlapping items serialize within the wave.
3. Dispatches each item via one of two tiers (B2).
4. On item completion, runs the item's `verify_commands`; only on green does
   the slot branch merge into the feature branch (B3).
5. At DAG convergence points (item with >1 blocking parents), composes a
   **synthesis digest** of what the merged parents actually did, inlined
   into the convergent item's dispatch prompt. In-context — no
   `SynthesisOutput` runtime state resurrected.
6. Owns failure handling: a failed slot gets one redo with feedback, then
   serial fallback in the foreman's own context (mirrors the existing
   subagent failure protocol in `roles/work.md`).

Wave advance is just the foreman's loop reaching the next iteration. Nothing
external advances anything.

### B2. Two dispatch tiers

| Tier | Mechanism | Use for | Properties |
| --- | --- | --- | --- |
| **In-context** | Harness `Agent` tool with worktree isolation | Cheap/mechanical items (< ~15 min) | Zero Overdeck machinery; dies with foreman; local only |
| **Registered slot** | `spawnRun` per-item agent in a per-item worktree on `feature/<issue>/slot-<n>` | Heavy items; anything to distribute | Durable state file; dashboard-visible via existing resource discovery; deacon-recoverable; **Fly-distributable** |

The foreman chooses the tier per item (size, model routing via Cloister,
local capacity). `kill.ts` already recognizes the `agent-<id>-<n>` naming.

### B3. Verify-then-merge

A slot branch merges into the feature branch only after:

1. The item's `verify_commands` pass in the slot worktree, and
2. A clean (or foreman-resolved) merge against the current feature branch.

This fixes the old swarm's biggest gap — slots merged with zero gates — and
keeps the feature branch continuously green, so a bad item is caught at item
granularity rather than at end-of-issue verification.

### B4. What is deliberately NOT rebuilt

- `routes/swarm.ts` (dispatch registry, persistence, auto-advance poller,
  slot-merged callback) — the foreman loop replaces all of it.
- `SwarmRuntime` / `SwarmSlotRuntime` / `SynthesisOutput` continue-state.
- The deacon `detectMergedSwarmSlots` patrol.
- `SwarmDispatchDialog` — registered slot agents appear in the dashboard
  through existing resource discovery; no new UI for v1.
- Per-slot PRs/review — the issue's PR remains the review unit; slots are
  implementation steps, reviewed together.

### B5. Phase 2: remote slots on Fly

Heavy slots dispatch to Fly VMs via the PAN-1676 per-agent remote machinery,
letting one big issue soak remote capacity (items spread across local +
remote workers). Out of scope for v1; the slot-branch architecture is what
makes it possible at all (shared-filesystem subagents never can).

## 5. Phasing

| Phase | Deliverable | Touches |
| --- | --- | --- |
| 1 | A1 slicing mandate + scope/verify metadata | `roles/plan.md`, planning skills (prompt/docs only) |
| 2 | A2 readiness lint | `pan plan finalize`, `projects.yaml` hotspot config |
| 3 | A4 scope-drift recording | `pan done` / verification gate |
| 4 | B1–B3 foreman swarm (local) | `roles/work.md`, `spawnRun` slot spawning, foreman protocol |
| 5 | A3 cross-issue signals | flywheel launch ordering, merge-queue predicted conflicts |
| 6 | B5 remote slots on Fly | remote-agents wiring |

Phases 1–3 are valuable standalone (they also shrink merge-train conflict
work) and are prerequisites for 4: the foreman cannot schedule without
populated `files_scope`/`verify_commands`. Per the no-partial-delivery rule,
phases here are implementation guidance — the issue ships when all phases
land, unless explicitly re-scoped into sibling issues at planning.

## 6. Non-goals

- Hard refuse-to-dispatch on predicted overlap, at any level (Directive's
  posture). The train is our guarantee; prediction only orders.
- Restoring the old runtime wholesale.
- Per-worker provider policy UI (Cloister routing already covers model
  choice).
- Cohort ceremonies (`swarm:complete-cohort` etc.) — issue close-out already
  owns lifecycle completion.

## 7. Risks & open questions

1. **Foreman context pressure.** Driving many waves may bloat the foreman's
   context. Mitigation: slot agents return terse structured reports; the
   foreman's continue state checkpoints wave progress so a respawned foreman
   resumes mid-swarm.
2. **Foreman dies mid-swarm.** Registered slots keep running (durable);
   deacon resumes the foreman, which re-reads wave state from the spec's
   `statusOverrides` + slot branch existence. Needs an explicit
   reconciliation step in the foreman protocol.
3. **Concurrent `bun install` / build contention** across slot worktrees on
   one machine. Mitigation: cap concurrent heavy slots (config), reuse the
   existing workspace-setup discipline.
4. **Scope prediction quality.** If planners produce junk `files_scope`,
   serialization decisions are junk. The drift loop (A4) measures this;
   `files_scope_confidence: low` items are treated as overlapping-with-
   everything for scheduling (conservative) but never blocked.
5. **Cost attribution per slot.** Registered agents already report cost per
   agent id; verify the dashboard rolls `agent-<issue>-<n>` up under the
   issue.

## 8. Acceptance criteria (sketch)

- A plan finalized without `files_scope` on every item produces lint
  warnings; `--strict` blocks.
- Two unconnected items with overlapping scopes produce a named warning
  listing the shared files (hotspots excluded).
- A work agent on a swarm-eligible plan fans out a wave of non-overlapping
  items as registered slot agents, each in its own worktree/branch; the
  dashboard shows them under the issue.
- A slot whose `verify_commands` fail does not merge; the foreman applies
  the redo-then-serial protocol.
- A convergent item's dispatch prompt contains the synthesis digest of its
  merged parents.
- `pan done` records scope drift in the workspace continue state.
- Killing the foreman mid-swarm and resuming yields a foreman that
  reconciles slot state and continues the correct wave.

## 10. Addendum (2026-06-11): further Directive practices to fold into Path A/B

A detailed re-review of the Directive repo (for PAN-1788) surfaced refinements
beyond what §3–§4 already specify. Fold these into the corresponding work items
during implementation — they are amendments, not new scope.

### 10.1 Broad-glob rejection for `files_scope` (amends A2)

Directive's story-quality lint rejects directory-only and broad-glob scope
declarations (`backend/**`, `frontend/**`, `docs/**`) outright — a scope that
names everything predicts nothing. Our readiness lint should equivalently
reject `src/**`, `src/dashboard/**`, bare directory names, and `**`-suffixed
roots as `files_scope` entries. This is complementary to the hotspot list:
hotspots are *excluded from overlap math*; broad globs are *rejected at
declaration time*.

### 10.2 Cross-field consistency rules (amends A2)

Directive's strongest lint checks are relational, not per-field:

- `readiness: ready` + `file_scope_confidence: low` → reject.
- `size: large` + `parallel_safe: true` → reject.

Our equivalents: an item is slot-dispatch-eligible only with high scope
confidence; `difficulty: complex|expert` items default to foreman-serialized
unless the plan explicitly marks them parallel-safe with a stated reason.

### 10.3 Readiness as a tri-state, not a boolean (amends A1/A2, B2)

Directive uses `readiness: ready | sequential | needs_refinement`. Adopt the
same vocabulary on items: `ready` (slot-eligible), `sequential` (fine, but must
run alone in its wave), `needs_refinement` (not dispatchable — decompose
first). `groupItemsByWave()` + `hasFileOverlap()` cannot express "fine but must
run alone"; the tri-state gives the foreman that signal directly.

### 10.4 `expected_outputs` as the evidence contract (amends A2, B3)

Alongside `verify_commands`, each slot-eligible item declares the observable
evidence it produces (named test passes, endpoint callable, file created,
fixture generated). B3's verify-then-merge asserts these before merging the
slot branch, and the foreman's synthesis digests at convergence points are
composed *from* them. Directive bans vague outputs ("story done", "works as
expected") — reuse PAN-1788's banned-phrase lists for this field.

### 10.5 Readiness report, not just pass/fail (amends A2)

Directive's `swarm_readiness.py` emits a report: dependency waves, conflict
groups, and a file-overlap matrix. We already have `groupItemsByWave()` and
`hasFileOverlap()` in `src/lib/vbrief/dag.ts` — render the same report at
`pan plan finalize` (and in the VBriefViewer) so dispatch decisions are
human-inspectable before any agent spawns.

### 10.6 Boundary maps at wave convergence (amends B3/B6)

For waves where item B consumes item A's output, the plan may optionally
declare produces/consumes pairs (concrete exported names — Directive
`contracts/boundary-maps.md`). At wave boundaries the foreman mechanically
checks "does the export A promised actually exist?" before dispatching B.
Complements `verify_commands`, which only prove A's own tests pass — not that
A delivered B's prerequisites.

### 10.7 Two-phase apply for issue splitting (amends A1)

Path A's "split big issues into independently shippable sibling issues" should
borrow Directive's decompose flow: emit a split *proposal* artifact (child
titles, scopes, dependency edges, cross-sibling overlap audit), validate with a
`--check` dry-run, then apply (create GitHub issues + cross-references). Keeps
the flywheel's audit trail clean and makes bad splits cheap to reject.

### 10.8 Scope-drift feedback refinement (amends A4)

Directive's `triage_scope_drift` machinery feeds drift records back into future
planning. Extend A4 the same way: areas of the codebase whose declared scopes
historically under-predict should default to `file_scope_confidence: low` in
subsequent plans (a per-project drift table in `.pan/context/`, consulted at
planning time).

### 10.9 Posture confirmed: still no refuse-on-overlap

Re-affirmed after the re-review: Directive hard-refuses dispatch on predicted
overlap because it has no integration safety net. We keep overlap as an
ordering signal (§1.4) — the merge train (PAN-1691/PAN-1737) is the guarantee.
Nothing in §10.1–10.8 changes that.
