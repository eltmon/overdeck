# PAN-2181 — Boot-reconciliation grace countdown: extend to 2 min + restyle modal to style-guide tokens (light/dark)

**Issue:** [PAN-2181](https://github.com/eltmon/overdeck/issues/2181)

## Glossary

- **Boot reconciliation** — the PAN-2076 startup gate. On a dashboard boot the Deacon holds all stopped work agents and shows the operator a countdown ("Auto-resuming all in 0:SS") with Resume-all / Keep-all / Review-each / Freeze controls before auto-resuming. State lives in `src/lib/cloister/boot-reconciliation.ts`.
- **Grace window / grace secs** — the countdown duration before the held agents auto-resume. Operator-overridable via `startup.reconciliation_grace_secs` in the Cloister config.
- **Theme token** — a CSS custom property defined on `:root` (light) and `.dark` (dark) in `src/dashboard/frontend/src/index.css`, referenced through a Tailwind utility (`bg-card`, `text-foreground`, `text-warning`, …). The style guide forbids hardcoded color classes (`bg-neutral-950`, `text-white`, etc.) so both modes render correctly.
- **Style guide** — `design/style-guide/STYLE-GUIDE.md`. STYLE-GUIDE.md:153 — "No component may use hardcoded color values like `bg-gray-800` or `text-white`." STYLE-GUIDE.md:25 — "Both light and dark modes are first-class citizens."

## Summary

Two contained changes:

1. **Extend the boot-reconciliation grace default from 30s → 120s (2 min).** The operator currently has only 30 seconds to react on boot.
2. **Restyle `BootReconciliationModal.tsx` to theme tokens.** It is hardcoded to a dark palette, so it renders dark-on-dark / unreadable in light mode. `GraceCountdown.tsx` is already token-based and only needs a both-modes verification.

This is a frontend restyle + two backend constants + tests. **No adjacent refactors** (Karpathy rule 3 — surgical changes).

## Requirements

### Functional

- **FR-1** — The boot-reconciliation grace countdown defaults to **120 seconds**. The existing `startup.reconciliation_grace_secs` config override still wins when set to a finite value > 0.
- **FR-2** — `BootReconciliationModal.tsx` contains **no hardcoded color classes** and renders legibly in **both** light and dark mode. Forbidden patterns: `bg-neutral-*`, `text-neutral-*`, `bg-zinc-*`, `text-zinc-*`, `text-orange-*`, `bg-orange-*`, `border-orange-*`, `bg-emerald-*`, `text-emerald-*`, `bg-sky-*`, `text-sky-*`, `bg-red-*`, `text-red-*`, `bg-gray-*`, `text-white`, `text-black`. Only theme tokens.
- **FR-3** — `GraceCountdown.tsx` is verified token-compliant and legible in both modes (expected: no code change — it already uses `badge-bg-warning`, `text-warning`, `text-foreground`, `bg-popover`, `text-destructive`).

### Non-functional

- **NFR-1** — `npm run typecheck`, `npm run lint`, and `npm test` (root + frontend) are green with no new failures.

## Current state (verified at writing time)

### Grace default — TWO sites, not one

The issue body cites only `boot-reconciliation.ts:14`. That is the **fallback constant**, not the effective default. The resolver reads config first:

`src/lib/cloister/boot-reconciliation.ts:40-45`:
```ts
export function getBootReconciliationGraceSeconds(): number {
  const value = loadCloisterConfigSync().startup.reconciliation_grace_secs;
  return Number.isFinite(value) && value > 0
    ? value
    : DEFAULT_BOOT_RECONCILIATION_GRACE_SECS;
}
```

Because `DEFAULT_CLOISTER_CONFIG.startup.reconciliation_grace_secs` is **30** (`src/lib/cloister/config.ts:289`), the config value of 30 always wins and the constant fallback is rarely reached. **Changing only `boot-reconciliation.ts:14` would NOT change the observed behavior.** Both must change:

- `src/lib/cloister/boot-reconciliation.ts:14` — `export const DEFAULT_BOOT_RECONCILIATION_GRACE_SECS = 30;`
- `src/lib/cloister/config.ts:289` — `reconciliation_grace_secs: 30,` (inside `DEFAULT_CLOISTER_CONFIG.startup`)

Tests that pin the old value:

- `tests/lib/cloister/config.test.ts:142` — `expect(startup.reconciliation_grace_secs).toBe(30);` → must become `120`.
- `src/lib/cloister/__tests__/boot-reconciliation.test.ts:16,87` — sets `mocks.graceSeconds = 30` as an **explicit fixture** (fed in via the config mock at `:32`). This is self-consistent regardless of the default and needs **no change** — but the suite must be re-run to confirm nothing asserts a 30-derived deadline.
