# PAN-1887 — Flywheel auto-merge for GitLab MRs

**Issue:** [PAN-1887](https://github.com/eltmon/overdeck/issues/1887)
**Author:** planning agent (claude-opus-4-8), `pan plan --auto`
**Date:** 2026-06-14

---

## Glossary

- **Forge** — the code-hosting platform behind a project. Overdeck supports two:
  `github` (PRs, `gh` CLI) and `gitlab` (MRs, `glab` CLI). `ForgeType =
  'github' | 'gitlab'` lives in `src/lib/forge.ts:27`.
- **Review artifact** — the forge-side object a branch is merged through: a GitHub
  **Pull Request** or a GitLab **Merge Request**. The `ForgeAdapter` interface
  (`src/lib/forge.ts:71`) abstracts create/merge/comment/approve/discover over both.
- **PR number / MR iid** — GitHub PRs are identified by a repo-scoped *number*
  (`/pull/42`); GitLab MRs by a project-scoped *iid* (`/-/merge_requests/62`).
  Both are integers in their respective URLs.
- **Auto-merge schedule** — the Flywheel's only merge lever. The orchestrator POSTs
  `/api/flywheel/auto-merge/schedule` (`src/dashboard/server/routes/flywheel.ts:658`);
  the handler `postAutoMergeSchedulePayload` (flywheel.ts:297) validates eligibility,
  then inserts a row into the `pending_auto_merges` table with a 5-minute cooldown.
- **Auto-merge executor** — a 30s server-side tick
  (`src/dashboard/server/services/auto-merge-executor.ts`) that picks up *due*
  `pending_auto_merges` rows and executes the merge by delegating to
  `triggerMerge(issueId)` (`src/dashboard/server/routes/workspaces.ts:4836`).
- **Eligibility check** — `isAutoMergeEligible(issueId)`
  (`src/lib/cloister/auto-merge-eligibility.ts:50`). Runs at BOTH schedule time
  (flywheel.ts:325) and execution time (auto-merge-executor.ts:92). Today it parses
  the PR URL with a **GitHub-only** regex and queries the **GitHub** API for CI /
  draft / mergeable state.
- **`readyForMerge`** — a per-issue `ReviewStatus` flag set true once Overdeck's
  own review and test roles have both passed. Forge-agnostic.

## Problem

Under `scope: all-tracked-projects` with `flywheel.require_uat_before_merge=false`,
the Flywheel is supposed to auto-merge ready work in **any** tracked project. It
cannot for GitLab-hosted projects (MYN). A ready GitLab MR
(`MIN-831`, `https://gitlab.com/eltmon/mind-your-now/-/merge_requests/62`,
review=passed test=passed) shows in `pan review pending --ready` but strands at
"ready for merge" — the operator must merge it by hand in the GitLab UI.

### Root cause — two GitHub-only gates in the *schedule* path

The schedule handler `postAutoMergeSchedulePayload` runs these checks in order:

1. **`isAutoMergeEligible` (flywheel.ts:325)** — fires FIRST. Its
   `parsePullRequestUrl` (`auto-merge-eligibility.ts:21`) matches only
   `github\.com/.../pull/(\d+)`. A GitLab MR URL returns `null`, so eligibility
   returns `{ eligible: false, reason: 'review status PR URL is missing or invalid' }`
   and the handler 422-rejects at flywheel.ts:327 — *before* `parsePrNumber` is
   even reached.
2. **`parsePrNumber` (flywheel.ts:272 → used at :333)** — matches only
   `/\/pull\/(\d+)/`. Returns `undefined` for a GitLab MR → 422 at flywheel.ts:335.

> **Refinement of the issue brief:** the brief pinpointed `parsePrNumber` as "the
> blocking gap." That is necessary but **not sufficient** — `isAutoMergeEligible`
> fires earlier and would still 422 a GitLab MR. Both gates must become
> forge-aware to unblock MIN-831.

### What is already correct (do NOT rebuild)

- **Execution is already forge-aware.** `triggerMerge` resolves `primaryForge` from
  the issue's merge set and merges via `getForgeAdapter(primaryForge)
  .mergeReviewArtifact(...)` for the single-repo path
  (`workspaces.ts:5251`, `:5532-5543`), the polyrepo path (`:5183`), and the remote
  path (`:4969`). For GitLab this runs `glab mr merge`. The auto-merge executor
  delegates to `triggerMerge` (`auto-merge-executor.ts:53-55`), so once a GitLab
  entry is *scheduled*, it *executes* correctly today. Issue work-item 3 ("route
  execution to the forge adapter") is therefore a **verification + missing test**,
  not new wiring.
- **The GitLab adapter is complete.** `gitlabForgeAdapter` (`forge.ts:284`) implements
  `mergeReviewArtifact` via `glab mr merge <iid> --squash` (`forge.ts:307-314`).

## Requirements

### Functional

- **FR-1** — A forge-aware parser maps a review-artifact URL to `{ forge, number }`,
  matching GitHub `/pull/(\d+)` → `{ 'github', N }` and GitLab
  `/-/merge_requests/(\d+)` → `{ 'gitlab', iid }`. Detection is by URL **path
  pattern** (supports self-hosted GitLab), not by hardcoded host. Returns `null`
  for unrecognised URLs.
- **FR-2** — The auto-merge schedule endpoint accepts a ready GitLab MR URL: it does
  NOT 422, and it persists a `pending_auto_merges` row carrying the MR iid in
  `prNumber` and `forge = 'gitlab'`. Existing GitHub behaviour is unchanged.
- **FR-3** — The `pending_auto_merges` table and `PendingAutoMerge` type carry a
  `forge` field. Schedules persist it; reads return it; pre-existing rows back-fill
  to `'github'`.
- **FR-4** — `isAutoMergeEligible` evaluates GitLab MRs without calling the GitHub
  API: it queries GitLab MR state (open + mergeable; not merged/closed/draft/conflict/
  CI-failing/CI-running) and returns `{ eligible: true }` for a ready, mergeable
  GitLab MR. GitHub eligibility behaviour is unchanged.
- **FR-5** — The GitLab merge-routing path is covered by a test asserting
  `getForgeAdapter('gitlab').mergeReviewArtifact(...)` issues `glab mr merge <iid>
  --squash` (the executor→triggerMerge→adapter route, validated at the adapter seam).

### Non-functional

- **NFR-1** — Zero regression to the GitHub auto-merge path (schedule, eligibility,
  execution behaviour byte-for-byte unchanged for GitHub inputs).
- **NFR-2** — The schema change is additive and idempotent: `ALTER TABLE ... ADD
  COLUMN forge TEXT NOT NULL DEFAULT 'github'` inside a `currentVersion < 54` block;
  the fresh `CREATE TABLE` def gains the column; historical migration CREATE blocks
  are left untouched.
- **NFR-3** — GitLab eligibility uses async exec only (promisified `execFile`, never
  `execSync` — server-reachable code, PAN-70) and degrades safely: a `glab` error or
  unparseable output yields `{ eligible: false, reason }`, never a throw.

## Decisions (made in this document)

- **D1 — Forge detection by path pattern, not host.** `/pull/(\d+)` ⇒ github;
  `/-/merge_requests/(\d+)` ⇒ gitlab. This supports self-hosted GitLab instances
  (MYN is gitlab.com, but the parser must not hardcode the host). Host is not used
  for detection.
- **D2 — Add a real `forge` column** rather than overloading `prNumber` alone. The
  brief requests it (work-item 2); it makes the row self-describing so no consumer
  ever misreads a GitLab iid as a GitHub PR number, and it gives downstream
  surfaces the data to render the correct artifact identifier. `NOT NULL DEFAULT
  'github'` back-fills existing rows correctly (every pre-existing entry was GitHub).
- **D3 — Keep GitLab eligibility a focused path, not a unified abstraction.** The
  issue notes a future "shared forge-abstraction" converging this with PAN-933.
  That convergence (adding `getArtifactState()` to `ForgeAdapter`) is deferred — it
  touches the adapter interface (cross-cutting) and PAN-933's review-poster surface.
  For now, GitLab eligibility is an injectable helper inside
  `auto-merge-eligibility.ts`, mirroring the existing injectable `getPullRequestState`
  dep so it stays unit-testable.
- **D4 — GitLab MR state field.** Use `detailed_merge_status` as the primary
  mergeability signal (`'mergeable'` ⇒ ready; any other value ⇒ ineligible with that
  value as the reason), with `state` ('opened') and `draft` as explicit pre-checks
  and `merge_status` ('can_be_merged') as a fallback when `detailed_merge_status` is
  absent. See **Implementation checkpoint** below.

### Implementation checkpoint (FR-4)

`glab mr view <iid> -R <projectPath> -F json` returns the GitLab MR object. The exact
JSON field names (`state`, `draft`, `merge_status`, `detailed_merge_status`,
`has_conflicts`, `pipeline.status`) are snake_case from the go-gitlab `MergeRequest`
struct. **Before relying on them, the work agent must confirm the field names** by
running `glab mr view 62 -R eltmon/mind-your-now -F json | jq 'keys'` against a real
MYN MR (or reading the go-gitlab struct). **Fallback** if `detailed_merge_status` is
absent on the installed GitLab/glab: gate on `state === 'opened'` && `draft !== true`
&& `merge_status === 'can_be_merged'` && `has_conflicts !== true`.

## Work items

### WI-1 (FR-1) — Forge-aware artifact-ref parser

**File:** `src/lib/forge.ts` (co-located with `ForgeType` and the adapters).

Add an exported pure function:

```ts
/** Parse a review-artifact URL into its forge + numeric id (PR number or MR iid).
 *  Detection is by path pattern so self-hosted GitLab is supported. */
export function parseArtifactRef(url: string | undefined): { forge: ForgeType; number: number } | null {
  if (!url) return null;
  const gh = url.match(/\/pull\/(\d+)(?:$|[/?#])/);
  if (gh) return { forge: 'github', number: Number.parseInt(gh[1], 10) };
  const gl = url.match(/\/-\/merge_requests\/(\d+)(?:$|[/?#])/);
  if (gl) return { forge: 'gitlab', number: Number.parseInt(gl[1], 10) };
  return null;
}
```

**Tests** (`tests/unit/lib/forge.test.ts`): a GitHub `/pull/42` URL → `{github,42}`;
a GitLab `/-/merge_requests/62` URL → `{gitlab,62}`; a self-hosted GitLab host →
still `{gitlab,N}`; a trailing-segment URL (`.../pull/42/files`) → `{github,42}`; a
non-artifact URL → `null`; `undefined` → `null`.
