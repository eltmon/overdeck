# PAN-1900 — UAT candidate branch codename is non-deterministic

**Issue:** [PAN-1900](https://github.com/eltmon/panopticon-cli/issues/1900)
**Type:** bug(flywheel)
**Planning mode:** `--auto` (no interactive Q&A)

---

## Glossary

- **UAT candidate / UAT generation** — a throwaway `uat/<label>-<codename>-<MMDD>`
  branch built off `origin/main` that merges the current bundle of ready feature
  branches together, so a human can UAT the whole batch in one sitting. Created by
  the PAN-1737 *UAT batch trains* engine (`uat-generation-engine.ts`). It never
  touches `main`.
- **Generations chain (PAN-1737)** — the current model: each assembly produces a
  distinct generation **row** in the `uat_generations` table. The newest `ready`
  generation is "the candidate"; older ones flip to `superseded` (still testable)
  and are trimmed to the newest `keep=3` by `cleanupUatGenerations`, which deletes
  trimmed branches locally **and on origin** (`git push origin --delete`).
- **codename** — the memorable word in the branch name (`otter`, `falcon`, …) from
  `UAT_CODENAMES` in `uat-candidate-name.ts`. Exists to disambiguate multiple
  generations on the same day (the date alone collides).
- **Reconciler** — `reconcileUatGenerations` (`uat-reconciler.ts`), run every 60 s by
  `startUatTrainReconciler()` (wired unconditionally in `main.ts:400`). Gated on
  `isMergeTrainEnabled()` AND an active flywheel run. It **auto-assembles** a new
  generation only when no live generation matches the desired bundle
  (`liveSignatureMatches`), so the non-forced path is already idempotent.
- **Forced reconcile** — `runUatTrainReconcile({ force: true })`. Skips the
  `liveSignatureMatches` idempotency guard and always assembles a new generation.
  Exposed as `POST /api/flywheel/assemble-uat` — meant as an explicit operator
  "rebuild" button.

---

## Problem

Under `require_uat_before_merge=true`, three `uat/*` branches were created for the
same day (0614), ~10 min apart, each with a different **random** codename
(`birch` → `willow` → `cobalt`). The operator can't tell which to ship, stale
branches accumulate on origin, and `GET /api/flywheel/current` exposes no field
naming the active candidate.

### Root cause (two independent defects)

**Defect 1 — the flywheel force-assembles every tick (the proliferation driver).**
`roles/flywheel.md:165-172` tells the orchestrator, under `require_uat_before_merge=true`,
to `POST /api/flywheel/assemble-uat` **every tick**, asserting the call is
"idempotent and safe to run every tick … deterministic per day … rebuilds the
*same* branch rather than proliferating new ones." **That assertion is false for
the current code.** PAN-1737 repurposed `/api/flywheel/assemble-uat` to
`runUatTrainReconcile({ force: true })` (`flywheel.ts:965-976`). `force: true`
bypasses the idempotency guard at `uat-reconciler.ts:212-217`, so **every tick
assembles a brand-new generation** — with a new branch. The role doc is doubly
stale: it also calls `GET /api/flywheel/uat-candidate` (line 168), an endpoint
that **no longer exists** (zero references in `src/`).

**Defect 2 — the codename is random.** `makeUatCandidateName`
(`uat-candidate-name.ts:42-61`) defaults its start index to
`Math.random` (`pick = (n) => Math.floor(Math.random() * n)`, line 45). So even a
*legitimately new* generation (the bundle genuinely changed — `cobalt` bundled
PAN-1242) gets an unordered, unpredictable name; the operator can't tell newest
from name alone, only by branch-tip time.

These are independent: Defect 1 creates redundant generations; Defect 2 makes all
generations (redundant or legitimate) un-orderable by name.

### What is NOT broken (verified — do not "fix")

- **Origin reaping works.** `buildUatGenerationCleanupGit().deleteBranch`
  (`uat-generation-deps.ts:145-148`) runs `git push origin --delete` + local
  delete; `cleanupUatGenerations` keeps the newest 3. The 4-branch pile-up is
  `keep=3` **outpaced by per-tick proliferation**, not a broken reaper. Once
  Defect 1 is fixed, the chain stabilises at ≤3 and origin branches reap on the
  next idle reconciler tick. **No reaper code change.**
- **The internal 60 s reconciler covers assembly.** It auto-assembles whenever
  `merge_train_enabled` + active run, independent of `require_uat`. So removing the
  every-tick forced call does **not** stop UAT assembly. (When `merge_train` is
  off, even the forced endpoint no-ops via `isEnabled()` → no edge regression.)

---

## Proposal

Fix both defects **within** the PAN-1737 generations architecture. Do **not**
revert to a single force-reset branch per day (see NonGoals + the architectural
fork in continue.json hazards).

1. **Stop the every-tick forced assemble** — rewrite the `require_uat_before_merge=true`
   section of `roles/flywheel.md` so the orchestrator *reads and surfaces* the
   server-assembled candidate instead of force-rebuilding it each tick. (Bead A)
2. **Make the codename deterministic + ordered** — default the start index to `0`
   instead of `Math.random`; the existing collision-walk then assigns
   otter → falcon → cedar in generation order. (Bead B)
3. **Surface the candidate authoritatively** — revive `GET /api/flywheel/uat-candidate`
   returning the newest `ready` generation (`{ branchName, bundled, status }`). (Bead C)

---

## Requirements

**Functional**
- **FR-1** — The candidate branch name is deterministic: identical
  (label, day, set-of-already-taken-names) inputs always produce the same name; no
  per-cycle randomness. Sequential generations get sequential codenames.
- **FR-2** — Under a steady-state bundle (unchanged ready set + member SHAs), the
  flywheel orchestration does not create a new `uat/*` branch per tick; the existing
  candidate is reused.
- **FR-3** — The active UAT candidate (newest `ready` generation: branch name +
  bundled issue IDs + status) is exposed via an authoritative read endpoint,
  returning an empty/`null` body when nothing is ready.

**Non-functional**
- **NFR-1** — The PAN-1737 generations chain is preserved: multiple live
  generations (newest supersedes, older stay testable) and per-generation live
  stacks remain supported. No collapse to one branch/day.
- **NFR-2** — `POST /api/flywheel/assemble-uat` remains available as an explicit
  operator forced-rebuild; it simply is no longer called on an automated loop.

## Work items

### Bead A — Stop the flywheel force-assembling the UAT candidate every tick
**Files:** `roles/flywheel.md` (the `require_uat_before_merge=true` block, ~lines 165-172).
**Why:** removes the proliferation driver (Defect 1) and the stale/false guidance.
**Repo rule restated:** operator-facing pipeline text must stand alone and define
its terms (explain-fully rule) — keep the rewrite in complete sentences, no
compression.

Before:
```
Allowed when `require_uat_before_merge=true` — **auto-assemble the UAT candidate** (this is the under-UAT flow; do it, don't ask the operator to flip UAT off):

- With UAT required you must **not** schedule merges — but you **should** keep a ready-to-UAT bundle assembled so the operator can review and ship a batch in one sitting. Each tick:
  1. `GET /api/flywheel/uat-candidate` → `{ branchName, bundled }`. `bundled` is the disjoint, batch-safe set of ready features (conflicting ones serialize and are excluded).
  2. If `bundled` is non-empty, `POST /api/flywheel/assemble-uat` (empty `{}` body). This (re)builds the per-day `uat/<label>-<codename>-<MMDD>` branch off current `origin/main` and merges the bundle onto it.
  3. Surface the candidate in your status/report: the branch name, the bundled issue IDs, and any merge conflicts it reported. The operator UATs that one branch, then clicks **Ship batch** (or `POST /api/flywheel/merge-next`).
- **This call is idempotent and safe to run every tick.** The branch name is deterministic per day and the branch is force-reset onto current main, so repeated assembly rebuilds the *same* branch from the current bundle rather than proliferating new ones. Assembling is *not* a merge — it never touches `main` — so the merge-policy gate below does not apply to it.
- Do **not** ask the operator "want me to flip `require_uat_before_merge=false`?" The UAT candidate *is* the answer under UAT. Only the operator changes that toggle.
```

After:
```
Allowed when `require_uat_before_merge=true` — **surface the auto-assembled UAT candidate** (this is the under-UAT flow; do it, don't ask the operator to flip UAT off):

- With UAT required you must **not** schedule merges. The server's UAT reconciler keeps a ready-to-UAT candidate assembled for you automatically: every 60 seconds, while a flywheel run is active and `flywheel.merge_train_enabled` is on, it builds — or rebuilds only when the ready set or its commits actually change — the current UAT generation branch off `origin/main`. You do **not** trigger assembly yourself each tick. Doing so used to mint a brand-new `uat/*` branch every cycle, because `POST /api/flywheel/assemble-uat` is a *forced* rebuild that skips the "nothing changed" check (PAN-1900). Each tick:
  1. `GET /api/flywheel/uat-candidate` → `{ branchName, bundled, status }`, or `null` when nothing is ready. `bundled` is the batch-safe set of issue IDs in the candidate (conflicting features are held out). This endpoint is the **authoritative** pointer at the current candidate — it always returns the newest `ready` generation, so you never infer it from branch tip times.
  2. Surface the candidate in your status/report: the branch name and the bundled issue IDs. The operator UATs that branch, then clicks **Ship batch** (or `POST /api/flywheel/merge-next`).
- The server auto-assembles; you only read and surface. Several UAT generations may be live at once (newest supersedes older, which stay testable until reaped) — the `uat-candidate` endpoint names the current one.
- `POST /api/flywheel/assemble-uat` is a **forced rebuild** — an explicit operator action that rebuilds the candidate even when nothing changed. Do **not** call it on a loop; the reconciler already keeps the candidate fresh. Assembling is *not* a merge — it never touches `main` — so the merge-policy gate below does not apply to it.
- Do **not** ask the operator "want me to flip `require_uat_before_merge=false`?" The UAT candidate *is* the answer under UAT. Only the operator changes that toggle.
```

Operational note (hazard H3): a **running** flywheel will not pick up role-doc
changes until a new run / restart.

### Bead B — Make the UAT candidate codename deterministic and ordered per day
**Files:** `src/lib/cloister/uat-candidate-name.ts` + its unit test.
**Why:** removes Defect 2; sequential generations get sequential codenames.

The function already walks forward from `start` on collision. Defaulting `start`
to `0` (instead of a random `pick`) makes the walk produce otter, falcon, cedar, …
as generations accumulate — deterministic and ordered, **no store changes**.

Before (`uat-candidate-name.ts:45`):
```ts
  const pick = deps.pick ?? ((n: number) => Math.floor(Math.random() * n));
```
After:
```ts
  // Deterministic, ordered-per-day start: the first untaken codename is the
  // first generation of the day (otter), the second is falcon, and so on. The
  // collision walk below advances as branches are taken. (PAN-1900: a random
  // start minted a fresh unordered codename per assembly cycle.)
  const pick = deps.pick ?? (() => 0);
```
Also update the file header comment that says the default wrapper "supplies real
randomness" → state it starts at index 0 (deterministic). Keep `pick` in
`UatNameDeps` (the injectable seam stays for tests).

### Bead C — Add authoritative `GET /api/flywheel/uat-candidate` read endpoint
**Files:** `src/dashboard/server/services/uat-train.ts` (new `getUatCandidatePayload()`),
`src/dashboard/server/routes/flywheel.ts` (new route + register in the router list
near `getUatGenerationsRoute`, ~line 1004), + a test.
**Why:** FR-3 — names the candidate authoritatively; revives the endpoint the role
doc (Bead A) references.

`getUatGenerationSync`/`listUatGenerationsSync` are already imported in
`uat-train.ts`. `listUatGenerationsSync` orders `created_at DESC` and accepts
`{ projectRoot, statuses, limit }`.

```ts
// services/uat-train.ts
export async function getUatCandidatePayload(): Promise<
  { branchName: string; bundled: string[]; status: UatGeneration['status'] } | null
> {
  const [gen] = listUatGenerationsSync({
    projectRoot: projectRoot(),
    statuses: ['ready'],
    limit: 1,
  });
  if (!gen) return null;
  return { branchName: gen.name, bundled: gen.members.map((m) => m.issueId), status: gen.status };
}
```
```ts
// routes/flywheel.ts — register alongside getUatGenerationsRoute
const getUatCandidateRoute = HttpRouter.add(
  'GET',
  '/api/flywheel/uat-candidate',
  httpHandler(Effect.gen(function* () {
    const { getUatCandidatePayload } = yield* Effect.promise(() => import('../services/uat-train.js'));
    const payload = yield* Effect.promise(() => getUatCandidatePayload());
    return jsonResponse(payload);
  })),
);
```
Test (extend `services/__tests__/uat-train.test.ts` or `routes/__tests__/flywheel.test.ts`):
seed a store with a newest `ready` generation → payload names it; no `ready`
generation → `null`.

## NonGoals
- Does **NOT** revert PAN-1737's generations chain to a single force-reset branch
  per day. The issue's literal "Suggested fix #1" (force-reset the same
  `uat/<label>-<MMDD>` branch) is **declined** — it would tear out the deliberately
  shipped multi-generation chain and per-generation live stacks (NFR-1).
- Does **NOT** change `POST /api/flywheel/assemble-uat` semantics — it stays a
  forced operator rebuild (NFR-2). It is simply no longer called on a loop.
- Does **NOT** add or modify the reaper / `cleanupUatGenerations`; origin reaping
  already works and self-corrects once proliferation stops.
- Does **NOT** add a field to the shared `FlywheelStatus` contract; surfacing uses a
  dedicated read endpoint instead.
- Does **NOT** change `makeUatCandidateName`'s collision-walk or numeric-suffix
  exhaustion fallback.
