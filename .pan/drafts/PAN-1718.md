# PAN-1718 â€” Duplicate successful `pan reload` restart-status writes from two concurrent processes

**Issue:** https://github.com/eltmon/overdeck/issues/1718

## Glossary

- **restart lock** â€” `~/.overdeck/restart.lock`, a fail-fast advisory mutex (`open(path, 'wx')`) acquired via `acquireRestartLock(caller)` in [`src/lib/restart-lock.ts`](../../src/lib/restart-lock.ts). A second overlapping holder gets `null` (EEXIST, not stale) and is expected to abort.
- **restart-status** â€” `~/.overdeck/restart-status.json`, a **single-entry** file (last-write-wins via tmp+`rename`) written by `writeRestartStatus()` in [`src/lib/restart-status.ts`](../../src/lib/restart-status.ts). Holds the most recent restart outcome only.
- **restart writers** â€” the four code paths that call `writeRestartStatus`: `pan reload` ([`reload.ts:63`](../../src/cli/commands/reload.ts)), `pan restart` ([`restart.ts:151`](../../src/cli/commands/restart.ts)), and the supervisor watchdog (two sites in [`watchdog.ts`](../../src/supervisor/watchdog.ts)).
- **restart announcer** â€” [`restart-announcer.ts`](../../src/dashboard/server/services/restart-announcer.ts) (PAN-1714), polls restart-status.json and emits one activity entry per new `ts`. This is what surfaced the duplicate.
- **supervisor cycle** â€” `stopSupervisorProcessSync()` + `startSupervisorProcessSync()` in [`src/lib/supervisor.ts`](../../src/lib/supervisor.ts), called by `pan restart --dashboard/--full` and `pan dev` (NOT by `pan reload`).

## Problem

On the restart announcer's first live run (2026-06-10 ~01:10Z) a single physical dashboard restart produced **two** successful `trigger='pan reload'` writes with different implied start times (01:10:10.5Z and 01:10:17.6Z) â€” definitively two processes. The fail-fast restart lock should have made the second visibly fail (`success=false`, "restart in progress"), yet both wrote `success=true`. In the same window `supervisor.pid` pointed at a dead pid that never logged "supervisor listening" â€” the signature of a supervisor stop/start cycle whose stop didn't take and whose start collided on EADDRINUSE.

## Root-cause analysis (verified against the code at planning time)

### Finding A â€” `pan reload`'s `pan dev` SIGUSR2 branch bypasses the restart lock entirely

`reloadCommand` ([`reload.ts:77-177`](../../src/cli/commands/reload.ts)) acquires the restart lock at **line 113**, but the `pan dev` marker branch at **lines 94-111** runs *before* that, calls `recordReloadStatus(startedAt, true)` and `return`s â€” **without ever acquiring the lock**. Any reload that finds a `pan dev` supervisor marker writes `success=true` with no mutual exclusion, so a concurrent restart flow is neither excluded nor made to fail. This is a provable violation of the second acceptance criterion. (The branch is fast, so it does not by itself explain two *35â€“40s* writes â€” see Finding B for why we cannot yet prove the exact pairing, and why the forensic journal is required.)

### Finding B â€” single-entry restart-status.json cannot identify the second writer

`writeRestartStatusPromise` tmp-writes then `rename`s a **single** file (last-write-wins). The pid instrumentation already landed (`pid` is recorded â€” [`reload.ts:71`](../../src/cli/commands/reload.ts)), but when two writers fire 2.6s apart the first entry is overwritten before anything reads it. With only the single file, "identify the second writer on next occurrence" is **structurally impossible** â€” the record of the first writer is gone. A forensic record must be **append-only**.

### Finding C â€” supervisor stop/start cycle leaves a dead pidfile (the `supervisor.pid` clue)

`stopSupervisorProcessSync` ([`supervisor.ts:131`](../../src/lib/supervisor.ts)) sends SIGTERM and **immediately** unlinks the pidfile without waiting for the process to exit. `startSupervisorProcessSync` ([`supervisor.ts:76`](../../src/lib/supervisor.ts)) then sees no live pidfile, spawns a new supervisor, and `writeFileSync`s its pid even if that child is about to die on EADDRINUSE (old supervisor still holding the port). Result: a recorded pid that never comes up â€” exactly the observed symptom. This corroborates "restart-flow machinery beyond the operator's single `pan reload` ran concurrently."

## Proposal

Three independent fixes, matching the two acceptance criteria:

1. **Correctness (lock):** acquire the restart lock at the top of `reloadCommand`, before the dev-marker branch, so every reload path (dev-signal and detached) is mutually excluded; a concurrent reload records `success=false` and exits non-zero. (Finding A)
2. **Diagnostics (forensics):** make `writeRestartStatus` also append every entry to an append-only `~/.overdeck/restart-events.jsonl` journal, add `readRestartEvents()`, and a pure `detectConcurrentRestartWriters()` helper. (Finding B)
3. **Diagnostics (operator surface):** `pan status` shows the restart writer's pid + initiator and warns when the journal shows overlapping concurrent writers. (Finding B)
4. **Correctness (supervisor):** `stopSupervisorProcessSync` waits (bounded) for the old process to actually exit before returning; `startSupervisorProcessSync` verifies the spawned child survived and clears the pidfile if it died. (Finding C)

> **Event-loop safety (verified at planning time):** every caller of `stopSupervisorProcessSync` / `startSupervisorProcessSync` is CLI-side â€” `src/cli/index.ts:958/1175`, `src/cli/commands/dev.ts:209/667`, `src/cli/commands/restart.ts:217-218/302/344`. None run on the dashboard server event loop, so a *bounded* synchronous exit-wait in these functions does not violate the no-blocking-the-event-loop rule. The bounded wait must stay short (â‰¤~1.5s total) and use `Atomics.wait` on a `SharedArrayBuffer` (or a tight `Date.now()` poll), never a busy-spin or `execSync('sleep')`.

## Requirements

- **FR-1** Every `pan reload` invocation â€” including the `pan dev` SIGUSR2 branch â€” acquires the restart lock before doing any work; if the lock is held, it records `success=false` ("restart in progress") and exits 2 without signalling or restarting.
- **FR-2** Every restart-status write is also appended, immutably, to `~/.overdeck/restart-events.jsonl`; the journal preserves both entries when two writers race.
- **FR-3** A pure helper identifies â‰Ą2 successful writes within a short window with different pids as concurrent writers.
- **FR-4** `pan status` shows the last restart's writer pid + initiator and, when the journal shows concurrent writers, prints a warning naming both.
- **FR-5** A supervisor stop/start cycle never leaves a pidfile pointing at a dead process: stop waits for exit; start verifies survival.
- **NFR-1** No new blocking calls on the dashboard server event loop (changes live in CLI-side `*_Sync` paths and the lib layer; bounded sync waits use `Atomics.wait`, not busy-spin).
- **NFR-2** Journal growth is bounded (cap to the last ~200 entries).
