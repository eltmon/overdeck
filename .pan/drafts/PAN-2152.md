## Proposal — five submodules + a thin barrel
Sibling naming follows the existing `workspace-<verb>.ts` convention.

1. `workspace-beads.ts` — `encodeBeadsVersion`, `getBeadsVersion`, `REDIRECT_MANAGED_BEADS_VERSION`,
   `initializeWorkspaceBeads`, **`__testInternals`** (defined here; barrel re-exports it). (~125 lines)
2. `workspace-remote.ts` — `convertToSshUrl`, `createRemoteWorkspace`, `destroyRemoteWorkspace`,
   `migrateCommand`, `MigrateOptions`, `syncAuthCommand`, `sshCommand`, `startCommand`, `stopCommand`.
   (~520 lines)
3. `workspace-create.ts` — `CreateOptions`, `createCommand`; imports `initializeWorkspaceBeads` from
   `./workspace-beads.js` and `createRemoteWorkspace` from `./workspace-remote.js`. (~420 lines)
4. `workspace-list.ts` — `ListOptions`+`listCommand`, `DestroyOptions`+`destroyCommand`; imports
   `destroyRemoteWorkspace` from `./workspace-remote.js`. (~255 lines)
5. `workspace-maintenance.ts` — `UseConfigOptions`+`useConfigCommand`, `UpdateOptions`+`updateCommand`,
   `AddRepoOptions`+`RepoGroups`+`loadRepoGroups`+`addRepoCommand`+`YAML`. (~265 lines)

Barrel `workspace.ts` keeps: imports needed by `registerWorkspaceCommands`, the command-wiring
function itself, and re-exports `export { __testInternals } from './workspace-beads.js'` +
`export { destroyCommand } from './workspace-list.js'`. Target < 250 lines.

### Why these seams
Cohesion by domain (beads / remote-VM / local-create / list+destroy / config+update+add-repo) and the
internal call graph: `createCommand` → {beads, remote}; `destroyCommand` → remote;
`createRemoteWorkspace` → `convertToSshUrl` (kept together in remote). Each module is independently
mergeable because the barrel re-exports/imports keep every external import resolving after each step.
Extraction order = beads → remote → create → list → maintenance → baseline-ratchet (leaves first).
