# PAN-1696 — Decouple merge-train from the Flywheel: per-project pipeline feature + multi-project view

**Issue:** https://github.com/eltmon/panopticon-cli/issues/1696
**Planning mode:** `pan plan --auto` (no interactive Q&A; defaults recorded in `plan.autoDecisions[]`)
**Date:** 2026-06-12

## Glossary

| Term | Meaning |
| --- | --- |
| **Flywheel** | The singleton orchestrator role (an LLM run) that keeps agents working. Run lifecycle in `src/lib/cloister/flywheel.ts`; dashboard surface `src/dashboard/frontend/src/pages/FlywheelPage.tsx`. |
| **Merge train** | The per-project landing machinery: conflict-aware merge queue, UAT batch-train generations, post-merge sibling reconciler, per-issue auto-merge policy. Shipped across PAN-1691/1737. |
| **Ready set** | All features of one project at the merge gate, in conflict-aware merge order. Today sourced from the flywheel run's `activePipeline`; after this issue, sourced from the review-status pipeline records. |
| **Review-status records** | The review-status SQLite store; the authoritative full listing is `getAllReviewStatusesFromDb()` (`src/lib/database/review-status-db.ts:235`; `getReviewStatusSync` at `src/lib/review-status.ts:463` reads the same store). Global, keyed by issue ID across ALL projects (PAN-*, MIN-*, …). |
| **Merge-gate eligibility** | `mergeGateEligibility()` (`src/lib/review-status.ts:140`) — the authoritative "allowed to merge" predicate (PAN-1759): review passed, test passed/skipped, verification not failed, not merged. |
| **Generation** | A throwaway `uat/<label>-<codename>-<MMDD>` branch containing the ready set with cross-feature conflicts resolved in-branch. Stored in the `uat_generations` SQLite table (name PK, has `project_root`). |
| **UAT train reconciler** | The 60s dashboard-server interval (`startUatTrainReconciler`, `src/dashboard/server/services/uat-train.ts:140`) that keeps one assembled generation ready per project. |
| **Sibling reconciler** | The post-merge rebase fan-out (`runMergeTrainReconcile(issueId)`, `src/lib/cloister/merge-train.ts:26`), fired from `firePostMergeLifecycle` in `src/dashboard/server/routes/specialists.ts`. |
| **Flywheel scope** | `roles.flywheel.scope` (`pan-only` \| `all-tracked-projects`, `src/lib/config-yaml.ts:343`) — which projects the *orchestrator* inventories. Distinct from merge-train per-project behavior. |
| **Tracked project** | An entry in `~/.panopticon/projects.yaml`, enumerated by `listProjectsSync()` (`src/lib/projects.ts:219`). Issue → project mapping via `resolveProjectFromIssueSync()` (`src/lib/projects.ts:351`). |
| **Project cockpit** | `ProjectOverview.tsx` in Command Deck; hosts `ProjectSettingsSection` (`src/dashboard/frontend/src/components/CommandDeck/ProjectOverview.tsx:152`). |

## Problem

The merge-train is a per-project pipeline concern (how ready work lands) but is presented and wired as a Flywheel feature (the orchestrator). Verified current state (2026-06-12):

1. **Flag is flywheel-namespaced.** `flywheel.merge_train_enabled` in `app_settings` (`src/lib/database/app-settings.ts:86`), read by `isMergeTrainEnabled()` (`:136`).
2. **Ready set requires a live flywheel run.** `getReadySet()` in `src/dashboard/server/services/uat-train.ts:63` calls `readCurrentFlywheelStatusForDashboard()` and returns `null` without a run; `getUatGenerationsPayload()` (`:236`) returns `[]` without a run. `GET /api/flywheel/merge-queue` (`src/dashboard/server/routes/flywheel.ts:891`) returns `[]` without a run.
3. **Single-project.** `projectRoot()` in `uat-train.ts:58` is `process.cwd()` — the Panopticon repo. A ready MIN-831 never assembles into a MYN `uat/*` generation (2026-06-11 operator report).
4. **UI lives only on the Flywheel page** (`MergeQueueCard.tsx` fetching `/api/flywheel/*`).
5. **Scope UX gap.** The Settings → Roles → Flywheel scope select exists (`RolesPanel.tsx:554-566`, shipped 2026-05-19 in PAN-1189) but the operator couldn't find it; the Flywheel page shows the *configured* scope from settings (`FlywheelConversationPane.tsx:84-90`), not the active run's *effective* scope; run records do not persist scope; `roles/flywheel.md:83` hardcodes "list PAN issues" and never defines what `all-tracked-projects` means.

Everything below the service layer is **already multi-project-capable**: `reconcileUatGenerations(projectRoot, …)` is parametrized and single-flight per root (`src/lib/cloister/uat-reconciler.ts:151`), `uat_generations` has a `project_root` column + index (`src/lib/database/schema.ts:355-374`), `buildUatGenerationGitDeps(projectRoot)` is parametrized (`src/lib/cloister/uat-generation-deps.ts:64`), the sibling reconciler resolves project from issue (`src/lib/cloister/merge-train-deps.ts:22`), and `ensureUatStack` uses `gen.projectRoot` (`src/lib/cloister/uat-stack.ts:179`).

## Requirements

### Functional

- **FR-1** The merge-train enable flag lives at app-settings key `merge_train.enabled`. `isMergeTrainEnabled()` reads the new key and falls back to the legacy `flywheel.merge_train_enabled` value when the new key is unset; `setMergeTrainEnabled()` writes only the new key. An installation with the legacy key `true` stays enabled after upgrade with no manual step.
- **FR-2** `ProjectConfig` gains an optional `merge_train?: 'enabled' | 'disabled'` field (projects.yaml). A resolution helper `isMergeTrainEnabledForProject(projectKey)` returns the per-project override when present, else the global flag. Both the UAT train reconciler (per-project tick) and the post-merge sibling reconciler (`runMergeTrainReconcile`) gate on this helper.
- **FR-3** The ready set is computed per project from review-status records filtered by `mergeGateEligibility().eligible` and `deaconIgnored !== true`, grouped by project via `resolveProjectFromIssueSync()` — with no read of `flywheel.activePipeline` and no requirement that a flywheel run exists. Titles resolve through an injectable resolver (dashboard: issue-data-service title map; fallback: the issue ID).
- **FR-4** The UAT train reconciler iterates every tracked project each tick; for each effectively-enabled project with ready work it reconciles against that project's root. It performs **no git commands** in a project whose ready set is empty AND whose generation chain has no live (`ready`/`superseded`/`assembling`) rows.
- **FR-5** Merge-train HTTP API moves to `/api/merge-train/*`: `GET /queues` (per-project ready sets), `GET /generations` (per-project chains), `POST /generations/:name/stack`, `POST /generations/:name/promote`, `POST /assemble` (`{project?}`), `POST /merge-next` (`{n, project}`). The old flywheel-namespaced merge-train routes (`/api/flywheel/merge-queue`, `/uat-generations`, `/uat-generations/:name/stack`, `/uat-generations/:name/promote`, `/assemble-uat`, `/merge-next`) are removed.
- **FR-6** `GET/POST /api/projects/:projectKey/merge-train` reads/writes the per-project override in projects.yaml, following the `auto-merge-default` endpoint pattern (`src/dashboard/server/routes/projects.ts:573-585`).
- **FR-7** A multi-project merge/UAT view exists with an explicit project selector/filter (selection persisted in localStorage), hosted on the Awaiting Merge page.
- **FR-8** The Flywheel page remains one viewer of the same data: its merge-train card renders from the new API and shows data even when no run is active; the global merge-train toggle keeps working (now writing `merge_train.enabled` through the renamed setters).
- **FR-9** The project cockpit's `ProjectSettingsSection` exposes the per-project merge-train control (Enabled / Disabled / Global default).
- **FR-10** UAT stack frontend URLs honor the project's DNS domain config (`ProjectConfig` `dns.domain`, `src/lib/projects.ts:37`) when present, falling back to `pan.localhost`.
- **FR-11** `pan flywheel status`'s merge-queue section (`src/cli/commands/flywheel.ts:818`) computes from the per-project ready set of the cwd project, not `activePipeline`.
- **FR-12** The active flywheel run's effective scope is persisted in the run record at start/resume and displayed on the Flywheel page, with copy stating that scope changes apply at the next run start or resume.
- **FR-13** The Settings → Roles → Flywheel scope select carries explanatory help text for `pan-only` vs `all-tracked-projects`, and the panel's apply-timing copy is accurate for scope (next run start/resume — not "next tick"). Saving updates `roles.flywheel.scope`.
- **FR-14** `roles/flywheel.md` defines scope semantics: under `all-tracked-projects` the orchestrator inventories ready work across all registered projects with the same author/assignee safety gates (generalized per tracker), and notes that merge/UAT trains are per-project and independent of the run.
- **FR-15** Docs updated: `docs/UAT-BATCH-TRAINS.md` (ready-set definition, "Operating it", API table), `docs/MERGE-TRAIN.md` §7 (resolved-by note), `docs/FLYWHEEL.md` (orchestrator scope ≠ batch-train scope; cross-project batches remain out of scope).

### Non-functional

- **NFR-1** `tests/unit/lib/cloister/in-flight-guard.test.ts` stays green; the post-merge path remains idempotent (PAN-328).
- **NFR-2** No `execSync` in server-reachable code; all new process exec is async argv-array `execFile` (PAN-70 rule).
- **NFR-3** The multi-project reconciler must not introduce per-tick git traffic in idle projects (see FR-4 skip condition).
- **NFR-4** Existing unit suites for the engine (`uat-generation-engine`, `uat-reconciler`, `uat-promote`, `merge-train*`, `flywheel-merge-order`) keep passing, updated only where the contract intentionally changed.

## Decisions (made in this document — do not re-derive)

| # | Decision | Rationale |
| --- | --- | --- |
| D1 | Flag migration is a **fallback read**, not a data migration: read `merge_train.enabled`, fall back to `flywheel.merge_train_enabled` when unset; writes target only the new key. | Zero-risk upgrade; the flag is currently ON in production and must not silently flip off. |
| D2 | Per-project override is **tri-state** (`enabled`/`disabled`/absent = follow global), mirroring `auto_merge_default` (`src/lib/projects.ts:88-241`). | Preserves current behavior on upgrade; gives operators an opt-out valve per project now that the global flag reaches every tracked repo. |
| D3 | Ready-set sourcing drops the `MERGE_GATE_VERBS` verb filter; `mergeGateEligibility` (+ `deaconIgnored` exclusion + branch-existence check) is the whole gate. | PAN-1759 already made eligibility authoritative; the verb was only the orchestrator's intent signal, needed solely because `activePipeline` was the data source. Remove `MERGE_GATE_VERBS` and the `FlywheelPipelineItem`-based wrapper if no non-test callers remain after the switch (verify with `grep -rn MERGE_GATE_VERBS src/`); update the contract comment near `FlywheelPipelineVerb` in `packages/contracts/src/flywheel.ts`. |
| D4 | No module file renames — `src/lib/flywheel-merge-order.ts` keeps its path. | Surgical-changes rule; the issue's coupling list is flag/data-source/UI, not file names. |
| D5 | Old `/api/flywheel` merge-train routes are **removed**, not aliased. | The dashboard frontend is the only consumer (verified: `MergeQueueCard.tsx:90-120`); aliases would be dead weight. `postAutoMergeSchedulePayload` and the pending-auto-merge routes are NOT merge-train routes and stay put. |
| D6 | The multi-project view lives on the **Awaiting Merge page**; the Flywheel page stays a viewer; no new top-level page. | Awaiting Merge is already the cross-project "ready work" surface; the issue names it explicitly. |
| D7 | Generation label stays the project's issue prefix (`codenameLabel`, `uat-train.ts:86`), which keeps generation names (global PK) collision-free across projects (`uat/pan-…` vs `uat/min-…`). Add a code comment stating this invariant. | `uat_generations.name` is a global PRIMARY KEY (`schema.ts:356`); prefix-distinct labels make cross-project collisions structurally impossible. |
| D8 | UAT stacks for non-PAN projects: resolve the frontend host from project `dns.domain` config with `pan.localhost` fallback; stack support beyond URL resolution is best-effort (devcontainer render already goes through the project's own template via `ensureDevcontainerSync`). | Minimal change that unblocks MYN; full non-PAN stack hardening is follow-up if a real project needs it. |
| D9 | Scope display truth: scope is baked into the run prompt at start/resume (`flywheel-actions.ts:223,294` → `flywheelRunConfigurationSection`, `src/lib/cloister/flywheel.ts:57-72`). Persist it on the run record; UI copy says "applies at next run start or resume". | Honest semantics; the running orchestrator's prompt cannot retro-change. |
| D10 | Reconciler keeps the existing `UatReconcilerDeps` interface; multi-project iteration happens in the service wiring (`uat-train.ts`), not in the pure orchestrator. | The pure layer is already per-root; only the wiring is single-project. |

## Out of scope (NonGoals)

- Cross-project UAT batches — generations remain strictly per-project; only the view/control surface spans projects.
- A new `pan merge-train` CLI verb or wrapper skill — the dashboard is the control surface; CLI exposure stays within `pan flywheel status`.
- Renaming `flywheel.require_uat_before_merge` or changing the `shouldHoldForUat` 3-tier policy resolution.
- Code-enforced multi-project flywheel inventory — scope semantics remain prompt-level instruction to the orchestrator LLM.
- PAN-1691 §5d live-cascade validation and closing the cascade bugs (operational task, separate).
- Backward-compat aliases for the removed `/api/flywheel` merge-train endpoints.
- Module/file renames (e.g. `flywheel-merge-order.ts` keeps its path).
- Hardening non-PAN UAT *stacks* beyond domain resolution (D8).

## Hazards

- **H1** Production currently runs with the merge-train flag ON; the rename (item 1) must not flip it off — the fallback read is the guarantee; verify `isMergeTrainEnabled()` returns true on an unmigrated DB in tests.
- **H2** Blast radius: after item 4, the global flag reaches every tracked project with ready work (git fetch/worktree/branch-push in *their* repos). The per-project `disabled` override (item 2) is the safety valve and the idle-skip (NFR-3) the noise guard. Review must confirm no git command runs for idle projects.
- **H3** The post-merge path must stay idempotent — `in-flight-guard.test.ts` green (NFR-1).
- **H4** Stale/cancelled tracker issues can linger in review-status records looking eligible; the `deaconIgnored` exclusion + branch-existence check are the backstops. Do not add tracker API calls to the ready-set hot path.
- **H5** `uat_generations.name` is a global PK; cross-project safety rests on label = issue prefix (D7). Keep the invariant comment next to `codenameLabel`.
- **H6** Browser verification of the dashboard changes must use an isolated Playwright browser instance/profile; do not rely on a shared logged-in session.
