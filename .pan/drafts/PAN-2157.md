# PAN-2157 — fix(flywheel): auto-merge schedule endpoint fails 'GitHub App not configured'

**Issue:** https://github.com/eltmon/overdeck/issues/2157
**Planned by:** agent:claude-opus-4-8 (`pan plan --auto`)

## Glossary

- **Auto-merge schedule endpoint** — `POST /api/flywheel/auto-merge/schedule`. When the
  Flywheel decides a ready PR may be merged autonomously, it calls this to enqueue a
  5-minute-cooldown auto-merge. Handler: `postAutoMergeSchedulePayload` in
  `src/dashboard/server/routes/flywheel.ts:293`.
- **GitHub App** — an installed GitHub App whose credentials live in
  `~/.overdeck/github-app/` (`app-id`, `private-key.pem`, `installation-id`). Used to mint
  installation tokens for GitHub REST calls. `isGitHubAppConfigured()` /
  `loadGitHubAppConfig()` in `src/lib/github-app.ts:100`/`:111`.
- **`gh` CLI fallback** — the already-established pattern: when the GitHub App is NOT
  configured, the code authenticates via the host's logged-in `gh` CLI instead. The forge
  adapter already does this for the actual merge (`mergeReviewArtifact`,
  `src/lib/forge.ts:208`).
- **Eligibility** — `isAutoMergeEligible(issueId)` in
  `src/lib/cloister/auto-merge-eligibility.ts:99`. Reads review status + live PR state to
  decide whether a PR may be auto-merged.
- **Merge backend** — whatever can authenticate a GitHub merge on this host: the GitHub App
  if configured, else the authenticated `gh` CLI, else nothing.
- **`require_uat_before_merge`** — global Flywheel toggle. When `false`, the operator has
  opted into autonomous merging (no human UAT gate before merge).

## Problem

With `flywheel.require_uat_before_merge=false`, the Flywheel's sanctioned autonomous merge
path is non-functional. `POST /api/flywheel/auto-merge/schedule` returns
`{"error":"GitHub App not configured. Run: node scripts/create-github-app.mjs"}` on a host
where the GitHub App is not provisioned (observed RUN-33, 2026-06-29, for PAN-1884 PR #2109
and PAN-2088 PR #2097 — both review+test passed, `mergeStateStatus=CLEAN`, main green).

### Verified root cause

The schedule handler `postAutoMergeSchedulePayload` (`flywheel.ts:293`) calls
`isAutoMergeEligible(issueId)` at `flywheel.ts:321`. For a GitHub PR, eligibility reads live
PR state via `defaultGetPullRequestState` →
`getPullRequestState` (the **GitHub App** Effect) at
`src/lib/cloister/auto-merge-eligibility.ts:39-41` and `:145`.
`getPullRequestState` → `generateInstallationTokenPromise` throws
`'GitHub App not configured. Run: node scripts/create-github-app.mjs'`
(`src/lib/github-app.ts:146`) when the App is absent.

Unlike the GitLab branch (`auto-merge-eligibility.ts:128-133`, which wraps the state read in
try/catch and returns `{eligible:false, reason}`), the **GitHub branch has no try/catch**
(`auto-merge-eligibility.ts:145`). The throw propagates out of `isAutoMergeEligible`, out of
`postAutoMergeSchedulePayload` (no try/catch at `flywheel.ts:321`), and the route wraps the
call in `Effect.promise` (`flywheel.ts:664`), so the rejection surfaces as the observed error
body.

### Why this is the ONLY blocker

The **actual merge** does not need the GitHub App. `triggerMerge`
(`src/dashboard/server/routes/workspaces/merge-ops.ts:440`) merges via the forge adapter
`getForgeAdapter(forge).mergeReviewArtifact(...)` (`merge-ops.ts:1141`), whose GitHub
implementation **already falls back to `gh pr merge` when `!isGitHubAppConfigured()`**
(`src/lib/forge.ts:212-225`). GitHub-App commit-status reporting in the merge path is gated
behind `isGitHubAppConfigured()` and simply skipped when absent (`merge-ops.ts:1122-1131`).
`getIssueLabels` in eligibility already uses the authenticated `gh` CLI directly and works
(`auto-merge-eligibility.ts:84-94`).

So the single missing piece is a `gh`-CLI fallback for **reading** GitHub PR state in
eligibility, mirroring the merge path's existing fallback. Once eligibility can read PR state
via `gh`, the whole schedule→cooldown→merge path works with no GitHub App.

## Proposal

1. **AC-1 (core):** Add a `gh`-CLI reader for GitHub PR state and route eligibility's PR-state
   read to it when `!isGitHubAppConfigured()`, mirroring `forge.ts:212`. (FR-1, FR-2)
2. **AC-1 (hardening):** Wrap eligibility's GitHub PR-state read in try/catch (like the GitLab
   branch) so any read failure yields a clean `{eligible:false, reason}` 422, never a 500. (FR-3)
3. **AC-2:** Add a reusable `getMergeBackendStatus()` helper and surface merge-backend
   availability on `pan flywheel status`, on a queryable dashboard endpoint + a conditional
   warning in the merge-queue card, and as a loud startup warning when
   `require_uat_before_merge=false` and no merge backend is available. (FR-4..FR-7)

This is the issue's direction #2 (alternative authenticated merge mechanism via `gh`) plus
direction #3 (loud health/startup warning). Direction #1 (provisioning the GitHub App) is NOT
pursued — the host already has an authenticated `gh` CLI, and the merge path already prefers
`gh` when the App is absent.

## Requirements

### Functional

- **FR-1** — A `gh`-CLI reader returns GitHub PR state (`GitHubPullRequestState` shape) by
  shelling `gh pr view`, and eligibility uses it when `!isGitHubAppConfigured()`.
- **FR-2** — With main green, a clean review+test-passed GitHub PR, and
  `require_uat_before_merge=false`, `POST /api/flywheel/auto-merge/schedule` returns 200 with
  a scheduled entry and never returns 'GitHub App not configured', on a host with no GitHub
  App but an authenticated `gh` CLI.
- **FR-3** — When the GitHub PR-state read throws, `isAutoMergeEligible` returns
  `{eligible:false, reason:<message>}` (caller maps to 422), rather than throwing.
- **FR-4** — `getMergeBackendStatus()` returns `{available, mode, detail}` with
  `mode: 'app'` when the App is configured, `mode: 'gh-cli'` when `gh auth status` succeeds,
  `mode: 'none'` otherwise; `available === (mode !== 'none')`.
- **FR-5** — `pan flywheel status` prints a "Merge backend:" line derived from
  `getMergeBackendStatus()`.
- **FR-6** — `GET /api/flywheel/merge-backend` returns `getMergeBackendStatus()` as JSON, and
  the dashboard merge-queue card renders a warning row when `available === false`.
- **FR-7** — On dashboard server startup, when `require_uat_before_merge === false` AND
  `getMergeBackendStatus().available === false`, a loud multi-line warning is logged
  (non-fatal).

### Non-functional

- **NFR-1** — The `gh` reader's `checksPending`/`checksFailed` match the App path's
  `getCommitCheckState` / `summarizeCiCheckRuns` semantics
  (`src/lib/github-app.ts:330`,`:278`): pending if any check run is not COMPLETED or any
  status context is PENDING; failed if any check run conclusion is in
  {FAILURE, TIMED_OUT, CANCELLED, STARTUP_FAILURE, ACTION_REQUIRED} or any status context
  state is in {FAILURE, ERROR}; SUCCESS/NEUTRAL/SKIPPED are OK.
- **NFR-2** — No behavior change on hosts where the GitHub App IS configured: the dispatch
  only changes the `!isGitHubAppConfigured()` path; the App path is byte-for-byte unchanged.
- **NFR-3** — Use `execFileAsync('gh', [...])` (argv form, no shell string) consistent with
  the existing `getIssueLabels` reader; never `execSync` (server event-loop rule).

## Work items

### b1 — gh-CLI fallback for reading GitHub PR state  (FR-1, FR-2, NFR-1, NFR-2, NFR-3)

**Files:** `src/lib/cloister/auto-merge-eligibility.ts` (+ test).

Add `getPullRequestStateViaGh(owner, repo, number): Promise<GitHubPullRequestState>` using
`execFileAsync('gh', ['pr','view', String(number), '--repo', `${owner}/${repo}`, '--json',
'state,mergeable,mergeStateStatus,isDraft,headRefOid,baseRefName,url,statusCheckRollup'])`.

Map JSON → `GitHubPullRequestState`:
- `state`: `'OPEN'` if gh `state === 'OPEN'` else `'CLOSED'` (gh `'MERGED'`/`'CLOSED'` → `'CLOSED'`)
- `merged`: gh `state === 'MERGED'`
- `mergeable`: `'MERGEABLE'`→`true`, `'CONFLICTING'`→`false`, else (`'UNKNOWN'`)→`null`
- `mergeableState`: `mergeStateStatus?.toLowerCase() ?? null`
- `draft`: `isDraft === true`
- `headSha`: `headRefOid ?? ''`
- `baseBranch`: `baseRefName ?? 'main'`
- `checksPending` / `checksFailed`: from `statusCheckRollup[]` per NFR-1 (CheckRun has
  `status`+`conclusion`; StatusContext has `state`).

Change `defaultGetPullRequestState` (currently
`auto-merge-eligibility.ts:39-41`) to dispatch:
```ts
async function defaultGetPullRequestState(owner, repo, number) {
  const { isGitHubAppConfigured } = await import('../github-app.js');
  if (!isGitHubAppConfigured()) return getPullRequestStateViaGh(owner, repo, number);
  return Effect.runPromise(getPullRequestStateEffect(owner, repo, number));
}
```

**Tests** (`src/lib/cloister/__tests__/auto-merge-eligibility.test.ts` or a new sibling):
inject a fake exec to assert the mapping (MERGEABLE→eligible, CONFLICTING→not, draft, merged,
closed, pending checks, failed checks) and that `getPullRequestState` dep can be overridden
(existing DI via `deps.getPullRequestState`).

### b2 — eligibility: contain GitHub PR-state read failures  (FR-3)

**Files:** `src/lib/cloister/auto-merge-eligibility.ts` (+ test).

Wrap the GitHub-branch PR-state read (`auto-merge-eligibility.ts:145`) in try/catch mirroring
the GitLab branch (`:128-133`):
```ts
let prState: GitHubPullRequestState;
try {
  prState = await (deps.getPullRequestState ?? defaultGetPullRequestState)(prRef.owner, prRef.repo, prRef.number);
} catch (cause) {
  const message = cause instanceof Error ? cause.message : String(cause);
  return { eligible: false, reason: `GitHub PR state lookup failed: ${message}` };
}
```

**Tests:** with `deps.getPullRequestState` rejecting, `isAutoMergeEligible` resolves
`{eligible:false, reason:/GitHub PR state lookup failed/}` and does not throw.
