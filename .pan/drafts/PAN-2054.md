# PAN-2054 — Make close-out terminal: stop closed/closed-out issues lingering in the pipeline

**Issue:** https://github.com/eltmon/overdeck/issues/2054
**Branch:** `feature/pan-2054`  •  **Base SHA at planning:** `91c90e104`
**Mode:** `pan plan --auto` (no interactive Q&A; inferred choices recorded under Auto-Decisions)

---

## Glossary

- **Close-out** — the terminal ceremony (`src/lib/close-out.ts` `executeCloseOutPromise`) that runs after a merged issue is verified on main: preserves the PRD, tears down the workspace, closes the tracker issue, applies the `closed-out` label, clears review status.
- **Durable record** — the per-issue git-tracked JSON at `<projectRoot>/.pan/records/<issue>.json` (PAN-1908). Its `pipeline` block (`PanIssuePipelineRecord`, `src/lib/pan-dir/record.ts:60`) is the **journal** — the durable source of truth for review/test/verification/merge verdicts.
- **Journal** — the `pipeline` block of the durable record, read by `readJournalStatusSync` (`src/lib/overdeck/review-status-record-sync.ts:57`).
- **DB cache** — the SQLite `review_status` row (`~/.overdeck/overdeck.db`). A disposable cache rebuilt from the journal.
- **`getReviewStatusSync`** (`src/lib/review-status.ts:707`) — the canonical read door for review status. Reconciles journal → DB on read: when the journal exists and is newer than the DB, it **re-derives** a `ReviewStatus` from the journal's `durable` fields.
- **Residue / reaping** — runtime leftovers of a closed issue: paused tmux session, workspace worktree (`workspaces/feature-<issue>/`), local+remote `feature/<issue>` branch, agent state dir (`~/.overdeck/agents/agent-<issue>/`). "Reaping" = removing them.
- **Reaper patrol** — the deacon patrol `reconcileClosedIssueAgents` (`src/lib/cloister/closed-issue-reaper.ts:77`, run from `runPatrol` at `deacon.ts:5128`) plus its reactive twin `handleIssueStatusChangedClosed` (`closed-issue-reaper.ts:35`, called from `service.ts:486`).
- **Terminal tracker state** — closed / done / canceled / completed (see `resource-discovery.ts` `isTerminalTrackerState`, ~`:638`).
- **WORKFLOW_LABELS** — the current-phase GitHub/Linear labels stripped by close paths (`src/lib/lifecycle/close-issue.ts:35`): `in-progress, in-review, planned, in-planning, needs-close-out, verifying-on-main, ready-for-merge`.

---

## Problem

A **closed + closed-out** issue keeps appearing in the Command Deck as active pipeline work because close-out is **not terminal** — it leaves runtime residue the dashboard reconstructs an active view from. Repro: **PAN-1832** (merged, GitHub CLOSED, `merged`+`closed-out` labels, spec `status=completed`) lingered for hours; `~/.overdeck/agents/agent-pan-1832` still exists today.

Four independent sources of residue, verified against `main` (2026-06-24) and re-verified at `91c90e104`:

1. **Paused agent's live tmux session** — `postMergeLifecycle` pauses the work agent (`deacon.ts:4947`); a paused agent keeps its tmux session, so `resource-discovery` `isLiveResource()` returns true. **(AC1 — already fixed: see below.)**
2. **Workspace + feature branch on disk** — `workspaces/feature-pan-1832`, local+remote `feature/pan-1832`, `~/.overdeck/agents/agent-pan-1832` survived. close-out's inline teardown (steps 3/5) did not take effect for PAN-1832 (agent re-created/re-paused after close-out, or teardown skipped).
3. **Stale durable `pipeline` block** — `.pan/records/pan-1832.json` `pipeline` is frozen pre-merge (`verificationStatus: running, mergeStatus: pending`). `getReviewStatusSync` reconciles journal→DB, so this stale block re-derives an active pipeline even after the DB row is cleared. `clearReviewStatus` (`review-status.ts:909`) only deletes the SQLite row — it never resets the journal, and close-out never resets it.
4. **Non-terminal GitHub labels** — the issue still carries `merged` and `ready`. close-out step 7 strips `WORKFLOW_LABELS` but **`merged` and `ready` are not in that list**, so they survive.

### Already shipped in this PR's base (AC1)

`src/dashboard/server/services/resource-discovery.ts` (~`:635-665`) now excludes terminal (closed/done/canceled/completed) issues from the resource tree unless they still have an **OPEN PR**. Verified: PAN-1832 no longer appears in `/api/issues/resource-allocated`. This stops the bleeding; the remaining work makes close-out *truly* terminal so the residue is removed at the source rather than filtered at the read.

## Requirements

**FR-1** — After close-out, the durable record's `pipeline` block is marked terminal: a new `closedOut: true` (+ `closedOutAt`) flag is set, `readyForMerge` is `false`, and no in-flight verification/merge status remains.
**FR-2** — `getReviewStatusSync` returns a terminal (non-active) result for a closed-out issue: it must NOT re-derive review/test/verification/merge as in-flight from a stale journal.
**FR-3** — After close-out, the issue carries none of `merged`, `ready` (in addition to the existing `WORKFLOW_LABELS`, which already includes `verifying-on-main`).
**FR-4** — A deacon patrol idempotently reaps residue (tmux session, workspace worktree, merged `feature/<issue>` branch local+remote, agent state dir) for a CLOSED issue whose branch is merged — including the pure-disk case where no agent is running (PAN-1832's exact state).
**FR-5** — A CLOSED + closed-out issue without an open PR does not appear in the Command Deck resource tree. *(Shipped — locked by a regression test here.)*

**NFR-1** — Reaping is **merge-guarded**: it reuses `isBranchMerged` (`close-out.ts:38`) and never deletes an unmerged branch or a workspace whose branch is not merged.
**NFR-2** — All new state writes go through the single write door (`src/lib/pan-dir/record.ts` writer + `writeIssueRecordSync`/`queueIssueRecordCommit`); all reads through `getReviewStatusSync`. No new direct `.pan/records` / SQLite / GitHub access is added (single-source-of-truth tenet).
**NFR-3** — The reaper stays gated by `getNoResumeMode()` (`OVERDECK_NO_RESUME=1` ⇒ no-op), matching every other recovery patrol.

---

## Proposal

Make close-out terminal at the **source of truth** and add an **idempotent, merge-guarded** reaper for residue that survives (or reappears after) close-out.

### Theme A — terminal pipeline block (FR-1, FR-2)

- **A1 (foundation):** Add `closedOut?: boolean` and `closedOutAt?: string` to `PanIssuePipelineRecord` (`record.ts:60`). Add a write-door helper `markRecordPipelineClosedOutSync(project, issueId)` in `record.ts`, modeled exactly on `clearRecordFeedbackSync` (`record.ts:520`): `ensureIssueRecordSync` → set `record.pipeline.closedOut = true`, `record.pipeline.closedOutAt = <ISO>`, `record.pipeline.readyForMerge = false`, `record.pipeline.verificationStatus = undefined`, `record.pipeline.mergeStatus = 'merged'`, `record.pipeline.updatedAt = <ISO>` → `writeIssueRecordSync` → `queueIssueRecordCommit`. Re-export from `pan-dir/records.ts` and (sync) from a place close-out can import.
- **A2:** Surface `closedOut`/`closedOutAt` in `readJournalStatusSync.durable` (`review-status-record-sync.ts:57`). In `getReviewStatusSync` (`review-status.ts:707`), when the journal reports `closedOut === true`, short-circuit: delete any stale DB row best-effort and `return null` (the issue has no active review status). This must run **before** the `journalNewer` reconcile block so a stale `verificationStatus: running` can never be re-derived.
- **A3:** In close-out `executeCloseOutPromise`, add a step (alongside step 8 "Clear review status") that calls `markRecordPipelineClosedOutSync` so the journal is reset to terminal at close-out time. Order: mark terminal **then** `clearReviewStatus` (DB), so a concurrent read can never see DB-cleared-but-journal-active.

### Theme B — strip terminal labels (FR-3)

- **B1:** In close-out step 7 (`close-out.ts:452-458`), after stripping `WORKFLOW_LABELS`, also strip the post-merge residue labels. Define `POST_MERGE_RESIDUE_LABELS = ['merged', 'ready']` (a new exported const next to `WORKFLOW_LABELS` in `close-issue.ts`, kept **separate** from `WORKFLOW_LABELS` so other in-flight callers that legitimately keep `merged`/`ready` are unaffected) and remove each (GitHub `gh issue edit --remove-label`, and the Linear branch). Keep `... || true` tolerance for labels that aren't present.

### Theme C — reap residue idempotently (FR-4, NFR-1)

- **C1 (foundation):** New module `src/lib/cloister/reap-issue-residue.ts` exporting `reapIssueResidue(projectPath, issueId): Promise<string[]>` — idempotent, merge-guarded. Steps (each best-effort, each appending an action string): (a) kill `agent-<issue>`, `test-<issue>`, `merge-<issue>` and `review-<issue>-*` tmux sessions via `killSession`; (b) **guard:** `isBranchMerged('feature/<issue>', projectPath)` — only proceed to disk/branch deletion when status is `merged` or `no-branch`; if `unmerged`, append a "skipped — branch unmerged" action and STOP; (c) `git worktree remove <workspacePath> --force` (fallback `rmSync`); (d) delete local + remote `feature/<issue>` branch; (e) `rmSync` `~/.overdeck/agents/agent-<issue>` and `planning-<issue>`. Mirrors close-out steps 3/5 but standalone and re-runnable. Uses `execAsync`/Effect `killSession` only — never `execSync` (no-execsync rule).
- **C2:** Wire C1 into the reaper. In `closed-issue-reaper.ts`: after the existing agent/session stop loops in BOTH `handleIssueStatusChangedClosed` and `reconcileClosedIssueAgents`, for each CLOSED issue call `reapIssueResidue(projectPath, issueId)`. Additionally add a **pure-disk scan** so residue with no running agent is caught: enumerate `<projectRoot>/workspaces/feature-*` dirs and `~/.overdeck/agents/agent-*` dirs, map each to an issue id, and for those that are closed call `reapIssueResidue` (the merge guard inside makes it safe). Keep the `getNoResumeMode().active` early-return (NFR-3). The merge guard (NFR-1) means an accidentally-still-open issue is never destroyed.

### Theme D — tests / regression locks (FR-1…FR-5)

- **D1:** Integration test: given a closed-out issue whose journal was stale (`verificationStatus: running`), after `markRecordPipelineClosedOutSync` + `clearReviewStatus`, `getReviewStatusSync` returns `null` (not an active state). Also assert the record `pipeline.closedOut === true`.
- **D2:** Regression test locking FR-5 in `tests/unit/dashboard/server/services/resource-discovery.test.ts`: a CLOSED + `closed-out` issue with no open PR is excluded from the resource tree; the same issue **with** an open PR is retained.

## Auto-Decisions (made in `--auto` mode)

- **AD-1 — Explicit `closedOut` marker, not merely `mergeStatus='merged'`.** A dedicated terminal flag is robust against future changes to `reviewGatesPassedSync` and gives `getReviewStatusSync` an unambiguous "not active" signal (the issue allows "reset … to a terminal state (or clear it)"). Returning `null` is the cleanest "no active pipeline".
- **AD-2 — Reaping lives in the deacon reaper patrol (extended), not only in close-out.** PAN-1832's residue survived *because an agent reappeared after* close-out ran; a one-shot teardown cannot catch that. An idempotent, merge-guarded patrol that reaps every cycle is the durable fix and auto-reaps PAN-1832 on first run — no manual operator step needed.
- **AD-3 — `merged`/`ready` stripped via a separate `POST_MERGE_RESIDUE_LABELS` const, not by extending `WORKFLOW_LABELS`.** `WORKFLOW_LABELS` is documented as *current-phase* labels and is imported by other close paths; `merged`/`ready` are post-merge/backlog labels that some in-flight callers legitimately keep, so they get their own terminal-only strip list.
- **AD-4 — close-out's existing inline teardown (steps 3/5) is left as-is; the new `reapIssueResidue` helper is additive.** Unifying the two is deferred (NonGoal) to keep these beads surgical and avoid destabilizing the working close-out path.
- **AD-5 — PAN-1832's specific disk residue is reaped automatically by the new patrol (C2) on first run; no manual teardown step is planned.** It is closed + merged, so the merge guard passes.

---

## NonGoals

- Refactoring close-out's inline teardown (steps 3/5) to call the new `reapIssueResidue` helper (dedup deferred).
- Changing `postMergeLifecycle`'s decision to *pause* the work agent at merge (the pause is intentional for verify-on-main).
- Any change to `resource-discovery` filtering beyond a regression-lock test (FR-5 is already shipped).
- Adding `merged`/`ready` to `WORKFLOW_LABELS` (would affect in-flight callers).
- Manual one-off scripts to delete PAN-1832 residue (the patrol does it).
- Touching the SQLite schema or adding new direct store access.
