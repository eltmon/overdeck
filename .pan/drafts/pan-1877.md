## Background

Multiple test files write to the developer's **real** `~/.panopticon` state because they build paths
from `homedir()` directly instead of the override-aware `getPanopticonHome()`. When such a test
crashes, is OOM-killed, or races another fork, its non-crash-safe `afterEach` restore fails and the
real state is left corrupted.

Verified offenders (built from `homedir()` + `.panopticon`, with destructive FS ops):

- `tests/lib/cloister/deacon-ci-retry.test.ts` — full-overwrites real `~/.panopticon/review-status.json`
  (`:69`, `:83`–`:84` build `join(homedir(), '.panopticon', 'review-status.json')`).
- `tests/unit/lib/cloister/pan-344-auto-merge.test.ts` — same real review-status.json.
- `tests/lib/cloister/deacon-orphan-recovery.test.ts` — `rmSync`/`writeFileSync` on real
  `~/.panopticon/agents/agent-<id>/` (`:189`, `:445`, `:516`). **Can delete real agent state.**

The source seam already exists and is correct in most code — it is the JSON module and these tests
that bypass it:

- `src/lib/paths.ts:9` — `getPanopticonHome()` returns `process.env.PANOPTICON_HOME || join(homedir(), '.panopticon')`.
- `src/lib/review-status-json.ts:17` — `const DEFAULT_STATUS_FILE = join(homedir(), '.panopticon', 'review-status.json')`
  **(the bug: uses `homedir()` directly, ignoring `PANOPTICON_HOME`).**

This is a real-impact bug even though the JSON file itself is vestigial (PAN-1883): the same test
pattern clobbers real **agent state**, and the parallel/single-fork variant is part of the red-main
test-pollution family.

---

## Glossary

- **`PANOPTICON_HOME`** — env var; when set, relocates all Panopticon state to that dir. Read by
  `getPanopticonHome()` (`src/lib/paths.ts:9`).
- **globalSetup / setupFiles** — vitest hooks that run before tests; the place to point the whole
  suite at a throwaway state dir.
- **Crash-safe isolation** — the real dir is never the target, so a crashed/raced test cannot corrupt
  it (vs. the current "overwrite real file then try to restore in afterEach").

---

## Requirements

- **FR-1** — `src/lib/review-status-json.ts` resolves its file path from `getPanopticonHome()`, not
  `homedir()`.
- **FR-2** — The entire test suite runs against a throwaway `PANOPTICON_HOME` (per-run temp dir); no
  test reads or writes the developer's real `~/.panopticon`.
- **FR-3** — Tests that build state paths from `homedir()` directly are changed to use
  `getPanopticonHome()` / the path consts so they land in the temp dir.
- **FR-4** — A regression guard fails CI if any test file builds a `~/.panopticon` path from
  `homedir()` for a write.
- **NFR-1** — Retry/backoff tests in the touched files must keep using fake timers (repo rule), not
  real `setTimeout`.

## Work items

### WI-1 — `review-status-json.ts` honors `PANOPTICON_HOME` (FR-1)

**File:** `src/lib/review-status-json.ts`.

**Current (`:12`, `:17`):**

```ts
import { homedir } from 'os';
...
const DEFAULT_STATUS_FILE = join(homedir(), '.panopticon', 'review-status.json');
```

**After:**

```ts
import { getPanopticonHome } from './paths.js';
...
// Resolved per-call so tests that set PANOPTICON_HOME after import are honored.
function defaultStatusFile(): string {
  return join(getPanopticonHome(), 'review-status.json');
}
```

Replace each `filePath = DEFAULT_STATUS_FILE` default parameter (verified at `:19`, `:30`, `:45`,
`:116`, `:121`, `:134`, `:141`, `:153`, `:164`, `:171`) with `filePath = defaultStatusFile()` so the
path is resolved dynamically at call time. Remove the now-unused `homedir` import and the
`DEFAULT_STATUS_FILE` const.

### WI-2 — Point the whole suite at a throwaway PANOPTICON_HOME (FR-2)

**File:** `vitest.config.ts` (+ a new `tests/setup/panopticon-home.ts`).

First inspect `vitest.config.ts` for an existing `globalSetup` / `setupFiles`. Add (or extend) a
**globalSetup** that, before any test module imports, sets `process.env.PANOPTICON_HOME` to a fresh
temp dir and removes it on teardown:

```ts
// tests/setup/panopticon-home.ts
import { mkdtempSync, rmSync } from 'fs';
import { tmpdir } from 'os';
import { join } from 'path';

let dir: string;
export function setup() {
  dir = mkdtempSync(join(tmpdir(), 'pan-test-home-'));
  process.env.PANOPTICON_HOME = dir;
}
export function teardown() {
  if (dir) rmSync(dir, { recursive: true, force: true });
}
```

Because `src/lib/paths.ts:6` evaluates `PANOPTICON_HOME` at module load, globalSetup (which runs
before test modules import) guarantees the path consts and `getPanopticonHome()` both resolve to the
temp dir.

### WI-3 — Fix tests that build paths from `homedir()` directly (FR-3)

In `deacon-ci-retry.test.ts`, `pan-344-auto-merge.test.ts`, and `deacon-orphan-recovery.test.ts`,
replace every `join(homedir(), '.panopticon', …)` with `join(getPanopticonHome(), …)` (import from
`src/lib/paths.js`). With WI-2 in place these resolve to the temp dir; the existing backup/restore
dances become unnecessary but may stay (now harmless).

**Repo rule restated (fake timers):** `deacon-ci-retry.test.ts` exercises CI-retry backoff. Any
delay/retry assertions MUST use `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()` — never real
`setTimeout`. If WI-3 surfaces real-timer usage, convert it.

### WI-4 — Regression guard (FR-4)

Add `tests/meta/no-real-panopticon-home.test.ts` that greps every file under `tests/` for the
pattern `homedir()` within ~2 lines of `'.panopticon'` and a write op (`writeFileSync`, `rmSync`,
`unlinkSync`, `mkdirSync`), and fails listing offenders. This prevents the pattern reappearing.
