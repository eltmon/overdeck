# PAN-1989 — Replace Pi harness with oh-my-pi (omp) and evaluate advanced features

**Issue:** https://github.com/eltmon/overdeck/issues/1989
**Planning model:** agent:claude-opus-4-8
**Status:** draft → proposed (on finalize)

---

## Glossary

- **Pi** — the *current* coding-agent harness Overdeck integrates: npm package `@mariozechner/pi-coding-agent`, binary `pi` (installed: v0.79.0), by Mario Zechner (`badlogic/pi-mono`). OAuth helpers reuse `@earendil-works/pi-ai`.
- **oh-my-pi / omp** — the *replacement*: npm package `@oh-my-pi/pi-coding-agent`, binary **`omp`** (no `pi` alias), by `can1357/oh-my-pi`. A **superset fork of the same pi-mono**, adding hashline edits, LSP/DAP tools, persistent Python + Bun worker, subagents, browser/search, ACP, and a 40+ provider registry. Config/auth under `~/.omp/agent/`.
- **Harness identifier** — the internal string union `RuntimeName`/`Harness` = `'claude-code' | 'pi' | 'codex'`, in `src/lib/runtimes/types.ts:20` and `packages/contracts/src/types.ts:49`.
- **RPC mode** — `pi --mode rpc` / `omp --mode rpc`: a headless NDJSON stdin/stdout server used by *work agents*. Overdeck feeds JSONL commands `{id,type:'prompt'|'abort',message}` through a per-agent named pipe (`~/.overdeck/agents/<id>/rpc.in`).
- **TUI mode** — interactive pane used by *conversations*; messages delivered via tmux paste-buffer, not the FIFO.
- **Vendored extension** — `packages/pi-extension` (`@overdeck/pi-extension`), loaded with `--extension <dist/index.js>`. It writes `ready.json`, heartbeats, cost events, and registers the `pan-done` slash command via the harness's extension API hooks (`session_start`, `tool_execution_end`, `turn_end`).
- **No-loss audit** — the mandatory enumerate-then-verify gate (repo rule `additive-refactor-no-loss`): list every old Pi surface item, then prove each has a home in the omp surface or is intentionally retired with rationale.
- **Drop-in track** vs **capability-research track** — the two linked tracks the issue defines: (a) replace Pi with omp preserving behavior; (b) research how omp lets Overdeck route/observe/tune *all* models better, then file follow-ups.

## Problem

Overdeck has first-class Pi support across runtime, FIFO RPC, a vendored extension, cost parsing, conversation parsing, harness routing/policy, settings/auth, docs, dashboard, and tests. We are moving to **oh-my-pi (`omp`)** — a superset fork of the same pi-mono. The runtime contract is *largely preserved* (omp keeps `--mode rpc`, the extension model, and the JSONL session shape), but **the binary name (`pi`→`omp`), the npm package, the config/auth directory (`~/.pi`→`~/.omp`), and the extension import package all change.** So this is **not** a string-rename: it requires a verified no-loss audit and real-output verification of the transcript/cost format.

Four in-flight Pi-surface defects must be explicitly dispositioned (resolved by the migration, or carried forward into the omp surface) — they must not be silently dropped:

| Defect | Symptom | Disposition |
| --- | --- | --- |
| **PAN-1859** | RED main — `agent-spawning.test.ts` "resumeAgent delivers continue prompt through Pi FIFO" fails (`writePiCommand` not called) | **Resolved** — runtime/FIFO rename bead must land a green ohmypi resume→FIFO test |
| **PAN-1833** | "Pi extension not built" when `pan start`/`pan strike` runs from a non-repo-root dir — extension detected via `process.cwd()` | **Resolved** — extension bead resolves the bundle path from the module location, not cwd |
| **PAN-1827** | Conversation view blank for pi-harness sessions — resolver handles only claude-code & codex (breaks flywheel orchestrator) | **Resolved** — conversation-parser/dispatch bead routes ohmypi + legacy-pi to a non-blank render |
| **PAN-1912** | Pi transcripts hide tool-call detail; agent panes lack the Tools show/hide toggle | **Split** — tool-call *data* surfaced in the observability bead; the frontend Tools-toggle *UI* carried forward as a filed follow-up |

## Proposal

Clean rename `pi` → `ohmypi` across the codebase (the operator's chosen strategy; matches the issue title and the in-flight `.pan`→`.overdeck` rebrand), with **read-only back-compat** for legacy `harness='pi'` rows and existing on-disk Pi session paths (the path-pattern session detector is harness-agnostic, so old transcripts still parse).

Execute as a **widen → migrate → narrow** type-rename so every intermediate commit stays green:
1. **Widen** the harness union to a transient 4-value set `'claude-code' | 'pi' | 'ohmypi' | 'codex'` + a `normalizeHarness()` legacy shim. Nothing breaks.
2. **Migrate** each subsystem (runtime, extension, launcher, parsers, auth, policy/routing, doctor, frontend) to emit/consume `'ohmypi'`, swapping the actual binary/package/paths to omp per the **contract doc** produced by the gated foundation bead.
3. **Narrow** the union back to `'claude-code' | 'ohmypi' | 'codex'`, keeping `'pi'` only in `normalizeHarness()` for legacy reads; the compiler flags any residual `=== 'pi'` site.

Because omp's upstream docs deliberately defer exact flags/hook-names/paths to its `docs/` tree, the **first bead installs omp, captures a real `--mode rpc` transcript, and records the exact pi→omp contract** (binary, package, every flag, config/auth path, extension import + hook names, real session JSONL shape vs Pi v3). Every downstream bead consumes that contract instead of guessing.

Per the operator's "parity + low-risk wins now" choice, two **bounded** low-risk wins land in this issue: (a) surface omp's additional providers Overdeck can already route to; (b) capture extra omp observability/cost fields + tool-call detail in the parsers. All higher-risk capabilities (LSP/DAP, hashline edits, subagents, model-control UIs) are deferred to filed follow-ups.

## NonGoals

- Adopting LSP/DAP tools, hashline edits, subagents, browser/search, ACP, or model-control UIs in this issue — each becomes a filed follow-up once parity is proven.
- Keeping the `pi`/`@mariozechner/pi-coding-agent` binary as a runnable harness — omp replaces it; `'pi'` survives only as a legacy *read* alias.
- Running Pi and omp side-by-side as co-equal harnesses.
- Rebuilding the frontend Tools show/hide toggle UI (PAN-1912 UI half) — carried forward as a follow-up; only the tool-call *data* is in scope here.
- Migrating existing `harness='pi'` DB rows destructively — they are normalized to `'ohmypi'` on read; the DB is a rebuildable cache.
- Changing the ToS policy posture — the single blocked cell (harness + Anthropic + subscription) is preserved, re-keyed to `'ohmypi'`.

- **FR-9** — Dashboard harness pickers, conversation rows/menus, and labels show `'ohmypi'` / "oh-my-pi".
- **FR-10** — A bounded low-risk provider expansion surfaces omp's additional routable providers in the provider registry/routing.
- **FR-11** — A bounded low-risk observability win captures extra omp usage/cost fields and tool-call detail in the parsers and cost events (PAN-1912 data half).
- **FR-12** — Docs (`configuration/harnesses.mdx`, `reference/harness-landscape.mdx`) explain installing and using omp, replacing stale Pi package/binary guidance.
- **FR-13** — A no-loss audit artifact enumerates every Pi surface item *before* replacement; a closing audit gate verifies each item is preserved, replaced, or intentionally retired with rationale, and that each of the four in-flight defects is dispositioned.
- **FR-14** — A capability-research deliverable (`docs/ohmypi-capabilities.md`) covers omp's full model/provider matrix and Overdeck improvements grouped by model-routing, role-defaults, runtime-delivery, observability/cost, dashboard-UX, and agent-reliability.

- **NFR-2** — Drop-in behavior and user-facing semantics are preserved unless a documented upstream omp incompatibility forces a deliberate change recorded in `docs/ohmypi-contract.md`.
- **NFR-3** — Tests cover the new launcher command, FIFO delivery path, resume behavior, settings/auth paths, transcript/cost parsing, and dashboard/settings harness behavior.
- **NFR-4** — No advanced omp feature beyond the two bounded low-risk wins (FR-10, FR-11) lands in this issue; the ToS gate is never weakened or reordered.
