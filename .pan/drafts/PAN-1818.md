# PAN-1818 — Role-agent context-overflow handling for review convoys

**Issue:** [PAN-1818](https://github.com/eltmon/overdeck/issues/1818)
**Title:** Review-role agents on gpt-5.5/claude-code (CLIProxy) die with 400 context errors — convoys lose a reviewer per cycle; no overflow recovery for role agents
**Planning mode:** `pan plan --auto` (non-interactive; decisions recorded below)

---

## Glossary

- **Review convoy** — the parallel set of four reviewer sub-role agents (security,
  correctness, performance, requirements) plus one synthesis (orchestrator) agent that
  Overdeck spawns to review a feature branch. Spawned by
  `spawnReviewRoleForIssuePromise` in `src/lib/cloister/review-agent.ts:296`.
- **Reviewer sub-role agent** — one convoy reviewer. tmux/agent session name is
  `agent-<issue-lower>-review-<subRole>` (e.g. `agent-pan-1815-review-correctness`).
  Created by `spawnReviewSubRoleForIssuePromise` (`review-agent.ts:219`).
- **Synthesis agent** — the convoy orchestrator, session `agent-<issue-lower>-review`
  (no sub-role suffix). Waits for `REVIEWER_READY/FAILED/TIMEOUT` signals, then writes a
  verdict. `roles/review.md`.
- **CLIProxy** — local sidecar (`127.0.0.1:8317`) that bridges a ChatGPT subscription
  OAuth token to an Anthropic-compatible `/v1/messages` endpoint, letting Claude Code run
  GPT models. `src/lib/cliproxy.ts:44-47`.
- **200k window illusion (PAN-1672)** — CLIProxy advertises gpt-5.5's 200k *marketing*
  window, but the upstream backend 400s with "input exceeds the context window" well
  before that. Claude Code's native auto-compact therefore never fires in time and the
  session hard-wedges. Conservative effective ceiling = `CLIPROXY_CODEX_CONTEXT_WINDOW`
  = 150_000 (`src/lib/model-capabilities.ts:122`).
- **Context overflow** — a terminal 400 whose tail matches `CONTEXT_OVERFLOW_PATTERNS`
  (`src/lib/context-overflow.ts:6`). Detected by `isContextOverflowTail()` (same file,
  line 13). NOT a transient error: retrying re-sends the same oversized context.
- **`REVIEWER_FAILED`** — the convoy signal the deacon sends the synthesis agent when a
  reviewer crashed/failed before writing a usable report (`deacon.ts:5903`). The synthesis
  agent treats it as blocking (`roles/review.md:80-86,114`).
- **`monitorReviewConvoySignals()`** — deacon patrol that owns reviewer lifecycle:
  detects each reviewer as ready / idle-failed / timed-out and signals synthesis
  (`deacon.ts:5788`). This is the convoy's *sole* intended lifecycle owner.
- **`checkApiErrorAgents()`** — generic deacon patrol that nudges/recovers *any*
  `agent-*`/`specialist-*`/`planning-*` session out of transient API errors and context
  overflow (`deacon.ts:964`). Work-agent-oriented recovery.

## Problem

For a large PR (#1815 was +2514/−711 over a 24-item vBRIEF), reviewer sub-role agents
running gpt-5.5 over the claude-code harness via CLIProxy overflow the real upstream
window while reading the diff + 15-20 files and die on a **400 context error**. Observed
2026-06-12: two consecutive review cycles on PR #1815 each lost a reviewer; every failed
reviewer forces a full re-review cycle and the work agent blocks 12+ min on
`pan review request` waiting for the next convoy.

Three structural gaps cause the bleed:

1. **No fast-fail on overflow.** `monitorReviewConvoySignals()` never inspects a reviewer's
   tmux tail for the 400. A reviewer that overflows sits idle until the 3-minute idle
   threshold (`REVIEWER_IDLE_FAILURE_MS`, `deacon.ts:5723`), then PAN-1806 respawns it
   **identically** against the same manifest — which deterministically re-overflows —
   before finally signalling `REVIEWER_FAILED`. ~6+ minutes burned on a guaranteed failure.

2. **Two patrols race and mis-handle the reviewer.** `checkApiErrorAgents()` treats every
   `agent-*` session as a work agent and derives the issueId by stripping `agent-` and
   uppercasing (`deacon.ts:1005-1007`). For `agent-pan-1815-review-correctness` that
   produces the garbage issueId `PAN-1815-REVIEW-CORRECTNESS`, then runs the work-agent
   compact-respawn path (`resumeAgent({compact:true})`, seeded with "run `bd ready` for
   your open beads" — meaningless for a reviewer) and can mark a non-existent workspace
   stuck. This races `monitorReviewConvoySignals()`, the actual owner.

3. **Window honesty is inconsistent in the registry.** gpt-5.5's
   `contextWindow` is correctly `CLIPROXY_CODEX_CONTEXT_WINDOW` (150k), but its `notes`
   string still claims "Effective Claude Code/CLIProxy ceiling is 200K"
   (`model-capabilities.ts:480`) — internally contradictory and nothing guards it.

The literal first ask — "cap inlined diff size in reviewer prompts" — is **already
satisfied** by PAN-1125: reviewer prompts inline only a Tier-1 summary (diff stat + top-15
risk-ranked file list), never the raw diff (`review-context.ts:25-27,194-196`). The
residual overflow vector is the reviewer's own *runtime* reads, not the spawn prompt.

## Proposal

Three surgical, harness-independent changes. None rearchitects review into chunked passes
(explicit NonGoal). This issue is the **role-agent surface**; the conversation-side window
work stays with [PAN-1672](https://github.com/eltmon/overdeck/issues/1672) and convoy
retry mechanics stay with [#1806](https://github.com/eltmon/overdeck/issues/1806).

1. **Fast-fail reviewers on overflow (ask #2, core fix).**
   - In `monitorReviewConvoySignals()`, when a reviewer has written no output for this run,
     capture its pane tail and run `isContextOverflowTail()`. On a match, signal
     `REVIEWER_FAILED <subRole> context-window overflow (no retry — deterministic)`
     **immediately and without respawning** (overflow re-occurs on the same manifest).
   - Exclude convoy reviewer sub-role sessions from `checkApiErrorAgents()` so the review
     monitor is their sole lifecycle owner — no race, no garbage-issueId work-agent recovery.

2. **Large-changeset reading guardrail (ask #1 residual).**
   - Compute a `largeChangeset` signal (file count + total changed lines) in the review
     context manifest, and surface a conditional selective-reading block in the Tier-1
     summary so a reviewer facing a huge diff reads the highest-risk files first, reads
     only changed regions, and — if it nears its context limit — writes a report on what it
     reviewed and flags the uncovered high-risk files as a **blocking coverage gap** rather
     than reading everything and overflowing.

3. **Window-honesty consistency (ask #3).**
   - Fix the contradictory gpt-5.5 `notes` string and lock
     `MODEL_CAPABILITIES['gpt-5.5'].contextWindow === CLIPROXY_CODEX_CONTEXT_WINDOW` with a
     test so the drift cannot reappear.

## Requirements

### Functional

- **FR-1** — A convoy reviewer that has written no output and whose pane tail matches a
  context-overflow pattern is signalled `REVIEWER_FAILED` (reason names "context-window
  overflow") on the next `monitorReviewConvoySignals()` pass, with **no respawn**.
- **FR-2** — `checkApiErrorAgents()` performs no recovery action (no nudge, no
  compact-respawn, no stuck-marking) on a session whose name matches
  `agent-*-review-(security|correctness|performance|requirements)`.
- **FR-3** — The review context manifest carries a large-changeset signal (file count and
  total changed lines); when the changeset exceeds the threshold, the Tier-1 summary
  reviewers receive includes a selective-reading guardrail block. Small changesets are
  unchanged (no extra block).
- **FR-4** — `MODEL_CAPABILITIES['gpt-5.5'].notes` is consistent with its 150k
  `contextWindow`, and a test asserts `contextWindow === CLIPROXY_CODEX_CONTEXT_WINDOW`.

### Non-functional

- **NFR-1** — All changes async-only and server-safe: reuse the already-imported
  `capturePane` Effect primitive; no `execSync` in deacon/server-reachable code.
- **NFR-2** — `npm run typecheck`, `npm run lint`, `npm test` pass with no new failures.
- **NFR-3** — The existing PAN-1806 idle-respawn behavior (transient API errors →
  respawn once) is preserved; only the deterministic-overflow case skips respawn.

## Work items

### WI-1 (FR-1) — Fast-fail reviewers on context overflow in `monitorReviewConvoySignals()`

**File:** `src/lib/cloister/deacon.ts` (`monitorReviewConvoySignals`, ~:5788).

Inside the per-reviewer loop, after `outputWrittenForThisRun` and `reviewerSessionAlive`
are computed and before the existing `signal` if/else ladder (`deacon.ts:5862`), add an
overflow check that runs only when no output has been written for this run:

- When `!outputWrittenForThisRun`, capture the reviewer's pane tail via
  `await Effect.runPromise(capturePane(agentId, 100))` (already imported, `deacon.ts:156`)
  and test it with `isContextOverflowTail()` (import from `../context-overflow.js`).
- If overflow is detected, set `signal = 'failed'` and
  `reason = 'context-window overflow (no retry — deterministic)'`, and ensure this branch
  is reached **before** the `idleAndNoOutput`/`respawnIdleReviewer` branch so an overflow
  is never sent down the respawn path.
- Leave the `REVIEWER_FAILED` message construction (`deacon.ts:5899-5903`) and the
  `reviewMonitorSignaled` write unchanged — the new branch just feeds them a terminal
  `failed` signal.

**Tests:** add to `src/lib/cloister/__tests__/deacon-stash-janitor.test.ts` (the file that
already exercises `monitorReviewConvoySignals`/`respawnIdleReviewer`) a case: reviewer
alive, no output, pane tail contains "input exceeds the context window" →
`messageAgent` receives `REVIEWER_FAILED ... context-window overflow ...` and
`respawnIdleReviewer` is NOT invoked.

### WI-3 (FR-3) — Large-changeset reading guardrail

**Files:** `src/lib/cloister/review-context.ts` (manifest build + `formatTier1Summary`,
~:331).

- Add a `largeChangeset` field to `ReviewContextManifest` (`review-context.ts:67`):
  `{ fileCount: number; changedLines: number; isLarge: boolean }`. Compute `changedLines`
  as the sum of `additions + deletions` over `changedFiles`, `fileCount` as
  `changedFiles.length`, and `isLarge = fileCount > REVIEW_LARGE_CHANGESET_FILES || changedLines > REVIEW_LARGE_CHANGESET_LINES`.
- Define module constants `REVIEW_LARGE_CHANGESET_FILES = 25` and
  `REVIEW_LARGE_CHANGESET_LINES = 1500` (decision D5).
- In `formatTier1Summary()`, when `manifest.largeChangeset.isLarge`, append a guidance
  block: read the highest-risk files first, read only the changed regions (not whole
  files), and if you approach your context limit write a report on what you reviewed and
  flag the uncovered high-risk files as a **blocking coverage gap** — do not attempt to
  read every file and overflow. When not large, append nothing.

**Tests:** add to `src/lib/cloister/__tests__/review-context.test.ts`: (a) a changeset over
threshold → `manifest.largeChangeset.isLarge === true` and `formatTier1Summary` output
contains the guidance block; (b) a small changeset → `isLarge === false` and the block is
absent.

## NonGoals

- Rearchitecting review into chunked / multi-pass reviewers (the "chunk" alternative in
  ask #1). Reference-by-path already exists (PAN-1125); chunking is out of scope.
- Generalizing the conservative CLIProxy ceiling to **all** routed models inside
  `computeContextUsage()` (it currently reads only the static per-model window, so only
  gpt-5.5 gets 150k there). That is conversation-side window-honesty work owned by
  PAN-1672. `getEffectiveTargetWindow()` already applies it for the model/harness switch path.
- Fixing the synthesis agent's (`agent-<issue>-review`) garbage-issueId derivation in
  `checkApiErrorAgents()`. WI-2 excludes only reviewer **sub-role** sessions; synthesis
  overflow handling is a separate latent concern not in this issue's evidence.
- Changing convoy retry counts / deadlines (#1806) or the harness default for openai
  models (PAN-1787 / PAN-1797).
- Touching `canUseHarnessSync` (ToS gate) or the `in-flight-guard` idempotency test.

## Planning Q&A (auto-mode decisions)

`pan plan --auto` was used; no interactive questions. Decisions and their defensible
defaults:

- **Q: Is "cap inlined diff size in reviewer prompts" (ask #1) still needed?**
  Options: (a) implement diff truncation/chunking; (b) treat as already-done by PAN-1125
  and address only the residual runtime-read overflow. **Chosen: (b)** — the manifest no
  longer embeds raw diff (`review-context.ts:25-27`); the spawn prompt cannot overflow.
  The residual is runtime reads, handled by a lightweight guardrail (WI-3), not a chunking
  rearchitecture (NonGoal). Principle: don't rebuild what already shipped; fix the real gap.

- **Q: On reviewer overflow, respawn (like #1806) or fast-fail without respawn?**
  Options: (a) respawn once then fail; (b) fail immediately, no respawn. **Chosen: (b)** —
  overflow on the same diff/manifest is deterministic; respawning re-overflows and burns
  ~6 min (the issue says this explicitly). #1806's respawn stays for *transient* API errors.

- **Q: Fix `checkApiErrorAgents` by excluding reviewers, or by correcting the issueId
  derivation?** Options: (a) correct the derivation so reviewers get work-agent recovery;
  (b) exclude reviewers entirely so the review monitor owns them. **Chosen: (b)** — a
  reviewer should never receive work-agent compact-respawn (its seed is bd/continue-based,
  meaningless for review). Single-owner lifecycle is cleaner and avoids the race.

- **Q: Scope of ask #3 (window honesty)?** Options: (a) generalize conservative ceiling to
  all CLIProxy models in `computeContextUsage`; (b) fix only the gpt-5.5 notes drift + a
  lock test, defer generalization to PAN-1672. **Chosen: (b)** — gpt-5.5 (the evidenced
  model) already keys proactive compaction off 150k; generalization is conversation-side
  PAN-1672 work. Proactive compaction (a ~60s patrol) cannot catch a reviewer's fast
  read-burst overflow anyway — fast-fail (WI-1) is the real protection.

- **Q (D5): Large-changeset threshold?** **Chosen: > 25 files OR > 1500 changed lines.**
  Evidence PR #1815 was 24 vBRIEF items / +2514/−711 (~3225 changed lines); the threshold
  must trigger well below that yet above ordinary small PRs. 25 files / 1500 lines is a
  defensible "large" boundary; encoded as named constants so it is tunable.
