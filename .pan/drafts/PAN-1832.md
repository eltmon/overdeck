# PAN-1832 — Role Models: multiple models per role with weighted distribution

**Issue:** [PAN-1832](https://github.com/eltmon/overdeck/issues/1832)
**Status:** Planning
**Author:** plan role (claude-opus-4-8)

## Problem

Each pipeline role (`plan`/`work`/`review`/`strike`/`flywheel`/`test`/`sequencer`)
binds to a **single** model in Settings → Roles (or a workhorse tier). When a
fleet runs many agents of one role, they all hit the same provider and exhaust
its rate limits — we have repeatedly hit gpt-5.5 rate-limit modals stalling
reviewers (#1830) and now hand-split the fleet ~30/70 gpt-5.5/kimi to spread
load. This issue makes that split first-class: a role's model may be a list of
`{model, weight}` entries, and each spawn picks one model by weighted
distribution.

## Glossary

- **Role** — a pipeline lifecycle stage (`plan`, `work`, `review`, `test`,
  `ship`, `strike`, `sequencer`, `flywheel`). Configured under `roles.<role>` in
  `~/.overdeck/*.yaml`.
- **Sub-role** — a configuration slot under a role (`work.inspect`,
  `review.security`, …). Has its own `model` field. **Out of scope** for
  distributions (stays single-model).
- **Workhorse** — a named model tier (`expensive`/`mid`/`cheap`) a role's model
  can reference as `workhorse:<slot>`. Stays single-model.
- **Model ref (`ModelRef`)** — a string: a concrete model id (`gpt-5.5`), a
  workhorse ref (`workhorse:mid`), or the `parent` sentinel (sub-roles only).
- **Distribution** — a list `[{ model: ModelRef, weight: number }, …]` assigned
  to a top-level role's `model` field.
- **Weighted (statistical) sampling** — at each spawn, deterministically map a
  per-spawn key to one entry, with probability proportional to its weight.
- **Relative weights / normalize** — weights are proportions, not required to
  sum to 100. `{70,30}` and `{7,3}` behave identically.
- **`resolveModel(role, subRole?, config, spawnKey?)`** — the single role→model
  resolver (`src/lib/config-yaml.ts:1640`).
- **`resolveHarness({model})`** — provider-default-only harness resolver
  (`src/lib/harness-resolve.ts:62`, PAN-1984). Harness derives **solely** from
  the chosen model's provider, so a per-agent model choice yields a per-agent
  harness with no extra work.
- **`determineModel({model, role, spawnKey?})`** — central work-agent spawn
  model resolver (`src/lib/agents.ts:2791`); an explicit `model` is returned
  unchanged (override), so sampling must happen once and then thread.

## Planning Q&A

**Q1 — Split exactness.** How exact should the realized split be across a fleet?
- Options: (1) Statistical hash per spawn — no new state, converges over a
  fleet; (2) Exact persistent counter — stride scheduling, new state store.
- **Answer: (1) Statistical hash per spawn.** Goal is load-spreading, not exact
  accounting; avoids a new stateful write path (single-store tenet).

**Q2 — Weight validation.** When weights don't sum to 100?
- Options: (1) Normalize (relative weights); (2) Require sum = 100.
- **Answer: (1) Normalize.** Weights are relative proportions; removes a class
  of typo/validation failures.

**Q3 — Sub-role scope.** Distributions on sub-roles too, or top-level only?
- Options: (1) Top-level roles only; (2) Roles and sub-roles.
- **Answer: (1) Top-level roles only.** Rate-limit motivation is role-fleet
  load; sub-roles run a fixed handful per issue. Smallest surface.

**Q4 — Observability.** Add an aggregated realized-split view in this issue?
- Options: (1) Existing per-agent display is enough; (2) Add a realized-split view.
- **Answer: (1) Existing per-agent display is enough.** The realized model is
  already recorded in `state.json` (`agents.ts:3315 model: selectedModel`) and
  shown per agent; an aggregate chart is a separable follow-up.

## Proposal

Make `RoleConfig.model` polymorphic: `ModelRef | WeightedModelRef[]`. Add a
deterministic stateless sampler. Sample once at the role-level spawn paths,
thread the concrete model downstream (harness then follows for free). Extend
both validation surfaces (config-load and settings API) to the list form, but
only for top-level role models. Extend the Settings → Roles picker to edit
distributions for top-level roles.

### Sampler (stateless, deterministic, no Math.random / Date.now)

```ts
// FNV-1a 32-bit over the spawn key; weights treated as relative integers.
export function pickWeightedModelRef(entries: WeightedModelRef[], spawnKey: string): ModelRef {
  const positive = entries.filter(e => e.weight > 0);
  if (positive.length === 0) throw new Error('distribution has no positive-weight entries');
  const total = positive.reduce((s, e) => s + e.weight, 0);
  const bucket = fnv1a32(spawnKey) % total;          // [0, total)
  let acc = 0;
  for (const e of positive) { acc += e.weight; if (bucket < acc) return e.model; }
  return positive[positive.length - 1].model;
}
export function representativeModelRef(entries: WeightedModelRef[]): ModelRef {
  return [...entries].sort((a, b) => b.weight - a.weight)[0]?.model;
}
```

Spawn key = `${role}:${issueId}` (one role-fleet agent per issue → varies across
the fleet; identical key → identical pick → idempotent on restart).

## Requirements

### Functional
- **FR-1** A top-level role's `model` accepts a single ref (today) OR a list of
  `{model, weight}`.
- **FR-2** At spawn, a role with a distribution resolves to one model by
  deterministic per-spawn statistical sampling.
- **FR-3** The harness follows from the sampled model via `resolveHarness`; one
  role can mix harnesses across agents.
- **FR-4** Weights are relative (normalized); they need not sum to 100.
- **FR-5** Validation (config-load + settings API) rejects malformed
  distributions (empty, non-positive/non-integer weight, invalid model ref) and
  rejects distributions on sub-roles and workhorses.
- **FR-6** The Settings → Roles picker switches a top-level role between single
  and distribution, adds/removes weighted models, and edits weights; sub-roles
  stay single-model.
- **FR-7** Back-compat: scalar role configs, YAML round-trips, and all
  display/router paths are unchanged.
- **FR-8** Sampling happens exactly once per spawn and is threaded; the realized
  model is still recorded in `state.json`; restart/resume reuse the persisted
  model (no resample).

### Non-functional
- **NFR-1** Sampling is deterministic, no `Math.random` / `Date.now`; seeded by a
  stable per-spawn key so display/router/test paths stay reproducible.
- **NFR-2** No new persistent state store (statistical hash, not a counter).
- **NFR-3** Distribution applies only to top-level roles; sub-role and workhorse
  `model` fields keep the scalar type.

## NonGoals

- No exact/stride distribution (no persistent per-role spawn counter).
- No distributions on sub-roles (`work.inspect`, `review.security`, …) or on
  workhorse slots.
- No per-model `maxAgents` governor accounting (issue's open question — deferred).
- No new aggregated "realized split" dashboard view (per-agent model already
  recorded/shown).
- No change to `resolveHarness` precedence or the ToS gate.
- No change to the resume/JSONL behavior beyond honoring the persisted model.

## Restated repo rules that intersect this work

- **No `Math.random` / `Date.now` in the sampler** — `resolveModel` is imported
  by the server and exercised by display/router/test paths; non-determinism would
  break reproducibility. Use a seeded hash of the spawn key.
- **Single source of truth** — do not add a new state store; the statistical hash
  needs none.
- **No hardcoded model fallbacks** — a distribution with no positive weight is a
  loud error, never a silent default.
- **Fake timers** — no delay logic here, but any test that ever waits must use
  `vi.useFakeTimers()`.
- **pan-style-guide** (frontend) — picker rows: no pill badges, muted text,
  reuse existing input/select classes already in RolesPanel.
- **Skills ↔ CLI lockstep** — only if a `pan` verb/flag/help changes (none
  expected here).
