## Glossary

- **pi** — the `earendil-works/pi` (a.k.a. `badlogic/pi-mono`) coding agent. Binary `pi`. Overdeck harness name `pi`.
- **oh-my-pi / omp** — Overdeck's vendored fork of pi. Binary `omp`. Overdeck harness name `ohmypi`. **Both harnesses are in scope; they share an identical extension/launcher shape** (`packages/ohmypi-extension` mirrors `packages/pi-extension`; `buildOhmypiCommand` mirrors the pi launcher).
- **Conversation** — a human-attended chat session (DB table `conversations`, tmux session `conv-<name>`), run as an interactive **TUI** so the operator can also type directly in the pane.
- **Work agent** — a pipeline implementation agent. Runs pi/omp in **`--mode rpc`** with a FIFO; **out of scope here** (already structured).
- **TUI mode** — `omp …` (no `--mode rpc`); renders the interactive terminal UI. What conversations use today.
- **RPC mode** — `omp --mode rpc`; **headless**, reads JSONL commands from stdin, no TUI. Mutually exclusive with the TUI (confirmed in pi's README). Used by work agents only.
- **Extension API** — the in-process API object passed to the vendored extension's default export (`overdeckOhmypiExtension(omp)`). Runs **alongside the live TUI**. This is the channel this PRD uses.
- **Control channel / inbox** — a per-agent on-disk drop the extension watches; the server writes typed commands into it; the extension dispatches them to `pi.*` calls. The generalization of #1958's "inbox".
- **steer** — deliver a queued message **after the current assistant turn's tool calls finish, before the next LLM call**. For course-correcting a *running* agent.
- **follow_up** — deliver a queued message **after the agent fully finishes** the current run.
- **thinking level** — pi's reasoning-effort setting: `off | minimal | low | medium | high | xhigh` (`xhigh` OpenAI-only). Maps to the dashboard composer's **effort** control.
- **ack** — an extension→server confirmation, keyed by a command `id`, that a control command was dispatched. Replaces today's "wait for the message to echo in the transcript" guess.

## Background / the bug that triggered this

An operator sent two messages to a busy GLM-5.2 / oh-my-pi conversation (agent "Working for 26s"). Both sat on **"Sending…"** indefinitely and were never confirmably delivered; the session later ended with no recoverable transcript, so the messages were lost. Cause: pi/omp **conversations deliver composer messages by tmux paste-buffer** — fire-and-forget keystrokes into the TUI pane, with no ack, no correlation id, ambiguous submit timing while the agent is mid-turn, and silent loss if the pane dies.

The fix is to drive the **pi extension API** (already loaded into every conversation) over a per-agent control channel, giving structured `steer`/`follow_up` delivery, live `setThinkingLevel`/`setModel`/`compact`, source attribution, and acks — **without losing the attended TUI**. The one capability the extension API lacks is **abort**, which gets a dedicated mechanism (Work Item 7).

## Requirements

**Functional**

- **FR-1** Operator composer messages to pi/omp conversations are delivered through the extension control channel (not tmux paste); the "Sending…" state clears on a real **ack**, not on transcript echo.
- **FR-2** A message sent while the agent is busy is delivered as **steer** (default) or **follow_up** (operator-selectable), and the agent acts on it at the correct point; no message is lost or stuck.
- **FR-3** A visible **Stop/Abort** control interrupts the current turn (mid-LLM-call or mid-tool-call) with sub-second visible feedback.
- **FR-4** The composer **effort** control changes the agent's thinking level **live** via `setThinkingLevel`, and the change is reflected/persisted for the conversation.
- **FR-5** The operator can switch the model of a running conversation live via `setModel` (extend the existing switch-model endpoint to use the channel for pi/omp).
- **FR-6** The operator can trigger a manual **compact** on a pi/omp conversation via `ctx.compact`.
- **FR-7** Messages injected by the orchestrator/another agent are tagged with a `source` distinct from operator input (folds in [#1958](https://github.com/eltmon/overdeck/issues/1958)); the agent is system-prompted how to treat extension-sourced messages.
- **FR-8** The vendored extension exposes a typed control-command dispatcher for both `ohmypi` and `pi` packages, behind a runtime-capability check that fails safe (logs + no-op) if the installed runtime lacks a method.

**Non-functional**

- **NFR-1** No `execSync` anywhere server-reachable; tmux interactions use the async primitives only (`sendKeysAsync`). (Repo rules.)
- **NFR-2** The control channel is per-agent, mode `0600`, under `~/.overdeck/agents/<id>/`, and never logs message bodies at info level.
- **NFR-3** Watchers/sockets/timers are started **only** from `session_start`, never from the extension module factory (pi extension rule, already followed at `index.ts:652`).
- **NFR-4** The tmux paste-buffer delivery path for pi/omp conversations is retired once the channel is proven; the only permitted remaining tmux write for pi/omp conversations is the abort control key (Work Item 7), and that decision is recorded in-code.
- **NFR-5** No regressions to Claude Code conversation delivery (its supervisor/eaten-message path is untouched).

## Design

```
 Composer / Stop button / effort dropdown        Orchestrator (server-side)
              │  POST /api/conversations/:name/{messages,abort,thinking-level,compact}
              ▼
   conversations.ts route handlers
              │  writeConversationControlCommand(agentId, { id, type, …, source })
              ▼
   ~/.overdeck/agents/<id>/control/<id>.json   ← per-agent inbox dir (atomic write+rename)
              │  fs.watch (started in session_start)
              ▼
   extension dispatcher  ──►  pi.sendUserMessage({deliverAs}) | pi.setThinkingLevel | pi.setModel | ctx.compact
              │  HTTP POST ack { id, ok } to dashboard (reuses existing extension→dashboard POST)
              ▼
   server resolves the pending command promise  ──►  route returns ok  ──►  UI clears "Sending…"
```

- **Transport decision:** a **file-drop inbox directory** (`control/`), one JSON file per command written via temp-file + `rename` (atomic), drained + `fs.watch`-ed by the extension from `session_start`. Chosen over a FIFO because the extension is in-process with the TUI and a dir gives natural persistence, simple draining of a startup backlog, and no open-fd/backpressure handling. (Work agents keep their FIFO; unchanged.)
- **Ack decision:** the extension **HTTP-POSTs an ack** to the dashboard (it already POSTs lifecycle signals), carrying `{ id, ok, error? }`. The server keeps an in-memory `Map<id, resolve>` so the awaiting route resolves on ack, with a timeout fallback that moves the optimistic bubble to the retry outbox (reuse the existing failSend path).
- **History plane unchanged:** the dashboard keeps parsing pi/omp JSONL for message **history** (the work-agent renderer already does this); the channel is the **control/input plane only**. This keeps the change incremental and avoids re-rendering Pi's event stream.
- **Abort is special (Work Item 7):** not in the extension API. Primary mechanism = send pi/omp's TUI **interrupt key** via `sendKeysAsync` (a single control key is reliable, unlike text paste). Verify the exact key as an implementation checkpoint.

## Decisions made in this document

- **Transport = file-drop inbox dir** (`~/.overdeck/agents/<id>/control/`), atomic temp+rename, watched from `session_start`. (Not a FIFO; not headless RPC.)
- **Ack = extension HTTP POST** to a new `control-ack` endpoint; server resolves the awaiting route; timeout → retry outbox.
- **History stays on JSONL parsing**; the channel is control/input only.
- **Abort = TUI interrupt keystroke via async tmux**, key verified at an implementation checkpoint; upstream `pi.abort()` requested in parallel; `501` if no clean key exists (never faked).
- **Both `ohmypi` and `pi` packages** get identical changes.
- **Claude Code conversations are untouched.**
- **Optional extension methods** so a stale runtime degrades gracefully rather than throwing.

## Intersecting repo rules (restated for the executor)

- **Async tmux only:** use `sendKeysAsync` / async primitives from `src/lib/tmux.ts`; never `sendKeys` (sync). (WI-7.)
- **No `execSync`** in any server-reachable code.
- **Fake timers** for any retry/backoff/timeout test (WI-4, WI-7): `vi.useFakeTimers()` + `vi.advanceTimersByTimeAsync()`; never real `setTimeout` delays.
- **Extension watchers start in `session_start`, never the module factory** (NFR-3).
- **JSONL session files are sacred** — the control channel and acks live under `control/`; never touch `sessions/*.jsonl`.
- **Surgical changes:** do not refactor the claude-code delivery path; touch only what each work item names.
