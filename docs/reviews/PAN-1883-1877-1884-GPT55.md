## Executive take

The three drafts are much stronger than normal "cheap executor" briefs: they have good terminology, concrete anchors, before/after snippets, and acceptance criteria. I would still tighten them before dispatching implementation.

My main concern is [PAN-1877](https://github.com/eltmon/panopticon-cli/issues/1877): the current draft fixes the obvious `review-status.json` path, but its stated goal is "no tests write real `~/.panopticon`", and the codebase has more real-home writes than the three listed offenders. A cheap executor following the draft literally can pass the named checks and still leave real `~/.panopticon` mutation in place.

[PAN-1883](https://github.com/eltmon/panopticon-cli/issues/1883) is directionally correct. SQLite/API is the live source of truth, and `review-status.json` is legacy/test scratch. The UI work item is underspecified at the type/design-system boundary: adding `MERGE BLOCKED` and `CI BLOCKED` is not only a `ProjectOverview.tsx` helper change; it also requires extending the `VerbBadge` variant contract and tests.

[PAN-1884](https://github.com/eltmon/panopticon-cli/issues/1884) is useful, but it mixes tracked repo changes with local Claude memory mutation. The draft should separate those execution phases, update the startup brief as well as the role prompt where contradictions exist, and make the "commit immediately" rule apply only to tracked repo changes.

## PAN-1877 review

### What I verified

The named bug exists:

- `src/lib/review-status-json.ts` builds `DEFAULT_STATUS_FILE` from `homedir()` and `.panopticon`, ignoring `PANOPTICON_HOME`.
- It uses that default in multiple function signatures, so the default must become a function call or each function must resolve the path dynamically.
- `tests/lib/cloister/deacon-ci-retry.test.ts` still hardcodes and writes the real `~/.panopticon/review-status.json`.
- `tests/unit/lib/cloister/pan-344-auto-merge.test.ts` hardcodes the real path, though its file-level comment says `fs` is mocked to intercept those writes.
- `tests/lib/cloister/deacon-orphan-recovery.test.ts` writes real `~/.panopticon/agents/...` paths.

The `getPanopticonHome()` direction is right. `src/lib/paths.ts` exposes a dynamic getter, and this is the right primitive for code that must honor test-time `PANOPTICON_HOME` changes.

### Major gap: the offender list is incomplete

I found additional tests that appear to touch real `~/.panopticon` today or rely on production modules that do:

- `tests/lib/shadow-state.test.ts`
  - Uses `join(homedir(), '.panopticon', 'shadow-state')`.
  - Calls real shadow-state functions.
- `tests/lib/shadow-mode.test.ts`
  - Same real shadow-state directory pattern.
- `tests/unit/dashboard/server/routes/show.test.ts`
  - The file comment explicitly says it writes unique-prefix files to real `~/.panopticon/shadow-state`.
- `tests/lib/status-context.test.ts`
  - Uses `join(homedir(), '.panopticon', 'agents')` and writes agent context marker files.
- `src/dashboard/server/routes/__tests__/conversations-supervisor.test.ts`
  - Sets `PANOPTICON_HOME`, but its helper still builds `~/.panopticon/conversations/...` from `homedir()`, and cleanup touches real `~/.panopticon/agents/...`.

The production-side issue is not limited to `review-status-json.ts`.

Examples:

- `src/lib/shadow-state.ts` hardcodes `join(homedir(), '.panopticon', 'shadow-state')`.
- `src/lib/review-status.ts` still has a `DEFAULT_STATUS_FILE` built from `homedir()`, even though the live default path no longer uses the JSON backend.
- Other runtime modules also contain direct `homedir(), '.panopticon'` paths.

If [PAN-1877](https://github.com/eltmon/panopticon-cli/issues/1877)'s FR-2 is "no test may write the real `~/.panopticon` tree", the PRD must include the shadow-state and agent-state offenders, not only review-status JSON.

### Suite isolation plan needs more rigor

A single temp `PANOPTICON_HOME` in `globalSetup` protects the real home only for code that honors `PANOPTICON_HOME`. It does not isolate tests from each other inside the same suite. With Vitest parallelism, one temp home per entire run can still allow cross-file pollution.

I would change WI-2 to one of these explicit strategies:

1. Global safe root plus per-worker home:
   - `globalSetup` creates `pan-test-root-*`.
   - `tests/setup.ts` sets `PANOPTICON_HOME` to `${root}/worker-${VITEST_POOL_ID}` before test file imports, after verifying Vitest setup-file ordering.
   - Each worker gets an isolated DB/state tree.

2. Global safe root only, but broaden the work to remove cached home constants:
   - Convert stateful modules from top-level `PANOPTICON_HOME` constants to dynamic getters where tests need env changes.
   - Accept that tests share one suite home, but make that an explicit non-goal.

Because `src/lib/paths.ts` caches `PANOPTICON_HOME` at import time while also exposing `getPanopticonHome()`, the PRD should be clear about which modules are still safe to import before env setup and which are not.

If the PRD wants to say it likely resolves [PAN-1720](https://github.com/eltmon/panopticon-cli/issues/1720), add tmux socket isolation to the scope or explicitly say "this fixes the real-home file writes; tmux socket isolation remains separate." The [PAN-1720](https://github.com/eltmon/panopticon-cli/issues/1720) issue comments mention both.

### The regression guard is too weak

The proposed static guard, "homedir within about 2 lines of `.panopticon` and a write op", will miss common real writes:

- A test calls a production helper that hardcodes real home.
- The path constant is far from the write call.
- The write uses async `writeFile`, `mkdir`, `rm`, `rename`, or a helper wrapper.
- A module caches a real path at import time before `PANOPTICON_HOME` is set.

I would add a runtime guard in `tests/setup.ts` that wraps the relevant `fs` and `fs/promises` write APIs and throws if the target path is under the real `${homedir()}/.panopticon`, unless the test has an explicit allowlist. Keep the static grep as a second line of defense, but do not rely on it as the main proof.

The runtime guard should cover at least:

- `writeFileSync`, `appendFileSync`, `mkdirSync`, `rmSync`, `renameSync`, `cpSync`
- `fs.promises.writeFile`, `appendFile`, `mkdir`, `rm`, `rename`, `cp`
- Potentially `createWriteStream` if tests use streams for state.

### Acceptance changes I would add

- A test that intentionally attempts to write `${homedir()}/.panopticon/pan-test-guard` fails under the test guard.
- Shadow-state tests write under the temp `PANOPTICON_HOME`, not real home.
- Agent-state/context tests write under the temp `PANOPTICON_HOME`, not real home.
- `stat` or `find` checks before/after the relevant test files show no mtime or content changes under real `~/.panopticon/review-status.json`, `~/.panopticon/shadow-state`, `~/.panopticon/agents`, or `~/.panopticon/conversations`.
- The PRD either removes the "likely resolves [PAN-1720](https://github.com/eltmon/panopticon-cli/issues/1720)" claim or includes tmux socket isolation in the implementation and acceptance criteria.

### Gaps to fix in the PRD

Separate tracked repo changes from local memory mutation.

The bundled rules and role docs can be committed. The Claude memory files under `/home/eltmon/.claude/.../memory` are local machine state and are not part of the repo. The "commit immediately" rule should be rewritten as:

- Commit the tracked repo changes immediately after they build and `pan sync` verifies rendered context.
- Then slim/delete local memory files and update `MEMORY.md`.
- Do not imply the local memory mutation itself is captured by the git commit unless there is a separate memory persistence mechanism.

Update both flywheel instruction surfaces, not only `roles/flywheel.md`.

If flywheel startup reads `docs/flywheel-brief.md` as the scope contract, adding rules only to `roles/flywheel.md` leaves the agent with contradictory instructions. The PRD should explicitly update:

- `roles/flywheel.md`
- `docs/flywheel-brief.md`

for:

- wakeup cadence
- red-main merge gate
- admin-merge override limits
- closing merged/verifying-on-main issues, if the current contradiction is in scope

Be careful with rule wording so new bundled rules do not conflict with existing bundled rules.

Specific risky spots:

- `handoff-not-pipeline-agents` must not contradict the existing rule that work/plan/review/test/ship agents run through `pan`. It should say ad hoc conversation handoffs use `pan handoff`; managed work agents still use `pan start` / `pan swarm` / `pan plan`.
- `no-oneshot-agents` should match current platform reality. The memory says to stop and surface when an operator asks for a Codex-harness work agent because current codex work agents run one-shot. If there is no reliable TUI path today, the new rule should say "do not spawn; surface to operator" rather than implying `codex` TUI is already available and wired.
- `facts-not-guessing` and `explain-fully-no-jargon` are broad behavior rules. They may be worth keeping, but they add recurring context. Consider combining small style memories where one rule can carry the actionable behavior.

The red-main/admin-merge rule should be mechanically checkable.

I would include exact verification commands or required data fields, for example:

- Inspect the latest main CI run with status, conclusion, head SHA, URL, and created time.
- Treat `status != completed` as not green.
- Treat missing/unknown conclusion as not green.
- Do not admin-merge a PR with failing checks merely because the operator wants progress; if main is green and the PR is red, inspect the PR failure first and require explicit operator override for that specific failure.

`pan sync` may write outside the repo.

The PRD says to verify:

- `~/.claude/CLAUDE.md`
- `~/.panopticon/context/pi-global.md`
- `~/.panopticon/context/codex-global.md`

That is correct functionally, but in a sandboxed Codex environment it may require approval because those paths are outside the repo. Add that as an implementation note so a cheap executor does not treat a permission failure as a code failure.
