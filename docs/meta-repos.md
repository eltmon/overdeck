# Meta Repos

**Pattern for managing shared skills, documentation, and team conventions across a polyrepo project.**

---

## Overview

A meta repo is a dedicated repository that contains everything an AI agent needs to work effectively in your project:
- Shared skills and commands
- Architecture documentation
- Team conventions (injected into every workspace)
- Onboarding materials for new team members

For large polyrepo projects (10+ repos), the meta repo is the single source of truth for agent configuration.

---

## Why Use a Meta Repo?

Without a meta repo:
- Each workspace has different or missing agent instructions
- Conventions are undocumented or scattered across repos
- Onboarding new team members requires manual setup

With a meta repo:
- Every workspace gets consistent agent configuration
- Conventions are centralized and version-controlled
- New team members can self-onboard from documentation

```

│       ├── CLAUDE.md             # Project conventions for agents

│       └── README.md             # Getting started guide
└── README.md
```

## .agent-template/

The `.agent-template/` directory is copied into every workspace created in the project. It provides the agent with project-specific context.

### CLAUDE.md

The main agent instruction file. Include only non-obvious conventions:

```markdown
# Project Conventions

## PR Conventions
All PRs target the `qa` branch, NOT main/master.

## Per-Repo Notes
- `api-service/` — Each microservice has its own package.json
- `providers/*/` — Provider integrations, don't modify directly

## Architecture
This is a Lambda-based microservices architecture. Each provider
has its own repo under `providers/`.

## What NOT to Do
- Never run `terraform apply` without approval
- Never modify provider repos directly — use the bridge repos
```

### Skills

Team-specific skills live in `.agent-template/.claude/skills/`:

### machine-setup.sh

Automated setup script for new machines:

```bash
#!/bin/bash
# Auto-generated by Panopticon Setup Agent

echo "=== Team Onboarding Setup ==="

# Check prerequisites
command -v git >/dev/null || { echo "Install git first"; exit 1; }
command -v gh >/dev/null || { echo "Install GitHub CLI"; exit 1; }
command -v node >/dev/null || { echo "Install Node.js 20+"; exit 1; }

# Authenticate GitHub CLI
gh auth status || gh auth login

# Clone repos
REPOS=(
  "org/int-micros-simphony"
  "org/int-provider-a"
)

for repo in "${REPOS[@]}"; do
  name=$(basename "$repo")
  if [ ! -d "$name" ]; then
    gh repo clone "$repo" "$name"
  fi
done

echo "Done! Run 'pan status' to verify setup."
```

## How Meta Repo Changes Propagate

### When a Workspace is Created

1. Panopticon reads `.agent-template/` from the meta repo
2. Copies contents to the new workspace's `.claude/`
3. Workspace has the latest meta repo contents

### Updating Existing Workspaces

Since workspaces are git worktrees, they don't automatically receive meta repo updates. To update:

1. **Sync the meta repo** in the workspace:
   ```bash
   cd team-meta
   git pull origin main
   ```

2. **Re-copy the agent template**:
   ```bash
   cp -r team-meta/.agent-template/.claude/ .claude/
   ```

---

## Symlinking the Meta Repo

For progressive polyrepo projects, the meta repo is typically symlinked rather than worktreed:

```yaml
# In projects.yaml
repos:
  - name: meta
    path: team-meta
    link_type: symlink    # Symlink, not worktree
    readonly: true        # Agent doesn't commit to meta repo
```

Benefits:
- Always reflects the latest state of the meta repo
- No feature branch management for the meta repo
- Agent can read but not modify

## Onboarding Flow

### New Team Member

1. Clone the meta repo
2. Run `machine-setup.sh` to install prerequisites and clone repos
3. Run `pan setup` to register the project with Panopticon
4. Run `pan workspace create` to test workspace creation
5. Ready to work on issues

### Existing Team Member

1. Pull latest from meta repo
2. Update workspace's `.claude/` from meta repo if needed
3. Continue working
