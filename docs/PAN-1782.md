# PAN-1782 — Handoff forks stall at "Injecting…" then die on double 300s summary timeout

**Issue:** https://github.com/eltmon/panopticon-cli/issues/1782
**Diagnosed in:** conversation 2734 (`pan.localhost/conv/2734`), 2026-06-11
**Status:** solution approved in principle ("separate precompactor" shape); not yet implemented

## Incident

Two handoffs of source conversation `20260611-0b30` — one with a GPT 5.5 handoff
author, one with GPT 5.4-mini — sat at "Injecting…" in the Command Deck for many
minutes, then failed. No target tmux session, no handoff document, no fallback
conversation. The server logs show both attempts:

1. Read a **748,262-character** source transcript.
2. Started smart-compaction pre-summary **with the user-selected GPT model**.
3. Died with `Smart summary generation failed: Summary generation timed out after 300000ms`.
4. Fell back to a summary fork, which ran **another** LLM smart summary over the
   same oversized input, and timed out again — killing the fork entirely.

## Root cause — three stacked defects

### 1. "Injecting…" is a fabricated label (UI/state defect)

Handoff fork rows are created with the fork *mode* string as their progress
*status*:

- `src/dashboard/server/routes/conversations.ts:4282` —
  `forkStatus: forkMode === 'plain' ? 'spawning' : forkMode === 'handoff' ? 'handoff' : 'summarizing'`
- `src/dashboard/frontend/src/components/CommandDeck/ConversationRow.tsx:202` —
  renders any status that isn't `summarizing`/`spawning` as **"Injecting..."**,
  so the unknown `handoff` status displays as injection.
- `src/dashboard/frontend/src/components/chat/ConversationPanel.tsx:944-956` —
  `ForkProgressView` computes `activeIdx = -1` for the `handoff` status, so the
  3-step timeline (Summarizing → Spawning → Injecting) shows no active step,
  and on failure marks step 0 ("Summarizing") failed regardless of what
  actually failed.

The status never changes during the entire authoring phase — `runForkPipeline()`
only writes `spawning` *after* authoring succeeds (`conversations.ts:4005`), and
`injecting` after the session spawns (`conversations.ts:3840`). So a 5-minute
authoring phase reads as a 5-minute "Injecting…" stall.

### 2. The handoff author model is forced to do precompaction (the actual timeout)

`authorHandoffExternal()` in `src/lib/conversations/summary-fork.ts:214` handles
long transcripts (> `HANDOFF_TRANSCRIPT_COMPACT_THRESHOLD` = 100k chars) by
precompacting via `sourceAdapter.compactSummary()` — but passes
`effectiveModel`/`effectiveHarness`, i.e. **the user-selected handoff author
model** (lines 257–266). A GPT model routed through CLIProxy summarizing a 748k-char
transcript cannot finish inside the 300s `runModelSummary` timeout. The user's
model choice was meant to pick *who writes the handoff document*, not *who
performs the hidden bulk-summarization step*.

### 3. The fallback path can fail exactly the same way (no last-resort degradation)

When authoring throws, both fallback sites re-run an LLM smart summary:

- `runForkPipeline()` → `buildSummary()` → `generateSummaryForFork()`
  (`conversations.ts:3936-3942`, used at 3962-3967, 3971-3975, 3981-3986)
- `createSummaryForkPromise()` → `generateSummarySeed()`
  (`summary-fork.ts:699-706`, 712-719, 729-736)

If that summary also times out (same oversized transcript, and in the legacy
path even the same model — `summaryModel = options.model || conv.model`), the
exception propagates and the whole fork fails. The cheap local heuristic
summarizer `generateFallbackSummaryPromise()` (`summary-fork.ts:445`) is never
tried unless `localSummaryOnly` was set up front.

## Decision

Operator-approved shape (AskUserQuestion in conv 2734): **Separate
precompactor** — keep the explicit user-selected model as the handoff *author*,
but run the hidden precompaction step on the selected summary model or a safe
large-context default. If LLM summarization still fails during handoff
fallback, degrade to the local heuristic summary instead of failing the fork.

Rejected alternative: "same model everywhere" with smaller GPT chunks and
better timeouts — preserves one-model semantics but stays slow/expensive and
keeps the failure mode in play.

## Implementation plan

### 1. Decouple precompaction from the handoff author model

`src/lib/conversations/summary-fork.ts`:

- Add a constant next to the existing handoff constants (line ~185):

  ```ts
  // Long transcripts are precompacted before authoring. The precompactor must
  // be a large-context summarizer — the user's handoff author model (e.g. a
  // GPT model behind CLIProxy) routinely times out summarizing ~750k-char
  // transcripts (PAN-1782). The author model still writes the final doc.
  const DEFAULT_HANDOFF_PRECOMPACT_MODEL = 'claude-sonnet-4-6';
  ```

- Extend `authorHandoffExternal()`'s `options` parameter:

  ```ts
  options: { now?: Date; precompactModel?: string; precompactHarness?: RuntimeName } = {}
  ```

- In the long-transcript branch (lines 257–269), call
  `sourceAdapter.compactSummary()` with
  `model: options.precompactModel ?? DEFAULT_HANDOFF_PRECOMPACT_MODEL` and
  `harness: options.precompactHarness ?? 'claude-code'` instead of
  `effectiveModel`/`effectiveHarness`. Keep the final authoring
  `runModelSummary(prompt, effectiveModel, …, effectiveHarness, ['Write'])`
  call unchanged — the user's model still authors the document.

- In `runForkPipeline()` (`src/dashboard/server/routes/conversations.ts:3953`),
  thread the route's summary selection into the override:

  ```ts
  authorHandoffExternal(parentConv, parentSessionFile, handoffFocus,
    handoffAuthorModel, handoffAuthorHarness,
    { precompactModel: summaryModel, precompactHarness: summaryHarness })
  ```

  `summaryModel`/`summaryHarness` are `undefined` unless the user explicitly
  picked them, so the safe default applies otherwise. The legacy CLI path
  (`createSummaryForkPromise`) passes no override — default precompactor.

### 2. Last-resort local fallback for handoff failures

`src/lib/conversations/summary-fork.ts` — add and export:

```ts
export async function generateHandoffFallbackSeed(
  sourceSessionFile: string,
  summaryModel: string | undefined,
  localSummaryOnly: boolean | undefined,
  includeThinkingInSummary: boolean | undefined,
  summaryHarness?: RuntimeName,
  sourceHarness?: RuntimeName,
): Promise<{ summary: string; summaryModel: string | null }> {
  try {
    return await generateSummarySeed(sourceSessionFile, summaryModel,
      localSummaryOnly, includeThinkingInSummary, summaryHarness, sourceHarness);
  } catch (error) {
    console.warn(`[summary-fork] handoff-fallback LLM summary failed (${msg(error)}) — using local heuristic summary`);
    return { summary: await generateFallbackSummaryPromise(sourceSessionFile), summaryModel: null };
  }
}
```

Use it **only** in handoff fallback sites (a summary-mode fork's LLM failure
should still fail loudly):

- `createSummaryForkPromise()` — the three `generateSummarySeed` calls at
  `summary-fork.ts:703`, `:717`, `:733`.
- `runForkPipeline()` — the three `buildSummary()` calls in the handoff branch
  (`conversations.ts:3966`, `:3975`, `:3985`); the summary-mode call at `:3990`
  keeps `buildSummary()`.

`forkFallbackReason` (`handoff-timeout` / `handoff-validation` /
`handoff-request-failed`) and `prependFallbackFocus()` behavior are unchanged —
the successor conversation still sees why it got a summary instead of an
authored handoff.

### 3. Honest fork progress status

- **Server:** initialize handoff rows with a real phase status —
  `conversations.ts:4282` becomes
  `forkMode === 'plain' ? 'spawning' : forkMode === 'handoff' ? 'authoring' : 'summarizing'`.
  In `runForkPipeline()`'s handoff fallback paths, write
  `updateForkStatus(convName, 'summarizing')` when entering the summary
  fallback so the UI tracks the real phase.
  Recovery compatibility is free: `getStuckForks()`
  (`src/lib/database/conversations-db.ts:696`) matches any non-null,
  non-`failed` status, so both new `authoring` rows and legacy `handoff` rows
  remain recoverable. Nothing else in live code compares
  `forkStatus === 'handoff'`.

- **`ConversationRow.tsx:202`:** replace the implicit "anything else is
  Injecting" ternary with an explicit map; unknown statuses display as
  "Preparing...", never "Injecting...":

  ```ts
  const FORK_STATUS_LABELS: Record<string, string> = {
    summarizing: 'Summarizing...',
    authoring: 'Authoring handoff...',
    handoff: 'Authoring handoff...',   // legacy rows mid-flight during deploy
    spawning: 'Spawning...',
    injecting: 'Injecting...',
  };
  ```

- **`ConversationPanel.tsx` `ForkProgressView`:** map statuses to step indices
  explicitly (`summarizing`/`authoring`/`handoff` → 0, `spawning` → 1,
  `injecting` → 2) instead of `FORK_STEPS.findIndex`, and when the live status
  is `authoring`/`handoff`, render step 0 as "Authoring — writing a handoff
  document from the source conversation" instead of "Summarizing". Unknown
  statuses must not render as a failed/empty injection step.

### 4. Regression coverage

- New test file `src/lib/conversations/__tests__/summary-fork-precompact.test.ts`
  (separate file so it can `vi.mock('../transcript-adapter.js')` without
  disturbing the existing handoff tests that rely on the real adapter):
  - **Precompaction decoupling:** mock adapter returns a >100k-char
    `serializeTranscript`; assert `compactSummary` is invoked with
    `claude-sonnet-4-6`/`claude-code` (and with the override when
    `precompactModel` is passed), while the authoring `runModelSummary` call
    still receives the requested GPT author model, with the compact summary —
    not the raw transcript — in the prompt.
  - **Double-failure degradation:** authoring fails AND the mocked
    `compactSummary` throws (simulated timeout); assert `createSummaryFork()`
    still returns a `summary`-mode fork seeded by the local heuristic
    (`## Conversation Summary Fork`), with `summaryModel: null` and the
    original `forkFallbackReason` + focus preserved.
- Extend `src/lib/conversations/__tests__/summary-fork-handoff.test.ts` only if
  signatures shift (existing 5-arg `authorHandoffExternal` calls keep working —
  the new parameter is the existing optional `options` object).
- Route/recovery: extend
  `src/dashboard/server/routes/__tests__/conversations-fork-recovery.test.ts` —
  legacy `fork_status='handoff'` rows already covered (lines 223, 251, 283);
  add an `authoring` row case. Add/extend a route test asserting new handoff
  rows are created with `fork_status='authoring'`.

   fallback conversation with a `Fallback:` badge rather than a dead fork.

## Non-goals

- No change to the 300s `runModelSummary` timeout itself — with a
  large-context precompactor it is sufficient, and raising it only delays
  honest failure.
- No GPT-side chunk-size tuning (rejected "same model everywhere" option).
- No backfill/migration of persisted legacy `handoff` status rows — the
  recovery path and the legacy UI label mapping handle them until they drain.
