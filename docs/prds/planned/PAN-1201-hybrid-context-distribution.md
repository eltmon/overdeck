# PAN-1201 — Hybrid Context Distribution

**Issue:** [PAN-1201](https://github.com/eltmon/panopticon-cli/issues/1201)
**Parent epic:** [PAN-1200](https://github.com/eltmon/panopticon-cli/issues/1200)
**Status:** Planned
**Date:** 2026-05-18

---

## Problem

`sync.devroot` in Panopticon's config is the gate that controls whether skills/agents/rules get distributed to `~/.claude/skills/`. For most users this field is unset, so `pan sync` runs but doesn't actually push anything. The original design also assumed every project lives under one root directory (`~/Projects/`), which is no longer true:

- Krux lives at `~/Projects/krux`
- Tap2Open could live anywhere a consulting checkout puts it
- Battle of Words was recovered into `~/Projects/BoW` from an unrelated source
- Future projects (clients, archives) shouldn't be forced under `~/Projects/`

Additionally, the existing model only knows about Claude Code's locations. Pi, Codex, Cursor, Gemini — each has its own conventions and Panopticon writes nothing to them.

## Goal

Replace `sync.devroot` with a layered context model where:

1. Projects can live anywhere on disk (identified by registration, not filesystem layout)
2. Three layers of context compose at sync time: global, project, workspace
3. A single canonical source per layer renders harness-specific outputs (CLAUDE.md, AGENTS.md, etc.) via templating
4. The dashboard provides a Context tab for editing layers with live per-harness preview

## Design Goals

- **Single source of truth per layer** — write once in markdown, render N times for N harnesses
- **Registration over location** — `pan projects add <path>` is the contract; no filesystem assumption
- **Backwards compatible migration** — old `~/Projects/.claude/` contents can be moved without losing anything
- **Cheap to extend** — adding a new harness is implementing one adapter, not redesigning the model

## Architecture

### Three Layers

| Layer | Location | Writer | When Applied |
|---|---|---|---|
| Global | `~/.panopticon/context/global.md` + `~/.panopticon/context/global/{skills,agents}/` | User (CLI or dashboard) | Every harness invocation, everywhere |
| Project | `<projectRoot>/.panopticon/context/project.md` + sibling dirs | User; lives in project repo | When CWD is anywhere under a registered project |
| Workspace | `<workspace>/.panopticon/context/workspace.md` | Auto-generated by Panopticon at spawn | Inside that specific issue workspace |

The workspace layer is composed at spawn time from:

1. The project layer of the workspace's parent project
2. PAN-1052 memory injection (`MEMORY_CONTEXT` block)
3. Issue-specific metadata (title, vBRIEF summary, current phase)
4. Live workspace status (from PAN-1052's status.json)

### Templating

Single canonical markdown per layer. Harness-specific divergence via Mustache-style blocks:

```markdown
# Project Rules

All agents must follow the Panopticon engineering philosophy: no bandaids, fix root causes.

{{#harness:claude}}
When using TodoWrite, prefer the beads CLI (`bd`) per CLAUDE.md guidance.
{{/harness:claude}}

{{#harness:pi}}
Pi RPC mode: write completion markers via `/pan-done` when ready for review.
{{/harness:pi}}

{{#harness:claude}}{{#harness:pi}}
Shared: never commit secrets. Never use --no-verify.
{{/harness:claude}}{{/harness:pi}}
```

Blocks are processed per-harness during render. Unrecognized blocks pass through (forward compatibility with future harnesses).

### Sync Output Map (v1)

| Layer | Harness | Output File |
|---|---|---|
| Global | claude-code | `~/.claude/CLAUDE.md` + `~/.claude/skills/<sym>` + `~/.claude/agents/<sym>` |
| Global | pi | `~/.claude/skills/<sym>` (Pi reads Claude's per current HARNESSES.md) — revisit if Pi adds its own location |
| Project | claude-code | `<projectRoot>/CLAUDE.md` (composed: project layer rendered for Claude) |
| Project | pi | Same as above; Pi uses project CLAUDE.md if present |
| Workspace | claude-code + pi | `<workspace>/CLAUDE.md` + `<workspace>/.panopticon/context/workspace.md` |

The workspace `CLAUDE.md` is the assembled bundle (project layer + memory + status). The `workspace.md` file is the same content, harness-neutral, used by the briefing system in PAN-1204.

### Project Registration

`pan projects add <path>` already exists for the projects.yaml registry. Extend it so registration triggers:

1. Create `<path>/.panopticon/context/` if missing, with a stub `project.md`
2. Add `<path>/.panopticon/context/` to git's expected committed paths (project.md is committed; workspace.md is gitignored)
3. Update `~/.panopticon/projects.yaml` with the absolute path

`pan sync` reads from this registry — not from `~/Projects/`.

### Migration

`pan context migrate` is a one-shot:

1. Detect `~/Projects/.claude/` if it exists
2. Copy `~/Projects/.claude/CLAUDE.md` (or the per-project ones it finds) → `~/.panopticon/context/global.md`
3. Copy `~/Projects/.claude/skills/` → `~/.panopticon/context/global/skills/`
4. Copy `~/Projects/.claude/agents/` → `~/.panopticon/context/global/agents/`
5. Discover projects under `~/Projects/` and register each via `pan projects add <path>` (interactive: prompt y/n per project)
6. Print "old location preserved at `~/Projects/.claude/` — delete with `rm -rf ~/Projects/.claude` when ready"
7. **Does NOT auto-delete** the source

`sync.devroot` in `~/.panopticon/config.yaml` is read at startup. If set, a deprecation warning is printed:

```
[WARN] sync.devroot is deprecated. Run `pan context migrate` to move to the layered model.
       Old location: ~/Projects/.claude/
       New location: ~/.panopticon/context/global/
       Set sync.devroot to null in config.yaml to silence this warning after migration.
```

## CLI Surface

```
pan context list                    # Show all three layers' files
pan context list --layer global     # Just one layer
pan context edit                    # Open global.md in $EDITOR
pan context edit --layer project    # Open this project's project.md
pan context edit --layer workspace  # Open this workspace's workspace.md (rare; mostly auto)
pan context sync                    # Render harness-specific outputs (alias for `pan sync` w/ context only)
pan context diff                    # Show what would change on next sync
pan context diff --harness claude   # Show what Claude would receive
pan context diff --harness pi       # Show what Pi would receive
pan context migrate                 # One-shot from old devroot
pan context validate                # Lint templates (unclosed blocks, unknown harnesses, etc.)
```

## Dashboard: Context Tab

New tab in the dashboard at `panopticon.localhost/context`:

- **Left rail:** layer selector (Global / Project / Workspace) and file tree (md files + skills/agents dirs)
- **Center:** Monaco editor with markdown highlighting and Mustache-block awareness (highlight `{{#harness:*}}` blocks)
- **Right rail:** three preview tabs
  - **Claude Code output** — what `~/.claude/CLAUDE.md` would look like after render
  - **Pi output** — what Pi receives (same in v1; future-proof)
  - **Full system prompt** — what an agent's full system + append + memory + briefing would look like (debug view)
- **Footer:** "Save & Sync" button runs `pan context sync` and shows a diff toast

## Wrapper Script Integration

The two harness wrappers handle the workspace-layer injection at spawn:

### Claude Code

`pan workspace create` / `pan start` generates a launcher script that does:

```bash
claude --append-system-prompt-file "$WORKSPACE/.panopticon/context/workspace.md" "$@"
```

The wrapper resolves the workspace at launch time and points the append-flag at the assembled bundle (which already contains project layer + memory + status).

### Pi

Pi's extension at `packages/pi-extension/` already reads lifecycle events. Extend it to load the same `workspace.md` at `session_start` and prepend it to the model's system prompt:

```typescript
// packages/pi-extension/src/index.ts (new addition)
async function onSessionStart(session) {
  const contextPath = join(workspaceRoot, '.panopticon/context/workspace.md')
  if (existsSync(contextPath)) {
    const content = await readFile(contextPath, 'utf-8')
    session.appendSystemPrompt(content)
  }
}
```

(API surface depends on what pi-mono exposes; may need a minor patch upstream.)

## Config Schema Changes

`~/.panopticon/config.yaml`:

```yaml
# Deprecated — emits warning on read. Will be removed in v2.0.
# sync:
#   devroot: ~/Projects/.claude

# New: layered context model
context:
  global:
    path: ~/.panopticon/context  # default; rarely overridden
  layers:
    global: true
    project: true
    workspace: true  # auto-assembled at spawn

  # Harness output targets (auto-discovered; override if needed)
  outputs:
    claude-code:
      home: ~/.claude
    pi:
      home: ~/.claude  # shares with Claude for now
```

## Acceptance Criteria

- `pan sync` no longer references `sync.devroot`; reading it from config emits the deprecation warning
- `~/.panopticon/context/global.md` exists after `pan install` (with a starter template)
- Editing `~/.panopticon/context/global.md` and running `pan sync` updates `~/.claude/CLAUDE.md`
- `{{#harness:claude}}` blocks render correctly when targeting Claude; absent when targeting Pi (and vice versa)
- Mixed blocks (`{{#harness:claude}}{{#harness:pi}}…`) render for both
- Workspace `workspace.md` is auto-assembled at `pan workspace create` time with project layer + memory placeholder + status
- `pan projects add ~/clients/tap2open` works regardless of filesystem location; project registered in projects.yaml
- Dashboard Context tab renders + edits all three layers; live preview panes update on change
- `pan context migrate` copies, registers, and prints next-step instructions; does NOT delete source
- `pan context validate` catches unclosed Mustache blocks and unknown harness names
- Existing `tests/cli/commands/sync-mirror.test.ts` updated to layered model; new tests for harness templating + workspace assembly

## Test Plan

Unit:
- `renderContext(layer, harness, content)` — block extraction correctness
- `assembleWorkspaceContext(workspace)` — composition order, deduplication
- `migrateDevroot(oldPath, newPath)` — idempotency, no-overwrite of existing target

Integration:
- `pan context migrate` end-to-end with fixture old layout
- `pan sync` materializes files to correct harness locations
- Dashboard Context tab edit → save → file on disk reflects change
- Spawn a Claude Code workspace; verify `--append-system-prompt-file` points at correct workspace.md
- Spawn a Pi workspace; verify Pi extension reads same workspace.md

## Out of Scope (Phase 2)

- Codex, Cursor, Gemini, OpenCode harness adapters (model supports them; not implemented in v1)
- `pan context git` (track global layer in user's own git repo)
- Per-user multi-machine sync (today, context is per-machine)
- Encrypted context layers (for secrets in context — use `~/.panopticon.env` for now)
