# PAN-800: Effect-native agent state consolidation

Single source of truth for agent runtime state. Canonical in-memory `SubscriptionRef` derived from the append-only `DomainEvent` stream. Files become bootstrap/persistence only. Hooks stop authoring state; they emit typed events. `tmux capture-pane` goes away as a consequence.

**Status:** Planned. PAN-800 is a complete, standalone deliverable. No cross-issue prerequisites. All development is halted on this until it ships end-to-end.

## Corrections to prior research

`PAN-798-research.md` and `PAN-798.md` are preserved for context but several of their conclusions do not hold up against the current codebase. PAN-800 replaces them; do not port their specifics forward uncritically.

1. **"Claude Code has no `SessionStart` hook event type" (PAN-759; referenced in PAN-798.md:33–46).** Wrong — it does. Evidence: `src/lib/caveman/workspace.ts:94–101` injects a `SessionStart` hook into workspace settings; `src/lib/caveman/__tests__/workspace.test.ts` asserts the injection; `src/lib/caveman/caveman-activate.js:3` is a live `SessionStart` hook script. The reason `~/.claude/settings.json` doesn't configure one today is that Panopticon's hook installer simply never registered it, not that the event type is missing.
2. **"Extend `runtime.json` with more fields" (PAN-798.md:80–93).** Preserves the root cause (files are canonical; three hooks read-modify-write the same JSON). PAN-800 inverts the model instead: canonical = event stream; file = non-canonical bootstrap fallback.
3. **"Add sibling JSON files per specialist: `review-results.json`, `merge-results.json`, `git-events.jsonl`, `test-results.json` (PAN-798.md:110–130).** Creates yet more file-based sources of truth. PAN-800 emits these as typed domain events in the existing `events` table and derives the snapshots from the fold. No new files.
4. **Event store table name (`event_log`)** — the actual table is `events` (see `src/dashboard/server/event-store.ts:131`). Earlier drafts had this wrong.
5. **"Add a new `POST /api/agents/:id/runtime-event` endpoint" — partial overstatement.** `POST /api/agents/:id/heartbeat` already exists at `src/dashboard/server/routes/agents.ts:764–782` but is a thin wrapper that writes to `runtime.json` via `saveAgentRuntimeState`. PAN-800 reshapes this route (same path) into the typed-event ingestion endpoint. A second path is not needed; the existing one is Schema-validated in place.
6. **`SubscriptionRef` is not yet used in the codebase.** The dashboard pattern today is `ServiceMap.Service` with closure-held state + `PubSub`/`EventEmitter` (see `src/dashboard/server/read-model.ts`). PAN-800 introduces `SubscriptionRef` for the runtime-state ref specifically, because atomic `update` + `.changes` stream is what we need to prevent the races the file-based hooks currently have. It is additive, not a rework of existing services.
7. **Append API choice.** `EventStore.append` is synchronous (blocking SQLite write); `appendAsync` batches via `setTimeout(..., 0)` and must be used from server-reachable code. PAN-800 writes use `appendAsync`; only startup bootstrap (one call to `readFrom(0)`) is allowed to be synchronous.

## Problem

### Five-way state duplication (current, verified)

| Location | Writer | Reader | Source of truth for |
| --- | --- | --- | --- |
| `~/.panopticon/agents/<id>/state.json` | CLI (`pan work start`), dashboard lifecycle routes | dashboard, CLI | `AgentState` config (workspace, runtime, model) |
| `~/.panopticon/agents/<id>/runtime.json` | bash hooks (`heartbeat-hook`, `stop-hook`, `work-agent-stop-hook`) and `saveAgentRuntimeState` | `getAgentRuntimeState` callers, dashboard, deacon | `activity`, `currentTool`, `claudeSessionId`, `waiting*` |
| `~/.panopticon/heartbeats/<id>.json` | `heartbeat-hook` only | dashboard health, cloister deacon | tool_name, last_action, git_branch, workspace, pid, timestamp |
| SQLite `agents` row | dashboard route handlers | dashboard routes | `status` (active/idle/stopped), `startedAt`, `branch`, `costSoFar` |
| `ReadModel.state.agentsById` (in-memory) | `applyEvent` in `read-model.ts` | RPC `getSnapshot` / `subscribeDomainEvents` | `AgentSnapshot` projection |

No single location has all the answers. The UI assembles an `AgentSnapshot` by merging four of the five. A change in any one silently diverges.

### Concrete failure modes

1. **Hook race on `runtime.json`.** Three hooks (`heartbeat-hook`, `stop-hook`, `work-agent-stop-hook`) each do `jq --arg ... '.a = $x | .b = $y'` then `mv temp runtime.json`. Parallel Claude tool calls fire hooks concurrently. We already hit this with cost events and had to wrap them in `flock` — same pattern will bite runtime state, but it has no `flock`.
2. **Blocking FS in the event loop.** `getAgentRuntimeState()` in `src/lib/agents.ts:399` uses `readFileSync`. It is called from Effect route handlers. This is the exact footgun CLAUDE.md prohibits ("No Blocking Calls in Dashboard Server Code").
3. **`AgentRuntimeState` is plain TypeScript** at `src/lib/agents.ts:359`. Not an Effect `Schema`. Not in `packages/contracts`. Frontend cannot share the type.
4. **`tmux capture-pane` is a universal fallback** for "what is this agent doing?" because there is no typed surface to ask. Twenty-two call sites, seventy-plus regexes.
5. **No `agent.*` runtime events.** `DomainEvent` has `agent.{created, started, stopped, status_changed, output_received, enrichment_changed}`, but nothing for per-tool-call activity, thinking, waiting, or model changes. Those signals live only in files.
6. **Ambient agent-ID inference (PAN-69).** Hooks resolve `AGENT_ID` from `$PANOPTICON_AGENT_ID || tmux display-message || "main-cli"`. When run outside tmux this can resolve to the wrong session. The HTTP ingestion endpoint fixes this by requiring the ID on the wire.

## Architectural principle

```
  writers                          canonical state                           readers
  -------                          ---------------                           -------

  session-start-hook       \
  heartbeat-hook            \
  stop-hook                  \     POST /api/agents/:id/heartbeat   (Schema-validated, typed body)
  specialist-stop-hook        -->       |
  work-agent-stop-hook       /          v
  cloister transitions      /     EventStore.appendAsync(AgentRuntimeEvent)
  review/test/merge        /             |
                                          v  (append-only, SQLite `events` table)
                                    EventEmitter fan-out
                                          |
                    +---------------------+----------------------+
                    v                     v                      v
           AgentStateService      ReadModel.applyEvent      WebSocket subscribers
           (SubscriptionRef)      (dashboard snapshot)      (frontend Zustand store)
                    |                     |                      |
                    v                     v                      v
           Effect route handlers   RPC getSnapshot / stream   same shared reducer
           cloister, deacon        subscribeDomainEvents      (T3Code pattern)
```

- **Canonical state** = `SubscriptionRef<HashMap<AgentId, AgentRuntimeSnapshot>>` inside `AgentStateService`.
- **Persistence** = the existing `events` SQLite table (7-day retention, monotonic sequence).
- **Bootstrap** = `EventStore.readFrom(0)` replayed into the ref once at service construction. No polling of `runtime.json`.
- **Files** = bootstrap fallback only. If the `events` table has no events for a known agent ID, the service reads its `runtime.json` once via `fs.promises.readFile` and emits a synthesized `agent.state_restored` event so the fold produces a snapshot. Subsequent state comes only from events.
- **Hooks** = emitters. They POST a typed event and exit. No `jq` read-modify-write. No direct file writes to `runtime.json` (file writes happen only as a POST-failure fallback buffer).

### Why `SubscriptionRef` specifically

`Ref` gives atomic `update`; `SubscriptionRef` adds a `.changes` stream for reactive consumers. That is the in-memory analog of what `EventStore` gives at the persistence layer: one writer, many readers, ordered. The existing `ReadModelService` uses closure-held state + `EventEmitter`, which works for "broadcast every event" but has no way to "read current state atomically from inside another `Effect.gen`". For runtime state we need both.

### Why HTTP ingestion (not file-watching)

- Runtime direction: Panopticon is moving toward Electron/npx (`memory/panopticon-electron-npx.md`). HTTP ingestion works in both; fs-watching doesn't port cleanly.
- One wire format, one Schema, one validation point. Bash hook becomes ~30 lines of `curl`; no `jq` read-modify-write.
- The route already exists (`POST /api/agents/:id/heartbeat`). We are repurposing it, not inventing one.

## `AgentStateService` — the single interface

```typescript
// src/dashboard/server/services/agent-state-service.ts (new)

export class AgentStateService extends ServiceMap.Service<
  AgentStateService,
  AgentStateServiceShape
>()("panopticon/dashboard/AgentStateService") {}

interface AgentStateServiceShape {
  get: (id: AgentId) => Effect.Effect<Option.Option<AgentRuntimeSnapshot>>
  getAll: Effect.Effect<HashMap.HashMap<AgentId, AgentRuntimeSnapshot>>
  changes: Stream.Stream<HashMap.HashMap<AgentId, AgentRuntimeSnapshot>>
  emit: (event: AgentRuntimeEvent) => Effect.Effect<void>
}

export const AgentStateServiceLive = Layer.effect(
  AgentStateService,
  Effect.gen(function* () {
    const eventStore = yield* EventStore
    const ref = yield* SubscriptionRef.make(HashMap.empty<AgentId, AgentRuntimeSnapshot>())

    // 1. Bootstrap from the persisted event log (synchronous readFrom is fine at startup).
    for (const ev of eventStore.readFrom(0)) {
      if (isAgentRuntimeEvent(ev)) {
        yield* SubscriptionRef.update(ref, (m) => applyAgentRuntimeEvent(m, ev))
      }
    }

    // 2. Bootstrap fallback:
    //    - First: query projection_cache for any persisted AgentRuntimeSnapshots and
    //      load agents with no live events in the retention window.
    //    - Then: for agents missing from both events and projection_cache, read
    //      ~/.panopticon/agents/<id>/runtime.json via fs.promises.readFile
    //      (never readFileSync — the zero-readFileSync success criterion applies
    //      here) and emit a synthesized agent.state_restored event. This path is
    //      only exercised during the migration from today's state.
    yield* seedRuntimeFromProjectionCache(ref, eventStore)
    yield* seedRuntimeFromFiles(ref, eventStore)

    // 3. Subscribe forward — fold every future AgentRuntimeEvent into the ref.
    yield* Effect.acquireRelease(
      Effect.sync(() => eventStore.subscribe((ev) => {
        if (isAgentRuntimeEvent(ev)) {
          // fire-and-forget: the service's single job is to keep the ref current
          Effect.runFork(SubscriptionRef.update(ref, (m) => applyAgentRuntimeEvent(m, ev)))
        }
      })),
      (unsubscribe) => Effect.sync(unsubscribe)
    )

    return {
      get: (id) => SubscriptionRef.get(ref).pipe(Effect.map((m) => HashMap.get(m, id))),
      getAll: SubscriptionRef.get(ref),
      changes: ref.changes,
      emit: (event) => Effect.promise(() => eventStore.appendAsync(event)).pipe(Effect.asVoid)
    }
  })
)
```

Writers use `AgentStateService.emit(event)`. Readers use `.get(id)` or subscribe to `.changes`. That is the entire public surface.

Invariants enforced by this service:

- The `HashMap` inside the ref is only mutable via `applyAgentRuntimeEvent` — i.e., via the event reducer. There is no public `set(id, snapshot)`. Every field mutation goes through one of the seven runtime events.
- `.emit` is async (`appendAsync` under the hood); the caller awaits durable append before proceeding. HTTP handlers stay non-blocking.
- Bootstrap is a one-shot, called once at Layer construction. The forward-subscription is a daemon.
- Every successful fold writes the resulting `AgentRuntimeSnapshot` to `projection_cache` keyed by `agentId`. This preserves stopped-agent state across the 7-day event-log retention window.

## HTTP ingestion endpoint

Replace the body of `POST /api/agents/:id/heartbeat` (`src/dashboard/server/routes/agents.ts:764–782`) with a Schema-validated, event-emitting handler. Same path; new semantics.

```typescript
const RuntimeEventBody = Schema.Union(
  Schema.Struct({ kind: Schema.Literal("activity"), activity: Activity, tool: Schema.optional(Schema.String) }),
  Schema.Struct({ kind: Schema.Literal("thinking_start"), lastToolAt: Schema.String }),
  Schema.Struct({ kind: Schema.Literal("thinking_stop"), resolvedBy: Schema.Literal("tool", "waiting", "idle", "stopped") }),
  Schema.Struct({ kind: Schema.Literal("waiting_start"), reason: WaitingReason, message: Schema.optional(Schema.String) }),
  Schema.Struct({ kind: Schema.Literal("waiting_clear"), clearedBy: Schema.Literal("user_response", "timeout", "stopped", "tool_resumed") }),
  Schema.Struct({ kind: Schema.Literal("message_received"), direction: Schema.Literal("to_agent", "from_agent"), source: Schema.Literal("user", "cloister", "specialist", "automated") }),
  Schema.Struct({ kind: Schema.Literal("model_set"), model: Schema.String, claudeSessionId: Schema.optional(Schema.String) })
)
```

One handler decodes the body, translates to the matching `agent.*` event, and calls `AgentStateService.emit`. The legacy body shape `{state, tool, timestamp}` that the current heartbeat endpoint accepts is mapped to `{kind: "activity", activity: state, tool}` for a short compatibility window, then removed in the same PR once all hooks have migrated.

### Why POST-per-event, not PUT of a runtime snapshot

PUT of the full snapshot re-introduces the read-modify-write problem at the HTTP layer. `emit` of atomic transitions preserves append-only semantics and gives replay for free.

### Backpressure

If the dashboard is down, POST fails. Hooks must not block Claude Code. The fallback:

1. POST with 0.5s timeout.
2. On failure, append the request body as a JSONL line to `~/.panopticon/agents/<id>/pending-events.jsonl`.
3. On the next successful POST, replay pending lines first (oldest first) and truncate the file.

The dashboard already makes sequence numbers server-assigned, so replayed events are distinguishable and ordered deterministically. No idempotency key needed beyond the server-side sequence.

**Concurrency.** Claude Code fires parallel tool calls, which spawn concurrent hook processes. The append *and* the replay paths both need `flock -x` around the `pending-events.jsonl` file — same treatment cost-event recording already uses (`$HOME/.panopticon/costs/state/<session>.lock`). Without the lock, two hook processes could both read the file, both succeed on their next POST, and replay the same events twice (not catastrophic, because server sequences are unique, but sequence inflation and log spam). A single per-agent lock at `$HOME/.panopticon/agents/<id>/pending.lock` serializes both operations.

## Hook rewrite protocol

All installed hooks become POST emitters. No `jq` read-modify-write on `runtime.json`. `runtime.json` continues to be written *only* as the POST-failure fallback buffer mentioned above (separate from the `state.json` config file, which is a different concern).

| Hook | Fires on | Emits |
| --- | --- | --- |
| `session-start-hook` (**new, installed by PAN-800**) | `SessionStart` | `agent.model_set` if model is inferable from env; `agent.activity_changed` to `idle` |
| `pre-tool-hook` | `PreToolUse` | `agent.thinking_stopped` (`resolvedBy: "tool"`); `agent.activity_changed` to `working` with tool name |
| `heartbeat-hook` | `PostToolUse` | `agent.activity_changed` with current tool (so "working" re-stamps while tool chains continue); cost-event recording stays as today |
| `notification-hook` | `Notification` | `agent.waiting_started` when the notification indicates a permission/question prompt |
| `stop-hook` | `Stop` | `agent.activity_changed` to `idle`; `agent.waiting_cleared` if prior `waiting`; still chains to specialist/work-agent hooks |
| `specialist-stop-hook` | chained from `stop-hook` | `specialist.completed` (existing) |
| `work-agent-stop-hook` | chained from `stop-hook` | `agent.activity_changed` to `idle` + existing nudge behaviour |

`heartbeat-hook` today is ~200 lines of `jq`. After this change it is ~40 lines: resolve `AGENT_ID`, assemble the typed body, `curl -X POST` with 0.5s timeout, on failure append to `pending-events.jsonl`. Cost-event recording stays on its own `flock`-serialized path.

### The missing `session-start-hook`

Claude Code supports `SessionStart` (proven by `src/lib/caveman/workspace.ts:94–101` and `src/lib/caveman/caveman-activate.js`). Panopticon's hook installer at `src/cli/commands/setup/hooks.ts` does not register it. PAN-800 adds it to:

- `hookScripts` array (line ~151)
- `hooksAlreadyConfigured()` check (line ~93)
- settings.json registration block (lines 256–322)

Installing the hook is part of PAN-800's work; no separate prerequisite issue.

## Consolidation: what is deleted

Direct consequences of PAN-800 landing:

- **`getAgentRuntimeState()`** (`src/lib/agents.ts:399`) — deleted. All server callers go through `AgentStateService.get(id)`. The CLI gets a thin wrapper that calls `GET /api/agents/:id/runtime` if the dashboard is up, falling back to `fs.promises.readFile` of `runtime.json`.
- **`getAgentRuntimeStateAsync()`** (`src/lib/agents.ts:438`) — deleted, same reason.
- **`saveAgentRuntimeState()`** — deleted from the happy path. Retained only as the `pending-events.jsonl` fallback writer.
- **`~/.panopticon/heartbeats/`** directory writers (in `heartbeat-hook`) — deleted. `runtime.json` subsumes the signal.
- **`confirmDelivery`** (`src/lib/tmux.ts`) — deleted. Replaced by checking `AgentStateService.get(id)` for `activity` ∈ {"working", "thinking"} after the last `agent.message_received` event.
- **`parseThinkingDuration`** — deleted. `ThinkingState.since` is a timestamp; duration is `now - since`.
- **`checkLazyAgent`** with its 34 regexes (`src/lib/cloister/deacon.ts`) — deleted. Replaced by: `activity === "idle"` AND `!waiting` AND `now - lastActivity > threshold`.
- **`checkStuckWorkAgents`** — simplified to the same predicate.
- **All `tmux capture-pane`** calls in `src/lib/cloister/` and `src/dashboard/server/` — migrated one-for-one to `AgentStateService` reads, per `PAN-798-audit.md`. After migration the function references are deleted.
- **`readFileSync(runtime.json)`** everywhere in server code — deleted. The bootstrap-fallback path in `AgentStateService` uses `fs.promises.readFile`; the zero-`readFileSync` success criterion applies to the fallback too.

What stays:

- **`state.json`** — `AgentState` *config* (workspace, runtime, model selection). Not runtime state. Unaffected.
- **`activity.jsonl`** (per-agent append-only tool log) — redundant with the event log but kept as a cheap, `grep`-able forensic artifact.
- **Claude Code JSONL session files** — SACRED, untouched.
- **SQLite `agents` row** — still the source of truth for lifecycle fields (`startedAt`, `branch`, `costSoFar`). Runtime fields come from the service.

## Delivery plan (one PR, five phases internal to PAN-800)

PAN-800 ships as one complete feature. The phases below are implementation ordering inside the issue — they are not separate shippable sub-issues. A partial landing is not valuable: consumers are still tied to the old paths, and the old and new surfaces disagree.

**Phase 1 — Contracts.** Land `AgentRuntimeSnapshot`, `Activity`, `ThinkingState`, `WaitingState` in `packages/contracts/src/types.ts`. Land the seven `agent.*` events plus `agent.state_restored` in `events.ts`. Extend `event-reducers.ts` to fold them. Unit-test the reducer against every event variant.

**Phase 2 — Service.** Ship `AgentStateService` (Layer, ServiceMap, `SubscriptionRef`). Wire into the dashboard server composition alongside `ReadModelService`. Subscribe to the event store. Verify bootstrap-from-empty (zero agents) and bootstrap-from-populated (replay N events, final `HashMap` matches expected).

**Phase 3 — Ingestion endpoint + SessionStart install.** Reshape `POST /api/agents/:id/heartbeat` into the typed-event ingestion handler. Add the legacy-body mapping for the transition. Install `session-start-hook` and register it in `~/.claude/settings.json` via the `pan setup hooks` path.

**Phase 4 — Hook migration.** Rewrite `heartbeat-hook`, `stop-hook`, `pre-tool-hook`, `notification-hook`, `specialist-stop-hook`, `work-agent-stop-hook`, and `session-start-hook` to the POST-emitter protocol. Preserve `pending-events.jsonl` fallback (with `flock`). The rewritten scripts live in `scripts/` in the repo, but the *running* system executes from `~/.panopticon/bin/`; Phase 4 is not done until the hooks have been reinstalled via `pan setup hooks` (which copies the scripts and updates `~/.claude/settings.json` to register `session-start-hook`). The definition-of-done includes running the installer, not just editing the scripts. Verify via running agents end-to-end.

**Phase 5 — Consumer migration + cleanup.** Migrate the 22 `tmux capture-pane` call sites from `PAN-798-audit.md` to `AgentStateService` reads. Migrate all `getAgentRuntimeState`/`getAgentRuntimeStateAsync` callers. Delete `confirmDelivery`, `parseThinkingDuration`, `checkLazyAgent`, `checkStuckWorkAgents`, heartbeat file writers, and `tmux capture-pane` imports. Remove the legacy body-shape mapping from the HTTP handler.

Ordering is load-bearing:
- Contracts before service (service imports the types).
- Service before ingestion endpoint (endpoint calls `AgentStateService.emit`).
- Ingestion endpoint before hook migration (hooks POST into the typed endpoint).
- Hook migration before consumer migration (consumers read snapshots that are only populated by migrated hooks).
- Consumer migration before cleanup (removing `capture-pane` before callers migrate breaks the build).

## Open questions to resolve during implementation

1. **CLI reads from HTTP or file?** `pan status` runs outside the dashboard process. Simplest: CLI calls `GET /api/agents/:id/runtime`. If the dashboard is down, fall back to `fs.promises.readFile(runtime.json)` for lifecycle context only, and print "runtime state unavailable". Decide once, apply uniformly.
2. **Frontend runtime store timing.** Reducer cases can land in Phase 1 (no harm — no events fire until Phase 4). Frontend subscription wiring lands in Phase 5. Don't couple them.
3. **Retention for stopped agents — resolved.** Event log is 7 days; if we do nothing, a stopped agent's `AgentRuntimeSnapshot` disappears after compaction. This is a **regression from today** (the existing `runtime.json` on disk persists indefinitely, and `getAgentRuntimeState()` happily reads a years-old one). Resolution: on every event, `AgentStateService` upserts the current `AgentRuntimeSnapshot` into the existing `projection_cache` table keyed by `agentId` (the same table `EventStore` already uses). On bootstrap, agents with no live events in the 7-day window are seeded from `projection_cache` instead of (or in addition to) `runtime.json`. Stopped-agent snapshots persist indefinitely, matching current behaviour. The fallback file read from `runtime.json` remains for the one-time migration from today's state.
4. **`runtimeSnapshotSequence` re-render volume.** Every runtime event bumps `AgentSnapshot`, potentially spamming frontend diffs. If profiling shows this is hot, split the frontend into two stores: `agentsById` (lifecycle) and `agentRuntimeById` (runtime). Defer until we have numbers.

## Non-goals

- **Specialist runtime state.** Specialists have a similar five-way-duplication problem but their lifecycle is different. A sibling issue (not this one) migrates specialists after PAN-800 proves the pattern.
- **`pan audit` CLI.** The event log will satisfy the audit-trail half of PAN-198; the query CLI is a separate surface feature.
- **Restructuring `AgentSnapshot`.** Only one field added (`runtimeSnapshotSequence`); existing fields unchanged.
- **Cost-event recording changes.** Stays on its own `flock`-serialized path.
- **Active intervention (PAN-188).** PAN-800 makes intervention reliable but does not implement it.
