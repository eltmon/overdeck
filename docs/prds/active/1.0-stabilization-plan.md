# Panopticon 1.0 Stabilization Plan

**Status:** Planning complete, ready for execution
**Created:** 2026-04-22
**Owner:** eltmon
**Target release:** 1.0

## Context

During PAN-698 (typography work) the work agent attempted a destructive `mv` of `.pan/`, `.planning/`, and `.claude/skills/` directories to `/tmp/`. User pressed ESC and halted the operation. Investigation revealed the `mv` was a symptom ŌĆö the actual cascade is architectural.

## Root cause cascade

1. GitHub labels are used as the state machine source of truth for issue/review status.
2. State writes are fire-and-forget ŌĆö `fetch()` without `.ok` check, no retry, no audit log.
3. On every dashboard boot, 5 `repair*` functions fire in parallel against the GitHub API (for every issue the system knows about ŌĆö 600+ calls observed).
4. On every work-agent spawn, `transitionIssueToInProgress` fires unconditionally (no idempotency check).
5. GitHub secondary rate limits hit silently; label mutations drop on the floor.
6. PAN-698's `in-review` transition silently failed. Today's respawn then clobbered the state back to `in-progress`.
7. Work agent, following `work.md:203` literally, ran `git fetch origin main && git rebase origin/main` manually ŌĆö contradicting the `rebase-and-submit` skill which says never rebase manually.
8. Workspace had untracked files (`.pan/`, `.claude/skills/*`) blocking the rebase. Agent decided to `mv` them to `/tmp/` as a workaround.

## Decisions (resolved 2026-04-23)

1. **PAN-676 mechanism** ŌåÆ **(b) Remove `Closes #NNN` from PR bodies**. Panopticon owns the close explicitly via API after merge. Keeps GitHub labels trustworthy for anyone browsing the repo directly. Handled in Epic A (#805). If a PR is ever merged outside Panopticon (GitHub web UI), the reconciler catches it on its next sweep.
2. **Uncommitted `issue-lifecycle.ts` edits** ŌåÆ **Salvaged** to branch `refactor/issue-status-events` (commit `812cc2cf`, pushed to origin). Follow-up ticket: #808 tracks the eventual PR into main. The workspace `feature-pan-698` can now be safely reset in Epic D.
3. **Execution sequence** ŌåÆ Epic D first, then A/B/C on solid foundation (confirmed by user).

### Epic A ŌĆö Labels as display, internal state as source of truth

Replace fire-and-forget label writes with a reconciler. GitHub labels become a display mirror; internal SQLite state is authoritative.

**Acceptance criteria:**
- [ ] `issue_state` table added to SQLite (columns: issue_id, canonical_state, last_synced_at, pending_mutation, updated_at)
- [ ] Reconciler service runs on fixed interval (e.g. 30s), diffs desired labels vs actual labels, writes ONLY deltas
- [ ] Reconciler checks `response.ok` on every API call, retries with exponential backoff on rate limit (Retry-After header honored)
- [ ] Reconciler writes to `label_sync_audit` table on every attempt (success or failure, with reason)
- [ ] All 5 `repair*` functions deleted from `src/dashboard/server/main.ts` boot path
- [ ] `transitionIssueToInProgress` becomes idempotent (no-op if already in-progress per internal state)
- [ ] Silent fetch failures in `src/dashboard/server/done.ts` / `updateGitHubToInReview` fixed (explicit `.ok` checks + error logging)
- [ ] All label writes go through reconciler ŌĆö zero direct `gh issue edit` calls outside reconciler
- [ ] PAN-676 resolved per chosen mechanism (option (b) or (c) ŌĆö see Open Decisions above)
- [ ] Integration test: issue state survives 1000 agent respawn simulations without label flood

### Epic C ŌĆö Workspace state sanity on spawn

Spawn flow stops destroying local state. Pre-flight checks guarantee the workspace is safe before an agent starts.

**Acceptance criteria:**
- [ ] Root cause identified: the specific code path that hard-reset `feature/pan-698` at 2026-04-22 15:30:09 (reflog-confirmed event)
- [ ] Pre-flight check before spawn: `git fetch origin <branch>`, compare local vs remote
- [ ] Pre-flight aborts spawn with clear error if local ahead-of-remote (refuses to risk losing unpushed work)
- [ ] Pre-flight aborts spawn if untracked files exist outside ignore set (no more `.pan/` ŌåÆ `/tmp/` attempts)
- [ ] When stashing planning debris pre-spawn: stash message format includes issueId + ISO timestamp + `salvageable` flag (for Epic D-style audits going forward)
- [ ] Integration test: spawn on ahead-of-remote branch fails fast with error (no reset occurs)
- [ ] Integration test: spawn on clean branch succeeds normally
- [ ] No hard-reset of local branches anywhere in spawn flow (confirmed via code review + reflog test)
