## Issue Details
- **ID:** PAN-506
- **Title:** Start Agent silently fails when beads DB not initialized — no error shown to user
- **URL:** https://github.com/eltmon/panopticon-cli/issues/506

## Description
## Summary

Clicking \"Start Agent\" does nothing when the beads Dolt database hasn't been initialized in the workspace. The API returns a 422 error but the dashboard shows no feedback to the user.

## Steps to Reproduce

1. Run planning for an issue — planning session completes
2. Click \"Start Agent\" in the dashboard
3. Nothing happens — no spinner, no error message

## Root Cause

The \`POST /api/agents\` endpoint calls \`bd list --json -l <issue-id>\` to check for beads. If the Dolt database doesn't exist (never initialized), this fails with:

\`\`\`
failed to open database: database "beads" not found on Dolt server
\`\`\`

The endpoint returns a 422, but the frontend's Start Agent button doesn't surface the error.

## Why the DB might not exist

The planning flow creates a \`.beads/redirect\` file pointing to the project root's \`.beads/\`, but \`bd init\` is never called in the workspace. The Dolt server has no database for the issue prefix until \`bd init --prefix <issue>\` is run.

## Fix

Two-part:
1. **Frontend**: Show the error message from 422 responses on the Start Agent button (same pattern as other mutation error states)  
2. **Backend**: Either auto-run \`bd init\` when the database is missing, or ensure \`complete-planning\` verifies the DB exists before calling \`createBeadsFromVBrief\`

## Workaround

\`\`\`bash
cd workspaces/feature-pan-<id>
bd init --prefix pan-<id>
bd backup restore  # if backup files exist
# Then trigger complete-planning to recreate beads from vBRIEF:
curl -X POST http://localhost:3011/api/issues/PAN-<id>/complete-planning -H 'Content-Type: application/json' -d '{"skipKill": true}'
\`\`\`
