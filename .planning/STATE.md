## Key Decisions
- D1: Implementing Phase 1 first (server-side fan-out fix + JSONL fix) — these are root causes; without them the tree shows ghost reviewers and conversations never load. Frontend liveness is layered on top.
- D2: Existing CommandDeck dir at `src/dashboard/frontend/src/components/CommandDeck/` already has IssueHeader/SessionPanel/ConversationList — Phase 2 is therefore a refactor, not a greenfield build. Keeping existing components when they map cleanly to Zone A/B/C; introducing new ZoneA/ZoneB/ZoneCConversation/ZoneCOverview components for the additive surfaces (sparkline, ToolFlash, tabbed overview).
- D3: Round metadata is read from `~/.panopticon/agents/<reviewer-id>/round-N.json` artifacts written by `archiveReviewerRound`. The aggregated state is exposed via the existing command-deck snapshot endpoint.
- D4: Liveness components are pure presentational (no data fetching, no event subscriptions) — Phase 2 components will plumb data into them. `ReviewerRole` is defined locally in `RoleBadge.tsx` rather than imported from `@panopticon/contracts` (which doesn't export it) or from `src/lib/cloister/specialists.ts` (server-side, cross-workspace import boundary). Keep the duplicate small and stable; widen the contracts package later if needed.
- D5: Per-issue session selection lives in a sibling Zustand store (`useCommandDeckSelection`) rather than the existing `useDashboardStore`. Rationale: `useDashboardStore` is reduced from contracts-shared event reducers; coupling UI selection to that pipeline would force the contracts package to model UI state. The sibling store is local-only, type-safe, and trivially testable. `selectedSessionId` in `CommandDeck/index.tsx` is *derived* from the slice (read by the active `selectedFeature`) so the existing `selectedSessionId` prop on `<ProjectNode>` continues to work without any tree-API change.
- D6: `roundMarkers` plumbing on the MessagesTimeline is intentionally deferred to pan-y6ge — `ZoneCConversation` is a one-line `<SessionPanel>` wrapper for now. Phase 2's demo criterion ("click any session → Zone B updates and Zone C shows the conversation") is met by the slice + tree wiring without live event subscription, since session data already flows from `subscribeProjectSessionTree`.
- D7: PR/Diff and Discussions tab content deferred to follow-up beads (pan-9yn5, pan-1r7j) — neither endpoint exists yet (`gh pr view --json`/`gh pr diff` shell-out + Linear+GitHub combined timeline) and shipping placeholders today is honest about the state. Tabs render `<DeferredTab>` so the navigation surface is complete and discoverable.
- D8: INFERENCE tab visibility is conditional on `planning.data.inference` being non-empty. Empty INFERENCE.md is the steady state for most issues today; hiding the tab keeps the strip from advertising a placeholder. Same single `usePlanningQuery` powers tab visibility + the rendered body, so the cache is shared.
- D9: `OverviewTab.tsx` derives "stage" from the most recent active session's role/type rather than a server-side stage field — this avoids adding a new endpoint just for the billboard. When a verification-gate stage feed lands later, swap `deriveStageFromSections` for the canonical source.
- D10: `roundMarkers` is shipped as a transparent pass-through prop from `MessagesTimeline` up through `ConversationPanel` → `SessionPanel` → `ZoneCConversation`. The bead deliberately does NOT derive markers from reviewer `roundMetadata` because deriving them requires mapping each round artifact to a specific JSONL message id, which depends on review-agent log conventions still in flux. Shipping the wiring now means a single follow-up can drop a `useRoundMarkers(issueId, sessionId)` hook into `ZoneCConversation` without touching the timeline component itself.
- D11: Round-divider rows are not separate virtualizer indices — instead each marker is rendered inside the wrapper of the row whose `id` it follows. The virtualizer's `measureElement` includes the divider's height automatically, which keeps the divider non-virtualized (always present in the DOM) without forcing a parallel marker-row index that would have to participate in `getItemKey`/`estimateSize`/`measureElement`. Same approach works for the non-virtualized tail rows via a `<Fragment>`.
- D12: The `ResizeObserver` jsdom shim was promoted from `XTerminal.test.tsx` into the global `src/test-setup.ts`. Reason: at least four components (MessagesTimeline, XTerminal, GodView CanvasTerminal/ConnectionLines) instantiate `new ResizeObserver(...)` in `useLayoutEffect`, so any future test that renders one of them needs the shim. A single global no-op stub is cleaner than per-file setup and matches the existing pattern for `scrollIntoView` and `matchMedia`.
- D13: PR/Diff endpoint shells out to `gh` instead of going through the GitHub REST API. Reason: the existing route (`closeIssuePullRequest`, lines 136-165 in issues.ts) already uses `gh pr list/close` via `execAsync`, so the user is already authenticated. Adding a parallel REST/GraphQL pipeline would require GITHUB_TOKEN management + new client wiring + duplicate path resolution. Three sequential shell-outs (list → view → diff) keep the route under 60 lines and reuse `isGitHubIssue` for repo resolution. 30s `refetchInterval` matches the costs cadence — PR data doesn't change second-by-second, and the user can drive a manual refresh by switching tabs (the queryClient cache is keyed per-issueId).
- D14: Discussions endpoint uses partial-success `errors[]` instead of returning HTTP 500 when any one source fails. Reason: 5 sources × ~30 polls/hr × N viewers means a single rate-limited GitHub call would otherwise blank the entire feed. The Linear comments and the issue conversation are independent of the PR review pipeline; collecting per-source failures into a `string[]` lets the UI render every source that did succeed and show a non-blocking warnings strip listing the failures. The frontend treats `errors[]` as advisory — items that did resolve still render, and the React Query cache still hands the data back on the next 30s refetch attempt. The Effect `LinearClient` lookups are injected into `fetchIssueDiscussions(deps)` so route-level tests can exercise Linear paths without mocking the Effect service map (mirrors how the route handler injects them via `Effect.runPromise`).
